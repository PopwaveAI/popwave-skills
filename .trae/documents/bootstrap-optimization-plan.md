# Bootstrap 优化执行计划

---

## 一、总结

基于优化 PRD 中列出的 12 个方向（A-L），通过三本差异度大的小说（遮天/吞噬星空/深渊主宰）模拟 L1 分类反推验证后，确认以下结论并形成执行顺序。

## 二、验证结论

### 2.1 L1 六件套分类缺陷已确认（原方向 C）

模拟反推验证结果：

| 原有维度 | 问题 | 严重度 |
|:--------|:-----|:------|
| 01-世界观底层逻辑 | **过载**。法则/时空/力量/矛盾/信仰/位面全部塞在一起 | 致命 |
| 02-世界观表层规则 | **与01边界模糊**。无法一致判定什么是"底层"什么是"表层" | 致命 |
| 01+02 | **二分法脱离创作直觉**。网文作者想的是"力量怎么升级？发生过什么大事？谁说了算？"而非"这是底层还是表层" | 致命 |
| 04-金手指 | **错误地设为独立+可跳过**。遮天的荒古圣体不是外挂而是世界观本身；在03(种族)和04(金手指)间反复拉扯 | 严重 |
| 无 | **修炼/力量体系无独立维度**。被迫挤在01中，但这是三部小说中最核心的设定 | 严重 |
| 无 | **历史背景/大事件无维度收纳**。遮天成仙路、吞噬大涅槃、深渊圣者浩劫——三部都有前提事件但无处处理 | 严重 |

### 2.2 新分类方案已验证

```
01 · 世界蓝图         (世界的基本面貌：时空结构、物理/魔法规则、世界基调)
02 · 力量体系         (如何变强：境界等级、修炼路径、职业体系、力量天花板)
03 · 历史与驱动力      (世界的前因后果：纪元划分、大事件、核心矛盾)
04 · 物种与天赋        (谁生活在这里：种族、异兽、特殊体质/血脉/天赋)
05 · 势力格局          (谁说了算：势力组织、权力分布、派系关系)
06 · 资源与物品        (什么好东西：关键道具、战略资源、货币体系)
```

**金手指改为 `[主角]` 标签**：嵌套在各维度的对应条目中标注，不设独立维度。

三部小说验证全部通过（详见规划阶段产出）。

### 2.3 Phase 0 问题已确认

灵魂三问（为什么是我写/读者为什么非看不可/核心情绪）是内向自省，产出 PRD.md 后无下游消费。

正确的方向：用户说"我想写灰骑士穿越博德之门3" → agent 帮用户完善想象 → 产出结构化 engine.yaml（各下游管线可直接消费）。

## 三、执行顺序

### Phase 1：Phase 0 重写 + Reader Profile 直出

**涉及文件**：
- `phases/phase-0.pe.md`（全量重写）
- `phases/phase-0.ref.md`（全量重写）
- `phases/phase-4.pe.md`（删除或降级）
- `SKILL.md`（更新步号和质量红线）

**做什么**：

1. 删除"灵魂三问"，改为：
   - Agent 接住用户的原始想象（一个场景/一个画面/一个概念）
   - 用追问帮用户完善：具体场景？核心冲突？世界观基调？
   - 结构化输出 **story-engine.yaml** 取代 PRD.md

2. `story-engine.yaml` schema：
   ```yaml
   engine:
     core_premise: "一句话：一个XX的人，想做XX，但XX挡着"
     world_anchor: "时间/空间/力量基调（凡人→超凡→破格 三选一）"
     core_conflict_type: "战斗/解谜/生存/种田/复仇/守护（可多选）"
     reader_promise: "读者每5章期待到什么"
     protagonist:
       one_line: "一句话描述"
       driving_force: "他想什么/怕什么/要什么"
   ```

3. **Reader Profile 从 Phase 4 挪到 Phase 0 直出**——不再从 PRD.md 二次提取。Phase 4 降级为"确认/校对"步骤。

4. 更新 SKILL.md：
   - 删除"灵魂三问"相关红线
   - 新增 `story-engine.yaml` 产出要求
   - 前 5 条红线改为 story engine 相关检查

**验证**：产出 story-engine.yaml，确认 plot/writer/QC 三个管线能直接读取其中字段作为输入。

### Phase 2：L1 分类重设计 + Phase 1 全量重写

**涉及文件**：
- `phases/phase-1.pe.md`（全量重写）
- `phases/phase-1.ref.md`（全量重写）
- `phases/phase-1.2.pe.md`（全量重写）
- `phases/phase-1.2.ref.md`（全量重写）
- `phases/phase-1.3.pe.md`（全量重写或删除）
- `phases/phase-1.5.pe.md`（全量重写）
- `SKILL.md`（更新产出物和红线）

**做什么**：

1. L1 分类从旧六件套改为新六件套：
   - 01-世界蓝图
   - 02-力量体系
   - 03-历史与驱动力
   - 04-物种与天赋
   - 05-势力格局
   - 06-资源与物品

2. 每份文件给 schema + before/after 示例：
   - 每个维度列出 4-6 个必填子字段
   - 每个子字段搭配：好的写法示例（from 遮天/吞噬/深渊）+ 坏的写法示例
   - 字数建议（不是硬门槛，是引导）

3. **金手指标签化**：在各维度中用 `[主角]` 前缀标注主角专属内容。例：
   - 02-力量体系：[主角] 荒古圣体修炼路径——比常人多九个小台阶
   - 04-物种与天赋：[主角] 金角巨兽分身天赋——基因倍数×10
   - 每篇文件末尾附 `[主角]` 标签汇总清单（跨维度的主角专属信息一览）

4. Phase 1.5（世界稳定性检验）从自问模式改为 **checklist 模式**（参考 plot 的节奏自检）：
   ```
   □ 力量体系与世界蓝图矛盾？（如：02中的境界突破条件与01中的法则冲突）
   □ 历史事件中的因果链完整？（03中的大事件是否有明确的触发-发展-结果）
   □ 势力之间的关系有张力？（05中的势力不是"各管各的"而是有互动/对抗）
   □ 物种的分布符合世界蓝图？（04中的种族出现在01设定的正确时空）
   □ [主角] 特殊能力在该世界中存在理论支撑？
   □ 是否有"一次性设定陷阱"？（用一次就忘的设定）
   □ 力量体系是否有明确的天花板和突破路径？
   ```

5. Phase 1.3（交叉关联）改为：
   - 不往每篇文件插污染性的 `【关联：XX】` 标注
   - 改为产出独立的 `_交叉引用记录.md`，记录三件事：
     a. 已确认一致的对子
     b. 发现的矛盾点
     c. 待确认（超出当前幕范围的设定）

**验证**：用遮天/吞噬/深渊三部小说分别按新 schema 产出 L1，确认信息量均衡、不透支。

### Phase 4：剩余 Phase 模板化 + 量化标准重写

**涉及文件**：
- `phases/phase-0.3.pe.md` + `.ref.md`（差异化模板 + 示例）
- `phases/phase-0.4.pe.md` + `.ref.md`（加 before/after 示例）
- `phases/phase-0.5.pe.md` + `.ref.md`（量化重写）
- `phases/phase-3.pe.md` + `.ref.md`（constitution.yaml 深度模板）
- `phases/phase-5.pe.md` + `.ref.md`（3个 yaml 字段模板）
- `SKILL.md`（同步更新）

**做什么**：

1. **Phase 0.3**（参考书甄别）：
   - 差异化三列表加模板：
     ```
     ✅ 借鉴点 | [一句话] | 理由 | 我们会怎么做
     ❌ 偏离点 | [一句话] | 我们的判断依据
     🆕 空白 | [一句话] | 为什么我们认为市场有这个需求
     ```
   - 加一个差异化示例（已从遮天/吞噬/深渊的对比中得到）

2. **Phase 0.4**（金手指）：
   - 保持原有的五问 + 四级评估
   - 新增 before/after 示例：
     ```
     ❌ 坏的金手指：系统签到流——每天自动签到送经验，主角不需要做任何决策
     ✅ 好的金手指：荒古圣体——给了主角"更强"的可能，但也需要付出"更长的修炼时间"的代价
     ```
   - 补充金手指标签化的使用说明

3. **Phase 0.5**（跨域素材）：
   - 删除"领域≥4/种子≥8/字数≥1500"等数字游戏标准
   - 改为质量导向的三层检查：
     ```
     L1：至少 2 个种子被明确标记为"已嵌入 L1 的哪个文件哪段"
     L2：至少 1 个种子是该领域独有的结构（不是"魔法=火焰"这种通用知识）
     L3：嵌入 L1 后，找一段正文试写确认能不能自然融入
     ```

4. **Phase 3**（项目骨架）：
   - constitution.yaml 从"一句话"加深度模板：
     ```yaml
     tone: "热血/黑暗/轻松/悬疑"  # 核心情绪基调
     cant_do:
       - rule: "主角不杀无辜"
         example: "ch12 放过俘虏"
         reason: "主角人格定位"
     must_do:
       - rule: "每20章有段位晋升"
         example: "ch1-20 凡人→超凡"
         reason: "平台节奏要求"
     ```

5. **Phase 5**（数值体系）：
   - 3 个 yaml 文件加字段模板（combat_capability / monster_rank_map / act_rank_schedule）

## 六、关键决策记录

| 决策 | 选择 | 原因 |
|:----|:-----|:------|
| L1 分类方案 | 5 维度 + 1（新 6 件套） | 三本书验证通过，正交性优于原方案 |
| 金手指处理 | 标签化 `[主角]` | 不设独立维度，避免跨维度拉扯 |
| Phase 2/6 | 删除 | plot skill 已覆盖，不重复 |
| Phase 4 去向 | reader_profile 直出 + Phase 4 降级校对 | 避免 Phase 0→Phase 4 绕路 |
| 量化标准 | 从数字游戏改质量导向三层检查 | 数字达标不等于质量达标 |
| 交叉引用 | 独立 `_交叉引用记录.md` | 不在 L1 文件中插污染性标注 |
| Reverse 事件日志 | 增加 L1_candidates 对齐 forward | 保障 forward/reverse 数据可互通 |
