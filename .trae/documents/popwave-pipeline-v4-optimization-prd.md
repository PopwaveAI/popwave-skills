## 二、攻击面 A：小说质量门禁（pop-novel-qa 双层改造）

### 2.1 证据链

**g113593**：quality-checklist.md 提供 8维×10分=80分制，每维度下有具体可判断的是非题，>60分交付、>70分优秀。这是唯一一个把"质量"做成可量化门禁的 skill。

**fiction-crafter**：6步审查——快速审查(3维) + 完整审查(9维)，含穿帮检测清单(死亡角色复活/地点矛盾/等级倒退/时间线错误)、套路重复检测(与前5章相似度，>80%预警)、角色成长曲线追踪、伏笔超期预警(40/30/20章分级)、配角消失预警(15章)。还有一个"总体评级规则"：优秀(全维度正常)/良好(1-2维预警)/需调整(3-4维)/需重写(5+)。4类评级带具体行动指引。

**novel-evaluator**：6维×权重×10分=100分制，含评分参考(每维度有1-10分的具体表现描述)、去AI味检查清单(10条具体可见模式)、短文节奏检查、S/A/B/C/D 5级制。

**humanizer**：24种AI模式分类——内容模式(过度强调意义/促销语言/虚化归因)、语言模式(AI高频词/系动词逃避/"不仅…而且…"/三段式/优雅变异/虚假范围)、风格模式(破折号过度/加粗过度/内联标题列表)、沟通模式(机器人语气/知识截断声明/谄媚语调)、填充词/过度hedging/假正面结论。每种配 Before→After 实例。

**现有基线(v1.0.0)**：pop-novel-qa 当前明确声明"不评分、不给建议、不评价节奏"。只有感觉型三层介入QC——Step 1大纲层/Step 2骨架层/Step 3正文层。desideratum"只回答'我读的时候怎么了'"。

### 2.2 改造方案

**改造目标**：将 pop-novel-qa 从"纯感觉型三层介入"改造为"双层QA：L1硬门禁(量化扫描) + L2感觉型三层介入"。

**不动的部分**：Step 1/2/3 的感觉型QC(我读的时候怎么了)保留为 L2。

**新增 L1 硬门禁**：

| 门禁维度 | 检查内容 | 来源 |
|---------|---------|------|
| **穿帮检测** | 死亡角色复活/地点矛盾/等级倒退/时间线错误/已毁地点出现/装备消失 | fiction-crafter |
| **套路重复检测** | 与前5章对比：冲突类型/情绪落点/打脸结构相似度，>80%预警 | fiction-crafter |
| **AI味硬扫描** | 基于中文特化 AI 模式清单的正则扫描(详见攻击面B) | humanizer |
| **伏笔超期预警** | >40章未回收=严重、>30章=中度、>20章=轻度 | fiction-crafter |
| **字数合规** | 每章字数在目标区间内(由project配置设定) | g113593 |
| **钩子强度匹配** | 高潮章钩子强度≥L4、过渡章≥L1(详见攻击面D) | g113593 |

**L1 评级规则**：

```
优秀 ✅：所有门禁通过
良好 ⚠️：1-2个门禁预警
需调整 ⚠️：3-4个门禁预警
需重写 ❌：严重问题(穿帮/套路重复>80%)或5+门禁预警
```

**L2 保留**：原有的三层感觉型QA（Step 1/2/3）作为 L2，继续做"我读的时候怎么了"。

**新增量化评分**：每个维度 1-10 分，满分 80 → L1 40分(硬门禁) + L2 40分(感觉型)。>60 交付，>70 优秀。

## 三、攻击面 B：去 AI 味深度化

### 3.1 证据链

**humanizer**（最强证据源）：24种AI模式 + 每模式 Before→After + Personality & Soul 章节。独特价值在于不仅告诉我们"不要什么"，还告诉我们"要什么"——有观点、节奏多变、承认复杂性、用"I"、允许一点乱、具体感受。

**novel-evaluator**：去AI味检查清单——"深吸一口气""嘴角微微上扬""眼神变得坚定""心中涌起一股暖流""无奈地摇了摇头""紧紧地握着""缓缓地开口"——这些是中文网文特化的AI高频模式，humanizer 没有覆盖。

**khazix-writer**：Personality & Soul 章节——"避免AI模式只是工作的一半。无菌无声音的写作和垃圾一样明显。" 活人感注入技巧：有观点(Vary your rhythm)、承认复杂性、用"I"、允许一点乱、具体感受。

**现有基线**：prose-render v3.0 的 Step 3(风格验证)只有3条粗略规则："无AI观感词/无套话/骨架事件不可跳过"。缺系统化扫描清单和正向注入指引。

### 3.2 改造方案

**两件事**：
1. 创建中文特化 `styles/anti-ai-patterns.md`（可被 prose-render 消费）
2. 在 prose-render 的 Step 3 加入 AI 味硬扫描 + 活人感注入

```
# 中文网文 AI 写作模式清单（去 AI 味专用）

## ❌ 质量红线
- 本章 AI 模式命中 >3 种 → 退回 Step 2 重写
- "深吸一口气""嘴角微微上扬""眼神变得坚定" → 零容忍，命中即替换

## 一、高频 AI 词汇（中文特化）
- 璀璨/瑰丽/绚烂/无垠/浩瀚…（过度修饰形容词）
- 心中涌起一股… / 一股…涌上心头（模板句式）
- 目光中闪过一丝… / 眼中闪过一抹…（AI最爱）
- 嘴角微微上扬 / 无奈地摇了摇头 / 紧紧地握着 / 缓缓地开口（AI机械动作）
- 深吸一口气 / 咬了咬牙 / 握紧了拳头（AI紧张三件套）

## 二、句式模式
- "不是…而是…" → AI标志性句式，改"说白了就是…"
- "与其说…不如说…" → 同上
- "这一刻，他…" → 过度煽情启动
- 连续3句以上以"他"开头 → 节奏呆板

## 三、内容模式
- 虚化归因："据说""有人说过""在XX的历史长河中"
- 过度强调意义："这是XX的转折点""标志着XX的开始"
- 教科书科普："首先需要了解…""众所周知…"
- 假正面结论："未来可期""一切才刚刚开始"

## 四、风格模式
- 破折号过度(每500字>1个)
- 环境描写与情节脱节(写景≥150字无事件推进)
- 形容词连用≥3个 → 精简

## 五、活人感注入指引（正向）
- [ ] 有至少一处自嘲或承认不足
- [ ] 有至少一处口语化感叹("操""卧槽""尼玛")
- [ ] 句子长度有明显变化(短句≤5字 + 长句≥25字交替)
- [ ] 有至少一处迟疑/打断/没说完整的话
- [ ] 情绪表达是体感记忆("我手都在抖")而非知识性描述("我感到非常紧张")
```

## 五、攻击面 D：钩子体系精细化

### 5.1 证据链

**g113593**：10种经典悬念钩子——突然揭示、紧急危机、未完成动作、身份反转、两难选择、神秘物品/线索、时间限制、承诺/威胁、离奇消失、言外之意。5级强度——L1好奇悬念(这很有趣)→L2关切悬念→L3迫切悬念→L4生存悬念→L5终极悬念。钩子类型轮换机制(防止同一个套路用腻)。

**fiction-crafter**：每章末尾必设钩子，与下一章"核心任务"对应，钩子强度与章节位置匹配(高潮章用L4+)。

**现有基线**：
- prose-render: Step 2 L67-L73 章末钩子仅两型(悬念/情绪)，无分级/无闭环
- chapter-design: fact-skeleton L110-L112 钩子字段仅 type+content，无强度/无驱动因素/无闭环
- plot: hook 分散在三处(卷间/章末/首卷)，无统一管理

### 5.2 改造方案

两个改动点：
1. **chapter-design 骨架模板** — 钩子字段从 type+content 扩展为 10种类型 + 5级强度 + 驱动因素
2. **plot 幕级设计** — 加入钩子强度递进约束(高潮章≥L4，过渡章L1-2，全书总趋势上升)

```
5. dilemma           两难选择 → 章末停在决定前
6. mysterious_clue   神秘线索 → 发现意义不明的重要物品
7. time_pressure     时间限制 → 倒计时/截止时间被设定
8. threat_promise    威胁承诺 → 某人做出威胁或承诺
9. disappearance     离奇消失 → 某人/物突然消失
10. subtext          言外之意 → 表面正常但暗示深层
```

**5级钩子强度**：

```
L1 好奇悬念 — "这很有趣" → 过渡章
L2 关切悬念 — "接下来会发生什么" → 中间章
L3 迫切悬念 — "他必须马上行动" → 推进章
L4 生存悬念 — "他会活下去吗" → 高潮章
L5 终极悬念 — "一切到底是什么意思" → 卷终
```

**修改文件 2**: `pop-novel-plot/6.1.0/steps/step-2-act.md`

| 改动 | 位置 | 内容 |
|------|------|------|
| 新增钩子强度递进约束 | L62-L84(节奏自检区) | 新增检查项：本章钩子强度≥前章-1？/高潮章钩子≥L4？/全书钩子强度总体上升？ |

## 六、攻击面 F：Skill 文档范式对标竞品

### 6.1 为什么这是最关键的改造

攻击面 A/B/C/D 加的都是**功能**（门禁、去 AI 味、评论区诱导、钩子体系），但攻击面 F 改的是**agent 如何消费这些功能**。一个写得像架构文档的 skill，agent 读完需要先"理解系统"再"找到入口"再"执行"——这个认知成本直接决定了输出质量和稳定性。

竞品 skill 的文档范式之所以有效，不是因为他们遵守了某种形式规范，而是因为他们遵循了一条**第一性原则**：

> **Agent 消费 skill 的方式 = 人类读菜谱，不是工程师读系统设计文档。**

菜谱告诉你：先开火 → 倒油 → 下葱姜 → 炒肉 → 调味 → 出锅。每步标注原料和工具。如果油温不够怎么办？翻下一页看 troubleshooting。你不会在菜谱开头看到"本菜谱采用三层烹饪架构：预处理层→加热层→调味层"。

### 6.2 竞品 skill 的文档范式（证据链）

以下是 g113593、fiction-crafter、humanizer、khazix-writer 的实际做法，逐个审视：

#### 范式 1：描述是触发式

| 竞品 | 实际写法 |
|------|---------|
| g113593 | `当用户要求：写番茄小说、创作网文、分章节写作、连续剧情、章节悬念、长篇小说时使用。` |
| fiction-crafter | `当用户要求：写小说/生成爽文/创作故事，或提及 爽文/小说/写作 时启用。` |
| novel-evaluator | `Invoke when user needs to evaluate/rate a novel, assess story quality, or get professional feedback on fiction writing.` |
| **我们** | `网文创作元 Skill（专家模式）。Think→Execute→Reflect 三层工作流。自动识别创作意图并路由子 Skill。` |

触发式 vs 定义式的差别：**agent 读触发式描述，0.2 秒内就知道"这个 skill 是不是该由我来用"。读定义式描述，还需要再往下翻才能确认。**

#### 范式 2：开篇一屏扫清（速查表）

g113593 和 fiction-crafter 的共同点：正文第一屏不是理论阐述，而是一张**速查表**，agent 不需要读完全文就能定位到正确的执行路径。

```
g113593 速查表示例（SKILL.md 第二屏）：
| 场景 | 操作 |
|------|------|
| 用户提供小说方向/题材 | 执行「提示词生成流程」|
| 开始创作新章节 | 先读取大纲、章节规划、记忆文件 |
| 新卷开始前 | 生成本卷的章节规划文件 |
```

```
fiction-crafter 速查表示例（SKILL.md 第二屏）：
| 场景                  | 操作 |
|----------------------|------|
| 用户提供小说方向/题材 | 执行「提示词生成流程」，产出完善的创作提示词 |
| 开始创作新章节       | 先读取大纲、章节规划、记忆文件，再按模板生成 |
```

我们的 skill 没有这个。agent 需要先读架构概述、管线声明、路由表，翻到 100+ 行才能找到"第一步做什么"。

#### 范式 3：流程是 SOP，每步标输入输出

g113593 的逐章创作流程：

```
#### 1. 写前分析
1. 读取 00-大纲.md - 查看TODO list和已完成章节的摘要
2. 读取上一章的摘要和结尾钩子
3. 读取 02-情绪曲线.md - 确认本章在该情绪单元中的位置和任务
4. 更新 00-大纲.md TODO list - 将本章标记为"进行中"
...
#### 2. 撰写
7. 创建章节文件
8. 撰写正文 - **每章必须达到2200-2800字**
...
#### 3. 撰写后优化
11. 连贯性检查
12. 深度润色
13. 质量自查
14. 字数检查
```

每一步都有**操作动作**（读/写/检查/润色）和**操作目标**（文件名/字数/检查清单引用）。agent 读完直接执行，不需要自行推演。

**对照我们的 expert-writer：**

```
## Think—Execute—Reflect 三层工作流
### Step 0 全局感知
1. 读 workspace-index.yaml → 感知文件树 → 锚定当前阶段
### Think：路由前强制加载
加载 skill 目录结构
路径解析 → 当前 project.yaml → 快照
...
```

这是**描述一个系统怎么工作**，不是**告诉 agent 你现在该做什么**。agent 读完 Step 0 要做 3 个判断（读到什么、怎么感知、怎么锚定），每个判断背后有未言明的规则。这就是为什么我们的 agent 需要"思考"而不是"执行"。

#### 范式 4：检查清单是门禁（不是建议）

| 竞品 | 门禁写法 |
|------|---------|
| g113593 quality-checklist | `总分 /80，>60 可交付，>70 优秀` |
| fiction-crafter 审查 | `❌ 需重写：严重问题或5+维度预警` |
| humanizer | 没有显式门禁分数，但每个模式都配 Before→After + "Below is WRONG → must rewrite" |

**对照我们 prose-render 的红线：**

```
- [ ] 设计包存在 — chXXX-设计包.md 完整可读
- [ ] 设计包已读取
...
```

这是**checklist**格式，不是**门禁**格式。checklist 的隐含语义是"尽量做到"，门禁的语义是"不通过=退回"。

正确的门禁写法：
```
❌ 设计包不存在 → 退回，禁止凭记忆渲染。必须等待上游 chapter-design 产出。
❌ AI 模式命中 ≥ 3 种 → 退回 Step 2，逐条替换后重新渲染。
```

#### 范式 5：实例 > 抽象规则

humanizer 定义了 24 种 AI 模式。每个模式的结构：

```
### N. Pattern Name
**Words to watch:** [具体词汇清单]
**Problem:** [一句话说清为什么这是问题]
**Before:** [3-5句具体错误文本]
**After:** [3-5句修正后文本]
```

不是"避免过度强调意义"，而是给你看一个具体段落 BEFORE 和 AFTER。agent 不需要理解"过度强调意义"这个抽象概念，它只需要对照 before 文本和自己的输出，相似就替换。

**khazix-writer** 也是如此——不是"用口语化表达"，而是给你 30+ 个推荐口语词组 + 禁用词清单 + 正反例。

**对照我们 prose-render 的规则：**

```
- 叙事者不解释 — 没有"他意识到/他感到/他仿佛"等 AI 观感词
```

这条规则是**抽象的**。"AI 观感词"是一个概念，agent 需要自己判断什么算、什么不算。对比 humanizer，agent 直接扫到 `深吸一口气 → 替换` 就行。

#### 范式 6：异常有兜底

g113593：

```
| 场景 | 处理方式 |
|------|---------|
| 中断后继续 | 读取所有记忆文件 + 上一章，继续生成 |
| 跨卷续写 | 先生成新卷的章节规划，再续写 |
| 修改后续写 | 确认修改内容已更新到记忆文件，再续写 |
```

onkos（更工程化）：

```
| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 不在 git 仓库 | git rev-parse 失败 | 提示用户；若拒绝，用 cp 文件备份代替 revert |
| results.tsv 缺失 | 文件不存在 | 新建并写表头 |
| 分支已存在 | git checkout -b 失败 | 分支名末尾加 -2，第3次失败则询问 |
```

**我们**：没有一个 skill 有体系性的异常兜底表。agent 遇到文件缺失时自行决策，可能编造数据、跳过步骤、或者报错后停滞。

### 6.5 OPC 团队深度对标：g113593 没覆盖的高级范式

workbuddy 专家团 (opc-team) 的 skill 设计是 g113593/fiction-crafter/humanizer 的"升级版"——他们在 SOP 化的基础上又多了几层我们和 g113593 都没做到的机制。

#### 范式 7：阶段边界 + 越界检测（g113593 没有，我们也没有）

**OPC 的做法**：每个 skill 有一个强制性的「本阶段边界」区块：

```
### 本步做什么
- 分析新杠杆、元杠杆可以在哪些方向发挥（环1）
- 识别行业里正在发生的边界变动（环2）
- ...

### 本步不做什么（以下属于后续阶段，不要提前进入）
- ❌ 不做内容策略或内容形态规划（属于 MVP 设计阶段）
- ❌ 不做文案撰写、话术设计、帖子骨架（属于资产沉淀阶段）
- ...

**越界检测**：如果对话出现"发什么内容""怎么写文案""用哪种格式""怎么发小红书"等执行话题，立即说：
> "这个方向很有价值，不过它属于后续 [MVP 设计 / 转化闭环] 阶段的范围。我们先把利基定位确认好，后面的执行才有准确的靶心。"
然后把话题拉回当前阶段。
```

**为什么这比 g113593 强**：g113593 的阶段划分（6问→规划→逐章创作）有顺序但**没有越界检测**。agent 可能在利基定位阶段就滑入了内容策略讨论。

**对我们意味着什么**：我们的 pipeline 有严格的阶段顺序（bookstrap → plot → chapter-design → prose-render → QA），但当前 skill 文档**没有越界检测**。agent 可能在 plot 阶段就提前设计了章节细节（属于 chapter-design 的活），或者在 chapter-design 阶段就开始纠结渲染用词（属于 prose-render 的活）。

> **可学**：给每个子 skill 加入「本阶段边界 + 越界检测」区块。expert-writer 的速查表里每行加一个"本阶段不做什么"列。

#### 范式 8：落盘检查点协议（g113593 有但粗，OPC 做到了极致）

**OPC 的做法**：每个 skill 有一个强制执行（不可跳过）的「落盘检查点」：

```
## 落盘检查点（阶段必须完成，不可跳过）

用户明确确认目标利基和定位陈述后，**立即**使用 Write 工具创建以下文件。在对话中描述结论不等于落盘。

**写入文件：**
- `opc-doc/outputs/02-niche-positioning/three-ring-analysis.md`
- `opc-doc/outputs/02-niche-positioning/candidates.md`
- `opc-doc/outputs/02-niche-positioning/target-segment.json`（含结构化的 JSON schema）
- `opc-doc/outputs/02-niche-positioning/positioning-statement.md`

**更新状态文件：**
- `opc-doc/state/current-stage.json`（写入：`{"stage": "02-niche-positioning", "status": "completed", "next_stage": "03-value-proposition", "summary": "一句话定位核心"}`）
- `opc-doc/state/decisions.json`（追加确认的利基选择）

**落盘完成后，在对话中告知用户：**
> "✅ 利基定位结论已保存。下次对话可以从价值主张继续。"

**只有落盘完成后，才可以提示进入 `opc-value-proposition`。**
```

**为什么这比 g113593 强**：g113593 的产出物是隐含在流程里的（"创建 00-大纲.md"），但没有统一的"落盘门禁协议"。agent 可能在对话里描述了结论但忘记写文件。

**对比我们的现状**：我们的 skill 产出物规范分散在各自的 step 文件里，没有统一格式。expert-writer 有 pipeline-check 做前置校验，但没有"写完才算完成"的后置门禁。

> **可学**：在 PRD 的改造中，给每个子 skill 的 SKILL.md 新增统一的「落盘检查点」区块。格式对齐 OPC：写入哪些文件（含精确路径）+ 更新哪些状态 + 告知用户什么 + 硬性"只有落盘后才进入下一步"。

#### 范式 9：对话轮次控制（g113593 没有）

**OPC 的做法**：interaction-protocol.md 定义了统一的对话轮次上限：

```
- 第 1 到 4 轮：正常推进当前模块
- 第 5 轮：进入提醒阶段，明确告诉用户需要逐步回到本阶段目标
- 第 9 轮：必须强制总结，询问用户是否还有关键补充
- 第 10 轮：默认完成阶段，输出阶段总结并进入确认或下一步
```

**为什么重要**：没有轮次控制，agent 可能在某个阶段无限展开。g113593 靠"6 问"自然限制了轮次，但我们的 skill 没有。

> **部分可学**：我们的对话轮次不如 OPC 可控（创作型对话天然需要更多轮次），但可以在 skill 文档中标注"本阶段建议对话轮次上限"，给 agent 一个软约束。

#### 范式 10：方案选择协议（g113593 没有，OPC 所有 skill 统一使用）

**OPC 的做法**：所有 skill 的决策点全部使用：

```
方案 A：[名称] — 适用情况 + 优点 + 代价
方案 B：[名称] — 适用情况 + 优点 + 代价
方案 C：[名称] — 适用情况 + 优点 + 代价
4. 我有自己的方案 — 可以直接说一个完全不同的版本 / 可以混合前几个 / 可以在某个方案基础上修改
```

**为什么重要**：统一降低了用户的决策成本。"选 A/B/C 或提出你的"比"你想怎么做？"更高效。而且每个方案都标注了"适用情况、优点、代价"，用户的选择是信息充分的。

**对比我们的决策点闸门**：我们有 4 个子 skill 确认点（bookstrap/plot/chapter-design/prose-render），但确认方式没有统一格式——有时是"你觉得可以吗？"有时是"要继续吗？"。

> **可学**：在我们的决策点闸门里引入"方案选择协议"——不只是"确认继续？"而是"这里有 3 个方向，每个的得失是什么，选一个或自己提"。

#### 范式 11：会话恢复协议（g113593 没有，OPC 的 orchestrator 独有）

**OPC 的做法**：每次会话开始时，**第一件事**是读 `current-stage.json` + `decisions.json` + 当前阶段 output 文件。然后向用户展示进度摘要，不重复提问。

```
"我找到了上次的记录。你已完成资源盘点和利基定位，上次停在价值主张阶段。要继续吗？"
```

**对比我们**：expert-writer 的 Step 0 做全局感知（读 workspace-index.yaml），但不如 OPC 的会话恢复协议系统化。

> **可学**：在 expert-writer 的"第一步：感知项目状态"里，加入"会话恢复协议"逻辑——先读状态文件，展示进度摘要，再问继续还是从某阶段重来。**不重复提问**。

#### 范式 12：教学模式/引导模式/直通模式

**OPC 的做法**：orchestrator 在首轮判断用户对方法论的熟悉度，然后选择三种模式之一：
- 教学模式：每步先解释概念和作用
- 引导模式：解释关键概念，保持问题简洁
- 直通模式：减少解释，保留确认

**对比我们**：expert-writer 的 reference 目录下有 think-开书设定/正文写作/审稿/续写 等教学模式模板，但没有系统性地按模式切换交互方式。

> **部分可学**：我们的用户群体相对固定（都是老板自己），不需要三种模式切换。但可以在 expert-writer 速查表里加一行"如果用户你说'直接写/快一点/跳过解释' → 切换到精简模式（少解释、多执行、保留闸门确认）"。

#### 能学 vs 不能学的总结

| 范式 | 可学性 | 原因 |
|------|:---:|------|
| 阶段边界 + 越界检测 | ✅ 完全可学 | 纯文档模式。给每个子 skill 加"本步不做什么 + 越界时说什么" |
| 落盘检查点协议 | ✅ 完全可学 | 纯文档模式。要求每个 skill 明确列出写入文件+状态更新+用户消息 |
| 对话轮次控制 | ⚠️ 部分可学 | 创作型对话轮次更难控制。加"建议轮次上限"作为软约束 |
| 方案选择协议 | ✅ 完全可学 | 纯文档模式。在决策闸门处统一"3 方案 + 4. 自己来"格式 |
| 会话恢复协议 | ✅ 完全可学 | 纯文档模式。并入 orchestrator 第一步 |
| 交互模式切换 | ⚠️ 部分可学 | 用户单一，不需要三模式。加"精简模式"开关即可 |
| 子 agent 调用 | ❌ 不能学 | pop 没有子 agent 功能（OPC 也没显式用子 agent，交互协议是纯文本指令） |
