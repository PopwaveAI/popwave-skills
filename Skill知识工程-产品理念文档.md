## 1. 核心理念：什么样的知识才能变成好 Skill

### 1.1 一句话总结

> **源材料的「抽象层级」决定 Skill 的复用质量。教「方法」的能变好 Skill，教「故事」的只能变摘要。**

一个 Skill 能做多好，不取决于 AI 有多聪明，而取决于源材料里藏了多少「可复用的东西」。

### 1.2 「换皮测试」——判断一篇文章能不能变成好 Skill

把源材料里所有项目名、公司名、具体数据、技术栈名字全部抹掉。然后看看剩下的部分，读者还能不能从中学到一套可以直接用的方法。

| 结果 | 结论 |
|:-----|:-----|
| 能 | 源材料的「骨架」是硬的，转出来的 Skill 会很好用 |
| 不能 | 源材料的价值和具体案例粘在一起，删掉案例就什么都不剩 |

### 1.3 三类知识 → 三类 Skill

| 源材料类型 | 特征 | 转化成的 Skill 形态 | AI 体感 |
|:-----------|:-----|:-------------------|:--------|
| **SOP 型**（菜谱） | 有明确步骤、先后顺序、操作指引、踩坑经验 | **工作流型**（Workflow）— AI 引导你一步步做 | 🟢 最惊艳——AI 真的在帮你干活 |
| **复盘型**（项目心得） | 结合特定项目/模型/数据，有少量方法论 | **能力参考型**（Capabilities）— AI 随叫随到给方案参考 | 🟡 有用但不够 wow |
| **分析型**（横向对比） | 案例丰富但缺独立分析框架 | **决策参考型**（Reference）— AI 辅助信息梳理 | 🟠 高级摘要机器人 |

## 2. 支撑论据

### 2.1 正面案例：SOP 型 → AI 漫剧工作流 Skill

**原文特征**：
> "先用大模型写剧本，再用即梦生成分镜图，再用可灵把静态图变成动态视频，最后用剪映合成"

- 步骤之间有严格的先后顺序
- 每一步有具体的操作指引和参数建议
- 「从零到一做 AI 漫剧」的通用流程，换成任何主题、任何风格，这套流程都成立

**转化效果**：用户说「帮我做个 AI 漫剧」，AI 真的能一步步带你从剧本写到出片。

**为什么成功**：文章教的不是「怎么做这一条片子」，而是「怎么做这一类片子」。抽象层级够高。

### 2.2 中庸案例：复盘型 → AI 解说能力建设 Skill

**原文特征**：
- 提出了「专业性、拟人性、趣味性」三个维度的优化框架（**这个很好——是方法**）
- 但文章大部分篇幅在讲「原案例在特定项目里是怎么做的」
- 方案与特定模型、数据管线、基础设施绑定紧密

**转化效果**：AI 能把这个方法论复述得不错，但用户问「我们的项目怎么做」时，AI 只能说「你们可以参考……」因为原文没给够「脱离特定项目也成立」的通用方法。

**为什么不够好**：框架只占两成，案例占八成。抹掉案例后骨架太薄。

### 2.3 反例：分析型 → AI 商业化分析 Skill

**原文特征**：
> 对比六家游戏公司的 AI 策略，案例很全

- 核心内容 = 「这六家公司分别做了什么」
- 用户如果问的恰好是文章里讲到的公司 → OK
- 用户如果问一个文章没提到的公司或品类 → AI 只能泛泛分析

**为什么失败**：缺少一个「不依赖具体案例也能成立的分析框架」。

### 2.4 Skill 的技术本质

```
传统 RAG           Skill
────────────       ────────────
处理陈述性知识      处理程序性知识
（API 参数是多少）    （遇到这类问题怎么做）
喂碎片给模型        给完整操作手册
AI 猜答案           AI 照着做
```

**Skill 的三层渐进式披露架构：**

```
┌─────────────────────────────────────┐
│ L1: 元数据摘要（~100 token）          │  ← AI 启动时读取，判断「需要哪个 Skill」
│ 「我是谁、我能干什么」                │
├─────────────────────────────────────┤
│ L2: 完整操作指令（按需加载）          │  ← AI 执行任务时读取
│ 步骤、规则、红线、常见错误             │
├─────────────────────────────────────┤
│ L3: 参考文档 + 模板（按需加载）        │  ← AI 需要细节时读取
│ 模板、结构定义、示例                   │
└─────────────────────────────────────┘
```

**关键约束：**

| 约束 | 说明 |
|:-----|:-----|
| **上限** | = 源材料里藏了多少「可复用的东西」，和 AI 聪明不聪明没关系 |
| **下限** | = 底层模型的能力边界，Skill 不创造能力，只传递能力 |
| **窗口** | 上下文窗口约 128K-200K token，天然适合「一个领域的一套方法」，不是「一个领域的所有知识」 |

> Skill 不会让 AI 变得更聪明，但它能让 AI 的聪明用在正确的地方。它不创造能力，只传递能力。

## 3. 对我们的启发（对照 pop-novel-plot v4.1 改造实战）

### 3.1 启发一：SKILL.md 的「骨架剥离」就是一次「换皮测试」

#### 文章说法
> 把具体案例抹掉，看剩下的框架还能不能站住。

#### 我们做了什么

**改造前**：pop-novel-plot 的 SKILL.md = **437 行**，混合了路由导航、详细执行指令、YAML 字段定义、错误示例、异常处理——全部塞在一个文件里。

**改造成果**：

```
SKILL.md（147行）          ← L1：路由层——「我是谁、干什么、12步顺序」
steps/（12个文件）          ← L2：执行指令——每步按需加载
templates/（10个文件）      ← L3：参考文档+模板——产出格式标准
```

**对照换皮测试**：

| 测试 | 旧 SKILL.md（437行一体） | 新 SKILL.md（拆分后） |
|:-----|:----------------------|:---------------------|
| 抹掉具体 YAML 字段定义 | ❌ 只剩错误示例和异常表 | ✅ 12步执行顺序 + 质量红线依然成立 |
| 抹掉异常处理表 | ❌ 只剩字段定义 | ✅ 执行顺序 + 步骤索引依然成立 |
| 结论 | 骨架和皮肉混在一起 | 骨架独立，皮肉按需加载 |

**对应的文章结论**：**渐进式披露不是炫技，是保证「骨架」独立存在、不被「皮肉」淹没。**

### 3.2 启发二：templates/ 做的就是「从案例中提取框架」

#### 文章说法
> 文章的「抽象层级」决定 Skill 的质量。教「方法」的能变好 Skill，教「故事」的只能变摘要。

#### 我们做了什么

pop-novel-plot 旧版 act-XX.yaml 的演进史，就是文章三类知识的真实投影：

| 版本 | 项目 | 源材料质量 | 产出物 | 对应文章分类 |
|:-----|:-----|:-----------|:-------|:-----------|
| v2.0 | 海贼法典 | 简化的 act-01.yaml，无 plotlines、无 info_release | 1 个 yaml | **分析型**——像摘要 |
| v2.9 | 北帝镇诡录 | act-01.yaml + 节点B + 情节线草案 + info-release | 4 个文件 | **复盘型**——有框架但不完整 |
| v2.9.1 | 新书-诸天万界 | 更精简的 act-01.yaml，关键字段丢失 | 1 个 yaml | **分析型**——退化了 |

**v4.1 改造后**：10 个 templates/ 模板就是把「方法」从「案例」中提取出来：

| 模板 | 提取的是什么 | 对应原则 |
|:-----|:------------|:---------|
| `act-guide.md` | 「一卷 yaml 应该怎么填」— 不是给你一个现成的 yaml 复制，而是给你字段说明+示例+常见错误 | **教方法，不是教故事** |
| `character-list.md` | 「一卷人物清单应该怎么设计」— 角色分类、出场节奏、关系矩阵 | **框架独立于具体角色** |
| `map-design.md` | 「一卷地图应该怎么画」— 关键地点、移动线路、空间情绪对应 | **框架可用于任何世界观** |
| `checkpoint-b.md` | 「设计前应该想清楚什么」— 6 个思维检查点 | **思维框架，不是填表清单** |

**对应的文章结论**：**好的模板 = 把「红烧这道菜的做法」升级为「红烧的方法」——换一条鱼、换一块豆腐，这个方法照样能用。**

### 3.3 启发三：Canvas 依赖链路就是对「Skill 不创造能力，只传递能力」的工程实践

#### 文章说法
> Skill 不创造能力，只传递能力。你能做的事是确保 LLM 不在错误的输入上浪费 token。

#### 我们做了什么

pop-novel-plot v4.1 的核心架构改进是 **Canvas → act-XX.yaml 的强制消费链路**：

```
Step 5  人物（act-XX-人物.md）   ──→  act-XX.yaml 的 characters_active 字段
Step 6  地图（act-XX-地图.md）   ──→  act-XX.yaml 的 locations 字段
Step 7  世界（act-XX-势力.md）   ──→  act-XX.yaml 的章节编排参考
Step 8  信息释放（info-release） ──→  act-XX.yaml 的 info_release 字段
         │
         └──→ 全部在 _meta.canvas_refs 显式声明依赖
```

**为什么这是「传递能力」的工程化：**

- **act-XX.yaml 不自己编角色**——它必须声明 `characters_active`，角色名必须在 act-XX-人物.md 中存在
- **act-XX.yaml 不自己编地点**——它必须声明 `locations`，地名必须在 act-XX-地图.md 中存在
- **writer 不会收到空投的信息**——它通过 act-XX.yaml 的 `info_release[].source_doc` 知道去哪读 L1 设定

| 对比 | 旧架构（v4.0 之前） | 新架构（v4.1） |
|:-----|:-------------------|:---------------|
| 人物 | Writer 凭记忆写角色出场 | act-XX.yaml 显式声明每章登场角色 → writer 去 act-XX-人物.md 读细节 |
| 地图 | Writer 凭记忆写场景 | act-XX.yaml 显式声明每章发生地点 → writer 去 act-XX-地图.md 读描述 |
| 信息释放 | Writer 凭记忆决定放多少设定 | act-XX.yaml 的 info_release 规划好 P0/P1 释放 → writer 按 source_doc 读原文 |

**对应的文章结论**：**Writer 不会因为有了 Skill 就突然学会创作——但我们确保它拿到的输入是正确的、可追溯的、有上游文件的。这就是「传递能力」的工程意义。**

### 3.4 启发四：分级强制（P0/P1/P2）是对「上下文窗口有限」的回应

#### 文章说法
> 上下文窗口约 128K-200K token，Skill 天然适合「一个领域的一套方法论」，不是「一个领域的所有知识」。

#### 我们做了什么

pop-novel-plot v4.1 的产物被分为三级：

| 等级 | 定义 | 产物 | 对应原则 |
|:-----|:-----|:-----|:---------|
| **P0 强制** | 缺失则不进章纲 | act-XX.yaml + act-XX-人物.md + act-XX-地图.md | **一套方法论**——writer 必需的输入 |
| **P1 建议** | 有则更好 | 节点B + 情节线草案 + info-release + 势力表 + 装备表 + 自检报告 | 补充上下文，但不阻塞流程 |
| **P2 可选** | 锦上添花 | 情节线纲汇总表 | 不强制，不占用 token |

**为什么分三级**：

如果把 10 个产物全标为 P0 强制，每产出一卷 writer 就要读 10 个文件——token 窗口撑不住。分级 = **把「一套方法」塞进上下文，把「所有细节」放在外部文件按需加载。**

**对应的文章结论**：**P0/P1/P2 分级不是为了偷懒跳过步骤——是为了在 128K token 窗口里，只喂给 writer 它真正需要的最少必要知识。**

### 3.5 启发五：模板的 error list 是对「文章质量决定 Skill 质量」的工程补偿

#### 文章说法
> Skill 的上限完全取决于你往 context 里塞了什么。瓶颈不在 AI 的转化能力，瓶颈在源头的内容质量。

#### 我们做了什么

`act-guide.md` 模板末尾有**常见错误表**（第 8 节）：

| 错误 | 正确做法 |
|:-----|:---------|
| plotlines_active 里出现不存在的线 id | 先完成 plotlines 定义，再填每章 |
| characters_active 里的角色不在 act-XX-人物.md | 先确保人物清单包含该角色 |
| locations 里的地点不在 act-XX-地图.md | 先确保地图包含该地点 |
| info_release 的 source_doc 路径错误 | 严格按照 L1 目录格式 |
| end_hook 和下一章 emotional_goal 不衔接 | 逐章检查钩子→情绪路径连续性 |

**为什么这是「补偿」而非「创造」**：

这些错误是新版 v2.9.1（新书-诸天万界）实际犯过的错误——plotlines 缺失、info_release 缺失、版本号涨了内容反而缩水。我们把已知坑点写成模板的 error list，本质上是在**补偿源材料的不足**——AI 不知道「没填 plotlines 会导致 writer 收不到剧情线」，但我们知道，所以我们把这条写进了模板。

**对应的文章结论**：**好模板不会让 AI 变聪明，但它把已知的踩坑经验「编译」成了 AI 能理解的规则——这是从「记录者」到「知识工程师」的跨越。**

## 4. 方法论汇总：Skill 知识工程的五条原则

| # | 原则 | 文章出处 | 我们的实践 |
|:-:|:-----|:---------|:-----------|
| 1 | **抽象层级决定复用性** | 「换皮测试」——抹掉案例后骨架能否独立 | SKILL.md 路由层只留骨架，steps/ 和 templates/ 是皮肉 |
| 2 | **教方法，不教故事** | 「好文章教的是红烧的方法，不是红烧这条鱼」 | templates/ 每份模板有字段说明 + 示例 + 错误表，面向通用场景 |
| 3 | **渐进式披露，按需加载** | 元数据→指令→参考，三层不一次性全吃 | SKILL.md（L1）+ steps/（L2）+ templates/（L3） |
| 4 | **不创造能力，只传递能力** | Skill 不会让 AI 变聪明，但能让聪明的方向对 | Canvas 强制消费链路：让 writer 读正确的输入而不是自己编 |
| 5 | **文章质量 = 瓶颈，模板是补偿** | Skill 上限 = 源材料里藏了多少可复用的东西 | 模板末尾的 error list：把踩坑经验编译成 AI 可执行的规则 |

## 5. 对后续 Skill 改造的指导意义

### 5.1 改造优先级判断矩阵

| 判断维度 | 改造该 Skill | 暂缓改造 |
|:---------|:-----------|:---------|
| 该 Skill 的 SKILL.md 是否骨肉混杂 | 是 → 拆分为 L1/L2/L3 | SKILL.md 已经足够精简则跳过 |
| 该 Skill 的产出物是否有格式不统一问题 | 是 → 补模板 files | 产出格式已统一则跳过 |
| 该 Skill 的下游消费是否缺失显式依赖 | 是 → 补 canvas 消费字段 | 下游链路已可追溯则跳过 |
| 该 Skill 的模板是否有 error list | 否 → 补充常见错误 | 已有则跳过 |

### 5.2 所有 Skill 的统一架构标准

```
{skill-name}/
├── SKILL.md              ← L1: 路由层
│   「我是谁、干什么、步骤索引、质量红线」
├── steps/                ← L2: 执行指令（按需加载）
│   └── step-N-xxx.md
├── templates/            ← L3: 参考文档 + 产出模板
│   └── xxx-guide.md      模板必须有：字段说明 + 示例 + 常见错误表
├── skill.json
└── CHANGELOG.md
```

### 5.3 模板质量自检清单

每个 `templates/` 下的模板文件必须通过：

- [ ] 是否有「字段说明」—— 每个字段的含义、可取值、示例
- [ ] 是否有「使用场景说明」—— 这个模板在什么时机使用
- [ ] 是否有「上下游关系」—— 输入来自哪个上游文件、产出被哪个下游消费
- [ ] 是否有「常见错误表」—— 至少 5 条已知踩坑经验
- [ ] 是否有「产出自检清单」—— 填完模板后的验收标准
- [ ] 是否通过「换皮测试」—— 抹掉所有具体案例后，指南依然有效
