## P0 阻塞级

### 问题 1：子agent 隔离机制全面失效

**严重度：** P0  
**影响范围：** 所有路由到子skill的任务（writer、plot、deconstructor、bookstrap）

**事实依据：**

- 会话 `2e5e943d` 中，三次路由到子skill（writer×2、plot×1），**三次全部失败**
- 失败原因一致：`gateway closed (1006 abnormal closure)`
- 会话 `0bcaa832` 中同样全部失败，最终 OpenClaw 彻底崩溃：

  ```
  Error: Cannot find module 'D:\popwave\resources\app.asar\dist-electron\electron\openclawAgentStream.js'
  ```
- **100% 失败率**，自记录以来没有一次子agent成功启动

**核心矛盾：**  
expert-writer 的 SKILL.md 要求子agent隔离执行（"不继承主agent历史上下文，防止污染"），但实际路径中所有子skill逻辑在主agent内执行。这导致：

1. **上下文污染**：前一步的认知错误向后传递（多弗线案例）
2. **精读被绕过**：主agent"觉得自己已经读过了"，跳过前文精读
3. **QC自我认可**：QC和writer共享上下文，事实错误检不出来
4. **思考膨胀**：主agent在同一段推理流中混合设计决策、执行规划、自我质疑

**详见专项文档：**`子agent隔离机制失效分析-2026-06-08.md`

### 问题 2：正文写作缺乏"写前精读"硬闸门

**严重度：** P0  
**影响范围：** 续写/拆书后写作环节，尤其是长篇小说续写

**事实依据（会话 `0bcaa832`，海贼法典续写）：**

1. **写 CH389 前只读了 CH388 末尾**，CH370-387 没读
2. 结果：**多弗朗明哥的关系线完全写错**

   - 原文事实：CH372 威廉和多弗已经私下谈妥合作（镀膜换列车股份）
   - AI 写成："多弗试探威廉这个陌生人"
3. 被用户质问后，AI 自我辩解："因为读者不知道他们私下谈过"
4. **用户直接驳回**："这是什么逻辑？" → AI 承认"吃设定"
5. 精读倒数20章后，发现**5个关键信息偏差**：

   - 多弗已合作（非陌生人）
   - 卡普交接时汉库克在楼上看
   - 卡斯兰娜是海军线人
   - 古罗莉欧萨观察的是蒸汽船（非码头）
   - 战国对威廉的态度是"大局为重"（非纯粹敌意）

**根因：**  
主agent的"记忆"替代了"读取"。因为会话历史中已经出现过海贼法典的内容，主agent"以为"自己知道前文状态，跳过了精读。这不是懒惰——而是上下文持续累积带来的认知错觉。

## P1 建议级

### 问题 3：风格文档只"加载"不"执行"

**严重度：** P1  
**影响范围：** 正文质量一致性、文风还原度

**事实依据（会话 `0bcaa832` 第29-30轮）：**

关键对话摘录：

> 用户："你有走 正文写作的 风格文档吗"  
> AI："用了，但没用到位。写之前读了 styles/default.md...但写正文时是凭记忆遵守而不是对照执行"

**问题表现：**

- 红线清单"否定句结构（不是A而是B）0容忍"——写时容易不自觉踩到
- 对话/叙述比的均衡型（50/50）——会议场景天然对话多，容易超
- 直到用户明确要求"用遮天风格试试"，才真正去读风格文件

**根因：**  
Render 阶段没有"风格文件全程开放"的硬约束。风格文件只在"读入"那一刻存在于上下文，实际写正文时已经被后续内容挤出了注意窗口。

---

### 问题 4：管线冗余，大量废弃中间产物

**严重度：** P1  
**影响范围：** 工作效率、项目文件整洁度

**事实依据：**

1. **北帝 v2 黄金三章全部删除：**

   - 用户新建 v2 目录 → 写 ch01-05 + 对应的 Director 设计说明 + 事实骨架 + 正文 + QC 自检
   - 然后说"v2 版本删除吧" → 全部清除
   - 5章 × 每章4个文件 = 20个文件**全部废弃**
2. **海贼法典写了三版 CH389：**

   - v1：初版（多弗线有偏差）
   - v2：精读后重建版（修正多弗线，幕纲+spec+正文全改）
   - v3：遮天风格重写版（风格注入）
3. **每次写一章产生4-5个文件**（Director设计说明、事实骨架、正文、QC自检、global-summary更新），但核心质检维度（事实一致性、文风执行度）却没有对应的检查节点

**根因：**  
管线定义了"做什么"（写几份文件），但没定义"做对了没有"（事实一致性检查、风格执行检查）。产出物多不等于质量高。

---

### 问题 5：spec 审批流程的认知负担

**严重度：** P1  
**影响范围：** 流程阻塞、用户信任

**事实依据（会话 `0bcaa832` 第15-16轮）：**

用户第一次遇到 spec.md 时的反应：

> "spec.md 需要审批 在哪里 路由给我 另外这个spec 是干吗的？"

这说明：

1. spec 作为"写前硬闸门"的概念对用户**不够直观**
2. 用户需要先理解 spec 的概念，再决定"通过"或"改"
3. **spec 本身也是 AI 生成的**——用户需要审批一个 AI 写给自己看的文档
4. 这变成了一种认知负担：用户需要判断"AI 的写作计划是否合理"

**对比：** 同样的闸门概念在 milestone 设计（"必须用户确认"）中运行良好，因为用户确认的是"剧情走向"，是自己能直观判断的内容。spec 确认的是"写作参数"，用户需要先掌握一套知识才能判断。

## P2 优化级

### 问题 6：OpenClaw 运行时稳定性风险

**严重度：** P2  
**影响范围：** 长线写作的中断风险

**事实依据：**

- 会话 `0bcaa832` 最后一次调用完全失败（`MODULE_NOT_FOUND`）
- 会话 `00c4e0e3` 完整记录了崩溃日志
- 子agent启动链路存在**双点故障**：

  1. Gateway connection 可能失败（`gateway closed (1006)`）
  2. OpenClaw 的 Node 模块路径可能变更（硬编码的 app.asar 路径）

**影响：**  
用户在海贼法典会话进行到一半（写完 CH390、正要在 CH391 继续推进时），工具突然崩溃。对话中断，需要重新建立上下文。

### 问题 7：项目间无知识共享机制

**严重度：** P2  
**影响范围：** 学习曲线不累积、重复踩坑

**事实依据：**

- 北帝项目的经验日志（`01-写作资产/经验日志/experience-log.md`）只记录了该项目自己的经验
- 海贼法典的 reverse mode 产出物（PRD、L1、锚定章库）各自独立
- 三个作品（北帝镇诡录、海贼法典、诸天万界）之间没有任何跨项目知识传递机制

  - 比如北帝项目的 QC 发现"首章无金手指即弃"的风险
  - 这个经验没有传递给海贼法典的续写
  - 也没有传递给诸天万界的开书

**根因：**  
项目级别的经验数据没有全局索引。每个项目的经验日志只记录本项目的得失，但跨项目可以复用的经验（黄金三章节奏、平台适配规则、常见写作红线）没有被集中管理。
