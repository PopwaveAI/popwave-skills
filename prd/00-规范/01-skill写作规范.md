## 十、Skill 规范写作标准（v6 草案）

> 适用范围：所有 Popwave Skill。
> 对齐基线：`skills/pop-shared-skill-create/SKILL.md`。
> 本文目标：统一 `SKILL.md`、`skill.json`、`steps/`、`references/` 的写法与边界。

### 10.1 总原则

| # | 规则 |
|:-:|:-----|
| 1 | **先定义执行，再定义表达。** 文档首要服务对象是 agent，不是人类读者。 |
| 2 | **双文件必备。** 新建或重构 skill 时，`SKILL.md` 与 `skill.json` 必须同时存在，缺一不可。 |
| 3 | **改造前先全读。** 改造存量 skill，必须完整读取 `SKILL.md`、`skill.json`、`CHANGELOG.md` 后再动手。 |
| 4 | **先扫描后修改。** 改造前必须输出一份扫描报告：结构、缺口、越界、落盘风险。 |
| 5 | **禁令式，不建议式。** 统一使用“必须 / 禁止 / 退回 / 终止”，不用“建议 / 尽量 / 可以考虑”。 |
| 6 | **路径必须可执行。** 所有读写动作写到具体路径或目录级别，不能写成抽象名词。 |
| 7 | **每段都有门禁。** 任何步骤都要写明触发条件、失败动作、回退路径。 |
| 8 | **冷启动友好。** 不假设 agent 记得上一轮上下文，首屏必须能独立启动执行。 |
| 9 | **正文不讲目的论。** 禁止出现“本步骤的目的是……”，只写“读什么 / 做什么 / 产出什么 / 不通过怎么办”。 |
| 10 | **SKILL.md 只放路由与约束。** 长解释、方法论、赛道差异、模板细则全部下放。 |

### 10.2 文件职责分层

| 文件/目录 | 只负责什么 | 不应该放什么 |
|:----------|:-----------|:-------------|
| `SKILL.md` | 路由、红线、速查表、步骤总览、边界条件、落盘检查点 | 大段方法论、长案例、模板字段说明、3 行以上版本史 |
| `skill.json` | 机器可读元数据：名称、描述、版本、路由、依赖、产出 | 面向人的长说明、执行 SOP |
| `steps/` | 每一步的详细执行指令 | 顶层路由、版本历史 |
| `references/` | 方法论、架构图、赛道差异、匹配 SOP | 主流程步骤说明 |
| `templates/` | 产出模板与内嵌注释 | 执行规则说明书 |
| `examples/` | 合格样例、反例、审计样本 | 当前正式规则 |
| `CHANGELOG.md` | 版本历史与变更摘要 | 现行规范正文 |

### 10.3 读取协议（强制）

| # | 规则 |
|:-:|:-----|
| 1 | **读取 skill 文件禁止用 Read 工具。** 优先使用 `skill_view`；无法使用时改用 `Get-Content -Encoding UTF8 -Raw`。 |
| 2 | **读取后做截断校验。** 对比读取内容长度与文件大小；显著偏小视为疑似截断。 |
| 3 | **疑似截断必须重读。** 第一次异常后立刻回退到 `Get-Content -Encoding UTF8 -Raw` 重读。 |
| 4 | **连续两次不过即终止。** 不允许在未确认全文的前提下继续改造或评估。 |

**最低执行模板：**

```text
1. 读 `SKILL.md`
2. 读 `skill.json`
3. 读 `CHANGELOG.md`
4. 做截断校验
5. 输出扫描报告
6. 再进入创建 / 改造 / 评估 / 审计
```

### 10.4 四类任务的写法约束

| 模式 | 触发场景 | 允许做什么 | 禁止做什么 | 必备产出 |
|:-----|:---------|:-----------|:-----------|:---------|
| A 设计 | 新建 skill / 从零搭建 | 创建 `SKILL.md` + `skill.json` + 目录骨架 | 跳过元数据文件 | 双文件骨架 + 目录清单 |
| B 改造 | 修复 / 重写 / 升级 skill | 先扫描，再改 `SKILL.md`/`skill.json`/配套目录 | 不读全文直接动手 | 扫描报告 + 改造后文件 |
| C 评估 | review / 打分 / 审核 | 评分、指出缺口、给改进建议 | 直接改文件 | 评分卡 + 建议 |
| D 审计 | 会话数据分析 / 执行回溯 | 分析 session 数据、定位 toolCall 膨胀与执行偏差 | 直接改 skill 正文 | 审计报告 |

### 10.5 `SKILL.md` 标准结构

#### 首屏

```markdown
# {skill 名称}

> **定位：从 X 到 Y。**
> **核心约束：不碰 Z。**
```

**要求：**
1. 第一屏必须能回答“这个 skill 消费什么、产出什么、不能碰什么”
2. 不能写成长段背景介绍
3. 越界风险必须前置到首屏

#### 质量红线

统一使用表格：

```markdown
## ❌ 质量红线

| # | 红线 |
|:-:|:-----|
| ❌1 | **{违规情形}** — {判定标准} → {动作} |
```

**铁律：**
1. 每条红线都要有唯一序号
2. 每条红线都要包含“触发条件 + 后续动作”
3. 红线是硬门禁，不分 P0/P1/P2

#### 速查表

按 skill 类型选择格式：

| 类型 | 速查表列 |
|:-----|:---------|
| 产文件型 | `步骤 \| 操作 \| 读什么 \| 产出什么 \| 门禁` |
| 路由型 | `意图 \| 路由到 \| 前置条件` |
| 工具型 | `我要 \| 操作 \| 前置条件` |

#### 核心流程

```markdown
## 核心流程

### 步骤 1：{步骤名}
**读什么：** `{路径}`
**做什么：** {祈使句}
**产出：** `{路径}` 或 “无落盘”
**❌ 门禁：** {失败条件} → {动作}

详细指令见 `steps/step-1-xxx.md`。
```

**要求：**
1. 每步必须有 I/O
2. 每步必须指向 `steps/`
3. 不能只画箭头流程图，不写门禁

#### WRONG 示例

```markdown
## ❌ WRONG 示例
> ❌ {真实错误动作}
> ✅ {正确替代动作}
```

**要求：**
1. 只能基于真实错误记录
2. 不写虚构“可能犯错”
3. 一条错法必须对应一条可执行改法

#### 边界条件

```markdown
## 边界条件

| 场景 | 触发条件 | 处理 |
|:-----|:---------|:-----|
```

**至少覆盖 6 类：**
1. 前置文件缺失
2. 数据不足
3. 格式不匹配
4. 循环依赖
5. 用户中断
6. 子 skill 不可用

#### 落盘检查点

```markdown
## 落盘检查点

| 确认项 | 状态 |
|:-------|:----:|
| `{文件路径}` 已写入 | [ ] |
```

**要求：**
1. 每个产出文件至少一行
2. 允许加入少量内容型检查项
3. 禁止把完整产出正文粘回 `SKILL.md`

#### 版本

```markdown
## 版本
vX.Y.Z | YYYY-MM-DD | {一句摘要} → [CHANGELOG.md](CHANGELOG.md)
```

### 10.6 `skill.json` 标准字段

> `skill.json` 是执行入口的机器配置，不是说明书。

**最小字段集合：**

```json
{
  "name": "pop-xxx",
  "version": "1.0.0",
  "description": "一句话说明输入到输出",
  "entry": "SKILL.md",
  "pipeline": [],
  "inputs": [],
  "outputs": [],
  "dependencies": [],
  "tags": []
}
```

**字段要求：**

| 字段 | 要求 |
|:-----|:-----|
| `name` | 与目录名一致，稳定不可漂移 |
| `version` | 与 `SKILL.md`、`CHANGELOG.md` 保持一致 |
| `description` | 一句话，不写长段说明 |
| `entry` | 固定指向 `SKILL.md`，除非该 skill 有特殊入口 |
| `pipeline` | 仅写真实上下游，不写愿景中的未来节点 |
| `inputs` | 写文件/状态/工具级输入，不写空泛概念 |
| `outputs` | 写真实落盘物和状态更新 |
| `dependencies` | 写真实依赖 skill、脚本、知识库 |
| `tags` | 只写检索标签，不拿来承载规则 |

### 10.7 改造前扫描报告模板

改造存量 skill 前，先输出一份扫描报告，最少包含：

```markdown
## 扫描报告

### 1. 文件完整性
- `SKILL.md`：有 / 无
- `skill.json`：有 / 无
- `CHANGELOG.md`：有 / 无

### 2. 结构检查
- 首屏是否有定位与约束
- 是否有红线表
- 是否有速查表
- 是否有 steps 指向
- 是否有边界条件
- 是否有落盘检查点

### 3. 越界与膨胀
- 方法论是否内联过多
- 版本史是否过长
- 路由与执行是否混写
- 是否缺少 steps 下放

### 4. 改造建议
- 必改项
- 可后置项
```

### 10.8 不允许出现的内容

| 不应该出现在 `SKILL.md` | 应该放哪 |
|:------------------------|:---------|
| 超过 3 行的版本历史 | `CHANGELOG.md` |
| 大段架构图或全链路说明 | `references/` |
| 方法论正文 | `references/` |
| 模板字段说明书 | `templates/` 注释 |
| 长案例解析 | `examples/` |
| “本 skill 的目的是……” | 不写 |
| 机器元数据 | `skill.json` |

### 10.9 行数预算

| skill 类型 | 目标行数 |
|:-----------|:--------:|
| 领域型 skill | ≤150 行 |
| 专家 / 路由型 skill | ≤120 行 |
| 工具型 skill | ≤80 行 |

> 超预算优先处理顺序：先下放 `steps/`，再下放 `references/`，最后裁剪案例。

### 10.10 验收清单

| 检查项 | 状态 |
|:-------|:----:|
| `SKILL.md` 存在 | [ ] |
| `skill.json` 存在 | [ ] |
| `CHANGELOG.md` 存在或已说明缺失原因 | [ ] |
| 首屏含“定位 + 核心约束” | [ ] |
| 红线为表格格式且带动作 | [ ] |
| 速查表类型正确 | [ ] |
| 核心流程每步都有 I/O 与 `steps/` 指向 | [ ] |
| 含真实 WRONG 示例 | [ ] |
| 含 6 类边界条件 | [ ] |
| 含落盘检查点 | [ ] |
| `skill.json` 字段与正文一致 | [ ] |

## 十一、案例对照（本轮修订结论）

### 案例 1：产文件型 skill

**合格最小结构：**

```markdown
# {标题}
> **定位：从输入到输出。**
> **核心约束：不碰越界域。**

## ❌ 质量红线
## 速查表
## 核心流程
## ❌ WRONG 示例
## 边界条件
## 落盘检查点
## 版本
```

**配套文件最小结构：**

```text
skill/
├── SKILL.md
├── skill.json
├── CHANGELOG.md
├── steps/
├── references/
└── templates/
```

### 案例 2：路由型 skill

**合格关键点：**
1. `SKILL.md` 只保留路由规则、前置检查、异常协议
2. 具体执行策略下放到 `steps/`
3. skill 清单、能力表、扩展知识优先放进 `skill.json` 或 `references/`

### 案例 3：存量 skill 改造

**正确顺序：**
1. 读 `SKILL.md`
2. 读 `skill.json`
3. 读 `CHANGELOG.md`
4. 做扫描报告
5. 再开始改造

**错误顺序：**
1. 打开 `SKILL.md`
2. 看几行
3. 直接改

## 十二、这一版相对上一版新增的硬约束

| 新增项 | 说明 |
|:-------|:-----|
| 双文件约束 | `SKILL.md` + `skill.json` 必须同时存在 |
| 读取协议 | 禁止用 Read，必须做截断校验 |
| 四模式区分 | 设计 / 改造 / 评估 / 审计边界明确分离 |
| 扫描报告前置 | 改造前必须先出扫描报告 |
| `skill.json` 规范化 | 增加机器元数据标准字段 |
