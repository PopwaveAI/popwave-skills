# 拆书→写书闭环现状与迭代建议

> 版本：v0.1 | 2026-06-22
> 用途：从“拆书 PRD → 拆书 skill → 入库规范 → pop-trope-library → 写书 PRD → 写书 skill → 出文/状态更新/QA → 反向暴露问题”的闭环角度，梳理当前现状、断点与优先迭代项。

---

## 一、闭环定义

当前系统不是两条独立管线，而是一个连续闭环：

```text
拆书PRD
  ↓
拆书skills（pop-decon-* / pop-shared-*）
  ↓
入库协议（入库清洗SOP）
  ↓
pop-trope-library
  ↓
写书PRD / expert-writer 管线合同
  ↓
写书skills（creative / reservoir / world / plot / chapter / prose / qa）
  ↓
正文 / 状态快照 / QA反馈 / 继续写
  ↓
反向暴露：library 缺口、字段不足、粒度不稳、命名冲突
  ↓
回流修订拆书PRD / 拆书skill / 入库协议
```

核心判断：

1. 拆书端定义“产什么”。
2. library 定义“怎么存、怎么找、怎么被消费”。
3. 写书端定义“真正需要什么”。
4. QA / continue / prose 的失败点，才是闭环里最真实的迭代信号。

因此，判断拆书循环是否成熟，不能只看拆书端，必须同时看：

- 写书端实际消费接口
- library 是否可检索
- 下游失败是否能反向定位到上游规范

---

## 二、闭环各段当前角色

### 1. 拆书端

- PRD：定义拆书主线、阶段产物、入库目标、下游消费关系。
- skills：`pop-decon`、`pop-decon-design-pack`、`pop-decon-volume`、`pop-decon-setting`、`pop-shared-dna`、`pop-shared-reader`
- 当前作用：
  - 产出设计包、卷幕结构、L1设定、剧情单元、文风DNA
  - 把“原书经验”转成可供写书消费的中间资产

### 2. library 中间层

- `pop-trope-library` 不是展示层，而是协议层。
- 它承担两件事：
  - 入库：拆书端把资产沉淀进来
  - 出库：写书端按协议检索并消费

### 3. 写书端

- PRD：`01-写作专家全链路依赖图-PRD.md`
- 总调度：`expert-writer`
- 核心消费链：`creative → reservoir → world → character → plot → chapter → prose → qa`
- 当前作用：
  - 验证拆书资产是否真的可用
  - 暴露 library 的字段、路径、粒度和命名问题

### 4. 反馈层

- `prose`：最敏感，最容易暴露 DNA 路径、scene 对齐、设计包精度问题
- `plot` / `chapter`：最容易暴露套路库、剧情库粒度与检索协议问题
- `world`：最容易暴露设定库结构、L1接口、命名体系问题
- `qa`：最适合作为“反向问题采样器”，把症状归因到上游

## 三、当前闭环已经成立的部分

以下部分说明闭环不是概念，而是已经在仓库中被正式承认：

### 1. 写书主线已正式依赖 library

- 写书 PRD 明确写了 `pop-trope-library` 为公共知识库。
- `expert-writer` 已有查询矩阵。
- `creative / world / plot / chapter / prose / qa` 均写了“必须查库”或“查库优先”。

这意味着 library 已经从“参考资料夹”升级为“主干协议层”。

### 2. 拆书端已开始按“给写书消费”来定义产物

- 拆书 PRD 已明确把设计包、卷幕、L1、DNA分别路由到 creative / world / prose 等环节。
- `pop-decon-volume` 已开始强调剧情单元结构化。
- `pop-shared-dna` 已开始按 prose 的 scene 对齐来约束产物。

### 3. expert-writer 已把“先查库再创作”提升为系统纪律

这说明后续迭代重点不再是“要不要做闭环”，而是“闭环的接口要不要收口”。

## 四、当前闭环的主要断点

以下断点不是局部 bug，而是闭环中真正会阻断迭代的结构性问题。

### 4.1 library 检索协议未真正落地

现状：

- `调用匹配SOP.md` 假设消费侧可按 `layer / track / meta_joy` 三维筛选。
- `入库清洗SOP.md` 也要求原子化素材写 frontmatter。
- 但当前素材实际情况是：
  - 原始素材 371 张，仅部分有 frontmatter
  - 剧情库 20 个实际内容文件，基本无 frontmatter
  - 套路库 15 张抽象模式卡，无 frontmatter

影响：

- creative / plot / chapter / qa 虽然“理论上能查库”，但很难稳定按协议自动筛选。
- 当前更多还是“人工看目录+经验性点读”，不是规范化检索。

结论：

`library 已被声明为协议层，但仍停留在半人工资产仓阶段。`

### 4.2 设定库结构规范前后冲突

现状：

- `pop-trope-library/SKILL.md` 和 `目录.md` 仍在描述 `设定库/框架/`、`设定库/质感/`
- `设定库/00-索引.md` 已明确说重构为“按书分目录”
- `入库清洗SOP.md` 同时保留了：
  - 单卡式框架/质感路径
  - 整书 L1 入库流程

影响：

- world / expert-writer / 写书 PRD 仍然习惯性把设定库当成“框架+质感”双层库来消费
- 实际文件组织已经更偏“按书整包消费”
- 导致“设定库到底是卡库，还是书级设定包库”没有定案

结论：

`设定库是当前闭环里最明显的协议漂移点。`

### 4.3 DNA 的路径与存储哲学仍未统一

现状：

- 一部分规范写 `styles/`
- 一部分规范写 `写作资产/文风DNA/`
- 当前 `pop-shared-dna` 主文档又直接写 `pop-trope-library/文风DNA/{书名}.md`
- 入库清洗SOP写“只留摘要不留原文”
- 但 DNA skill 的真实产物要求保留大量原文摘录

影响：

- prose 只能做多路径 fallback，说明上游协议未统一
- DNA 到底是“library 内长期参考档案”还是“项目内临时写作资产”，边界不清
- “场景卡=原文参考页”与“入库只留摘要”两种理念直接冲突

结论：

`DNA 模块已可用，但尚未完成从历史路径到统一协议的收敛。`

### 4.4 剧情库粒度不稳定

现状：

- volume skill 规范说要输出结构化剧情单元
- 但库里实际同时存在：
  - 单元卡式（U01/U02...）
  - 幕级聚合式
  - 卷级大文件式

影响：

- plot 端无法假设“剧情库条目”的统一粒度
- 后续做按元爽点、按套路链、按冲突弧复用时，会遇到颗粒度不一致问题

结论：

`剧情库已经有内容，但还没有稳定文件学。`

### 4.5 Phase 3 / world 接口已经开始分叉

现状：

- 拆书 PRD 还在写 `战力体系.md / 动态升级表.md`
- `pop-decon-setting` 真实输出已转向 `combat_capability.yaml`、`起点快照.md`
- 写书 world 已经升级到 `L1-01~07`
- 写书 PRD 仍主要按 `L1-01~06` 叙述

影响：

- 拆书端不知道未来要对齐哪个 world 接口版本
- 写书端正在进化，但 PRD 与中间库规范尚未同步

结论：

`world 接口已经进入升级期，但拆书侧尚未完成反向对齐。`

### 4.6 QA 还没有成为“正式反哺上游”的故障采样器

现状：

- `pop-writer-qa` 已经能检查套路对齐、AI味、原文污染、元爽点多样性等
- 但这些问题目前主要停留在“审查结果”
- 还没有形成：
  - 哪类 QA 问题应该归因到拆书
  - 哪类归因到 library
  - 哪类归因到写书 skill

影响：

- 闭环缺少稳定反馈回路
- QA 发现问题后，系统不知道“该修哪一层协议”

结论：

`QA 已具备反馈价值，但还没有被制度化接入上游迭代。`

## 五、从闭环角度看，当前最该补齐的不是“更多内容”，而是“文档规范”

原因：

1. 当前问题主要不是“库里没东西”，而是“有东西但协议不稳定”
2. 只要接口不稳，拆书越多、素材越多，后期越难清洗
3. 文档规范是这个闭环的 API

可把文档规范分成四层：

### A. 产物定义层

谁产、产什么、文件名是什么、静态还是动态、谁消费。

例：

- `剧情单元` 到底是单元卡、幕级卡还是卷级卡
- `DNA` 的主路径到底是什么
- `设定库` 到底按书消费还是按卡消费

### B. 存储结构层

目录结构、索引结构、frontmatter 规范、命名规范。

### C. 查询协议层

哪些模块按三标签查，哪些先查 PRD 再读 L1，哪些只支持按书名或场景类型取。

### D. 反馈归因层

当 prose / plot / qa 出问题时，问题应该回流到：

- 拆书 skill
- 入库 SOP
- library 内容清洗
- 写书 skill

如果不补这四层，闭环就只能靠“懂系统的人脑补”，不能靠系统自身稳定演进。

## 六、建议的迭代优先级

### P0：先统一协议，再扩内容

#### P0-1 统一设定库模型

必须先定一个方向：

方案 A：`设定库 = 按书整包消费`
- 先看 PRD
- 再读 L1
- 框架/质感只是书内维度，不再作为物理目录

方案 B：`设定库 = 框架/质感卡库`
- 拆成单卡
- world / character 按标签检索

当前仓库更接近方案 A。  
建议：`PRD / SKILL / 目录 / SOP 全部统一到方案 A`，不要再双轨并存。

#### P0-2 统一 DNA 主路径与存储哲学

必须定清楚：

- 项目消费主路径
- library 归档主路径
- 是否保留原文摘录
- prose 的 fallback 是否只作为兼容层，不再写入新规范

建议：

- 项目消费：`写作资产/文风DNA/{书名}.md`
- library 归档：`pop-trope-library/文风DNA/{书名}.md`
- 文档中明确：library 档案保留原文摘录；对话输出只留摘要

#### P0-3 统一剧情库粒度

建议先定“默认单位”为：

- `单元卡 = 2-5章的可复用叙事单元`

幕级/卷级文件可以保留，但应改为：

- 幕级索引 / 幕级聚合页
- 卷级索引 / 卷级总览页

不要把不同粒度文件都算作“剧情库主条目”。

### P1：把 library 从“资产仓”升级为“协议仓”

#### P1-1 补齐三标签 frontmatter

至少补齐：

- 原始素材
- 套路库
- 剧情库主条目

否则 `调用匹配SOP` 只是纸面协议。

#### P1-2 重写 `目录.md`

把它从“宣传页”改成“系统状态页”，明确：

- 当前采用的目录模型
- 兼容层有哪些
- 哪些文件是历史格式
- 哪些模块已稳定
- 哪些模块仍试水

#### P1-3 给每个模块补“主消费方式”

例如：

- `套路库`：按三标签查
- `设定库`：先查 PRD 卖点，再读书级 L1
- `剧情库`：按赛道+元爽点+冲突类型查
- `文风DNA`：按书名+scene 精确匹配

### P2：把 QA / prose / world 的反馈正式回流到拆书

建议加一份“闭环反馈归因表”：

| 下游症状 | 优先怀疑层 |
|:---------|:-----------|
| prose 找不到场景卡 | DNA 命名 / 路径协议 |
| world 无法稳定借设定 | 设定库结构协议 |
| plot 难以复用剧情库 | 剧情库粒度 |
| QA 套路对齐失败 | 章节设计包 / 套路库公式 / 拆书套路抽象 |
| 连续同质爽点 | creative / plot 查询策略 / 元爽点映射 |

这样 QA 才不是“终点报告”，而是“回流触发器”。

## 八、最终判断

当前状态可以概括为：

### 已完成的

- 闭环方向已经成立
- 上下游已经互相引用
- library 已经进入主回路
- 一部分内容资产已经足够支持试运行

### 未完成的

- 统一接口没有收口
- library 仍处于早期演进态
- 文档规范存在历史残留和双轨并存
- QA 反馈还没有制度化回流

### 结论

下一轮迭代最该做的不是继续大量加内容，而是：

1. 定接口
2. 清历史
3. 补索引
4. 建反馈回流规则

只有这样，拆书循环才会从“能跑”进入“能稳定迭代”。

---

## 九、建议的执行顺序

1. 统一 `设定库` 模型
2. 统一 `DNA` 主路径与存储规则
3. 统一 `剧情库` 主条目粒度
4. 补齐 library frontmatter 与索引
5. 回写拆书 / 写书 PRD 与关键 skill 文档
6. 建立 QA → 上游归因表

如果这 6 步做完，后续再扩书、扩素材、扩 skill，系统成本会低很多。
