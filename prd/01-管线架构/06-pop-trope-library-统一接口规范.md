# pop-trope-library 统一接口规范

> 版本：v0.2 | 2026-06-22
> 用途：把 `pop-trope-library` 从“四模块素材库”升级为“写书与拆书共用的上游知识总库”，并定义五库结构、消费接口、兼容层与容错策略。

---

## 一、总定位

`pop-trope-library` 不再只表示“套路素材库”，而是写书系统的上游知识总库。

目标结构：

```text
pop-trope-library/
├── 立项库/
├── 设定库/
├── 文风库/
├── 剧情库/
├── 套路库/
├── 原始素材/
└── references/
```

五库职责：

| 模块 | 服务对象 | 解决的问题 |
|:-----|:---------|:-----------|
| `立项库/` | creative, expert-writer | 这本书为什么成立，PRD 怎么立，元爽点怎么组合 |
| `设定库/` | reservoir, world, character | 世界怎么搭，力量/制度/命名/文明底色怎么借鉴 |
| `文风库/` | prose | 语言怎么写，scene 如何匹配文风DNA |
| `剧情库/` | plot | 故事怎么推进，剧情单元如何改建 |
| `套路库/` | creative, plot, chapter, qa | 模式怎么复用，套路公式和使用红线是什么 |

`原始素材/` 是拆书中间层，不是主消费层。它服务套路库晋升与复盘，不直接作为写书 skill 的稳定接口。

---

## 二、设计原则

### 1. 总库统一，模块差异保留

五库都属于 `pop-trope-library`，但不能强行使用同一种条目粒度。

- `立项库`：PRD 级模式
- `设定库`：书级设定包
- `文风库`：书级文风档案
- `剧情库`：剧情单元
- `套路库`：抽象模式卡

### 2. 消费侧优先

接口由写书侧真实需求反推：

- `creative` 需要立项模式、元爽点组合、套路总览
- `world` 需要按书读取完整 L1 设定
- `plot` 需要剧情单元与套路链
- `chapter` 需要具体套路卡的公式与节奏控制
- `prose` 需要可全量加载的文风档案和 `scene` 匹配
- `qa` 需要套路卡的使用红线

### 3. 主接口稳定，兼容层明确

写书 skill 仍在高速迭代，因此不能一次性切断旧路径。

每个模块都应标明：

- `canonical`：主接口
- `fallback`：兼容读取
- `legacy`：历史保留，不鼓励新逻辑依赖

---

## 三、五库接口

## 3.1 立项库

### 主定位

`立项库` 是 creative 的上游经验库，沉淀 PRD 级知识。

它不存“桥段怎么写”，而存：

- 什么立项结构稳定
- 什么元爽点组合可持续
- 什么题材焊接方式容易成立
- 什么 PRD 会在 world / plot 阶段崩掉

### 主路径

```text
pop-trope-library/立项库/
```

建议结构：

```text
立项库/
├── 00-索引.md
├── 立项模式/
├── 元爽点组合/
├── 题材焊接/
├── 加工哲学/
└── 失败案例/
```

### 主条目类型

| 类型 | 用途 | 示例 |
|:-----|:-----|:-----|
| 立项模式卡 | 帮 creative 生成可落地的书型 | 阶级压迫型修仙、黑色幽默反派成长 |
| 元爽点组合卡 | 判断 2-3 个主爽点是否兼容 | 信息差 + 阶级革命 + 绝境反杀 |
| 题材焊接卡 | 指导跨域融合 | DND 规则骨架 + 中文网文升级节奏 |
| 加工哲学卡 | 决定保留什么、改写什么 | 借机制不借质感、保留情绪换设定 |
| 失败案例卡 | 防止 PRD 高概念但下游不可施工 | 卖点过多、世界规则互斥、主爽点打架 |

### 建议 frontmatter

```yaml
---
source_type: prd_pattern
track: 修仙
meta_joy: 信息差
stage: creative
status: trial
---
```

### 消费方式

- `creative`：生成故事概念前，优先读 `立项库/00-索引.md`
- `expert-writer`：开新书时，把立项库作为 creative 前置参考
- `world / plot`：当 PRD 下游施工卡住时，可回查立项库判断是不是立项层矛盾

### 与套路库边界

| 问题 | 归属 |
|:-----|:-----|
| 这本书为什么成立 | 立项库 |
| 这个爽点组合能不能支撑长篇 | 立项库 |
| 这个具体桥段怎么部署 | 套路库 / 剧情库 |
| 这个套路怎么写才爽 | 套路库 |

---

## 3.2 设定库

### 主定位

`设定库` 是 world / reservoir / character 的设定参考库。

主模型：按书整包消费。

### 主路径

```text
pop-trope-library/设定库/{书名}/
```

最低稳定内容：

```text
PRD.md
L1-01-世界蓝图.md
L1-02-力量体系.md
L1-03-历史与驱动力.md
L1-04-物种与天赋.md
L1-05-势力格局.md
L1-06-资源与物品.md
世界宪法.md
```

可选增强内容：

```text
L1-07-术语与文明底色.md
起点快照.md
终点快照.md
数值体系/
动态升级表.md
```

### 主索引

```text
pop-trope-library/设定库/00-索引.md
```

### 消费方式

固定顺序：

1. 查 `设定库/00-索引.md`
2. 按赛道、卖点、命名气质、机制类型找到参考书
3. 读该书 `PRD.md`
4. 再读对应 L1 文件

### 兼容层

历史上的 `设定库/框架/` 和 `设定库/质感/` 降级为 legacy。

新规范中，“框架/质感”是书内维度，不再是设定库主目录模型。

---

## 3.3 文风库

### 主定位

`文风库` 是 prose 的风格锚定库。

主模型：一本书一份文风档案，内部按 `scene` 匹配。

### 主路径

目标主路径：

```text
pop-trope-library/文风库/{书名}.md
```

当前兼容路径：

```text
pop-trope-library/文风DNA/{书名}.md
```

### 迁移策略

短期：

- `文风库/` 作为 canonical 入口
- `文风DNA/` 作为 fallback 内容目录
- prose 继续允许读取 `文风DNA/`

中期：

- 新 DNA 入库优先写入 `文风库/`
- `文风DNA/` 保持只读兼容

### 必备内容

文风档案必须提供：

- 书名级标题
- 场景卡
- `scene:` 字段
- 通用维度
- 时间演变
- 原文摘录

### 消费方式

`prose` 标准读取顺序：

```text
1. pop-trope-library/文风库/{书名}.md
2. pop-trope-library/文风DNA/{书名}.md
3. 写作资产/文风DNA/{书名}.md
4. 章节设计包/文风DNA/{书名}.md
5. styles/{书名}.md
```

### 边界

文风库只描述笔触，不提供内容替换规则。

禁止把 DNA 观察翻译成：

- 替换设定体系
- 替换战斗机制
- 替换角色身份
- 改写剧情骨架

---

## 3.4 剧情库

### 主定位

`剧情库` 是 plot 的剧情改建参考库。

主模型：标准剧情单元。

### 主路径

目标主路径：

```text
pop-trope-library/剧情库/{书名}/units/{单元ID}-{单元名}.md
```

当前兼容：

```text
pop-trope-library/剧情库/{书名}/{现有文件}.md
```

### 主条目粒度

默认主条目：2-5章的可复用叙事单元。

允许辅助文件：

- 本书索引
- 幕级聚合页
- 卷级总览页

但辅助文件不应替代 `units/` 下的主条目。

### 建议 frontmatter

```yaml
---
source_type: plot_unit
unit_scope: chapter_unit
track: 修仙
meta_joy: 信息差
status: stable
---
```

### 消费方式

- `plot`：优先读标准单元卡
- 如果只有聚合页，允许 fallback，但必须标注“聚合页参考，不是标准单元卡”

---

## 3.5 套路库

### 主定位

`套路库` 是抽象模式库，服务 creative / plot / chapter / qa。

### 主路径

```text
pop-trope-library/套路库/{模式名}.md
```

### 主索引

```text
pop-trope-library/套路库/00-总索引.md
```

### 建议 frontmatter

```yaml
---
source_type: abstract_pattern
layer: 机制
track: 修仙
meta_joy: 信息差
status: stable
---
```

### 消费方式

- `creative`：读总索引和元爽点映射
- `plot`：按剧情线需求查模式族
- `chapter`：读取具体套路卡的公式和节奏控制
- `qa`：读取具体套路卡的使用红线

---

## 四、统一查询协议

五库不强行共用一种查询方式。

| 模块 | 主查询键 | 次查询键 | 主消费方式 |
|:-----|:---------|:---------|:-----------|
| 立项库 | 书型 / 元爽点组合 / 题材焊接 | `track/meta_joy/stage` | creative 前置参考 |
| 设定库 | 书名 / 卖点 | 赛道 / 质感方向 / 机制类型 | 先读索引和 PRD，再读 L1 |
| 文风库 | 书名 | `scene` | 书级定位 + 场景精配 |
| 剧情库 | 单元ID / 本书索引 | `track/meta_joy/unit_scope` | 先读单元卡，聚合页辅助 |
| 套路库 | 模式名 / 总索引 | `track/meta_joy/layer` | 精确读卡 + 三维筛选 |

---

## 五、兼容层与容错

## 5.1 canonical / fallback / legacy

建议所有索引都标记条目状态：

- `canonical`：当前主接口
- `fallback`：可读兼容接口
- `legacy`：历史保留，不鼓励新逻辑依赖

### 当前建议

| 模块 | canonical | fallback | legacy |
|:-----|:----------|:---------|:-------|
| 立项库 | `立项库/` | 无 | skill 内散落的 PRD 经验 |
| 设定库 | `设定库/{书名}/` | `设定库/00-索引.md` | `设定库/框架/`, `设定库/质感/` |
| 文风库 | `文风库/{书名}.md` | `文风DNA/{书名}.md` | `styles/{书名}.md` |
| 剧情库 | `剧情库/{书名}/units/*.md` | 现有书目录文件 | 卷级/幕级大文件直接作主条目 |
| 套路库 | `套路库/{模式名}.md` | `套路库/00-总索引.md` | 无 frontmatter 的旧卡 |

## 5.2 写书侧容错要求

### creative

- 立项库为空时，可继续走原有 creative 流程
- 查到立项模式时，只作为 PRD 推导参考，不替代用户选择

### world

- 允许 `L1-07` 缺失
- 不允许缺 `PRD.md` 与 `L1-01~06`
- 设定库缺匹配书时，必须声明库缺

### plot

- 剧情库缺标准单元卡时，可读聚合页
- 但要标注“粒度不足”

### chapter / qa

- 套路库旧卡无 frontmatter 时，允许精确读取
- 需要筛选时，优先使用已补 frontmatter 的条目

### prose

- 必须支持文风库与文风DNA双路径
- DNA 缺失时不得静默降级
- scene 缺失时走外部DNA适配，而不是改剧情

## 六、入库协议调整

### 6.1 拆书端新增入库目标

拆书或复盘完成后，除原四库外，应允许沉淀到 `立项库`：

- 成功 PRD 模式
- 失败 PRD 模式
- 题材焊接经验
- 元爽点组合经验
- 加工哲学

### 6.2 写书端也可以反向入库

写书不是只消费 library。以下结果也应回流：

- creative 产出的稳定 PRD
- world 阶段证明可施工的设定架构
- plot 阶段证明可持续的爽点组合
- qa 发现的立项层问题

这类回流优先进入 `立项库`，而不是塞进 `套路库`。

---

## 七、当前迁移顺序

建议按以下顺序推进：

1. 新增 `立项库/00-索引.md`
2. 新增 `文风库/00-索引.md`，先指向现有 `文风DNA/`
3. 更新 `pop-trope-library/目录.md` 为五库结构
4. 更新 `pop-trope-library/SKILL.md` 的定位与目录说明
5. 更新 `调用匹配SOP.md`：从“四模块三维查询”改成“五库分模块查询”
6. 更新 `入库清洗SOP.md`：新增立项库入库规则，修正设定库双轨问题
7. 后续再迁移实际内容，不急着一次性搬文件

---

## 八、最终判断

`pop-trope-library` 应升级为总库，而不是继续被理解成“套路库的外壳”。

最稳的长期结构是：

- `立项库`：决定书怎么立
- `设定库`：决定世界怎么搭
- `文风库`：决定语言怎么写
- `剧情库`：决定故事怎么推
- `套路库`：决定模式怎么复用

这样 creative 到 prose 的每个阶段都有自己的上游知识接口，也能承接写书 skill 持续迭代带来的新需求。
