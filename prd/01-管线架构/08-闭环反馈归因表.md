# 闭环反馈归因表

> 版本：v0.1 | 2026-06-22
> 用途：当 prose / plot / chapter / world / qa 暴露问题时，系统化地归因到拆书 skill、入库 SOP、library 内容、或写书 skill。让 QA 从"终点报告"变成"回流触发器"。

---

## 一、归因模型

```
下游症状
  ↓
四层排查（逐层向上一一排除）
  ├── ① 写书 skill 本身（prompt/参数/上下文问题）
  ├── ② library 内容（缺/错/粒度不对/路径失效）
  ├── ③ 入库 SOP（门槛未执行/清洗不完整/索引未更新）
  └── ④ 拆书 skill（设计包格式/事件提取遗漏/命名不一致）
```

**排查原则**：从近到远。先排除写书 skill 自身问题，再往上游追溯。不要一跳直接归因到拆书。

---

## 二、归因速查表

### prose（正文渲染）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 找不到场景卡 | **library 内容** | ① 文风库/{书名}.md 是否存在 ② 文风DNA/{书名}.md 是否存在 ③ 场景卡 `scene:` 字段是否与设计包的 scene 值一致 | 补 DNA 档案 / 对齐 scene 命名 |
| 渲染风格漂移（AI味） | **写书 skill** | ① prose 的 delegation contract 是否正确传递 DNA 路径 ② 子 agent 是否被错误注入了"改写指令"而非原始 DNA | 修 prose 的 step-0-delegation-contract |
| 场景卡原文不足 | **入库 SOP** | DNA 档案是否满足 6 条红线（原文>=500字/场景卡>=6/通用维度>=4/有时间演变/无降级替换） | 退回 pop-shared-dna 扩采 |
| 多路径 fallback 混乱 | **入库 SOP** | 文风库 vs 文风DNA vs styles 三路径是否一致 | 统一到 canonical/fallback 双路径 |
| DNA 路径找错了书 | **library 内容** | 设计包中的书名/文风DNA 文件名是否匹配 | 修正文件名或路径引用 |

### plot（剧情设计）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 剧情库查不到可复用剧情线 | **library 内容** | ① 剧情库/{标签}/ 中是否有标准剧情线 ② 线卡是否满足六段格式 ③ 是否误读了 _LEGACY/ 中的卷幕切片 | 从拆书资产提炼标准剧情线 |
| 剧情线粒度不合适（太粗/太细） | **入库 SOP** | 入库时是否执行了剧情线提炼核查清单（≥4阶段/可迁移边界≥3条） | 退回重提炼 |
| 套路链与剧情线不配套 | **写书 skill** | plot 是否按调用匹配SOP 分模块查询（而非一次查全库） | 修 plot 的查询流程 |
| 新书剧情线强行套用原书卷节奏 | **写书 skill** | plot 是否直接消费了原书卷/幕聚合页（违反消费纪律） | 修 plot 的消费协议 |

### world（世界设定）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 无法按赛道找到参考书 | **library 内容** | ① 设定库/00-索引.md 是否按赛道标注了参考书 ② PRD.md 的卖点摘要是否写清了匹配逻辑 | 补索引 / 补 PRD 匹配逻辑 |
| 读到的设定不完整 | **入库 SOP** | 设定库/{书名}/ 下是否包含 PRD.md + L1-01~06 + 世界宪法 | 补全设定包 |
| 读到了旧版框架/质感路径 | **library 内容** | 是否仍在消费 `设定库/框架/` `设定库/质感/` 路径（已降级为 legacy） | 改为 `设定库/{书名}/` |
| 设定与数值体系不匹配 | **拆书 skill** | Phase 3 setting 产出的 L1-02 力量体系与 combat_capability.yaml 是否一致 | 回溯 Phase 3 重新归纳 |
| L1 设定中混入了编造数据 | **拆书 skill** | 是否有凭空发明的境界链/名词（无 chXX 原文证据） | 退回 Phase 3 补充证据 |

### chapter（章节设计）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 套路公式缺少参数 | **library 内容** | 套路库/ 中该模式的 10 字段是否齐全（通用公式/变体家族/使用红线） | 补全抽象模式卡 |
| 套路卡无 frontmatter 无法筛选 | **library 内容** | 原始素材/ 或 套路库/ 的对应条目是否有 layer/track/meta_joy 标签 | 补三标签 |
| 节奏控制数据缺失 | **入库 SOP** | 套路卡是否包含"节奏控制"段（篇幅×前置期×密度×位置） | 退回补全 |
| 使用了未晋升的 L3 具体卡 | **写书 skill** | chapter 是否直接从原始素材/ 读取而非从套路库/ | 修 chapter 读取路径 |

### qa（质量审查）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 套路对齐失败（正文与公式不符） | **写书 skill → 拆书 skill** | ① prose/chapter 是否正确消费了套路公式 ② 公式本身是否准确（原始素材→套路库晋升过程是否有误） | 逐层排查 |
| 连续同质爽点 | **写书 skill** | creative 的元爽点组合是否过于单一；plot 的剧情线多样性是否不足 | 修 creative/plot |
| AI 味/原文污染 | **写书 skill** | prose 的 DNA 消费是否正确（观察≠改写指令） | 修 prose delegation contract |
| 兑现缺口（伏笔未回收） | **拆书 skill** | Phase 2 volume 的契诃夫枪追踪是否遗漏了关键伏笔 | 回溯 volume 补充 |
| 命名冲突（不同书同名词） | **library 内容** | 设定库/ 的命名/术语是否缺少作用域标注（书级命名空间） | 补命名冲突记录 |

### creative（立项）

| 下游症状 | 优先怀疑层 | 具体排查路径 | 修复方向 |
|:---------|:-----------|:------------|:---------|
| 立项库空，无法复用经验 | **library 内容** | 立项库/00-索引.md 是否为空；Phase 4 trace SOP 转化是否未执行 | 执行 SOP 转化 / 写书复盘入库 |
| 元爽点组合无法匹配变体 | **library 内容** | 元爽点-变体映射表.md 是否覆盖本书赛道 | 补变体映射 |
| PRD 高概念但下游不可施工 | **写书 skill** | creative Step 9 验证是否跳过了 world/plot 约束检查 | 修 creative PRD 验证 |

## 三、通用归因规则

### 当不确定归因层时

按以下顺序排查（从近到远）：

1. **写书 skill 自身**：检查 prompt、参数、上下文是否完整。该 skill 的 SKILL.md 是否有红线违规。
2. **写书 skill 的 library 消费**：是否按调用匹配SOP 正确路由到了对应模块？是否走了错误的 fallback？
3. **library 内容**：目标文件是否存在？格式是否完整？索引是否指向正确路径？
4. **入库 SOP**：入库时是否过了质量门槛？去重是否执行？索引是否更新？
5. **拆书 skill**：设计包是否有缺层/缺字段/格式漂移？卷幕划分是否正确？设定归纳是否有 chXX 证据？

### 标记规则

每个发现的归因应标注：

```markdown
> **归因**：{拆书 skill / 入库 SOP / library 内容 / 写书 skill} · {具体文件或步骤}
> **修复**：{一句话动作}
> **触发轮次**：{第 X 章 prose / plot Step Y / qa 全检}
```

## 四、回流触发协议

### 何时触发回流

| 触发条件 | 回流目标 |
|:---------|:---------|
| prose 多次找不到场景卡（≥3次） | → 检查 DNA 入库质量 → 可能退回 pop-shared-dna |
| plot 连续 2 卷剧情线单一 | → 检查剧情库多样性 → 可能需新拆一本书补充 |
| world 无法找到任何参考书 | → 检查设定库 00-索引 → 可能需补索引或拆新书 |
| qa 发现同一套路连续踩同一红线 | → 检查套路卡的红线段是否写清楚 → 修卡 |
| chapter 3 次以上找不到合适套路 | → 检查套路库抽象度是否足够 → 可能需提炼新模式 |

### 回流执行者

- **library 内容问题** → pop-trope-library 维护者
- **入库 SOP 问题** → 拆书管线 or 入库 agent
- **拆书 skill 问题** → pop-decon-* skill 维护者
- **写书 skill 问题** → pop-writer-* skill 维护者

---

> **维护约定**：每次 QA 或 prose/plot 暴露新类型问题后，追加到归因速查表。表不追求一次性完备——它随管线迭代持续生长。
