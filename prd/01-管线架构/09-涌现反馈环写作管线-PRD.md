## 0. 文档定位

本文档是 popwave-skills 写作管线的**终极重构方向**，不是立即实施的代码方案。

当前 v2 管线已转正为默认模式，作为从 v1 雪花式管线到涌现式管线的过渡。本 PRD 描述的是 v2 之后的下一步演进——一次从哲学层面到架构层面的系统性重构。

文档内容源于对目标小说《我在北美搞内战》的拆解分析、v1/v2 管线产出的 A/B 对比、以及对 agent 创作能力边界的深度思考。

## 1. 问题诊断：雪花方法的结构性劣势

### 1.1 当前管线的八阶段雪花结构

当前管线采用雪花式写作方法，自顶向下逐层细化：

```
creative（创意研究）→ world（世界构筑）→ character（角色设计）→ plot（剧情架构）→ chapter（章节设计）→ prose（正文渲染）→ qa（质检）
```

每一阶段产出文档，下一阶段消费上一阶段的文档。信息的传递路径是：设定 → 大纲 → 章纲 → 正文。

### 1.2 agent 信息损失链

雪花方法的底层假设是"更多前置规划 = 更好产出"。对人类作者成立——人类作者脑子里有一个丰富的隐性模型，大纲是对这个模型的有损压缩，但即使压缩了，写的时候原模型仍在脑子里托底。

对 agent **完全不成立**。agent 没有隐性模型。每一层传递都是信息损失：

```
设定层：agent 产出的设定本身就是浅薄的（第一次损失）
  ↓ 压缩为大纲
大纲层：浅薄设定的摘要（第二次损失）
  ↓ 压缩为章纲
章纲层：大纲摘要的摘要（第三次损失）
  ↓ 扩展为正文
正文层：章纲摘要的摘要的扩展（三次损失后渲染）
```

每经过一层，agent 离"创作"更远，离"填表"更近。prose skill 拿到的是第三手信息——章纲的摘要，而不是原始的创作冲动。

### 1.3 A/B 测试验证的差距

通过 v1（纯雪花式）与 v2（改进版雪花式+场景流设计）的 A/B 对比，以及与目标小说的 gap 分析，确认了以下结构性差距：

| 维度 | 目标小说 | v1 | v2 |
|:-----|:---------|:---|:---|
| 每章事件密度 | 11-13个 | 1-2个 | 5-7个 |
| 金手指类型 | 压力型（血量倒计时） | 观察型（看标签） | 工具型（评估/路径） |
| 爽点兑现 | 每章即时兑现 | 三章无实质爽点 | 爽点延迟且偏弱 |
| 危机链 | 不间断（解决一个立刻来一个） | 几乎无 | 间歇式 |
| 主角主动性 | 全程主动出击 | 全程被动应对 | 半主动 |

v2 相比 v1 有改进，但改进是**执行层面的量变**，不是**设计哲学的质变**。核心差距在于：管线关注了世界观一致性、角色弧线完整性、剧情线追踪——这些都是"工程正确性"层面的设计，但**没有显式设计网文的核心爽感机制**。

### 1.4 拆解资料的认知陷阱

目标小说的拆解资料（事件链层、爽点层、设定提取区）是分析者从成品**反推**出来的结构，不是原作者写之前就有的文档。原作者手里可能根本没有这么细的设计包——他脑子里有一座冰山（海明威冰山原则），写出来的只是水面上的部分。

如果照着拆解资料的格式去改设计包模板（加爽点层、加事件链表），我们是在模仿"分析工具"，不是在模仿"创作能力"。模板再精细，agent 仍然是"填表模式"。

## 2. 替代方案：涌现+反馈环

### 2.1 核心范式转变

从"逐层施工图"（雪花方法）转向"最小种子 → 涌现写作 → 反馈纠偏"。

不是纯涌现——纯涌现必定写崩，因为 agent 缺少人类作者的三样东西：长期方向感、节奏直觉、伏笔记忆。

答案是**涌现+反馈环**：用最小种子提供方向，用涌现写作释放创作决策，用反馈环防止失控。

### 2.2 五步闭环

```
最小种子 → 涌现写作 → 反思 → 活记忆更新 → 弧线校准
                ↑                          |
                └──────────────────────────┘
                    （每章循环）
```

1. **最小种子**：极小但极浓的故事引擎，包含压力机制、主角引擎、金手指、冲突轴、目的地
2. **涌现写作**：agent 在写作时刻做创造性决策，种子完整地在上下文中，不是执行预制计划
3. **反思**：后验评估——这章读者获得了什么？压力在吗？节奏对吗？读者会弃书吗？
4. **活记忆更新**：更新种子追踪、压力状态、角色状态、节奏数据
5. **弧线校准**：每10-20章做一次宏观检查，不达标则回退重写

### 2.3 与雪花方法和纯涌现的对比

| 维度 | 雪花方法 | 纯涌现 | 涌现+反馈环 |
|:-----|:---------|:-------|:------------|
| 信息损失 | 3次压缩 | 1次（种子→正文） | 1次（种子→正文） |
| 创作决策点 | 规划阶段做决策，执行阶段填表 | 写作时刻做决策 | 写作时刻做决策 |
| 上下文利用 | 正文 agent 只看到章纲（3手信息） | 写作 agent 直接看到种子（1手信息） | 写作 agent 看到种子+活记忆 |
| 适应性 | 大纲一旦确定，偏离即回退 | 每章自由发挥 | 每章根据反馈调整方向 |
| 防崩机制 | 靠前置规划的完整性 | 无 | 靠后验反馈环的纠偏能力 |

## 3. 三阶段管线（取代八阶段）

### 3.1 阶段一：种子设计

种子不是设定文档，是**故事引擎**。包含五个要素，每个都必须能"生成剧情"：

**压力机制**（最重要）
- 什么倒计时在逼迫主角？不行动的代价是什么？
- 目标小说的范例：血量≤5，每小时掉1点——不是"设定"，是"剧情发电机"
- 种子必须设计可量化的损失倒计时，而非模糊威胁

**主角引擎**
- 不是完整角色卡，是驱动力+缺陷+压力的三元组
- 驱动力：他想要什么？缺陷：什么阻碍他得到？压力：如果得不到会怎样？
- 这三个要素的张力天然生成剧情

**金手指**
- 不是功能清单，是能力+限制+成长路径
- 限制比能力更重要——限制创造困境，困境驱动剧情
- 成长路径给读者"养成感"

**冲突轴**
- 不是势力清单，是核心矛盾的张力线
- 一句话能说清，但能支撑全书

**目的地（5-10个）**
- 不是大纲，是模糊的路标
- "某个时刻，主角会第一次获得真正的力量"
- 不指定章节，只指定方向——故事在涌现过程中自然到达

### 3.2 阶段二：涌现写作环

每章是一个"写→反思→调整"的循环：

**写（涌现）**
- 上下文：种子 + 活记忆（前N章状态）+ 上一章正文
- agent 自由创作，做实时决策
- 没有"章纲"约束——agent 自己决定这章发生什么

**反思（后验评估）**

五个读者体验问题：
1. 这章读者获得了什么？（奖励/技能/信息/情感——至少1个）
2. 这章的压力在吗？（倒计时在走吗？新威胁在逼近吗？）
3. 这章埋了什么？（新种子？伏笔？债务？）
4. 这章的节奏对吗？（对比目标小说同阶段的密度）
5. 读者会继续翻页吗？（章末有没有即时钩子？）

**调整（活记忆更新）**
- 更新种子追踪（哪些已回收，哪些仍在活跃）
- 更新压力状态（倒计时到哪了）
- 更新角色状态（主角的心理/能力/资源变化）
- 更新节奏数据（最近几章的事件密度趋势）
- 设定下一章的"方向提示"（不是章纲，是1-2句话的方向感）

### 3.3 阶段三：弧线校准

每完成一个弧线（10-20章），做一次宏观检查：
- 活了哪些种子？死了哪些？该回收没回收的？
- 战力曲线合理吗？膨胀了吗？
- 角色弧线连贯吗？有没有漂移？
- 宏观节奏波形对吗？有没有连续5章平淡？
- 目的地进度：走到哪了？偏离了吗？

不达标则回退重写，不是打补丁。

### 3.4 三阶段 vs 八阶段映射

| 当前八阶段 | 涌现式三阶段 | 映射关系 |
|:-----------|:-------------|:---------|
| creative | 种子设计·压力机制+冲突轴 | 压缩为种子五要素之一 |
| world | 种子设计·金手指+世界规则 | 压缩为种子五要素之一，深度按爽感分配 |
| character | 种子设计·主角引擎 | 压缩为种子五要素之一 |
| plot | 种子设计·目的地 | 压缩为种子五要素之一，不再产出L2/L3表 |
| chapter | 涌现写作环·写 | 不再有"设计包"，写作时刻做决策 |
| prose | 涌现写作环·写 | chapter与prose合并为单一写作环节 |
| qa | 涌现写作环·反思+弧线校准 | 从事后修补变为实时反馈环 |

## 4. 五个设计原则

### 4.1 北极星重定义

| | 当前 | 改造后 |
|:---|:---|:---|
| 核心数据流 | 设定→角色→剧情→章节→正文（信息传递） | 读者体验目标→爽感循环→事件密度→正文渲染（体验传递） |
| 衡量标准 | 信息完整性、路径正确性 | 读者弃书概率、爽点密度、危机紧迫度 |
| 设计包核心问题 | "这章发生什么？" | "这章读者感受到什么？" |

### 4.2 读者心理模型

当前 pipeline 没有"读者"这个概念，只关注"故事"。但目标小说的成功本质上是对读者心理的精确操控——每章5-6个爽点、危机-喘息-新危机循环、短线债务快速回收。这些都是读者心理学，不是叙事学。

改造方向：在种子设计阶段就建立"读者体验设计"——这本书靠什么吸引读者？压力来自哪里？爽感来自哪里？节奏怎么安排？这份读者体验设计应该**驱动**下游所有环节。

### 4.3 决策驱动

从"模板填空"转向"决策驱动"。每个设计决策都有"为什么这样选"的理由。

- 模板式："ch003 爽点A：老约翰来访，类型：信息获取，强度★★★"（填空）
- 决策式："ch003 选择'信息交换'而非'战斗冲突'，因为前两章都是逃亡，第三章需要降速但不能无趣，用'老约翰知道主角秘密'制造信息不对等的紧张感。代价：本章危机感下降，用'工业路空白区'的长线悬念补偿。"（决策+理由+代价）

### 4.4 设定深度按爽感贡献度分配

当前 world skill 产出均匀的设定清单。但目标小说的设定是不均匀的——力量体系、经济系统、社会映射深到吓人，地理历史浅尝辄止。因为作者知道哪些设定能产生爽感。

设定深度分配：
- 金手指系统/压力机制 → 极深（直接影响每章爽感）
- 经济系统/奖励循环 → 深（支撑"获得感"）
- 社会映射/势力格局 → 中（支撑冲突设计）
- 地理/历史/术语 → 浅（够用即可）

### 4.5 约束+范例+门禁三重机制

如何让没有网文直觉的 agent，产出一个有网文直觉的设计？不可能直接注入直觉，但可以用三重机制逼近效果：

**约束（少而精，体验级而非工程级）**
- "每章必须让读者获得至少1次实质性奖励"（体验级）
- "金手指必须包含可量化的损失倒计时"（体验级）
- "章末必须设置一个下章开头需解决的即时危机"（体验级）

约束要少，但每条都直指读者体验。红线太多了约等于没有红线。

**范例（目标小说拆解资料作为金标准）**
- 不是让 agent 照抄格式，而是让 agent 在做设计决策时参照"目标小说在这个节点是怎么做的"

**门禁（体验模拟环节）**
- 在涌现写作产出正文后，反思层检查"读者读到这章会不会弃书"
- 不达标则回退重新写

## 5. 活记忆组件设计

活记忆不是大纲，不是设定文档，是**随故事增长的状态追踪器**。它不预设"第20章该发生什么"，而是记录"前19章发生了什么、埋了什么、欠了什么"，让 agent 在写第20章时自己决定怎么推进。

| 组件 | 职责 | 防崩类型 | 数据形态 |
|:-----|:-----|:---------|:---------|
| 种子追踪 | 已埋伏笔（章号+内容+预期回收点） | 防线索遗失 | 列表，每条含状态（活跃/已回收/超期预警） |
| 压力状态 | 所有倒计时的当前值 | 防节奏崩坏 | 键值对，含触发条件与剩余值 |
| 角色状态 | 主角的数值/心理/资源/关系变化 | 防人设漂移 | 章末追加的状态增量 |
| 世界规则 | 已确立的规则清单 | 防世界观矛盾 | 只增不改的规则表 |
| 节奏日志 | 每章事件密度/爽点数/钩子类型 | 防节奏崩坏 | 章号+三项指标的日志 |
| 战力曲线 | 主角能力值的时间序列 | 防战力膨胀 | 章号+能力值的折线数据 |
| 目的地进度 | 5-10个目的地的到达状态 | 防失去方向 | 目的地+状态（未到达/接近/已到达） |

活记忆只增不改（除了状态更新），它是**故事的账本**，不是**故事的蓝图**。

### 与现有 state-log.yaml 的关系

活记忆是现有 state-log.yaml 概念的扩展。state-log.yaml 已实现了"append-only narrative logs"的基本思路，但只追踪事件状态。活记忆在此基础上增加了"读者体验追踪"维度（节奏日志、爽点记录、危机链状态），以及"防崩保险"维度（种子追踪、战力曲线、目的地进度）。

演进路径：state-log.yaml → 扩展为活记忆（增加读者体验追踪+防崩保险模块）。

## 6. 与当前实现的过渡路径

### 6.1 v2 转正作为过渡起点

v2 管线已转正为默认模式。v2 相比 v1 的改进（场景流设计、DNA硬阻塞、金手指行动引擎）是有价值的执行层面优化，但不涉及哲学层面重构。v2 在当前雪花式框架内做到了最好，但仍然受限于雪花方法的结构性劣势。

### 6.2 分阶段迁移建议

迁移不应一步到位，应按依赖关系分阶段推进：

**阶段A：验证种子设计**
- 在 creative skill 中试验"种子五要素"设计，替代当前的完整设定文档
- 用种子设计目标小说前3章，验证种子能否驱动涌现写作
- 这是整个架构的最小验证实验

**阶段B：验证涌现写作环**
- 在 chapter+prose skill 中试验"写→反思→调整"循环
- 反思层用目标小说拆解数据作为校准标准
- 验证 agent 能否在无章纲的情况下产出高质量正文

**阶段C：验证活记忆**
- 扩展 state-log.yaml 为活记忆，增加读者体验追踪+防崩保险模块
- 验证活记忆能否在20章尺度上保持连贯性

**阶段D：验证弧线校准**
- 实现每10-20章的宏观检查机制
- 验证回退重写是否可行

### 6.3 风险与应对

| 风险 | 应对 |
|:-----|:-----|
| agent 涌现创作能力不够 | 种子设计必须极强——压力机制能自动生成剧情方向 |
| 反思层不够聪明 | 用目标小说拆解数据作为校准标准，不凭空判断 |
| 长期连贯性失控 | 弧线校准每10-20章宏观检查 + 活记忆种子追踪超期预警 |
| 目的地永远到不了 | 目的地是模糊路标，允许调整；连续20章无进展则强制纠偏 |

## 7. 开放问题

1. **种子设计的最小可行形态**：五要素各自的"最小够用"标准是什么？过简则驱动力不足，过繁则回到雪花方法的老路。
2. **反思层的判定标准**：如何避免反思层变成另一个"填表"——agent 应付式地回答五个问题但不真正反思？
3. **活记忆的膨胀控制**：随章节增长，活记忆会越来越大，如何在保持完整性的同时控制上下文窗口消耗？
4. **弧线校准的回退机制**：回退重写10章的代价比打补丁大得多，如何判断"该回退"vs"该打补丁"的临界点？
5. **多角色视角的涌现**：当前设计以单主角为假设，多视角叙事如何适配涌现+反馈环？
6. **与 trope-library 的集成**：种子设计阶段如何利用 trope-library 的素材原型，而不退回到"填表模式"？
