## 0. 文档定位与背景

### 0.1 本 PRD 的性质

09 愿景 PRD 回答"为什么要做涌现式管线"。本 PRD 回答"怎么做"。

v3.1 管线（涌现式 3 阶段）由 expert-writer v9.0.0 专用调度，skill 命名带 `v3` 前缀。v2 skill 文件保持在 `skills/` 目录不动，但 expert-writer 不再引用 v2。

### 0.2 验证基础

种子文档实验（项目c）已验证：600 字种子产出的 5 章正文，在事件密度、爽点兑现、危机链、主角主动性四个维度上接近目标小说，优于 v1/v2。压缩比 50:1，信息损失从 3 次降为 1 次。

### 0.3 四个架构调整

| 调整 | v3.0 的问题 | v3.1 的方案 |
|:-----|:-----------|:-----------|
| 文风从种子和创作端拆出 | 文风DNA硬阻塞在创作端，分散故事涌现注意力 | 文风DNA降为项目资产（不写入种子），新增修订层子agent负责文风对齐 |
| 信息获取强制化+本地沉淀 | agent自主判断"要不要查"=选择不查，ch12-23持续跳过 | 每章强制读索引→有则读→无则WebSearch→写入知识库→更新索引 |
| 反思+种子生长拆给独立子agent | 同一agent自审天然偏向"没问题"，27章全✅零回退 | 质检子agent（context隔离）做五问反思+种子生长判断+爽点终验 |
| expert-writer 去掉 v2 双轨 | v2逻辑占60%篇幅，每次加载都读v2路由表分散注意力 | v9.0.0 全面服务于v3.1，不再引用v2 skill，认知负荷降低40-60% |

### 0.4 与 v2 的隔离原则

- expert-writer v9.0.0 全面服务于 v3.1，不再引用 v2 skill
- v2 skill 文件保持在 `skills/` 目录不动，如需 v2 可另建 v2 专用调度器
- 共享资产：pop-trope-library、pop-shared-dna
- v3 项目与 v2 项目数据结构不兼容，不可切换

## 1. 问题诊断回顾

### 1.1 雪花方法的结构性劣势

v2 管线采用雪花式方法：creative → world → character → plot → chapter → prose → qa。每一阶段产出文档，下一阶段消费上一阶段。对人类作者成立（脑子里有隐性模型托底），对 agent 完全不成立——agent 没有隐性模型，每一层传递都是信息损失：设定层浅薄（第一次损失）→ 大纲摘要（第二次损失）→ 章纲摘要的摘要（第三次损失）→ prose 拿到第三手信息渲染。

### 1.2 A/B 测试验证的差距

| 维度 | 目标小说 | v2 |
|:-----|:---------|:---|
| 每章事件密度 | 11-13 个 | 5-7 个 |
| 金手指类型 | 压力型 | 工具型 |
| 爽点兑现 | 每章即时 | 延迟且偏弱 |
| 危机链 | 不间断 | 间歇式 |
| 主角主动性 | 全程主动 | 半主动 |

### 1.3 拆解资料的认知陷阱

目标小说的拆解资料是分析者从成品反推的结构，不是原作者写之前就有的文档。照着拆解格式改模板 = 模仿分析工具而非模仿创作能力。

### 1.4 v3.0 28章质量诊断

v3.0 管线运行 28 章后暴露 8 个 Gap（5 执行 + 3 设计），根因是六步同会话执行导致每步都不专注。

**执行 Gap（5 个）：**

| Gap | PRD 设计 | 实际执行 | 根因 |
|:----|:-------|:---------|:-----|
| E1 | Step 1 四类信息检查 | ch12-23 持续跳过 16 章 | "自主判断要不要查"=选择不查 |
| E2 | 红线 5 法则全部验证 | ch002 起从未执行 | 红线无验证，跳过成本为零 |
| E3 | 五问反思引用正文证据 | 证据退化为剧情摘要 | 同一 agent 自审天然偏向"没问题" |
| E4 | 种子自主生长 | 冻结 27 章，0 次生长 | 种子生长判断在 context 污染下退化 |
| E5 | Step 2 自由涌现 | 字数坍塌 53%（ch1-6 avg 3600 → ch20-27 avg 1680） | 6 步同会话，过程产物污染写作 context |

**设计 Gap（3 个）：**

| Gap | PRD 设计 | 实际执行 | 根因 |
|:----|:-------|:---------|:-----|
| D1 | 场景设计 1-3 个 | 只到"场景+目的"级，无内部 beat | 缺乏场景设计粒度指引 |
| D2 | 法则 4"8-12 事件密度" | 只验证不指导 | 法则作为检查项而非创作指引 |
| D3 | "涌现"vs"高密度"矛盾 | 放弃高密度 | 涌现与密度未分到不同 agent 层 |

## 2. 管线全景图

### 2.1 三阶段循环管线（活种子版）

```
                    ┌─── 活种子（六要素）──────────────────┐
                    │  初始化：pop-writer-v3-seed 塑造      │
                    │  生长：质检子agent判断→主会话机械写入  │
                    │  修剪：弧线校准时 agent 自主清理       │
                    │  新鲜度：每条要素带last_updated_ch     │
                    └──────────┬─────────────────────────────┘
                               │
  ┌── expert-writer v9.0.0（v3.1专用调度引擎）──────────────────┐
  │                                                              │
  │  Think：状态感知 + 信息需求判断 + 写作任务包 + 智能调度       │
  │  Execute（调度+3子agent，context隔离）：                     │
  │    ├─ Step 0：本章规划（场景/线索/爽点/危机/钩子+法则对照）   │
  │    ├─ Step 1：信息获取（强制读索引→读/搜→写本地→更新索引）   │
  │    ├─ Step 2：创作子agent（纯故事涌现，不管文风）            │
  │    ├─ Step 3：修订子agent（文风对齐/人设丰富/爽点验证/bug）  │
  │    ├─ Step 4：质检子agent（五问反思+种子生长判断+爽点终验）  │
  │    ├─ Step 5：记忆更新+种子生长+方向提示（主会话机械执行）   │
  │    └─ Step 6：落盘+项目总控更新                              │
  │  Reflect：方向提示 + 弧线校准检查 + 引导                     │
  │                                                              │
  │  校准触发：定期/事件/用户 → pop-writer-v3-arc（修剪+回退）    │
  └──────────────────────────────────────────────────────────────┘
```

### 2.2 管线设计特点

- **活种子（六要素）**：不是一次性产出，是随故事生长的故事引擎——质检子agent在写作中发现新压力源、新线索、新规则时判断是否生长，主会话机械执行写入
- **信息获取强制化**：每章必须读知识库索引，有匹配则读本地文件，无匹配则 WebSearch 并写入知识库沉淀——不再"自主判断要不要查"
- **文风拆出修订层**：文风DNA不进种子、不在创作端硬阻塞，由独立修订子agent负责文风对齐——创作端专注故事涌现
- **3子agent context隔离**：创作/修订/质检三个子agent各自context隔离，消除过程产物污染——解决字数坍塌和反思走过场
- **chapter+prose 合并**：消除章纲→正文的第三次信息损失
- **活记忆替代大纲**：不预设"第 20 章该发生什么"，记录"前 19 章发生了什么"
- **智能中控**：expert-writer v9.0.0 从路由器升级为 v3.1 专用调度引擎，新增信息获取调度、3子agent调度、智能调度三大能力

### 2.3 辅助 Skill

| Skill | 入 | 出 | 说明 |
|:---|---:|---:|:---|
| pop-writer-game v2.0.2 | 种子 + 活记忆 | AI 互动文字游戏 | 不变 |
| pop-writer-html v1.3.2 | 正文/种子 | 单文件 HTML 发布页 | 不变 |

## 4. 阶段一：种子设计——从 creative 复制重建

### 4.1 设计原则：复制不耦合

pop-writer-v3-seed 从 pop-writer-creative 复制而来，但完全独立重建。不引用 creative 的任何文件，不依赖 creative 的任何逻辑。creative 的 4 步流程（生态调研→金手指→文风DNA→PRD）被涌现式改造为 seed 的 4 步（生态调研→压力矩阵→金手指→文风DNA→种子组装）。

**creative 4 步 → seed 4 步映射：**

| creative 步骤 | creative 产出 | 涌现式改造 | seed 步骤 | seed 产出 |
|:---|:---|:---|:---|:---|
| step-1 生态调研 | 生态图谱（≥3 角色群） | 生态图谱→压力矩阵+冲突轴（从"角色群生态"扩展到"制度性压迫维度"） | step-1-ecology-to-pressure | 种子§压力矩阵 + §冲突轴 |
| step-2 金手指 | 金手指行动引擎（5 维度） | 压缩为种子金手指（能力+限制+成长路径） | step-2-golden-finger | 种子§金手指 |
| step-3 文风DNA | 文风DNA锁定声明 | 蒸馏为文风库文件（项目资产，不写入种子） | step-3-dna-lock | `写作资产/文风库/{书名}.md` |
| step-4 PRD产出 | 立项PRD（10 块结构） | 替换为种子组装（主角引擎+成长路径+目的地，creative 不产出的新增） | step-4-seed-assembly | 种子§主角引擎+§成长路径+§目的地+种子验证 |

### 4.2 设定/世界观的压缩原则

v2 的 world 阶段产出（L1-01~07、数值体系、宪法）不单独进种子。世界观以两种形态压缩进种子：

- **压力矩阵**：制度怎么吃人——"医疗缺失""帮派勒索""离婚斩杀"比"北美医疗体系概述"能生成更多剧情
- **冲突轴**：四层生态怎么交叉——"帮派×流浪者×系统×地下"比"势力格局清单"能生成更多困境

数值体系压缩为"成长路径"（能力+心态+资源的成长台阶，不是属性数值表）。

**核心原则：能生成剧情的设定才进种子，不能生成剧情的不进。**

### 4.3 种子六要素总览

| 要素 | 核心问题 | 最小标准 |
|:-----|:---------|:---------|
| 压力矩阵 | 什么在逼迫主角？不行动的代价是什么？ | ≥3 个压力源，至少 1 个可量化倒计时 |
| 主角引擎 | 他想要什么？什么阻碍他？如果得不到会怎样？ | 驱动力三层（表层/中层/深层）+ 缺陷 + 决策风格 |
| 金手指 | 能做什么？不能做什么？怎么变强？ | ≥1 个行动驱动机制，≥2 个限制条件 |
| 冲突轴+活跃线索 | 核心矛盾是什么？当前有哪些暗流？ | ≥3 层生态，≥3 条活跃线索 |
| 成长路径 | 主角的能力/心态/资源如何成长？ | ≥4 个成长阶段，每个对应至少 1 个目的地 |
| 目的地 | 故事的模糊路标在哪？ | 5-10 个目的地 |

> 文风DNA 不再是种子要素，降为项目资产（详见 §4.10）。

种子规模约束：约 600-1500 字（对比 v2 等价前置文档约 31000 字，压缩比 20-50:1）。

### 4.4 压力矩阵

不是单一倒计时，是一整张压力源矩阵。每个压力源能独立杀死主角，多个压力源交叉时产生道德困境。

格式规范：

| 压力源 | 机制 | 杀人方式 | 显性表现 | 活跃阶段 |
|:-------|:-----|:---------|:---------|:---------|
| 医疗缺失 | 无医保，药价极高 | 病了不治就死 | 血量斩杀线（HP<=5） | ch1-3 显性 |
| 帮派勒索 | MS-13/Aryan 控制街区 | 不交钱=被打死 | 巡逻搜人 | 全程 |

每个压力源标注来源（种子搜索法调研结果 / library 参考 / 用户指定），确保不是大模型空想。

### 4.5 主角引擎

驱动力三层设计（解决长线不足的关键）：

| 层级 | 内容 | 激活时机 | 与压力矩阵的关系 |
|:-----|:-----|:---------|:----------------|
| 表层 | 活下去，活好 | ch1 即激活 | 直接应对压力矩阵 |
| 中层 | 保护身边的人 | 遇到需要保护的人时激活 | 天然冲突——保护别人消耗自己资源、暴露位置 |
| 深层 | 推翻/改变制度 | 能力成长+身边人被制度碾碎后浮现 | 从"在被压榨中求生"到"向压榨者开战" |

缺陷：身体/社会身份/资源的三重约束。决策风格：一句话定义。

### 4.6 金手指

能力（按模块解锁）+ 限制（比能力更重要）+ 系统错位/核心创意 + 数值风格（戏剧优先 vs 精确模拟）。

### 4.7 冲突轴+活跃线索

冲突轴：核心矛盾的张力线。四层生态结构（帮派/流浪者/系统/地下），每层独立产生危机，多层交叉产生困境。

活跃线索：3-5 条当前活跃的张力线。每条线索格式：ID + 描述 + 状态 + 涉及生态层。涌现写作时由 agent 自主决定哪条线索在本章推进。

### 4.8 成长路径

主角能力+心态+资源的成长台阶。每一步对应一次"质变时刻"——主角的能力、选择范围、与制度的关系发生不可逆变化。成长路径在 ch4-ch10 的表层驱动力真空期提供中期驱动力。

### 4.9 目的地

模糊路标，不指定章节。允许调整原则：连续 20 章无进展则弧线校准强制纠偏。

### 4.10 文风DNA（项目资产）

项目资产，非种子要素。seed 的 step-3 蒸馏产出文风库文件 `写作资产/文风库/{书名}.md`，供 emerge 修订层子agent加载。硬阻塞（缺失=项目初始化终止）。与 pop-shared-dna 的关系：pop-shared-dna 蒸馏产出 → 写入文风库 → 修订层子agent加载（不写入种子）。

### 4.11 种子验证

种子设计完成后，验证六要素能否生成前 3 章的全部剧情走向。验证不通过 = 种子不够强，退回补强。

## 5. 活种子机制

### 5.1 活种子哲学

种子不是一次性产出的静态文档，是随故事生长的故事引擎。在小说不同阶段，压力源、驱动力、金手指的活跃度都不一样——开局显性的医疗缺失可能在 ch15 后被解决，而强化剂压力源可能在 ch20 才浮现。活种子允许质检子agent在写作过程中发现新要素并判断是否生长，主会话机械执行写入，在弧线校准时自主清理不活跃要素。

**活种子 vs 活记忆：**

| 维度 | 活种子 | 活记忆 |
|:---|:---|:---|
| 性质 | 故事引擎（方向性要素） | 状态账本（append-only 记录） |
| 可变性 | 可生长可修剪 | append-only（除压缩） |
| 内容 | 六要素（压力/引擎/金手指/冲突/成长/目的地） | 七组件状态记录 |
| 作用 | 生成剧情的方向 | 记录已发生的剧情 |

### 5.2 初始化

pop-writer-v3-seed 塑造初始种子（§4）。初始化产出：
- 种子文档.md（v1.0.0，六要素）
- 种子变更日志第一条
- 活记忆 baseline #0

### 5.3 生长：质检子agent判断，主会话机械执行

**核心原则：种子生长由质检子agent（Step 4）判断，主会话（Step 5）机械执行写入。** 质检子agent在审视正文时发现新压力源、新线索、新规则、新目标，判断是否触发种子生长。主会话只负责机械写入，不做判断。

生长场景（示例，非穷举）：

| 触发场景 | 生长要素 | 示例 |
|:---|:---|:---|
| 发现新压力源 | 压力矩阵 | ch5 发现强化剂依赖风险→追加 |
| 发现新线索 | 活跃线索 | ch8 发现巫毒教与裂隙关联→追加 |
| 确立新世界规则 | 世界规则 | ch10 发现系统等级与精神力关系→追加 |
| 成长路径细化 | 成长路径 | ch12 选择巫医职业→阶段4细化 |
| 发现新目标 | 目的地 | ch15 发现裂隙本质线索→追加 |

**质检子agent判断原则：**
- 如果新发现的要素能生成未来剧情 → 建议生长
- 如果只是当前章节的一次性事件 → 不建议生长，只记入活记忆
- 如果不确定 → 建议生长（宁多勿少，弧线校准时可修剪）

生长执行流程：
1. 质检子agent（Step 4）在审视正文时发现新要素 → 判断是否触发种子生长
2. 触发 → 主会话（Step 5）机械写入种子文档对应要素
3. 版本号 minor+1（如 1.0.0→1.1.0）
4. 变更日志追加记录
5. 同步更新活记忆对应组件

### 5.4 修剪：弧线校准时 agent 自主判断

**核心原则：权限全给 agent。** 弧线校准时 agent 自主判断哪些要素不再活跃，标记关闭或回收。

修剪场景（示例，非穷举）：

| 修剪场景 | 修剪要素 | 判定依据 |
|:---|:---|:---|
| 压力源已解决 | 压力矩阵 | 倒计时归零且不再激活 |
| 活跃线索已回收 | 活跃线索 | 线索悬念已兑现 |
| 目的地已到达 | 目的地 | 故事已涌现到该路标 |
| 线索长期休眠 | 活跃线索 | 连续 10 章未触及 |
| 要素新鲜度过期 | 任意要素 | last_updated_ch距今>15章且无活跃迹象→标记休眠 |

**last_updated_ch新鲜度判据**：每条种子要素带 `last_updated_ch` 字段（初始化为0，种子生长时更新为当前章号）。弧线校准时检查：`当前章号 - last_updated_ch > 15` 且无活跃迹象（活记忆中无引用） → 标记休眠。

**修剪原则：不删除历史，只标记状态。** 已关闭/已完结/已休眠的要素保留在种子文档的"已关闭区"，供版本追溯。

种子文档头部版本元数据 + 底部变更日志：

...

...

---
## 变更日志
| 版本 | 章号 | 类型 | 变更内容 | 触发原因 |
|:-----|:-----|:-----|:---------|:---------|

### 5.6 活种子与活记忆的同步关系

| 活种子要素 | 活记忆组件 | 同步动作 |
|:---|:---|:---|

### 5.7 last_updated_ch 新鲜度感知

每条种子要素（压力源/线索/规则/目的地等）带 `last_updated_ch` 字段，用于感知要素变化新鲜度。

| 字段 | 初始化 | 更新时机 | 消费者 |
|:---|:---|:---|:---|
| `last_updated_ch` | 0（种子设计时） | 种子生长时更新为当前章号 | 质检子agent、弧线校准 |

**agent 可感知要素变化新鲜度：**
- `当前章号 - last_updated_ch <= 5`：要素近期有变化，活跃
- `当前章号 - last_updated_ch 6-15`：要素稳定，正常
- `当前章号 - last_updated_ch > 15` 且无活跃迹象：要素可能休眠，需关注（修剪/进化/激活）

质检子agent在种子生长判断时参考 `last_updated_ch`：如果一条压力源 `last_updated_ch` 距今已超过 15 章且活记忆中无引用，可在质检报告中建议标记休眠。弧线校准时作为正式修剪判据（§5.4）。

## 6. 信息获取机制

### 6.1 从自主判断到强制化获取

v3.0 把信息获取交给 agent 自主判断"要不要查"，结果是 ch12-23 持续跳过 16 章——"自主判断要不要查"等于"选择不查"。v3.1 改为强制化获取：每章必须读知识库索引，有匹配则读本地文件，无匹配则 WebSearch 并写入知识库沉淀。

种子设计阶段仍保留种子搜索法（用于压力矩阵的制度性压迫维度调研），但写作过程中的信息获取改为强制化流程。

**token 便宜原则：** 告诉 agent 不要吝啬搜索新信息沉淀到本地。WebSearch + 写入知识库的 token 成本远低于因信息缺失导致的质量退化成本。宁可多搜多沉淀，不可跳过。

### 6.2 强制化流程

每章写作前（pop-writer-v3-emerge step-1），**必须读知识库索引.md**（强制），然后做 4 类检查：

| 需求类型 | 判断问题 | 信息源 | 示例 |
|:---|:---|:---|:---|
| 制度/流程细节 | 这章涉及什么我不熟悉的现实制度/流程？ | WebSearch → 知识库/制度流程/ | 美国催债法流程、庇护所申请、帮派等级制 |
| 同类场景写法 | 这章的场景类型需要参考技法吗？ | library 套路库 → 知识库/场景技法/ | 帮派对峙场景、系统面板展示、绝境反杀 |
| 文风渲染参考 | 这章的文风渲染需要参考吗？ | library 文风库 | 冷硬对话节奏、暴力场景感官描写 |
| 已埋伏笔/前文细节 | 这章需要呼应哪些已埋伏笔？ | 项目本身（活记忆种子追踪+前章正文） | MS-13 在找谁、老约翰之死、前身背景 |

**强制化获取流程：**
1. **读索引**：读 `写作资产/知识库/索引.md`（强制，门禁：未读取=退回补读）
2. **有匹配**：索引中有相关条目 → 读知识库本地文件
3. **无匹配**：索引中无相关条目 → WebSearch 搜索 → 提取关键信息 → 写入知识库对应子目录 → 更新索引.md
4. **无需求**：4 类检查均无明确需求 → 允许 `needs: []`，但索引必须已读

**核心原则：索引必须读，信息必须沉淀。** 即使本章无信息需求，索引也必须读取（确认无遗漏）。搜索到的新信息必须写入知识库本地沉淀，不可只用不存。

### 6.3 获取流程

1. **读索引**：读 `写作资产/知识库/索引.md`，检查是否有与本章相关的已沉淀信息
2. **有匹配 → 读本地**：索引中有相关条目 → 读知识库对应文件（如 `知识库/制度流程/美国催债法.md`）
3. **无匹配 → WebSearch**：搜索制度/流程细节/场景技法，提取关键信息
4. **写入知识库**：将搜索结果写入知识库对应子目录（本地沉淀，不丢失）
5. **更新索引**：在 `索引.md` 中追加新条目（文件路径+摘要+关联章号）
6. **library**：查询套路库/文风库，提取场景技法/渲染参考
7. **项目本身**：查阅活记忆种子追踪+前章正文，提取待回收伏笔

**知识库目录结构（详见 §6.6）：** `写作资产/知识库/` 下设 4 个子目录——制度流程/、场景技法/、创意参考/、世界观素材/，外加 `索引.md`。

获取的信息**必须沉淀到知识库本地文件**（WebSearch 结果），不可只用不存。library 查询和项目本身的信息直接融入写作上下文，在信息获取记录中登记。

## 7. 阶段二：涌现写作环

### 7.1 调度+3子agent循环

```
Step 0        Step 1          Step 2           Step 3          Step 4          Step 5              Step 6
本章规划  →  信息获取     →  创作子agent  →  修订子agent  →  质检子agent  →  记忆更新+种子生长  →  落盘
(主会话)     (强制化)         (context隔离)   (context隔离)   (context隔离)   (主会话机械执行)     (主会话)
                                ↑                  ↑               │
                                └──────────────────┴───────────────┤ 不通过→回退Step 2或3
                                                                  │
    ↑                                                             │
    └─────────────────────────────────────────────────────────────┘
```

### 7.2 本章规划（先想再查再写）

写之前先想明白。读种子六要素+上一章正文+活记忆+方向提示，判断这章要怎么写——5个决策点（场景/线索/爽点/危机/钩子），然后对照网文爽感机制10条法则检查。规划做完，再进信息获取（强制化）。

网文爽感机制是可配置的写作法则文件（`references/网文爽感机制.md`），包含10条法则：压力机制/即时奖励循环/不间断危机链/高事件密度/主角主动决策/养成感/信息披露节奏/多线交叉/情绪闭环。（注：80%展示20%内省法则移至修订层子agent执行，不在创作端检查。）当法则冲突时按优先级处理（压力机制>即时奖励>危机链>...）。未来探索其他领域时，替换为对应领域的写作法则。

### 7.3 信息获取（强制化）

详见 §6。每章强制读知识库索引，有匹配则读本地文件，无匹配则 WebSearch 并写入知识库沉淀。产出 `info_acquired` 记录。

### 7.4 创作子agent（context隔离）——chapter+prose 合并

**子agent，context隔离。** 输入：种子六要素+活记忆（最后 baseline + event）+ 上一章正文末尾+chapter_plan+info_acquired+创作模板。

合并理由：消除 chapter→prose 的信息损失。设计决策和渲染执行在人类作者脑中是同时发生的，分离导致 prose 拿到的是"设计摘要"而非"创作冲动"。

**纯故事涌现，不管文风。** agent 自由创作，做实时决策，没有章纲约束。核心任务：
- 场景流渲染（逐事件展开，不跳跃）
- 压力源逼近（每章至少一个压力源推进）
- 章末钩子（即时危机/悬念）

**不做的事（移到修订层）：** 文风对齐、80%展示20%内省、AI观感词清理——这些由修订子agent负责。

输出：正文初稿+创作决策记录。

### 7.4.1 修订子agent（context隔离）

**子agent，context隔离。** 输入：初稿+文风DNA（文风库文件）+种子六要素+活记忆+修订checklist。

5 项修订任务：

| 任务 | 内容 | 门禁 |
|:---|:---|:---|
| 文风对齐 | 对照文风库文件，调整句式/感官/距离/节奏 | 文风匹配度<80%=退回 |
| 人设丰富 | 补充角色心理/行为细节，防人设漂移 | — |
| 爽点验证 | 检查爽点是否兑现，不足则强化 | 无爽点=退回 |
| bug修复 | 逻辑漏洞/时间线矛盾/设定冲突 | 有硬bug=退回 |
| AI观感词清理 | 清理"不禁""竟然""仿佛"等AI高频词 | AI观感词>3种=退回 |

**80%展示20%内省法则在此层执行。** 创作端不管文风，修订端统一对齐。

输出：修订稿+修订记录（YAML）。

### 7.5 质检子agent（context隔离）

**子agent，context隔离。** 输入：修订稿+活记忆+种子六要素+网文法则+质检模板。

3 项质检任务：

| 任务 | 内容 | 不通过处理 |
|:---|:---|:---|
| 五问反思 | 五个读者体验问题，**每个问题必须引用正文具体段落作为证据** | 无正文引用=补引用 |
| 种子生长判断 | 审视正文是否发现新压力源/新线索/新规则，判断是否触发种子生长（§5.3） | — |
| 爽点终验 | 爽点是否兑现、压力是否在走、章末是否有钩子 | 不达标=回退 |

**五问反思表：**

| 问题 | 判断标准 | 不达标处理 |
|:-----|:---------|:-----------|
| 读者获得了什么？ | 至少 1 次实质奖励 | 回退Step 2（故事层） |
| 压力在吗？ | 倒计时在走 或 新威胁在逼近 | 回退Step 2（故事层） |
| 节奏对吗？ | 事件密度 8-12 个/章 | 回退Step 2（故事层） |
| 线索追踪正常吗？ | 超期伏笔预警；新种子已登记 | 标记方向提示 |
| 读者会翻页吗？ | 章末有即时钩子 | 回退Step 2（故事层） |

**质检不通过 → 回退目标：**
- 故事层问题（事件密度/压力/爽点/钩子）→ 回退 Step 2（创作子agent重写）
- 文风/人设层问题（文风匹配/人设漂移/AI观感词）→ 回退 Step 3（修订子agent重修）

防"应付式填表"机制：每个问题必须有具体正文引用作为证据。

### 7.6 记忆更新+种子生长写入+方向提示（主会话机械执行）

质检通过后，主会话机械执行写入。读质检报告 → 机械写入活记忆七组件 + 种子生长（如质检子agent建议生长）+ 方向提示。**主会话不做判断，只机械执行质检子agent的结论。**

### 7.7 方向提示生成

1-2 句话的方向感，不是章纲。格式：

```
方向提示 ch004:
- 必须给奖励（ch3只有信息，无实质获得）
- 压力需加码（血量倒计时已3章未推进）
- 回收短线：MS-13追杀线已3章未出现，需触面
- 钩子必须是即时危机
```

方向提示不指定具体事件——只说"需要给奖励、需要加压、需要触面MS-13、需要即时钩子"。具体怎么实现，由写作 agent 涌现。

### 7.8 每章执行 SOP

1. **Step 0**：本章规划（种子六要素+上章+活记忆+方向提示+网文爽感机制→5决策点+法则对照）
2. **Step 1**：信息获取（强制读索引→有则读→无则WebSearch→写知识库→更新索引，产出info_acquired）
3. **Step 2**：创作子agent（context隔离，纯故事涌现：场景流+压力源+钩子，不管文风）
4. **Step 3**：修订子agent（context隔离，文风对齐/人设丰富/爽点验证/bug修复/AI观感词清理）
5. **Step 4**：质检子agent（context隔离，五问反思引用正文证据+种子生长判断+爽点终验，不通过→回退Step 2或3）
6. **Step 5**：记忆更新+种子生长写入+方向提示（主会话机械执行质检报告结论）
7. **Step 6**：落盘 + 项目总控更新

## 8. 活记忆系统

### 8.1 与 state-log.yaml 的演进关系

v2 用 state-log.yaml（双组件：baseline+event）。v3 用活记忆.yaml（七组件），概念同源结构不同。演进路径：state-log.yaml → 活记忆（七组件）。

### 8.2 七大组件

| 组件 | 职责 | 防崩类型 | 数据形态 |
|:-----|:-----|:---------|:---------|
| 种子追踪 | 已埋伏笔（章号+内容+预期回收点） | 防线索遗失 | 列表，每条含状态（活跃/已回收/超期预警） |
| 压力状态 | 所有倒计时的当前值 | 防节奏崩坏 | 键值对，含触发条件与剩余值 |
| 角色状态 | 主角的数值/心理/资源/关系变化 | 防人设漂移 | 章末追加的状态增量 |
| 世界规则 | 已确立的规则清单 | 防世界观矛盾 | 只增不改的规则表 |
| 节奏日志 | 每章事件密度/爽点数/钩子类型 | 防节奏崩坏 | 章号+三项指标的日志 |
| 战力曲线 | 主角能力值的时间序列 | 防战力膨胀 | 章号+能力值的折线数据 |
| 目的地进度 | 5-10 个目的地的到达状态 | 防失去方向 | 目的地+状态（未到达/接近/已到达） |

### 8.3 防膨胀压缩机制

压缩触发条件：每 20 章 event 累积后。压缩策略：读最后一个 baseline + 之后全部 event → 合并成新 baseline → 删旧 baseline 及其后 event → 追加新 baseline。压缩后文件回到 5-15KB。

### 8.4 谁读谁写

| skill | 步骤 | 操作 |
|-------|------|------|
| pop-writer-v3-seed | 产出后 | 初始化活记忆 baseline #0 |
| pop-writer-v3-emerge | Step 1 | 读活记忆（最后 baseline + event）→ 取当前状态 |
| pop-writer-v3-emerge | Step 4 | 追加 event 到活记忆 + 种子生长同步 —— 唯一写入者 |
| pop-writer-v3-arc | 检查时 | 读活记忆全量 → 宏观检查 |
| pop-writer-v3-arc | 压缩时 | 合并 baseline + event → 新 baseline |
| pop-writer-v3-arc | 回退时 | 删 entries 中 chapter > N 的条目 |

## 9. 阶段三：弧线校准

### 9.1 校准触发条件

- 定期触发：每 10-20 章自动触发（建议值 15 章）
- 事件触发：活记忆任意组件触发预警
- 用户触发：用户主动要求宏观检查

### 9.2 宏观检查清单

| 检查项 | 检查内容 | 预警条件 |
|:-------|:---------|:---------|
| 种子追踪 | 活了哪些？死了哪些？该回收没回收的？ | >20 章未回收 |
| 战力曲线 | 膨胀了吗？跳级了吗？ | 连续跳级 |
| 角色弧线 | 连贯吗？驱动力三层演化进度 | 漂移或停滞 |
| 宏观节奏 | 波形对吗？有没有连续 5 章平淡？ | >=3 章低密度 |
| 目的地进度 | 走到哪了？偏离了吗？ | 连续 20 章无进展 |
| 冲突轴 | 活跃线索还活着吗？需要新线索吗？ | >=10 章未触及 |

### 9.3 种子修剪

弧线校准时，agent 自主判断哪些种子要素不再活跃（§5.4）。修剪不删除历史，只标记状态（已关闭/已完结/已到达/休眠）。

### 9.4 回退机制

| 问题严重度 | 判定标准 | 处理方式 |
|:----------|:---------|:---------|
| 轻微 | 单章质量问题，不影响后续 | 打补丁（重写该章） |
| 中等 | 2-3 章累积偏差，活记忆可吸收 | 微调方向提示 + 后续章节自然纠正 |
| 严重 | 弧线级偏移 | 回退重写（删 entries + 重写 10-20 章） |

回退执行流程：输出校准报告 → 用户确认回退范围 → 删活记忆 entries 中 chapter > N 的条目 → 删正文 ch > N 的文件 → 从第 N+1 章重新进入涌现写作环。

## 10. expert-writer 中控层（v9.0.0 v3.1专用调度引擎）

### 10.1 从 8 阶段线性到 3 阶段循环 + v3.1专用调度

| 维度 | v3.1（3 阶段循环+调度+3子agent） |
|:-----|:-------------------|
| 管线结构 | 种子设计→涌现写作环（调度+3子agent）↔弧线校准 |
| 中控层 | v3.1专用调度引擎（信息获取调度+3子agent调度+智能调度） |
| 执行模式 | 循环迭代，每章一个循环，3子agent context隔离 |
| 回退粒度 | 按章回退（删 entries + 正文），质检不通过按层回退Step 2/3 |
| v2引用 | 无（v9.0.0全面服务于v3.1，不再引用v2 skill） |

### 10.2 Think 层（v3.1）

v3.1 Think 层 5 项职责：
1. **状态感知**：种子状态感知（当前版本+变更日志+六要素）+ 活记忆感知（七组件）+ 章号+弧线感知 + 方向提示感知
2. **意图识别**：开新书/继续/写第X章/检查/回滚
3. **信息需求判断**：识别本章需要的外部信息（4 类检查），交由 Step 1 强制化获取
4. **写作任务包生成**：整合种子六要素+活记忆+信息需求+方向提示，生成本章写作任务包
5. **智能调度判断**：根据状态决定下一步动作（正常→写作环 / 校准触发点→弧线校准 / 质检不通过→回退Step 2/3 / 严重偏差→校准）

### 10.3 Execute 层（v3.1，调度+3子agent）

v3.1 Execute 层 6 项职责（调度+3子agent，context隔离）：
1. **Step 0 本章规划调度**：调度本章规划（场景/线索/爽点/危机/钩子+法则对照）
2. **Step 1 信息获取调度**：强制读索引→读/搜→写知识库→更新索引
3. **Step 2 创作子agent调度**：传入精简context（种子六要素+活记忆+chapter_plan+info_acquired+创作模板），纯故事涌现
4. **Step 3 修订子agent调度**：传入初稿+文风库文件+修订checklist，文风对齐/人设/爽点/bug/AI观感词
5. **Step 4 质检子agent调度**：传入修订稿+活记忆+种子+网文法则+质检模板，五问反思+种子生长判断+爽点终验
6. **Step 5+6 记忆更新+落盘**：主会话机械执行质检报告结论 → 落盘+项目总控更新

### 10.4 Reflect 层（v3.1）

v3.1 Reflect 层 4 项职责（种子生长判断已移至质检子agent）：
1. **方向提示生成**：基于质检报告结论生成1-2句方向感
2. **弧线校准检查**：检查是否触发弧线校准条件
3. **项目总控回写**：更新 3 阶段+章号+弧线计数+种子版本
4. **引导**：完成后引导用户下一步

### 10.5 三大核心能力（v3.1）

| 核心能力 | 所在层 | 职责 |
|:---|:---|:---|
| 信息获取强制化 | Think 层+Execute 层 | 每章强制读索引→读/搜→写知识库→更新索引，不再自主判断 |
| 3子agent调度 | Execute 层 | 创作/修订/质检3子agent context隔离调度，消除过程产物污染 |
| 智能调度 | Think 层+Execute 层 | 根据状态决定写作/校准/回退，质检不通过按层回退Step 2/3 |

### 10.6 闸门机制

| 闸门 | 触发时机 | 内容 |
|:--------|:---------|:-----|
| 种子确认 | 种子设计完成 | 种子六要素完整+文风库文件已蒸馏+验证通过 |
| 弧线校准确认 | 每 10-20 章 | 校准报告确认（含种子修剪确认+last_updated_ch新鲜度检查） |

### 10.7 v3.1 路由表

| 用户说 | v3.1 路由 |
|:-------|:----------|
| "开新书/启动项目" | pop-writer-v3-seed |
| "继续/下一步" | 查项目总控（章号+阶段）→ 涌现写作环或弧线校准 |
| "写第 X 章" | 涌现写作环（调度+3子agent，直接写） |
| "检查/审稿" | 弧线校准（或质检子agent报告） |

## 17. 开放问题与风险

1. **活种子生长的边界**：质检子agent判断生长后，主会话机械执行可能导致过度生长——靠弧线校准修剪兜底，种子规模超 1500 字时 agent 自主优先修剪
2. **信息获取强制化的开销**：每章强制读索引+搜索+沉淀可能增加 token 开销——token 便宜原则：信息沉淀的 token 成本远低于质量退化成本，宁可多搜多沉淀
3. **活记忆膨胀控制**：七组件比 state-log 更大——压缩后是否可控（已定义压缩机制，待验证）
4. **弧线校准回退代价**：回退重写 10 章的代价——已定义判定标准，待验证
5. **多角色视角的涌现**：当前设计以单主角为假设——待后续迭代
6. **3子agent context隔离的调度开销**：每章调度3个子agent可能增加执行时间——context隔离带来的质量提升应大于调度开销，待v3.1 28章验证
