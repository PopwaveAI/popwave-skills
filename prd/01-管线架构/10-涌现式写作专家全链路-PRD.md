## 0. 文档定位与背景

### 0.1 本 PRD 的性质

本文档已吸收原 09 愿景 PRD 的全部内容（"为什么要做涌现式管线"的哲学论证），并回答"怎么做"。

v3.3 管线（涌现式 3 阶段）由 expert-writer v9.2.0 唯一调度，skill 命名带 `v3` 前缀。v2 skill 文件保持在 `skills/` 目录不动，但 expert-writer 不再引用 v2。

### 0.2 验证基础

种子文档实验（项目c）已验证：600 字种子产出的 5 章正文，在事件密度、爽点兑现、危机链、主角主动性四个维度上接近目标小说，优于 v1/v2。压缩比 50:1，信息损失从 3 次降为 1 次。

### 0.3 五个架构调整（v3.1→v3.2）

| 调整 | v3.1 的问题 | v3.2 的方案 |
|:-----|:-----------|:-----------|
| emerge 拆分为独立 skill | 3子agent是emerge内嵌step文件，context隔离靠纪律非架构 | emerge降为纯调度器，create/revise/qa各自独立skill，context隔离从纪律变架构边界 |

### 0.4 v3.1→v3.2 迭代动机

v3.1 运行3章后暴露4个核心问题：

## 1. 问题诊断回顾

### 1.1 雪花方法的结构性劣势

v2 管线采用雪花式方法：creative → world → character → plot → chapter → prose → qa。每一阶段产出文档，下一阶段消费上一阶段。对人类作者成立（脑子里有隐性模型托底），对 agent 完全不成立——agent 没有隐性模型，每一层传递都是信息损失：设定层浅薄（第一次损失）→ 大纲摘要（第二次损失）→ 章纲摘要的摘要（第三次损失）→ prose 拿到第三手信息渲染。

### 1.2 A/B 测试验证的差距

| 维度 | 目标小说 | v2 |
|:-----|:---------|:---|
| 每章事件密度 | 11-13 个 | 5-7 个 |
| 金手指类型 | 压力型 | 工具型 |
| 爽点兑现 | 每章即时 | 延迟且偏弱 |
| 危机链 | 不间断 | 间歇式 |
| 主角主动性 | 全程主动 | 半主动 |

### 1.3 v3.0 28章质量诊断

v3.0 管线运行 28 章后暴露 8 个 Gap（5 执行 + 3 设计），根因是六步同会话执行导致每步都不专注。

### 1.4 v3.1 3章质量诊断

v3.1 运行3章后暴露4个问题：creative研究档案缺失、搜索沉淀不足（3章仅2次搜索每次~15-20行）、主角行为出戏（CIA精锐不灭口）、emerge未真正拆分（3子agent是step文件非独立skill）。

### 2.2 管线设计特点

- **活种子（核心卖点+六要素+行为准则）**：不是一次性产出，是随故事生长的故事引擎。核心卖点定义"这本书卖什么"，六要素定义"故事引擎怎么运转"，行为准则定义"主角怎么行动"
- **seed step-0创意共创**：六要素不是agent自己产出，是和用户逐步共创出来的。step-0通过元素融合（相似/相反/盲区）+核心卖点提炼+用户QA，在调研前建立创作共识
- **expert-writer主会话7步循环→3独立子skill**：create/revise/qa各自独立skill，context隔离从纪律约束变为架构边界——解决字数坍塌和反思走过场
- **主角行为准则贯穿**：种子主角引擎新增行为准则字段，创作层逐场景检查→修订层对齐→质检层终验，三层贯穿解决主角出戏
- **统一素材库**：研究档案（seed产出）+场景素材（emerge产出）+知识沉淀（emerge搜索）统一管理，MD内容+YAML front matter索引
- **设定库（v3.3新增）**：v2雪花法设定资料按需嵌入——seed可选产出（世界宪法/力量体系/社会结构/角色档案/卷纲路标），emerge Step 1按需读取。参考但不写死，不是线性消费链
- **信息缺口引导搜索**：emerge Step 0本章规划产出信息缺口字段，引导Step 1强制搜索——从"判断要不要查"变为"Step 0已列出缺口，Step 1必须填"
- **搜索深度标准**：每篇知识沉淀≥500字+必须含具体案例/流程/数据+多轮搜索策略（最多3轮）+无需求需逐类说明理由
- **文风拆出修订层**：文风DNA不进种子、不在创作端硬阻塞，由独立修订子skill负责文风对齐
- **chapter+prose 合并**：消除章纲→正文的第三次信息损失
- **活记忆替代大纲**：不预设"第 20 章该发生什么"，记录"前 19 章发生了什么"

## 4. 阶段一：种子设计——创意共创→六要素构建

### 4.1 设计原则：seed吸收creative最佳实践，独立不耦合

pop-writer-v3-seed 吸收 creative 的最佳实践（元素融合方法、搜索SOP），但完全独立。不引用 creative 的任何文件，不依赖 creative 的任何逻辑。

**第一步是seed不是creative。** seed新增step-0创意共创阶段，在调研前与用户共创核心卖点和创作方向。六要素不是agent自己产出，是和用户逐步共创出来的——每步产出后强制用户QA确认。

**seed 5步流程：**

| seed 步骤 | 内容 | 产出 | 用户QA |
|:---|:---|:---|:---|
| **step-0 创意共创** | 元素融合（相似/相反/盲区）+核心卖点提炼+创作方向共识 | 创意共识单（核心卖点+融合元素清单+6维度方向） | ✅ 用户确认创意共识单 |
| step-1 三界压力调研→压力矩阵 | 基于创意共识做三界压力调研→压力矩阵+冲突轴+研究档案持久化 | 种子/压力矩阵.md+种子/冲突轴.md + 素材库/研究档案/ | ✅ 用户确认压力源和冲突轴 |
| step-2 金手指 | 金手指行动引擎设计 | 种子/金手指.md | ✅ 用户确认金手指能力+限制 |
| step-3 文风DNA | 文风DNA蒸馏 | `写作资产/文风库/{书名}.md`（项目资产） | ✅ 用户确认文风方向 |
| step-4 种子组装 | 主角引擎（含行为准则）+成长路径+目的地+种子验证 | 种子/主角引擎.md+成长路径.md+目的地.md+_index.yaml+_log.md | ✅ 用户确认完整种子 |

### 4.2 核心卖点（种子元数据，v3.2新增）

核心卖点回答"读者为什么要看这本书"，是种子文件夹的顶层元数据（非第七要素），引导六要素的构建方向。

| 提炼维度 | 问题 | 示例 |
|:---------|:-----|:-----|
| 核心爽点 | 读者最期待什么体验？ | CIA精锐用战术碾压底层混混的反差爽感 |
| 差异化 | 和同类作品比有什么不同？ | 不是重生/穿越，是真实技能+系统面板的混合 |
| 情绪承诺 | 读者读完会 feeling 什么？ | 从绝望到掌控的逆袭快感 |

核心卖点在step-0创意共创阶段提炼，经用户确认后写入 `种子/核心卖点.md`。后续emerge写作时，create子skill可参考核心卖点确保每章兑现情绪承诺。

### 4.3 step-0创意共创流程

**核心原则：用户不一定在这个时候就有非常好的创作idea，需要共创。**

1. **用户初始元素收集**：用户可能只有一句话（"我想写CIA特工在美国底层生存"），哪怕只有一句话也要开始共创
2. **元素融合（agent主动补充）**：在用户提供元素基础上，agent主动补充三类元素供用户判断

| 补充类型 | 做什么 | 示例 |
|:---------|:-----|:-----|
| **相似元素** | 与用户元素相近的变体，拓宽选择面 | "CIA特工"→补充"退役/叛逃/卧底/被陷害" |
| **相反元素** | 与用户元素对立的选项，制造张力 | "底层生存"→补充"上层堕落/中产破产" |
| **盲区元素** | 用户完全没提到但相关的元素 | "系统面板"→补充"系统是敌对的/碎片化的/需献祭" |

3. **核心卖点提炼**：从融合元素清单中提炼核心卖点（一句话+展开）
4. **创作方向共识**：6维度方向引导后续六要素构建
5. **用户QA确认**：用户确认创意共识单后才进step-1

### 4.4 种子六要素总览

| 要素 | 核心问题 | 最小标准 |
|:-----|:---------|:---------|
| 压力矩阵 | 什么在逼迫主角？不行动的代价是什么？ | ≥3 个压力源，至少 1 个可量化倒计时 |
| 主角引擎 | 他想要什么？什么阻碍他？**他会怎么行动？** | 驱动力三层+缺陷+决策风格+**行为准则**（背景映射/战斗风格/道德底线/行为禁忌） |
| 金手指 | 能做什么？不能做什么？怎么变强？ | ≥1 个行动驱动机制，≥2 个限制条件 |
| 冲突轴+活跃线索 | 核心矛盾是什么？当前有哪些暗流？ | ≥3 层生态，≥3 条活跃线索 |
| 成长路径 | 主角的能力/心态/资源如何成长？ | ≥4 个成长阶段，每个对应至少 1 个目的地 |
| 目的地 | 故事的模糊路标在哪？ | 5-10 个目的地 |

> 文风DNA 不再是种子要素，降为项目资产。行为准则是主角引擎的子字段（v3.2新增），不是独立第七要素。

### 4.5 压力矩阵（三界框架）

不是单一倒计时，是一整张压力源矩阵。每个压力源能独立杀死主角，多个压力源交叉时产生道德困境。

**三界压力框架**（v3.3新增）：不同类型小说的压力来源截然不同，单一框架无法覆盖。将所有压力源归入三个域，按需调研：

| 压力界 | 核心问题 | 适用类型 |
|:-------|:---------|:---------|
| 社会界 | 人吃人的制度是什么？ | 现代/都市/历史/职场 |
| 自然界 | 环境怎么杀人？ | 末日/生存/荒野/科幻 |
| 超自然界 | 超自然法则怎么压迫？ | 玄幻/仙侠/灵异/异能 |

大多数小说混合使用2-3个界。基于 Step 0 创意共识单判定涉及哪些界，分界调研。

格式规范：

| 压力源 | 界属 | 机制 | 杀人方式 | 显性表现 | 活跃阶段 | 来源标注 | last_updated_ch |
|:-------|:-----|:-----|:---------|:---------|:---------|:---------|:----------------|
| {压力源1} | 社会/自然/超自然 | {怎么运作} | {不行动的后果} | {读者看到什么} | {什么章节显性} | {调研来源} | 0 |
| {压力源2} | | | | | | | 0 |

每个压力源标注来源（种子搜索法调研结果 / library 参考 / 用户指定），确保不是大模型空想。交叉困境至少1个跨界困境。

### 4.6 主角引擎（含行为准则）

驱动力三层设计（解决长线不足的关键）：

| 层级 | 内容 | 激活时机 | 与压力矩阵的关系 |
|:-----|:-----|:---------|:----------------|
| 表层 | 活下去，活好 | ch1 即激活 | 直接应对压力矩阵 |
| 中层 | 保护身边的人 | 遇到需要保护的人时激活 | 天然冲突——保护别人消耗自己资源、暴露位置 |
| 深层 | 推翻/改变制度 | 能力成长+身边人被制度碾碎后浮现 | 从"在被压榨中求生"到"向压榨者开战" |

缺陷：身体/社会身份/资源的三重约束。决策风格：一句话定义。

**行为准则（v3.2新增）**：主角的背景决定其行为模式。每本书的主角风格不同，行为准则是"这个背景的人在这本书里怎么行动"的映射。

| 子项 | 内容 | 示例 |
|:-----|:-----|:-----|
| 背景映射 | 主角背景→行为规则映射 | CIA黑色行动精锐→不留活口/先手压制/战术优先/消灭威胁优先于谈判 |
| 战斗风格 | 一句话定义战斗方式 | CQB近身压制+无声击杀+利用环境 |
| 道德底线 | 不可逾越的红线 | 不杀平民/不背叛盟友 |
| 行为禁忌 | 绝不会做的事 | 不谈判/不留活口/不暴露身份 |

设计原则：
- 行为准则必须与驱动力三层一致（驱动力决定"为什么做"，行为准则决定"怎么做"）
- 行为准则必须与缺陷有张力（如"不留活口"的准则 × "资源匮乏"的缺陷 = 不得不冒险处理尸体）
- 行为准则不是设定堆砌，是行为预测器——写作时能据此判断"主角在这个场景会怎么行动"

三者叠加才能预测主角的完整行为链：为什么→怎么决定→怎么做。

### 4.7 金手指

能力（按模块解锁）+ 限制（比能力更重要）+ 系统错位/核心创意 + 数值风格（戏剧优先 vs 精确模拟）。

### 4.8 冲突轴+活跃线索

冲突轴：核心矛盾的张力线。四层生态结构（帮派/流浪者/系统/地下），每层独立产生危机，多层交叉产生困境。

活跃线索：3-5 条当前活跃的张力线。每条线索格式：ID + 描述 + 状态 + 涉及生态层 + last_updated_ch。

### 4.9 成长路径

主角能力+心态+资源的成长台阶。每一步对应一次"质变时刻"——主角的能力、选择范围、与制度的关系发生不可逆变化。

### 4.10 目的地

模糊路标，不指定章节。允许调整原则：连续 20 章无进展则弧线校准强制纠偏。

### 4.11 文风DNA（项目资产）

项目资产，非种子要素。seed 的 step-3 蒸馏产出文风库文件 `写作资产/文风库/{书名}.md`，供 revise子skill加载。硬阻塞（缺失=项目初始化终止）。

### 4.12 研究档案持久化（v3.2新增）

seed step-1 的三界压力调研独立存盘到 `素材库/研究档案/`，不再只压缩进种子表格：

| 产出文件 | 内容 | 与种子的关系 |
|:---------|:-----|:-------------|
| `种子展开图-{书名}.md` | 三界压力调研详情（界属判定+分界调研+压力源候选+外部参照） | 种子压力矩阵是压缩版，展开图是详情版 |
| `交叉困境分析-{书名}.md` | 压力源交叉困境详细分析（组合/困境描述/道德张力/可衍生剧情线） | 种子冲突轴是压缩版，交叉困境分析是详情版 |

**核心原则：种子是引擎索引，研究档案是引擎详情。** 创作子skill可读种子获取压缩版压力矩阵，也可读研究档案获取完整调研深度。

### 4.13 种子验证

种子设计完成后，验证六要素能否生成前 3 章的全部剧情走向。验证不通过 = 种子不够强，退回补强。

## 5. 活种子机制

### 5.1 活种子哲学

种子不是一次性产出的静态文档，是随故事生长的故事引擎。活种子允许质检子skill在写作过程中发现新要素并判断是否生长，主会话机械执行写入，在弧线校准时自主清理不活跃要素。

### 5.2 生长：质检子skill判断，主会话机械执行

**核心原则：种子生长由质检子skill（Step 4）判断，主会话（Step 5）机械执行写入。**

生长场景（示例，非穷举）：

| 触发场景 | 生长要素 | 示例 |
|:---|:---|:---|
| 发现新压力源 | 压力矩阵 | ch5 发现强化剂依赖风险→追加 |
| 发现新线索 | 活跃线索 | ch8 发现巫毒教与裂隙关联→追加 |
| 确立新世界规则 | 世界规则 | ch10 发现系统等级与精神力关系→追加 |
| 成长路径细化 | 成长路径 | ch12 选择巫医职业→阶段4细化 |
| 发现新目标 | 目的地 | ch15 发现裂隙本质线索→追加 |
| **主角行为准则演化** | **主角引擎.行为准则** | **ch20从"不留活口"浮现"选择性留活口"→行为禁忌更新** |

### 5.3 修剪：弧线校准时 agent 自主判断

修剪场景包括：压力源已解决、活跃线索已回收、目的地已到达、线索长期休眠、要素新鲜度过期。

**last_updated_ch新鲜度判据**：每条种子要素带 `last_updated_ch` 字段。`当前章号 - last_updated_ch > 15` 且无活跃迹象 → 标记休眠。

### 5.4 版本管理

版本号规则（语义化）：
- **major**（X.0.0）：种子要素结构变化（新增/删除要素类型）
- **minor**（1.X.0）：生长（追加新压力源/新线索/新规则/新目的地/行为准则演化）
- **patch**（1.0.X）：修剪/调整（关闭压力源、回收线索、调整细节）

## 6. 统一素材库与信息获取机制

### 6.1 统一素材库目录结构（v3.2新增）

```text
素材库/
├── 索引.md                    # 统一索引（YAML front matter + MD表格，强制读取入口）
├── 研究档案/                  # seed阶段产出（研究过程持久化）
│   ├── 种子展开图-{书名}.md   # 三界压力调研详情
│   └── 交叉困境分析-{书名}.md # 压力源交叉困境详细分析
├── 场景素材/                  # emerge阶段产出（替代v2素材储备池，按需生成）
│   └── {场景类型}-素材卡.md
└── 知识沉淀/                  # emerge搜索产出（替代v2散落知识库）
    ├── 制度流程/              # 现实制度/流程细节（WebSearch 沉淀）
    ├── 场景技法/              # 场景写法参考（library 沉淀）
    ├── 创意参考/              # 创意灵感/对标参考
    └── 伏笔追踪/              # 伏笔追踪记录
```

**YAML vs MD 决策**：MD文件做内容（agent直接读取无需解析），YAML front matter做索引元数据（机器可读的路由信息）。纯YAML不适合长文本内容，纯MD无结构化元数据。

**索引.md 格式：**

```markdown
---
last_updated: ch{章号}
last_updated_by: {seed/emerge-ch{N}}
total_files: {N}
categories:
  研究档案: {N}
  场景素材: {N}
  知识沉淀: {N}
---

# 素材库索引

## 研究档案/
| 文件 | 创建阶段 | 关键词 | 简述 |
|:-----|:---------|:-------|:-----|
| 种子展开图-{书名}.md | seed | 制度/压力/调研 | 三界压力调研详情（社会界/自然界/超自然界） |
| 交叉困境分析-{书名}.md | seed | 困境/交叉/道德 | 压力源交叉困境详细分析 |

## 场景素材/
| 文件 | 创建章号 | 关键词 | 简述 |

## 知识沉淀/
### 制度流程/
### 场景技法/
### 创意参考/
### 伏笔追踪/
```

### 6.2 信息获取强制化流程

每章写作前（expert-writer Step 1），**必须读素材库索引.md**（强制），然后做 4 类检查：

| 需求类型 | 判断问题 | 信息源 | 示例 |
|:---|:---|:---|:---|
| 制度/流程细节 | 这章涉及什么我不熟悉的现实制度/流程？ | WebSearch → 素材库/知识沉淀/制度流程/ | 美国催债法流程、庇护所申请、帮派等级制 |
| 同类场景写法 | 这章的场景类型需要参考技法吗？ | library 套路库 → 素材库/知识沉淀/场景技法/ | 帮派对峙场景、系统面板展示、绝境反杀 |
| 已埋伏笔/前文细节 | 这章需要呼应哪些已埋伏笔？ | 项目本身（活记忆种子追踪+前章正文） | MS-13 在找谁、老约翰之死 |
| 创意参考 | 这章需要灵感参考吗？ | WebSearch/library → 素材库/知识沉淀/创意参考/ | 金手指变体、特殊场景设计 |

**强制化获取流程：**
1. **读索引**：读 `素材库/索引.md`（强制，门禁：未读取=退回补读）
2. **有匹配**：索引中有相关条目 → 读素材库本地文件
3. **无匹配**：索引中无相关条目 → WebSearch 搜索 → 写入素材库对应子目录 → 更新索引.md
4. **无需求**：4 类检查均无明确需求 → 允许 `needs: []`，但**必须逐类说明不需求理由**

### 6.3 搜索深度标准（v3.2新增）

| 维度 | 标准 | 不达标处理 |
|:-----|:-----|:-----------|
| 文件字数 | 每篇知识沉淀文件≥500字 | 补充搜索结果至达标 |
| 内容深度 | 必须含具体案例/流程步骤/数据数值（至少其一） | 追搜补充具体细节 |
| 多轮搜索 | 首次搜索结果不足→追搜（换关键词/换角度），最多3轮 | 3轮仍不足→标注"信息有限"并继续 |
| 结构化 | 按知识沉淀文件模板结构化 | 重新整理 |

**多轮搜索策略：**
1. 首次搜索：用 chapter_plan 提取的核心关键词搜索
2. 首次结果评估：是否有具体案例/流程/数据？字数是否≥500？
3. 不足→追搜：换关键词（更具体）/ 换角度（从案例反查流程）/ 换信息源
4. 最多3轮，3轮后仍不足→标注"信息有限，创作时酌情处理"

### 6.4 无需求门槛提高（v3.2新增）

4类检查均无明确需求 → `needs: []` 仍合法，但**必须逐类说明不需求理由**：

```yaml
info_acquired:
  needs: []
  no_need_reasons:
    制度流程: "{一句话理由，如'本章为已建立的捡垃圾机制日常运作，无新制度涉及'}"
    场景技法: "{一句话理由，如'本章场景类型已熟练，无需技法参考'}"
    已埋伏笔: "{一句话理由，如'本章为全新场景，与前文无直接关联'}"
    创意参考: "{一句话理由，如'本章为常规生存场景，无需灵感参考'}"
```

**门禁：no_need_reasons 四项未全部填写 = 退回补填。** 连续3章 `needs: []` → 标记信息获取退化预警。

### 6.5 信息获取记录（info_acquired）

```yaml
info_acquired:
  chapter: 3
  index_read: true
  needs:
    - type: 制度/流程
      question: 美国催债公司的追债流程和手段
```

## 7. 阶段二：涌现写作环

### 7.1 expert-writer主会话7步循环→3子skill循环

```
Step 0        Step 1          Step 2                Step 3               Step 4               Step 5                 Step 6
本章规划  →  信息获取     →  调度create子skill  →  调度revise子skill  →  调度qa子skill     →  记忆更新+种子生长   →  落盘
(主会话)     (强制化)         (context隔离)        (context隔离)        (context隔离)        (主会话机械执行)       (主会话)
                                ↑                       ↑                      │
                                └───────────────────────┴──────────────────────┤ 不通过→回退Step 2或3
                                                                                │
    ↑                                                                         │
    └─────────────────────────────────────────────────────────────────────────┘
```

### 7.2 本章规划（Step 0，主会话）

写之前先想明白。读种子核心卖点+六要素+上一章正文+活记忆+方向提示，判断这章要怎么写——5个决策点（场景/线索/爽点/危机/钩子），然后对照网文爽感机制10条法则检查。**规划产出信息缺口（info_gaps）**，列出本章需要搜索的信息点，引导Step 1强制搜索。规划做完，再进信息获取（强制化）。

### 7.3 信息获取（Step 1，主会话）

详见 §6。**优先处理Step 0产出的信息缺口（info_gaps）**——`must_search: true`的缺口必须WebSearch，`must_search: false`的先查素材库再决定。处理完缺口后做4类补充检查。每章强制读素材库索引，有匹配则读本地文件，无匹配则 WebSearch 并写入素材库沉淀。搜索方法论复用creative的"关键词→搜→筛→存"四步法。产出 `info_acquired` 记录。搜索深度标准≥500字+多轮搜索。无需求需逐类说明理由。

### 7.4 调度创作子skill（Step 2，context隔离）

**pop-writer-v3-create v1.0.0，context隔离。**

expert-writer组装精简context（不传会话历史）→调用pop-writer-v3-create→收集产出→门禁检查。

**输入：** 种子六要素（含行为准则）+活记忆+上章末尾+chapter_plan+info_acquired+创作模板

**核心任务：**
- 场景流渲染（逐事件展开，不跳跃）
- 压力源逼近（每章至少一个压力源推进）
- 章末钩子（即时危机/悬念）
- **主角行为一致性检查（v3.2新增）**：加载行为准则→逐场景行为预测→行为一致性验证→行为禁忌检查

**主角行为一致性检查流程：**

| 步骤 | 检查内容 | 不一致时处理 |
|:-----|:---------|:-------------|
| 1. 加载行为准则 | 背景映射/战斗风格/道德底线/行为禁忌 | 准则缺失=退回seed补全 |
| 2. 逐场景行为预测 | 基于行为准则预测"主角在这个场景会怎么行动" | — |
| 3. 行为一致性验证 | 正文中的主角行为是否符合预测 | 不符合=修正行为描写 |
| 4. 行为禁忌检查 | 正文是否触碰行为禁忌 | 触碰=修正（除非有显性破例剧情） |

**出戏检查红线：**

| 出戏类型 | 表现 | 处理 |
|:---------|:-----|:-----|
| 背景不符 | CIA精锐不灭口、战术动作简单 | 退回重写该场景 |
| 道德底线越界 | 无显性破例剧情却触碰底线 | 退回重写 |
| 行为禁忌违反 | 无合理原因却做禁忌之事 | 退回重写 |

**不做的事（移到修订层）：** 文风对齐、80%展示20%内省、AI观感词清理。

**输出：** 正文初稿+创作决策记录。

### 7.5 调度修订子skill（Step 3，context隔离）

**pop-writer-v3-revise v1.0.0，context隔离。**

expert-writer组装精简context→调用pop-writer-v3-revise→收集产出→门禁检查。

**输入：** 正文初稿+文风DNA（文风库文件）+种子六要素+活记忆+修订checklist

5 项修订任务：

| 任务 | 内容 | 门禁 |
|:---|:---|:---|
| 文风对齐 | 对照文风库文件，调整句式/感官/距离/节奏 | 文风匹配度<80%=退回 |
| 人设丰富 | 补充角色心理/行为细节，防人设漂移 + **行为准则对齐** | — |
| 爽点验证 | 检查爽点是否兑现，不足则强化 | 无爽点=退回 |
| bug修复 | 逻辑漏洞/时间线矛盾/设定冲突 | 有硬bug=退回 |
| AI观感词清理 | 清理"不禁""竟然""仿佛"等AI高频词 | AI观感词>3种=退回 |

**行为准则对齐（v3.2新增）**：检查正文行为是否符合种子行为准则（背景映射/战斗风格/道德底线/行为禁忌），战斗场景的战术动作是否符合战斗风格定义。不符合=修正行为描写。

**输出：** 修订稿+修订记录（YAML）。

### 7.6 调度质检子skill（Step 4，context隔离）

**pop-writer-v3-qa v1.0.0，context隔离。**

expert-writer组装精简context→调用pop-writer-v3-qa→收集质检报告→门禁检查。

**输入：** 修订稿+活记忆+种子六要素+网文法则+质检模板

4 项质检任务：

| 任务 | 内容 | 不通过处理 |
|:---|:---|:---|
| 五问反思 | 五个读者体验问题，**每个问题必须引用正文具体段落作为证据** | 无正文引用=补引用 |
| **行为一致性终验** | 背景映射一致性+战斗风格一致性+道德底线/禁忌检查 | 任一❌=回退create子skill |
| 种子生长判断 | 审视正文是否发现新压力源/新线索/新规则，判断是否触发种子生长 | — |
| 爽点终验 | 爽点是否兑现、压力是否在走、章末是否有钩子 | 不达标=回退 |

**行为一致性终验（v3.2新增）：**

| 检查项 | 判断标准 | 正文引用证据 | 结果 |
|:-------|:---------|:-------------|:-----|
| 背景映射一致性 | 主角行为符合背景映射的行为规则 | {引用正文} | ✅/❌ |
| 战斗风格一致性 | 战斗场景的战术动作符合战斗风格定义 | {引用正文} | ✅/❌ |
| 道德底线/禁忌 | 无显性破例剧情时不触碰底线/禁忌 | {引用正文或"无触碰"} | ✅/❌ |

> 行为一致性任一❌ → 回退 pop-writer-v3-create（故事层问题）

**五问反思表：**

| 问题 | 判断标准 | 不达标处理 |
|:-----|:---------|:-----------|
| 读者获得了什么？ | 至少 1 次实质奖励 | 回退create子skill（故事层） |
| 压力在吗？ | 倒计时在走 或 新威胁在逼近 | 回退create子skill（故事层） |
| 节奏对吗？ | 事件密度 8-12 个/章 | 回退create子skill（故事层） |
| 线索追踪正常吗？ | 超期伏笔预警；新种子已登记 | 标记方向提示 |
| 读者会翻页吗？ | 章末有即时钩子 | 回退create子skill（故事层） |

**质检不通过 → 回退目标：**
- 故事层问题（事件密度/压力/爽点/钩子/行为一致性）→ 回退 Step 2（调度create子skill重写）
- 文风/人设层问题（文风匹配/人设漂移/AI观感词/行为准则对齐）→ 回退 Step 3（调度revise子skill重修）

**输出：** 质检报告+通过/不通过+回退目标。

### 7.7 记忆更新+种子生长写入+方向提示（Step 5，主会话机械执行）

质检通过后，主会话机械执行写入。读质检报告 → 机械写入活记忆七组件 + 种子生长（如质检子skill建议生长）+ 方向提示。**主会话不做判断，只机械执行质检子skill的结论。**

### 7.8 落盘+项目总控更新（Step 6，主会话）

正文落盘 → 项目总控（章号+1/种子版本/弧线触发）→ 检查弧线校准触发条件 → 触发则交接v3-arc，未触发则回Step 0。

### 7.9 每章执行 SOP

1. **Step 0**（主会话）：本章规划（种子六要素+上章+活记忆+方向提示+网文爽感机制→5决策点+法则对照）
2. **Step 1**（主会话）：信息获取（强制读素材库索引→有则读→无则WebSearch→写素材库→更新索引，≥500字+多轮搜索，产出info_acquired）
3. **Step 2**（调度create子skill，context隔离）：纯故事涌现（场景流+压力源+钩子+**行为一致性检查**），不管文风
4. **Step 3**（调度revise子skill，context隔离）：文风对齐/人设丰富含**行为准则对齐**/爽点验证/bug修复/AI观感词清理
5. **Step 4**（调度qa子skill，context隔离）：五问反思引用正文证据+**行为一致性终验**+种子生长判断+爽点终验，不通过→回退Step 2或3
6. **Step 5**（主会话机械执行）：记忆更新+种子生长写入+方向提示
7. **Step 6**（主会话）：落盘 + 项目总控更新 + 弧线触发检查

## 8. 活记忆系统

### 8.1 七大组件

| 组件 | 职责 | 防崩类型 | 数据形态 |
|:-----|:-----|:---------|:---------|
| 种子追踪 | 已埋伏笔（章号+内容+预期回收点） | 防线索遗失 | 列表，每条含状态（活跃/已回收/超期预警） |
| 压力状态 | 所有倒计时的当前值 | 防节奏崩坏 | 键值对，含触发条件与剩余值 |
| 角色状态 | 主角的数值/心理/资源/关系变化 | 防人设漂移 | 章末追加的状态增量 |
| 世界规则 | 已确立的规则清单 | 防世界观矛盾 | 只增不改的规则表 |
| 节奏日志 | 每章事件密度/爽点数/钩子类型 | 防节奏崩坏 | 章号+三项指标的日志 |
| 战力曲线 | 主角能力值的时间序列 | 防战力膨胀 | 章号+能力值的折线数据 |
| 目的地进度 | 5-10 个目的地的到达状态 | 防失去方向 | 目的地+状态（未到达/接近/已到达） |

### 8.2 防膨胀压缩机制

压缩触发条件：每 20 章 event 累积后。压缩策略：读最后一个 baseline + 之后全部 event → 合并成新 baseline → 删旧 baseline 及其后 event → 追加新 baseline。压缩后文件回到 5-15KB。

### 8.3 谁读谁写

| skill | 步骤 | 操作 |
|-------|------|------|
| pop-writer-v3-seed | 产出后 | 初始化活记忆 baseline #0 |
| expert-writer | Step 1 | 读活记忆（最后 baseline + event）→ 取当前状态 |
| expert-writer | Step 5 | 追加 event 到活记忆 + 种子生长同步 —— 唯一写入者 |
| pop-writer-v3-arc | 检查时 | 读活记忆全量 → 宏观检查 |
| pop-writer-v3-arc | 压缩时 | 合并 baseline + event → 新 baseline |
| pop-writer-v3-arc | 回退时 | 删 entries 中 chapter > N 的条目 |

## 9. 阶段三：弧线校准

### 9.1 校准触发条件

- 定期触发：每 10-20 章自动触发（建议值 15 章）
- 事件触发：活记忆任意组件触发预警
- 用户触发：用户主动要求宏观检查

### 9.2 宏观检查清单（含行为准则演化追踪）

| 检查项 | 检查内容 | 预警条件 |
|:-------|:---------|:---------|
| 种子追踪 | 活了哪些？死了哪些？该回收没回收的？ | >20 章未回收 |
| 战力曲线 | 膨胀了吗？跳级了吗？ | 连续跳级 |
| 角色弧线 | 连贯吗？驱动力三层演化进度 + **行为准则演化追踪** | 漂移或停滞 |
| 宏观节奏 | 波形对吗？有没有连续 5 章平淡？ | >=3 章低密度 |
| 目的地进度 | 走到哪了？偏离了吗？ | 连续 20 章无进展 |
| 冲突轴 | 活跃线索还活着吗？需要新线索吗？ | >=10 章未触及 |

**行为准则演化追踪（v3.2新增）**：检查主角行为准则是否随成长路径合理演化，是否存在行为准则与驱动力矛盾。

### 9.3 种子修剪

弧线校准时，agent 自主判断哪些种子要素不再活跃。修剪不删除历史，只标记状态（已关闭/已完结/已到达/休眠）。

### 9.4 回退机制

| 问题严重度 | 判定标准 | 处理方式 |
|:----------|:---------|:---------|
| 轻微 | 单章质量问题，不影响后续 | 打补丁（重写该章） |
| 中等 | 2-3 章累积偏差，活记忆可吸收 | 微调方向提示 + 后续章节自然纠正 |
| 严重 | 弧线级偏移 | 回退重写（删 entries + 重写 10-20 章） |

## 14. 开放问题与风险

1. **活种子生长的边界**：质检子skill判断生长后，主会话机械执行可能导致过度生长——靠弧线校准修剪兜底
2. **素材库膨胀控制**：研究档案（seed一次性）+知识沉淀（emerge每章可能新增）——索引.md提供路由，agent按需读取
3. **行为准则演化管理**：行为准则随成长路径演化时需保持与驱动力一致——arc层行为准则演化追踪兜底
4. **3子skill调度开销**：每章调度3个独立子skill可能增加执行时间——context隔离带来的质量提升应大于调度开销
5. **qa子skill自包含副本同步**：网文爽感机制.md和活种子生长触发规则.md在qa中复制副本——需同步更新
6. **旧种子兼容性**：旧种子（单文件版，无行为准则字段）——create子skill加载时检测缺失→退回seed补全
