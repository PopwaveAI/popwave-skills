## 0. 文档定位与背景

### 0.1 本 PRD 的性质

09 愿景 PRD 回答"为什么要做涌现式管线"。本 PRD 回答"怎么做"。

v2 管线（雪花式 8 阶段）保持在 `skills/` 目录不动，作为 AB 测试对照组。v3 管线（涌现式 3 阶段）独立新建，skill 命名带 `v3` 前缀，与 v2 零耦合。

### 0.2 验证基础

种子文档实验（项目c）已验证：600 字种子产出的 5 章正文，在事件密度、爽点兑现、危机链、主角主动性四个维度上接近目标小说，优于 v1/v2。压缩比 50:1，信息损失从 3 次降为 1 次。

### 0.3 四个修正方向

| 修正 | v1.0 的问题 | v2.0 的方案 |
|:-----|:-----------|:-----------|
| 种子复用 creative 流程 | v1.0 从零新建 seed skill | 从 creative 复制重建为 pop-writer-v3-seed，独立不耦合 |
| 信息注入自主判断 | v1.0 把注入限定在种子阶段，路径预设 | 涌现写作前 agent 自主判断"这章需要什么信息" |
| 活种子机制 | v1.0 明确写"种子设计阶段（一次性）" | 种子可生长可修剪，权限全给 agent 自由判断 |
| 中控层强化 | v1.0 仅把 8 阶段换成 3 阶段，中控层太薄 | expert-writer 升级为智能写作指挥官，Think/Execute/Reflect 各层新增核心能力 |

### 0.4 与 v2 的共存原则

- v2 skill 保持在 `skills/` 目录不动
- v3 skill 独立新建在 `skills/` 目录，带 `v3` 前缀
- 共享资产：pop-trope-library、pop-shared-dna、state-log 概念
- expert-writer 支持双轨（v2/v3），项目初始化时确定模式，不可切换
- AB 测试：同一种子，双管线产出，三方对比（v2 vs v3 vs 目标小说）

## 1. 问题诊断回顾

### 1.1 雪花方法的结构性劣势

v2 管线采用雪花式方法：creative → world → character → plot → chapter → prose → qa。每一阶段产出文档，下一阶段消费上一阶段。对人类作者成立（脑子里有隐性模型托底），对 agent 完全不成立——agent 没有隐性模型，每一层传递都是信息损失：设定层浅薄（第一次损失）→ 大纲摘要（第二次损失）→ 章纲摘要的摘要（第三次损失）→ prose 拿到第三手信息渲染。

### 1.2 A/B 测试验证的差距

| 维度 | 目标小说 | v2 |
|:-----|:---------|:---|
| 每章事件密度 | 11-13 个 | 5-7 个 |
| 金手指类型 | 压力型 | 工具型 |
| 爽点兑现 | 每章即时 | 延迟且偏弱 |
| 危机链 | 不间断 | 间歇式 |
| 主角主动性 | 全程主动 | 半主动 |

### 1.3 拆解资料的认知陷阱

目标小说的拆解资料是分析者从成品反推的结构，不是原作者写之前就有的文档。照着拆解格式改模板 = 模仿分析工具而非模仿创作能力。

## 2. 管线全景图

### 2.1 三阶段循环管线（活种子版）

```
                    ┌─── 活种子 ────────────────────────────┐
                    │  初始化：pop-writer-v3-seed 塑造       │
                    │  生长：涌现写作中 agent 自主追加新要素  │
                    │  修剪：弧线校准时 agent 自主清理        │
                    └──────────┬─────────────────────────────┘
                               │
  ┌── expert-writer v8.0.0（智能中控）──────────────────────────┐
  │                                                              │
  │  Think：状态感知 + 信息需求判断 + 写作任务包 + 智能调度       │
  │  Execute：                                                    │
  │    ├─ 信息获取（WebSearch/library/项目·agent 自主判断）       │
  │    ├─ 涌现写作（种子+活记忆+文风DNA+增量信息 → 正文）         │
  │    ├─ 反思（五问后验评估）                                    │
  │    └─ 活记忆更新 + 种子生长                                   │
  │  Reflect：方向提示 + 弧线校准检查 + 种子生长判断 + 引导       │
  │                                                              │
  │  校准触发：定期/事件/用户 → pop-writer-v3-arc（修剪+回退）    │
  └──────────────────────────────────────────────────────────────┘
```

### 2.2 管线设计特点

- **活种子**：不是一次性产出，是随故事生长的故事引擎——agent 在写作中发现新压力源、新线索、新规则时自主追加到种子
- **信息需求自主判断**：每章写作前 agent 自己判断"需要什么信息"，主动去获取（搜索/library/项目本身），不是预设路径
- **chapter+prose 合并**：消除章纲→正文的第三次信息损失
- **活记忆替代大纲**：不预设"第 20 章该发生什么"，记录"前 19 章发生了什么"
- **智能中控**：expert-writer 从路由器升级为调度器，新增信息需求判断、活种子管理、智能调度三大能力

### 2.3 辅助 Skill

| Skill | 入 | 出 | 说明 |
|:---|---:|---:|:---|
| pop-writer-game v2.0.2 | 种子 + 活记忆 | AI 互动文字游戏 | 不变 |
| pop-writer-html v1.3.2 | 正文/种子 | 单文件 HTML 发布页 | 不变 |

## 4. 阶段一：种子设计——从 creative 复制重建

### 4.1 设计原则：复制不耦合

pop-writer-v3-seed 从 pop-writer-creative 复制而来，但完全独立重建。不引用 creative 的任何文件，不依赖 creative 的任何逻辑。creative 的 4 步流程（生态调研→金手指→文风DNA→PRD）被涌现式改造为 seed 的 4 步（生态调研→压力矩阵→金手指→文风DNA→种子组装）。

**creative 4 步 → seed 4 步映射：**

| creative 步骤 | creative 产出 | 涌现式改造 | seed 步骤 | seed 产出 |
|:---|:---|:---|:---|:---|
| step-1 生态调研 | 生态图谱（≥3 角色群） | 生态图谱→压力矩阵+冲突轴（从"角色群生态"扩展到"制度性压迫维度"） | step-1-ecology-to-pressure | 种子§压力矩阵 + §冲突轴 |
| step-2 金手指 | 金手指行动引擎（5 维度） | 压缩为种子金手指（能力+限制+成长路径） | step-2-golden-finger | 种子§金手指 |
| step-3 文风DNA | 文风DNA锁定声明 | 前置到种子阶段，引用路径+风格关键词 | step-3-dna-lock | 种子§文风DNA |
| step-4 PRD产出 | 立项PRD（10 块结构） | 替换为种子组装（主角引擎+成长路径+目的地，creative 不产出的新增） | step-4-seed-assembly | 种子§主角引擎+§成长路径+§目的地+种子验证 |

### 4.2 设定/世界观的压缩原则

v2 的 world 阶段产出（L1-01~07、数值体系、宪法）不单独进种子。世界观以两种形态压缩进种子：

- **压力矩阵**：制度怎么吃人——"医疗缺失""帮派勒索""离婚斩杀"比"北美医疗体系概述"能生成更多剧情
- **冲突轴**：四层生态怎么交叉——"帮派×流浪者×系统×地下"比"势力格局清单"能生成更多困境

数值体系压缩为"成长路径"（能力+心态+资源的成长台阶，不是属性数值表）。

**核心原则：能生成剧情的设定才进种子，不能生成剧情的不进。**

### 4.3 种子七要素总览

| 要素 | 核心问题 | 最小标准 |
|:-----|:---------|:---------|
| 压力矩阵 | 什么在逼迫主角？不行动的代价是什么？ | ≥3 个压力源，至少 1 个可量化倒计时 |
| 主角引擎 | 他想要什么？什么阻碍他？如果得不到会怎样？ | 驱动力三层（表层/中层/深层）+ 缺陷 + 决策风格 |
| 金手指 | 能做什么？不能做什么？怎么变强？ | ≥1 个行动驱动机制，≥2 个限制条件 |
| 冲突轴+活跃线索 | 核心矛盾是什么？当前有哪些暗流？ | ≥3 层生态，≥3 条活跃线索 |
| 成长路径 | 主角的能力/心态/资源如何成长？ | ≥4 个成长阶段，每个对应至少 1 个目的地 |
| 目的地 | 故事的模糊路标在哪？ | 5-10 个目的地 |
| 文风DNA | 这本书读起来什么感觉？ | 引用文风库路径 + 3-5 个风格关键词 |

种子规模约束：约 600-1500 字（对比 v2 等价前置文档约 31000 字，压缩比 20-50:1）。

### 4.4 压力矩阵

不是单一倒计时，是一整张压力源矩阵。每个压力源能独立杀死主角，多个压力源交叉时产生道德困境。

格式规范：

| 压力源 | 机制 | 杀人方式 | 显性表现 | 活跃阶段 |
|:-------|:-----|:---------|:---------|:---------|
| 医疗缺失 | 无医保，药价极高 | 病了不治就死 | 血量斩杀线（HP<=5） | ch1-3 显性 |
| 帮派勒索 | MS-13/Aryan 控制街区 | 不交钱=被打死 | 巡逻搜人 | 全程 |

每个压力源标注来源（种子搜索法调研结果 / library 参考 / 用户指定），确保不是大模型空想。

### 4.5 主角引擎

驱动力三层设计（解决长线不足的关键）：

| 层级 | 内容 | 激活时机 | 与压力矩阵的关系 |
|:-----|:-----|:---------|:----------------|
| 表层 | 活下去，活好 | ch1 即激活 | 直接应对压力矩阵 |
| 中层 | 保护身边的人 | 遇到需要保护的人时激活 | 天然冲突——保护别人消耗自己资源、暴露位置 |
| 深层 | 推翻/改变制度 | 能力成长+身边人被制度碾碎后浮现 | 从"在被压榨中求生"到"向压榨者开战" |

缺陷：身体/社会身份/资源的三重约束。决策风格：一句话定义。

### 4.6 金手指

能力（按模块解锁）+ 限制（比能力更重要）+ 系统错位/核心创意 + 数值风格（戏剧优先 vs 精确模拟）。

### 4.7 冲突轴+活跃线索

冲突轴：核心矛盾的张力线。四层生态结构（帮派/流浪者/系统/地下），每层独立产生危机，多层交叉产生困境。

活跃线索：3-5 条当前活跃的张力线。每条线索格式：ID + 描述 + 状态 + 涉及生态层。涌现写作时由 agent 自主决定哪条线索在本章推进。

### 4.8 成长路径

主角能力+心态+资源的成长台阶。每一步对应一次"质变时刻"——主角的能力、选择范围、与制度的关系发生不可逆变化。成长路径在 ch4-ch10 的表层驱动力真空期提供中期驱动力。

### 4.9 目的地

模糊路标，不指定章节。允许调整原则：连续 20 章无进展则弧线校准强制纠偏。

### 4.10 文风DNA

种子第七要素。引用路径 `写作资产/文风库/{书名}.md`，硬阻塞（缺失=种子设计终止）。与 pop-shared-dna 的关系：pop-shared-dna 蒸馏产出 → 写入文风库 → 种子引用。

### 4.11 种子验证

种子设计完成后，验证七要素能否生成前 3 章的全部剧情走向。验证不通过 = 种子不够强，退回补强。

## 5. 活种子机制

### 5.1 活种子哲学

种子不是一次性产出的静态文档，是随故事生长的故事引擎。在小说不同阶段，压力源、驱动力、金手指的活跃度都不一样——开局显性的医疗缺失可能在 ch15 后被解决，而强化剂压力源可能在 ch20 才浮现。活种子允许 agent 在写作过程中自主发现新要素并追加，在弧线校准时自主清理不活跃要素。

**活种子 vs 活记忆：**

| 维度 | 活种子 | 活记忆 |
|:---|:---|:---|
| 性质 | 故事引擎（方向性要素） | 状态账本（append-only 记录） |
| 可变性 | 可生长可修剪 | append-only（除压缩） |
| 内容 | 七要素（压力/引擎/金手指/冲突/成长/目的地/文风） | 七组件状态记录 |
| 作用 | 生成剧情的方向 | 记录已发生的剧情 |

### 5.2 初始化

pop-writer-v3-seed 塑造初始种子（§4）。初始化产出：
- 种子文档.md（v1.0.0，七要素）
- 种子变更日志第一条
- 活记忆 baseline #0

### 5.3 生长：agent 自主判断

**核心原则：权限全给 agent。** 不设严格的生长门槛——agent 在涌现写作中发现新压力源、新线索、新规则、新目标时，自主决定是否追加到种子。

生长场景（示例，非穷举）：

| 触发场景 | 生长要素 | 示例 |
|:---|:---|:---|
| 发现新压力源 | 压力矩阵 | ch5 发现强化剂依赖风险→追加 |
| 发现新线索 | 活跃线索 | ch8 发现巫毒教与裂隙关联→追加 |
| 确立新世界规则 | 世界规则 | ch10 发现系统等级与精神力关系→追加 |
| 成长路径细化 | 成长路径 | ch12 选择巫医职业→阶段4细化 |
| 发现新目标 | 目的地 | ch15 发现裂隙本质线索→追加 |

**agent 自主判断原则：**
- 如果新发现的要素能生成未来剧情 → 追加到种子
- 如果只是当前章节的一次性事件 → 不追加，只记入活记忆
- 如果不确定 → 追加（宁多勿少，弧线校准时可修剪）

生长执行流程：
1. 涌现写作中确立新要素 → agent 判断是否触发种子生长
2. 触发 → 更新种子文档对应要素
3. 版本号 minor+1（如 1.0.0→1.1.0）
4. 变更日志追加记录
5. 同步更新活记忆对应组件

### 5.4 修剪：弧线校准时 agent 自主判断

**核心原则：权限全给 agent。** 弧线校准时 agent 自主判断哪些要素不再活跃，标记关闭或回收。

修剪场景（示例，非穷举）：

| 修剪场景 | 修剪要素 | 判定依据 |
|:---|:---|:---|
| 压力源已解决 | 压力矩阵 | 倒计时归零且不再激活 |
| 活跃线索已回收 | 活跃线索 | 线索悬念已兑现 |
| 目的地已到达 | 目的地 | 故事已涌现到该路标 |
| 线索长期休眠 | 活跃线索 | 连续 10 章未触及 |

**修剪原则：不删除历史，只标记状态。** 已关闭/已完结的要素保留在种子文档的"已关闭区"，供版本追溯。

### 5.5 版本管理

种子文档头部版本元数据 + 底部变更日志：

```markdown

> lastUpdatedAt: 2026-06-25T00:00

## [七要素当前活跃内容]
...

## 已关闭区（修剪记录）
| 要素 | 原内容 | 关闭章号 | 关闭原因 |
...

---
## 变更日志
| 版本 | 章号 | 类型 | 变更内容 | 触发原因 |
|:-----|:-----|:-----|:---------|:---------|
| 1.0.0 | 0 | init | 初始化七要素 | 种子设计 |
| 1.1.0 | 5 | grow | 压力矩阵追加"强化剂" | ch5发现新压力源 |
| 1.2.0 | 8 | grow | 活跃线索追加T006 | ch8发现巫毒教与裂隙关联 |
| 1.2.1 | 15 | prune | 压力源"医疗缺失"标记已关闭 | ch15弧线校准，HP已稳定 |
```

版本号规则（语义化）：
- **major**（X.0.0）：种子要素结构变化（新增/删除要素类型）
- **minor**（1.X.0）：生长（追加新压力源/新线索/新规则/新目的地）
- **patch**（1.0.X）：修剪/调整（关闭压力源、回收线索、调整细节）

### 5.6 活种子与活记忆的同步关系

| 目的地 | 目的地进度 | 种子增减目的地→活记忆目的地进度同步 |

## 6. 信息需求判断机制

### 6.1 从预设注入到 agent 自主判断

v1.0 把信息注入限定在种子设计阶段，注入源和路径都是预设的。v3 改为：涌现写作环中 agent 自主判断"这章需要什么信息"，主动去获取。

种子设计阶段仍保留种子搜索法（用于压力矩阵的制度性压迫维度调研），但写作过程中的信息获取改为 agent 自主判断。

### 6.2 判断逻辑

每章写作前（pop-writer-v3-emerge step-1），agent 做 4 类检查：

| 需求类型 | 判断问题 | 信息源 | 示例 |
|:---|:---|:---|:---|
| 制度/流程细节 | 这章涉及什么我不熟悉的现实制度/流程？ | WebSearch | 美国催债法流程、庇护所申请、帮派等级制 |
| 同类场景写法 | 这章的场景类型需要参考技法吗？ | library 套路库 | 帮派对峙场景、系统面板展示、绝境反杀 |
| 文风渲染参考 | 这章的文风渲染需要参考吗？ | library 文风库 | 冷硬对话节奏、暴力场景感官描写 |
| 已埋伏笔/前文细节 | 这章需要呼应哪些已埋伏笔？ | 项目本身（活记忆种子追踪+前章正文） | MS-13 在找谁、老约翰之死、前身背景 |

**agent 自主判断"需不需要查"：**
- 4 类检查均无明确需求 → 跳过查询，直接进入涌现写作
- 有明确需求 → 列出信息需求清单，逐项获取

**核心原则：不是每章都必须查。** 常规章节（日常生存、已建立机制的运作）不需要查。agent 自主判断。

### 6.3 获取流程

1. **WebSearch**：搜索制度/流程细节，提取关键信息
2. **library**：查询套路库/文风库，提取场景技法/渲染参考
3. **项目本身**：查阅活记忆种子追踪+前章正文，提取待回收伏笔

获取的信息不存为独立文档，直接融入写作上下文。在信息需求判断记录中登记。

### 6.4 信息需求判断记录

```yaml
info_needs:
  - chapter: 3
    needs:
      - type: 制度/流程
        question: 美国催债公司的追债流程和手段
        source: WebSearch
        result: 催债公司可上门、可诉讼、可申请强制执行
      - type: 已埋伏笔
        question: 前身"布鲁斯·韦恩"的背景细节
        source: 活记忆种子追踪 + ch1正文
        result: 前身童星→娈童丑闻→三婚三离→流浪
    merged: true
  - chapter: 4
    needs: []  # 常规章节，无需查询
    merged: false
```

### 6.5 library 查询矩阵

种子设计阶段（预设查询）：

| 种子设计子步骤 | 查询模块 | 用途 |
|:-------------|:---------|:-----|
| 种子搜索法 | 设定库/{对标书}/ + WebSearch | 压力矩阵设计 |
| 书型+元爽点 | 立项库/00-索引 + 元爽点-变体映射表 | 确定本书书型 |
| 冲突轴+活跃线索 | 套路库/00-总索引 + 剧情库/{标签}/ | 冲突轴设计 |
| 金手指 | 套路库/金手指体系 | 金手指设计 |
| 文风DNA | 文风库/{对标书}.md | 种子第七要素 |

涌现写作阶段（agent 按需查询）：

| 查询模块 | 查什么 | 用途 |
|:---------|:-------|:-----|
| 套路库/{具体套路名}.md | 场景技法+节奏控制 | 写作时按需参考 |
| 文风库/{书名}.md | 风格渲染场景卡 | 文风锚定（全量驻留） |

## 7. 阶段二：涌现写作环

### 7.1 六步循环

```
本章规划 → 信息需求判断 → 写（涌现）→ 反思（五问）→ 活记忆更新+种子生长 → 方向提示
    ↑                                                                    │
    └────────────────────────────────────────────────────────────────────┘
```

### 7.2 本章规划（先想再查再写）

写之前先想明白。读种子+上一章正文+活记忆+方向提示，判断这章要怎么写——5个决策点（场景/线索/爽点/危机/钩子），然后对照网文爽感机制10条法则检查。规划做完，再进信息需求判断。

网文爽感机制是可配置的写作法则文件（`references/网文爽感机制.md`），包含10条法则：压力机制/即时奖励循环/不间断危机链/高事件密度/主角主动决策/养成感/信息披露节奏/80%展示20%内省/多线交叉/情绪闭环。当法则冲突时按优先级处理（压力机制>即时奖励>危机链>...）。未来探索其他领域时，替换为对应领域的写作法则。

### 7.3 信息需求判断

详见 §6。agent 自主判断本章需要什么外部信息，逐项获取后融入写作上下文。

### 7.4 写（涌现）——chapter+prose 合并

上下文构成：种子（当前版本）+ 活记忆（最后 baseline + event）+ 上一章正文末尾 + 文风DNA + 方向提示 + 增量信息（如有）。

合并理由：消除 chapter→prose 的信息损失。设计决策和渲染执行在人类作者脑中是同时发生的，分离导致 prose 拿到的是"设计摘要"而非"创作冲动"。

agent 自由创作，做实时决策，没有章纲约束。文风DNA 硬阻塞（缺失=终止）。80% 展示 + 20% 内省。

### 7.5 反思（后验评估）

五个读者体验问题：

| 问题 | 判断标准 | 不达标处理 |
|:-----|:---------|:-----------|
| 读者获得了什么？ | 至少 1 次实质奖励 | 标记下章必须给奖励 |
| 压力在吗？ | 倒计时在走 或 新威胁在逼近 | 标记下章必须加压 |
| 节奏对吗？ | 对比目标小说同阶段的事件密度趋势 | 标记下章需加速/减速 |
| 线索追踪正常吗？ | 超期伏笔预警；新种子已登记 | 标记需回收的伏笔 |
| 读者会翻页吗？ | 章末有即时钩子 | 标记需重写章末 |

反思防"应付式填表"机制：每个问题必须有具体正文引用作为证据。反思不通过 = 回退重写本章。

### 7.6 活记忆更新+种子生长

反思通过后，更新活记忆七组件。同时 agent 判断是否触发种子生长（§5.3）——如果本章发现了新压力源/新线索/新规则，自主决定是否追加到种子。

### 7.7 方向提示生成

1-2 句话的方向感，不是章纲。格式：

```
方向提示 ch004:
- 必须给奖励（ch3只有信息，无实质获得）
- 压力需加码（血量倒计时已3章未推进）
- 回收短线：MS-13追杀线已3章未出现，需触面
- 钩子必须是即时危机
```

方向提示不指定具体事件——只说"需要给奖励、需要加压、需要触面MS-13、需要即时钩子"。具体怎么实现，由写作 agent 涌现。

### 7.8 每章执行 SOP

1. 本章规划（种子+上章+活记忆+方向提示+网文爽感机制→5决策点+法则对照）
2. 信息需求判断（基于本章规划，agent 自主判断需要什么信息，逐项获取）
2. 涌现写作（种子+活记忆+文风DNA+增量信息→正文，文风DNA硬阻塞，80/20）
3. 反思（五问后验评估，不通过=回退重写）
4. 活记忆更新+种子生长（七组件刷新 + agent自主判断是否生长种子）
5. 方向提示生成（1-2 句下一章方向）
6. 落盘 + 项目总控更新

## 8. 活记忆系统

### 8.1 与 state-log.yaml 的演进关系

v2 用 state-log.yaml（双组件：baseline+event）。v3 用活记忆.yaml（七组件），概念同源结构不同。演进路径：state-log.yaml → 活记忆（七组件）。

### 8.2 七大组件

| 组件 | 职责 | 防崩类型 | 数据形态 |
|:-----|:-----|:---------|:---------|
| 种子追踪 | 已埋伏笔（章号+内容+预期回收点） | 防线索遗失 | 列表，每条含状态（活跃/已回收/超期预警） |
| 压力状态 | 所有倒计时的当前值 | 防节奏崩坏 | 键值对，含触发条件与剩余值 |
| 角色状态 | 主角的数值/心理/资源/关系变化 | 防人设漂移 | 章末追加的状态增量 |
| 世界规则 | 已确立的规则清单 | 防世界观矛盾 | 只增不改的规则表 |
| 节奏日志 | 每章事件密度/爽点数/钩子类型 | 防节奏崩坏 | 章号+三项指标的日志 |
| 战力曲线 | 主角能力值的时间序列 | 防战力膨胀 | 章号+能力值的折线数据 |
| 目的地进度 | 5-10 个目的地的到达状态 | 防失去方向 | 目的地+状态（未到达/接近/已到达） |

### 8.3 防膨胀压缩机制

压缩触发条件：每 20 章 event 累积后。压缩策略：读最后一个 baseline + 之后全部 event → 合并成新 baseline → 删旧 baseline 及其后 event → 追加新 baseline。压缩后文件回到 5-15KB。

### 8.4 谁读谁写

| skill | 步骤 | 操作 |
|-------|------|------|
| pop-writer-v3-seed | 产出后 | 初始化活记忆 baseline #0 |
| pop-writer-v3-emerge | Step 1 | 读活记忆（最后 baseline + event）→ 取当前状态 |
| pop-writer-v3-emerge | Step 4 | 追加 event 到活记忆 + 种子生长同步 —— 唯一写入者 |
| pop-writer-v3-arc | 检查时 | 读活记忆全量 → 宏观检查 |
| pop-writer-v3-arc | 压缩时 | 合并 baseline + event → 新 baseline |
| pop-writer-v3-arc | 回退时 | 删 entries 中 chapter > N 的条目 |

## 9. 阶段三：弧线校准

### 9.1 校准触发条件

- 定期触发：每 10-20 章自动触发（建议值 15 章）
- 事件触发：活记忆任意组件触发预警
- 用户触发：用户主动要求宏观检查

### 9.2 宏观检查清单

| 检查项 | 检查内容 | 预警条件 |
|:-------|:---------|:---------|
| 种子追踪 | 活了哪些？死了哪些？该回收没回收的？ | >20 章未回收 |
| 战力曲线 | 膨胀了吗？跳级了吗？ | 连续跳级 |
| 角色弧线 | 连贯吗？驱动力三层演化进度 | 漂移或停滞 |
| 宏观节奏 | 波形对吗？有没有连续 5 章平淡？ | >=3 章低密度 |
| 目的地进度 | 走到哪了？偏离了吗？ | 连续 20 章无进展 |
| 冲突轴 | 活跃线索还活着吗？需要新线索吗？ | >=10 章未触及 |

### 9.3 种子修剪

弧线校准时，agent 自主判断哪些种子要素不再活跃（§5.4）。修剪不删除历史，只标记状态（已关闭/已完结/已到达/休眠）。

### 9.4 回退机制

| 问题严重度 | 判定标准 | 处理方式 |
|:----------|:---------|:---------|
| 轻微 | 单章质量问题，不影响后续 | 打补丁（重写该章） |
| 中等 | 2-3 章累积偏差，活记忆可吸收 | 微调方向提示 + 后续章节自然纠正 |
| 严重 | 弧线级偏移 | 回退重写（删 entries + 重写 10-20 章） |

回退执行流程：输出校准报告 → 用户确认回退范围 → 删活记忆 entries 中 chapter > N 的条目 → 删正文 ch > N 的文件 → 从第 N+1 章重新进入涌现写作环。

## 10. expert-writer 中控层强化

### 10.2 Think 层强化

| v2 职责 | v3 强化职责 |

5. **智能调度判断（新增）**：根据状态决定下一步动作（正常→写作环 / 校准触发点→弧线校准 / 反思不通过→回退重写 / 严重偏差→校准）

### 10.3 Execute 层强化

| v2 职责 | v3 强化职责 |

v3 Execute 层 5 项职责：
1. **信息获取调度（新增）**：根据 Think 层信息需求清单，调度 WebSearch/library 查询
2. **涌现写作调度**：加载种子+活记忆+信息+文风DNA，涌现写作（chapter+prose 合并）
3. **反思调度**：五问后验评估
4. **活记忆更新+种子生长调度（新增）**：反思通过后更新活记忆七组件 + agent 自主判断是否触发种子生长
5. **闸门（变形）**：种子确认 + 弧线校准确认

### 10.4 Reflect 层强化

| v2 职责 | v3 强化职责 |
|:---|:---|
| 通用审视+回写+引导 | 方向提示+弧线校准检查+种子生长判断+项目总控回写+引导 |

v3 Reflect 层 6 项职责：
1. **方向提示生成（保留+强化）**
2. **弧线校准检查（新增）**：检查是否触发弧线校准条件
3. **种子生长判断（新增）**：反思中发现新要素时，判断是否触发种子生长
4. **项目总控回写（保留）**：更新 3 阶段+章号+弧线计数+种子版本
5. **活记忆压缩检查（保留）**
6. **引导（保留+强化）**

## 11. expert-writer 双轨支持

### 11.2 改造点清单

## 17. 开放问题与风险

1. **活种子生长的边界**：权限全给 agent 后，agent 可能过度生长导致种子膨胀——靠弧线校准修剪兜底，种子规模超 1500 字时 agent 自主优先修剪
2. **信息需求判断变成"每章都查"**：agent 过度保守——在 SOP 中明确"常规章节不查"原则+给出"不需查"的示例
3. **活记忆膨胀控制**：七组件比 state-log 更大——压缩后是否可控（已定义压缩机制，待验证）
4. **弧线校准回退代价**：回退重写 10 章的代价——已定义判定标准，待验证
5. **多角色视角的涌现**：当前设计以单主角为假设——待后续迭代
6. **v2/v3 双轨切换混乱**：模式一旦确定不可切换——项目总控"管线模式"字段强制
