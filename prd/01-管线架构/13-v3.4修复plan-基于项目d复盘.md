## 修复1：expert-writer SKILL.md 内联7步核心门禁

### 问题
框架"强制载入"只载入SKILL.md作为system prompt，但7步循环的步骤细节和红线都在steps/和references/里。agent读完ch001的step文件后认为"流程已知"，ch002/ch003不再重新加载，导致plan check丢失、质检退化。

### 修复方案
在SKILL.md中内联**7步循环核心门禁表**（而非完整step内容），让框架加载SKILL.md时就包含关键约束。

**SKILL.md新增内容（约40行，紧凑）**：

```markdown
## 7步循环核心门禁（每章必须全部通过）

| 步骤 | 动作 | 硬门禁 | 验证证据 |
|:--|:--|:--|:--|
| Step0 本章规划 | 读种子六要素+活记忆→做5决策点→产出chapter_plan | plan文件状态必须为"已确认"才能进Step1 | 章节规划/chXXX-plan.md 顶部状态字段 |
| Step1 信息获取 | 强制读素材库/索引.md→处理info_gaps→4类补充检查 | 索引未读=退回补读 | 素材库/索引.md被read记录 |
| Step2 调度创作 | 组装精简context→调度create子skill→产出正文初稿 | create子skill必须读取种子六要素全量 | 创作决策记录中的种子读取清单 |
| Step3 调度修订 | 组装精简context→调度revise子skill→产出修订稿 | 文风DNA必须完整加载（禁止limit截断） | 修订记录dna_loaded:true |
| Step4 调度质检 | 组装精简context→调度qa子skill→产出质检报告 | 质检报告必须落盘+引用正文证据 | 质检报告/chXXX-qa.md文件存在 |
| Step5 记忆+生长 | 读质检报告→追加活记忆event→种子生长回写 | 种子生长必须更新_log+_index+要素文件 | 种子/_log.md新增版本记录 |
| Step6 落盘 | 修订稿落盘+质检报告落盘+更新项目总控 | 正文+质检报告+项目总控三文件更新 | 三文件file-change记录 |

### 每章执行红线
- ❌1 每章Step0开始前，必须重新读取本门禁表（不依赖记忆）
- ❌2 plan未经用户确认，禁止进入Step1
- ❌3 质检报告未落盘，禁止进入Step5
- ❌4 文风DNA加载必须完整（Get-Content -Raw），禁止read limit截断
- ❌5 子agent失败降级时，必须重读完整种子context+独立质检，禁止凭记忆直写
```

## 修复2：plan从"场景list"重构为"任务list"

### 问题
当前plan主体是scenes（场景list），scene.description写得极细（连台词都写死），创作决策前置到plan阶段，create环节变成"渲染执行"而非"创造"。

### 修复方案
重构chapter_plan结构，从"场景list"改为"任务list"——定义"本章要达成什么"，不定义"怎么达成"。

**新chapter_plan结构（Markdown格式，非YAML）**：

## 修复3：revise从"逐行修改"重构为"完全重写" + 文件加载规范

### 问题
1. 当前revise是"5项修订任务"逐行修改模式（文风对齐/人设丰富/爽点验证/bug修复/AI观感词清理），第34行明确"不改变故事结构，只做渲染层面的打磨"
2. 用户预期是"用正文+文风DNA完全重写一次"
3. 文风DNA加载用read工具且agent自加limit:60，只读了40.7KB文件的前60行

### 修复方案

**3a. revise核心逻辑重构**：

从"5项修订任务逐行修改"改为"基于正文+文风DNA直接完全重写"：

```markdown
## 修订核心逻辑

revise不是逐行修改，而是**基于正文初稿 + 文风DNA直接完全重写**。

### 输入
- 正文初稿（故事事实：情节/场景/角色行为/对话/人名/动作——这些全部保留不改）
- 文风DNA（渲染引擎：句式/叙事距离/感官序列/段落呼吸/对话引导——这些完全重写）

### 执行
完整加载文风DNA（Get-Content -Encoding UTF8 -Raw，禁止limit截断），然后基于初稿直接重写正文。

### 硬约束：不可改写的事实信息
重写时**绝不改写已发生的事实信息**，包括但不限于：
- 人名（角色叫什么就是什么）
- 动作（谁打了谁、用了什么武器、怎么打的）
- 事件顺序（先发生了什么后发生了什么）
- 对话内容（说了什么信息就是什么信息）
- 场景设定（在哪里发生的就在哪里）
- 数值（系统面板的属性/经验/HP等数值不变）

重写只改**渲染层**：怎么描述这些事实、用什么句式、什么感官顺序、什么节奏——这些按文风DNA重做。

### 门禁
- 重写稿与初稿的**事实信息一致性**（人名/动作/事件/对话内容/场景/数值不变）
- 文风DNA逐条对照（7项核心特征每项附正文证据）
- 字数控制（2500-3500字）
- AI观感词清理（≤3种命中）
```

**3b. 文件加载规范**：

在revise step和expert-writer SKILL.md中增加文件加载硬规范：

```markdown
### 文件加载规范（红线）
- 文风DNA文件 `写作资产/文风库/{书名}.md`：必须用 `Get-Content -Encoding UTF8 -Raw` 加载完整文件，**禁止用read工具的limit参数截断**
- 种子文件 `种子/*.md`：必须用 `Get-Content -Encoding UTF8 -Raw` 加载完整文件
- 大文件（>10KB）一律用 Get-Content -Raw，不用read limit
- read工具仅用于查看文件片段（如确认文件是否存在），不用于加载创作/修订所需的完整内容
```

### 改动文件
- `skills/pop-writer-v3-revise/steps/step-1-revise.md`：核心逻辑从"5项修订任务"重构为"故事骨架提取+文风DNA完全重写"
- `skills/pop-writer-v3-revise/references/修订指南.md`：同步更新
- `skills/expert-writer/SKILL.md`：新增文件加载规范红线

### 验证标准
- revise产出的正文与初稿故事骨架一致，但渲染层完全按DNA重写
- 文风DNA完整加载（不再limit:60）
- 修订记录含"故事骨架一致性"检查 + "文风DNA逐条对照"检查（每项附正文证据）

## 修复4：create→revise自动连贯，只check两次

### 问题
当前流程：plan→(check)→create→(check?)→revise→(check?)→qa→(check?)。中间产物验收冗余，且ch002/ch003的plan check被跳过。

### 修复方案
重构为**双check点**流程：

```
Step0 plan → 【CHECK 1：用户确认plan】→ Step1信息获取 → Step2 create → Step3 revise → Step4 qa → 【CHECK 2：用户验收最终正文】→ Step5记忆生长 → Step6落盘
```

**关键变化**：
- plan确认后，Step1→Step2→Step3**自动连贯执行**，不需要人验收中间产物（create初稿、revise重写稿）
- revise完成后，最终正文交付用户check
- **只有两个人工check点**：plan确认 + 最终正文验收
- **qa从流程中剔除**（v3.4决策）：qa skill文件暂不改动，但expert-writer的7步循环不再调用qa。质检职责由revise层的文风DNA逐条对照+事实信息一致性检查承担

**dispatch流程调整**：

```markdown
### Step 2-3 自动连贯执行（dispatch连续调度，无qa环节）

plan确认后，主会话连续调度 create→revise，中间不暂停：

1. Step2 调度create → 收到正文初稿 → 不交付用户，直接传给Step3
2. Step3 调度revise → 收到重写稿（含文风DNA逐条对照+事实一致性检查）→ 交付最终正文给用户check

无qa环节。revise层承担文风验证+事实一致性+字数控制+AI观感词清理全部职责。

子agent失败降级时：
- 主会话接管create→revise顺序执行
- 但必须：重读完整种子context（Get-Content -Raw）+ 文风DNA完整加载 + 每步产出落盘
```

### 改动文件
- `skills/expert-writer/steps/step-2-2-dispatch-create.md`：去掉"交付用户"环节，改为直接传给Step3
- `skills/expert-writer/steps/step-2-3-dispatch-revise.md`：revise完成后交付用户check；revise新增文风DNA逐条对照+事实一致性检查职责
- `skills/expert-writer/steps/step-2-4-dispatch-qa.md`：**保留文件但从流程中剔除调用**（qa skill暂不动）
- `skills/expert-writer/steps/step-2-execute.md`：7步循环改为6步（Step0-3+Step5-6，跳过Step4），标注两个check点

## 修复5：子agent降级策略（P0方案B兜底）

### 问题
ch001子agent spawn失败后，agent选择"主会话直写"降级，但降级时未重读种子context、未独立质检，且固化为后续所有章节的默认策略。

### 修复方案
在dispatch step中写明spawn失败的降级策略：

```markdown
### 子agent失败降级策略（方案B兜底）

优先使用子agent（sessions_spawn）。子agent失败时，降级为主会话执行，但必须：

1. **重读完整种子context**：用 Get-Content -Raw 重新加载种子六要素全量文件，不依赖会话历史记忆
2. **重读文风DNA**（revise环节）：用 Get-Content -Raw 加载完整文风DNA文件
3. **独立质检**：降级执行qa时，必须引用正文证据（每问附正文片段），不能纯声明式通过
4. **降级标记**：在产出物中标注"降级主会话执行"，便于事后追溯
5. **重试限制**：子agent最多重试1次，第2次失败才降级

降级不等于跳过门禁——所有7步门禁在降级模式下同样必须通过。
```

## 修复6：revise承担文风DNA验证（qa已剔除，职责下沉到revise）

### 问题
原设计qa子skill不传入文风DNA，只验8020比例，不查句式/叙事距离/感官密度/对话留白。文风对齐完全依赖revise层自检自报，无第三方复核。v3.4决策：qa从流程中剔除，文风验证职责全部下沉到revise层。

### 修复方案
revise在重写完成后，必须执行文风DNA逐条对照验证（替代原qa的文风验证职责）：

```markdown
### revise新增：文风DNA终验（重写完成后执行）

重写稿产出后，revise层必须执行文风DNA逐条对照（每项附正文证据）：

| 检查项 | 标准 | 不通过 |
|:--|:--|:--|
| 场景卡公式 | 正文是否套用了DNA场景卡的公式 | 重新重写该场景 |
| 数据嵌入格式 | 系统面板是否按DNA格式嵌入 | 重新重写 |
| 感官序列 | 嗅觉优先序列是否执行 | 重新重写该场景 |
| 段落呼吸 | 独句段+短句链是否执行 | 重新重写 |
| 叙事者距离 | 是否全程贴主角视角 | 重新重写 |
| 对话引导词 | 是否符合DNA引导词规范 | 修正后输出 |

文风DNA终验结果写入修订记录，随正文一起交付用户check。
```
