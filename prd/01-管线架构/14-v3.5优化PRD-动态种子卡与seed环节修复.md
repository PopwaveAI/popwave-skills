## 一、设计理念

### 1.1 MMORPG 隐喻映射

小说世界是一个 MMORPG 游戏。种子文档是玩家任务日志——持续更新、驱动前进。任务来源由三股力量混合构成：主角阵营的内在驱动力（主角引擎）、压力源矩阵不断施加的外在压力、目的地终点的方向锚定。设定库/素材库/人物库构成这个世界的舞台和规则。

不走大纲/章纲。传统大纲是"导演脚本"（预设每场戏怎么演），信息层层衰减。任务表是"游戏设计"（设定目标和约束，让 agent 自己玩）——种子文档→正文是单次跃迁，信息损失最小。大纲和卷纲的功能被内化为任务表的结构：主线任务链取代剧情大纲，任务阶段取代卷纲划分，当前可执行层取代章纲。

### 1.2 与 v3.4 的关系

v3.4 解决了"流程不执行"的合规性问题（6步门禁内联+文件加载规范+降级策略）。v3.5 解决的是"执行了但产出质量不够高"的结构性问题——种子与创作脱节、种子冻结无生长、信息获取无优先级。

## 二、五个核心改动

### 改动1：种子文档统一化

#### 问题

v1.4.0 种子是"6个分要素文件 + _index.yaml + _log.md"的文件夹结构。问题：
- 压力矩阵和冲突轴本质是世界设定，混在种子里职责不清
- 6个分文件增加 agent 加载成本和交叉引用复杂度
- _log.md 的三个职责（版本管理/前3章验证/已关闭区）都有更好的归属

#### 方案

**统一为 `种子文档.md`（单文件活文档）+ 设定分流到设定库。**

种子文档结构：

```markdown
# 种子文档 · {书名}

> version: 1.0.0 | last_updated_ch: 0 | current_ch: 0 | current_arc: "第一卷"

## 任务表（核心段，持续维护，≤800字）

### 主线任务（带当前阶段映射）
| ID | 终点 | 阶段链 | 当前阶段 | 当前可执行 | 状态 |
|:---|:-----|:-------|:---------|:-----------|:-----|
| D001 | 生存 | D001 | D001 | 逃出CIA黑场 | 当前 |
| D002 | 净化河谷市 | D002→D003→D004→D005 | — | — | 锁定 |
| D005 | 推翻混沌美利坚 | （同上链） | — | — | 锁定 |

### 压力任务（带倒计时）
| ID | 压力源 | 状态 | 当前倒计时 | 本章必须推进? |
|:---|:-------|:-----|:-----------|:-------------|
| S2 | 伤口败血症 | 激活 | 24h | 是 |
| S4 | 暗网悬赏 | 持续 | — | 背景 |

### 线索任务（带回收窗口）
| ID | 线索 | planted | recall窗口 | 状态 |
|:---|:-----|:-------|:-----------|:-----|
| T001 | CIA恐虐渗透 | ch001 | ch3-6 | 进行中 |

### 本章聚焦（≤300字，每章Step0更新）
- 推进: {本章必须推进的压力/线索}
- 新增: {本章可能新增的任务}
- 钩子方向: {即时危机型钩子}
- 本章三问: ①读者必须知道什么 ②哪个压力必须推进 ③章末留什么危机

## 要素切片（支撑段，≤850字）
### 主角引擎
（驱动力三层+缺陷+行为准则，≤200字）
### 金手指
（能力+限制+行动驱动，≤150字）
### 成长路径
（阶段里程碑，≤100字）
### 当前阶段上下文
（从目的地投影的当前阶段描述+完成条件，≤150字）
### 核心卖点
（一句话+核心爽点+情绪承诺，≤100字）
### 压力聚焦+活跃线索
（从设定库压力矩阵/冲突轴切片本章相关项，≤150字）
```

**关键设计：**

| 维度 | 设计 | 理由 |
|:-----|:-----|:-----|
| 文档形态 | 单文件活文档，每次加载每次维护 | 任务表和要素间有引用关系，统一文档减少加载成本 |
| 长度控制 | 核心段≤800字 + 支撑段≤850字 = 总计≤1650字 | 每次加载的文档体积可控；核心段是硬约束 |
| 任务状态 | 锁定/当前/已完成，持续维护 | 任务不丢失，ch001的线索在ch005仍是"进行中" |
| 主线三层映射 | 终点/当前阶段/当前可执行 | agent只看到当前可执行层，防止跳到大结局 |
| 设定分流 | 压力矩阵全量+冲突轴全量→设定库 | 它们是世界设定不是种子；种子只保留本章切片 |

**设定库结构（ground truth，按需读取）：**

```
写作资产/设定库/
├── 世界宪法.md（世界法则+核心矛盾）
├── 力量体系.md
├── 社会结构.md
├── 压力矩阵全量.md（所有压力源+交叉困境）
├── 冲突轴全量.md（四层生态+线索池全量）
├── 人物库/
│   └── {角色名}.md（NPC档案：身份/关系/动机/可用任务）
└── 已废弃/（arc修剪的废弃设定归档）
```

#### _log.md 取消评估

站在整个项目层面，"记录/日志"类文件体系：

| 文件 | 职责 | 更新频率 |
|:-----|:-----|:---------|
| `活记忆/活记忆.yaml` | 每章event追加（含seed_growths字段） | 每章 |
| `种子文档.md` | 当前有效蓝图（头部带版本号） | 每次生长 |
| `设定库/` | ground truth（含已废弃分区） | 按需 |
| `修订记录/chXXX-revise.md` | 每章修订记录归档 | 每章 |

_log.md 承担的三个职责替代方案：

| _log.md职责 | 替代方案 | 理由 |
|:-----------|:---------|:-----|
| 版本管理（版本号+变更记录表） | 种子文档头部 version+last_updated_ch + 活记忆 seed_growths字段 | 每次生长已在活记忆event记录变更内容+章号+触发原因 |
| 前3章走向验证 | 一次性验证，不持久化 | 验证完即失效；长线偏离由arc校准发现 |
| 已关闭区 | 设定库"已废弃/"分区 | 关闭的要素本质是已废弃设定，归设定库管理 |

**决策：取消 _log.md。** 三个职责都有更好的归属，取消后每个文件职责单一无重叠。

### 改动2：新建 pop-writer-v3-plot skill（任务链设计）

#### 问题

当前管线没有独立的任务链设计 skill。v2 的 pop-writer-plot 有卷战略+剧情线+金手指任务驱动节点的基础，但含章锚点等雪花式残留。种子文档的任务表需要专业的设计阶段，不应混在 seed skill 的创意共创中。

#### 方案

新建 `pop-writer-v3-plot`，从 v2 plot 继承卷战略+剧情线设计理念，大改重构为涌现式任务链设计。

**v2 plot 继承什么 / 砍什么 / 改什么：**

| v2 plot 组件 | 决策 | 理由 |
|:-------------|:-----|:-----|
| 卷战略（step-1） | 继承但简化 | 保留卷定位+起点终点+主冲突；砍掉篇幅预算/线数预算（涌现式不预设章数） |
| 危机交叉设计（step-2） | 砍掉 | 压力矩阵已覆盖危机来源，不需要独立交叉设计 |
| 剧情线（step-3） | 大改→任务链 | L3剧情线骨架节点→主线任务链三层映射 |
| 分幕+章锚点（step-4） | 砍掉 | 涌现式不需要章锚点；章级规划由expert-writer Step0负责 |
| 密度审计（step-5） | 砍掉 | 下沉到revise层 |

**pop-writer-v3-plot 2步流程：**

**Step 1：卷方向锚设计**
- 卷定位（本卷兑现全书哪个承诺）
- 起点终点（角色/世界/力量/关系四维度状态变化）
- 主冲突（谁阻碍谁，争夺什么，失败代价）
- 产出：`卷纲/卷N-方向锚.md`（方向参考非强制，≤300字）

**Step 2：任务链设计**
- 从目的地（seed产出）展开为任务链
- 三层映射设计：终点→阶段链→当前可执行
- 每个阶段的完成条件+解锁条件
- 金手指任务驱动节点设计（继承v2的"金手指任务"概念）
- 产出：写入种子文档任务表的主线任务段

**任务链三层映射设计：**

```
D005 推翻混沌美利坚（终点）
  └─ D004 建立抵抗组织（阶段4，锁定）
       └─ D003 对抗系统层（阶段3，锁定）
            └─ D002 净化河谷市（阶段2，锁定）
                 └─ D001 生存（阶段1，当前）
                      └─ 当前可执行: 逃出CIA黑场→搞抗生素→搞假身份→...
```

每个阶段定义：
- 完成条件：满足什么条件算这个阶段完成
- 解锁条件：前一阶段完成后，下一阶段自动解锁
- 当前可执行：当前阶段下主角能做的事（初始3-5个，随剧情更新）

**阶段切换规则：**
- expert-writer 每章 Step5 落盘时检查当前阶段完成条件
- 满足→标记"待解锁"，提示用户进入 arc
- arc 校准时确认完成→解锁下一阶段→更新种子文档任务表的当前阶段+当前可执行

### 改动3：Seed Skill 维护SOP补全

#### 问题

seed skill v1.4.0 只定义初始化，不定义维护。实战结果：项目d 种子冻结在 v1.1.0，28章0生长。无进度维度、无长度控制、已关闭区写入SOP缺失。

#### 方案

新增 `step-5-maintenance.md`，定义种子维护SOP（供 expert-writer Step4 和 arc 调用参考）：

**3a. 种子生长SOP标准化：**
- 生长判断在 revise 子skill（重写过程中发现新要素→修订记录记种子生长建议）
- 生长执行在 expert-writer Step4（机械执行修订记录的生长建议）
- 生长执行4步：更新种子文档对应段→更新 last_updated_ch→版本号 minor+1→活记忆 seed_growths 字段记录
- 新增：生长时的长度检查——种子文档超1650字时触发压缩

**3b. 种子进度维度：**
种子文档头部新增进度追踪：
```markdown
> version: 1.2.0 | last_updated_ch: 5 | current_ch: 5 | current_arc: "第一卷"
```
- current_ch：当前写到第几章（expert-writer Step5落盘时更新）
- current_arc：当前叙事弧（arc切换时更新）
- last_updated_ch：最后一次种子生长/检查时的章号

**3c. 种子长度控制红线：**
- 种子文档总计 ≤1650字（核心段800+支撑段850）
- 活跃线索 ≤8条（超过时旧线索标记"休眠"或"回收"）
- 压力源聚焦 ≤3个/章（全量压力源在设定库，不限制）
- 当前可执行任务 ≤5个（超过时旧任务标记"已完成"或"过期"）

**3d. 种子健康检查（arc触发，每10-15章）：**
- 新鲜度检查：last_updated_ch > 15章的要素标记"需关注"
- 膨胀检查：超1650字触发修剪
- 一致性检查：种子文档与活记忆/设定库是否一致
- 阶段切换检查：当前阶段完成条件是否满足

**3e. 已废弃设定管理（替代原_log.md已关闭区）：**
- 写入时机：arc校准确认某要素已完结/废弃
- 写入位置：`设定库/已废弃/`分区
- 写入内容：要素名+原内容摘要+关闭章号+关闭原因
- 关闭后不可恢复，只能新增替代要素

### 改动5：Create 输入适配 + 本章聚焦更新权限

#### 问题

create 当前输入是 task-list（v3.4格式），v3.5 改为种子文档。同时需要明确 create 对种子文档的写入权限边界。

#### 方案

**5a. create 输入改为种子文档：**
create 子skill 接收完整种子文档（任务表+要素切片），从中读取"本章聚焦"作为本章创作约束，读取"要素切片"作为故事引擎。

**5b. create 的种子文档写入权限：**
create 在涌现写作过程中，如发现需要调整本章聚焦（如发现了新的线索、压力变化、任务状态变更），**仅可更新种子文档的"本章聚焦"段**。不得修改任务表其他段（主线/压力/线索任务的状态变更由 expert-writer Step4 负责）。

权限边界：

| 种子文档段 | create权限 | 维护者 |
|:-----------|:-----------|:-------|
| 主线任务 | 只读 | expert-writer Step0（状态查询）+ arc（阶段解锁） |
| 压力任务 | 只读 | expert-writer Step0+Step4（状态更新） |
| 线索任务 | 只读 | expert-writer Step0+Step4（状态更新） |
| 本章聚焦 | **可写** | expert-writer Step0（初版）+ create（写作中微调） |
| 要素切片 | 只读 | expert-writer Step4（种子生长时更新） |

**5c. create 输出新增：**

## 五、风险与决策

### 风险1：种子文档体积 vs 信息密度

1650字要装下任务表+六要素切片+三问，可能信息密度过高导致 agent 理解困难。

**缓解**：核心段（任务表+本章聚焦）800字是硬约束，支撑段（要素切片）850字可按需压缩。如果实战中发现不够，优先压缩要素切片中的成长路径+核心卖点（这两个最稳定，不需要每章更新），保留压力聚焦+活跃线索（每章变化大）。

### 风险2：plot skill 与 seed skill 的职责边界

plot 设计任务链写入种子文档，seed 初始化种子文档。两者都写种子文档，可能职责不清。

**决策**：seed 先产出种子文档初版（含目的地但无任务链），plot 在 seed 之后运行，设计任务链并写入种子文档的主线任务段。管线顺序：seed → plot → 涌现写作环。

### 风险3：create 写本章聚焦的权限边界

create 在写作中更新本章聚焦，可能与 expert-writer Step4 的种子生长职责冲突。

**决策**：create 只能更新"本章聚焦"段（≤300字），不能修改任务表其他段。create 的调整记录在创作决策记录中，expert-writer Step4 读取后决定是否正式回写到任务表（如新增线索任务、压力状态变更）。

### 决策记录

| 决策 | 选择 | 理由 |
|:-----|:-----|:-----|
| 种子文档形态 | 单文件活文档 | 减少加载成本，任务表与要素间有引用关系 |
| _log.md | 取消 | 三职责都有更好归属；项目记录体系职责单一 |
| plot skill | 新建v3，从v2继承+大改 | v2有基础但含雪花式残留；涌现式需要独立任务链设计 |
| create 写入权限 | 仅本章聚焦段 | 裁判选手分离；create有涌现自主权但不承担任务表维护 |
| 主线任务可见性 | 三层映射，锁定不显示路径 | 防止agent跳到大结局；当前可执行层驱动涌现 |
| 大纲/卷纲 | 内化为任务表结构 | 不走雪花式；任务链取代大纲，阶段取代卷纲 |
