# v3.5 涌现式写作管线重构 PRD

> 版本：v3.5-current-state
> 更新日期：2026-07-06
> 状态：实施级
> 取代文档：10-涌现式写作专家全链路-PRD.md
> 关联 skill：pop-emergent, pop-emergent-seed, pop-emergent-research, pop-emergent-write, pop-emergent-review

---

## 1. 结论

涌现式写作不再走“复杂元 skill 每轮总控”，也不回到雪花法填表。新架构是：

```text
库负责记忆
current-state 负责注意力
soul 负责主卖点和表达魂
write 负责成文
review/ledger 负责验收和压缩
```

`pop-emergent` 不再作为每轮正文写作常驻总控，而是改为**项目初始化、修复和规范审计 skill**。每轮执行由四个执行 skill 完成：

- `pop-emergent-seed`
- `pop-emergent-research`
- `pop-emergent-write`
- `pop-emergent-review`

## 2. 背景

旧方案的失败不在于“规划不够复杂”，而在于复杂 SOP 无法稳定执行：

| 方案 | 优点 | 死穴 |
| --- | --- | --- |
| 雪花式复杂管线 | 控制力强 | 机械填表，正文缺生命力，agent 选择性执行 |
| 简化直出 | 正文可能更活 | 中长线弱，剧情和设定易漂 |
| 旧涌现式 | 试图兼顾 | 沉淀不被下一轮强制消费，等于无效 |

新方案不要求 write 记住所有规范，而是把下一章必需信息压缩成唯一入口文件：`涌现/current-state.md`。

## 3. 核心设计

### 3.1 三层文件

| 层 | 文件 | 定位 | write 是否直接读 |
| --- | --- | --- | --- |
| 入口层 | `current-state.md` | 下一章唯一入口包：事实、人物、设定、燃料、伏笔、硬推进、禁区 | 是，每次全量 |
| 入口层 | `soul.md` | 主卖点 + 叙事魂 + 正文气口 | 是，每次全量 |
| 库层 | seed/research/设定库/人物库/剧情线/content-mechanics | 长期记忆，可大、可杂 | 否，由 review 筛入 current-state |

### 3.2 current-state

`current-state.md` 是下一章 write 的唯一剧情入口，控制在 1000-2500 字。

必须包含：

- 当前章位
- 不可改事实
- 人物状态
- 设定状态
- 可用燃料队列
- 伏笔债务
- 下一章硬推进
- 禁止漂移

没有 `current-state.md` 时，`pop-emergent-write` 不得标 `formal`。

### 3.3 soul

`soul.md` 不是单纯文风文件，而是“主卖点 + 叙事魂 + 正文气口”。

必须包含：

- 主卖点/元爽点
- 读者追读承诺
- 主角主动方式
- 爽点外显方式
- 叙事人格
- 句子气口和段落呼吸
- 对白方式
- 信息释放
- 禁区

soul 不记录可变事实、具体剧情节点、人物当前状态、数值、招式名、系统规则。事实进入 current-state 或库文件。

## 4. 文档元数据协议

长期文档必须在正文前放置元数据块：

```markdown
---
doc_type: current-state | soul | seed | research | setting | character | plotline | fuel | mechanics | review-log | chapter
role: 本文档在管线里的定位
read_policy: full-required | full-if-targeted | summary-allowed | index-only
compression: forbid | allow-into-current-state | archive-after-10
primary_consumer: write | review | seed | research | human
source_of_truth: true | false
last_updated: YYYY-MM-DD
---
```

读取策略：

| read_policy | 含义 |
| --- | --- |
| `full-required` | 消费者每次必须全量读，不能用摘要替代 |
| `full-if-targeted` | 本轮直接相关才全量读 |
| `summary-allowed` | 可被 review 压缩进 current-state |
| `index-only` | write 不直接读，只供 review/人类按需查 |

## 5. Skill 定位

### 5.1 pop-emergent

定位：初始化、修复、审计。

触发场景：

- 新建涌现式项目
- 项目跑偏
- 缺 `current-state.md`
- 缺 `soul.md`
- 需要检查项目是否符合规范
- skill scope/版本疑似错乱

不做：

- 不写正文
- 不生成燃料长表
- 不审稿长评
- 不作为每轮常驻总控

### 5.2 pop-emergent-write

定位：只负责成文。

每次只读：

1. `涌现/current-state.md`
2. `涌现/soul.md`
3. 最近一章正文或用户指定锚定正文
4. 用户本轮要求

禁止默认读取 seed/research/设定库/人物库/剧情线/review-沉淀。缺关键信息时停止，要求 review/ledger 修复 current-state。

### 5.3 pop-emergent-review

定位：审稿 + ledger。

核心产物不是长评，而是新版 `current-state.md`。写完一章后必须执行 review，把新增事实、人物变化、设定变化、燃料使用、伏笔债务压缩进下一轮 write 的入口。

### 5.4 pop-emergent-seed

定位：确定故事靠什么成立、靠什么长期爽、哪些判断不能丢。

输出：

- 轻量 seed
- 主卖点
- soul 草案
- 题材机制待 research
- 伏笔/长线债务

### 5.5 pop-emergent-research

定位：找能入场面的燃料和题材机制。

research 结果不直接交给 write，而是由 review/ledger 筛选进 `current-state.md`。

## 6. 标准流程

### 6.1 新项目

```text
pop-emergent 初始化骨架
-> pop-emergent-seed 生成 seed 和 soul 草案
-> pop-emergent-research 补燃料和 content-mechanics
-> pop-emergent-review 初始化 current-state
-> pop-emergent-write 写正文
-> pop-emergent-review 更新 current-state
```

### 6.2 续写

```text
检查 scope 中是否有 pop-emergent-write
检查 current-state.md 和 soul.md 是否存在
读取 current-state + soul + 最近正文 + 用户要求
写正文并落盘
执行 review 更新 current-state
```

### 6.3 项目跑偏

```text
pop-emergent 审计
-> 标出缺口：scope / version / current-state / soul / review闭环
-> 调用 review/ledger 修复 current-state
-> 必要时调用 seed/research 补主卖点或燃料
```

## 7. PE 系统提示词注入契约

仅靠 skill description 触发不稳定，因此建议在 Paopao/agent harness 的 expert prompt 中加入 PE 底线提示，替代旧提示：

```text
你是一个网文爽文创作 agent，目标是写出商业男频可读、清晰、有推进、有反馈的正文，规避空泛严肃文学感。

当可用 skill 列表中存在 pop-emergent 系列时：
1. 涉及涌现式小说项目、续写、审稿、燃料、seed、current-state、soul、剧情账本时，必须优先选择 pop-emergent 系列中匹配的 skill。
2. 每次回复必须说明本次采用的 skill 和 execution.mode。
3. 只能声明采用当前可用 skill scope 中真实存在的 skill；如果目标 skill 不在可用列表，不得自称已采用。
4. 写正文前，如果使用 pop-emergent-write，必须确认已读取 current-state.md、soul.md、最近正文和用户本轮要求；缺 current-state 或 soul 时不得标 formal。
5. 正文必须落盘到 txt，对话只回摘要、钩子和创作记录。
6. review 的核心产物是更新 current-state.md，而不是只写体感长评。
```

这段提示词只负责底线，不替代 skill。

## 8. 验收标准

### 8.1 单轮写作验收

- scope 中真实存在 `pop-emergent-write`
- response 不得自称采用不可用 skill
- `current-state.md` 存在且被消费
- `soul.md` 存在且被消费
- 正文落盘到 txt
- 对话只回摘要、钩子、创作记录
- 写完后执行或建议执行 `pop-emergent-review`

### 8.2 review 验收

- 读了待审正文
- 检查上一版 current-state 的硬推进是否兑现
- 检查 soul 是否落到正文
- 输出或更新新版 `current-state.md`
- 标出待确认的设定库/人物库/剧情线/soul/content-mechanics 修改

### 8.3 项目规范验收

- `pop-emergent` 版本不与内容错位
- `pop-emergent-write` 不再要求全项目全文加载
- 所有长期文档带元数据块
- `current-state.md` 控制在 1000-2500 字
- write 不直接读大型库文件
- research 燃料通过 review/ledger 筛入 current-state

## 9. 版本规划

| skill | 版本 | 改动 |
| --- | --- | --- |
| pop-emergent | 1.8.0 | 改为初始化/修复/审计入口 |
| pop-emergent-write | 2.3.0 | 只读 current-state + soul + 最近正文 + 用户要求 |
| pop-emergent-review | 1.1.0 | 增加 ledger 职责，核心产物为 current-state |
| pop-emergent-seed | 1.1.0 | 增加主卖点/soul 草案输出 |
| pop-emergent-research | 1.1.0 | 燃料池改为供 review 筛入 current-state |

## 10. 迁移策略

旧项目迁移时：

1. 保留已有正文。
2. 从 seed/research/设定库/人物库/剧情线/最近正文生成 `current-state.md`。
3. 从项目目标、现有正文和文风资产生成 `soul.md`。
4. 给长期文档补元数据。
5. 下一章起按新版 write 规则执行。

## 11. 废弃内容

以下旧设计不再作为涌现式主线：

- L2/L3 卡作为每章硬运行时结构
- create/revise 双阶段正文流水线
- 每章全项目全文加载
- 由 write 直接读取 seed/research/设定库/人物库/剧情线
- 把文风DNA当剧情机制或内容素材库
- 只在 review 文件里沉淀但不更新下一轮入口
