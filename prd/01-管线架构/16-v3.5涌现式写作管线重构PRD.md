## 1. 背景与问题诊断

### 1.1 两个范式各自的死穴

| 维度 | v2 雪花法 | v3 涌现法 |
|:-----|:---------|:---------|
| 核心逻辑 | 宏观→中观→微观层层细化 | 撒下种子让它自己长 |
| 死穴 | 信息不足被层层放大——宏观决策在信息最少时做出，却被传递到执行层，agent 变成填表机器 | 种子是静态的，剧情是动态的，两者 gap 随章节指数级扩大——第一章 OK，第二章开始失控 |
| 实测表现 | 剧情平庸，无创造空间 | 剧情控不住，设定穿透不达预期 |
| 本质问题 | 把"规划"当"创作" | 设计了更新机制但执行不了 |

雪花法失败的根因是**信息注入不足**——agent 没见过好的结构，设计出来的东西是空洞的。不是方法本身错误。

### 1.2 v3 执行失败的根本原因

v3.4 设计了精巧的更新机制——7组件活记忆、8种种子生长场景、6步循环。但项目d实测中执行率极低：

| 设计 | 实测执行率 | 失败原因 |
|:-----|:----------|:---------|
| 种子生长（8种场景） | 0次/28章 | 太复杂——8种判断场景，agent 注意力不够 |
| 信息获取强制化 | 跳过16章（ch12-23） | 太重——6步循环5步是 meta-processing，创作 token 被挤压 |
| 反思五问后验评估 | 0次回滚 | 没咬合力——反思是自评，没有否决机制 |
| 网文法则执行 | 仅ch001完全遵循 | 更新时机错——状态更新放在流程最后，agent 上下文已耗尽 |

设计了但执行不了，等于没有设计。四个原因：太复杂、太重、没咬合力、更新时机错。

### 1.3 v3.5 要解决的四个核心问题

| # | 问题 | v3.5 解法 |
|:-:|:-----|:---------|
| 1 | 剧情从第二章开始失控 | L2 剧情单元卡提供跨章结构约束，导演意图提供单章质量框架 |
| 2 | 设定穿透不达预期 | L2卡设定引用指针 + 主会话强制读取 + manifest验证 |
| 3 | 创作 token 占比低 | 6步循环简化为5步，子agent context 隔离让 80%+ token 给创作 |
| 4 | 子agent 上下文注入是黑盒 | context manifest 机制实现可观测、可验证、可定位 |

## 2. 核心设计理念

### 2.1 约束驱动涌现

不在"预设"和"涌现"之间二选一，而是建立"动态约束系统"——不预设路径，但预设约束；不写大纲，但注入导演意图。

人类网文作者写作时脑子里转的不是详细大纲，而是四件事：主角当前状态、什么在逼他行动、有哪些未了线索、这一章让读者感受什么。v3.5 把这四件事结构化为可注入的约束。

### 2.2 设计原则

| 原则 | 含义 | 解决的问题 |
|:-----|:-----|:-----------|
| 少而强 | 3个核心约束比30个检查项有效 | v3 太复杂导致执行率为0 |
| 约束不预设 | 给方向不给路径 | v2 预设路径导致平庸 |
| 创作 token 优先 | 80%+ token 给创作 | v3 meta-processing 吃掉60% token |
| library 作为增量信息源 | 写作时按需查询而非预先翻译 | 设定穿透预翻译量级爆炸 |
| L2 卡卡在导演意图粒度 | 不下探到场景设计层 | 保留创作空间同时提供质量框架 |
| 判断前移 | 信息需求判断在 plot 设计时完成 | 写作时 agent 只执行不判断 |

### 2.3 关键概念定义

**导演意图**：单章级别的创作约束（≤150字），从 L2 卡结构分析表提取。定义这章的叙事功能、核心事件方向、情绪曲线、子线推进、章末钩子。说"这章要达到什么效果"，不说"这章怎么写"。

**L2 剧情单元卡**：跨章叙事单元的活文档（4-10章/卡）。定义一个单元的叙事功能、子线交叉、起终点状态变化、节奏控制。它是唯一运行时活文档——每章写作前从中提取导演意图，每单元结束时由 arc 更新。它吸收了 v3.4 种子文档的全部运行时功能。不预设具体场景和台词。

**设定引用指针**：L2 卡结构分析表中每章行的一个列，标注本章涉及的设定文件路径。plot 设计 L2 卡时做信息需求预判，写作时主会话看到指针强制读取对应文件。判断责任从"写作时 agent 自己判断"前移到"plot 设计时一次性判断"。

**活记忆**：章节级状态变化的自然语言日志（YAML append-only）。每章一段话高度概括"这章发生了什么状态变化"，不做结构化字段拆分。作为 L2 卡物理坐标段（单元级更新）的章节级补充。

**状态快照**：从活记忆最新 events + L2卡物理坐标投影出的当前状态（≤400字）。包含主角当前状态、压力仪表盘、未了事项。不持久化，每章实时投影。

**context manifest**：子agent 上下文注入的清单+回执机制。主会话产出 manifest 声明注入了什么，子agent 产出 receipt 确认实际接收了什么，主会话对照检查一致性。详见第8章。

## 4. seed 环节：初始设定设计

### 4.1 职责

与用户共创六要素，初始化整个管线的设定资产。seed 产出的不再是"种子文档"——而是写入参考/设定/ 下的全局设定文件 + 活记忆 baseline#0。

六要素不是 agent 自产，是和用户逐步共创——每步产出后强制用户 QA 确认。

### 4.2 library 在 seed 中的角色

| seed 步骤 | library 查询模块 | 查询内容 | 用途 |
|:---------|:----------------|:---------|:-----|
| step-0 创意共创 | 立项库/00-索引 + 元爽点-变体映射表 | 同类书型参考 | 确定书型和核心卖点 |
| step-1 生态→压力 | 设定库/{对标书}/ + 套路库/00-总索引 | 对标书世界观设计、压力源参考 | 压力矩阵设计 |
| step-1 生态→压力 | 剧情库/L2卡（同类L1大类） | 同类剧情单元的子线结构 | 冲突轴设计参考 |
| step-2 金手指 | 金手指库/00-索引 + 套路库/金手指体系 | 金手指类型参考 | 金手指行动引擎设计 |
| step-3 文风DNA | 文风库/{对标书}.md | 对标书文风DNA档案 | 蒸馏项目文风DNA |

seed 的调研工作由 pop-research 承担（详见第9章），seed 不再内嵌调研SOP。

### 4.3 seed 产出清单

## 5. plot 环节：L2 剧情单元卡设计

### 5.1 职责

plot 环节是 v3.5 的核心。它设计跨章的叙事结构——L2 剧情单元卡。L2 卡定义一个4-10章跨度的叙事单元要完成什么叙事功能、有哪些子线、起终点的状态变化、节奏控制。

关键设计决策：L2 卡的粒度卡在"导演意图"层。它定义"这组章节要达到什么效果"，不预设具体场景和台词。这保留了 v3 的涌现空间，同时提供了 v2 缺失的结构约束。

### 5.2 L2 卡 = 唯一运行时活文档

v3.5 取消种子文档。L2 卡吸收其全部运行时功能：

| 种子文档原功能 | L2卡如何承担 |
|:---------------|:-----------|
| 本章聚焦（导演意图） | 结构分析表每章行 |
| 任务表（主线/压力/线索） | 结构分析表+嵌套子线 |
| 状态快照投影 | 物理坐标段（人物表+力量体系现状） |
| 活跃线索 | 嵌套子线（延伸/闭合标注） |
| 本章三问 | 从结构分析表本章行推导 |

种子文档中跨单元不变的全局设定（主角引擎/金手指/压力矩阵/冲突轴/核心卖点）移到写作参考/设定/，作为 ground truth。

### 5.3 L2 卡的来源：事前设计 + 多层信息依赖

v2 雪花法失败的根本原因是信息注入不足。v3.5 的解法是让 plot 环节在设计 L2 卡时，从多个信息源获取支撑。

plot 设计 L2 卡时的信息依赖优先级：

| 优先级 | 信息源 | 用途 | 必须/可选 |
|:-------|:-------|:-----|:---------|
| 1 | 前一个L2卡实际内容（arc更新后） | 衔接——主角当前状态、子线延伸状态、未回收伏笔 | 必须（L2-001除外，首张无前序） |
| 2 | 写作参考/设定/ | 全局设定 ground truth | 必须 |
| 3 | library匹配的同类L2卡 | 结构标杆——同类单元的设计维度参考 | 必须 |
| 4 | L3剧情线卡（如有） | 宏观参考——剧情线已推进到哪、哪些债务待回收 | 可选（L2-001除外，首张无L3） |

关键设计决策：plot 不事前设计 L3。L3 由 arc 从实际写作中事后提炼（详见第10章）。plot 设计 L2 卡时，如果 L3 已存在则作为宏观参考，不存在则不依赖。

plot 设计 L2 卡的流程：

```
1. 读取前一个L2卡的实际内容（arc更新后的物理坐标+子线状态）——L2-001除外
2. 读取写作参考/设定/下的全局设定
3. 确定当前L1大类，从library匹配同类L2卡（2-3张）作为结构参考
4. 读取L3卡（如已有）作为宏观参考——剧情线推进进度+待回收债务
5. 综合以上4层信息，设计当前项目L2卡
6. 信息需求预判：扫描事件brief，标注设定引用指针
7. 识别信息缺口：本地没有的，调用pop-research调研补缺
8. 产出L2卡，存入卷纲/
```

agent 见过好的 L2 卡长什么样，设计时就有标杆。参考的是结构骨架和设计维度，内容是当前项目自己的。

### 5.4 L2 卡标准格式

```markdown
# L2-NNN · {单元名称}（chXXX-chXXX）

## 元数据
| 字段 | 内容 |
|:-----|:-----|
| 单元编号 | L2-NNN |
| 单元名称 | |
| 单元类型 | 建置型/推进型/高潮型/余波型 |
| 章节范围 | chXXX-chXXX |
| 叙事功能 | 本单元在全书中完成什么 |
| 核心成就 | 单元结束时主角的状态变化（含数值） |
```

### 5.5 设定引用指针

结构分析表新增"设定引用"列。这是 v3.5 的核心机制——plot 设计 L2 卡时做信息需求预判，将每章涉及的设定文件路径标注在指针列中。

```markdown
| 阶段 | 功能 | 章节 | 事件brief | 情感 | 节奏 | 设定引用 |
|:-----|:-----|:-----|:---------|:-----|:-----|:---------|
| 推进 | 首次猎杀 | ch003 | 废弃厂房双杀→系统升级→街头感知解锁 | 紧迫→冷酷→满足 | 极快 | 写作参考/设定/金手指.md#升级机制, 写作参考/设定/主角引擎.md#行为准则 |
```

设定引用是路径指针，不是内容副本。emerge 写作时主会话看到指针，强制读取对应文件并通过 manifest 注入子agent。agent 不需要自己判断"要不要读设定"——plot 设计时已经判断过了。

### 5.6 事件 brief 的粒度要求

事件 brief 用**动作链**而非功能描述：

| 粒度 | 示例 | 是否合格 |
|:-----|:-----|:---------|
| 功能描述 | "索伦趁乱清理铁钳帮据点" | 不合格——太抽象 |
| 动作链 | "索伦从系统面板看到据点坐标→趁群龙无首潜入仓库→黑暗中逐个清除3个守卫→发现账本→带走战利品→撤离时看到教团资助文件" | 合格——有具体动作和因果 |

### 5.7 子线咬合的要求

### 5.8 plot 信息需求判断流程

### 6.4 Step2：信息获取

信息获取由两部分组成：

**A. 设定指针强制读取（机械执行）**

**B. 按需查询（agent判断）**

| 需求类型 | 判断问题 | 信息源 |
|:---------|:---------|:-------|

### 6.5 Step3：子agent创作（create→revise）

**create 阶段（故事结构层）：**
- 注入：L2卡导演意图段+状态快照+上章末尾+设定文件(指针强制读取)+info_acquired+L2卡(可选)
- 不注入：文风DNA（避免创作时模仿文风）
- 产出：正文初稿+创作决策记录+context receipt
- 自主涌现：在导演意图框架内自由创作，不预设场景台词

**revise 阶段（渲染层）：**
- 注入：初稿+文风DNA（硬阻塞，Get-Content -Raw完整加载）+L2卡要素切片(主角引擎+金手指+活跃线索)
- 不注入：导演意图（创作决策不回传）、info_acquired
- 产出：重写稿+修订记录+context receipt
- 完全重写+文风DNA终验+导演意图验证（新增）+事实一致性+字数终检

**导演意图验证（revise新增）：** revise 重写后，对照导演意图检查产出：
- 叙事功能是否匹配
- 事件链是否覆盖（核心事件是否都出现了）
- 情绪曲线是否对齐
- 子线是否推进
- 章末钩子是否有

### 6.6 自动连贯规则

- Step0（导演意图确认）和 Step3（重写稿验收）是仅有的两个人工check点
- Step1 → Step2 → Step3 自动执行不暂停
- Step4 → Step5 自动执行不暂停

## 9. pop-research：独立调研 skill

### 9.1 为什么独立

当前 WebSearch 的问题是流程问题——agent 临时搜两下就完事，没有深度标准，容易停在浅层。seed 环节的 research 之所以好，是因为有完整 SOP 约束（三界框架+分界调研+交叉困境+质量检查），但被锁在 seed 的 step-1 里，其他环节用不到。

v3.5 将种子搜索法 SOP 提取为独立 skill，三个环节共用。

### 9.2 职责

接收其他 skill 传入的调研需求，按规范化流程执行深度调研，产出存盘到写作参考/知识沉淀/并返回路径指针。

### 9.3 调研流程

```
1. 接收调研需求（调用方传入：领域+具体问题+期望深度）
2. 三界框架定界（社会界/自然界/超自然界）
3. 分界调研（社会界6维度/自然界6维度/超自然界5维度）
4. 压力源转化（6字段结构化）
5. 交叉困境识别（6种交叉类型）
6. 质量检查（≥3个压力源、≥3个交叉困境、含具体案例）
7. 存盘到写作参考/知识沉淀/{主题}.md
8. 更新写作参考/索引.md
9. 返回路径指针给调用方
```

### 9.4 三环节调用方式

| 调用方 | 调研需求 | 传入参数 | 产出 |
|:-------|:---------|:---------|:-----|
| seed | 世界观+压力矩阵调研 | 书型+核心卖点 | 写作参考/设定/压力矩阵全量.md + 研究档案 |
| plot | L2卡设计时的场景调研 | 事件brief+涉及领域 | 写作参考/知识沉淀/xxx.md + L2卡设定引用指针更新 |
| emerge | 写作中发现信息缺口 | 缺口描述 | 写作参考/知识沉淀/xxx.md + 索引更新 |

### 9.5 调研深度标准

- 每篇知识沉淀≥500字
- 必须含具体案例/流程步骤/数据数值
- 多轮搜索（最多3轮）
- 无需求需逐类说明理由

```
...
```

触发条件：L2 单元最后一章的 Step5（落盘）完成后，自动触发 arc。不等"每10-20章"——每个单元结束时立即跑。

### 10.3 arc 执行流程（5步）

| Step | 职责 | 说明 |
|:-----|:-----|:-----|
| step-1 宏观检查 | 六项检查（主线进度/子线闭合/伏笔回收/节奏连贯/压力推进/设定一致） | 检查对象为活记忆+L2卡完成度+L3剧情线进度 |
| step-2 设定修剪 | 关闭过期线索/标记休眠要素 | 修剪对象为L2卡子线+写作参考/设定 |
| step-3 L2卡更新 | 单元结束时更新L2卡物理坐标+子线状态 | 把L2卡从"设计方案"更新为"实际记录" |
| step-4 L3卡提炼/更新 | 从已写L2单元提炼剧情线追踪卡 | 跨单元的伏笔回收+子线延伸+力量体系变化曲线 |
| step-5 活记忆压缩 | 合并baseline+events→新baseline | 自然语言合并 |

### 10.3 L2 卡更新（v3.5新增）

单元结束时（如ch005写完，L2-001覆盖ch001-005），arc 更新 L2 卡：

```
1. 读取本单元所有章节的正文+活记忆events
2. 更新L2卡：
   - 物理坐标段：回填实际的时间轴/地点/人物状态/力量体系数值
   - 嵌套子线：标注实际闭合/延伸状态
   - 子线咬合：回填实际因果链
   - 结构分析表：核对事件brief与实际发生的偏差
3. 产出更新后的L2卡（供下一个L2设计时优先级1参考）
```

### 10.4 L3 卡提炼/更新（v3.5新增）

L3 卡不是事前设计的——它从实际写作中事后提炼。第一个 L2 单元完成后，arc 提炼 L3 卡的初始版本；后续每个 L2 单元完成后，arc 更新 L3 卡。

L3 卡追踪的是**剧情线**（不是叙事单元）。一张 L3 卡覆盖一条剧情线贯穿多个 L2 单元的全过程：

```
L2-001完成 → arc提炼L3-001初始版（从L2-001实际内容提取剧情线起点+阶段S1）
L2-002完成 → arc更新L3-001（追加阶段S2+更新力量体系快照+更新叙事债务表）
L2-003完成 → arc更新L3-001（追加阶段S3+...）
...
```

L3 卡提炼流程：

```
1. 读取本单元更新后的L2卡+活记忆events
2. 识别本单元推进了哪些剧情线（从L2卡嵌套子线提取）
3. 对每条已有L3卡：追加新阶段（S+1）+更新力量体系快照+更新叙事债务表
4. 对新出现的剧情线：创建新L3卡（L3-NNN），填写起点快照+初始阶段
5. 更新L3卡与其他L3线的咬合关系
6. 产出更新后的L3卡（供后续L2设计时优先级4参考）
```

L3 卡的标准格式继承 library 拆书产出的 L3 卡结构（元数据+物理坐标+一句话剧情线+核心驱动力+阶段推进表+起点终点快照+叙事债务与回收+与其他L3线咬合+可迁移要点+不可迁移部分）。

### 10.5 L2 卡完成度检查

| 检查项 | 检查内容 | 预警条件 |
|:-------|:---------|:---------|
| 导演意图执行率 | 已写章节中导演意图5项验证通过的比例 | <70% 触发预警 |
| L2卡偏离度 | 实际剧情与L2卡设计方案的偏差程度 | 单元结束偏差>40% 触发预警 |
| 子线推进覆盖率 | L2卡设计的子线在实际写作中是否都推进了 | <60% 触发预警 |
| 事件密度达标率 | 已写章节的事件密度（8-12个/章）达标比例 | <50% 触发预警 |
| L3叙事债务回收 | L3卡追踪的伏笔是否在预期窗口内回收 | 超期>20章触发预警 |

### 11.3 验收标准

**功能性验收：**
1. seed 产出写作参考/设定/下的全局设定文件（不再产出种子文档）
2. plot 能从 library 匹配同类 L2 卡并参考设计当前项目 L2 卡
3. L2 卡结构分析表含设定引用指针列
4. plot 能识别信息缺口并调用 pop-research 调研补缺
5. emerge Step0 能从 L2 卡提取导演意图
6. emerge Step2 能根据设定指针强制读取设定文件
7. create 在导演意图框架内自由涌现
8. revise 对照导演意图验证产出（5项检查）
9. context manifest 每次dispatch产出，receipt每次附带，一致性检查执行
10. 活记忆为自然语言段落，每章一段话
11. arc 能更新 L2 卡（单元结束时）
12. arc 能提炼/更新 L3 卡（剧情线追踪）
13. plot 设计 L2-002 时实际读取了 L2-001 实际内容（优先级1）
14. pop-research 能被 seed/plot/emerge 三环节调用

**质量验收（10章试写）：**
15. 10章中导演意图执行率≥70%
16. 10章中事件密度达标（8-12个/章）比例≥60%
17. 10章中文风DNA完整加载率100%
18. 10章中设定文件读取执行率100%（指针指向的全读到）
19. 10章创作token占比≥60%
20. L2卡偏离度≤40%

**library 整合验收：**
21. plot 设计 L2 卡时实际读取了 library 参考卡
22. emerge Step2 查询 library 的次数≥3次/10章
23. library 查询结果在 receipt 中确认应用

**L3 剧情线验收：**
24. L2-001完成后arc能提炼L3-001初始版
25. L2-002设计时读取了L3-001作为宏观参考（如有）
26. L3卡叙事债务表追踪伏笔回收

### 11.4 风险与缓解

| 风险 | 影响 | 缓解措施 |
|:-----|:-----|:---------|
| L2卡设计质量不达标 | 导演意图约束力弱 | library参考注入+动作链粒度要求+arc偏离度检查 |
| 设定指针指向的文件不存在 | 强制读取失败 | plot设计时验证路径存在；不存在则调用pop-research补缺 |
| 导演意图过于约束 | 回到v2填表模式 | 卡在导演意图粒度，不下探场景设计层 |
| context manifest token开销过大 | 创作token占比受限 | manifest+receipt总量控制在单章创作token的30%以内 |
| pop-research调研深度不足 | 知识沉淀质量低 | 继承seed的调研SOP+质量检查（≥3个压力源、≥3个交叉困境） |
