## 1. 背景与问题诊断

### 1.1 起点skill群组现状

起点skill群组现有9个skill，但与番茄skill群组（8个成熟skill）相比存在结构性缺失和版本落后。同时起点群组有两个独特优势：world已实现三层骨架理论（力量体系+动力引擎+金手指），以及流派write分离架构（dndlike/onepiece）。

| skill | 起点版本 | 番茄对应版本 | 差距 |
|:--|:--|:--|:--|
| 总控 | pop-qidian v4.0.1（初始化审计入口，非pipeline） | pipeline v3.2.0（Phase路由+state可视化） | 无pipeline总控 |
| seed | v7.0.0 | v13.17.0 | 差6个大版本，缺七维底牌/双轨发散/故事纲领/黄金首章 |
| world | v1.0.0 | v1.5.1 | 版本低但理论更深（已有三层骨架+动力引擎） |
| plot | v3.0.0 | v3.5.2 | 缺分支剧情线/分幕设计/困难三层面 |
| write | v2.0.1 | v8.3.1 | 差6个大版本，缺DNA三态协议/设定库精选注入 |
| review | v2.0.1 | v4.2.1 | 缺四维审核框架/好看度4问 |

| research | v3.5.1（仅燃料+机制） | v2.2.0（decon-lite 9表） | 无decon-lite拆书 |
| 流派write | dndlike v1.0.1 + onepiece v1.0.1 | 无（番茄单一write内化） | 起点独有优势 |

### 1.2 与三层骨架SOP的八个核心差距

| # | 差距 | 现状 | SOP要求 | 影响 |
|:--|:--|:--|:--|:--|
| 1 | 无pipeline总控 | pop-qidian只是初始化审计入口，无Phase路由 | Phase 0→5路由+state追踪 | agent无法按依赖链推进，靠自觉 |
| 2 | seed严重落后 | v7.0.0，无七维底牌/双轨发散/故事纲领/黄金首章 | seed是骨架层+主角层核心 | 立项质量不可控，缺市场校准和首章试读 |
| 3 | 骨架层位置错误 | world v1.0.0 有三层骨架理论但放在world | 骨架应在seed（立项第一优先级） | 骨架成为世界构筑的子模块而非立项核心 |
| 4 | 无character skill | 角色管理在seed的Step 4角色储备池 | character是plot→write之间必经站 | 角色无独立闭环，write缺角色库唯一源 |
| 5 | research无decon-lite | 拆书分散在pop-decon系列5个独立skill | decon-lite 9表统一档位供骨架设计 | 力量体系/动力引擎/金手指缺乏参考书结构化输入 |

| 8 | plot缺分支剧情线 | 2步精简版（素材→幕纲白描） | 分支剧情线≥6条+分幕设计+困难三层面 | 剧情结构单薄，缺多线交织 |

差距 3 是起点的独特情况——world已经有了三层骨架理论（比番茄world更深），但放错了位置。SOP要求骨架在seed（立项核心），起点需要把world的三层骨架设计能力前移到seed，world改为消费骨架。

### 1.3 起点的两个独特优势

**优势一：world的三层骨架理论**。起点world v1.0.0 已经实现了"力量体系（广义坐标系）+动力引擎（众生攀登系统六组成）+金手指（助力定位）"的三层骨架理论，基于24本男频小说验证。这比番茄world v1.5.1（只有力量体系四层结构，无动力引擎概念）理论更深。重构时这份方法论资产是起点的核心竞争力，前移到seed后作为骨架设计的指导方法论。

**优势二：流派write分离架构**。起点有dndlike和onepiece两个独立流派write skill，每个自带DNA+流派技法+战斗模式分等级段。番茄是单一write内化。流派分离架构让起点可以按赛道快速扩展写作引擎，重构时保留这一架构。

## 2. 重构目标与原则

### 2.1 目标

基于男频SOP三层骨架（骨架→主角→血肉）重构起点skill群组，补齐缺失的pipeline和character，升级落后的seed/write/review/research，利用world已有的三层骨架理论优势和流派write分离优势，建成与番茄skill群组对等且理论更深的起点创作管线。

### 2.2 原则

**骨架前移**：将world的三层骨架设计能力（力量体系+动力引擎）前移到seed Phase 1，world改为消费骨架。world的三层骨架理论作为方法论资产指导seed的骨架设计。

**依赖链优先**：骨架就绪才能设计主角，主角就绪才能填充血肉。pipeline总控保证依赖链不被跳过。

**优势保留**：流派write分离架构（dndlike/onepiece）保留不动，只微调对齐新架构。world的三层骨架理论方法论保留，从"world的设计能力"变为"seed的设计指导"。

**decon-lite收敛**：research引入decon-lite 9表统一档位，收敛分散在pop-decon系列的拆书功能。

## 4. 各 skill 重构方案

### 4.1 新建 pop-qidian-pipeline：Phase路由总控

起点完全缺少pipeline总控，pop-qidian v4.0.1只是初始化审计入口。新建pop-qidian-pipeline，参考番茄pipeline v3.2.0，实现Phase 0→5路由+project-state.md状态追踪。

**职责**：
- 项目初始化（创建标准化四文件夹：素材/设计/正文/审核）
- project-state.md/html状态追踪（当前Phase/骨架就绪/主角就绪/血肉就绪/最近产出）
- Phase路由（按依赖链硬约束路由到对应skill）
- 版本快照管理

**Phase 路由**：

| Phase | 路由目标 | 触发条件 |

| Phase 0 | research（decon-lite 9表）+ dna-style | 用户给出参考书 |

| Phase 3 | world（地图+势力+全书设定）→character（角色库）→plot（剧情白描） | Phase 2完成 |
| Phase 4 | write（兜底模板或流派write） | Phase 3完成 |
| Phase 5 | review | Phase 4每章完成 |

**project-state.md 新增字段**：

| 字段 | 内容 |

### 4.4 新建 pop-qidian-character：角色管理引擎

## 7. 风险与依赖

### 7.1 seed跨版本升级风险

seed从v7.0.0直接跳到v8.0.0，跨了6个大版本（番茄seed从v7到v13经历了多次迭代）。一次性引入七维底牌+骨架层+主角层+双轨发散+故事纲领+黄金首章，改动量巨大。

**对策**：分两阶段交付（阶段2骨架层+阶段3主角层），每阶段独立验证。seed的SKILL.md速查表必须明确标注Phase 1和Phase 2的Step顺序和依赖关系。

### 7.2 world三层骨架理论前移风险

world v1.0.0的三层骨架理论是起点的核心竞争力，前移到seed后world的核心价值从"设计骨架"变为"消费骨架生长血肉"。如果前移过程中方法论丢失，seed的骨架设计质量会下降。

**对策**：world的三层骨架理论方法论完整移植到seed的Step 1d/1e设计指导中。world保留对骨架的消费验证能力——如果骨架.md无法支撑血肉生长，打回seed重做。world的拆解模式和融合模式不受影响。

### 7.3 character新建的plot依赖风险

character作为Phase 3.5必经站，依赖plot的分幕设计出场角色清单。如果plot升级（阶段5）晚于character新建（阶段4），character会因plot未产出分幕设计而无法运行。

**对策**：阶段4新建character时，先基于现有plot v3.0.0的幕纲产出设计角色清单（过渡方案）。阶段5 plot升级后，character切换为消费分幕设计的正式方案。

### 7.4 流派write兼容性风险

dndlike和onepiece是独立完整skill，微调对齐新架构时可能破坏现有流派技法体系。

**对策**：流派write只微调两点（加载主角设计.md+角色库.md，爽感层锚定坐标系跃迁），不动流派技法和DNA目录。微调后跑一次dndlike测试章节验证兼容性。

### 7.5 decon-lite与pop-decon系列共存风险

research引入decon-lite后，与现有pop-decon系列5个独立skill功能重叠。agent可能混淆两者触发条件。

**对策**：明确边界——decon-lite是轻量拆9表（4步完成），pop-decon系列是重拆（完整立项圣经）。pipeline的路由规则明确：七维底牌摸底时用户给参考书→decon-lite；用户主动要求精拆一本书→pop-decon。

### 7.6 版本一致性风险

本次重构涉及8个skill（含2个新建），必须保证SKILL.md/skill.json/CHANGELOG.md三处版本号一致。

**对策**：每阶段交付时统一检查三处版本号。pipeline的版本快照表同步更新。
