## 一、问题总览

写作管线项目在推进 15-20 轮后出现两个连锁问题：

1. **Context overflow 硬拦截。** 6-12项目a 15 次 run 中 5 次 overflow（33%），6-14-项目a 24 次 run 中 19 次 overflow（79%）
2. **overflow 期间 agent 完全失忆。** 每个被拦截的 session 只有 668 字节——agent 看到的只有一行 `prompt too large`

**根因是架构级的不对齐：** OpenClaw/Hermes 的 session 模型把 toolCall payload（文件写入的完整内容）当作对话历史来存储和回灌，而写作管线的 toolCall 本质上是**产出物落盘**而非**需要回忆的对话**。

---

## 二、根因分析

### 2.1 toolCall payload 的链式累积

```
Phase           每轮 toolCall payload  →   累积
─────────────────────────────────────────────
creative        ~5KB × 3 轮           →  15KB
world           ~16KB × 3 轮          →  63KB
plot            ~12KB × 4 轮          →  111KB
chapter-design  ~8KB × 5 轮           →  151KB
deconstructor   ~29KB × N 轮          →  ＞325KB  ← 此时溢出
```

### 2.2 三层截断的叠加效应

| 层面 | 问题 | agent 感知 |
|:-----|:-----|:-----------|
| System prompt | limitChars=32768，截掉 15% | 丢失 ~6K chars 的 skill/tool 描述 |
| Compaction | 只压缩自然语言对话，不碰 toolCall payload | 记住了"用户要写跨界小说"，不知道"L1 六件套 / 3卷13章" |
| toolCall payload | write 工具的完整文件内容被视为对话历史 | 每轮 5-30KB 历史累积，75K tokens 的产出废物 |
| Overflow | 不是截断，是完全拒绝 | agent 看到 668 字节 + "prompt too large"，以前所有创作进度归零 |

### 2.3 Compaction 压缩机制空转

session 中存在 4 个 compaction 事件，但全部内容为空摘要（`No prior history. None.`）。4 个 compaction 总计 37,379 字符，没有压缩任何历史内容，反而自身占用了 10% 的 prompt 预算。

### 2.4 产出摘要化已生效但不够

expert-writer v3.0+ 在 18 个 skill 文件中新增产出纪律："写入文件后对话中只留摘要（≤ 200 字）"。assistant text 从 29KB → 700 字（-97%）。**但产出纪律管的是 assistant text 输出，管不了 write 工具调用的 payload。**

## 三、框架层改造方案

### 3.1 核心方案：toolCall payload 截断

**改造位置：** `agent/context_compressor.py` 的 `_prune_old_tool_results` 方法

**改造前行为：** `write` 类型的 toolCall 在注入历史时，完整存储 `input.content`（文件全部内容），导致每轮 5-30KB 累积。

**改造后行为：** `write` 类型的 toolCall 在注入历史时，不存 `input.content`，仅存元数据：

```json
{
  "name": "write",
  "input": {
    "file_path": "00-原始设定/L1-元设定层/01-世界蓝图.md",
    "content_size": 5234,
    "summary": "含地理/位面/时间/法则四个字段"
  }
}
```

**效果：** 每轮 5-30KB → ~100 字节。完整管线全程累积节省 ~150,000 tokens。

**副作用：** 模型在后续轮次看不到完整文件内容，需要 agent 通过 `Read` 工具重新加载。这是可接受的——assistant text 摘要已经指引了文件路径。

### 3.2 已有截断机制（源码确认）

`_truncate_tool_call_args_json` 函数已存在于 `context_compressor.py` 中，能解析 tool_call 的 arguments JSON 并截断所有长字符串字段到 200 字。但当前只在 `_prune_old_tool_results` 的第一遍扫描中使用，未覆盖 protected tail 区域内的 `write_file` 调用。

**改造要点：** 在 `_prune_old_tool_results` 中增加第二遍扫描，对 protected tail 区域内的 `write_file` 工具调用参数也做截断。

### 3.3 辅助方案

| 方案 | 层面 | 可节省 | 状态 |
|:-----|:-----|-------:|:----:|
| 清理 Workspace 引导文件 | 客户端 | ~3,300 tokens | ❌ 无效（OpenClaw 从内部模板注入） |
