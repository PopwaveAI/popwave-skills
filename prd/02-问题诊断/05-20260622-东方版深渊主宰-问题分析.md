## 2. 根因详解

### Gap① 无「重大变更影响范围声明」协议

**证据链：**

```
#47 (14:43) 用户：「修改下，主角新增穿越者设定...强化圣者浩劫」
#48 (14:46) agent 回应：「关键改动全部完成。已更新文件清单：
  江轩-主角角色卡 / 起点快照 / L1-03 历史与驱动力」
           ——更新 3 个文件，但实际依赖树涉 25+ 文件

#48 末尾：「需要我继续更新 动态升级表（新增穿越者觉醒线）
         和 L1-01 世界蓝图（补充圣者浩劫层级描述）吗？」
           ——问号结尾 → 把「应该做的事」推给用户决定

#49 (14:47) 用户炸：「剧情设计文档还是旧的？
         我说要追加剧情线你对应文档都没碰？
         然后幕纲里都缺乏剧情线太少了...」
```

**Skill 层问题链：**

1. **expert-writer\references\think\typical-errors.md** — 8 条典型错误，**无一条覆盖「重大设定变更后的全链影响声明」**
2. **expert-writer\references\pipeline\manifest.md** — 定义了管线顺序，但**没有「当上游设定被修改后，下游哪些文件需要同步更新」的映射表**
3. **pop-writer-world\SKILL.md** — 边界条件有「宪法更新→标记受影响维度→重推→连锁重建下游」，但这是针对"宪法更新"不是"穿越者这种颠覆级设定变更"。穿越者同时影响 World（L1 全部）+ Character + Plot（全部剧情线+幕纲）+ Chapter + Prose，**无任何技能定义了这种跨阶段变更的传播协议**
4. **pop-writer-plot\SKILL.md** — 边界条件有「用户中断→用户改卷目标或主元爽点→回到步骤1」，但那是"正在写剧本时用户改方向"，不是"剧本已经写完了，用户改了上游的世界设定"

**缺失设计：**

```yaml
# 应当存在的 Skill 级协议
change_magnitude:
  minor: # 局部润色（改一个名字）→ 只改本文件
  medium: # 单维度设定扩展（加一个神系）→ 本技能 + 相邻1-2技能
  major: # 颠覆级设定（加穿越者 / 改全书核心矛盾）
    action: |-
      ① 禁止直接编辑
      ② 先输出「影响范围声明」——列出所有可能过时的文件路径
      ③ 用户确认范围后，逐项标 Updated / N/A / Rebuild needed
      ④ 再逐文件执行更新
```

### Gap② 无「设计意图捕获」前置步骤

**证据链：**

```
#38-#41 (14:22-14:27)：
  Agent 建的 plot 结构：1条主线（江轩成长）+ 6条支/设/反派/关系线 = 7条线平铺
  用户接受，但潜意识里用的是不同框架

#49 (14:47) 用户揭露真正框架：
  「skill没要求 主线1 是世界危机 主线2 是主角成长线 主线3是主角行动线吗？」

  关键点：这个分歧从 plot 阶段一开始就存在，但 agent 从未主动问过
  「你心中的主线结构是怎样的？」
  用户只能在 5 轮对话后、跨过 chapter+prose 再回到 plot 时才被迫说出真正结构
```

**Skill 层问题：**

1. **pop-writer-plot v7.9.0** — Step 1 直入「创建卷纲·战略层」，读 PRD / L1 / 角色卡 / 数值体系，**没有任何"先和用户确认主线结构的心智模型"的步骤**
2. **expert-writer** — 路由到 plot 时，子 skill 的 think 阶段只做「前置校验」（文件是否存在），**不做「用户心智模型校验」（你的主线结构是几条？怎么分工？）**
3. 对比 **pop-writer-world** — 至少有一个"写 L1-02 机制段前必做：向用户确认等级数量/分段范围"的指令。**Plot 阶段缺少对应的"确认主线结构"指令**

**缺失设计：**

```yaml
# pop-writer-plot Step 0 应当增加
step-0-user-model-capture:
  ask: |-
    「在开始设计剧情线之前，我想先确认你对主线结构的理解。
    你心中全书的主线是：
    A. 1条主线 + 多条支线（主线负责所有叙事功能）
    B. 2条主线并行（如：世界危机 + 主角成长）
    C. 3条主线并行（如：世界危机 + 主角成长 + 主角行动）
    D. 其他结构——请描述一下」

    「每条主线的核心职责是什么？它们之间的关系是平行、交织还是嵌套？」
  gate: 用户未确认结构之前不进 Step 1
```

### Gap③ Pipeline 文档标记 "S"（静态）与事实不符

**证据链：**

```yaml
# expert-writer\references\pipeline\manifest.md
小说世界设定/L1-01~06.md          S  ← 标为 Static
小说世界设定/数值体系/*.md         S  ← 标为 Static
小说世界设定/起点快照.md           S  ← 标为 Static
小说世界设定/动态升级表.md         S  ← 标为 Static
```

这些文件在 project.json 的 3 小时内是"一次性产出"的。但项目生命周期（按年计的写作管线）中，**设定在 character/plot 阶段被修改是完全正常的**。标记 "S" 给了 agent 错误的安全感——"这些文件已经锁定了，不需要再碰"。

**实际证据：** #47 穿越者设定加入后，L1-01(世界蓝图)未更新、L1-02(力量体系)未更新、L1-04/05/06/07 均未更新、数值体系未更新、动态升级表未更新。

**修正建议：**

```yaml
# 修正 manifest 的 S/D 标记逻辑：
# S = 相对于当前写作阶段的"基线版本"
# 但每次上游变更后，标记自动失效，下游检测到 staleness 后重算
# 所有标注为 S 的文件其实应理解为: Snapshot(版本化快照，带全局版本号)
```

```

  ✓ 起点快照（加穿越者板块+圣者浩劫）
  ✓ L1-03 历史与驱动力（加圣者浩劫描述）

  未更新且未标记为过期：
  ✗ L1-01 世界蓝图（无圣者浩劫层级描述）
  ✗ L1-02 力量体系（无穿越者"先知已知"视角）
  ✗ L1-04/05/06/07（全部无穿越者影响）
  ✗ 数值体系（无穿越者认知偏差对等级感知的影响）
  ✗ 世界宪法（无穿越者约束条款）
  ✗ 动态升级表（无穿越者觉醒路径）
  ✗ 卷一-卷纲（仍在讲"普通练气士孤儿逐步发现世界"）
  ✗ 主线1-世界危机线（无穿越者"先知验证"引擎）
  ✗ 主线2-主角成长线（无穿越者知行不合一设计）
  ✗ 主线3-主角行动线（无"过期剧本决策"框架）

  ✗ ch01 设计包（无穿越者内心独白）

```

**Skill 层问题：** 每一个子 Skill 只定义了「我的输入是什么、我的产出是什么」，但没定义**「我的产出被变更后，哪些下游文件应被标记为 stale」**。这需要跨技能共享的依赖映射表。

**缺失设计：**

```yaml
# 应在 expert-writer 层（而不是子 Skill）维护一个全链依赖追踪表

dependency_tracking:
  L1-01_世界蓝图:
    consumed_by: [character(江轩卡), plot(卷纲), prose(ch01背景)]
  L1-02_力量体系:
    consumed_by: [character(all), plot(卷纲+幕纲), chapter(设计包), prose(正文)]
  江轩-主角角色卡:
    consumed_by: [plot(卷纲+剧情线), chapter(设计包), prose(正文)]
  卷一-卷纲:
    consumed_by: [chapter(ch01), prose(ch01)]
  起点快照.md:
    consumed_by: [plot(卷纲), chapter(entity-snapshot), prose(ch01)]
```

### Gap⑤ 下游 Skill 的 I/O 边界过于严格，无法感知上游过期

**证据链：**

```
pop-writer-prose v3.6.0 红线#2：「不读角色卡 — 角色台词风格/行为基线已在设计包的登场人物表中」

这条红线在正常流程下合理——prose 不应从角色卡推断新行为。
但当角色卡核心设定变化时（如「新增穿越者身份」），
这条红线变成了信息孤岛的铁幕：
  - prose 写了 ch01 正文（3276字），基于"普通练气士孤儿"版本
  - 用户加了穿越者设定 → 角色卡更新
  - prose 因为「不读角色卡」红线 → 完全不知道角色卡变了
  - ch01 正文保持旧版本，无穿越者内心独白
```

**Skill 层问题：** 红线是好的，但**「设计包是唯一输入源」这条红线缺少配套的依赖过期校验**。prose 应当在启动时检查：我的上游（设计包、实体快照）是否基于最新版本的角色卡。

**修正建议：**

```yaml
# 在 prose 的红线中增加：
红线#2.5: 「上游版本校验 — 启动渲染前检查 entity-snapshot 的版本戳
           是否与角色卡最后修改时间匹配。不匹配=上游过期→终止并回退到 chapter」
```

### Gap⑥ 无「询问 vs 行动」判定规则

**证据链：**

```
#48：「需要我继续更新 动态升级表（新增穿越者觉醒线）和 L1-01 世界蓝图（补充圣者浩劫层级描述）吗？」

这个问题把本应由 agent 做的机械性更新推给了用户。
"穿越者设定→动态升级表需要更新"是逻辑推导的结果，不是创意决策。
agent 应当直接更新，而不是询问。

对比：#13 用户自己指出"文档还是旧的"——agent 本可以不触发这个纠正。
```

**Skill 层问题：** 当前所有 Skill 都在教 agent "不确定时问用户"。但缺少区分：

| 类型 | 特征 | 应该 |
|:-----|:-----|:-----|
| **创意决策** | 影响风格/方向/品味 | 问用户（给 2-3 个选项） |
| **机械传播** | A 变了→B 必须同步更新 | **直接做**，不询问 |
| **技术修正** | 某个数算错了/不准确 | 直接做 |
| **可逆推演** | 从已知新设定能逻辑推导 | 做且展示推导链 |

穿越者→动态升级表更新属于 **"可逆推演"**，不是"创意决策"。

```
#8-#18 命名迭代全过程：

  #9  agent 搜文抄公→提案6阶
```

- "通用审视+质量信号"
- "引导语"

**没有这个问题：** "用户刚刚的纠正是指向表面症状（名字不好听/剧情线少），还是指向结构层错配（主线模型/力量体系骨架）？"

### Gap⑧ 管线无回溯机制

**证据链：**

```
Pipeline 硬性顺序：creative → reservoir → world → character → plot → chapter → prose → qa

当穿越者设定在 chapter+prose 之后加入时：
  正确做法：回溯到 creative（更新 PRD → re-run reservoir → re-run world → ...）
  实际做法：驻留在 prose 阶段，补丁式更新就进文件
```

**Skill 层问题：** Manifest 定义了"不可跳跃"，但**没定义"当需要回溯时如何倒退"**。回溯需要：
1. 声明回溯目标阶段
2. 标注受影响范围
3. 倒序推进（从当前阶段退到回溯目标阶段之间已产出的文件，逐级检查）

**缺失设计：**

```yaml
# pipeline manifest 需增加的章节
管线回溯协议:
  触发条件: 用户修改了已固化阶段的设定
  步骤:
    1. 影响范围声明（输出文件列表）
    2. 回溯目标阶段判定
    3. 倒序: 从当前阶段到目标阶段之间的产出物逐级标记 "stale"
    4. 正序: 从目标阶段开始重新推进
  门禁: 未完成全链 stale 标记之前不进任一阶段的编辑
```

### Gap⑨ 无文件版本向量

**证据链：**

```
#49 用户：「剧情设计文档还是旧的？」
此时项目 53 个文件，没有一个机制能让用户或 agent 快速回答：
  - 「哪些文件基于穿越者前版本？」
  - 「哪些已经更新？」
  - 「哪些不需要更新？」

agent 只能凭记忆回答，用户只能逐个文件检查。
```

**Skill 层问题：** 项目总控.md 记录了"管线进度"（哪些阶段完成），但没有**文件级版本戳**。L1-01.md 的内容版本是多少？什么时候最后一次更新？基于哪个"宪法版本"？

**最小实现：**

```yaml
# 在 项目总控.md 中增加文件版本区块
文件版本向量:
  L1-01-世界蓝图.md: v1 (基于 PRD v1)  # stale → 需更新

  卷一-卷纲.md: v1 (基于 PRD v1 + 角色卡 v1) # stale → 需更新
```

### Gap⑩ 用户心智模型与 Skill 模板之间的层错配

**证据链：**

```
这是贯穿全流程的元问题。它解释了为什么所有其他 Gap 都未被执行拦截：

- pop-writer-world 模板的 L1-07（术语与文明底色）是 8 个维度中最不重要的一个，
  但对这位用户的暗黑东方史诗来说，术语命名直接决定了用户是否愿意继续推进

- pop-writer-prose 的「设计包是唯一输入源」红线，
  在穿越者设定加入后变成了信息屏障
```

**Skill 层问题：** Popwave 的写作管线（从 creative 的 PRD 到 prose 的正文）是一个**"通用网文"流水线模型**。它隐含假设了：
- 用户想要"传统的单主线+多支线结构"
- 力量体系用"花哨命名优于传统命名"
- 剧情线是"1 条主线负责所有叙事功能"

但这位用户（高水平作者）的心智模型完全不同：
- 3 条主线并行（世界危机/成长/行动）
- 传统武侠命名（开脉/洗髓/明劲/暗劲/宗师）
- 剧情线之间需要清晰的职责分工

**修正方向：** 不是改模板——而是在**管线入口处（expert-writer Think 阶段）增加「用户心智模型征询」步骤**，让后续 Skill 知道应该用哪个"风格分支"。

```yaml
user_model_profile:
  命名偏好: traditional  # traditional / ornate / minimalist / poetic
  主线结构: multi-main   # single-main / dual-main / multi-main
  剧情密度: high         # low / medium / high
  黑暗程度: dark         # light / dark / grimdark
  确认密度: detailed     # minimal / moderate / detailed（给选项让他定）
```

### Gap⑪ PRD 爽点体系写在了错误的抽象层级 + plot 未做逐卷映射

> **用户纠正（2026-06-23）：** "PRD写的太僵了。说是果位晋升，但本质是变强。第一卷不太可能接触果位、洞天这种中期才会露面的设定。"
> **结论：** Gap⑪ 有两个子问题，一半在 creative 一半在 plot。PRD 的爽点体系用错了抽象层级，导致 plot 既无法验证也无法适配。

**问题定性：**

`pop-writer-creative v4.4.0` 的 PRD 模板（`templates/prd-模板-空白.md`）要求填入的爽点类型是**具体的游戏化术语**（果位晋升、洞天经营、权柄争夺），而不是**抽象的元爽点**（变强、经营建设、权力博弈、探索解谜、危机合作）。

后果：
- PRD 写的 5 种爽点中有 4 种的**字面描述**与卷一无关（果位、洞天、神系权柄、神战）
- 但卷一的**实际爽点意图**（底层逆袭、反杀、发现世界真相）对应的是元爽点"变强"和"探索解谜"——PRD 没有用这些词
- 所以 §⑪ 的"爽点一致性检查"无法机械执行：它要么 0 分过关（字面不匹配），要么跳过检查（无法判定）

**证据链：**

PRD §九 当前写法：
```yaml
# pop-writer-creative 模板产出的当前 PRD
爽点类型: 果位晋升(30%), 洞天经营(20%), 权柄争夺(25%),
          神话发掘(15%), 危机共斗(10%)
# ↑ 这些全是中后期的"叙事事件类型"，不是元爽点
```

应该写的写法：
```yaml
# 元爽点抽象层（适用于全书各阶段）
元爽点: 变强(30%), 经营建设(20%), 权力博弈(25%),
        探索解谜(15%), 危机合作(10%)

# 卷一特化（plot 阶段填充）
vol-1-爽点落地: 变强=境界突破(开脉→洗髓), 探索解谜=发现练气士遗迹,
                底层逆袭=反杀疤脸, ...

# 卷三特化（预留）
vol-3-爽点落地: 变强=首次接触果位碎片, 经营建设=第一座道场, ...
```

再对比 ch01 正文的实际读者体验：
```
读者读完 ch01 感受到的爽点：
? 穿越者信息差（我知道你不知道）— 属于「智识爽点」，PRD 未列出
? 底层生存压迫感 — 属于「困境建置」，PRD 未列出
✗ 果位晋升 — PRD 说 30% 但根本不该在 ch01 出现（用户明确说这是中期设定）
```

**Skill 层根因（两个）：**

**原因 A — creative skill 的 PRD 模板抽象层级过低：**

```yaml
# pop-writer-creative/templates/prd-模板-空白.md 的问题
wrong: |
  | 爽点类型 | 占比 | 实现方式 |
  | 果位晋升 | 30% | 收集果位碎片→提升神力等级 |
  | 洞天经营 | 20% | 建设洞天→聚拢香火 |
  # ↑ 这些是"实现方式的例子"，不是"爽点本身"

correct: |
  | 元爽点 | 占比 | 卷一落地示例 | 中期落地示例 |
  | 变强 | 30% | 开脉→洗髓→明劲 | 接触果位碎片→权柄觉醒 |
  | 经营建设 | 20% | —（卷二启动） | 第一座道场·聚拢香火 |
  | 权力博弈 | 25% | 帮会恩怨→宗门暗流 | 神系拉锯·合纵连横 |
  | 探索解谜 | 15% | 废弃道观·壁画·老瘸子身份 | 上古遗迹·封神榜下落 |
  | 危机合作 | 10% | —（卷三启动） | 神系联手对抗幽冥 |
  # ↑ 元爽点不变，每卷的落地形式不同
```

**原因 B — plot skill 无「元爽点 → 本卷特化」步骤：**

`pop-writer-plot v7.9.0` 的 Step 1（卷纲战略）要求「提取 PRD 的爽点」，但它是照着字面提取的——拿到"果位晋升 30%"就直接在卷一找果位碎片，找不到就标 0%。

它应该做的：
```yaml
# pop-writer-plot step-prd-adaptation 应当增加的逻辑
step: 将 PRD 的元爽点特化到本卷
  1. 读 PRD §九 的元爽点（变强/建设/博弈/探索/合作）
  2. 问用户或用世界观推断：本卷（武侠底层阶段）这些元爽点长什么样？
     - "变强" = 境界突破（开脉→洗髓→明劲）
     - "探索" = 发现废弃道观、练气士壁画
     - "博弈" = 帮会争斗、疤脸上线
  3. 输出「本卷爽点落地表」— 每类元爽点的本卷具体实现形式
  4. 门禁：每类元爽点在本卷至少有一种实现形式，否则退回 creative 协商
```

**修正后的 Gap⑪ 等级：** 仍为 **P0**（creative ↔ plot 之间的抽象层级不对齐是系统性问题，不只是单次执行失误）

### Gap⑫ prose skill 无「对话比例门禁」

**问题定性：** PRD §十 明确承诺"对话比例 ≥ 40%"。实际 ch01 正文的对话比例目测 <15%。大量信息通过主角内心独白释放——恰恰是 PRD 明确禁止的"靠旁白堆设定"。但 `pop-writer-prose` 没有任何检查对话比例的机制。

**证据链：**

ch01 正文（217 行）的对话分布：
```
老瘸子对白：3 句（"疤脸昨晚来过了/三天/收好，别让人看见"）
阿七对白：4 句（"我还以为疤脸把你剁了/疤脸上头有人/今晚别走西门/你自己小心"）
疤脸对白：2 句（"三天码头/不来我就不是找你一个人了"）
疤脸手下：1 句
路人：1-2 句
→ 总计约 12-15 句对白，占正文篇幅 <15%
```

对比 PRD 承诺「≥40%」，偏差超过 60%。而且大量信息通过主角内心独白释放：

```
「圣者浩劫。他记得自己知道这件事——或者更准确地说，记得"前世知道"这件事。」
「他在心里快速对比了一下——"我记得疤脸只是个普通压迫者，三章就没了。这个版本里他有上线。"」
「"我的剧本有缺页。" 江轩在心里说。"而且我不知道缺了多少。"」
```

这完全违反了 PRD §十 的"信息通过对话释放，不靠旁白堆设定"的承诺。

**Skill 层根因：**

`pop-writer-prose v3.6.0` 的验证步骤（`steps/step-2-verify-output.md`）检查：
- ✅ 风格验证（偏差≥2 处退回）
- ✅ 宪法检查（AI 词/解说员句式/视角一致性）
- ✅ 文本脉冲（500 字/微脉冲）
- ✅ 字数检查（>50% 退回）
- ❌ **对话比例检查 — 全无**

**关键缺失设计：**

```yaml
# pop-writer-prose step-2-verify-output.md 应当增加
dialog-ratio-gate:
  check: 对话段字数 / 正文总字数 ≥ 0.4
  trigger: PRD §十 承诺"对话比例 ≥ 40%"
  action: |
    if ratio < 0.3: ❌ 退回 Step 1，标注「对话比例严重不足」
    if ratio < 0.4: ⚠️ 警告，标注「需增加对话释放信息」
  note: 信息通过角色对白释放，不通过主角内心独白
```

### Gap⑬ world skill 与 plot skill 之间无「节奏约束传递」协议

**问题定性：** PRD 定义了清晰的力量阶段节奏（武侠 ch1-ch30 → 高武 ch30-ch100 → 神话 ch100+）。`pop-writer-world` 的 Phase 7 动态升级表本应是这个节奏的具象化工具。但实际传递到 plot 时，动态升级表被解释为"主角从哪层爬到哪层"，而不是"每卷必须覆盖多宽的能级跨度"。结果卷一 50 章只覆盖了半阶跨度（开脉未成→开脉圆满）。

**证据链：**

PRD §六 原文承诺：
```
第一层：武侠（前期·ch1-约ch30）
  力量基础：内力/真气、经络、丹田
  战斗特色：拳脚兵器、轻功身法、点穴、暗器
  对标深渊主宰：1-5级，盗贼的潜行背刺

第二层：高武（中期·ch30-约ch100）

```

实际卷一卷纲：
```
卷起点（ch1）：后天·开脉未成
卷终点（ch50）：开脉圆满，传承第一层觉醒
```

——**整整 50 章的跨度，连 PRD 承诺的"武侠层"都没走完。** PRD 说 ch1-ch30 是武侠层（意味着 30 章内要从开脉走到先天顶峰/宗师门口），但实际 50 章才到开脉圆满（武侠层最低段）。

再对比《深渊主宰》的原文节奏（s1 对标）：
| 对标项 | 深渊主宰·索伦 | 东方版·江轩 |

| 初始等级 | 10 级平民 | 开脉未成（对标 1 级不到） |
| 第一次质变 | 约 40 章（达成二阶） | **ch50（才开脉圆满，离宗师还差 4 阶）** |
| 前中期跨度 | 10 级→15 级（1-5 级约 40 章） | 0 级→1 级用了 50 章 |

**Skill 层根因：**

`pop-writer-world v2.0.0` Phase 7 动态升级表只定义了：

```yaml
# 当前 Phase 7 产出格式（节选）
全书跃迁路径：开脉未成 → 开脉圆满 → 洗髓 → 明劲 → 暗劲 → 宗师 → ...
每次跃迁：触发条件、社会地位重置、代价
```

但它**没有定义**：
```yaml
# 缺失的关键信息
per-volume-span:  # 每卷必须覆盖的能级跨度
  volume-1: 开脉未成 → 洗髓（目标：2 阶跨度，而非 0.5 阶）
  # 门禁：单卷能级跨度小于 1 阶 → 退回 creative 协商节奏
  
escrow_guard:  # 卷级防崩
  max_chapters_per_stage: 15  # 同一小阶不超过 15 章
  # 开脉未成→开脉圆满花了 50 章 = 违规
```

**缺失设计：**

```yaml
# 应在 pop-writer-world → pop-writer-plot 的传递协议中增加
rhythm-transfer-contract:
  # world 产出
  per-volume-pace:
    vol-1: {start: 开脉未成, end: 洗髓以上, chapters: 50}

  # plot 接收时必须验证
  gate: |
    如果 plot 设计的单卷跨度 < world 动态升级表要求的 60%：

    → 标记为「PRD 节奏偏离」
```

### Part C · 执行层（"Skill 加载机制缺陷"）—— 1 个根因

> 跨于 Part A（结构层）和 Part B（体验层）之间的元问题。它解释了为什么 Part A 和 Part B 的根因在本次执行中全部爆了：**agent 没有加载完整的 Skill 指令就推进了管线。**

### Gap⑮ 子 Skill 步骤链无加载门禁 —— agent 只加载 step-1 就完成了整个 plot 阶段

> **根因修正（2026-06-23）：** 初始版本认为"agent 找不到路标"。精确追溯后确认——agent **完整加载了 SKILL.md，看到了速查表中的所有文件路径和步骤名，但仍然选择不加载**。所以这不是路径可见性问题，是 agent 的行为模型问题。

**问题定性：** SKILL.md 的速查表包含"读什么文件"和"产出"两列。agent 的行为模式是：**只看"产出"列决定自己要写什么、不看"读什么文件"列去加载具体指令和模板**。结果：知道要产出"剧情线文档"，但不知道剧情线文档必须包含 17 个模块——因为模板从未被加载。

**精确证据链（源自消息 #36→#37 的工具调用日志）：**

```
第1步: 加载 SKILL.md（完整）→ 速查表进入上下文 ✅
       速查表列出：
         | 第3步 | steps/step-3-plotline-docs.md + templates/plotline-doc.md | 剧情线文档 | ... |
         | 第4步 | steps/step-4-act-plan.md | 幕纲·分幕层 | ... |
         | 第5步 | steps/step-5-chapter-anchors.md | 章锚点与Canvas | ... |
         | 第6步 | steps/step-6-canvas-audit.md | Canvas验核报告 | ... |
         | Step3 | templates/plotline-doc.md | 剧情线文档 | |
         | 4/5/6 | templates/act-outline.md | 幕纲 | |
         | 5/6 | references/payoff-design-guide.md | payoff规则 | |

第2步: 加载 step-1-volume-strategy.md（完整）✅
第3步: 加载 templates/volume-outline.md（完整）✅
第4步: — (steps/step-2~6)           ❌ 未加载
第5步: — (templates/plotline-doc)  ❌ 未加载
第6步: — (templates/act-outline)   ❌ 未加载
第7步: — (references/payoff-*)     ❌ 未加载
第8步: 直接 write 卷纲、剧情线、act-01 ✅

→ agent 的决策：看到"产出=剧情线文档"→ 径自用自己的理解写剧情线。
   "读什么文件"列在速查表中明明白白，但 agent 没有执行。
```

**不是"找不到路标"——路标就在速查表里，agent 选择了不看。**

同样的模式发生在其他阶段的速查表处理中：

```
prose 阶段（消息 #46）：
  SKILL.md 速查表：
    | 第1步 | steps/step-1-render.md | 正文初稿 | ...
    | 第2步 | steps/step-2-verify-output.md | 验证通过→正文 | ...
  实际加载：
    ✅ step-1-render.md（完整）
    ❌ step-2-verify-output.md（未加载）
  后果：ch01 正文写完后没有经过验证步骤就直接交付
```

**为什么 world 阶段没有出这个问题？**

对照 world 的 SKILL.md 写法——它没有依赖速查表来驱动 agent，而是在**核心流程部分直接把每步的指令内联写出来了**：

```markdown
# 收到任务后，按顺序执行

### Phase 0：融合适配
**读什么：** creative 的 PRD.md、故事引擎.md、素材储备池.md...
**做什么：** 逐维回答"融合决策对本 L1 维度的影响是什么"
**产出：** 小说世界设定/融合适配清单.md
**门禁：** 全部 8 个维度标注完成前不进 Phase 1

### Phase 1：L1 设定层推演
...
```

速查表是参考索引，核心逻辑在正文中。agent 读完全文就知道每一步该做什么。

而 plot 的 SKILL.md 把核心逻辑放在了**独立的 step 文件**中，SKILL.md 本身只是索引：

```markdown
### steps/ — 执行层

| 什么时候 | 读什么文件 | 产出 | 门禁 |
|:---------|:----------|:-----|:-----|
| 第1步 | steps/step-1-volume-strategy.md | 卷纲 | ❌ ... |
| 第3步 | steps/step-3-plotline-docs.md | 剧情线文档 | ❌ ... |
```

**当 agent 读完 SKILL.md 准备行动时，它的大脑是这样工作的：**
1. "第1步要产出卷纲" → 加载 step-1 看看怎么写 ✅
2. "第3步要产出剧情线文档" → **"我知道剧情线是什么，直接写"** ❌
3. 它不认为"读什么文件"列是一个需要执行的指令——那列是参考信息，不是命令

**对比 world 的写法：**
1. "Phase 0 读 creative 的 PRD.md + 故事引擎.md" → **"这是命令，去加载"** ✅

**Skill 层根因（修正版）：**

```
根因 A：速查表格式对 agent 不够"指令式"
  人类读到「产出=剧情线文档」会问"那我要用什么模板？"→ 翻速查表
  Agent 读到「产出=剧情线文档」→ 直接用自己的理解干
  → 修复：在速查表的"读什么文件"列加指令性前缀，如：
    「必须先加载 steps/step-3-plotline-docs.md 再写」

根因 B：核心逻辑放到独立 step 文件 = 需要 agent 自愿多加载一步
  world 把每步指令内联在 SKILL.md 中 → agent 读 SKILL.md 时全部拿到
  plot 把每步指令外置到 step-{N}.md → agent 需要主动去加载
  → 修复：在 step-1 末尾增加强制门禁，命令加载 step-2
    并在速查表的名词后加指令性语言

根因 C：没有任何机制检查"速查表的文件是否真的被加载了"
  → 修复：增加产出后审计，对比速查表中的文件路径
    vs agent 上下文中出现的路径
```

**修正后的 Gap⑮ 等级：** 仍为 **P0**——它解释了 Part A 和 Part B 所有根因的共同执行失败模式：**速查表写的是参考信息，agent 当参考信息处理。需要它执行时，必须把指令内联在 body 里或加显式门禁。**
