**之后 PRD 没有再被重新审视或修改。** 整个 54 轮对话中没有一轮是 agent 主动加载 PRD 来验证后续产出是否对齐的。

### 产出物

- `全书立项PRD.md` v1.0 ✅ — 10 块结构齐全
- 但 PRD 的爽点体系用的是中后期术语（"果位晋升 30%""洞天经营 20%"），非元爽点（"变强""探索解谜"）

### Gaps

| # | Gap | 证据 |
|:-:|:----|:------|
| G-C1 | **PRD 爽点写在了过低的抽象层级** | 「果位晋升」是中期事件名，「变强」才是元爽点 |
| G-C2 | **PRD 产出后从未被回溯校验** | 后续 world/plot/prose 阶段没有一次"加载 PRD 检查对齐"的工具调用 |
| G-C3 | **PRD 的"阶段承诺"未传递到下游** | PRD 说「武侠层 ch1-ch30」，但 plot 设计了 50 章武侠最低段——但没有任何环节检查过这个偏离 |

### 根因

1. `pop-writer-creative` 的 `templates/prd-模板-空白.md` 用具体事件名（"果位晋升""洞天经营"）而非元爽点（"变强""探索解谜"），导致 PRD 的爽点体系对卷一不可操作。
2. **没有任何 skill 要求在进入 plot 阶段前重新加载 PRD 进行一致性校验。**

### 优化建议

```yaml
# 修正 PRD 模板
prd-payoff-template:
  instead_of: ["果位晋升 30%", "洞天经营 20%"]
  use: |
    | 元爽点 | 占比 | 卷一落地 | 卷三落地 |
    | 变强 | 30% | 境界突破 | 果位碎片 |
    | 经营建设 | 20% | (卷二起) | 洞天道场 |
    | 权力博弈 | 25% | 帮会暗流 | 神系拉锯 |
    | 探索解谜 | 15% | 遗迹发现 | 封神榜下落 |
    | 危机合作 | 10% | (卷三起) | 神系联手 |
    
# 在 pop-writer-world 和 pop-writer-plot 的入口步骤中增加
step-0-reload-prd:
  action: 进入本阶段前，重新加载 PRD.md，提取与本书卷对应的承诺
  gate: 如果本卷设计与 PRD 的阶段承诺偏差 > 30%，终止并提示
```

## 2. pop-writer-reservoir

### Skill 设计

| 条目 | 内容 |
|:-----|:------|
| 版本 | v3.1.0 |
| 定位 | 外部素材 → 剧情储备卡 / 设定储备卡 |
| 核心红线 | 不读 PRD 就评估；储备卡缺元卖点；未查 trope-library |
| 模式 | 被 creative 调用 / 独立调起 / 主动丰富 |

### 实际发生了什么

本项目在进入 agent 对话前，**素材储备池已经存在**（7 张神系素材卡 + 1 张设计原则卡）。这是 pre-existing 资产，不是 agent 在本会话中产出的。项目开始时的状态：

```
素材储备池/00-三层结构-设计原则.md
素材储备池/01~06-神系素材卡（山海古神/楚巫神系/封神正统/天师道系/藏传密教/民间神祇）
```

整个 54 轮对话中，**没有一轮是 reservoir 阶段**。Agent 在消息 #1 检查到素材储备池已存在后，直接跳过 reservoir 进入 world。

### 加载情况

| 文件 | 加载？ | 备注 |
|:-----|:------:|:-----|
| pop-writer-reservoir/SKILL.md | ? 未检测到主动加载 | 因为 reservoir 阶段被跳过 |
| 素材储备卡 01-06 | ⚠️ 仅有初始检查时的列表读取 | world 阶段 L1-05 势力格局写作时引用了素材卡内容 |

### Gaps

| # | Gap | 证据 |
|:-:|:----|:------|
| G-R1 | **reservoir 阶段被完全跳过** | 消息 #1 中 agent 发现 reservoir "已完成"后直接推进到 world。没有验证储备卡是否完整、质量是否达标 |
| G-R2 | **skill 的"被 creative 调用"模式未触发** | 按 skill 设计，creative 产出 PRD 后应自动触发 reservoir（读 PRD→逐条素材产出储备卡）。本项目未触发 |

### 根因

`expert-writer` 的管线调度器假设 reservoir 阶段如果是"已完成"状态就直接跳过。但 **reservoir 本应是 creative 后的第一个阶段，它的职责是"把 PRD 的创意决定转换为可消费的储备卡"**。如果 PRD 是预先写好的（不在本会话中），reservoir 应该被触发来"验证储备卡与 PRD 的对齐度"——但没有任何 skill 定义了这种"存量资产验证模式"。

### 优化建议

```yaml
# expert-writer step-1-think.md 应增加
存量阶段验证模式:
  触发条件: 管线的某个阶段标记为 "已完成" 但不在本会话中执行
  动作: |
    1. 加载该阶段 SKILL.md 的质量红线和落盘检查点
    2. 逐项检查现有产出物是否满足
    3. 输出验证报告
  门禁: 检查不通过 → 退回重做，不得直接推进到下一阶段
```

## 3. pop-writer-world

### Skill 设计

`pop-writer-world v2.0.0` 是这个项目中**执行得最好的 skill**。它的设计特点是：

- **核心逻辑内联在 SKILL.md 正文中**（非速查表驱动）
- 从 Phase 0 到 Phase 7，每步都写清了"读什么/做什么/产出/门禁"
- 大量 WRONG 示例（14 条），覆盖了常见偏差
- 速查表是参考索引，不是执行指令 —— 实际执行靠正文中的顺序指令

### Gaps

| # | Gap | 证据 |
|:-:|:----|:------|
| G-W1 | **PRD 的神格阶梯三级制未落地** | PRD §三 定义了六大神系的弱小/中等/强大神力命名对应表，但 L1-02 力量体系完全没有体现这条——它只描述了修行境界，没有神系果位命名对应 |
| G-W2 | **L1-02 迭代效率低下（7 轮）** | Agent 先给了 3 个花哨命名方案（凶兽系/兵戈系/身劫系），用户否了→搜参考→再提案→用户再简化，共 7 轮才定稿。如果 agent 在第一轮就问"你想要传统武侠命名还是花哨原创命名？"，可以省 80% 的来回 |
| G-W3 | **动态升级表缺少"每卷必须覆盖的能级跨度"约束** | Phase 7 动态升级表定义了全书跃迁路径，但没定义"每卷至少 X 阶跨度"。导致 plot 阶段卷一只用半阶跨度通过了 gate |

### 根因

1. **world 阶段工作好的原因**：SKILL.md 把指令内联在正文里，agent 读完就知道每一步要做什么。不需要自己判断"该加载哪个 step 文件"。
2. **迭代效率问题**：`references/power-system-naming.md` 提供了"多方案提案法"和"双概念融合法"，但 agent 在第一轮没有先问用户的命名偏好（传统 vs 原创），而是直接编造 3 个花哨方案——这违反了 pop-writer-world 自己的红线 #13：「质感层直接照搬类型标配命名=零原创性」，但 agent 的反向错误是"过度原创，不看用户想要传统命名"。
3. **神格阶梯遗漏**：`pop-writer-world` 的 Phase 4 数值体系部分要求"从 L1 力量体系推导"，但 PRD 的神格阶梯属于"神明体系"而非"修行体系"，L1-02 只写了修行境界，没有写神系果位——两个体系被割裂了。

### 优化建议

```yaml
# pop-writer-world 在 Phase 0 融合适配中增加
prd-section-consistency-check:
  sections_to_verify: [§三 神格阶梯, §六 力量体系, §九 爽点体系]
  action: |
    检查 PRD 每一节是否在对应的 L1 文件中有了落地
    如 §三 神格阶梯 → 应在 L1-02 力量体系中有 "各神系果位命名与修行境界的对应关系"
    如 §九 爽点体系 → 应在动态升级表中转化为每卷的能级跨度约束

# L1-02 机制段前增加命名偏好征询
step-before-naming:
  ask: "你更倾向传统武侠命名（开脉/洗髓/明劲...）还是原创抽象命名（械/裂/噬...）？"
  gate: 用户未确认命名方向前不进入命名段
```
