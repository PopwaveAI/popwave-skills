## 二、根因分析

### 根因 1（P0）：子Agent结果捕获失败 → 连锁导致无自动续推

**严重度：P0**

#### 2.1.1 核心故障：结果捕获 100% 失败

每一批子Agent都返回 **"未能读取到子Agent最终文本"**。从 batch-1 到 batch-21，**100% 复现**。

实际文件全部写盘成功（61个设计包 + 135个套路文件都在磁盘上），但 orchestrator 无法读取子Agent的结束文本。

#### 2.1.2 连锁效应：结果捕获失败 → 无法确认完成 → turn 终止 → 无自动续推

**这不是两个独立问题，而是一条因果链：**

```
正常路径（如果结果捕获成功）：
  spawn 子Agent → 收到 "completed" + 产出摘要 → 确认成功 → 自动 spawn 下一批 → 继续

实际路径（结果捕获失败）：
  spawn 子Agent → 收到 "未能读取到子Agent最终文本"
  → 无法确认子Agent是否真正完成
  → 防御性行为：生成进度报告（"X批全接，已完成Y章"）
  → turn 结束
  → 不会自动启动下一批
  → 等用户 push "继续"
```

orchestrator 拿不到完成信号，自然不敢自动往下跑——它不知道上一批是成功了还是失败了，只能生成一个"我做了X，你看看"的响应然后停下来。**"无自动续推"不是独立的架构缺陷，而是结果捕获失败的直接下游后果。**

50批 × 3章/批 ÷ 3并发 = 17轮，每轮都需要用户手动push。

#### 2.1.3 Skill 层缺失：无降级策略

`pop-decon-design-pack/references/batch-scaling.md` 设计了批量编排协议，但未定义"子Agent结果捕获失败时的降级策略"。

`batch-scaling.md` 设计的循环逻辑：

```python
for round_batches in rounds_of_3:
    results = delegate_task(tasks=[{b} for b in round_batches])
    # 检查全部 completed，失败批次退回重跑
```

这个循环假设 `delegate_task` 能返回有效的 `results`。当 results 全部是"未能读取"时，"检查全部 completed"这一步无法执行——agent 退化为"生成报告 → 等待"模式。

**orchestrator 应当在收到"未能读取"结果后，直接用文件系统校验（`Get-ChildItem` 检查文件是否落盘）替代文本结果捕获。** 文件在磁盘上 = 子Agent完成了工作，不管文本结果是否能读到。

#### 2.1.4 修复方案

```yaml
# pop-decon SKILL.md + batch-scaling.md 应增加的降级 + 续推协议
result_capture_fallback:
  trigger: 收到 "未能读取到子Agent最终文本"
  action: |
    1. 不依赖文本结果判断完成状态
    2. 直接用 Get-ChildItem 检查目标文件是否落盘
    3. 文件存在且非空 = 该章成功完成
    4. 文件缺失 = 标记失败，退回重跑

batch_continuation:
  rule: |
    批量拆解任务中，单次用户消息内必须完成全部批次。
    每批子Agent返回后（无论文本结果是否可读）：
      1. 用文件系统校验替代文本结果捕获
      2. 校验通过 → 立即 spawn 下一批（不生成中间响应）
      3. 校验失败 → 重跑失败批次
      4. 全部批次完成后才生成最终响应
  prohibition: |
    ❌ 禁止在批次间生成"等待用户确认"的响应
    ❌ 禁止在 sessions_yield 后结束 turn
    ❌ 禁止依赖子Agent文本结果作为唯一完成判据
```

### 根因 2（P1）：上下文窗口膨胀（放大因子）

**严重度：P1**

上下文膨胀不是中断的直接原因，但它**放大了根因 1 的影响**——当上下文接近满载时，模型更容易遗忘任务目标（不再尝试续推），输出质量下降，章节进度追踪丢失。

最后一次成功 run 的 `injectedHistoryTurns: 14`，意味着 14 轮对话历史被注入上下文。每轮包含：
- 子Agent spawn 详情（每个含完整 prompt ~2K tokens）
- 子Agent结果（即使"未能读取"，session 路径和错误信息仍消耗 tokens）
- orchestrator 的进度表格响应（含设计包/套路库的详细产出清单）

按 900K 上下文窗口计算，14 轮历史 + skill 文件 + 格式规范 + Wiki 骨架 ≈ 接近上限。

**与根因 1 的关联：** 如果根因 1 修复（文件校验替代文本捕获 + 自动续推），每轮不再生成进度报告响应，上下文膨胀问题会大幅缓解——从 17 轮历史降到 1 轮（全部批次在单次消息内完成）。

**与历史问题的关联：** 此问题与 `04-20260614-toolCall膨胀与框架改造-问题分析.md` 中记录的 toolCall payload 膨胀是同一类问题——子Agent的 spawn 详情和结果信息在历史中累积，挤占了上下文空间。

## 三、因果链总览

```
子Agent结果捕获 100% 失败（P0 根因）
    │
    ├──→ orchestrator 无法确认子Agent完成状态
    │       │
    │       └──→ 防御性行为：生成进度报告 → turn 结束 → 无自动续推
    │                                               │
    │                                               └──→ 用户必须手动 push 5 次
    │
    ├──→ 每轮生成进度报告 → 上下文膨胀（P1 放大因子）
    │       │
    │       └──→ 模型遗忘任务目标 → 更不可能自动续推
    │
    └──→ 无法自动校验产出质量
            │
            └──→ 依赖人工检查文件落盘
```

**关键洞察：** 如果结果捕获正常，orchestrator 拿到 "completed" 信号后会自然地 spawn 下一批——"无自动续推"根本不会成为一个问题。结果捕获失败是因，无自动续推是果。修复结果捕获（或用文件校验替代）就能同时解决两个问题。

## 五、修复方案

### 立即可做（恢复任务）

1. 在 Paopao 中重新发一条消息 "继续跑 ch062-ch150"，让 orchestrator 从断点续推
2. 或者直接在当前对话中继续，因为 61 个设计包已安全落盘

### 根治方案

| 修复项 | 做法 | 预期效果 | 优先级 |
|:-------|:-----|:---------|:------:|
| **文件校验替代文本捕获** | orchestrator 不依赖子Agent文本结果，改为 spawn 后直接 `Get-ChildItem` 检查文件是否落盘 | 绕过结果捕获 bug，获得可靠的完成信号 | P0 |
| **续推循环** | pop-decon SKILL.md 增加"单次用户消息内完成全部批次"的指令，文件校验通过后直接 spawn 下一批，禁止中间生成响应 | 消除手动 push 需求（根因 1 修复后自然实现） | P0 |
| **上下文瘦身** | 每轮结束后只保留进度计数（"已完成 N/150"），丢弃子Agent详细产出表 | 14轮历史从 ~200K 降到 ~20K（根因 1 修复后此问题大幅缓解） | P1 |
| **版本锁更新** | 更新 `config.json` 的 `lockedVersions` 和 `installed` 到实际版本 | 消除版本不一致风险 | P2 |

> **修复优先级逻辑：** 文件校验是根因修复——拿到可靠完成信号后，续推循环和上下文瘦身都自然跟进。不需要把三个修复看作独立工作项。
