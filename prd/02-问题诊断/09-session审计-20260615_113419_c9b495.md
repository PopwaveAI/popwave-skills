## 二、问题 1：产出摘要纪律不达标

**现象**：13 条助理消息 >500 chars，最大 1,748 chars。SKILL.md 红线要求写入文件后对话中只留 ≤200 字摘要。

## 三、问题 2：Design Package 跳过 — 已知 Bug 复现

**现象**：CH2 直接写正文，跳过了 `pop-writer-chapter` 的设计包步骤。用户当场发现并校正。

**证据链**：
```
msg_id=1384: 用户要求 CH1 正文 → prose-render
msg_id=1390: CH1 正文完成
msg_id=1394: CH2 正文直接写入（跳过 design-pack）
msg_id=1396: 用户 → "你跳过了 skill 要求先出设计包 后出正文的规则吗"
msg_id=1397: agent 承认并补救
```

**根因**：agent 认为"CH1 你已经确认过风格了，CH2 直接写省一道工序"。这正是 `PRD-Skill调用机制失效.md` 中描述的长会话惰性侵蚀——设计包→正文的强制管线在 CH2 被压缩。

**修复建议**：`pop-writer-prose` SKILL.md 中应增加红线：**正文输入源必须是 chXXX-设计包.md 文件，不得直接从 act-XX.yaml canvas 渲染**。当前规则只说"消费设计包"，没说"不消费设计包就禁止写正文"。

## 四、问题 3：act-01.yaml 三次重写 — 48KB 浪费

**现象**：act-01.yaml 被连续写了 3 遍，总计 48KB write payload。

```
msg_id=1332: 18,014 bytes → YAMLError（markdown 混入 YAML）
msg_id=1336: 14,218 bytes → markdown 代码块包裹 YAML
msg_id=1348: 16,055 bytes → 最终修复版
```

**问题**：Hermes 的 write_file 自带 YAML lint，第一次已经报了 `error`，但 agent 只修了一半（去掉了 markdown 注释但把 YAML 包在了 markdown 代码块里）。第二次没报 YAMLError，但结构不对，agent 自己发现后手动重写了第三次。

**根因**：YAML lint 只检查语法结构，不检查格式约定（YAML 文件中不应有 markdown 代码块包裹）。agent 也没有在 write 前做格式验证。

## 五、问题 4：文风DNA 加载过晚 — 先写后改

**现象**：CH1-3 正文在没有文风DNA（`pop-shared-dna`）的情况下产出，用户提供 `@file:深渊主宰.md` 后全部删除重写。

**证据**：
```

msg_id=1460: 用户 → "用这个 删除正文1~3章 用设计包重写产出"
→ CH1-3 全部删除并重写
```

**根因**：prose-render 的 SKILL.md 没有要求"写正文前检查文风DNA目录是否非空"。当前规则是"加载文风DNA（如有）"，不做强制检查。agent 降级使用默认风格写了 3 章，用户不满意。

**修复建议**：`pop-writer-prose` SKILL.md 增加红线：**正文渲染前必须检查 `写作资产/文风DNA/` 目录是否有内容。为空时输出警告，让用户选择：继续默认风格 / 先运行 pop-shared-dna 产出风格文件**。
