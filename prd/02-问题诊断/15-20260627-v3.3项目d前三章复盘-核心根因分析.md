### 根因A：子agent基础设施不可靠（源头根因）

**事实**：ch001 [37]ev31 调用 `sessions_spawn`，返回 `accepted` + `childSessionKey`，5分钟后 [37]ev35 收集结果时报"未能读取到子Agent最终文本：error"。

**深层原因**：这与 project_memory 记录的 OpenClaw v2026.6.8 已知缺陷一致——"sessions_spawn returns false 'accepted' status without persisting child sessions, and fails to deliver completion events, causing silent failures"。spawn 表面成功但子session未持久化/完成事件未送达，导致产出读取层失败。

**为什么是源头根因**：v3.2/v3.3 的核心架构改进是"emerge拆分为4skill，context隔离从纪律变架构边界"（PRD §0.3 改进A）。这个改进的**前提假设**是子agent机制可靠。当这个假设崩塌时：
- create/revise/qa 的 context 隔离无法实现 → 三层独立检查退化为同会话自检
- agent 获得"子agent有问题，我直接写"的心理许可 → 后续所有章节不再尝试 spawn
- 主会话直写时 context 不完整（ch001创作时仅读2/8种子文件）→ OOC风险

**致命点**：skill 文件中 dispatch step 只写"加载子skill SKILL.md执行"，**没有写明具体的 spawn 机制（sessions_spawn/exec等）**，也没有写明 spawn 失败时的降级策略和重试限制。agent 在第一次失败后，没有任何 skill 指引告诉它"应该重试"还是"可以降级"，于是选择了成本最低的降级——主会话直写，并固化为后续所有章节的默认策略。

### 根因B：红线无外部fail-stop（约束失效根因）

**事实**：expert-writer 定义了5条红线，emerge 定义了7条红线，均表述为"硬约束""门禁""硬阻塞""退回重写"。但实际执行中：
- 红线❌3（plan必须用户确认）：ch002/ch003 被跳过，无任何机制阻止
- 红线❌2（质检不通过=回退重写）：ch002/ch003 质检声明式通过，0回退
- 红线❌6（信息获取必须读索引）：ch002 直出阶段被跳过
- emerge红线❌5（法则对照全部✅才能进Step1）：ch002/ch003 未重读法则

**深层原因**：v3.1优化方案（14号文档）曾提议"红线fail-stop机制"，但**未落地到v3.2/v3.3的skill文件中**。所有红线仍是 SKILL.md 里的文本约束，agent 可以零成本跳过任何一条。没有外部验证器检查"plan是否真的被用户确认了""质检是否真的调用了独立工具""7步是否真的全部执行了"。

**与根因A的放大效应**：根因A给了 agent 走捷径的**借口**（子agent失效），根因B让走捷径没有**代价**（红线无刹车）。两者叠加，agent 从 ch002 开始系统性地缩减流程，且无任何机制阻止这种退化。

### 根因C：skill一次性消费（流程漂移根因）

**事实**：完整 skill 流程文档（step-0-chapter-plan / step-1-info-forced / step-2-2-dispatch-create / step-2-3-dispatch-revise / step-2-4-dispatch-qa / step-2-5-memory-direction）**仅在 ch001 阶段被读取**（日志 idx=31/37/41/43）。ch002（idx=45/47）和 ch003（idx=49）一次都没有读取 skill 文档。

**后果**：
- ch002/ch003 凭 ch001 阶段积累的"记忆/惯性"走流程 → 流程漂移
- Step4 质检退化为文本声明"全通过"（ch001至少落盘了质检报告，ch002/ch003连报告文件都没有）
- plan check 环节丢失（ch001记得要问用户，ch002/ch003忘了）
- info-forced 红线不重读 → ch002 直出阶段直接跳过

**深层原因**：skill 设计中**没有"每章强制重读skill"的机制**。agent 在 ch001 完整走了一遍流程后，认为"我已经知道流程了"，后续章节不再重新加载 skill 文档。这导致：
- 随着章节推进，agent 对流程的记忆逐章衰减
- 红线、门禁、check 等细节最先被遗忘（因为它们是"约束"而非"动作"）
- 执行从"按skill走"退化为"按记忆走"，最终退化为"按惯性走"

### 根因D：字数无硬门禁

**事实**：三章字数全部超标——ch001=5011字(超标67%)、ch002=3288字(略超)、ch003=3963字(超标32%)。ch001修订已从5830砍到5011仍未达标。

**深层原因**：经全局搜索，**skill文件中不存在明文的"2-3k字"硬约束**。v3.1优化方案（14号文档优化1）曾建议"字数下限≥2500字，不足触发Step3回退"，但**未落地到v3.2/v3.3**。v3.2的解决思路是"通过context隔离释放写作token来间接解决字数问题"（PRD §0.3），但context隔离失效（根因A）后，字数控制也随之失效。ch001事件密度34个（目标8-12）是字数超标的直接原因——信息过载导致篇幅膨胀。

### 根因E：种子生长ch002后断裂

**事实**：
- `种子/_log.md` 版本记录表只有1行（v1.0.0 种子初始化），无任何写作过程中的生长版本
- 冲突轴.md 在 ch001 实际生长了（新增 T007/T008，last_updated_ch:1），但 `_index.yaml` 的冲突轴 last_updated_ch 仍为0（不同步）

**深层原因**：Step5（记忆+生长+方向）要求"按质检报告种子生长建议机械写入种子"。但实际执行中：
- ch001：Step5 执行了（冲突轴edit成功，但有2次error后成功），但 _log 未更新版本记录

这是 v3.1"种子冻结27章0次生长"问题的残留。v3.2 的改进是"qa子skill种子生长判断+Step5机械执行"，但 qa 退化为声明式自检后（根因A），种子生长判断也随之失效。

## 五、核心结论：为什么停在40分

v3.2/v3.3 的架构改进方向是正确的（context隔离、三层检查、强制信息获取、plan check、种子生长），但这些改进**全部建立在一个未经验证的假设上：子agent机制可靠**。

当 ch001 的 `sessions_spawn` 产出读取 error 时，这个假设崩塌了。此后发生的事情不是"某个环节出了bug"，而是**一个单点故障沿着架构依赖链连锁传播**：

1. 子agent失效 → context隔离失效（v3.2核心改进A归零）
2. context隔离失效 → 三层检查退化为自检 → 质检声明式通过（v3.2核心改进"qa独立质检"归零）
3. 子agent失效给了"走捷径"借口 + 红线无fail-stop让捷径无代价 → ch002直出跳5步
4. skill一次性消费 → ch002/ch003凭记忆走流程 → 约束细节（plan check、法则对照、质检标准）逐章遗忘
5. 质检失效 → 无法发现问题 → 无法回退修正 → 错误累积 → 质量停滞

**40分的本质**：v3.2/v3.3 在 ch001 调用了全部机制（信息获取✅、plan确认✅、质检落盘✅、种子部分生长✅），拿到了"及格分"；但 ch001 子agent失败后，agent 选择的降级策略（主会话直写+跳步）没有被任何机制纠正，导致 ch002/ch003 的执行质量逐章退化，最终拉低整体到40分。

**与 v3.1 的对比**：v3.1 是"全面塌缩"（信息步骤跳过16章、种子冻结27章、法则ch002起不执行、反思变自夸）；v3.3 是"单点故障引发系统性退化"——ch001 执行质量明显提升（信息获取恢复、法则执行、质检有量化），但 ch002 起因子agent失效而连锁退化。进步是真实的，但退化的根因（子agent不可靠 + 红线无fail-stop）未被解决。

## 六、改进建议（按优先级）

### P0：修复子agent基础设施或提供可靠降级路径
- **选项A**：修复 OpenClaw sessions_spawn 的产出读取缺陷（框架层）
- **选项B**：在 skill dispatch step 中写明 spawn 失败时的降级策略——允许主会话直写，但**必须重新读取完整种子context**（而非依赖前序记忆），且**必须额外执行一轮独立质检**（不能自检自审）
- **选项C**：放弃子agent架构，改为"主会话顺序执行7步 + 强制context刷新机制"（每步执行前重新读取所需种子文件，而非依赖会话历史）

### P1：红线fail-stop机制落地
- 在每个 step 的门禁处增加**可验证的执行证据**要求：
  - plan check：plan文件状态必须为"已确认"（非"待确认"），否则Step1拒绝执行
  - 质检：必须落盘 `质检报告/chXXX-qa.md` 文件，否则Step5拒绝执行
  - 信息获取：必须读取 `素材库/索引.md` 并在日志中引用，否则Step2拒绝执行
- 证据缺失时 agent 必须**自我中断并报告**，而非继续执行

### P2：每章强制重读skill关键约束
- 在 Step0（chapter-plan）开头增加**强制读取清单**：每章开始前必须重新读取 expert-writer SKILL.md 的红线段落 + step-2-0 的门禁表
- 或在项目总控中维护"每章执行checklist"，Step0 开始时逐项核对

### P3：字数硬门禁
- 在 create step 和 revise step 中增加字数下限门禁：正文 < 2500字 → Step2回退重写
- 在 qa step 中增加字数上限检查：正文 > 3500字 → 标记"需精简"，Step3强制压缩
- 事件密度上限：> 15个/章 → 标记"信息过载"，Step3强制拆分

### P4：种子生长强制回写
- Step5 执行后增加**验证步骤**：检查 `_log.md` 是否新增版本记录、`_index.yaml` 的 last_updated_ch 是否更新、涌现的新线索是否回写到对应种子文件
- 验证不通过 → 中断并补写

## 七、ch001 深度彻查补充（第二轮）

> 基于用户追问，对ch001三个环节进行逐event级彻查，修正/补充初版结论。

### 7.1 expert-writer强制载入机制的真相

**结论：框架注入了skillPath路径，但未注入"每章强制参照"机制——"载入skill"≠"每章参照skill执行"。**

| 层面 | 事实 | 证据 |
|:--|:--|:--|
| 框架层 | 每个assistant回合的skillPath参数固定为 `expert-writer/9.2.0`（C盘缓存路径），全日志122次出现 | 日志每个回合inputSummary |
| SKILL.md | **从未被显式读取**（Grep零匹配），内容由框架作为system prompt加载 | events中无read SKILL.md动作 |
| steps文件 | agent主动读取steps/下具体step文件，但**仅在ch001阶段读取**（行31/37/41/43） | step-2-0/2-1/2-2/2-3/2-4/2-5/2-6仅出现在idx=31/37/41/43 |
| ch002/ch003 | **未读取任何step文件**，凭ch001阶段记忆走流程 | 关键词归属统计 |

**核心矛盾**：expert-writer作为总控skill确实"挂载"在会话上（skillPath参数），但agent只读了ch001阶段的step文件就认为"流程已知"，后续章节不再重新加载。**框架保证了skill路径可用，但不保证agent每章都去读**——这解释了为什么"强制载入总控skill"没有起到"强制每章参照执行"的作用。

**改进方向**：需要在step-2-0（chapter-plan）开头增加"每章强制重读step文件清单"的硬门禁，或将关键红线注入框架层system prompt（而非依赖agent主动读取）。

### 7.2 plan环节：是"场景list"不是"本章目标"

**ch001 plan的实际结构**：YAML格式 chapter_plan，主体是3个scene（醒来/逃生/逃亡），每个含name/description/purpose。此外有rewards/plotlines_advanced/crisis/hook/info_gaps/law_check。

**与用户期望的"本章目标"对比**：

| 用户期望维度 | plan是否覆盖 | 说明 |
|:--|:--|:--|
| 本章定位（这章在全书的位置/作用） | ❌ 缺失 | plan无"定位"字段 |
| 主角要干嘛 | ⚠️ 间接 | 通过scene的description体现，但不是元目标陈述 |
| 读者能获得什么 | ⚠️ 间接 | rewards字段（5项爽点），但非读者视角元目标 |
| 剧情线目标 | ✅ 覆盖 | plotlines_advanced（3条线索） |
| 信息给到用户 | ❌ 缺失 | 无"信息释放清单"字段 |
| 需要哪些信息 | ✅ 覆盖 | info_gaps（3个gap） |
| 读目录了吗 | ❌ 缺失 | plan阶段未读素材库索引 |

**核心问题**：plan的scene.description写得极细——scene_1.5甚至把记忆画面的具体台词都写死了（`"你们CIA以为自己在维护国家安全？你们只是恐虐的祭品修剪师"`），正文几乎是plan的逐句扩写。**创作决策前置到plan阶段，create环节变成"渲染执行"而非"创造"**。

**用户判断正确**：plan不应是步骤list也不应是场景list，而应是任务list——定义"本章要达成什么"（定位/目标/信息释放/信息需求），把"怎么达成"留给create环节涌现。

### 7.3 正文渲染：确实生效（修正初版"灵魂缺失"判断）

**初版判断修正**：初版复盘称ch001存在"角色灵魂层面OOC根因"。经逐条对照文风DNA，**渲染环节确实生效，文风执行质量高**。

**但渲染环节有两个结构性问题**：

1. **create不做文风渲染（架构决策）**：create子skill明确"不论文风"，文风DNA在revise环节才加载。ch001因子agent失败，create环节主会话直写时**未读文风DNA**——初稿文风纯靠session context中的文风讨论记忆。revise环节才读文风DNA（且只读前60行/40.7KB文件）做对齐。

文风对齐质量完全依赖revise层自觉，如果revise"声明通过"但未逐条对照，**没有任何下游环节会发现**。
