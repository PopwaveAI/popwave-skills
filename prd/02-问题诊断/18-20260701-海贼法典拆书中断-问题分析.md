# 海贼法典拆书中断：100章任务断在47章 — spawn-unmaterialized 大规模复发

> **问题诊断 · 2026-07-01 · Platform Engineering**
> **P0 · 拆书管线中断** | spawn-unmaterialized | fallback-coverage-gap | orchestrator-no-recovery
> 基于 project `海贼法典` 的 10 个 runs 数据分析

---

## 1. 问题现象

用户指令"拆完100章"，管线在 Phase 1（设计包提取）中段断流，Phase 2-4 全部未启动。

| 阶段 | 预期 | 实际 | 完成率 |
|:-----|:-----|:-----|:-------|
| 源章节拆分 | 100章 | 100章（`_temp/chapters/`） | 100% |
| Phase 1 设计包 | 100个 | **47个**（ch001-ch047） | **47%** |
| Phase 2 L2/卷纲 | ≥1份 | 0 | 0% |
| Phase 3 设定 | 1套 | 0 | 0% |
| Phase 4 PRD | 1份 | 0 | 0% |

用户连续催促两轮（"继续任务呀 跑完100张为止"、"你有子agent还在跑吗？你没跑完100章呀"），仍未能恢复。

## 2. 根因关联

根因与 [16-OpenClaw并发Spawn崩溃](16-20260630-OpenClaw并发Spawn崩溃-问题分析.md) 完全一致：

| 历史根因 | 本次表现 |
|:---------|:---------|
| RC-1: Gateway WebSocket 断裂 | 一致——`spawn-unmaterialized` |
| RC-2: sessions_spawn 虚假 accepted | 一致——accepted 后 session 未落盘 |

**不复现 16 号文档的触发条件**：16 号文档中并发 spawn 是在"同一秒内 spawn 2+"触发崩溃。本次 orchestrator 按批次 spawn（每批3章），batch 间有 yield 等待，并非同秒并发——但仍大面积失败。说明 **spawn-unmaterialized 不限于并发触发，长会话积累到一定 run 数后也会自发崩溃**。

## 3. 新发现（增量于 16 号文档）

### 3.1 Fallback 覆盖不全：部分失败批次无人补救

Paopao 的 fallback 机制（`paopao-background-task` 隐藏支线会话）在检测到 `spawn-unmaterialized` 后会启动补救。但本次 10 个 batch 中：

| 批次 | spawn 结果 | fallback 补救 | 最终产出 |
|:-----|:-----------|:-------------|:---------|
| batch 1-5 | ✅ 正常 materialized | 不需要 | ch001-ch015 ✅ |
| batch 6 | ⚠️ 部分失败 | 部分补救 | ch016-ch018 ⚠️ |
| batch 7-8 | ❌ unmaterialized | ✅ fallback 补救 | ch019-ch024 ✅ |
| batch 9-11 | ❌ unmaterialized | ❌ **未触发 fallback** | **缺口** |
| batch 12-13 | ❌ unmaterialized | ✅ fallback 补救 | ch025-ch030+ ✅ |
| batch 14-15+ | ❌ unmaterialized | ❌ **未触发 fallback** | **ch048-ch100 全缺** |

**关键发现**：fallback 机制本身不可靠——同一类型的失败，有的批次触发了 fallback，有的没有。batch 9-11 和 batch 14+ 的失败完全没有被补救，导致 53 章设计包永久缺失。

### 3.2 Orchestrator 无自动恢复机制：Phase 1 断流后管线死亡

pop-decon orchestrator 的 `delegate_task` 编排模式假设所有子 agent 都会成功返回。当 spawn 大面积失败时：

1. orchestrator 收到空结果或错误，但没有**断点续跑**逻辑
2. 用户催促后，系统重新跑了几轮到 ch047，但 spawn 持续失败
3. Phase 1 从未到达质量门禁（Step 2.5），Phase 2-4 因门禁阻塞全部未启动
4. 最终 run `e05dfa4d`（用户催促轮）连 response 都没产出——卡在 `[warning]` + `[tool]` 事件

**Skill 层缺陷**：orchestrator 缺少"检测已有产出 → 从缺口续跑"的断点恢复能力。每次重跑都从零开始或随机位置开始，而非从 ch048 精确续跑。

### 3.3 长管线对 spawn 失稳更敏感

| 维度 | 16号文档（6-30项目b） | 本次（海贼法典） |
|:-----|:---------------------|:----------------|
| 任务规模 | plot 环节，~5个 spawn | 100章拆书，~34个 spawn（15 batch × 2-3章） |
| 失败 spawn 数 | 2个 | ~20+个 |
| 补救覆盖 | 全部 fallback 成功 | **部分 fallback，多数未覆盖** |
| 管线影响 | 单环节中断 | **全管线死亡**（Phase 1 未完成 → 2/3/4 全阻） |

**结论**：spawn 失稳的破坏力随管线长度和 spawn 次数非线性增长。5个 spawn 时 fallback 能兜住，34个 spawn 时 fallback 覆盖率降至 ~40%。

## 4. 中断链路图

```
用户: "拆完100章"
  ↓
pop-decon orchestrator
  ├── batch 1-5 (ch001-ch015)     ✅ spawn 正常
  ├── batch 6 (ch016-ch018)       ⚠️ spawn 部分失败 → fallback 部分补救
  ├── batch 7-8 (ch019-ch024)     ❌ spawn 失败 → ✅ fallback 补救
  ├── batch 9-11                  ❌ spawn 失败 → ❌ fallback 未触发 → 缺口
  ├── batch 12-13 (ch025-ch030+)  ❌ spawn 失败 → ✅ fallback 补救
  ├── 后续零星产出                 ⚠️ 到 ch047 后停止
  ├── batch 14+ (ch048-ch100)     ❌ 完全未跑 → 53章设计包全缺
  ↓
Phase 1 质量门禁 (Step 2.5)        ❌ 覆盖率 47% < 95% → 门禁不通过
  ↓
Phase 2/3/4                       ❌ 全部阻塞 → 管线死亡
  ↓
用户催促两轮                       ❌ 仍无法恢复 → run e05dfa4d 无 response
```

## 5. 修复建议增量

### 5.1 Skill 层（pop-decon orchestrator）

| 修复项 | 优先级 | 说明 |
|:-------|:------:|:-----|
| **断点续跑** | P0 | orchestrator 启动 Phase 1 前先扫描 `设计包v4/` 已有文件，计算缺口，从缺失章节续跑而非从零开始 |
| **spawn 失败重试** | P0 | 单 batch spawn-unmaterialized 时，orchestrator 自行重试 1-2 次再降级，不依赖宿主 fallback |
| **并行度降级** | P1 | 检测到连续 spawn 失败时，自动从 5 batch/轮 降级到 2-3 batch/轮，降低 gateway 压力 |
| **Phase 1 完成度自检** | P1 | 每轮 yield 后检查 `设计包v4/` 文件数是否增长，若无增长则判定 spawn 系统不可用，告知用户而非静默继续 |

### 5.2 框架层（OpenClaw / Paopao）

| 修复项 | 优先级 | 说明 |
|:-------|:------:|:-----|
| **Fallback 全覆盖** | P0 | 每个 `spawn-unmaterialized` 都必须触发 fallback，不能部分触发部分跳过 |
| **Fallback 结果校验** | P1 | fallback 补救后校验目标文件是否真正落盘，落盘失败则标记为硬失败并通知用户 |
| **Materialization 超时自适应** | P2 | 10秒固定窗口太短，复杂 spawn 任务应按 task payload 大小动态调整超时 |
