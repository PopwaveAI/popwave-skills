## 十、SKILL.md 写作规范

### 10.1 总原则

| # | 规则 |
|:-:|:-----|
| 1 | **写给 agent 看，不是给人看。** 每一行回答一个问题："agent 看到这段该怎么执行？" |
| 2 | **禁令式，不建议式。** "不要跳过 Phase 2" → 不是"Phase 2 建议在 Phase 1 之后执行" |
| 3 | **具体到文件路径。** "读 act-XX.yaml" → 不是"读取幕纲文件" |
| 4 | **每段有判定标准。** "事件数 ≥ 章字数 ÷ 200" → 不是"事件要足够多" |
| 5 | **冷启动友好。** agent 没有上一轮的上下文。SKILL.md 是从零开始的唯一指令 |
| 6 | **不出现在正文里写"本 Step 的目的是…"** — agent 需要知道做什么、读什么、怎么判定好坏，不需要目的描述 |
| 7 | **不超过 200 行。** 详细执行逻辑下放到 steps/ 按需加载 |

### 10.2 各段写作规格

#### 首屏：标题 + 一句话定位 + 约束

```
# {标题} vX.Y.Z

> **定位：从 X 到 Y。** 一句话说清输入→输出。
> **核心约束：不碰 Z。** 如果有越界风险，在首屏亮出来。
```

**真实案例**（取自 `pop-writer-chapter` SKILL.md 第 13 行）：

```
> ✅ 核心约束：不碰文风。不知道文风DNA的存在。不写叙事者声音。
```

这条立即告诉 agent：如果它尝试在本步处理文风就是越界。

**反例**（虚构）：

```
> ❌ chapter-design 是写作管线中的一环……（一段自我定位）
```

自我定位放进 `references/pipeline-arch.md`，不在 SKILL.md 首屏。

---

#### 速查表：三种格式按 skill 类型选择

| skill 类型 | 用什么列 |
|:----------|:---------|
| **产文件型**（plot, deconstructor, world） | `步骤 \| 操作 \| 读什么 \| 产出什么 \| 门禁`（pop-dna 格式） |
| **路由型**（expert-writer） | `意图 \| 路由到 \| 前置条件` |
| **工具型**（工具 skill） | `我要 \| 操作 \| 前置条件`（chapter-design 格式） |

**真实案例**（pop-dna 格式，取自 `pop-shared-dna` SKILL.md 第 18-24 行）：

```
| 步骤 | 操作 | 读什么 | 产出什么 | 门禁 |
|:-----|:-----|:-------|:---------|:-----|
| 步骤0 | 取样 | 全书扫描 | 采样清单（≥30章，≥20,000行） | ❌ 不够退回 |
| 步骤1 | 逐章精读 | 采样章全文 | 场景类型标记 + 候选原文段 | — |
```

**门的写法**：`❌ 不够退回`（不是"采样不满足条件时退回"）。

---

#### 质量红线：先判定、后行动

每条红线格式：
```
❌{序号} | **{什么情况}** — {具体判定标准}。{不满足时的动作} | {怎么验证}
```

**真实案例**（取自 `pop-writer-chapter` SKILL.md 第 29-38 行，部分）：

```
- [ ] Canvas 就绪 — 设计/卷/volume-XX.md + 设计/幕/act-XX.yaml 全部存在
- [ ] 所有出场角色可追溯 — 每个角色在 volume-XX.md §三（角色池）中存在
- [ ] ★ 事件数 ≥ 章字数 ÷ 200 — 硬性下限
```

**升级到 v5 后应改为**：

| # | 红线 |
|:-:|:-----|
| ❌1 | **Canvas 未就绪** — `设计/卷/volume-XX.md` 或 `设计/幕/act-XX.yaml` 任一不存在 → 终止，提示用户先完成 plot |
| ❌2 | **出场角色不可追溯** — 事件链中的角色在 `volume-XX.md §三` 角色池中找不到 → 标注越界，退回修正 |
| ❌3 | **事件密度不达标** — 事件链事件数 < 章字数 ÷ 200，硬性下限 → 退回扩充事件 |

**红线的铁律**：
1. 每个红线有唯一的数字序号
2. 每个红线有判定条件（"什么情况下触犯"）+ 动作（"触犯后怎么办"）
3. 红线是绝对禁令，不是质量建议。不设 P0/P1/P2 分层（分层放在步骤内）

---

#### 核心流程段：每步 I/O + 指向 steps/

```
### 步骤 1：{步骤名}

**读什么：** {具体文件路径}
**做什么：** {祈使句描述动作}
**产出：** {产出文件名和路径}
**❌ 门禁：** {不通过的条件和处理}

详细指令见 `steps/step-1-xxx.md`。
```

**案例**（取自 `pop-shared-dna` SKILL.md 第 55-65 行）：

```
### 步骤0：取样
**读什么：** 全书（扫描章节目录 + 确认章节边界和全卷分布）。
**做什么：** 均匀采样——前 20% + 中段 40-60% + 后 20%，每段 ≥10 章
**产出：** 采样清单（章号列表 + 每章行数统计）
**❌ 门禁：** 不到 20,000 行 → 退回，继续读更多章
```

**不该出现的**：

```
❌ 步骤 1 的目的是建立本章的设计基线，包括角色池、地点池...
```

"目的是"是给人写的描述——agent 不需要目的，它需要**读什么文件、做什么判断**。

---

#### WRONG 示例段

每条 4 行：
```
> ❌ agent 实际会犯的错误做法
> ✅ 正确的替代做法
```

**必须基于真实 agent 出错记录**，不能编造。来源：changelog、用户反馈、自己测试时观察到的问题。

**案例**（取自 deconstructor v11.1 的 ETL 管道经验）：

```
> ❌ Phase S 写完"16岁、弯刀大师专长"但未先运行 extract.py
> ✅ 先跑 extract.py → 从 _temp/baseline-data.json 确认年龄"原文未显式说明"→ 从 world-data.json#class 确认职业路径
```

---

#### 边界条件表：覆盖至少 6 种异常

每个 skill 必须有一张边界条件表，逐条列举 agent 执行时可能遇到的异常。格式：

```markdown
## 边界条件

| 场景 | 触发条件 | 处理 |
|:-----|:---------|:-----|
| {agent 会遇到的异常} | {什么情况下触发} | {怎么做，不超过一行} |
```

**6 种必须覆盖的场景维度**：
1. 前置文件缺失 — "上游 skill 没产出依赖文件"
2. 数据不足 — "JSON 条目不够做出判断"
3. 格式不匹配 — "产出物类型与模板不符"
4. 循环依赖 — "A 依赖 B、B 依赖 A"
5. 用户中断 — "用户在中途改变了需求"
6. 子 skill 不可用 — "路由目标无法加载"

---

#### 落盘检查点

用 checklist 形式，agent 逐条勾选：

```markdown
## 落盘检查点

| 确认项 | 状态 |
|:-------|:----:|
| `设计/卷/volume-01.md` 已写入 | [ ] |
| `设计/幕/vol-01/act-01.yaml` 已写入 | [ ] |
| ... | [ ] |
```

每个产出文件一行。

**不该出现生成后的内容粘贴。** 落盘点只列路径。

---

### 10.3 不允许出现的内容

| 不应该出现在 SKILL.md | 该出现在哪 |
|:---------------------|:----------|
| > 3 行的版本历史 | `CHANGELOG.md` |
| 管线架构图 | `references/pipeline-arch.md` |
| 设计方法论（如"爽点有四级…"） | `references/` 下的独立文件 |
| 赛道差异说明 | `references/` 下的独立文件 |
| 模板字段的详细解释 | `templates/` 内嵌注释 |
| "本 skill 的目的是…" 自然语言段落 | 不需要——agent 从速查表 + 红线 + I/O 标注已经能理解 |

**例外**：专家 skill（expert-writer）的 SKILL.md 可以保留管线图段——因为专家的身份就是调度，不是执行。

## 十三、附录 B：pop-dna v4.0 — 完整批注版

以下是一个符合 v5 规范的真实 skill，逐段加批注说明"为什么这样写"。

```
# pop-dna · 文风DNA蒸馏引擎 v4.0                           ← 标题格式: {id} · {副标题} v{版本}

> **定位：从原文中按场景类型提取作者笔触。产出 prose-render 旁读原文。** ← 一行说清: 输入是什么、输出是什么
```

**批注**：首屏两行搞定。没有版本差异表（挪到 CHANGELOG）、没有管线说明（在 skill.json#pipeline 里）。

```
## ❌ 质量红线                                        ← 第一屏，六个字标题
| # | 红线 |                                          ← 表格 → agent 可以逐行扫描
|:-:|:-----|
| ❌1 | 原文不足退回 — 任何场景卡原文 < 500 字 → 退回扩采  ← 判定条件 + 动作
| ❌2 | 场景覆盖不足 — 场景卡 < 6 个 → 退回读更多章       ← 每个红线有数字
| ❌3 | 不接受降级替换 — 战斗遭遇战的原文内容是行军队列 → 退回读更多章 ← "差不多"不被允许
| ❌4 | 产出只留摘要 — 写入 styles/ 后不粘贴完整文件      ← 落地纪律
```

**批注**：四条红线。每条有"什么情况"+"怎么做"。不是"P0/P1/P2 分层"。

```
## 速查表                                             ← 生产型格式: |步骤|操作|读什么|产出|门禁|
| 步骤 | 操作 | 读什么 | 产出什么 | 门禁 |
|:-----|:-----|:-------|:---------|:-----|
| 步骤0 | 取样 | 全书扫描 | 采样清单（≥30章，≥20,000行） | ❌ 不够退回 |
| 步骤1 | 逐章精读 | 采样章全文 | 场景类型标记 + 候选原文段 | — |
| 步骤2 | 全书验证 | 全书扫描 | 验证过的判断 | — |
| 步骤3 | 写文件 | 步骤1+2 产出 | `styles/{书名}.md`（20-30K） | ❌ 维度不达标退回 |
| 步骤4 | 试写验证 | 风格文件 | 2 组 300 字对比稿 | — |
```

**批注**：5 行速查，agent 不出 SKILL.md 就能知道全流程。

```
## 核心流程                                           ← 每步一个子标题 + 指向 steps/
### 步骤0：取样
详细指令 → `steps/step-0-sampling.md`

### 步骤1：逐章精读
详细指令 → `steps/step-1-close-reading.md`

### 步骤2：全书搜索验证
详细指令 → `steps/step-2-verify.md`

### 步骤3：写风格文件（★ 核心）
详细指令 → `steps/step-3-write.md`

### 步骤4：试写验证
详细指令 → `steps/step-4-trial.md`
```

**批注**：每步一行 + 一条文件路径。不需要解释"这步做什么"——速查表已经说了。

```
## ❌ WRONG 示例
> ❌ 战斗原文不足 500 字 → 用行军队列描述代替
> ✅ 退回采样，读更多章直到找到真正的战斗场景原文

> ❌ 产出的分析报告说"作者喜欢用短句"
> ✅ 产出原文段落。让 prose-render 对照原文感受，不需要替它总结

> ❌ 场景卡原文来自不同类型的战斗 → "早期遭遇战"原文实际是 boss 战
> ✅ 严格匹配——设计包 scene 字段对应什么，DNA 场景卡就用什么原文
```

**批注**：3 条，都是真实遇到过的 agent 错误。不是编造的"可能犯错"。

```
## 边界条件
| 场景 | 触发条件 | 处理 |
|:-----|:---------|:-----|
| 采样章不够 | 全书 < 30 章 | 全部采样，标注"短篇" |
| 某场景类型无原文 | 全书扫描未找到该场景 | 标注"此场景类型未出现在原文中"，不做填充 |
| 风格文件超过 30K | 场景卡累计超过 | 删减原文长度但保持每卡 ≥ 500 字 |
| 试写验证不通过 | 对比稿与原文风格偏差 > 2 处 | 退回步骤 3 修正对应场景卡 |
| 用户中途改变场景列表 | 用户说"加一个战斗·水下场景" | 重新跑步骤 2-4 |
| 子 agent 不可用 | — | 主 agent 手动执行，遵守同上红线 |
```

**批注**：6 种异常。精确覆盖 10.2 要求的六个维度。
