# Hermes Agent vs OpenClaw：上下文管理代码级深度对比

> 报告日期：2026-06-10
> 数据来源：GitHub API 直读源码 + OpenClaw 架构拆解系列（经源码验证）
>
> 整理注：本文为技术选型前期的原始调查材料。最终决策结果见「为什么选择 Hermes Agent 作为 Popwave 技术底层」。

---

## 核心发现

**从代码层面看，上下文管理是这两个项目差距最大的模块。**

OpenClaw 在上下文管理上是一套「精密的装配线」——多个独立的守卫函数、策略接口、生命周期钩子围绕 Gateway 调度器工作。Hermes Agent 是一套「自适应的循环系统」——可插拔的 ContextEngine 抽象 + 有状态的记忆管理器 + 严格的 token 预算控制。

**一句话：OpenClaw 的上下文管理管得细，Hermes 的上下文管理管得深。**

---

## 一、Hermes Agent 上下文管理

### 1.1 上下文引擎：可插拔的策略抽象

`agent/context_engine.py` 定义了一个完整的 ABC：

```python
class ContextEngine(ABC):
    threshold_percent: float = 0.75      # 触发阈值——窗口 75%
    protect_first_n: int = 3             # 头部保护
    protect_last_n: int = 6              # 尾部保护

    def update_from_response(self, usage) -> None
    def should_compress(self, prompt_tokens) -> bool
    def compress(self, messages, current_tokens, focus_topic) -> List
    def on_session_start(self, session_id)
    def on_session_end(self, session_id, messages)
```

**比例阈值** `threshold_percent=0.75` 自适应所有模型窗口大小。

### 1.2 ContextCompressor 压缩策略

关键参数：
- `_SUMMARY_RATIO = 0.20` — 摘要占被压缩内容的 20%
- `_SUMMARY_TOKENS_CEILING = 12000`
- `_MIN_SUMMARY_TOKENS = 2000`
- 先裁剪工具输出 → 再用辅助模型做迭代摘要

### 1.3 记忆管理器：fence-tag 注入

```python
def build_memory_context_block(raw_context: str) -> str:
    return (
        "<memory-context>\n"
        "[System note: The following is recalled memory context...]\n\n"
        f"{clean}\n"
        "</memory-context>"
    )
```

配合 `StreamingContextScrubber` 防止记忆泄露。

### 1.4 SQLite + FTS5 持久化

- WAL 模式支持并发读 + 单写
- FTS5 全文检索 + trigram 索引双重检索
- parent_session_id 链支持压缩触发会话拆分

## 二、OpenClaw 上下文管理

### 2.1 ContextEngine 接口（7 方法）

```
bootstrap() → 初始化
ingest() → 摄取消息
assemble() → 组装上下文
compact() → 压缩
afterTurn() → 轮后处理
prepareSubagentSpawn() → 子 Agent 上下文准备
onSubagentEnded() → 子 Agent 合并
```

### 2.2 24 个 Plugin Hooks + 3 种执行模式

三种模式：Void Hook（并行/不返回值）、Modifying Hook（串行/返回值 merge）、Sync Hook（同步/热路径）

### 2.3 三层溢出防护

1. ToolResultContextGuard — 单条工具结果截断（maxToolResultTokens=4096）
2. Tool Loop Detection — 三种死循环模式 + 30 次熔断
3. dropThinkingBlocks — 清空历史 thinking

### 2.4 上下文装配链

```
工具列表 → Skills 元数据 → AGENTS.md → SOUL/TOOLS/IDENTITY/USER/MEMORY.md
→ 自更新指令 → 时间上下文 → 压缩摘要
```

三个关键截断参数：`bootstrapMaxChars=12000`、`bootstrapTotalMaxChars=60000`、`maxToolResultTokens=4096`

## 三、核心 Gap 对比

| 维度 | Hermes Agent | OpenClaw | 胜者 |
|:-----|:------------|:---------|:----:|
| 触发机制 | 比例阈值 75% | 绝对 token 数 | **Hermes** |
| 抽象层次 | ContextEngine ABC 7 方法 | 3 套系统（Internal/Plugin/ContextEngine） | 平手 |
| 记忆注入 | fence-tag 隔离 | MEMORY.md 全量拼入 | **Hermes** |
| 压缩质量 | 迭代摘要 + handoff prefix | 单次摘要 + 硬截断 | **Hermes** |
| 长期记忆 | FTS5 按需检索 | Markdown 全量注入 | **Hermes**（最大差距） |
| Token 效率 | 双通道 Skill 加载 + @语法 | 全量注入 + maxSkillsPromptChars | **Hermes** |
| 子 Agent 生命期 | 无等价物 | prepareSubagentSpawn/onSubagentEnded | **OpenClaw** |
| 循环保护 | 14 类结构化异常 | 3 种熔断 + 30 次阈值 | 平手 |

### 代码层最难追上的 Gap

1. **Hermes 的 FTS5 检索式记忆** — OpenClaw 要从文件全量拼接改为数据库按需检索，需改记忆架构底层
2. **OpenClaw 的子 Agent 上下文生命周期钩子** — Hermes 没有 `prepareSubagentSpawn`/`onSubagentEnded` 等价物
