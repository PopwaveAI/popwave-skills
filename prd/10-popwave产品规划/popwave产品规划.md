# popwave 产品规划

> 创建日期：2026-07-07
> 文档性质：0-to-1 产品立项与战略规划

---

## 一、立项定位

popwave 是一款面向网文创作的垂直 agent-ide 产品，以雪花式写作专家、涌现式写作专家、拆书专家、营销推书专家等 skill 群作为核心竞争力。但 skill 层的能力是片段化的、无状态的——任何通用办公 IDE 装一个同类 skill 就能跑起来。如果壁垒只停留在 skill 层，WorkBuddy、Trae 这类通用产品可以用极低成本复制并内化。

真正的分水岭在于：通用办公 IDE 优化的是"单次任务完成度"——用户给一个指令，产品完成一个任务，上下文用完即弃。而网文创作是超长链路的持续演进：一个项目 120 章横跨数月，第 87 章不能让第 12 章已经死掉的角色复活，设定库、人物库、剧情线要持续演化、互相校验。这种"跨章跨会话一致性追踪"是通用 IDE 产品形态天然不处理的区域。

popwave 的立项逻辑由此清晰：不靠 skill 层建立壁垒，而是把 skill 群验证过的领域逻辑沉淀到产品层，构建通用 IDE 即便复制了全部 skill 也够不到的领域基础设施。具体来说，产品层要建两块核心壁垒——创作状态引擎和资产关系图谱——再叠加一批依赖这两块壁垒才能跑起来的甜点功能。skill 层负责"会写"，产品层负责"记得住、算得清、看得见"。

---

## 二、三层架构与竞争分析

把一款 agent-ide 产品拆为三层：skill 层、harness 层、产品层。三层的可内化难度逐级递增。

### 2.1 Skill 层

雪花式写作、涌现式写作、拆书、营销推书等 skill 构成核心能力。这一层是片段化的能力封装——输入一段需求，输出一段创作结果。它无状态、可移植，通用 IDE 只需安装同类 skill 即可覆盖。这是最容易被复制的层，不能作为壁垒。

### 2.2 Harness 层

负责上下文注入、skill 编排、子 agent 调度、状态传递。这一层通用 IDE 也在持续补齐（WorkBuddy 的 context harness engineering 已有成熟实践）。部分能力可被内化，但网文创作管线的编排逻辑绑定特定创作阶段（seed → 卷纲 → 幕纲 → create），比通用任务调度更重度。这一层有一定差异化空间，但不足以构成护城河。

### 2.3 产品层

这是通用 IDE 产品形态天然缺失的区域。产品层要做的是：持久化的领域状态模型、结构化的资产关系、垂直场景的交互范式。通用 IDE 技术上做得到，但它的产品定位决定了不会为某个垂直领域专门建一套状态引擎和资产图谱——那等于自废"通用"的武功。

三层的关系不是叠加，而是依赖：产品层为 harness 层提供持久化基础，harness 层为 skill 层提供编排能力，skill 层提供具体创作能力。壁垒从底向上传导——产品层的领域数据模型一旦建立，跑在上面的 skill 体验会产生质的差异。

## 三、核心壁垒一：创作状态引擎

### 3.1 它解决什么问题

网文创作的最高频翻车点不是"写不好"，而是"写崩了"——角色前后矛盾、设定自己打脸、时间线对不上、剧情线断头。这些问题的根源是：创作过程横跨数月、上百个会话，没有任何机制保证跨章跨会话的一致性。通用 IDE 的 context 是"当前会话 + 当前文件"，用完即弃，对"第 12 章做了什么决定"这类历史状态毫无感知。

### 3.2 产品层要做什么

创作状态引擎是一个持久化的领域状态模型，跨章跨会话存活，记录并校验整个创作过程的演化。核心能力包括：

- **状态快照**：每个章节完成时生成一份完整状态快照（设定库版本、人物状态、剧情线进度、长线债务清单），支持回溯到任意历史节点
- **一致性校验**：新章节产出时，自动与历史状态交叉验证——角色是否已死亡、设定是否冲突、时间线是否合理、剧情线是否断头
- **长线债务管理**：追踪所有埋下的线（伏笔）和偿还的线（回收），标注触发条件，在合适时机提示"该还债了"
- **剧情线健康度**：监控每条 L2 剧情线的推进进度，对沉睡过久的线发出预警

### 3.3 为什么通用 IDE 内化不了

通用 IDE 的文件系统是扁平的、无类型的。一个 skill 调用结束后，状态就消失了——除非用户手动维护。要实现跨章一致性，需要在产品层定义领域状态模型、维护状态演化日志、做冲突检测算法。这不是一个 skill 能完成的，而是产品层的基础设施。通用 IDE 要内化它，等于在自己的产品层为网文单独建一个状态引擎，与其"通用办公"定位直接冲突。

### 3.4 与现有 skill 的衔接

项目里已有的 `state-log.yaml`（append-only 叙事日志）、设定库 / 人物库 / 剧情线分类存储、每 10 章压缩归档机制，是创作状态引擎的雏形。当前这些逻辑散落在 skill 层，由 agent 自觉执行，可靠性低（96.3% 的 runs 中 agent 跳过 skill 文件读取）。产品化方向是把这些逻辑从 skill 层上移到产品层，固化为引擎级能力——agent 不再需要"自觉"维护状态，引擎自动兜底。

## 四、核心壁垒二：资产关系图谱

### 4.1 它解决什么问题

网文创作的资产是结构化的、强关联的：设定卡有 L1 维度归属（世界蓝图 / 力量体系 / 势力格局等），人物有状态机和关系网，剧情线有 L2 追踪和功能位约束，卷纲幕纲有"不能抢的后续收益"限制。这些资产之间存在大量引用关系——改一个设定可能影响多条剧情线，一个人物的状态变化可能牵动多个伏笔。

通用 IDE 只有"文件"概念，没有"资产关系"。文件之间是扁平的、无类型的，一个 skill 没法凭空造出这层关系图谱。结果就是：创作者改了第 20 章的一个设定，根本不知道它会影响第 60 章的哪条剧情线。

### 4.2 产品层要做什么

资产关系图谱是网文创作的领域数据模型，维护所有创作资产及其引用关系。核心能力包括：

- **结构化资产定义**：设定卡、人物卡、剧情线、卷纲、幕纲都有明确的类型归属和字段规范，不再是扁平文件
- **引用关系维护**：设定被哪条剧情线引用、人物参与了哪些事件、伏笔在哪章埋下又在哪章偿还——所有引用关系自动追踪
- **影响传播检测**：修改一个资产时，自动高亮所有受影响的关联资产（改一个设定 → 标红受影响的剧情线）
- **关系可视化**：角色关系网、设定依赖图、剧情线交织图——让创作者一眼看清全局结构

### 4.3 为什么通用 IDE 内化不了

资产关系图谱需要产品层定义领域数据模型、维护引用完整性、做冲突检测。这不是文件系统天然具备的能力，而是需要为网文创作场景专门构建的中间层。通用 IDE 的文件系统是通用的、无类型的，一个 skill 调用结束后引用关系就丢失了。要内化这套图谱，通用 IDE 需要为网文单独建一个领域数据层——与其产品定位冲突。

### 4.4 与现有 skill 的衔接

项目已有的设定卡模板（含 L1 维度字段）、人物库、剧情线追踪、卷纲登场人物表（7 列结构化字段），是资产关系图谱的数据源。当前这些资产以 Markdown 文件形式散落在项目目录里，彼此之间的引用关系靠 agent 人工记忆。产品化方向是把这些文件升级为结构化数据对象，在产品层维护引用关系图谱。

## 五、甜点功能层

甜点功能本身不是护城河——单个拿出来，通用 IDE 抄个功能形态不难。但它们全部跑在两个核心壁垒之上：没有创作状态引擎，冲突雷达、时间轴回溯就没有数据可读；没有资产关系图谱，角色关系网、设定影响追溯就没有关系可算。通用 IDE 即便抄了界面，底层数据不存在，功能就是空壳。

### 5.1 依附创作状态引擎的甜点

| 功能 | 价值 | 为什么抄了也是空壳 |
|---|---|---|
| 时间轴回溯 | 任意章节状态快照，一键回滚到历史节点 | 没有跨章持久状态，无快照可滚 |
| 冲突雷达 | 设定 / 人物 / 时间线矛盾实时检测 | 没有结构化状态，无矛盾可检测 |
| 剧情线健康度 | 各 L2 线推进进度 + 沉睡预警 | 没有 L2 追踪数据，进度无从算起 |
| 长线债务仪表盘 | 埋线 / 偿线追踪 + 触发条件提示 | 没有债务登记，仪表盘是空的 |

### 5.2 依附资产关系图谱的甜点

| 功能 | 价值 | 为什么抄了也是空壳 |
|---|---|---|
| 角色关系网可视化 | 阵营 / 敌友 / 亲疏 + 章节演变动画 | 没有人物关系结构，画不出网 |
| 设定影响追溯 | 改一个设定 → 高亮受影响剧情线 | 没有设定-剧情引用关系，追溯无锚点 |
| 人物状态机视图 | 起点 → 当前轨迹 + 能力 / 关系变化 | 没有人物状态演化记录，轨迹不存在 |
| 世界观地图 | 地理 / 势力 / 资源 L1 维度空间化 | 没有设定卡归属，无法空间投影 |

### 5.3 吃垂直红利的独立甜点

这几个不依附单一壁垒，但依赖网文创作的垂直场景理解：

- **文风 DNA 指纹库**：跨项目沉淀个人文风格纹，越用越像自己。这是创作者私有资产，迁移成本极高。项目已有 `pop-shared-dna` 雏形，产品化后是软性锁定
- **灵感碎片池**：随时捕捉碎片，自动关联到设定 / 人物 / 剧情线。依赖资产图谱做关联，单独抄走也没东西可关联
- **创作数据看板**：日产量、有效字数、连续创作天数、章节节奏分布。纯垂直场景的统计，通用 IDE 不会为写作单独做

## 六、读者-市场反馈闭环（远期壁垒）

前两个壁垒解决"写得好"，这个壁垒解决"写得对"。网文最终面向读者，追读率、完读率、章末流失点应该反向驱动创作调整——哪章节奏拖了、哪个爽点没爆发、哪个角色读者不买账。

这需要打通起点、番茄等平台的数据，把读者行为信号变成创作建议。通用 IDE 连"读者"这个概念都不存在，一个 skill 不可能凭空接入平台数据并形成闭环。这是垂直产品独有的、且越跑数据越厚的护城河。

作为远期规划，当前优先级低于两个核心壁垒，但产品架构预留数据接入接口。

## 七、优先级与路线图

### 7.1 投入原则

甜点功能的投入顺序绑定核心壁垒的成熟度：状态引擎和资产图谱的数据结构定稿之前，甜点功能都是空中楼阁。

### 7.2 三阶段路线

**阶段一：领域数据层定稿**

定义创作状态引擎和资产关系图谱的数据模型。把现有 skill 中散落的 state-log、设定库、人物库、剧情线的结构升级为产品级数据对象，明确字段规范、引用关系、状态演化规则。这一阶段不碰 UI，只做数据层。

**阶段二：核心壁垒产品化**

把数据层能力固化为产品级引擎。创作状态引擎实现状态快照、一致性校验、长线债务追踪；资产关系图谱实现引用维护、影响传播检测。同步上线第一波甜点——冲突雷达和角色关系网可视化，前者解决最高频痛点，后者提供最直观的体感差异。

**阶段三：甜点层铺开 + 远期闭环**

在核心壁垒跑通后，低成本叠加其余甜点功能（时间轴回溯、剧情线健康度、设定影响追溯等）。远期启动读者-市场反馈闭环的平台数据接入。

### 7.3 第一波甜点选择依据

冲突雷达和角色关系网可视化作为第一波，原因：

- 冲突雷达：矛盾是网文创作最高频的翻车点，刚需痛点，用户感知最强
- 角色关系网可视化：用户打开就"这不像通用 IDE"，体感差异最直观，且数据源（人物库）已结构化，实现成本低

## 八、竞争定位总结

| 维度 | 通用办公 IDE（WorkBuddy / Trae） | popwave |
|---|---|---|
| 优化目标 | 单次任务完成度 | 跨章跨会话一致性 |
| context 生命周期 | 短时、用完即弃 | 持久化、持续演化 |
| 资产管理 | 扁平文件、无类型 | 结构化资产、引用关系图谱 |
| 状态追踪 | 无 | 创作状态引擎 + 一致性校验 |
| 读者侧 | 不存在 | 远期平台数据闭环 |
| skill 可内化 | 是 | 是（但产品层壁垒无法内化） |

通用 IDE 技术上做得到产品层的所有功能，但它的产品定位决定了不会为某个垂直领域专门建一套状态引擎、资产图谱和平台数据接入。popwave 要拉开差距，核心动作是：把 skill 层验证过的领域逻辑上移到产品层，固化为领域基础设施。skill 是可复制的弹药，产品层是打不穿的阵地。
