## 一、问题诊断

### 1.1 当前管线认知结构缺陷

写作专家管线（pop-writer-*）经过多轮迭代，已从单skill进化到11 skill的精密传送带。但核心结构缺陷从未被修复：**管线只有「顺序」的概念，没有「维度/层」的概念。**

| 维度 | 当前管线假设 | 真实创作行为 |
|:-----|:------------|:-------------|
| 时间关系 | 线性顺序：A→B→C→D→E | 并行+脉冲：四根线同时进展，偶尔交汇 |
| 素材来源 | 搜→存→用，单次消耗 | 跨域索引→按层提取→杂交→再杂交 |
| 层边界 | 不存在。world做力量体系+命名+历史 | 框架层≠质感层≠机制层≠调味层，各层独立选源 |
| 冲突检测 | qa在终点发现所有问题 | 接缝点应提前发现，但不阻塞 |
| 知识复用 | 素材只服务当前项目 | 跨域融合模式应跨项目积累 |

### 1.2 根因

11个skill、~50个产出文件的管线规模下，**没有任何机制让任何一个步骤知道自己正在处理的是创作的哪一层**。creative采集素材时不问「这是给框架层还是调味层的」→reservoir归档时不标「这是机制层素材」→world同时做框架层（力量体系）和质感层（命名）不分家→plot在一锅粥的层边界上拉种子→prose拿到一个不知道前面对各层做了什么承诺的输入。

> **问题不是缺少某个具体步骤，是管线没有「维度」的概念，只有「顺序」的概念。**

## 二、核心概念：层织机（Layer Loom）

### 2.1 一句话

> **从「顺序文件传送带」重构为「四台并行织机 + 接缝管理器 + 域引擎」。创作不是一根线，是四根线并行纺织，只在接缝点交汇。**

### 2.2 结构总览

```
                        ┌──────────────────┐
                        │   层架构卡        │ ← 管线宪法，一页定层
                        │   (Layer Charter) │
                        └────────┬─────────┘
                                 │ 四台织机同时读取
                   ┌─────────────┼──────────────┐
                   ▼             ▼              ▼
            ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
            │框架织机   │ │质感织机   │ │机制织机   │ │调味织机   │
            │(Framework)│ │(Texture) │ │(Mechanism)│ │(Flavoring)│
            │力量体系   │ │术语/命名  │ │冲突/剧情  │ │叙事/幽默  │
            │数值升级   │ │文明底色   │ │套路链    │ │情绪调控   │
            └─────┬────┘ └─────┬────┘ └─────┬────┘ └─────┬────┘
                  │             │             │             │
                  └─────────────┼─────────────┼─────────────┘
                               ▼             ▼
                        ┌──────────────────────────┐
                        │     接缝管理器            │
                        │     (Seam Manager)        │
                        │  检测·协调·标记·不阻塞    │
                        └──────────────────────────┘
                               │
                     ┌─────────┴─────────┐
                     ▼                   ▼
              ┌────────────┐     ┌──────────────┐
              │ 章节渲染器   │     │   域引擎      │
              │ (Renderer)  │     │ (Domain Engine)│
              │ 设计包→正文  │     │ 跨项目累积    │
              └────────────┘     └──────────────┘
```

## 三、五个概念详解

### 3.1 概念一：层架构卡（Layer Charter）

**替代对象：** creative 所有产出（PRD + 故事引擎 + 素材储备池 + 爽点引擎 + 样品试读）

**定位：** 管线宪法，一页纸。

**核心变化：** PRD 从 6 维度扩展到 7 维度，新增「层架构」维度。层架构卡不是「这本书是什么」的单一陈述——它是**四台织机的层契约**。

#### 模板

```markdown
# 层架构卡 · {书名}

> v1 | {日期} | 基于{前N章/全书}分析

---

## 1. 一句话定位

> {一句话，让脑中放电影}

---

## 2. 类型锚点

- 主类型：{玄幻/历史/悬疑/科幻}
- 辅类型/跨界元素：{DND/克苏鲁/战锤}
- 混合度：{轻量叠层/深度融合/完全重构}

---

## 3. 核心承诺

> {给读者的合同}

---

## 4. 层架构（★ 新增 — 管线宪法）

| 层 | 功能定义 | 源域 | 转化规则 | 行为禁区 | 设计节奏 | 接缝点 |
|:-|:---------|:-----|:---------|:---------|:---------|:-------|
| **框架层** | 世界怎么运转——力量/升级/属性/数值 | DND 3.5e | 属性→外功六要素 | ❌不碰命名、不碰历史 | 先纺 | world数值前 |
| **质感层** | 事物怎么称呼——术语/命名/文明底色 | 佛教名相+阴符经 | 名相化，不参与机制 | ❌不碰数值、不碰冲突结构 | 先纺 | world命名前 |
| **机制层** | 冲突怎么解决——套路/博弈/爽点节奏 | 三十六计+金融借贷 | 博弈结构→剧情线 | ❌不碰命名、不碰幽默密度 | 中纺 | plot分幕前 |
| **调味层** | 读者什么感受——叙事距离/幽默缓冲/情绪调压 | 穿越者网络梗 | 幽默不破基调 | ❌不突破框架层黑暗承诺 | 后纺 | prose前 |

---

## 5. 不可妥协的DNA

1. {第一条}
2. {第二条}
3. {第三条}

---

## 6. 边界（否定式约束）

- ❌ {不做什么}
- ❌ {不做什么}
```

**变化幅度：** creative 产出从 ~10 文件降为 1 卡。

### 3.2 概念二：四台织机

**替代对象：** pop-writer-world + pop-writer-plot + pop-writer-prose

**核心变化：** 不再是三个顺序skill各自吃饭，而是四台织机并行工作，各自只做自己层的事。

#### 框架织机（Framework Loom）

| 属性 | 值 |
|:-----|:----|
| 替代 | pop-writer-world（力量体系/数值部分） |
| 产出 | 力量体系文档、数值曲线、升级表、资源循环规则 |
| 输入 | 层架构卡（框架层源域+转化规则） |
| 不做什么 | 不碰命名（那是质感织机的事）、不碰情节（那是机制织机的事） |
| 行为禁区 | [❌不碰术语命名、❌不碰历史叙述] |
| 工作模式 | 脉冲式：按接缝点产出版本，不追求一次终稿 |
| 接缝点 | ①层架构卡锁定后启动 ②数值初版→机制织机读 ③机制反馈→数值调整 |

#### 质感织机（Texture Loom）

| 属性 | 值 |
|:-----|:----|
| 替代 | pop-writer-world（命名/术语部分） |
| 产出 | 术语体系表、命名规则、氛围指南 |
| 输入 | 层架构卡（质感层源域+转化规则） |
| 不做什么 | 不碰数值、不碰暴力结构、不碰情节走向 |
| 行为禁区 | [❌不碰数值、❌不碰冲突结构] |
| 接缝点 | ①框架力量体系锁定后→命名对齐 ②机制剧情线锁定后→角色名对齐 |

#### 机制织机（Mechanism Loom）

| 属性 | 值 |
|:-----|:----|
| 替代 | pop-writer-plot |
| 产出 | 剧情线文档、分幕规划、套路链、契诃夫枪链 |
| 输入 | 层架构卡 + 框架织机的数值约束 |
| 不做什么 | 不碰命名、不碰幽默密度 |
| 行为禁区 | [❌不碰术语命名、❌不碰幽默插入时间点] |
| 接缝点 | ①框架织机数值初版 ②质感织机角色名 ③→章节渲染器 |

#### 调味织机（Flavoring Loom）

| 属性 | 值 |
|:-----|:----|
| 替代 | pop-writer-prose |
| 产出 | 正文章节 |
| 输入 | 层架构卡 + 设计包 + 文风DNA |
| 不做什么 | 不碰剧情、不突破框架层基调承诺 |
| 行为禁区 | [❌不改变事件链、❌不突破框架黑暗承诺、❌不篡改角色核心设定] |
| 接缝点 | ①机制织机出设计包 ②渲染前检查「调味空间」 |

### 3.3 概念三：接缝管理器（Seam Manager）

**替代对象：** pop-writer-qa + 修宪回路

**定位：** 不做设计，只做检测·协调·标记。

**核心设计原则——不阻塞：** 检测发现问题→标记但不停止→各织机可选接或不接。只有当层契约明确声明「这两层互斥」时才阻塞。

#### 三职能

| 职能 | 做什么 | 不做什么 |
|:-----|:-------|:---------|
| **检测** | 每台织机的产出是否遵守自己的层契约和行为禁区 | 不做质量评估 |
| **协调** | 织机A的产出vs织机B的产出是否矛盾 | 不做修改决策 |
| **标记** | 矛盾输出为 `{层A}:{层B} tension @{章号}` | 不阻塞产出流程 |

#### 标记等级

| 等级 | 含义 | 处理 |
|:-----|:-----|:-----|
| ✅ 可共存 | 两层有张力但可协商 | 标记记录，织机自行决定 |
| ⚠️ 建议协调 | 微调可解决 | 通知两织机的下一次脉冲 |
| ❌ 必须阻塞 | 层契约明确互斥（如：调味层的搞笑破坏框架层的绝望基调） | 阻塞，送层架构卡重新裁决 |

### 3.4 概念四：域引擎（Domain Engine）

**替代对象：** pop-writer-reservoir + pop-writer-fuse

**定位：** 跨项目累积的域索引 + 转化规则库。

#### 结构

```
域引擎/
├── 域索引/
│   ├── DND 3.5e/
│   │   ├── 可借元素.json（属性/法术/怪物/专长/阵营/装备）
│   │   ├── 已验证转化规则.md
│   │   │   ├── 属性→外功六要素 ✅（深渊主宰项目验证）
│   │   │   └── 法术位→经脉开穴 ❓（验证中）
│   │   └── 已知风险.md（过度机制化会降低叙事流畅度）
│   ├── 三十六计/
│   │   ├── 可借元素.json（15计名+结构描述）
│   │   └── 已验证转化规则.md
│   │       └── 暗渡陈仓→双线叙事 ✅（让魔门再次伟大项目验证）
│   └── 佛教名相/
│       ├── 可借元素.json
│       └── 已验证转化规则.md
│           └── 他化自在天→功法名 ✅
│
└── 杂交记录/
    ├── 太平经案例：系统流[70%]×金融借贷[20%]×佛教他化自在天[10%]
    ├── 外功六要素：传统丹道[60%]×DND属性[40%]
    └── 五贼盗天功法：阴符经[50%]×仙侠修真[50%]
```

#### 变化：素材不再只为当前项目服务

skill reservoir 当前把素材内化后归档到当前项目的储备池，项目结束后消失。域引擎是**跨项目累积**的——每个项目结束时，将其融合模式提炼为杂交记录存入域引擎。

#### 层架构卡设计时的工作流

1. 作者说「我想写一本像DND+三十六计的黑暗玄幻」
2. 域引擎搜索 → DND所有可借元素列表 → 三十六计已验证转化规则
3. 四层各自选源：
   - 框架层建议：DND（有验证记录，风险低）
   - 机制层建议：三十六计（有黑暗修仙项目的验证记录）
   - 质感层建议：佛教名相或阴符经（可混合）
4. 层架构卡产出时自动引用域引擎中的验证记录

### 3.5 概念五：章节渲染器（Renderer）

**替代对象：** pop-writer-chapter + pop-writer-prose 的当前边界

**变化：** 渲染器不是被动执行的，它在事件骨架敲定之前就知道了自己的调味空间。

#### 工作流程

```
机制织机产出剧情线 → 确定第X章的基调（压抑/高潮/缓冲）
                ↓
调味织机提前感知：「第X章压抑基调→幽默空间很小→叙事距离需拉近」
                ↓
渲染器读设计包 + 调味空间判断 → 确定叙事距离 → 渲染正文
```

#### 调味空间判断（渲染前强制自问）

```
调味层空间判断：
- 当前场景基调：{从设计包情绪目标读取}
- 框架层黑暗承诺是否允许幽默插入：{是/否/有条件}
- 如果允许，幽默最大密度：{冷幽默/单次吐槽/完整玩梗}
- 如果不允许，替代调味手段：{细节描写/感官放大/叙事节奏}
```

## 四、工作流程：卷1全生命周期

```
[Phase 0 · 层诊断]
  作者说想法 → 路由判断 → 层架构卡设计
  （不同于当前：不是问「类型/基调/机制」然后写PRD
   而是先问「这书分几层、每层从哪借、层间怎么接缝」）
   产出：层架构卡（1页）
   ↓

[Phase 1 · 织机并行]
  框架织机启动：力量体系初版（脉冲1）
  质感织机启动：术语表初版（脉冲1）
  机制织机暂候：等框架数值就位
  调味织机暂候：等机制剧情线就位
  ↓ 接缝点

[Phase 1.5 · 接缝1]
  框架织机数值初版 → 机制织机读取
  框架织机基调约束 → 调味织机预注册基调
  接缝管理器检测层间一致性
  ↓

[Phase 2 · 机制启动]
  机制织机启动：从框架读约束→设计剧情线+分幕+套路链
  质感织机持续：角色命名与机制剧情线对齐
  框架织机持续：按机制反馈调整数值
  ↓ 接缝点

[Phase 2.5 · 接缝2]
  机制剧情线定型 → 调味织机感知每章调味空间
  接缝管理器标记所有 tension
  ↓

[Phase 3 · 渲染器启动]
  章节渲染器脉冲：每章
  ① 机制织机产设计包
  ② 调味织机检查「此章调味空间」
  ③ 渲染器按调味空间渲染正文
  ④ 接缝管理器标记层间 tension
  ↓

[Phase 4 · 章节交付]
  正文章节产出
  接缝管理器沉淀检测记录（供qa消费）
  项目结束时→域引擎沉淀杂交记录
```

## 五、产出一览

### 5.1 按概念

| 概念 | 产出文件 | 文件数 |
|:-----|:---------|:------:|
| 层架构卡 | 1 个 markdown | 1 |
| 框架织机 | 力量体系 + 数值曲线 + 资源循环规则 | 3 |
| 质感织机 | 术语体系 + 命名规则 + 氛围指南 | 3 |
| 机制织机 | 剧情线文档 + 分幕规划 + 套路链 + 契诃夫枪链 | ~5 |
| 调味织机 | 正文章节（卷内章数） | ~30-200 |
| 接缝管理器 | 检测日志（不留盘/或选留） | 0-1 |
| 域引擎 | 域索引 + 转化规则 + 杂交记录（跨项目） | 持续增长 |

### 5.2 与当前管线的对比

| 指标 | 当前管线 | 层织机方案 |
|:-----|:---------|:-----------|
| skill 数 | ~11 | 5 概念 |
| 产出文件数 | ~50/项目 | ~12/项目 |
| 文件类型 | PRD/故事引擎/储备池/L1×6/宪法… | 层架构卡/织机产出×3-5 |
| 层意识 | 无 | 层契约驱动 |
| 冲突发现时机 | qa（终点） | 接缝点（期间） |
| 阻塞模式 | 发现即阻塞 → 退回重来 | 标记不阻塞 → 织机自愈 |
| 跨项目复用 | 无 | 域引擎累积杂交记录 |
| 并行度 | 严格顺序 | 四织机脉冲并行 |

## 六、与当前管线的关系

### 6.1 覆盖关系

| 当前 skill | 去向 |
|:-----------|:------|
| pop-writer-creative | → 层架构卡（产出从10文件降为1卡） |
| pop-writer-reservoir | → 域引擎（从单项目到跨项目） |
| pop-writer-fuse | → 域引擎（杂交记录沉淀） |
| pop-writer-world | → 框架织机 + 质感织机（从1拆2） |
| pop-writer-plot | → 机制织机 |
| pop-writer-chapter | → 渲染器（读入部分） |
| pop-writer-prose | → 调味织机 + 渲染器（渲染部分） |
| pop-writer-qa | → 接缝管理器（检测部分） |
| pop-writer-character | → 质感织机（角色命名子集） |
| pop-writer-continue | → 机制织机（续写场景下恢复状态） |
| pop-writer-game | → 外部（独立） |
| pop-writer-html | → 外部（独立） |

### 6.2 可保留的现有资产

- **文风DNA档案（pop-shared-dna）** — 格式完全保留，调味织机和渲染器直接消费。
- **设计包格式（chXXX-设计包.md）** — 格式保留，渲染器的核心输入。
- **实体快照（entity-snapshot.yaml）** — 保留，渲染器每章后更新。
- **套路库（pop-trope-library）** — 保留，机制织机消费。
- **数值体系模板** — 保留，框架织机使用。

### 6.3 不受影响的拆书专家

层织机方案只影响**写作专家**管线。拆书专家（pop-novel-deconstructor / pop-shared-dna / pop-decon-trace）的产出一律保留并增强消费——域引擎的初始杂交记录正是从创意溯源的产出中提取。

## 七、迁移路径

### 7.1 原则

- **不是一次性推倒重建**。层架构卡可以先作为一个轻量步骤嵌入当前管线的 creative 末端。
- **每台织机可以独立验证**。框架织机先实现，不影响当前 plot+prose 的运作。
- **域引擎从零开始**。第一个项目不依赖域引擎的历史数据——用它新产生的杂交记录填充。

### 7.2 三步走

#### 步骤一：层架构卡先行（最低侵入）

- 当前管线的 creative 末端增加一个步骤：产出层架构卡
- 现有 PRD 模板新增第 7 维度「层架构」
- 层架构卡产出后，后续所有 skill 加载时读取（不改变现有执行逻辑，只增加感知）
- 预期变化：agent 开始意识到自己属于哪一层

#### 步骤二：织机分拆（中等侵入）

- world 拆为框架织机 + 质感织机（两个独立步骤，不是两个skill）
- plot 的拉种子步骤增加层过滤
- prose 渲染前加调味空间判断
- 接缝管理器作为一个轻量工具上线（只做检测和标记，不阻塞）
- 预期变化：产出的层归属清晰，交叉污染减少

#### 步骤三：域引擎替换 reservoir（高侵入）

- 当前项目储备池改为域引擎格式
- 每个项目结束时自动沉淀杂交记录
- 新项目启动时域引擎作为候选源
- 预期变化：跨项目复用开始生效

### 7.3 回退机制

每步都设计为可逆：
- 层架构卡 → 可以废弃，退回当前 PRD 格式
- 织机分拆 → world 可以合并回一个 skill
- 域引擎 → reservoir 可以恢复为单项目模式

## 八、风险与对策

### 8.1 风险矩阵

| # | 风险 | 概率 | 影响 | 对策 |
|:-:|:-----|:----:|:----:|:-----|
| 1 | 并行织机的注意力分散 | 中 | 高 | 每台织机设行为禁区列表，不做什么 > 要做什么 |
| 2 | 层契约是软约束，LLM易忽略 | 高 | 高 | 层契约印在每个接缝点的提示词中，不是一次性加载 |
| 3 | 域引擎冷启动期无数据 | 高 | 中 | 从创意溯源（pop-decon-trace）的初次产出自动填充 |
| 4 | 接缝管理器「不阻塞」被滥用 | 中 | 中 | 设至少一个硬阻塞点：调味渲染前基调一致性检查 |
| 5 | 四个概念过于抽象，落地困难 | 中 | 高 | 每概念拆为3个强制推理步骤+1个检查清单 |
| 6 | 旧用户习惯难以切换 | 高 | 低 | 三步迁移，每一步都是增量改进不是颠覆 |

### 8.2 关键机制：行为禁区

每台织机的行为禁区列表是层契约的**执行锚点**。它比「要做什么」更重要，因为：

- 当前管线的所有「层污染」问题都是织机**做了不该自己做的事**（world做了质感命名→污染了框架层数值）
- 行为禁区是LLM最容易理解的约束形式（「不要做X」比「做Y」更不会产生歧义）

#### 禁区设计原则

- 每条禁区以 `❌` 前缀
- 每条禁区对应一个具体的「**不做某件事**」，不是「不要做错某件事」
- 每个接缝点检查时，禁区遵守情况是第一优先级

## 九、待决策项

| # | 决策项 | 选项 | 建议 |
|:-:|:-------|:-----|:-----|
| 1 | **层数** | 4层（框架/质感/机制/调味）/ 5层（+数据感知层） | 先4层，需求验证后扩展 |
