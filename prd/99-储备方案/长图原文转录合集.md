# 第一篇：MCP 协议深度解析

> 【第12期/总第23期】AI模型上下文协议：Model Context Protocol (MCP)

---

## 为什么需要MCP？

大模型（LLM）的能力受限于训练数据，无法获取实时信息、执行外部操作或访问私有数据。传统解决方案是为每个场景单独开发集成代码，导致"N×M"的集成困境：

- **N个模型** × **M个工具** = **N×M个集成**

MCP通过标准化协议，让模型与外部世界的连接变得像"USB接口"一样即插即用。

---

## 先把模型看成一个无状态的函数

```
┌─────────────────┐
│   输入 Prompt    │
└────────┬────────┘
         ▼
┌─────────────────┐
│      LLM        │  ← 无状态：每次调用都是独立的
│  (大语言模型)    │
└────────┬────────┘
         ▼
┌─────────────────┐
│   输出 Response  │
└─────────────────┘
```

**核心特点：**
- 无记忆：不知道之前的对话历史
- 无外部连接：不能访问互联网、数据库、文件系统等
- 纯函数：给定输入，产生输出

---

## 模型与外部系统之间的结构化协议

```
┌─────────┐     ┌─────────────┐     ┌─────────┐
│  Client │◄───►│     MCP     │◄───►│ Server  │
│  (Host) │     │   Protocol  │     │(Tools)  │
└─────────┘     └─────────────┘     └─────────┘
      │                                  │
      │    ┌─────────────────┐          │
      └───►│  1. Tools       │◄─────────┘
           │  2. Resources   │
           │  3. Prompts     │
           │  4. Sampling    │
           └─────────────────┘
```

**MCP协议核心能力：**

| 能力 | 说明 | 示例 |
|:---|:---|:---|
| **Tools** | 模型可调用的函数 | 查天气、发邮件、操作数据库 |
| **Resources** | 只读的数据源 | 文件内容、数据库记录、API响应 |
| **Prompts** | 预定义的提示模板 | 代码审查模板、数据分析模板 |
| **Sampling** | 服务器请求模型生成 | 服务器让模型补全、总结、翻译 |

## 一次完整的调用流程

```
┌─────────┐                              ┌─────────┐
│  User   │──"查北京明天天气"─────────────►│  Client │
│         │                              │ (Host)  │
└─────────┘                              └────┬────┘
                                              │
                                              ▼
┌─────────┐    ┌─────────────────┐      ┌─────────┐
│ Weather │◄───│  3. 调用工具     │◄─────│   LLM   │
│ Server  │    │  4. 返回结果     │      │         │
│ (MCP)   │───►│                 │─────►│         │
└─────────┘    └─────────────────┘      └────┬────┘
     ▲                                        │
     └────────────────────────────────────────┘
              2. 识别需要调用工具
              5. 整合结果生成回复
```

**详细步骤：**

1. **用户提问**："查北京明天天气"
2. **LLM分析**：识别需要调用外部工具
3. **MCP调用**：通过协议调用Weather Server的`get_weather`工具
4. **执行返回**：Server执行查询，返回结构化数据
5. **生成回复**：LLM整合结果，生成自然语言回复

## MCP与Function Calling的关系

```
┌─────────────────────────────────────────┐
│           Function Calling              │
│    (OpenAI/Anthropic等模型原生能力)      │
│         "模型识别需要调用函数"            │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│              MCP Protocol               │
│      "标准化的函数描述与调用协议"         │
│    统一不同模型、不同平台的差异           │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│           MCP Server/Client             │
│      "具体的工具实现与连接管理"           │
└─────────────────────────────────────────┘
```

**关键理解：**
- **Function Calling** = 模型的"意愿"（想调用工具）
- **MCP** = 标准化的"方式"（怎么描述、怎么调用、怎么返回）
- **MCP Server** = 实际的"执行者"（真正干活的人）

## 工具定义：标准化的接口描述

```json
{
  "name": "get_weather",
  "description": "获取指定城市的天气信息",
  "inputSchema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称，如'北京'"
      },
      "date": {
        "type": "string",
        "description": "日期，格式'YYYY-MM-DD'"
      }
    },
    "required": ["city"]
  }
}
```

**关键字段：**
- `name`：工具唯一标识
- `description`：**至关重要**，LLM靠它理解工具用途
- `inputSchema`：JSON Schema格式，描述参数结构

## Resources：只读的数据上下文

```
┌─────────────────────────────────────────┐
│              MCP Resources              │
│         "给模型提供只读上下文"            │
├─────────────────────────────────────────┤
│  类型          │  示例                   │
├────────────────┼────────────────────────┤
│  File          │  本地文件、代码文件      │
│  Database      │  数据库记录、查询结果    │

```

**与Tools的区别：**

| | Tools | Resources |
|:---|:---|:---|
| **方向** | 模型 → 外部（主动调用） | 外部 → 模型（被动读取） |
| **操作** | 可执行操作（读/写/执行） | 只读访问 |
| **时机** | 模型决定何时调用 | 用户或系统预加载 |
| **示例** | 发送邮件、创建文件 | 读取代码、查看文档 |

## 开发一个MCP Server（Node.js示例）

```typescript
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  CallToolRequestSchema,
  ListToolsRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";

// 1. 创建Server实例
const server = new Server(
  {
    name: "weather-server",
    version: "1.0.0",
  },
  {
    capabilities: {
      tools: {},  // 声明支持Tools
    },
  }
);

// 2. 定义可用工具列表
server.setRequestHandler(ListToolsRequestSchema, async () => {
  return {
    tools: [
      {
        name: "get_weather",
        description: "获取指定城市天气",
        inputSchema: {
          type: "object",
          properties: {
            city: { type: "string" },
          },
          required: ["city"],
        },
      },
    ],
  };
});

// 3. 处理工具调用
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  if (request.params.name === "get_weather") {
    const { city } = request.params.arguments;
    // 实际调用天气API...
    const weather = await fetchWeather(city);
    return {
      content: [
        {
          type: "text",
          text: `${city}天气：${weather}`,
        },
      ],
    };
  }
  throw new Error("Unknown tool");
});

// 4. 启动服务
const transport = new StdioServerTransport();
await server.connect(transport);
```

## MCP vs 传统方案对比

| 维度 | 传统集成 | MCP协议 |
|:---|:---|:---|
| **开发成本** | N×M个集成，重复开发 | 一次开发，多模型复用 |
| **接入效率** | 每新增工具需改代码 | 配置即接入，无需改动 |
| **标准化** | 各平台标准不一 | 统一协议，跨平台兼容 |
| **安全性** | 分散管理，难以审计 | 集中管控，细粒度权限 |
| **可扩展性** | 紧耦合，难维护 | 松耦合，插件化架构 |
| **生态丰富度** | 依赖单一厂商 | 开源生态，社区驱动 |

## 关键设计原则总结

**1. 模型为中心（Model-Centric）**
协议设计围绕LLM的能力特点优化
- 自然语言描述工具用途
- 结构化输入输出（JSON Schema）
- 支持多轮、多工具组合调用

**2. 双向通信（Bidirectional）**
打破传统单向请求-响应模式
- Client ↔ Server 双向交互
- 支持Sampling反向调用模型

**3. 渐进增强（Progressive Enhancement）**
从简单到复杂的能力扩展
Tools → Resources → Prompts → Sampling

**4. 安全隔离（Security Isolation）**
本地优先，用户授权控制
- 工具执行需用户确认
- 文件访问限定白名单目录
- 敏感操作二次验证

# 第二篇：Procedural Memory — 让 AI Agent 拥有真正的长期记忆

> 来源：WorkBuddy 企业级 AI Agent 平台
> 技术博客：blog.workbuddy.ai
> GitHub：github.com/workbuddy-ai/procedural-memory

---

## 背景：做好 Prompt Engineering 就够了吗？

**每多走一轮，对话历史就越来越长**

Context Window 上下文窗口

```
用户：我想订一张从北京到上海的机票
第一轮 Assistant：好的，请问您想哪天出发？
第二轮 用户：明天下午
Assistant：好的，已为您查询到明天下午从北京到上海的航班...
第 N 轮 ...
→ 上下文爆炸
```

**传统方案：Context Engineering - 被动压缩**

- Truncation 截断
- Summarization 总结
- RAG 检索增强

问题：丢失细节、丢失状态、丢失上下文

---

## 核心概念：Procedural Memory 程序记忆 — 知道该做什么

**像人类一样，把"怎么做"内化为本能**

骑自行车 / 游泳 / 弹钢琴 —— 不需要回忆每一步，身体自动执行

**Procedural Memory = 可执行的程序 + 触发条件**

| 触发条件 When 什么时候做 | 执行程序 What 做什么 | 参数填充 How 怎么做 |
|:---|:---|:---|

### 示例：订机票 Agent 的程序记忆

```
程序 ID: book_flight
触发条件: 用户表达订票意图
  - 关键词：订机票、航班、飞机票
  - 实体识别：出发地、目的地、时间

执行程序:
  1. 确认出发地 → 参数: from_city
  2. 确认目的地 → 参数: to_city
  3. 确认日期 → 参数: date
  4. 查询航班 → 调用 API: search_flights(from, to, date)
  5. 展示结果 → 渲染卡片
  6. 确认预订 → 调用 API: book_ticket(flight_id)
```

### 相比 RAG，程序记忆的优势

| RAG | Procedural Memory |
|:---|:---|
| 检索相似文本片段 | 检索可执行程序 |
| 需要 LLM 重新理解 | 直接加载执行逻辑 |
| 可能检索不准 | 精确匹配触发条件 |
| 每次都要生成回复 | 按既定流程执行 |

## Memory：LLM 应用的三层记忆架构

### 三层记忆模型（金字塔结构）

**最上层 — Working Memory 工作记忆**
- 当前对话轮次
- 临时上下文
- 容量最小，速度最快

**中间层 — Semantic Memory 语义记忆**
- 知识库、文档
- 用户画像
- RAG 检索
- 容量中等，检索速度中等

**最底层 — Procedural Memory 程序记忆**
- 技能、流程、程序
- 触发-执行模式
- 容量最大，执行最稳定

### 对比表格

| 特性 | Working Memory | Semantic Memory | Procedural Memory |
|:---|:---|:---|:---|
| 存储内容 | 当前对话上下文 | 知识、事实、文档 | 程序、流程、技能 |
| 访问方式 | 直接读取 | 向量检索 | 条件触发 |
| 更新频率 | 每轮更新 | 定期更新 | 版本化管理 |
| 容量 | 有限（上下文窗口） | 较大（向量数据库） | 最大（程序库） |
| 典型应用 | 多轮对话保持 | 知识问答 | 复杂任务执行 |

**简单任务 → 复杂任务 / 单次调用 → 多轮交互 / 静态知识 → 动态决策**

## ReAct 范式：推理与行动交织

```
Thought（思考）
  ↓
Action（行动）
  ↓
Observation（观察）
  ↓
  ↻ 循环直至任务完成
```

```python
# ReAct 循环示例
def react_loop(query, max_steps=10):
    for step in range(max_steps):
        # 1. 思考：分析当前状态，制定下一步计划
        thought = llm.predict(
            f"Question: {query}\n"
            f"Previous steps: {history}\n"
            f"Think about what to do next:"
        )
        
        # 2. 行动：选择并执行工具
        action = parse_action(thought)
        observation = execute_tool(action)
        
        # 3. 观察：记录执行结果
        history.append({
            "thought": thought,
            "action": action,
            "observation": observation
        })
        
        # 4. 判断：是否已得到最终答案
        if is_final_answer(thought):
            return extract_answer(thought)
    
    return "Max steps reached"
```

## 七、Loop Engineering：任务如何同时被解决

Loop Engineering 是一种让 AI 能同时处理多个任务的方法。它通过把大任务拆成小任务，让小任务之间能互相配合，像流水线一样工作。这样 AI 就能更快、更准地完成任务。

这个概念来自软件工程里的 Pipeline（流水线），但 Loop Engineering 更强调"循环"——任务不是一次性走完，而是可以反复迭代、优化。

### 1. 什么是 Loop Engineering？

- Prompt Engineering → 告诉 AI 要做什么
- Loop Engineering → 让 AI 自己知道要做什么、怎么做、怎么改

Loop Engineering 的核心是：把复杂任务拆成多个小任务，让 AI 能自动循环执行、自我修正。

### 2. Loop Engineering 的四个步骤

**步骤1：分解（Decompose）**
把大任务拆成小任务。比如写代码可以拆成：理解需求 → 设计架构 → 写代码 → 测试 → 优化。

**步骤2：循环（Loop）**
让小任务形成循环。比如写代码时发现测试不通过，就回到"改代码"步骤，直到通过。

**步骤3：评估（Evaluate）**
每个步骤都要有评估标准。比如代码是否运行？是否符合需求？是否有 bug？

**步骤4：优化（Optimize）**
根据评估结果优化。比如发现某个步骤经常出错，就加强那个步骤的提示词。

### 3. Loop Engineering vs 传统方法

- 传统方法：一次性给 AI 所有信息，期待它一次做对。
- Loop Engineering：分步骤、可循环、能自我修正。

### 4. Loop Engineering 的实际应用

以写代码为例：

- **Loop 1：理解需求** — AI 读需求文档，输出理解摘要，人确认或修正。
- **Loop 2：设计架构** — AI 根据需求设计代码结构，人确认或修正。
- **Loop 3：写代码** — AI 写具体代码，自动运行测试。
- **Loop 4：测试与修复** — 如果测试失败，AI 自动分析错误、修改代码，再测试，直到通过。
- **Loop 5：代码审查** — AI 检查代码质量、安全性、性能，提出优化建议。
- **Loop 6：优化与重构** — AI 根据审查结果优化代码。

每个 Loop 都可以独立运行，也可以互相触发。比如 Loop 4 发现架构问题，可以回到 Loop 2。

### Loop Engineering 循环工程示意图

```
┌─────────────────────────────────────────┐
│  Loop Engineering · 循环工程            │
│  让 AI 像软件工程师一样迭代开发          │
├─────────────────────────────────────────┤
│                                         │
│    ┌─────────┐    ┌─────────┐    ┌─────────┐
│    │ 需求理解 │ → │ 架构设计 │ → │  编码   │
│    │Understand│   │ Design  │   │  Code   │
│    └────┬────┘    └────┬────┘    └────┬────┘
│         ↑              │              │
│         └──────────────┴──────────────┘
│                          ↓
│                    ┌─────────┐
│                    │ 测试验证 │
│                    │  Test   │
│                    └────┬────┘
│                         ↓
│                    ┌─────────┐
│                    │ 优化重构 │
│                    │Optimize │
│                    └─────────┘
│                         ↓
│                    [循环迭代]
│
│  关键原则：
│  ① 小步快跑  ② 快速验证  ③ 自动回滚  ④ 持续优化
│
└─────────────────────────────────────────┘
```

### 5. Loop Engineering 的关键技术

- **状态管理**：记录每个 Loop 的状态，知道现在在哪一步
- **上下文传递**：把上一个 Loop 的结果传给下一个
- **错误处理**：某个 Loop 失败怎么办？重试？跳过？回滚？
- **人机协作**：哪些 Loop 必须人确认？哪些可以全自动？
- **并行执行**：独立的 Loop 可以同时跑，提高效率

### 6. Loop Engineering 的工具支持

目前支持 Loop Engineering 的工具：

- **Cursor Composer**：可以分步骤执行，但循环能力有限
- **Claude Artifacts**：可以生成代码并预览，但缺乏自动迭代
- **GitHub Copilot Workspace**：有计划支持多步骤任务
- **自建系统**：用 LangChain、AutoGen 等框架自己搭

### 7. Loop Engineering 的挑战

- **循环次数控制**：无限循环怎么办？怎么判断"足够好"？
- **状态爆炸**：Loop 太多，状态太复杂，怎么管理？
- **成本问题**：每个 Loop 都调用 AI，token 消耗大
- **延迟问题**：循环越多，总时间越长
- **可解释性**：Loop 中间状态怎么展示给人看？

### 8. Loop Engineering 的未来

我认为 Loop Engineering 会成为 AI 编程的标配。未来的 IDE 不是帮你写代码，而是帮你管理整个开发流程：

- 自动分解任务
- 自动循环执行
- 自动测试验证
- 人机协作决策

这才是真正的"AI 软件工程师"。

- 需求变更时，AI 能增量修改吗？还是重写？
- 代码风格不一致，怎么统一管理？
- 技术债务怎么积累、怎么还？

### 3. 安全性和可靠性

- AI 会引入安全漏洞吗？（会的，已经有很多案例）
- 关键系统能用 AI 生成代码吗？（医疗、金融、航空）
- 怎么证明 AI 生成的代码是正确的？（形式化验证？）

### 4. 创造性工作的边界

AI 能设计新架构吗？还是只能模仿已有模式？

- 真正的创新需要打破常规，但 AI 基于统计学习
- 怎么平衡"安全"和"创新"？

## 九、评述 Cursor、Harmonic、Bolt 等工具

### 1. Cursor：最接近 Loop Engineering 的 IDE

Cursor 是目前最火的 AI 编程工具。它的核心能力：

- **Tab 补全**：基于上下文预测下一行代码
- **Chat**：对话式编程，可以问问题、改代码
- **Composer**：多文件编辑，可以跨文件修改
- **@ 符号**：引用代码、文件、文档、错误信息

Cursor 的局限：

- 还是"单次交互"思维，缺乏真正的 Loop
- 多文件编辑时，容易改坏其他地方
- 对大型代码库的理解有限

### 2. Harmonic：YC 新宠，100x 工程师的尝试

Harmonic 是 YC 投资的新公司，目标是打造"100x 工程师"——一个 AI 能顶 100 个人类工程师。

他们的思路：

- 不只是代码补全，而是完整的开发流程
- 自动分解任务、自动执行、自动验证
- 更接近 Loop Engineering 的理念

但 Harmonic 还很早期，具体产品形态不清楚。

### 3. Bolt：AI 生成完整应用

Bolt（bolt.new）让用户用自然语言描述，直接生成可部署的完整应用。

特点：

- **端到端**：从需求到部署，一键完成
- **可视化**：可以实时预览效果
- **简单应用效果好，复杂应用容易崩**

局限：

- 生成的代码质量参差不齐
- 难以定制和扩展
- 不适合生产环境

### 4. 其他值得关注的产品

- **Replit Agent**：Replit 推出的 AI 编程助手，集成在云端 IDE
- **GitHub Copilot Workspace**：GitHub 计划中的多步骤编程工具
- **Devin**：Cognition AI 的"AI 软件工程师"，演示很炫，但争议也大
- **SWE-agent**：普林斯顿开源的 AI 编程智能体

## 十、AI 编程的未来趋势

### 1. 从工具到伙伴

未来的 AI 不是"帮我写代码的工具"，而是"和我一起解决问题的伙伴"。

- 能理解业务上下文
- 能主动提出建议
- 能长期陪伴项目成长

### 2. 从单点到流程

不只是代码生成，而是覆盖完整软件生命周期：

需求 → 设计 → 编码 → 测试 → 部署 → 运维 → 迭代

### 3. 从通用到垂直

不同领域需要不同的 AI 编程能力：

- **前端开发**：重视 UI/UX、交互细节
- **后端开发**：重视性能、安全、可扩展
- **数据科学**：重视数据处理、模型训练
- **嵌入式**：重视资源限制、实时性

### 4. 从替代到增强

AI 不会完全替代程序员，而是增强程序员的能力：

- 初级程序员 → 更快成长
- 中级程序员 → 效率翻倍
- 高级程序员 → 专注架构和创新

## 总结

AI 编程正在从"玩具"走向"工具"，再到"伙伴"。

Loop Engineering 代表了下一代 AI 编程的方向：不是一次性生成完美代码，而是通过循环迭代、自我修正，逐步逼近目标。

但核心挑战依然存在：意图理解、长期维护、安全可靠性。这些不是技术问题，而是人机协作的根本问题。

未来的软件工程师，需要学会：

1. **和 AI 协作**：知道什么时候让 AI 做，什么时候自己做
2. **设计 Loop**：把复杂任务拆解成 AI 能执行的循环
3. **把控质量**：AI 生成的代码，最终责任在人
4. **持续学习**：AI 能力在进化，人的能力也要进化

> AI 不会取代程序员，但会用 AI 的程序员会取代不会用 AI 的程序员。
