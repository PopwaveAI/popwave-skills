# Skill 知识工程：团队内部分享

> 创建日期：2026-07-08
> 面向：全体团队成员（工程、产品、运营）
> 目标：讲清楚 Skill 是什么、Agent 怎么用它、能力边界在哪、哪些工作可以 Skill 化、怎么写一个合格的 Skill

---

## 1. 为什么要聊这个

我们的 Agent 平台跑了一段时间，大家可能注意到一个现象：同样一个 Agent，配上不同 Skill 后表现差异巨大。有的 Skill 让 Agent 像换了个人——精准、靠谱、一步不漏；有的 Skill 写了跟没写一样，Agent 该犯的错还是犯。

这背后不是模型聪明不聪明的问题，而是知识工程的问题。Skill 的本质是把人类专家的方法论「编译」成 Agent 能执行的指令。编译质量决定执行质量。

这篇文档把我们在网文创作 Skill 群里踩过的坑、总结出的规范整理出来，让团队每个人都能判断「这个工作要不要 Skill 化」以及「怎么写才合格」。

---

## 2. Skill 是什么

### 2.1 一句话定义

Skill 是一份写给 Agent 看的操作手册，告诉它遇到某类任务时该怎么做——读什么文件、按什么顺序执行、产出什么、什么情况必须退回。

### 2.2 和传统 RAG 的区别

| 维度 | 传统 RAG | Skill |
|:-----|:---------|:------|
| 处理的知识类型 | 陈述性知识（API 参数是多少、文档写了什么） | 程序性知识（遇到这类问题怎么做） |
| 喂给模型的方式 | 把知识碎片塞进上下文，让模型自己猜 | 给完整的操作手册，让模型照着做 |
| 模型行为 | 搜索 → 拼接 → 生成（本质是高级摘要） | 读取 → 执行 → 产出（本质是照章办事） |

RAG 解决的是「模型不知道」，Skill 解决的是「模型知道但不知道怎么做」。一个喂碎片，一个给方法。

### 2.3 技术本质

Skill 不会让 Agent 变得更聪明，但它能让 Agent 的聪明用在正确的地方。它不创造能力，只传递能力。

这句话的含义是：

- **上限** = 源材料里藏了多少可复用的东西，和模型能力无关
- **下限** = 底层模型的能力边界，Skill 不突破这个边界
- **窗口** = 上下文窗口约 128K-200K token，天然适合「一个领域的一套方法」，不是「一个领域的所有知识」

你往 Skill 里塞什么，决定了 Agent 能做什么。瓶颈不在模型的转化能力，在源头的内容质量。

## 3. Agent 如何使用 Skill

### 3.1 三层渐进式披露

Agent 不是一次性把整个 Skill 吃进去。Skill 被设计成三层，按需加载：

```
┌─────────────────────────────────────┐
│ L1: SKILL.md（≤100行，强保障）        │  ← host 层强制注入，100% 到达
│ 「做什么 + 怎么运作（SOP骨架）+ 红线」│
├─────────────────────────────────────┤
│ L2: steps/（视情况，按需加载）         │  ← 步骤多且复杂时才拆，Agent 按需读取
│ 每个 step 的详细操作指令               │
├─────────────────────────────────────┤
│ L3: references/ + templates/（按需）  │  ← Agent 需要细节时读取
│ 方法论、规范参考、产出模板             │
└─────────────────────────────────────┘
```

为什么要分三层？因为上下文窗口有限。但分层不等于「SKILL.md 只做路由」——SKILL.md 是唯一 100% 到达的文档，它的首要职责是让 Agent **完整理解这个 skill 怎么做**，而不是只给一张红线列表和文件索引。分层 = 把 SOP 骨架放第一层（Agent 即使不读 step 也能干活），把详细操作指令和参考资料放在二三层按需加载。

### 3.2 强弱加载保障

这里有一个关键的现实约束：**Agent 不一定会读你给它指的文件。**

我们的 435 runs 数据证明了一个残酷的事实：

| 对话轮次 | Skill 文件读取率 |
|:---------|:----------------|
| 第 1 轮 | 约 3.7% |
| 3 轮之后 | 0-3% |

Agent 在第一轮之后基本就切到了「对话记忆模式」，靠缓存的 Skill 摘要干活，不再主动读 step 文件。

所以 Skill 设计有一条铁律：

- **强保障**：SKILL.md 由 host 层每次 run 强制注入，100% 到达 Agent 上下文
- **弱保障**：steps/、references/、templates/ 需要 Agent 主动 readFile，天然不可靠

**设计 SKILL.md 时，必须假设 step 文件可能没被读到。** 这意味着 SKILL.md 不能只放红线——它必须自包含这个 skill 的 SOP 骨架（step1~n 的简要描述），让 Agent 即使不读任何 step 文件，也知道整个流程怎么做。红线是辅助约束，SOP 骨架才是 SKILL.md 的主体。

### 3.3 Agent 执行 Skill 的实际链路

以拆书管线为例，Agent 收到「拆这本书」指令后的实际执行过程：

```
用户: "拆这本书"
    ↓
Agent 读取所有可用 Skill 的 description（L1 元数据）
    ↓
匹配到 pop-decon（description 含"当用户说'拆书/解构/分析'时启用"）
    ↓
host 层强制注入 pop-decon 的 SKILL.md 全文（强保障）
    ↓
Agent 按 SKILL.md 速查表指引，主动读取 steps/step-1-pipeline.md（弱保障）
    ↓
按 step 文件执行操作：检查源文件 → ETL 拆章 → 逐章提取设计包
    ↓
需要模板时读取 templates/fact-skeleton.md（弱保障）
    ↓
产出设计包文件，写盘
```

整个链路中，只有 SKILL.md 是 100% 到达的。其余文件全靠 Agent 自觉。所以 SKILL.md 不能只是一张红线表和文件索引——它必须包含完整的 SOP 骨架，让 Agent 即使没读到任何 step 文件，也知道「拆书要经过 ETL → 设计包 → 设定 → PRD 四个阶段，每个阶段产出什么」。这是唯一可靠的信息传递通道，不用它传 SOP 就是浪费。

## 4. 能力边界

### 4.1 Skill 能做什么

| 能力 | 说明 |
|:-----|:-----|
| 传递方法论 | 把人类专家的「遇到这类问题怎么做」编译成 Agent 可执行的指令 |
| 约束行为 | 通过红线和门禁，阻止 Agent 做不该做的事（如跳过质量校验） |
| 保证产出一致性 | 通过模板统一产出格式，让不同 Agent、不同 run 的产出可比对 |
| 编译踩坑经验 | 把已知错误写成 error list，让 Agent 不重复踩坑 |

### 4.2 Skill 不能做什么

| 限制 | 说明 |
|:-----|:-----|
| 不创造能力 | 模型不会因为有了 Skill 就突然学会它原本不会的事 |
| 不突破上下文窗口 | 128K-200K token 的限制还在，Skill 只能优化窗口内的信息密度 |
| 不保证 Agent 一定读 step 文件 | 弱保障文件天然不可靠，关键信息必须在 SKILL.md 中 |
| 不替代人类判断 | Skill 传递的是方法，不是决策。该人确认的门禁还是要人确认 |

### 4.3 边界判断实例

以我们的拆书管线为例：

- **Skill 能做到的**：告诉 Agent「Phase 1 未完成不准进 Phase 2」「每 beat 必须有原文证据」「产出文件必须按 chXXX-设计包.md 命名」
- **Skill 做不到的**：让一个没读过小说的模型突然理解叙事节奏、让 Agent 自己判断「这个 beat 是不是凭空发明的」（这需要人类校验）
- **Skill 的补偿手段**：在模板里写 error list——「plotlines_active 里出现不存在的线 id」是已知错误，写进去让 Agent 检查。这是用工程手段补偿模型能力的不足

## 5. 日常工作中哪些可以 Skill 化

### 5.1 换皮测试

判断一个工作能不能 Skill 化，做一个测试：**把这个工作里所有具体项目名、公司名、数据、技术栈名字全部抹掉。看看剩下的部分，读者还能不能从中学到一套可以直接用的方法。**

| 结果 | 结论 |
|:-----|:-----|
| 能 | 骨架是硬的，可以变成好 Skill |
| 不能 | 价值和具体案例粘在一起，删掉案例就什么都不剩，不适合 Skill 化 |

### 5.2 三类知识对应三种 Skill 形态

| 源材料类型 | 特征 | Skill 形态 | Agent 体感 |
|:-----------|:-----|:-----------|:-----------|
| SOP 型（菜谱） | 有明确步骤、先后顺序、操作指引、踩坑经验 | 工作流型——Agent 引导你一步步做 | 真的在帮你干活 |
| 复盘型（项目心得） | 结合特定项目，有少量方法论 | 能力参考型——Agent 随叫随到给方案参考 | 有用但不够惊艳 |
| 分析型（横向对比） | 案例丰富但缺独立分析框架 | 决策参考型——Agent 辅助信息梳理 | 高级摘要机器人 |

**SOP 型最适合 Skill 化。** 如果你的工作有明确的步骤顺序、每步有具体的操作指引和踩坑经验，它天生就是好 Skill 的原料。

### 5.3 适合 Skill 化的工作特征

从我们的实践来看，以下特征的工作适合 Skill 化：

- **有固定流程**：每次做都按类似步骤走（如拆书管线的 Phase 1→2→3→4）
- **有明确产出**：每步都有文件落盘，可以校验（如设计包、卷纲、设定文件）
- **有踩坑经验**：团队踩过的坑可以写成规则避免重复（如「凭空发明 beat」「Phase 1 没完成就跳 Phase 2」）
- **有上下游依赖**：产出被下游消费，可以建立强制消费链路（如 Canvas → yaml 的依赖声明）
- **抹掉具体案例后框架依然成立**：换一本书、换一个项目，这套方法照样能用

### 5.4 不适合 Skill 化的工作

- **一次性任务**：做完不会再做，没有复用价值
- **高度依赖创意和灵感**：没有固定流程可以提炼
- **纯信息查询**：用 RAG 或搜索就够了，不需要 Skill
- **案例占比超过 80%**：抹掉案例后什么都不剩，Skill 化只是做了个摘要

## 6. Skill 书写规范

### 6.1 四层架构

```
my-skill/
├── SKILL.md          ← 骨架层：做什么 + 怎么运作（SOP骨架）+ 红线，≤100行
├── skill.json        ← 机读：平台元数据
├── CHANGELOG.md      ← 变更历史
├── steps/            ← 展开层：当步骤多且复杂时，展开详细操作指令
├── references/       ← 知识层：方法论、规范参考（读后理解）
└── templates/        ← 模板层：空模板、示例（复制填充产出）
```

| 层级 | 职责 | 行数目标 |
|:-----|:-----|:---------|
| 骨架层 (SKILL.md) | 做什么 + 怎么运作（step1~n 简述）+ 红线 + 速查表 | ≤100 行 |
| 展开层 (steps/) | 详细操作指令——视情况，不为拆而拆 | 50-150 行/文件 |
| 知识层 (references/) | 方法论、规范参考 | 按需加载 |
| 模板层 (templates/) | 空模板、输出格式骨架 | 按需加载 |

**SKILL.md 是 skill 的完整骨架。** 它必须自包含：这个 skill 做什么、怎么运作（step1~n 简要描述）、红线、速查表。因为 SKILL.md 是强保障（100% 注入），它是 Agent 唯一可靠的信息来源——不在这里写 SOP 骨架，等于浪费唯一的机会让 Agent 理解整个流程。红线是辅助，SOP 骨架才是主体。

**steps/ 是视情况的。** 简单 skill 可以没有 steps/，直接在 SKILL.md 里写完。只有当步骤多且单步复杂、SKILL.md 的简述放不下时，才拆到 steps/ 展开详细叙述。不为拆而拆。

### 6.2 SKILL.md 标准结构

```markdown
---
name: my-skill
description: "当用户说'做X/做Y'时启用。产出来物供给下游管线。"
---

# my-skill

> 一句话摘要

## 这个 Skill 做什么
（2-3句话说明职责和产出）

## 怎么运作
Step 1: xxx → 产出 xxx
Step 2: xxx → 产出 xxx
Step 3: xxx → 产出 xxx
（每个 step 一两句话简述，让 Agent 即使不读 step 文件也知道整个流程）

## 红线
| # | 红线 |
|:-:|:-----|
| ❌1 | **读取协议** — 用 Get-Content -Encoding UTF8 -Raw，禁用 Read 工具 |
| ❌2 | ... |

## 速查表（全文件目录引导）
| 我要 | 读什么文件 | 什么时候读 |
|:-----|:----------|:----------|

## 版本
vX.Y.Z | YYYY-MM-DD | 一句摘要 → [CHANGELOG.md](CHANGELOG.md)
```

内容优先级：**做什么 → 怎么运作（SOP 骨架）→ 红线 → 速查表**。SOP 骨架是主体，红线是辅助。如果 SKILL.md 放不下全部内容，优先压缩红线和速查表，保住 SOP 骨架。

### 6.3 红线怎么写

红线是 SKILL.md 的辅助约束层，不是主体。主体是 SOP 骨架——Agent 得先知道怎么做，红线才有意义。判断一条规则该不该当红线，标准是：**违反后会不会断裂下游流程？**

| 是红线 | 不是红线 |
|:-------|:---------|
| 读取协议（截断会丢内容） | frontmatter 不超过4行（格式偏好） |
| 双文件结构（缺文件平台无法注册） | description 含触发词（写法要求） |
| 版本三处一致（不一致导致 registry 错乱） | 红线在第一屏（布局建议） |

红线要求：

- **≤7 条**——红线太多了约等于没有红线
- 每条包含「触发条件 + 后续动作」
- 第一条必须是读取协议
- 统一使用禁令式：「必须 / 禁止 / 退回」，不用「建议 / 尽量 / 可以考虑」

### 6.4 skill.json 标准字段

```json
{
  "id": "my-skill",
  "version": "1.0.0",
  "displayName": "我的技能",
  "description": "当用户说'做X/做Y'时启用。产出来物供给下游管线。",
  "entry": "SKILL.md",
  "activation": {
    "slashCommands": ["my-skill", "做X"]
  },
  "permissions": {
    "readProjectFiles": true,
    "writeProjectFiles": true
  }
}
```

版本号必须在 SKILL.md、skill.json、CHANGELOG.md 三处保持一致。

### 6.5 description 写法

description 是 Agent 判断「要不要启用这个 Skill」的唯一依据。写不好，Agent 就不会在正确的时机调用你的 Skill。

```
❌ 差：description: "用户调研工具"
❌ 差：description: "五模式设计：模式A→B→C→D→E"
✅ 好：description: "当用户说'调研/社区/什么火'时启用。产出调研报告供给下游管线。"
```

必须含「当用户说…时启用」触发条件。说明做什么 + 何时触发 + 产出给谁。

### 6.6 references/ vs templates/ 的区别

这两个目录容易搞混：

| 目录 | 定位 | 使用方式 | 典型文件 |
|:-----|:-----|:---------|:---------|
| references/ | 知识层 — 读后理解 | 阅读理解，指导操作 | 搜索 SOP、调用规范、设计原理 |
| templates/ | 模板层 — 复制填充 | 复制 → 填充 → 产出文件 | 空白模板、输出格式骨架 |

模板文件不放 references/。SOP 文件不放 templates/。

### 6.7 行数预算

| 文件 | 目标行数 |
|:-----------|:--------:|
| SKILL.md（含 SOP 骨架） | ≤100 行 |
| steps/ 单文件 | 50-150 行/文件 |
| references/ 单文件 | 按需 |
| templates/ 单文件 | 按需 |

SKILL.md 放宽到 100 行，是因为要容纳 SOP 骨架。这不是浪费——SKILL.md 是 100% 到达的文档，多写 40 行 SOP 简述，比让 Agent 去读一个可能不会被读到的 step 文件划算得多。超 100 行时，先看 SOP 骨架能不能再压缩（每步一两句话），再考虑拆 step。

### 6.8 文件拆分原则

**steps/ 拆分是视情况的，核心原则是「不为拆而拆」：**

简单 skill（步骤少、单步不复杂）→ 不需要 steps/，直接在 SKILL.md 的「怎么运作」里写完。SKILL.md 的 SOP 骨架如果已经足够让 Agent 执行，就没有理由拆出去。

复杂 skill（步骤多、单步操作复杂）→ SKILL.md 写简述（每步一两句话），steps/ 展开详细叙述。只有当 SKILL.md 的简述不够 Agent 执行时，才拆。

| 该拆 | 不该拆 |
|:------|:--------|
| 单步操作复杂，SKILL.md 简述放不下 | 步骤简单，SKILL.md 简述已够用 |
| 有用户确认断点（天然分水岭） | 连贯推导链中间硬切 |
| 两个阶段彼此独立 | 两个阶段必须顺序执行且共享上下文 |

拆分决策：SKILL.md 的 SOP 骨架不够用？→ 单步太长？→ 有用户断点？→ 按断点拆 / 有知识层可下沉？→ 下沉 / 都没有 → 不拆，在 SKILL.md 里写完。

**references/templates**：越细越好。Agent 只在需要时读对应文件，不拆 = 每次全读浪费 token。

### 6.9 模板质量要求

每个 templates/ 下的模板文件必须通过「换皮测试」——抹掉所有具体案例后，指南依然有效。具体要求：

- 有字段说明（每个字段的含义、可取值、示例）
- 有使用场景说明（这个模板在什么时机使用）
- 有上下游关系（输入来自哪个上游文件、产出被哪个下游消费）
- 有常见错误表（至少 5 条已知踩坑经验）
- 有产出自检清单（填完模板后的验收标准）

好的模板 = 把「红烧这道菜的做法」升级为「红烧的方法」——换一条鱼、换一块豆腐，这个方法照样能用。

## 7. 五条核心原则

| # | 原则 | 含义 |
|:-:|:-----|:-----|
| 1 | SKILL.md 是骨架，不是路由表 | 100% 到达的文档必须自包含 SOP 骨架，Agent 不读 step 也能干活 |
| 2 | 抽象层级决定复用性 | 抹掉案例后骨架能否独立——能才是好 Skill |
| 3 | step 视情况拆，不为拆而拆 | 简单 skill 在 SKILL.md 里写完；复杂 skill 才拆 steps/ 展开详细叙述 |
| 4 | 不创造能力，只传递能力 | Skill 不让 Agent 变聪明，但让聪明的方向对 |
| 5 | 源材料质量 = 瓶颈 | Skill 上限取决于你往里塞了什么，模板是踩坑经验的补偿 |

## 8. 创建一个新 Skill 的流程

1. 向用户确认 Skill 名称和职责
2. 创建目录 `{skill-name}/`
3. 创建 `SKILL.md`（≤60 行，按标准结构）
   - 红线第一条必须是读取协议
   - 速查表必须是全文件目录引导
4. 创建 `skill.json`（按标准字段）
5. 创建 `CHANGELOG.md`（初始 v1.0.0）
6. 如有 steps/references/templates 需求，创建对应目录
7. 对照检查清单逐项验证
8. 输出创建完成报告（文件路径列表）

改造存量 Skill 时，顺序是：先读全文 → 输出扫描报告 → 只改结构/格式/红线位置 → 版本+1 三处同步 → 输出 BEFORE/AFTER 对比。不读全文直接动手是红线行为。
