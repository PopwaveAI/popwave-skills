# Bookstrap Phase 0 优化 PRD — 卖点与需求挖掘太浅的问题

> 版本：v1.0 | 2026-06-12
> 状态：分析阶段，待入库 储备方案/

---

## 一、问题诊断

### 1.1 核心问题：story-engine 的产出物没有"验证回路"

当前流程：
```
用户说"我想写X" 
  → Phase 0: 追问2-3轮 → 产出 story-engine.yaml 
  → 用户说"对" → 锁定 
  → 进 Phase 1 L1 设定 → ... → Phase 6/7 快照
  → 交 plot → 写了 50 章才发现"这个故事读者不买账"
```

**断裂点**：story-engine 锁定得太早，而且锁定后就没有人再回头检视它了。L1 设定全部基于一个未经外部验证的假说展开——50 章后方向偏了，源头就是 Phase 0 的卖点挖掘深度不够。

### 1.2 三个表层追问不够用

当前的追问方向表（来自 phase-0.pe.md）：

| 追问方向 | 问题本身 | 产生的答案类型 |
|:---------|:---------|:--------------|
| 场景追问 | "这个世界最关键的一个画面是什么？" | 画面描述（印象） |
| 冲突追问 | "主角面临的最大困难是什么？来自哪里？" | 冲突类型（战斗/探索/...） |
| 角色追问 | "主角是个什么样的人？他想做什么？怕什么？" | 角色描述 |
| 基调追问 | "这个故事的整体氛围是黑暗还是轻松？" | 氛围形容词 |
| 设定边界 | "这个世界跟现实世界最大的不同是什么？" | 差异点 |
| 读者预期 | "读者看完前三章，应该记住什么？" | 内容承诺 |

**问题**：这些追问全部是**主观感受层面的**。用户说"黑暗基调"，后续没有人问"你说的黑暗是什么程度的黑暗？像《道诡异仙》那种疯癫黑暗，还是像《诡秘之主》那种克系疏离黑暗？"

### 1.3 缺少"对标验证"环节

Phase 0.3 有参考书甄别，但它在 Phase 0 之后才执行，而且它的目标不是**验证 story-engine 的假说是否成立**，而是**寻找差异化定位**——这意味着如果在 Phase 0 锁定的方向本身有问题（比如用户说的"卖点"放在市场上其实没人看），Phase 0.3 也无法及时纠偏。

更关键的是：**story-engine.yaml 里没有"这本书的卖点是否经过验证"这个字段**。一个未经市场验证的个人幻想 = 一个可能写到 50 章才发现方向偏了的故事。

### 1.4 与拆书产出的脱节

现在 Lv1 拆书（Phase S）已经能产出 `story-engine.yaml`（参考书的）、主角参考卡、卷1快照。但 bookstrap Phase 0 **没有消费这些产出的入口**。

问题链：
1. 用户说"我要写一本像《吞噬星空》的科幻末世"
2. Lv1 拆书跑完 → 拿到罗峰的 story-engine、角色卡、卷1快照
3. bookstrap Phase 0 开始追问 → **没有加载拆书产出** → 用户说的"像吞噬星空"变成一个模糊的标签，而不是一个结构化的对照物

## 二、根因分析

| # | 根因 | 症状 |
|:--|:-----|:------|
| 1 | **story-engine 没有"外部验证"维度** | 所有字段都是用户主观陈述，无市场对照 |
| 2 | **追问深不深完全靠 Agent 临时判断** | 没有结构化的验证 checklist，Agent 问完 2-3 轮就觉得够了 |
| 3 | **Phase 0 与 Phase 0.3 是线性关系不是验证回路关系** | Phase 0.3 的差异化分析不反哺 story-engine |
| 4 | **拆书产出在 Phase 0 没有消费入口** | 用户说了对标书但 Phase 0 不看，到 Phase 0.6 才看——但那时 story-engine 已经锁了 |
| 5 | **"卖点"没有被拆解为可验证的子命题** | 用户说"升级爽"——几章升一次算爽？参考书是几章？不知道 |

## 三、优化方向

### 方向 1：story-engine 增加"卖点验证回路"

```
当前：
  Phase 0 锁定 story-engine → 后面所有人都拿这个当圣旨

优化后：
  Phase 0 锁定初版 story-engine
    → Phase 0.3 差异化分析反哺 story-engine（发现参考书实际卖点和用户说的不同 → 修正）
    → Phase 0.5 跨域素材反哺 story-engine（素材注入后修改 world_anchor）
    → Phase 0.6 拆书融合反哺 story-engine（参考书的 reader_promise 和用户说的不一致 → 重新确认）
    → Phase 1-7 维护 story-engine 的更新日志
    → plot 入口校验：如果 story-engine 版本号比上次进 plot 时更新了，重新检视
```

story-engine.yaml 增加字段：

```yaml
engine:
  version: 3     # 新增 v3
  core_premise: ""
  
  # 新增：卖点验证
  selling_point_validation:
    status: "unverified / validated / need_revision"
    validated_against:
      - source: "对标书"    # 来自 Phase 0.3 差异化分析
        finding: "参考书实际卖点是A，用户说想要B，两者差异是C"
      - source: "拆书融合"  # 来自 Phase 0.6
        finding: "参考书的读者预期是X章一个Y，用户预期是Z，是否需要调整？"
    last_validated_at: "Phase X"
    revision_count: 0      # story-engine 被修正的次数
```

### 方向 2：标准化"卖点拆解"清单

不是让 Agent 自由发挥追问，而是给一个**结构化的卖点验证矩阵**：

| 维度 | 用户原始说法 | Agent 追问目标 | 参考数值（来自拆书） | 最终确认 |
|:-----|:-----------|:--------------|:------------------|:--------|
| **升级节奏** | "升级爽" | 几章一次有效实力反馈？ | 吞噬星空卷1：平均8章一次等级突破 | 确认值：X章一次 |
| **战斗密度** | "战斗多" | 战斗场景占多少比例？日常 vs 战斗的章比例？ | 参考书卷1：战斗章占比40% | 确认值：X% |
| **世界观释放** | "设定有意思" | 前半卷放出多少%的世界观？读者何时完全理解核心设定？ | 参考书：第5章释放核心设定 | 确认值：第X章 |
| **情绪弧线** | "虐主再翻身" | 压抑章和爽章的比例？低谷持续最长几章？ | 参考书：压抑≤3章必给一个甜头 | 确认值：X |
| **信息密度** | "脑洞大" | 每章平均几个新信息点？类型分布？ | 参考书：平均3个/章 | 确认值：X个/章 |

**关键理解**：用户说的"升级爽"是一个感受——"每8章升一级"才是结构化的卖点。后面的 plot 和 chapter-design 需要的是后者，不是前者。

### 方向 3：拆书产出 → Phase 0 的消费协议

当前拆书 Lv1 产出的 `story-engine.yaml` 含有 `reading_level` 字段（核心卖点清单）。bookstrap Phase 0 应该：

1. 用户说"对标《吞噬星空》"
2. 检查 `_参考书/吞噬星空/` 下是否有 Phase S 产出
3. 有 → 加载 `story-engine.yaml` 的 `reading_level` 和 `protagonist_drive`
4. Phase 0 追问时用这些做**参考锚点**：

```
Agent："你说要像《吞噬星空》一样的升级爽感。罗峰的升级节奏是平均8章一次突破。
        你希望你的主角也是这个节奏，还是更快/更慢？"
```

而不是：

```
Agent："升级节奏大概是什么样的？"
```

### 方向 4：降低 story-engine 的"承诺风险"

当前 Phase 0 锁定的 story-engine 会被后续所有阶段视为权威。应该改成：

```
Phase 0 → 产出 "story-engine-v1.yaml（初版）"
Phase 0.3 → 基于差异化分析，可能产出 "story-engine-v2.yaml（修正版）"
Phase 0.6 → 基于拆书融合，可能产出 "story-engine-v3.yaml（确认版）"
Phase 1 → L1 设定阶段如果发现 engine 不合理 → 再修一版
```

每次版本变更的记录写入 `engine.version_history`。

## 四、与全链路的关系

### 4.1 拆书 Lv1 → Phase 0 最短路径

```
拆书 Lv1（Phase S）
  ↓
产出：参考书的 story-engine.yaml + 主角参考卡 + 卷1快照
  ↓
bookstrap Phase 0 检测到 _参考书/{对标书}/
  ↓
加载参考引擎 → 追问时用参考值做锚点 → 产出初版 story-engine
  ↓
Phase 0.3 差异化分析 → 确认或修正
  ↓
Phase 0.6 拆书融合 → 基于 T1/T4/T5/T7 再次修正
  ↓
→ 移交 plot 的 story-engine 至少经过 2 轮外部验证
```

### 4.2 与 Layer 2 拆书 PRD 的关系

方向 3 和方向 4 都依赖拆书 Lv1 稳定产出，且依赖 bookstrap Phase 0.3/0.6 的消费协议升级——这正好是之前**拆书v3.0-分级拆解PRD** 中定义的 Layer 2 范围。

**所以 bookstrap Phase 0 优化的依赖链是：**

```
拆书 Lv1 Phase S ✅（刚落地）
  → 拆书 Lv2/Lv3 消费协议 ⏳（Layer 2 待命）
    → bookstrap Phase 0 消费拆书产出 ⏳（本 PRD，Layer 2 的一部分）
      → plot Step 4.5/9 消费拆书产出 ⏳（Layer 2 的一部分）
```

### 4.3 是否等下游稳定？

**和拆书 Layer 2 一样：不卡在下游，先消化能消化的。**

可以现在就做的：
- **方向 1**：story-engine.yaml 加 `selling_point_validation` 字段，代码改动只在 bookstrap 内部
- **方向 2**：Phase 0 加"卖点验证矩阵"——不需要任何外部依赖，只是追问框架升级

需要等拆书产出稳定的：
- **方向 3**：Phase 0 消费拆书产出——需要等拆书 Lv1 的 Phase S 稳定运行一轮后再改消费端
- **方向 4**：story-engine 多版本迭代——需要 Phase 0.3/0.6 先有一个稳定的反哺协议

---

## 五、建议优先级

| P0 | 方向 1 + 方向 2 | 现在就能改，不影响任何下游 |
|:---|:---------------|:------------------------|
| P1 | 方向 3（Phase 0 消费拆书） | 等拆书 Phase S 稳定运行一次后推进 |
| P2 | 方向 4（story-engine 多版本） | 等方向 3 落地后再做 |
