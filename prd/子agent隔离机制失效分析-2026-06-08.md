## 一、设计意图

expert-writer 元技能的设计中，子agent扮演的关键角色是**强制上下文归零**：

> "执行时启动子agent，不继承主agent的历史上下文，防止污染"
> —— expert-writer SKILL.md

设计者预判的风险："前一步说了什么"会污染"后一步该做什么"。子agent通过完整的隔离会话来解决这个问题——每次路由到子skill，子agent从零启动，只看到自己的输入文件，不受主agent记忆的干扰。

## 三、五个实际后果（均来自会话证据）

### 后果 1：思考级上下文污染

**现象：** 主agent在同一段推理流中混合了原本应分属不同角色的逻辑。

**证据：** ch020 写作的思考日志（~39KB），截取结构：

```
1. 重新读 act-01.yaml → 确认 ch020 的爽点类型
2. 担心节奏问题 → "等等，act-01.yaml 说 ch20 是幕末，但第四幕才是血祭天蓬……"
3. PRD 自查 → "嗯 PRD 说 ch145-150 才是终极……"
4. 自我打断 → "但剧情需要 ch020 释放……"
5. 跳回去确认 ch019 的结尾状态
6. 又考虑要不要启动子agent
```

**本应是谁做的：**
| 事 | 本应由谁做 | 实际谁在做 |
|:---|:----------|:----------|
| 决定"不释完整天蓬咒，留第四幕" | Director（章纲设计子skill） | 主agent |
| 获得这个决策去执行 | Writer 子agent | 同上 |
| 质检这个决策是否正确 | QC 子agent | 同上 |
| 纠结子agent是否启动 | runtime 层 | 同上 |

**影响：** 三个层次（设计决策、执行规划、运行管理）混合在同一段推理流中。每一轮思考都拆分困难、逻辑混乱、token浪费。

---

### 后果 2：决策前后传导

**现象：** 前一步处理形成的"模糊认知"传导到后一步的事实判断。

**证据：** 海贼法典 CH389 多弗朗明哥关系线错误。

```
决策传导链：

Step 1 [探索/阅读阶段]
  → 主agent初次扫描海贼法典全书（388章）
  → 形成记忆："多弗是一个反派，喜欢试探和掌控别人"
  → 这个记忆是**粗略的、基于印象级的**，不是精确的

Step 2 [向导/分析阶段]  
  → 主agent做拆书分析、写节奏地图
  → 在上述记忆基础上工作，没有逐章验证
  → 形成"多弗和威廉的关系是敌对+试探"的认知基调

Step 3 [写作阶段]
  → 写 CH389（多弗和威廉同框）
  → 直接调用 Step 2 的认知基调
  → 写成"多弗试探威廉这个陌生人"
  → 但原文事实是：CH372 两人已经私下谈妥合作

Step 4 [被用户质问后]
  → 精读倒数20章，发现偏差
  → 用户质问："因为读者不知道他们私下谈过——这是什么逻辑？"
  → AI 承认："这不叫表演型人格，这叫吃设定"
```

**关键洞察：** 这个错误不是单点决策错误——它是一级一级传导过来的。如果 Step 1→Step 2→Step 3 各自是独立子agent，每个都会从原文文件重新读取事实，Step 2 就不会继承 Step 1 的模糊印象。

**影响：** 事实错误。而且是被用户一眼就能看出来的事实错误——信任伤害最大。

---

### 后果 3：精读被"记忆"绕过

**现象：** 主agent因为"记得自己已经读过"，跳过了必要的前文精读。

**证据链（海贼法典会话）：**

```
用户: "写389的时候 有读前文吗"
AI: "读了，但读得不够。CH388 末尾全部读完了……CH388之前那10-20章没有逐章重读"
用户: "倒数20章你都精度一下"

→ 精读后发现5个关键信息偏差
  1. 多弗已合作（非陌生人）
  2. 卡普交接时汉库克在楼上看
  3. 卡斯兰娜是海军线人
  4. 古罗莉欧萨观察的是蒸汽船
  5. 战国拦着卡普（大局为重态度）
```

**根本矛盾：**
- 隔离有效时 → 子agent从零启动 → **必须**读原文文件 → 精读不会被跳过
- 隔离失效时 → 主agent"觉得我已经看过了" → **跳过**读文件 → 依赖记忆写作

这不是"懒惰"——而是上下文持续累积导致的**认知错觉**。主agent的上下文里确实存在过"海贼法典"的内容，它"知道自己见过"，但它分不清"见过"和"最近 N 章逐字精读"之间的信息密度差异。

---

### 后果 4：QC 变成自我认可

**现象：** QC 自检与 writer 共享同一个上下文，所以 QC 检不出 writer 的事实错误。

**证据：** CH389 的 QC 自检报告查了什么？

| 检查项 | 结果 |
|:-------|:----:|
| 字数合规 | ✅ |
| 句式检查 | ✅ |
| 节奏评估 | ✅ |
| 爽点分布 | ✅ |
| **多弗关系线是否正确（事实一致性）** | **❌ 未检查（也未发现）** |

为什么检不出来？因为 QC "知道" writer 想表达什么。

```
writer 的意图：多弗在试探威廉（一个"陌生人"）
QC 的视角：同上（共享上下文）
QC 的结论：✅ 表达一致

→ 但原文事实是：多弗和威廉已经是盟友
→ QC 和 writer 都不知道这个事实存在偏差
```

如果 QC 是独立子agent，它会做的流程：
1. 从零启动
2. 读原文设定 + 原文最后 N 章
3. 读 CH389 正文
4. 逐条核对："多弗和威廉的关系在设定/原文中是'已合作'，在正文中是'试探' → **不一致**"

**影响：** 管线中最关键的质控环节变成了"作者自己的再次确认"，而非独立审计。

---

### 后果 5：思考膨胀 + token 浪费

**现象：** 主agent反复加载已读文件，思考日志异常膨胀。

**定量对比：**

| 任务 | 正常子agent预期 | 实际主agent执行 | 膨胀倍率 |
|:-----|:--------------:|:---------------:|:--------:|
| v2 ch01 重写思考日志 | ~50KB | ~**188KB** | **3.8x** |
| 单次 writer 执行 | ~30-50KB | ~80-120KB | **2-3x** |
| 单次 plot 执行 | ~20-40KB | ~55KB | **1.5-2x** |

**浪费在哪：**
- 重新读 SKILL.md（子agent启动时会自动加载，但主agent要自己读）
- 重新读审视框架（think-正文写作.md 等）
- 重新读 act-XX.yaml
- 自己跟自己做设计决策的辩论（写 ch020 时的 39KB 思考日志）
- 考虑"要不要启动子agent"这个 meta 层面的决策（每次都要）

## 四、核心问题链

```
隔离失效
   │
   ├──→ A. 决策不隔离
   │       └──→ 多弗线：从"已合作"错误传导为"试探"
   │
   ├──→ B. 精读不隔离
   │       └──→ 跳过前文精读，用"记忆"代替"读取"
   │
   └──→ C. QC 不隔离
           └──→ 自己审自己，事实错误审不出来
```

**三条不隔离 → 一条完整的事故链：**

```
Step 1: 主agent粗读全文 → 形成模糊认知
Step 2: [无隔离] → 模糊认知传递到拆书/分析阶段
Step 3: [无隔离] → 模糊认知传递到写作阶段
Step 4: [无隔离] → 跳过精读验证，依赖记忆写作
Step 5: 写错（多弗变成"试探的陌生人"）
Step 6: [无隔离] → QC 无法发现错误
Step 7: 用户发现 → 信任成本 + 重写成本
```

这个链条上的每一步，"有隔离"时都会被打断——因为每步都要从文件重新读取数据。

## 五、修复方向讨论

### 方向 A：修好子agent（根治）

恢复 gateway 连接 + 修复 OpenClaw 的硬编码模块路径。子agent启动成功后，隔离机制自动恢复。

**风险：** 从记录看，问题持续存在至少 5 天且未恢复，可能不是简单的配置修复。

---

### 方向 B：在现有架构下补闸门（替代方案）

如果不依赖子agent，则需要**在流程层面重建隔离效果**。核心原则是：

> 每个步骤的输入必须来自**文件系统**，而非来自**上下文记忆**

具体建议：

| 问题 | 替代措施 | 放置位置 |
|:-----|:---------|:---------|
| 精读被跳过 | 写前精读清单：强制读最后N章，输出事实提取报告 | pop-novel-writer Step 1 (Design) |
| 决策传导 | 每步骤输入输出协议：写死读哪些文件、产出哪些文件 | expert-writer 纪律协议 |
| QC 自我认可 | QC 强制读原文设定作为独立输入源 | pop-novel-qa 输入集 |
| 思考膨胀 | 步骤粒度协议：禁止在步骤A完成之前质疑步骤A | expert-writer 纪律协议 |
| 风格不执行 | Render 阶段要求风格文件内容显式写入 prompt | pop-novel-writer Step 2 (Render) |

---

### 方向 C：去掉子agent要求 + 补闸门（务实选择）

SKILL.md 中不再假设子agent可用，而是把隔离纪律写进流程规范本身。

**核心变更：**

```
当前: "路由到子skill → 启动子agent → 子agent做"
改为: "路由到子skill → 加载Skill规范 → 按以下步骤手动执行"
```

这意味着每个子skill 的步骤定义需要足够精细（读什么文件→做什么判断→出什么结果），让主agent在没有隔离的上下文中也能**机械地执行**，而不是**自由地推理**。

---

## 六、结论

子agent隔离失效不是"偶尔不好用"——它是**架构依赖失效**。整个 SOP 管线在假设"子agent会正常隔离"的基础上设计，但这个假设自始至终没有成立过。

在隔离缺席的情况下，产生了可追溯的 5 个负面后果，其中多弗线的事实错误是伤害最大的单点事件。

**修复的下限是：** 写入一个"主agent不依赖隔离也能正确执行"的流程协议。
