## 零、一句话核心

> **全链路图谱定义了"一本好小说的完整骨架"。拆书的本质，就是把这本参考书的骨架从血肉中还原出来，让它以同一种文件格式站在写作管线旁边，随时被参考。**

当前拆书产出的 T1~T7 是孤立的一套分类体系——和 bookstrap/plot/chapter-design 消费的文件**格式不同、维度不同、用不起来**。改造的核心就是把拆书产出对齐到全链路文件结构，让参考书分析直接成为写作管线的"参考答案"。

## 一、问题诊断：当前拆书的三个断裂

### 断裂 1：产出格式与消费格式不同

```
当前：
  拆书产出 T1~T7 (纯分析报告，人读)
          ↓ 用户自己对比
  写作管线产出的 story-engine / volume-XX / act-XX / 设计包 (结构化，Agent 读)

问题：Agent 读不了分析报告，人得手工把"参考书的精华"搬运到写作管线里。
```

### 断裂 2：维度覆盖不全

| 全链路核心文件 | 当前拆书覆盖 | 差距 |
|:------|:------|:-----|
| **story-engine（核心假说）** | ❌ 没有 | 这本书到底在讲一个什么本质命题？当前拆书不提取 |
| **起终点快照（卷级状态对比）** | ❌ 没有 | 角色在哪一章从什么心态变成什么心态？当前拆书不追踪 |
| **info_release_plan（信息释放规划）** | ❌ 没有 | 作者在第几章向读者释放了什么信息？当前拆书完全不分析 |
| **Lv1~Lv4 角色分级** | ❌ 没有 | 路人/功能/配角/主角混在一起拆，不知道谁承担什么叙事功能 |
| **数值体系** | ⚠️ 有概念无结构 | T1 有力量体系，但没有"段位排期""碰撞曲线"这类结构 |
| **剧情结构** | ⚠️ 只有宏观三幕 | 没有"卷-幕-章"三级拆解 |
| **爽点分布** | ❌ 没有 | 没有按幕标记爽点/情绪弧线/读者体验曲线 |

### 断裂 3：拆书不控制时间成本

当前拆书一条路走到底——下载完就 Phase 0→4 全跑。用户要等很长时间才能拿到第一批有用信息。期间用户流失风险极高。

**核心问题：用户需要的不是"完整的分析报告"，而是"第一批能立刻辅助开书的信息"。**

## 二、核心理念：全链路图谱 = 拆书产出模板

### 2.1 逻辑链

```
全链路图谱
  = "一本好小说在 popwave 体系下的完整文件结构"
  = story-engine.yaml + L1-01~06 + 角色卡(Lv1~Lv4) + 数值体系 
    + 全书架构 + 卷/volume-XX + 幕/act-XX + 设计包 + 文风DNA

反转视角：

拆解一本参考书
  = "把这本书还原成全链路图谱的文件结构"
  = 产出和写作管线同格式的 story-engine / 角色卡 / 卷分析 / 信息释放规划 / 文风DNA
```

### 2.2 核心价值

拆书产出的不再是"给人看的分析报告"，而是**和写作管线同一个格式的参考文件**。当用户说"我要写一本像《吞噬星空》一样的科幻"，拆书产出的 `_参考书/吞噬星空/story-engine.yaml` 可以被 bookstrap 直接当参考答案读。

### 2.3 与 character-schema 的协同

character-schema Lv1~Lv4 定义了"角色有多少维度"。拆书时把参考书的角色按这个分级体系归类，产出不同 Lv 的角色参考卡。当用户写自己的角色时，character-schema 模板是**兜底的最低保真度模板**，拆书产出的角色参考卡是**用户认可的、具体赛道的最高保真度模板**。

## 三、分级拆解体系：Lv1 / Lv2 / Lv3

### 3.1 设计原则

| 原则 | 说明 |
|:-----|:------|
| **Lv1 不阻塞开书** | Lv1 拆完 = 用户立刻拿到 story-engine + 主角参考卡 + 卷1快照，可以直接进 bookstrap |
| **每升一级追加，不重写** | Lv2 在 Lv1 基础上追加，Lv3 在 Lv2 基础上追加。存量信息不删 |
| **每级有明确的"可以开始写作"门槛** | Lv1 拆完就可以辅助开书；Lv2 拆完可以辅助 plot 做卷设计；Lv3 是全量参考 |

### 3.2 Lv1 · 快速扫描（~20-30 分钟）

```
目标：30 分钟内拿到第一批能立刻辅助开书的信息
范围：前 10-20 章（精读） + 目录级全卷概览（跳读）
人机分工：Agent 全自动（不需要人在场等）
```

**产出物清单：**

| # | 产出文件 | 对应全链路文件 | 产出方式 |
|:--|:---------|:--------------|:---------|
| 1 | `story-engine.yaml` | `00-原始设定/L0-产品层/story-engine.yaml` | Agent 从开头提取核心假说 |
| 2 | `Lv4-主角参考卡.md` | `状态/角色/{主角}-角色卡.md` | 按 character-schema Lv4 模板填写 |
| 3 | `卷1起点快照.md` | `00-原始设定/起点快照.md` | 第 1 章开篇状态 |
| 4 | `卷1终点快照.md` | `00-原始设定/终点快照.md` | 第 100 章（或卷末）终点状态 |
| 5 | `快速文风指纹-top5.md` | `写作资产/文风DNA/` (简化版) | 前 20 章统计：句式/节奏/对话比/描写密度等 Top 5 规则 |
| 6 | `Lv1-拆解摘要.md` | —（索引文件） | 一句话概括：这本书的核心卖点是什么 |

**Lv1 完成后的用户状态：**
- 知道了参考书的核心假说（"这本书到底在讲什么"）
- 知道了主角的完整角色卡（按 Lv4 标准）
- 知道了卷 1 的起点和终点状态（"主角从哪里走到哪里"）
- 有了 5 条可执行的文风规则
- **可以立刻启动 bookstrap，不再等待**

### 3.3 Lv2 · 结构拆解（~1-3 小时）

```
目标：完整还原参考书卷1的所有结构维度
范围：第一卷全部章节（精读）
前置：Lv1 已产出且用户确认
人机分工：Agent 自动跑，产出后用户审阅
```

**在 Lv1 基础上追加的产出物：**

| # | 产出文件 | 对应全链路文件 | 产出方式 |
|:--|:---------|:--------------|:---------|
| 7 | `卷1-volume-analysis.md` | `设计/卷/volume-01.md` | 还原卷1的角色池/地点池/剧情线/势力动机/卷级快照 |
| 8 | `卷1-act-mapping.yaml` | `设计/幕/vol-01/act-01~0N.yaml` | 卷1拆成 3-6 个幕，每幕标注情绪弧线 + 爽点类型 |
| 9 | `卷1-info-release-plan.md` | `act-XX.yaml § info_release_plan` | 作者在卷1的哪些章节向读者释放了哪些关键信息 |
| 10 | `Lv3-重要配角参考卡.md × N` | `状态/角色/{配角}-角色卡.md` | 按 character-schema Lv3 模板填写，每个重要配角一张 |
| 11 | `Lv2-功能角色池.md` | `状态/角色/龙套池.md` | 按 character-schema Lv2 模板汇总 |
| 12 | `T1-力量体系规则手册.md` | `00-总控/数值体系/combat_capability.yaml` | 升级版：增加段位排期推测 |
| 13 | `T2-世界观展开节奏.md` | `L1-01 世界蓝图` | 升级版：标注世界观信息的释放节奏 |
| 14 | `Lv2-拆解摘要.md` | — | 更新摘要，标注 Lv2 新增发现 |

**Lv2 完成后的用户状态：**
- Lv1 全部 + 卷1结构全貌
- 知道了每个幕的情绪弧线和爽点分布
- 知道了作者的信息释放节奏
- **可以辅助 plot 做卷级设计**

### 3.4 Lv3 · 全书深拆（数小时~天级）

```
目标：完整还原参考书的全书结构
范围：全书所有章节（精读 + sampling）
前置：Lv2 已产出且用户确认
人机分工：Agent 自动跑，产出后用户审阅
```

**在 Lv2 基础上追加的产出物：**

| # | 产出文件 | 对应全链路文件 | 产出方式 |
|:--|:---------|:--------------|:---------|
| 15 | `全书架构.md` | `设计/全书架构.md` | 还原全卷蓝图 |
| 16 | `volume-02~0N-analysis.md` | `设计/卷/volume-02~0N.md` | 每卷一份 |
| 17 | `act-mapping-all.yaml` | `设计/幕/vol-02~0N/act-*.yaml` | 所有幕的情绪弧线+爽点映射 |
| 18 | `entity-snapshot-chain.yaml` | `00-总控/entity-snapshot.yaml` | 关键转折点的角色状态变化链 |
| 19 | `Lv4-核心对立角色参考卡.md` | `状态/角色/{反派}-角色卡.md` | 按 character-schema Lv4 填写最终反派 |
| 20 | `完整文风DNA档案.md` | `写作资产/文风DNA/{书名}.md` | pop-dna v3.0 全量蒸馏 |
| 21 | `Lv3-拆解摘要.md` | — | 更新摘要，标注全书层面的核心发现 |

**Lv3 完成后的用户状态：**
- 完整全链路参考文件集
- 可以逐卷对照参考
- 文风 DNA 可以直接喂给 prose-render

## 五、拆书产出 → 写作管线消费协议

### 5.1 核心协议：bookstrap Phase 0.6 是消费入口

bookstrap Phase 0.6（deconstruct-融合摘要.md）是拆书产出进入写作管线的唯一官方入口。拆书产出不直接散落到写作管线各处，而是通过 Phase 0.6 做"翻译"——把参考书的发现翻译成"对我的项目意味着什么"。

```
拆书产出 (_参考书/{书名}/)
       │
       ▼
bookstrap Phase 0.6
  → 读取拆书产出的 story-engine.yaml
  → 读取拆书产出的角色参考卡
  → 读取拆书产出的卷分析
  → 产出 deconstruct-融合摘要.md
       │
       ├─→ "参考书的核心假说 vs 我的假设 —— 差距在哪？"
       ├─→ "参考书的角色配置 —— 哪些可以借用/哪些需要重新设计？"
       ├─→ "参考书的卷结构 —— 我的卷数/每卷字数应该怎么调整？"
       └─→ "参考书的文风 —— 哪几条规则可以直接用？"
```

### 5.2 消费协议表

| 拆书 Lv | 产出文件 | 消费 Skill | 消费阶段 | 消费方式 |
|:------|:---------|:----------|:---------|:---------|
| Lv1 | story-engine.yaml | bookstrap | Phase 0.6 | 对比我的 story-engine，找出差异 |
| Lv1 | Lv4-主角参考卡 | bookstrap | Phase 3 (角色设计) | 参考结构，对照我的主角是否在同维度上有设计 |
| Lv1 | 卷1起终点快照 | bookstrap | Phase 6/7 (快照设计) | 参考快照的维度和颗粒度 |
| Lv1 | 快速文风指纹 | bookstrap | Phase 0.6 | 提取 2-3 条可复用的文风规则 |
| Lv2 | 卷1-volume-analysis | plot | Step 4.5 (卷设计) | 参考卷1的角色池/地点池/剧情线结构 |
| Lv2 | 卷1-act-mapping | plot | Step 9 (幕设计) | 参考幕的情绪弧线和爽点配置 |
| Lv2 | 卷1-info-release-plan | plot | Step 9 (info_release_plan) | 参考信息释放的节奏和密度 |
| Lv3 | 全书架构 | plot | Step 1.5 (全书架构) | 参考全卷蓝图 |
| Lv3 | 完整文风DNA | prose-render | Step 1 (风格感知) | 直接加载为风格约束 |

### 5.3 不直接消费的边界

以下产出物**不直接写入写作管线**，而是作为 Phase 0.6 的参考输入：

- 拆书产出的角色参考卡 → 不直接变成我的角色卡。Phase 0.6 做翻译
- 拆书产出的数值体系 → 不直接变成我的数值体系。Phase 0.6 做翻译
- 拆书产出的 entity-snapshot-chain → 不直接变成我的 entity-snapshot。太贵，不需要

## 六、Skill 适配策略：模板借力拆书产出

### 6.1 核心命题

> **Skill 自带模板 = 低保真度通用兜底。拆书产出 = 用户认可的高保真度参考。当拆书产出存在且质量被用户确认时，Skill 应该优先参考拆书产出而非自己的模板。**

### 6.2 各 Skill 适配方向（暂不改造，先思考）

#### pop-novel-bookstrap

**当前：** Phase 1-7 依赖 skill 自带的模板和提问流程。

**适配后（有拆书时）：**

| Phase | 当前行为 | 适配行为 |
|:------|:---------|:---------|
| Phase 0 (story-engine) | 从零提问，构建核心假说 | **先从拆书的 story-engine.yaml 中提取参考模板**——"参考书的核心假说是 X，它在你的赛道上意味着什么？你是沿用还是反着来？" |
| Phase 3 (角色设计) | 按 character-schema Lv4 模板逐项填写 | **加载拆书的 Lv4-主角参考卡作为对照**——"参考书的主角在'驱力'维度上是这样配置的（展示），你的主角在这一维度上有什么不同？" |
| Phase 6/7 (快照) | 从零设计起终点状态 | **加载拆书的卷1起终点快照**——"参考书的卷1走了这么远（展示），你的卷1计划走多远？" |

**关键规则：** 拆书产出不是"答案"，而是"脚手架"。用户仍然要自己做设计决策，但有参考书的对照会让决策更快更准。

#### pop-novel-plot

**当前：** Step 4.5/9 依赖 L1 和 project.yaml 做卷设计和幕设计。

**适配后（有拆书时）：**

| Step | 当前行为 | 适配行为 |
|:------|:---------|:---------|
| Step 4.5 (卷设计) | 从零设计角色池/地点池/剧情线 | **加载拆书的 volume-01-analysis 作为结构参考**——"参考书的卷1有 X 个角色、Y 个地点、Z 条剧情线。你的卷1需要在哪些维度上更密集/更稀疏？" |
| Step 9 (幕设计) | 从零设计 Canvas 矩阵 | **加载拆书的 act-mapping 作为节奏参考**——"参考书卷1的幕级情绪弧线是这样的（展示），你的幕级情绪弧线要更陡还是更平？" |
| Step 9 (info_release) | 从零规划信息释放 | **加载拆书的 info-release-plan**——"参考书在第 X 章释放了核心设定，在第 Y 章确认了世界观。你的信息释放要不要比参考书更快/更慢？" |

#### pop-novel-character-schema

**这是适配收益最大的 Skill。**

**当前：** 模板是通用模板——"一句话概括：'一个[特质]的[身份]，他/她想[目标]但[障碍]'"。这个模板跨赛道通用，但**没有示范效果**。

**适配后（有拆书时）：**

```
character-schema 被调用
  │
  ├─ 检查: workspace-index.yaml → 是否有已拆参考书？
  │
  ├─ 有 → 加载拆书的角色参考卡作为"赛道示例"
  │       ┌──────────────────────────────┐
  │       │ 你用 Lv4 模板设计主角       │
  │       │                              │
  │       │ 对照：                        │
  │       │ 《吞噬星空》罗峰的 Lv4 参考卡 │
  │       │ （侧边展示：驱力 / 心理 / 演化 │
  │       │  的填写范例）                 │
  │       └──────────────────────────────┘
  │
  └─ 没有 → 用通用模板（兜底）
```

**关键理解：** 角色模板写完是"填空题"，拆书的角色参考卡是"参考题"。用户在填自己的角色卡时，对照参考卡可以大幅减少"这个词对不对""这个维度到底写什么"的困惑。

#### pop-dna / pop-novel-prose-render

**当前：** pop-dna 产出风格文件，prose-render 消费。

**适配后（有拆书时）：**

- Lv1 拆书的"快速文风指纹 Top 5"可以在 prose-render Step 1 加载为初始风格约束
- Lv3 拆书的"完整 DNA 档案"可以替代从零产出的 style file

#### 调度层（expert-writer）

**适配方向：** 在 workspace-index.yaml 中增加 `reference_deconstructions[]` 字段。当 bookstrap/plot/character-schema 被路由时，expert-writer 在 §3.1.5 信息增强阶段检查是否有可用拆书产出，有则将其路径注入子 skill 的上下文。

### 6.3 适配哲学：兜底 → 升级

```
Skill 自带模板
  = 低保真度
  = 通用，但不针对具体赛道
  = 不会因为赛道不同而变化
  = 永远存在，永远可兜底

拆书产出
  = 高保真度
  = 针对用户选择的赛道和参考书
  = 模板的示范版本——"这个赛道的主角卡实际上应该长这样"
  = 存在时优先使用，不存在时退回模板
```

**关键约束：拆书产出只做"参考"，不做"替代"。** 用户的写作决策永远是自己的。拆书产出提供的是"这本书是这么干的"，不是"你也应该这么干"。

## 七、迁移路径

### 7.1 从当前 deconstructor (v8.0) → 新 deconstructor (v9.0)

```
Phase 1: 补全产出
  Lv1 快速扫描（新增，当前没有）
  Lv2 结构拆解（升级当前的 Phase 0-4）
  Lv3 全书深拆（保留当前的全量分析能力）

Phase 2: 产出格式对齐
  T1~T7 → 从"分析报告"升级为"全链路结构文件"
  story-engine.yaml（新增）
  角色参考卡（新增，按 character-schema Lv1~Lv4）
  起终点快照（新增）
  info-release-plan（新增）
  act-mapping（新增）

Phase 3: Skill 适配
  bookstrap Phase 0.6（升级消费协议）
  plot Step 4.5/9（增加拆书参考模式）
  character-schema（增加对照模式）
  expert-writer（增加 reference_deconstructions 字段）

Phase 4: 渐进交付
  Lv1 → "你不需要等 Lv2/Lv3，现在就可以开书了"
  Lv2 → "你的卷设计可以更精准了"
  Lv3 → "你有了完整参考体系"
```

### 7.2 向后兼容

- Lv2 的 T1~T7 继续产出，供喜欢看分析报告的用户使用
- Lv3 保留 pop-dna 全量蒸馏能力
- fragment-pipeline 不砍（离线大文件处理），但不再作为唯一路径
- 当前 T1~T6 模板保留，作为 Lv2 阶段的兜底

### 7.3 不做的

- **不要求用户必须拆书才能写作** — 拆书是增强，不是前置依赖。用户永远可以直接开书
- **不做全自动 Skill 改造** — Phase 3 的 Skill 适配先出方案，用户确认后再逐 Skill 改
- **不砍 Lv2 的 T1~T7** — 传统分析报告对部分用户仍然有价值，保留
