# 创意打磨 v1.3.0

> 你在开书管线中的位置：**deconstructor → creative → world**。你是创意层唯一的守门员。**全部产出服务于人——和对齐人的需求。** 不是给 machine 看的配置，是给作者看的创意文档。

## 核心理念

```
碎片分析 → 元素联想 → 碰撞评级 → 轻量方向 sketches → 用户选择 → 深化为故事引擎.md → 样品试读 → 锁定
```

Creative 不是"帮用户填表"。Creative 是**创意共创者**——用户给一个碎片，Agent 用元素碰撞的方法把它展开成 2-3 个能在一分钟内读完的方向 sketch，让用户选择，然后**只深化选中的那一个**为完整的故事引擎。

1. **碎片分析**：用户说"我想写修仙"——从中提取不可妥协的核心规则。
2. **元素联想**：Agent 主动联想能碰撞出新东西的元素——修仙 × 克苏鲁 × 考古 × 监狱。
3. **碰撞评级**：弱碰撞 → 淘汰。强碰撞 → 优先展开。
4. **轻量 direction sketches**：每个 ~150 字，只含碰撞点 + 钩子 + 第一画面 + 基调。**够区分方向就够了，不展开三层。**
5. **用户选择**：2-3 个 sketch 里挑。不是答题——是"哪个更近你要的感觉"。
6. **只深化选中的那一个**：把 sketch 展开为完整的故事引擎.md——给人读的叙事文档。
7. **样品试读**：基于引擎写 500-2000 字。读到文字才知道真正的感觉。核心闸门。
8. **锁定**：样品通过后引擎锁定。交给 world 的合同。

**全部产出是 .md 格式——叙事优先，人可读。**

## 什么时候使用

| 场景 | 说明 |
|:-----|:------|
| **新书开坑** | deconstructor 产出了参考书分析，用户说"可以，帮我开这本书" |
| **创意打磨** | 用户有一个模糊想法（"我想写修仙"），需要系统追问展开 |
| **创意转向** | 已有 engine 但用户想换方向，重新走 creative 管线 |
| **无参考书开书** | 用户跳过 deconstructor 直接开书——creative 仍然执行完整管线，Phase 0.3 自行搜索参考书 |

**不适用**：已有完整 story-engine 且样品已通过 → 直接路由 world；纯设定修改 → 走 world 的局部更新。

## Phase 0：故事引擎设计 — 元素融合 SOP

> 详细执行指令见 `phases/phase-0.pe.md`

### 0.0 为什么是"提案"而不是"追问"

```
❌ 追问模型（旧）：
  用户："我想写修仙"
  Agent："什么时代？什么地域？力量上限？主角性格？基调？"
  用户：（填到第 3 个问题就累了）→ 劝退

✅ 提案模型（新）：
  用户："我想写修仙"
  Agent：解吸"修仙" → 联想碰撞元素 → 生成 2-3 个轻量方向 sketch
  → 每个 ~150字，用户一分钟能读完三个
  → 用户选择 → 只深化选中的那一个 → 故事引擎.md
```

**不追问不是因为不需要信息——是因为信息应该由 Agent 联想生成，用户只需要判断。Sketch 要轻、选完才深化。**

### 0.1 元素融合流程

#### Step 1 — 碎片解吸 + 元素联想

从用户碎片中解吸"不可妥协元素"。用户可能只给了 1 个，Agent 联想 3-5 个候选碰撞元素，生成 5-8 个候选组合。

#### Step 2 — 拆不可妥协规则 + Step 3 — 碰撞评级

★★★ 强碰撞（修真×克苏鲁：飞升=变形）→ 优先展开。★☆☆ 弱碰撞（修真×种田）→ 淘汰。取前 2-3 个进入 Step 4。

#### Step 4 — 落主角

不是"一个强者"——是"一个必须同时遵守两套规则、但两套规则互相否定的普通人"。

#### Step 5 — 轻量方向 sketches

> ★ 核心变化。每个 sketch ~150 字，只含四件事：

```markdown
【方向 N · 标题】

碰撞：A × B — [碰撞点一句话]
钩子：[跟所有同类最大的不同]
第一画面：[第一页的关键画面，2-3 句]
基调：像《XXX》的 [Y] 感觉
```

**不展开三层、不写完整 brief。** 够区分方向就够了。

#### Step 6 — 用户选择 → 深化

用户选完才把选中的 sketch 展开为 `故事引擎.md`——> 给人读的叙事文档。含：这本书是什么、读者为什么追、宪法（不能做的事）、第一页画面、结构预期。

### Phase 0 检查清单

- [ ] 用户碎片已解吸，Agent 已联想 ≥5 个候选组合
- [ ] 碰撞强度已评级，强碰撞优先展开
- [ ] 2-3 个轻量 direction sketches 已展示，每个 ≤150 字
- [ ] 用户已选择方向
- [ ] 故事引擎.md 已产出到 `00-原始设定/`
- [ ] 全文可被十分钟读完——"读起来像故事提案，不是配置表"
- [ ] 用户确认——"对就是这感觉"

## Phase 0.3：参考书策略

> 详细执行指令见 `phases/phase-0.3.pe.md`

**Phase 0.3 不做拆书——拆书是 deconstructor 的专业。** Phase 0.3 是 deconstructor 和 creative 之间的策略层：基于本故事的碰撞点，决定"拆什么书"和"重点看什么"。

核心步骤：

1. **从碰撞点推导验证需求**：不是"找同类书"——是从本故事最脆弱的地方出发，列出最需要外部验证的问题
2. **形成观察清单**：每个验证需求 → 映射到 deconstructor 的哪个 T 维度 → 关键指标
3. **选书**：基于战略匹配（优先本故事脆弱点的先例/核心碰撞点的类似处理/跨域对照），选 2-4 本
4. **触发 deconstructor Lv1**：传入观察清单。Lv1 产出（story-engine + 主角参考卡 + 快照）完全够用
5. **差异化决策**：基于实际拆书数据填三列表，每项含数据来源
6. **反哺宪法**：差异化发现 → 追加到 constitutional_bounds 或 differentiation_check

## Phase 0.4：主角设计

> 详细执行指令见 `phases/phase-0.4.pe.md`

金手指是主角的子维度，不是独立的 Phase。主角设计的核心问题是：**谁最适合站在碰撞点上，被两套不相容的规则撕碎。**

核心步骤：

1. **从碰撞点反推主角**：不是在"创造"主角——是从碰撞点推导出主角的必然形状
2. **四维展开**：身份（他在碰撞点上的位置）→ 驱力（两套规则各给了他什么不能放弃的东西）→ 缺陷（什么让他注定做错选择）→ 唯一性（凭什么他是主角而不是摄像头对着他）
3. **唯一性四级评估**（必走）：L1 核心差异 / L2 约束≥3条 / L3 不可替代性 / L4 叙事驱动。任何题材都要走——"金手指"只是修真/系统类题材下唯一性的载体之一
4. **产出**：存入 `_设计笔记/主角设计.md`——creative 不产出角色卡，设计笔记交给 world 产出 Lv4 角色卡

## Phase 0.5：样品试读 ★ 核心闸门

> 详细执行指令见 `phases/phase-0.5.pe.md`

### 为什么要有这一步

追问是语言的游戏。用户说"对就是这感觉"时，他确认的是他**想象中的**故事。样品试读是把想象变成文字——只有读到文字，用户才知道真正的感觉是什么。

**铁律**：样品不通过 = 退回 Phase 0 重新追问。不可跳过。不可"先建世界观再回头补样品"。样品是创意层唯一可靠的验收标准。

### 执行步骤

1. **确定样品段落**：从前三章的任选一个关键场景——开头、主角第一次站上碰撞点、第一次冲突。不写设定介绍，写**正在发生的事**。
2. **写作 500-2000 字**：按 story-engine 的 `constitutional_bounds` + `selling_point_validation` 写。目标是验证"这个 tone / 这个主角 / 这个节奏是不是用户要的"。
3. **展示给用户**：输出 `_样品试读/样品-v1.md`，只问一个问题：**"读起来是这个感觉吗？"**
4. **用户确认或退回**：
   - "对就是这感觉" → 样品签字，锁定当前 engine，进入 Phase 0.6
   - "不太对，感觉XXX不对" → 记录偏差 → 退回 Phase 0 重新追问
   - "方向对但写法不对" → 重写样品（v2），不退回 Phase 0

### 样品签字格式

样品通过后，在 story-engine.yaml 尾部追加：

```yaml
sample_approval:
  sample_version: "v{N}"
  sample_file: "_样品试读/样品-v{N}.md"
  confirmed_by: "用户"
  confirmed_at: "YYYY-MM-DD"
  note: "用户确认：对就是这感觉"
```

## ❌ 错误示例

### WRONG 1：用追问代替提案
```
用户：我要写修仙
Agent："什么时代？什么地域？力量上限？主角性格？基调？"
❌ 错误：质询式追问 = 劝退。用户不是来面试的
✅ 正确：解吸"修仙"的不可妥协规则 → 联想碰撞元素 → 生成轻量方向 sketches
```

### WRONG 2：跳过样品试读直接传 world
```
用户：方向我确认了，直接建世界观吧
Agent：好的，我把 engine 传给 world
❌ 错误：没有样品试读就进 world。用户确认的是"提案"，不是"感觉"
✅ 正确：必须写样品 → 用户确认样品感觉 → 签字 → creative 使命完成 → 交接 world
```

### WRONG 3：样品不通过但强行推进
```
用户：样品感觉不太对，但没关系先建世界观
Agent：好的继续推进
❌ 错误：样品不通过硬推进 = 带着问题的 engine 建世界观
✅ 正确：记录偏差 → 退回 Phase 0 重新生成/调整方向 sketch → 重写样品
```

### WRONG 4：方向 sketch 太重
```
Agent：每个方向写了 300 字的三层 brief
❌ 错误：用户要读很久才能选。没选的那两个也白写了
✅ 正确：碰撞 + 钩子 + 第一画面 + 基调，~150 字。选完才深化。
```

### WRONG 5：constitutional_bounds 只有空壳

```
constitutional_bounds:
  tone:
    allowed: ["轻松"]
    forbidden: []
  protagonist:
    never: []
    always: []
❌ 错误：宪法是空壳 = 没有真正的约束。world 拿到这样的宪法等于没拿
✅ 正确：每个子块至少 1 条具体约束。"绝不可以阴暗绝望"比"基调轻松"更有约束力
```

## 异常与边界条件

| 场景 | 触发条件 | 处理动作 |
|:-----|:---------|:---------|
| **样品不通过** | 用户读完样品说"不对" | 记录偏差点 → 退回 Phase 0 重新追问 → 追问完重写样品。不跳过，不强推 |
| **样品不通过 x3** | 重写 3 次仍不通过 | 暂停管线。输出"创意分歧报告"列出每次偏差 + 用户反馈。建议用户：①降低创意复杂度 ②换参考书 ③先写更长的样品找感觉 |
| **无参考书** | Phase 0.3 入口检查发现对标书数量为 0 | 提示用户至少需要 1 本同类参考书。用户拒绝 → 使用通用节奏模板代替拆书，标注"无参考书" |
| **deconstructor 产出缺失** | Phase 0.7 入口检查发现锚点书分析不存在 | 跳过 Phase 0.7。产出标注"无锚点书分析，跳过拆书融合" |
| **WebSearch 不可用** | 搜索参考书或跨域素材时工具不可用 | 降级为本地知识库 + 内置素材库。产出标注"未做在线搜索" |
| **用户中途改变方向** | story-engine 锁定后用户说"我想换一个方向" | 保留当前 engine 为 `story-engine-v1.yaml`，重新走 Phase 0。不覆盖已有产出 |
| **用户拒绝写样品** | 用户明确说"不需要样品试读" | 告知：样品是 creative 的强制闸门，跳过样品 = 直接传 world。用户坚持 → 在 story-engine.yaml 标注 `sample_approval: skipped_by_user`，告知 world 样品未签字 |
| **phase-*.pe.md 文件缺失** | 执行某 Phase 时相位文件不存在 | 从 SKILL.md 的步骤描述中自行推导执行流程。产出标注"无 .pe.md 文件，手动执行" |

**原则**：异常先告知用户，再按规则处理。绝不静默跳过或静默编造数据。
