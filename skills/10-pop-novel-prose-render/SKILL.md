# 正文渲染 / 上色表达 v3.0

消费 09-pop-novel-chapter-design 的设计包产物，全量加载 `写作资产/文风DNA/` 文风DNA（场景卡 v4 格式），对照设计包的 scene 字段定位场景卡后渲染。

**加载策略**：全量加载单文件（~20-30K），不拆分。过渡段/情绪缓冲/场景切换依赖整体风格感知。

**核心约束：不碰剧情。** 不读上游 Canvas、不验证设定、不判断角色出场是否合理。Design 说了这章发生什么 → 我只管写好。

**v3.0 核心变更**：对齐 03-pop-dna v4 场景卡格式。设计包 scene 字段 → DNA 场景卡 1:1 映射。全文加载为单一文件。

## ❌ 质量红线（开工前→完工后自检）

| 条件 | 不满足 → |
|:-----|:---------|
| 设计包存在 — chXXX-设计包.md 完整可读 | ❌ 终止渲染，告知用户缺少设计包。 |
| 设计包已读取 — 角色状态 + 事件链 + 场景类型已提取 | ❌ 退回 Step 1 重新读入。 |
| 文风DNA 已读取 — `写作资产/文风DNA/` 下有对应档案（Step 1 已读入 context） | ❌ 降级为默认风格渲染 + 标记警告。 |
| 锚定章已感知 — 本章场景类型对应的锚定章特征已建立（如有） | ⚠️ 未感知则继续，标注"锚定章缺失可能影响场景风格一致性"。 |
| 所有事件已渲染 — 每个骨架事件在正文中有对应段落 | ❌ 退回 Step 2，补齐缺失事件后再进入 Step 4 输出。 |
| 风格验证通过 — 对照 `写作资产/文风DNA/` 原文验证，无风格偏差 ≥ 2 处 | ❌ 退回 Step 3，逐偏差修补后重新验证。 |
| Canvas 约束未突破 | ❌ 标记违规项，退回上游修正或本章标注偏离。 |
| 叙事者不解释 — 没有"他意识到/他感到/他仿佛"等 AI 观感词 | ❌ 退回 Step 2，逐处改写后再验证。 |
| **产出只留摘要** — 写入文件后对话中不粘贴完整产出 | ❌ 说"已写入 {路径}。摘要：{核心}。需展开告诉我。" |

## ❌ 错误示例

### WRONG 1：叙事者跳出来解释
→ 「他手里的剑很锋利——这把武器是他三年前从师门继承的，那场变故之后他一直在熟悉它的重量。」  
→ ✅ 正确：「剑刃在月光下泛着冷白。缠绳握柄上被拇指磨出了一道光滑的凹痕。」

### WRONG 2：因为风格规则而跳过骨架事件
→ 骨架有「主角向盟友提问」事件，但 Render 觉得「对话太长不符合简洁风格」就直接删了。  
→ ✅ 正确：骨架说了有对话就必须写对话。语言可以简洁，事件不可跳过。

### WRONG 3：读上游 Canvas
→ Render 读了 act-XX.yaml 或 L1 设定。  
→ ✅ 正确：Render 不知道这些文件的存在。只读骨架 + 人物卡 + 风格材料。

### WRONG 4：重新提取风格契约
→ 先写一份"风格契约"，再用契约渲染。这是多此一举——文风DNA文件已经完整可读。  
→ ✅ 正确：Step 1 读完文风DNA，context 中已有。Step 2 直接渲染。

## 异常与边界条件

| 条件 | 处理方式 |
|:-----|:---------|
| 设计包不存在 | ❌ 终止渲染。告知用户"chXXX-设计包.md 不存在，请先执行 chapter-design 产出设计包。" |
| `写作资产/文风DNA/` 目录为空 | ⚠️ 降级为默认风格渲染。标记"文风DNA缺失，本章使用默认叙事风格。建议先执行 03-pop-dna 建立文风档案。" |
| 续写时上一章正文缺失 | ❌ 终止渲染。告知用户"上一章正文不存在，无法获取末尾800字做衔接。" |
| entity-snapshot 章节数不一致 | ⚠️ 继续渲染，但标记警告："entity-snapshot 声称{N}章，实际正文{M}章。章末状态更新块可能不准确。" |
| 设计包中 scene 字段缺失 | ⚠️ 降级为线性叙事渲染。文风DNA场景卡映射跳过，使用通用风格。 |
| 设计包中某事件描述不完整（缺少 POV/冲突类型/情绪目标） | ⚠️ 对该事件做最小推断补全（POV默认主角、冲突类型默认外部、情绪目标默认平），并标注推断来源。 |
| 本章为高潮章，事件数 > 8 | ⚠️ 检查是否节奏过密。超过8个事件时，Step 2 输出时提示"本章事件密度偏高，建议确认是否需拆分"。 |
| 上一章末尾有未闭合钩子 | ✅ Step 2 开幕段落优先回应该钩子（"衔接渲染"规则）。 |

## 本阶段边界

### 本步做什么
- 读设计包 + 文风DNA → 逐事件渲染 → 风格验证 → 输出正文
- 仅负责"写好"，不负责任何剧情判断

### 本步不做什么
- ❌ 不验证角色出场是否符合设定（那是 QA 的活）
- ❌ 不检查事件是否符合幕纲情绪弧线（那是 plot 的活）
- ❌ 不重新编排事件顺序（骨架说了顺序，Render 不改）
- ❌ 不自行增删事件（骨架有几个事件就写几个）
- ❌ 不输出中间产物（不生成风格契约、不生成渲染草稿文件——全部在 context 中）

**越界检测**：Render 发现某个事件"写不出来"或"不合理" → 说"该事件在渲染时遇到 [具体困难]，建议回到 chapter-design 确认事件定义。" — 不自作主张修改事件逻辑。
