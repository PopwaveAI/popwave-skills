# 网文书籍舆情追踪技能

## 概述

基于 cnovel-research 的 8 平台采集能力，接受一本书的关键词（书名/作者/角色/梗），自动扫描全平台讨论，输出结构化舆情报告。

## 输入格式

用户提供以下任意组合：

```
书名：《诡秘之主》
作者：爱潜水的乌贼
角色：克莱恩 / 奥黛丽 / 阿蒙
关键词：序列 / 源堡 / 愚者 / 诡秘
平台偏好：全平台 / 只看龙空+贴吧 / 排除豆瓣
时间范围：最近一个月 / 最近一年 / 全部
```

**最少输入**：书名即可。作者、角色、关键词为可选增强项。

## 搜索词构造规则（⚠️ 必须覆盖全部四类，不可只取前两类）

### 第一类：书名+作者（基础覆盖）
```
"{书名}"                        # 不加"小说"限定——后过滤而非前过滤
"{书名} {作者}"                  # 书名+作者
```

### 第二类：核心角色名（至少5个）
```
"{书名} {角色1}" / "{书名} {角色2}" / ...
# 用户输入角色名时直接用；未提供则用 WebSearch 快速补全角色列表
```

### 第三类：经典场景/梗/标签
```
# 根据书类型自动生成
# 大IP老书: "{书名} 退婚" / "{书名} 经典" / "{书名} 爽文"
# 争议书: "{书名} 抄袭" / "{书名} AI"
# 高口碑新书: "{书名} 好看" / "{书名} 推荐"
```

### 第四类：交叉书比较（可选但推荐）
```
"{书名} vs" / "{书名} 对比" / "{书名} 和 xx 比较"
# 大IP常被拿来和其他书比较——这类讨论含深度分析
```

### 第五类：传播层（可选）
```
"{书名} 实体书" / "{书名} 影视化" / "{书名} 动漫"
```

### ⚠️ "小说"限定词的陷阱

**不要前过滤**——不要在所有搜索词里强制加"小说"。
**改为后过滤**——先搜《书名》，拿到结果后人工删除 TV/动漫/手游内容。

原因：大量书评讨论不会在微博/知乎/评论区特意写"小说"二字——"斗破苍穹是我最爱的一本书"比"斗破苍穹小说好看"更常见。前过滤会漏掉这些。

## 平台采集策略

| 渠道 | 类别 | 怎么扒 | 用户操作 | 效果 |
|:---|:---:|:---|:---|:---|
| **微博** | 🔥必 | crawl4weibo 自动读Cookie搜，11关键词×2页 | 零 | 全自动 ✅ |
| **知乎** | 🔥必 | WebSearch + WebFetch 搜标题→点开拿全文 | 零 | 全自动 ✅ |
| **B站** | 🔥必 | 搜索API ⚠️必须爬评论区。标题搜→评论API逐条捞，超时重试3次 | 零 | 全自动 ✅ |
| **贴吧** | 🔥必 | aiotieba 扫6-7个相关吧 | 零 | 全自动 ✅，但老书可能被动漫/盗版淹没 |
| **龙空** | 🔥必 | lkong_scraper.py 双层降级：①Playwright读Cookie搜 ②WebSearch降级 | 首次 setup一次 | 有登录态→100%自动 ✅ |
| **豆瓣** | 🟡选 | WebSearch搜列表标题，正文403需手动 | 手动复制粘贴 | 半自动 ⚠️ |
| **晋江** | 🟡选 | HTTP直抓board 4+17，仅女频适用 | 零 | 全自动 ✅ |
| **阅文** | ⚪辅 | WebSearch，书评少 | 零 | 辅助渠道 |
| **公众号** | 🟡选 | 搜狗微信搜索 → 可获**标题+摘要+日期**。正文需过验证码，半自动 | 零 | 半自动 ⚠️ |
| **小红书** | ⚪辅 | ①外部索引 site:xiaohongshu.com ②用户登录态下浏览器采样（pip install playwright） | 登录态下零 | 无登录→仅索引 ⚠️ |
| **抖音** | ⚪辅 | 仅外部索引 site:douyin.com。搜索页依赖浏览器 JS 渲染，批量抓取不稳定 | 全部手动 | 仅索引 ⚠️ |
| **视频号** | ✖弃 | 无公开搜索入口，内容依赖微信客户端 | 全部手动 | 不可用 ❌ |

**类别说明**：🔥必=每次必扫、🟡选=按书类型决定、⚪辅=有就扫没有不强求、✖弃=已放弃不浪费时间

**默认扫描**：每次必扫 5 个 🔥 渠道，输出渠道复盘表标注每个平台跑通/卡点/跳过。

### 辅助渠道的三层采集方案

对于公众号、小红书、抖音这三个不能全自动的渠道，按优先级拆三层：

**第一层：稳定公开源（全自动）**
- 公众号：搜狗微信搜索 `weixin.sogou.com` → 可获取文章标题/摘要/公众号名/日期 → 正文需过验证码
- 小红书：外部搜索引擎 `site:xiaohongshu.com "{书名}"` — 实测零返回，基本不可用
- 抖音：外部搜索引擎 `site:douyin.com "{书名}"` — 搜索页为 JS 空壳，不可用

**第二层：用户授权采样源（半自动，需用户浏览器已登录）**
- 小红书：`pip install playwright` → 读浏览器 Cookie → 搜索+采样
- 抖音：同上。不做批量抓取，只采样 Top N 条结果

**第三层：人工补充源（需用户手动操作）**
- 视频号：用户提供链接或截图，仅做样本分析
- 微信内搜索：用户提供截图或复制文本

**报告标注规范**：
- 仅用第一层 → 标注「公开索引版」
- 用了第一层+第二层 → 标注「登录采样版」
- 用了全部三层 → 标注「全源版」

### 双路交叉验证规则

**每个平台必须同时走两条路**，任一命中即算覆盖：

| 平台 | 路径 A（专用工具/API） | 路径 B（AnySearch） |
|:---|:---|:---|
| 晋江 | HTTP 抓取 board=4 + board=17 | AnySearch site:bbs.jjwxc.net |
| 贴吧 | aiotieba get_threads（至少5个相关的吧） | AnySearch site:tieba.baidu.com |
| 龙空 | Playwright 浏览器 Cookie（搜） | lkong_scraper.py（提取） |

### 零结果标注规则

报告里不能笼统写"未命中"。零结果的平台必须标注原因：

| 场景 | 标注 |
|:---|:---|
| 搜了但确实无讨论 | "⚪ 无人讨论 — 该平台用户群与此书类型不重叠" |
| 搜了但路径未覆盖 | "⚠️ 待补搜 — 仅走了一条路径" |
| 搜不到因为工具限制 | "⚠️ 需人工 — 平台反爬/登录限制" |

### 龙空搜索原理（两层自动降级）

龙空 Fifth Era 把搜索功能关在了登录墙后面。提取层（爬 thread 全文）已攻克，发现层（找哪些 thread 在讨论这本书）靠两层自动降级：

```
第1层（优先）: Playwright 打开你的浏览器 → 读龙空 Cookie → 直接搜龙空
             前提: 你浏览器里登录过 lkong.com + pip install playwright
             效果: 100% 命中，跟手动上龙空搜一样
             
第2层（降级）: WebSearch "site:lkong.com/thread {书名}"
             前提: 无（只要有网）
             效果: 大IP能命中 1-5 条，新书大概率 0 条
```

**用户首次使用前**：
```bash
pip install playwright --break-system-packages
python -m playwright install chromium
python tools/lkong_scraper.py setup   # 检查环境 + 试搜
```

**降级方案的边界**：龙空所有内容在 JS 渲染层，Google/Bing 只能索引被人贴过外链的页面。新书/小众书的龙空讨论几乎不可能被搜索引擎抓到——这不是工具问题，是龙空架构设计。

**用户无需提供账号密码**。跟微博一样：你在浏览器里登录过龙空 → Playwright 打开你的浏览器 → 自动读 Cookie → 搜。零额外操作。如果你没登录过龙空，它会自动退回第 2 层。

## 多版本区分

当书有同名电视剧/电影/动漫时，搜索结果会被非书内容污染。策略：

1. **改为后过滤**：先不加"小说"限定词搜《书名》，拿到结果后人工删除 TV/动漫/手游内容
2. **一票否决**：标题含"电视剧集""电视剧""电影""真人版""分集剧情""在线观看""手游""游戏攻略"→ 标记为"非书讨论，已过滤"
3. **IP讨论保留**：标题含"改编""IP""网文"→ 保留，这是书的衍生讨论

## 开始使用

### 一次性安装

```bash
pip install playwright --break-system-packages
python -m playwright install chromium
pip install crawl4weibo aiotieba --break-system-packages
python tools/lkong_scraper.py setup
```

`setup` 会检查环境是否到位、浏览器是否能读龙空 Cookie。一次跑通，以后不用管。

### 搜一本书

> 「查一下《XXX》」

只需给书名。工具自动：
1. 龙空搜（优先走浏览器 Cookie，降级走搜索引擎）
2. 微博、知乎、B站、贴吧、豆瓣并行扫
3. 所有平台按热点提取亮点和差评
4. 输出一份有引用、有判断的舆情报告

报告末尾会有一张渠道复盘表，标注每个平台跑通了没、原因是什么。

### 报告结构（7段标准模板）

详见 `舆情报告模板.md`：

1. **基本档案** — 书名/作者/平台/状态/数据
2. **口碑速览** — 正面/负面/分析比例 + 全平台判断
3. **成功密码** — 3-4条成功原因解释
4. **翻车风险** — 具体问题与读者反馈
5. **读者怎么说** — 分观点分类（🟢🟡🔴⚪）
6. **适合参考/注意** — 可学之处与踩坑预警
7. **小结** — 拆书/开书建议

## 注意事项

1. **小红书/抖音/豆瓣详情页大概率拿不到正文**，依赖 AnySearch 摘要 + 人工补查
2. **微博搜索结果可能包含无关内容**（同名非书籍），需人工过滤
3. **龙空的帖子编号格式为 thread-{数字}**，可直接构建链接
4. **如果书很新/很小众**，搜索结果少 → 报告照出，标注"讨论量不足，可能为早期阶段"
5. **如果书是大IP/影视化中**，预计会有大量营销号内容混入 → 标注来源，区分自然讨论和推广
6. **⚠️ 龙空搜男频书不要搜女频版** — 根据书的类型选择对应版块。男频→网文江湖/原创评论，女频→女频版
7. **B站 API 超时自动重试** — 首次超时加 delay 3s 再试，最多 3 次。不能超时一次就放弃
8. **报告底部必须输出"渠道执行复盘"** — 逐条列出每个平台用的什么工具、结果、卡点（如有）

### 三条跨项目原则

**原则 1：写完立刻用真数据打脸。** 不要在文档里设计规则。写完 Skill 拿真书跑一遍，跑出来的漏洞当场修。v1.0→v1.4 的升级不是靠"想"，是靠千屿/斗破/宿环三本书把漏洞打出来的。

**原则 2：规则要按场景动态调整，不是一刀切。** "小说"限定词——对千屿适用、对斗破是毒药。龙空版块选择——对女频搜女频、对男频搜主版。这些规则不是"写对就行"，是看情况对不对。

**原则 3：报告的"我"要站出来说话。** 用户不花钱雇你搬运原文。引用是标尺，分析才是交付物。报告必须有独立一节讲清楚"基于以上数据，我的判断是什么、对你的产品意味着什么"。
