# L1 · 数值体系设计模板

> 开书阶段必填。定义全书级段位刻度、消耗量化、能力进化路径。
> 填完后数据进入 v3.db skills 表的 rank / unlock_condition / cost_visual 字段，
> 写作时 ESM before 加载到 context-bundle，Pass 2 据此约束主角的能力上限。

---

## 一、段位刻度

```yaml
数值体系:
  段位刻度:
    - name: "第1阶段名称"              # 如"石印"
      unlock_condition: "解锁条件"      # 如"初始状态" / "纹路≥4"
      abilities:
        - "能力A名称"
        - "能力B名称"
      cost_visual: "使用代价的视觉化标志"  # 如"指尖变白（第1指节）"
      description: "这个段位的战斗风格是什么？"
    
    - name: "第2阶段名称"
      unlock_condition: "解锁条件"
      abilities:
        - "能力A名称（升级版）"
      cost_visual: "代价标志的变化"
      description: "战斗风格的变化"
```

### 设计原则

1. **段位数 ≥ 3**（太少读者没有成长感）
2. **每段至少 1 个新能力**（不能同能力升段但无变化）
3. **cost_visual 逐段升级**（从指尖→手掌→前臂，逐步恶化）
4. **解锁条件明确**（"纹路≥4"可计算，"遇到某个BOSS"可预期）

## 二、消耗量化

```yaml
消耗量化:
  基础技能:
    skill_name: "基础盖印"
      cost_type: "生命力"
      cost_visual: "指尖变白"
      cooldown: "无（但每次累积消耗）"
      max_usage_before_consequence: "连续使用X次后XXX"
  
  大招技能:
    skill_name: "吞噬"
      cost_type: "生命力+印记"
      cost_visual: "手掌变白，消耗一道纹路"
      cooldown: "每章最多1次"
  
  特殊技能:
    skill_name: "复活"
      cost_type: "生命力+印记"
      cost_visual: "前臂变白"
      cooldown: "每幕最多1次"
```

### 设计原则

1. **每种消耗必须有一个视觉化锚点**——读者要能通过身体变化感知到代价
2. **大招和大招之间不能共用冷却**——防止玩家/读者发现"反正每章只能用一个，用哪个都一样"
3. **消耗不能只在逻辑中存在，必须在正文中可观测**——如果读者不知道角色还剩多少"蓝"，那就不是量化

## 三、能力进化树

```yaml
能力进化树:
  路线A:
    name: "盖印路线"
    进化路径:
      - "基础盖印（初始）→ 精准盖印（纹路≥4）→ 远程盖印（纹路≥7）→ 封印（纹路≥10）"
    关系: "其他能力的基础通路"
  
  路线B:
    name: "吞噬路线"
    进化路径:
      - "吞噬（纹路≥4解锁）→ 定向吞噬（只吞指定类型）→ ???"
    解锁依赖: "盖印路线至少到精准盖印"
    关系: "消耗一道纹路，释放一次超常规力量"
  
  路线C:
    name: "复活路线"
    进化路径:
      - "复活纸人（初始可用）→ 复活其他实体（纹路≥7）→ ???"
    关系: "消耗最大，CD最长"
```

### 设计原则

1. **能力之间的关系必须明确**——并列/分支/前置-后置。不明确的关系让读者无法预期成长路径
2. **每条路线有独立进化逻辑**——不是"变强了所以更厉害了"，是"解锁了这个所以能这么做"
3. **至少有一条路线的最终形态是明确的**——让读者对终点有期待

## 四、数据落地方式

设计阶段填好以上模板后，由 ESM 写入工具将数据注入 v3.db：

| 字段 | 目标表 | 说明 |
|:-----|:-------|:-----|
| `name` | skills | 能力名称 |
| `rank` | skills | 当前段位（石印/铜印/...） |
| `unlock_condition` | skills | 解锁条件 |
| `cost_visual` | skills | 代价视觉化锚点 |
| `description` | skills | 能力描述 |

## 五、combat_capability（战斗能力约束）（v2.8 新增）

定义角色在每个段位的战斗能力边界，用于约束战斗描写和剧情合理性。

```yaml
combat_capability:
  attack_range: "只有近战（必须接触到目标）"    # 攻击范围（近战/远程/范围）
  damage_level: "对煞级诡异=即死，对劫级=灼伤"  # 对不同等级敌人的伤害表现
  limit: "不能同时对多个目标用印"               # 战斗限制条件
  weakness: "盖印后印面需要冷却20-30秒"         # 弱点或冷却条件
```

### 设计原则

1. **attack_range 必须在每个段位明确**——近战就是近战，不能模糊处理
2. **damage_level 必须给出跨级对照**——对低一级=即死，对同级=五五开，对高一级=勉强
3. **limit 是战斗描写的约束条件**——限制越具体，战斗场景越有层次感
4. **weakness 必须可被敌人利用**——不能是"怕水但敌人不会用水"这样的伪弱点

## 六、monster_rank_map（敌我段位对照表）（v2.8 新增）

将每个敌人类型映射到玩家的段位体系，用于设计遭遇战难度和主角成长节奏。

```yaml
monster_rank_map:
  "影尸": "入行·辩烟 同级"            # 敌人名称 → 对应玩家段位
  "守门人": "上道·用墨 初期"
  "恶堕僧": "上道·用墨 巅峰"
  "祭坛守卫": "入道·结印 初期"
```

### 设计原则

1. **每个可战斗的敌人类型必须有段位映射**——不能有"实力不明"的敌人（神秘Boss除外）
2. **段位映射决定战斗结果预期**——同级=苦战，低一级=碾压，高一级=逃跑/智取
3. **至少设计 3 档难度差**——让读者感受到"主角确实在变强"

---

## 七、act_rank_schedule（幕级升级时间表）（v2.8 新增）

规划每幕（act）结束时主角应达到的段位，以及对应的关键解锁节点。

```yaml
act_rank_schedule:
  "第一幕":
    end_rank: "入行·辩烟（纹路3/13）"
    key_unlock: "ch003 基础盖印 / ch004 复活 / ch007 吞噬"
  "第二幕":
    end_rank: "入行·断木（纹路7/13）"
    key_unlock: "ch023 精准盖印 / ch031 定向吞噬"
  "第三幕":
    end_rank: "上道·用墨（纹路13/13）"
    key_unlock: "ch044 远程盖印 / ch050 ???"
```

### 设计原则

1. **每幕必须有明确的 end_rank**——不能"大概变强了"，要"到第X章是Y段位"
2. **key_unlock 必须指向具体章节编号**——让后续写作时知道"这一章要写角色解锁什么能力"
3. **段位跨度合理**——不能第一幕就从凡人到半神，也不能三幕都在原地踏步
4. **与情绪曲线对齐**——升级节点应该对应情绪曲线的小高潮
