# 写作质量基线

> 领域：内容创作质量
> 适用 skill：pop-novel-agent-pro / pop-seo-anything / pop-book-promo / pop-novel-game
> 来源：整合自 pop-novel-agent-pro 的 ESM v9.0 问题暴露记录、问题池与根因库、遗漏问题清单、密藏域复盘、5.27 问题整理、跨 skill 通用经验合辑，以及对话中暴露的 HTML 展示站返工案例
> 目的：独立可读，不依赖原始文档。每条基线给出"为什么"+"怎么做"+"验证方式"

---

> **写作质量的基线不是"写得好不好"，而是"该做到的事做到了没有"。**

---

## 一、章末钩子：决定读者翻不翻下一页

### 问题

多章节末尾以"旁白总结"收束（"本章说明了……"、"这就是为什么……"），不是事件断裂。连续 2 章以总结收尾 = 读者流失风险。

### 标准

章末最后 3 行必须在以下四种类型之一上收束，且不允许连续 2 章使用同一种：

| 类型 | 定义 | 示例 |
|:-----|:-----|:-----|
| **事件断裂** | 新事件突然切入，打断当前场景 | "门还没打开，身后传来了脚步声。" |
| **悬念收束** | 揭示一条线索或提出一个问题 | "那封信的落款，是一个已经死了三年的人。" |
| **代价提示** | 展示当前选择的代价 | "他低头看印——裂缝已经到了第四道。" |
| **过渡缓冲** | 为下一章做情绪铺垫 | "后天就到了。" |

### 验证

1. 末 200 字中是否出现"本章说明了/本章揭示了/这就是为什么/就这样"→ 违反
2. 末 3 行是否为事件断裂/悬念/代价/过渡之一 → 人工判断
3. 最近 2 章的钩子类型是否重复 → 标记提醒，不阻塞

## 二、否定式描写：最隐蔽的语法癌症

### 问题

用否定句来描述事物（"不是A而是B"、"说不清的颜色"），而不是直接呈现象。读者读到"不是X"时需要先理解X是什么，再理解"不是X"意味着什么——信息传递效率降低 50%。

### 标准

禁止以下 6 种否定式结构出现在正文中：

| 变体 | 示例（禁止） | 替代（推荐） |
|:-----|:------------|:------------|
| 不是A而是B | "那不是恐惧，是一种说不清的压迫" | "恐惧攥住了他的喉咙" |
| 不是A不是B是一种C | "不是风不是震动是一种频率" | "空气在低频震颤" |
| 说不清的X | "一种说不清的感觉" | "寒意从脚底爬到脊椎" |
| 直觉告诉 | "直觉告诉他不能回头" | "他停住了脚步，没有回头。" |
| 他没有/她没有问 | "他没有问为什么" | "他想问为什么，张了张嘴，没出声。" |
| 不知道为什么 | "不知道为什么，他感到不安" | "不安毫无来由地攥紧了他的胃。" |

### 验证

全文正则扫描上述 6 种模式。命中 > 0 处 → 标记为 WARN。

## 三、爽点密度：每章必须落地至少一个

### 问题

连续 2 章以上零爽点 = 填充章。番茄平台基准是每章至少 1 个微爽点，连续 2 章零爽点 = 读者弃书。

### 标准

每章必须落地至少 1 个微爽点。爽点分为四级：

| 等级 | 定义 | 读者反应 | 允许频率 |
|:-----|:-----|:---------|:--------|
| 微爽点 | 信息推进/小反转/成本展示 | "有点意思" | 每章至少 1 个 |
| 中爽点 | 解谜/协作/初次展示 | "不错" | 每 3-5 章至少 1 个 |
| 大爽点 | BOSS战/形态突破/情感高潮 | "太爽了" | 每 7-10 章至少 1 个 |
| 零爽点 | 纯过渡/信息补全 | "跳过去不亏" | 连续 ≤ 1 章 |

### 验证

1. 章级检查：本章是否有至少 1 个微爽点落地？→ 人工/LLM 判断
2. 连续检查：最近 3 章中是否有 ≥ 2 章零爽点？→ 触发告警

## 四、对话占比：打破叙事密度

### 问题

旁白叙事占正文 85% 以上，读者看一段长描写后容易走神。对话是调节叙事节奏的核心手段。

### 标准

- 对话字符 ≥ 全文字数的 **25%**
- 如果一个场景超过 **500 字** 没有一句直接对话（含角色内心独白），触发提醒

### 验证

1. 单章对话占比 < 25% → WARN
2. 连续 3 章对话占比 < 25% → 告警（需要调整整体节奏设计）

---

## 五、信息密度：一次只让读者吸收 2 个新概念

### 问题

信息密度失衡——前期单章引入 13 个新概念（过密），中期一整章只讲一件事（过稀）。读者在第一章就被信息淹没，在第六章又开始觉得无聊。

### 标准

- **过渡章**：新引入概念/实体 ≤ 2 个
- **核心章**：新引入概念/实体 ≤ 4 个
- 如果当前场景的信息量不够撑满一章 → **压缩到单场景，不要拉成多场景**
- 同一类信息只给一次（如查房场景不需要重复写每家每户的相同结论）

### 验证

提取本章出现的所有 {实体名} 和首次提及的概念名：
- 过渡章 > 2 个 → WARN
- 核心章 > 4 个 → WARN

## 六、场景时间设置：避免"天黑=危险"公式化

### 问题

几乎每章的事件触发节点都是"天黑"。读者 3 章后就建立了条件反射："天一黑就要开打"。

### 标准

连续 ≤ 3 章使用同一时间模式（如全在天黑触发）。如果过半事件在夜间发生，调整部分事件发生时间（清晨/白天/黄昏/黎明）。

### 验证

检查最近 3 章的场景时间设置，如果全部相同 → 触发提醒。

---

## 七、代价/钩子类型差异化

### 问题

同一代价类型或钩子呈现方式被反复使用——如每次战斗收尾都是"低头看印→纹路变多"，同一个物理动作+同一类信息增量使用 4 次。

### 标准

本章的代价类型和钩子呈现方式不能与最近 3 章重复：

| 维度 | 不允许重复 |
|:-----|:----------|
| **代价类型** | 体力消耗/装备损耗/寿命代价/人际关系代价/心理代价…… |
| **钩子呈现** | 低头看→纹路变多 / 远方传来声音 / 收到一条消息 / 门开了…… |

### 验证

手动/LLM 对比本章与最近 3 章的代价和钩子呈现方式，重复则标记提醒。

---

## 八、战斗场景：必须有视觉锚定

### 问题

战斗场景只有动作链（伸手→盖印→缩回），没有画面感。因为 Pass 2 渲染时没有锚定章参考，全靠 LLM 默认语感。

### 标准

本章包含战斗/对抗场景时，必须有：
1. **至少 1 个同类型的锚定片段参考**（从锚定章库或学习资料中选择）
2. **视觉细节要求**（动作/环境互动/感官表现）
3. 如果锚定章库无匹配战斗片段 → 在场景设计中补充视觉细节要求

### 验证

战斗章节完成后，检查正文中是否有：
- 至少 2 个感官细节（视觉/听觉/触觉）
- 战斗结果有明确的体力/状态反馈
- 不是单纯"你一下我一下"的动作链

## 九、角色辨识度：不能靠台词标签区分

### 问题

所有角色在压力下的反应趋同——沉默/观察/等待。读者读 10 章说不出的主角三个性格标签。

### 标准

每个登场角色必须有可区分的：
- **说话方式**（语速/用词/语气词/习惯句式）
- **行为模式**（压力下的第一反应：攻击/防御/观察/逃避）
- **至少一句能让人记住的主动台词或行动**

### 验证

本章涉及的核心角色中，是否有至少 1 人能通过单独的一句台词或一个行为被识别？→ 人工/LLM 判断

## 十、阅读体验铁律

### 10.1 避免抽象描写

禁止：用"一种说不清的感觉/某种频率/不是语言不是画面"代替感官呈现。要求：用动作、对话、环境细节传递氛围。

### 10.2 "不是语言不是画面"类抽象表达

正则命中以下模式之一 → WARN：`不是语言不是画面` / `某种频率` / `说不清道不明`

### 10.3 字数基线（适用于网文）

| 章节类型 | 字数区间 |
|:--------|:--------|
| 过渡/轻松章 | 1800-2200 |
| 核心/推进章 | 2200-2500 |
| 爆发/战斗章 | 1500-1800（动作场景精炼） |

### 10.4 "他感到/她感到/他意识到/仿佛" 冗余前缀

正文中正则命中以下模式 → WARN：
```
他感到|她感到|他意识到|她意识到|仿佛
```

推荐替代：直接用动作/对话/环境反映角色感受，不要用"感到"引出。

## 附：验证优先级

当时间有限时，按以下顺序检查：

| 优先级 | 检查项 | 验证方式 | 拦截级别 |
|:------|:-------|:--------|:--------|
| 🔴 P0 | 章末钩子（禁止旁白总结） | 正则+LLM | 可阻断 |
| 🔴 P0 | 否定式描写 | 正则 | 可阻断 |
| 🔴 P0 | 连续零爽点 ≥ 2 章 | LLM 判断 | 标记 |
| 🔴 P0 | 战斗无锚定（战斗章专用） | LLM 判断 | 标记 |
| 🟡 P1 | 对话占比 < 25% | 正则 | WARN |
| 🟡 P1 | 信息密度超标 | 正则 | WARN |
| 🟡 P1 | 场景时间连续重复 ≥ 3 章 | 人工/LLM | WARN |
| 🟡 P1 | 抽象描写 | 正则 | WARN |
