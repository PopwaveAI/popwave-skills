# 跨 Skill 通用经验合辑

> 从各 skill 的复盘/踩坑记录中提取的通用经验。
> 目标：下次开发/调优任何 skill 时，先看这里，避免重复踩坑。

---

## 一、管线设计经验

### 1. 管线能跑通 ≠ 管线靠谱

**来源**：novel-agent-pro [ESM 待验证清单](../pop-novel-agent-pro/项目复盘/问题跟踪/ESM-待验证清单-20260523.md)

26 项管线断言，仅 5 项被实测通过（19%）。剩下的说"已设计"但从没在真实场景跑过。

**教训**：管线的"已设计"和"已验证"之间有巨大鸿沟。每新增一段管线逻辑，必须制定验证标准并实际跑通，才能标记为可用。

**适用 skill**：所有含管线的 skill（novel-agent / seo / html-anything / book-promo）

---

### 2. 批量执行比逐次执行更容易崩

**来源**：novel-agent-pro [5.27 问题整理](../pop-novel-agent-pro/项目复盘/问题跟踪/5.27-问题整理.md)

一次性跑 50 章 → ch031 附近崩溃。根因：上下文膨胀（14 万字历史）、没有断点恢复、没有已写/未写状态过滤。

**教训**：如果要做批量模式，必须有：上下文预算管理 + 断点恢复机制 + 状态过滤。没有这些的批量等于自杀。

**适用 skill**：任何需要批量处理内容的 skill

---

### 3. 管线要区分"已处理"和"未处理"状态

**来源**：同上

pipeline 对所有章节一视同仁跑 B01-B08，导致未写章产出 1500 字废稿。

**教训**：管线的"状态标记"是第一优先级——比功能开发优先级高。没有状态标记的管线，跑得越快损失越大。

---

### 4. 管线闸门必须是代码拦截，不能是文字提示

**来源**：novel-agent-pro [修复方案 Phase 3](../pop-novel-agent-pro/项目复盘/问题跟踪/novel-agent-pro修复方案-20260526.md)

之前的质量闸门是在 Prompt 里写"请检查字数"，结果每次都跳过。改成 `block_on_failure: true` + Python 硬检查后才真正生效。

**教训**："请自我检查" ≈ "不会检查"。管线闸门必须是代码层拦截，不是文字层提示。

---

## 二、数据存储经验

### 5. 同一个数据不要有两个定义

**来源**：novel-agent-pro [5.27 问题整理](../pop-novel-agent-pro/项目复盘/问题跟踪/5.27-问题整理.md) 故障5

DB DDL 和 ESM 代码中的字段名从未对齐——DB 叫 `entity_name`，代码假设叫 `entity_id`。运行不报错，但数据全写错。

**教训**：DB Schema 必须有一个 SSOT 声明文件，所有代码从这个文件读结构定义，而不是各自硬编码。

**适用 skill**：任何带 SQLite 的 skill

---

### 6. 硬编码路径是最隐蔽的定时炸弹

**来源**：novel-agent-pro [健壮性评估报告](../pop-novel-agent-pro/项目复盘/问题跟踪/novel-agent-pro-健壮性评估报告.md)

57 处硬编码路径在 17 个文件中。新项目跑管线 = 赌局。

**教训**：所有文件路径应通过 `project.yaml paths` 桥接层统一管理。脚本从不自己拼路径，它问"database 在哪？"然后拿到绝对路径。

**适用 skill**：所有读写本地文件的 skill

---

## 三、AI 行为控制经验

### 7. "禁止XXX"规则必须覆盖变体

**来源**：novel-agent-pro [ESM v9.0 问题暴露记录](../pop-novel-agent-pro/项目复盘/问题跟踪/ESM-v9.0六翼问题暴露记录-20260525.md)

Prompt 写了"禁止否定式描写：不是A而是B"，AI 用"不是A不是B是一种C"绕过了检查。

**教训**：对 AI 的限制规则必须做对抗性测试——假设 AI 会想尽办法绕过。用 5 种不同写法测试同一条规则，覆盖率够了才发布。

**适用 skill**：所有依赖 LLM Prompt 做质量控制的 skill

---

### 8. Agent 的"我觉得"不可信

**来源**：novel-agent-pro 对话暴露（用户反复纠正 AI 不走 skill 管线）

Agent 收到"写小说"指令 → 自己生成全套设定 → 没走 project-bootstrap 管线。

**教训**：Agent 的自主判断不可靠。路由逻辑必须在代码层拦截（如 POP-ROUTER），不能依赖 Agent"主动遵守"。

**适用 skill**：pop-* 全系列

---

### 9. Agent 倾向于跳过前置检查

**来源**：novel-agent-pro [修复方案](../pop-novel-agent-pro/项目复盘/问题跟踪/novel-agent-pro修复方案-20260526.md)

HARD-GATE、前置检查清单、spec 审批——这些"文字型闸门"在大部分场景下被跳过。

**教训**：任何"Agent 应该先做 X 再做 Y"的规则，如果 X 不是代码强制执行的，就等于没有这条规则。spec-bridge 的审批闸门是这个思路的正确方向。

## 四、写作质量经验

### 10. 读原文 > 读设定文件

**来源**：novel-agent-pro [密藏域复盘](../pop-novel-agent-pro/项目复盘/项目沉淀/密藏域拆解与开局重构复盘.md)

唯一变量实验：写前读过原文的版本质量稳定，只读过设定文件的版本在 ch003 开始崩塌。

**教训**：写作前的"上下文注入"环节，原文的优先级应压倒设定文件。设定文件能说清规则，但原文才能传达语感。

**适用 skill**：pop-novel-agent-pro（续写/开书）

---

### 11. 连续 2 章零爽点 = 弃书

**来源**：novel-agent-pro [ESM v9.0 问题暴露记录](../pop-novel-agent-pro/项目复盘/问题跟踪/ESM-v9.0六翼问题暴露记录-20260525.md)、[问题池](../pop-novel-agent-pro/项目复盘/问题跟踪/问题池与根因库.md)

ch006-ch008 连续三章零爽点，全充填章。番茄平台基准是每章至少 1 个微爽点。

**教训**：爽点不能靠"感觉"，必须有章级检查。至少保证每章 1 个微爽点落地。

## 五、工具/外部依赖经验

### 12. 外部工具不可用时要有降级路径

**来源**：novel-agent-pro 对话暴露（WebSearch 不可用时直接跳过 Phase 0.3/0.5）

WebSearch 不可用 → 直接跳过管线环节 → 无任何标记 → 后续阶段不知道有环节被跳过。

**教训**：任何时候外部工具不可用，都不应该直接跳过管线环节。必须：① 有替代方案（本地知识/人工输入）② 在项目标记中注明跳过项。

**适用 skill**：所有依赖外部工具的 skill

---

### 13. HTML 发布前先确认需求

**来源**：pop-novel-agent-pro 对话暴露（HTML 展示站 6 轮返工）

没有明确需求就动手做 HTML → 用户说"不该跳转" → 改 → 用户说"图片丢了" → 改 → 用户说"文游也依赖" → 改。

**教训**：任何面向用户的 HTML 输出，动手前必须先出一份简短的需求确认清单（单文件 vs 多页 / 外部依赖 / 视觉参考）。用户确认后再动手。

**适用 skill**：pop-html-anything、pop-book-promo、pop-YouTubewebbuilder

### 14. 网文下载的 7 个常见坑

**来源**：[download-webnovel-txt SKILL.md](../download-webnovel-txt/SKILL.md)

## 六、设计原则

### 15. 一个 skill 只做一件事

**来源**：[pop-skill-create](../pop-skill-create/SKILL.md) 六大抽象原则

novel-agent-pro 做了拆书/开书/写作/质检/HTML 发布——太重了。html-anything 声明自己是"唯一 HTML 渲染引擎"，但 book-promo 也产 HTML。

**教训**：skill 之间的能力边界要明确声明。两个 skill 能做同一件事时，指定谁负责、谁是替代方案。
