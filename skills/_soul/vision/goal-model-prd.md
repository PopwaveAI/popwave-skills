## 1. 问题定义

### 1.1 场景

用户给一个高智能 agent 下达了一个多步骤、长时间的任务：

> "帮我写完 50 章小说。"
> "调研 20 个竞品并输出对比报告。"
> "把这份 300 页的合同逐条审核并标注风险。"
> "持续追踪 10 个关键词的舆情，每周出一份简报。"

用户的核心诉求是一致的：**我不想坐在旁边一直按继续键。**

### 1.2 当前做法的失败模式

传统上这个需求通过两种方式解决，都失败了：

**方式一：人在回路（成功率 100%，但违背目标）**

人每步确认"继续" → 但用户就是想离开座位。

**方式二：Pipeline 自动化（试图模拟按继续键）**

把任务拆成微步骤 → 脚本逐个按步骤执行 → 夹中间闸门检查。

| 失败模式 | 根因 |
|:--------|:-----|
| 质量持续下降 | 上下文累积膨胀，剩余能力不足以支持高质量输出 |
| 对未就绪的任务空跑 | 未区分"待执行"和"已完成" |
| 上下文窗口溢出 | 长任务的历史残留撑爆上下文 |
| 崩溃后无法恢复 | 无断点恢复标记 |
| 辅助数据从未被使用 | 知识库/参考文件写了但流程不读 |
| 质量闸门形同虚设 | 发现问题记录但不阻断、不修复、不学习 |
| 失败经验丢失 | 犯过的错下次还犯 |

### 1.3 核心矛盾

```
旧 Pipeline 的设计假设：
  agent = 流水线工人 → 需要拆成最细的步骤 → 每步一个闸门

实际结果：
  agent 被逼成盲执行的机器 → 丢弃了判断力 → 闸门发现的问题无人修复
  → 最终需要人在回路 → 减负目标失败
```

**换一个假设：**

```
Goal Model 的设计假设：
  agent = 有判断力的协作者 → 给它目标，让它自己决定步骤
  → 但用不可绕过的契约约束它的执行边界
```

## 2. 产品目标

### 2.1 核心价值主张

> **人不在回路的情况下，系统能稳定地、可预期地、可审计地完成多步骤复杂任务。**

### 2.2 关键指标

| 指标 | 目标值 | 说明 |
|:----|:-----:|:-----|
| 连续运行时间 | 不限 | 理论可无限延长，不随时间推移而降质 |
| 单任务质量 | 不劣于人在回路 | 产出质量不低于人工逐步骤监督 |
| 审计完整度 | 100% | 每一步决策都有日志可查 |
| 异常自愈率 | > 80% | 常见异常 agent 自行修复，无需人工介入 |
| 上下文隔离 | 每步独立 | 执行层每次新上下文窗口，不跨步骤积累 |
| 跨任务学习 | 持续 | 前一个任务的教训自动注入下一个任务 |

### 2.3 非目标

- 不是让 agent 完全自由发挥（模板池约束 + 质量基线约束）
- 不是替代 skill/能力层（skill 提供领域能力，Goal Model 提供编排）
- 不是实时协作系统（用户不在回路，事后验收）

│  约束：模板池只读 / 质量基线只读 / 审计日志强制写                  │
│                                                                   │

│    1. 持有目标                                                    │
│    2. 轮询共享状态层 → 判断当前进度                              │

│  职责：                                                            │

│    5. 执行 self_audit → 对照 quality_gates 逐项检查              │

│    ├── 质量基线（只读）                                          │

│    ├── 审计日志                                                  │
│    └── 经验库                                                    │
└─────────────────────────────────────────────────────────────────┘

## 4. Tier 1 · 编排层

### 4.1 形态

**LLM Agent。** 不是脚本。不是规则引擎。

### 4.2 为什么是 Agent 而不是脚本

| 场景 | 脚本的行为 | Agent 的行为 | 谁对 |
|:----|:----------|:------------|:----:|
| 产出距标准差 5%，但整体质量可接受 | ❌ 僵化解回 | ✅ 判断边界情况，弹性通过 | Agent |
| Tier 2 返回资源路径错误 | ❌ 报错退出 | ✅ 检查路径，修正后重试 | Agent |
| 经验库发现上次教训"避免 X" | ❌ 忽略 | ✅ 在本次 Message 中注入提醒 | Agent |
| 连续多次产出偏弱 | ❌ 逐次独立判断 | ✅ 识别趋势问题，暂停并报人 | Agent |

### 4.3 三大约束

Tier 1 有判断力，但必须被约束在轨道内：

#### 约束 1：模板池只读

Tier 1 不能自己发明 Message 结构。只能从预写的模板池中选择。

```
允许：  "当前任务匹配模板 'batch-research-v2' → 填充变量"
不允许："我觉得模板池不够好 → 自己写一个 Message 结构"
```

#### 约束 2：质量基线只读

质量标准定义在共享状态层，Tier 1 可以读，不能改。

```
允许：  "质量基线要求 2200-2800 → 检查 Result 是否达标"
不允许："这次质量放宽吧 → 改质量基线"
```

#### 约束 3：审计日志强制写

Tier 1 每次决策前必须写日志。日志记录是执行任何操作的前置条件。

```
2026-05-27 22:00 | 选了模板 chapter-write-v2
2026-05-27 22:00 | 填入变量：ch026
2026-05-27 22:01 | 调起 Tier 2 | message_hash=abc
2026-05-27 22:06 | Tier 2 返回 | pass
2026-05-27 22:06 | 质量基线通过 | 更新进度
```

### 4.4 工作流

```
Loop:
  1. 读进度 → "当前进度到 ch25，下一步 ch26"
  2. 读任务设计 → act-01.yaml
  3. 读当前状态 → DB/状态文件
  4. 读经验库 → "上次有什么教训"
  5. 从模板池取模板 → "chapter-write-v2"
  6. 填充变量 → 组装 Message
  7. 写审计日志 → "即将调起 Tier 2"
  8. 调起 Tier 2 → 传入 Message
  9. 等待返回
  10. 检查 Result → 质量基线对比
  11. PASS → 更新进度 + 写审计日志 → 下一轮
  12. FAIL → 附加修复指令 → 回到 8（≤ N 次）
  13. 超过 N 次 → 写异常日志 → 暂停 → 等人工
```

### 4.5 核心原则

**Tier 1 绝不越过模板池直接生成 Message。** 它的创造力体现在：
- 选择最合适的模板
- 判断质量边界
- 异常处理
- 经验回传

**不在**发明新流程、修改质量标准、跳过步骤。

## 5. Tier 2 · 执行层

### 5.1 形态

**LLM Agent 实例。** 每次调用创建独立的上下文窗口，执行完毕后销毁。

### 5.2 生命周期

```
Tier 1 调起 → 创建新 session
           → 注入 Message（唯一的输入）
           → Tier 2 执行
           → Tier 2 返回 Result
           → session 销毁 → 上下文清零
```

### 5.3 执行义务

#### 步骤 1：解析 Message

理解契约中的所有条款。格式异常或缺失关键字段 → 返回错误 Result。

#### 步骤 2：执行 mandatory_reads

Message 中列出的所有资源读取，必须在执行主要任务**之前**完成。

#### 步骤 3：执行主要任务

根据 execution_flow 自主执行。可以调整顺序，不能跳过步骤。

#### 步骤 4：执行 mandatory_writes

Message 中列出的所有写操作，必须在返回 Result **之前**完成。

#### 步骤 5：执行 self_audit

对照 quality_gates 逐项自检。

```
self_audit 伪代码：
  for each quality_gate:
    if not pass → fail
  if any fail:
    尝试修复 → 回到步骤 3
    最多 3 次 → 仍旧 fail 则返回附带 fail 原因
```

#### 步骤 6：打包 Result

```json
{
  "task_id": "ch026",
  "status": "pass",
  "self_audit": {
    "pass": true,
    "checks": { "word_count": "pass (2700)" },
    "fixes": []
  },
  "files_created": ["ch026.md"],
  "db_updated": true,
  "summary_updated": true,
  "lesson_learned": "",
  "issues_found": [],
  "execution_log": "读 act → 读 DB → 写正文 → 自检"
}
```

### 5.4 和传统模式中 agent 的区别

| 维度 | 传统 Pipeline 中的 agent | Tier 2 |
|:----|:----------------------|:-------|
| 输入粒度 | 指令级 | 目标级（契约） |
| 上下文 | 依赖对话历史 | 自包含 |
| 自检 | 外部审查 | 自己执行 |
| 修复 | 报告给上游等处理 | 自己修 |
| 学习 | 无 | 写教训到经验库 |
| 销毁时机 | 整个任务结束 | 每次返回后 |

## 6. 共享状态层

### 6.1 组件

```
共享状态层/
├── progress/          # 进度状态
├── baselines/
│   ├── quality_gates     # 质量标准（只读）
│   └── template_pool/    # Message 模板（只读）
├── audit/             # 审计日志
└── knowledge/
    └── experience-log    # 经验教训库
```

### 6.2 访问权限

| 组件 | Tier 1 | Tier 2 | 人 |
|:----|:------|:-------|:---|
| 进度状态 | 读写 | 按需读 | 读写 |
| 质量基线 | 只读 | 通过 Message 传递 | 写 |
| 模板池 | 只读 | 不直接访问 | 写 |
| 审计日志 | 强制写 | 不访问 | 读 |
| 经验库 | 读写 | 读 + 追加 | 读写 |

### 6.3 进度状态设计

```json
{
  "goal": "写完 50 章",
  "steps": {
    "ch001": { "status": "done", "passed": true },
    "ch026": { "status": "pending", "retries": 0 }
  },
  "last_checkpoint": "ch025",
  "running": true
}
```

## 7. Message 契约设计

这是整个系统的核心。Message 是 Tier 1 和 Tier 2 之间的唯一通信媒介，也是质量保障的载体。

### 7.1 通用结构

```yaml
# Message 模板

task_id: "唯一标识"
project_path: "资源根路径"
goal: "一句话任务描述"

# 必读清单 — 执行前必须读取
mandatory_reads:
  - "资源路径 1"
  - "资源路径 2"

# 可选参考
optional_reads:
  - "资源路径 3"

# 执行流程 — 可调顺序，不可跳过
execution_flow:
  - "读 mandatory_reads"
  - "执行主要任务"
  - "自检"

# 必写清单 — 返回前必须完成
mandatory_writes:
  - "产出文件路径 1"
  - "状态更新路径 2"

# 质量标准
quality_gates:
  # 具体指标由模板定义，例如：
  # word_count: { min: 2200, max: 2800 }
  # forbidden_patterns: [...]

# 自检要求
self_audit:
  required: true
  on_fail: "自行修复 → 重跑自检 → 最多 3 次"

# 结果格式要求
result_format:
  status: "pass | fail | partial"
  self_audit:
    pass: bool
    checks: {}
    fixes: []
  files_created: []
  lesson_learned: ""
  issues_found: []
```

### 7.2 模板池示例

| 模板名 | 适用场景 | 关键特征 |
|:-------|:--------|:--------|
| `chapter-write` | 小说写作 | 字数/违禁句式/钩子检查 |
| `batch-research` | 批量调研 | 信源数量/引用格式/覆盖度 |
| `contract-review` | 合同审核 | 条款覆盖率/风险等级标注 |
| `data-report` | 数据报告 | 数据完整性/可视化/摘要 |
| `qa-check` | 批量质检 | 多文件同时检查 |

### 7.3 模板不可绕过

模板存储在共享状态层，Tier 1 只能选和填，**不能修改结构，不能创建新模板，不能绕过。**

修改模板池的唯一方式：**人工提交。**

## 8. 审计机制

### 8.1 设计原则

**审计是所有操作的前置条件。**

### 8.2 审计日志格式

每步决策写一行 JSONL：

```
{"ts":"...","action":"select_template","template":"chapter-write"}
{"ts":"...","action":"launch_tier2","task_id":"ch026","message_hash":"abc123"}
{"ts":"...","action":"tier2_returned","status":"pass"}
{"ts":"...","action":"update_progress","ch026":"done"}
```

### 8.3 可检测的违规

| 行为 | 检测方式 | 后果 |
|:----|:--------|:-----|
| 一次发送了多个任务 | 审计日志 absence 中间轮次 | 打回 |
| 跳过 mandatory_read | Result 缺少预期读操作记录 | 打回 |
| 未更新状态 | Result 中状态更新标记为 false | 打回 |
| 质量不达标 | quality_gates 检查 | 修复或打回 |
| 修改模板结构 | 模板哈希校验 | 系统报警 |
| 修改质量基线 | 基线哈希校验 | 系统报警 |
| 未写审计日志 | 日志轮次断裂 | 系统报警 |

## 9. 质量基线

### 9.1 定义

质量基线由人定义，存储在共享状态层。

```yaml
# 示例：小说写作的质量基线
word_count:
  min: 2200
  max: 2800
forbidden_patterns:
  - "regex_pattern_1"
  - "regex_pattern_2"
hook:
  required: true
```

```yaml
# 示例：调研报告的质量基线
source_count:
  min: 5
citation_style: "APA"
coverage:
  required_categories:
    - "竞争对手"
    - "市场趋势"
```

### 9.2 所有权

- **只能由人修改。** Tier 1 和 Tier 2 均无修改权限。
- 每次修改附带版本号和理由。

## 10. 经验库

### 10.1 目的

解决"下一次犯同样的错误"的问题。

### 10.2 写入时机

- Tier 2 自检发现问题并修复后
- Tier 1 发现模式性问题后
- 人工审查后

### 10.3 格式

```yaml
- id: lesson-001
  source: "Tier2自检"
  issue: "正文出现违禁句式"
  fix: "修改为替代表达"
  prevention: "写完后运行违禁句式检测"
```

### 10.4 注入机制

Tier 1 在每条 Message 中，从经验库提取与当前任务最相关的 2-3 条教训，注入 Message。

Tier 2 收到后必须注意这些教训。

---

## 11. 异常处理

### 11.1 异常分类

| 类型 | 示例 | 处理 |
|:----|:-----|:-----|
| 可修复 | 路径错误 | Tier 1 修正后重试 |
| 可重试 | 超时 | 重试 ≤ 3 次 |
| 需降级 | 资源缺失 | 跳过，记录缺失 |
| 需暂停 | 连续重试失败 | 暂停，等人工 |

### 11.2 重试策略

```
首次失败 → 写日志 → 重试
再次失败 → 同一原因 → 记经验库 → 降级
三次失败 → 暂停 → 等人工
```

## 12. 对比：Goal Model vs 传统 Pipeline

| 维度 | Pipeline | Goal Model |
|:----|:---------|:-----------|
| 编排主体 | 脚本 | LLM Agent |
| 执行粒度 | 指令级 | 目标级 |
| 上下文 | 累积不释放 | Tier 2 每步独立 |
| 质量闸门 | 外部脚本 | 契约内 self_audit |
| 异常处理 | 记录不阻断 | 自愈或暂停 |
| 学习能力 | 无 | 经验库 |
| 审计 | 无 | 每步日志 |

## 13. MVP 验证路径

### 13.1 范围

| 组件 | MVP | 不做 |
|:----|:---|:-----|
| Tier 1 | 手动触发 | 自动轮询 |
| Tier 2 | 执行单一任务类型 | 多任务类型 |
| 模板 | 2-3 个 | 全部 |
| 审计 | 写日志 | 自动化违规检测 |

### 13.2 验证步骤

```
Step 1: 写一个 Message 模板
Step 2: 手动组装一条 Message
Step 3: 调起 Tier 2 → 传入 Message
Step 4: Tier 2 执行 → 返回 Result
Step 5: 人工检查 → 确认质量标准
Step 6: 迭代模板 → 质量稳定
Step 7: 引入 Tier 1（手动触发）
Step 8: 扩展模板池
```

### 13.3 通过标准

| 标准 | 目标 |
|:----|:----|
| 连续任务无人工干预 | 5 次 |
| 自愈率 | > 80% |
| 审计可读 | 完整回溯 |

## 14. 开放问题

### 14.1 Tier 1 如何被约束不偷懒？

三道约束：模板池只读、质量基线只读、审计强制写。

但 Tier 1 可能一次发送多个任务（跳过审计日志中的中间轮次）。

**待决策**：是否需要引入独立于 Tier 1 的监督进程，专门检测审计日志轮次完整性？

### 14.2 模板池演化机制

模板池由人初始编写。随着经验库积累可能需要更新。

选项：
A. 人定期 Review（安全但慢）
B. Tier 1 提议，人审批（灵活但有风险）
C. 只增不修改（安全）

**待决策**：模板池升级策略。

### 14.3 Tier 2 权限边界

Tier 2 有完整文件/API 权限。如何防止污染不属于当前任务的范围？

**待决策**：沙箱路径 / 权限隔离策略。
