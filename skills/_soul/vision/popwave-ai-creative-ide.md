## 1. 背景：代际框架回顾

基于此前与老板讨论形成的 AI 产品代际框架：

| 代际 | 协作模式 | 核心特征 | 代表产品 |
|:----|:---------|:---------|:---------|
| **Gen 1** | 人 → AI | 大模型套壳 + 提示词工程，点到点诉求满足 | SeaArt、AI 工具站 |
| **Gen 2** | 人 → AI → 交付物 | 云端 agent，端到端任务外包，受限于云端 tools | Manus、Lovable |
| **Gen 3** | 人 ⟷ AI | 本地 agent + IDE，双向实时协作 | Cursor、Codex |
| **Gen 4** 🎯 | 人 ⟷ AI ⟷ AI ⟷ 人 | **多节点协作网络**，多个人 + 多个 AI 在共享空间协同 | 尚未出现 |
| **Gen 5** | AI = OS | AI 即操作系统，系统级自主调度 | 远期概念 |

**PopWave 的定位**：直接以 Gen 4 为原生设计起点，不做 Gen 1-3 的追赶者，而是**重新定义创意协作的产品范式**。

## 2. PopWave 当前的代际位置

### 2.1 当前能力（Goal Model 架构）

基于现有 Goal Model PRD，PopWave 当前处于 **Gen 3 到 Gen 3.5**：

| 维度 | 当前状态 | 代际对应 |
|:----|:---------|:---------|
| 编排架构 | 单 Tier 1 + 单 Tier 2 | Gen 3（高效的单线执行） |
| 协作模式 | 人 → 系统 → 产出（人不在回路） | Gen 2 后端 + Gen 3 执行 |
| Agent 数量 | 一个编排 agent + 一个执行 agent | 单线 |
| 用户参与 | 一次性下达目标 → 事后验收 | 离线模式 |
| 学习机制 | 经验库（跨任务） | 超越 Gen 3，接近 Gen 4 |

### 2.2 瓶颈识别

1. **单线执行**：只有一个 Tier 1 编排，只能串行执行，无法并行
2. **单一角色**：所有任务用一个 agent 做，没有角色分工
3. **用户不在回路**：目标是一次性下达，后续人无法参与中间过程
4. **无协作网络**：多人的创作无法在同一空间内协同

## 3. PopWave 的第四代形态：AI Creative IDE

### 3.1 一句话定义

> **一个多人多 AI 协作的创作 IDE——像 Cursor 对编程的改造一样，重新定义"创作"这件事的工作方式。**

### 3.2 核心主张

```
传统的创作工具（Word、飞书文档、Notion）：
  人写 → 人改 → 人审 → 人发
  所有 cognitive load 在人身上

Gen 1 的 AI 写作工具（各类 AI 写作网站）：
  人提需求 → AI 生成 → 人复制粘贴 → 人手动整理
  工具是孤岛，产出是碎片

Gen 3 的 AI 创作工具（当前 PopWave）：
  人定目标 → AI 自主执行 → 人验收
  高效但人不在创作流中

Gen 4 的 PopWave Creative IDE：
  人的位置是"创作流中的决策节点"，不是"流水线起点"
  AI 是岗位化角色（角色分工 × 能力叠加）
  多人可以在同一个创作空间里，各自带着自己的 AI 协作
```

### 3.3 对标逻辑

| 领域 | Gen 3 产品 | Gen 4 的想象 |
|:----|:-----------|:-------------|
| 编程 | Cursor（一人一 AI 结对） | **多人多 AI 协作的代码空间**（已初现：GitHub Copilot Workspace） |
| 设计 | Figma（多人协作，但 AI 只是插件） | **AI 作为设计团队成员的 Figma** |
| **创作** 🎯 | **各类 AI 写作工具** | **PopWave Creative IDE** |

## 4. 设计哲学：涌现式协作

### 4.1 一个底层判断

PopWave 的选择取决于对两个核心变量的判断：

**变量一：创作内容的复杂度天花板**——持续暴涨，没有收敛趋势
- 单作品从几十万字进入百万字级，未来可能是持续演化的"活"作品
- 世界观从单线剧情进入多层嵌套宇宙、跨作品联动
- 协作人数从 1-2 人扩展为 10+ 人 + 数十个 AI 角色

**变量二：Agent 能力的跃迁速度**——只快不慢
- 模型能力半年一迭代，agent 框架从单轮到自主编排
- 当你的 agent 比我的 agent 强一个版本，协作关系就不再对等

**两个变量交汇的方向只有一个解：中心化管控的边际收益趋近于零，管理成本趋近于无穷大。**

一个钉钉式的管理员不可能理解 20 个 AI agent 在 10 个并行剧情线上的所有推理和决策。试图去管——要么管不过来（失控），要么管死了（扼杀创造力）。

### 4.2 三种组织模式的对比

#### 微信模式（去中心化 P2P）

```
特征：
  - 每个 agent 是独立"号"，自主拉群
  - 群是临时的、可变的、可退的
  - 关系是网状 + 自组织的

适用：Agent 角色发现、作品内自由协作
不适：项目治理、质量基线、版本控制
```

#### 钉钉模式（中心化强管控）

```
特征：
  - 组织是元单位，个人隶属于组织
  - 有管理员、审批流、权限层级
  - 关系是树状 + 强管控的

适用：项目管理、权限控制、正式发布
不适：高复杂度创作场景（管不过来也管不活）
```

#### 涌现式模式（PopWave 的选择 🎯）

```
特征：
  - 没有中央调度，靠简单规则 + 局部信号协作
  - 系统整体行为从局部交互中"涌现"
  - 类似蚁群——没有蚁后发号施令，但能造出复杂巢穴

核心差异：
  - 不是"谁来管"，而是"怎么让自组织跑起来"
  - 没有人（也没有 AI）知道全貌，但系统整体做出智能行为
  - 管理成本不随节点数量增长
```

### 4.3 为什么是涌现式？

这个选择背后的第一性原理推导：

```
前提 1：内容复杂度只会越来越复杂
        → 中心化组织的信息汇聚成本指数级上升
前提 2：Agent 能力只会越来越强
        → 强管控会抑制 Agent 自主性的价值
前提 3：创作的底层逻辑本身就是涌现
        → 好世界观从几条核心规则涌现
        → 好剧情从角色和世界的相互作用涌现
        → 好团队从协作中涌现，而非组织结构图定义
─────────────────────────────────────────────
结论：PopWave 的职责不是"管理 AI"
      而是"让创造性能量自然涌现的空间"
```

### 4.4 涌现的四条简单规则

PopWave 不提供钉钉式的管理后台，而是提供让涌现发生的"简单规则"：

```
规则 1：自主权
  - 每个 agent 有权拒绝它认为不合理的任务（附理由）
  - 每个 agent 有权主动提议任务（不需要上级批准）

规则 2：局部共识
  - 涉及世界观设定的修改 → 必须得到所有依赖该设定的 agent 同意
  - 跨 agent 的质量争议 → 由"最近的共同相关方"仲裁，不是由上层管理者

规则 3：完全可观测
  - 所有决策对所有人可见（但不是所有人必须关注所有决策）
  - 任何 agent / 人可以查询任何决策历史

规则 4：自由进出
  - 任何 agent / 人可以随时退出协作（不锁定）
  - 退出时自动移交知识（基于结构化设定集）
```

### 4.5 和前面代际框架的呼应

| 代际 | 组织模式 | 产品体现 |
|:----|:---------|:---------|
| Gen 1 | AI 是工具 | 你用它，用完丢 |
| Gen 2 | AI 是外包工 | 你派活，它干完 |
| Gen 3 | AI 是结对伙伴 | 你们一起干活 |
| **Gen 4** 🎯 | **AI 是自主节点** | **你们在一个生态里自组织协作** |

### 4.6 产品的设计语言

> **"像加好友一样邀请 AI 伙伴，像生态一样自然协作。"**

而不是：
> "请选择你的组织架构，系统将为你分配 AI 和审批流。"

具体体现：

| 层面 | 模式 | 体现 |
|:----|:-----|:-----|
| Agent 市场/发现 | 微信 🎯 | 每个 agent 独立"号"，用户自主邀请 |
| 作品内协作 | 涌现 🎯 | agent 之间直接通信，简单规则自协调 |
| 跨项目 Agent | 微信 🎯 | 一个 agent 可同时在多个作品服务 |
| 项目治理 | 轻钉钉 🎯 | 质量基线、版本控制（仅约束产出，不约束过程） |
| 全局一致性 | 涌现 🎯 | 从局部共识涌现，而非中心指令 |

## 5. Agent 角色系统设计

### 5.1 角色化 Agent 的设计哲学

> **不是"一个 AI 全能"，而是"多个 AI 各司其职"。**

每个 Agent 角色 = 领域专家 × 固定人格 × 工具集 × 权限边界

### 5.2 角色矩阵

| 角色 | 职责 | 输入依赖 | 产出 | 协作对象 |
|:-----|:-----|:---------|:-----|:---------|
| **世界观师** | 构建世界设定、物理规则、文明体系 | 作者的零散 notes | 世界观文档（结构化） | 情节师、人设师 |
| **情节师** | 设计剧情线、冲突、节奏、钩子 | 世界观 + 人设 | 大纲、章节梗概 | 世界观师、节奏师 |
| **人设师** | 人物小传、成长弧、关系网 | 世界观 | 人设卡（结构化） | 情节师、对话师 |
| **文风师** | 维持全文风格一致性 | 样章、风格指南 | 风格约束、词库 | 所有写作类角色 |
| **对话师** | 角色对白设计、语气一致性 | 人设、当前章节 | 对话段落 | 人设师、文风师 |
| **节奏师** | 章节节奏控制、悬念密度、情绪曲线 | 全文进度 | 节奏分析报告 | 情节师 |
| **逻辑师** | 时间线校对、伏笔回收、设定矛盾检测 | 全文 + 设定集 | 逻辑报告 | 所有角色 |
| **校对师** | 错别字、语法、表达流畅性 | 当前章节 | 校对标记 | 文风师 |
| **梗概师** | 生成简介、章节摘要、推荐语 | 全文 | 各类摘要文本 | — |
| **封面师** | 根据作品风格生成封面视觉方案 | 文风 + 世界观 | 封面描述/图 | 文风师 |
| **知识库** | 提供外部知识参考（历史/科学/文化） | 作者的查询 | 知识卡片 | 所有角色 |

### 5.3 角色间协作协议

角色之间通过**结构化消息**进行通信，而非自由对话：

```
【情节师 → 人设师】的协作消息示例：

from: Plot_Agent
to: Persona_Agent
type: character_need
context: "第8章需要主角在压力下做出违背本性的决定"
request: "评估是否符合主角lihua的人物成长弧"
constraint: "不能破坏第3章建立的核心人物冲突"
urgency: high

→ 人设师返回评估结果，附带建议修改
```

**关键原则**：角色间的通信对用户可见、可干预、可驳回。用户是所有 AI 之间协作的**最终裁决者**。

## 6. 协作模式：人 × AI × AI × 人

### 6.1 四种协作拓扑

```
┌──────────────────────────────────────────────────────────────┐
│               PopWave 的四层协作拓扑                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  A. 人 + 专属 AI（一对一）                                    │
│     ┌─ 一个作者 + 一套个人 agent 集群                         │
│     └─ 类似 Cursor，但 agent 是角色化的                       │
│                                                              │
│  B. 人 ↔ AI ↔ 人（跨角色传递）                                │
│     ┌─ 作者写完初稿 → AI 校对 → 发给编辑                     │
│     └─ 编辑的 AI 和作者的 AI 可以直接交互                     │
│                                                              │
│  C. 人 × 多人 AI（创作团队）                                  │
│     ┌─ 多人 + 各自的 AI 在同一作品空间                        │
│     └─ AI 之间自动协作，人在关键节点裁决                      │
│                                                              │
│  D. AI ↔ AI（无人值守协作）                                   │
│     ┌─ 作者设定目标 → 多个 AI 自主协作完成                    │
│     └─ 人只参与验收（类似 Goal Model 的"人不在回路"）         │
│                                                              │
│  用户可在任何时候在这四种模式之间切换。                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 一个典型创作流程

```
① 作者 A 在 Workspace 中创建项目 "仙侠宇宙：归墟纪"

② 作者 A 激活 世界观师 + 情节师 + 人设师
   → 三个 AI 同时从作者的零散 notes 中读数据
   → 世界观师开始构建世界设定
   → 人设师开始生成人物卡片
   → 情节师等待世界观和人设初步成型

③ 世界观师生成初版后，自动通知 情节师："设定就绪"
   → 情节师结合人设卡开始设计核心剧情线
   → 同时自动通知 逻辑师："开始同步跟进"

④ 作者 A 的朋友（编辑 B）被邀请进入 Workspace
   → 编辑 B 有自己的一套 AI（校对师 + 逻辑师）
   → 编辑 B 的校对师和作者 A 的文风师自动建立连接
   → 每次作者 A 更新章节，校对师自动工作，结果发给文风师

⑤ 作者 A 的工作流：
   → 写大纲阶段：和情节师深度协作（人 ⟷ AI）
   → 写作阶段：让 agent 集群自主执行（AI ↔ AI），每天看产出
   → 修改阶段：和编辑 B 的 AI 集群协同（人 × AI × AI × 人）

⑥ 每周自动出一份"作品健康报告"：
   → 逻辑师：伏笔回收率 XX% | 设定矛盾 0 处
   → 节奏师：情绪曲线 | 悬念密度
   → 文风师：风格一致性评分
```

## 8. 关键设计决策

### 决策 1：角色能力 vs 全能 agent

| 选项 | 优劣 |
|:----|:-----|
| **A. 角色化 agent（推荐 ✅）** | ✅ 职责清晰，分工明确，每步上下文可控<br>✅ 可独立优化每个角色的 prompt<br>✅ 质量可归因（哪个角色出问题）<br>⚠️ 协作协议设计成本高 |
| B. 单一全能 agent | ❌ 上下文快速膨胀<br>❌ 创作不同阶段需要不同能力，冲突<br>❌ 质量难以归因 |

**结论**：采用角色化 agent，从核心 4-5 个角色开始（世界观师、情节师、人设师、文风师、逻辑师）。

### 决策 2：协作模式优先级

| 模式 | 优先级 | 理由 |
|:----|:------|:-----|
| **A. 人 + 专属 AI（一对一）** | **P0** | 单人创作是最大基数场景 |
| **C. AI ↔ AI（无人值守）** | **P0** | 复用现有 Goal Model 架构 |
| **B. 人 ↔ AI ↔ 人** | **P1** | 多人协作的门槛较低的场景 |
| **D. 人 × 多人 AI** | **P2** | 最复杂的场景，需要生态成熟 |

### 决策 3：设定集的结构化程度

| 选项 | 优劣 |
|:----|:-----|
| **A. 强结构化（推荐 ✅）** | ✅ AI 可确定性读取和写入<br>✅ 版本控制、diff、回滚<br>✅ 可作为 agent 协作的共享状态<br>⚠️ 初始编写成本高 |
| B. 自由文本 | ❌ AI 解析不稳定<br>❌ 难以做一致性检测 |

**结论**：设定集采用 YAML/Markdown 结构化格式，像代码一样管理。

### 决策 4：AI 间的通信协议

| 选项 | 优劣 |
|:----|:-----|
| **A. 结构化消息（推荐 ✅）** | ✅ 可追踪、可审计、可回放<br>✅ 用户可以查看和干预<br>✅ 便于调试 |
| B. 自由对话 | ❌ 不可控、易偏题<br>❌ 无法做质量闸门 |

**结论**：AI 间通信走结构化消息格式，参考 Goal Model 的 Message 契约设计。

## 10. 演进路线图

### Phase 1：Gen 3.5 基础（当前 → 3 个月）

**目标**：在现有 Goal Model 基础上，引入**角色化 agent** 概念

```
成果：
  - 将当前单一 Tier 2 拆分为 3 个角色 agent：情节师、文风师、逻辑师
  - Tier 1 可根据任务特征路由到不同角色
  - 设定集结构化（世界观 + 人设 + 大纲）
  - 用户可见的角色切换和角色配置界面
```

**验证指标**：
- 角色化后质量是否有提升（vs 单一 agent）
- 用户是否能感知到不同角色的存在价值

### Phase 2：AI ↔ AI 协作（3 → 6 个月）

**目标**：实现角色间自动协作，复用 Goal Model 的无人值守能力

```
成果：
  - 角色间结构化通信协议
  - 自动协作流（如：情节师完成 → 通知逻辑师做 consistency check）
  - 协作流对用户可见、可干预
  - 作品健康报告（角色间自动生成）
```

**验证指标**：
- AI 间协作的准确率
- 用户干预率（期望：逐渐降低）
- 单作品无人工干预的连续运行时长

### Phase 3：多人 × 多 AI（6 → 12 个月）

**目标**：引入多人协作，实现完整的 Gen 4 协作网络

```
成果：
  - 多人 Workspace（邀请、权限、操作历史）
  - 每个人可配置自己的专属 AI 角色
  - AI 之间的协作跨用户边界
  - 编辑 / 审校角色的入驻
```

**验证指标**：
- 多人协作场景的使用率
- 团队创作效率 vs 单人创作
- AI 跨用户协作的准确率

### Phase 4：生态化（12 个月+）

**目标**：从产品走向平台

```
成果：
  - Agent 角色市场（第三方开发角色）
  - 质量基线市场（风格指南、类型模板）
  - API 开放（外部工具集成）
```

## 11. 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|:----|:------|:-----|:-----|
| **多 agent 协作通信成本过高** | 中 | 高 | Phase 1 先验证 3 个 agent 的效果；通信协议从简到繁 |
| **角色化增加用户理解成本** | 高 | 中 | 默认给用户一套"推荐配置"；用户无需理解角色细节也能用 |
| **AI 间协作产生不可预期的结果** | 中 | 高 | 结构化消息 + 用户可见 + 可回滚 |
| **多人协作场景市场时机未到** | 中 | 高 | Phase 1-2 围绕单人场景打磨价值；多人作为自然延伸 |
| **LLM 成本随 agent 数量线性增长** | 高 | 中 | 角色级 prompt 压缩；高频角色可用小模型 |
| **用户习惯"一个 AI 干所有"** | 高 | 中 | 产品设计上让角色化不增加用户负担；角色的协作在后台进行 |

## 附录：场景用例

### 场景 A：单人作者，AI 协作写作

```
用户：网文作者，日更 6000 字

工作流：
  ① 早上打开 PopWave → 情节师已根据大纲自动生成了今日章节的节奏建议
  ② 作者接受建议 → 开始写作 → 文风师实时建议措辞
  ③ 写完一节 → 逻辑师自动检查：时间线/地名/人物关系是否一致
  ④ 修改 → 校对师过一遍 → 发布
  ⑤ 晚上睡前看一眼节奏师生成的"明日预告" → 确认明天方向

AI 在后台一直运转，但作者只在自己想参与的节点出现。
```

### 场景 B：双人合著，AI 做桥梁

```
用户：两个朋友合写一本无限流小说

工作流：
  ① A 负责世界观和主线 → B 负责具体副本设计
  ② A 的世界观师和 B 的副本师自动对齐设定
  ③ 如果 B 设计了一个违背世界观的副本 → 逻辑师报错 → 自动发消息给两人
  ④ 两人在共享 Workspace 中讨论 → 决策 → AI 同步更新设定集
  ⑤ 成稿后 → 校对师和文风师统一全文风格

AI 解决了合著中最头痛的问题：设定不一致 + 风格不统一。
```

### 场景 C：作者 + 编辑 + AI 质检

```
用户：作者写完初稿，提交给编辑审核

工作流：
  ① 作者在 Workspace 中标记"初稿完成"
  ② 编辑收到通知 → 带着自己的 AI 加入
  ③ 编辑的校对师过一遍 → 标记问题
  ④ 编辑的逻辑师和作者的逻辑师交叉检查 → 发现 3 处伏笔未回收
  ⑤ 编辑在问题旁直接批注 → 作者的文风师自动建议修改方案
  ⑥ 作者确认修改 → 发布

编辑和作者的 AI 在后台完成了 80% 的质检工作。
```
