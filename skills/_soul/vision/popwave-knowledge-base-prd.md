## 1. 问题与机会

### 1.1 当前痛点

| 痛点 | 描述 |
|------|------|
| **知识散落** | 网文作者的创作方法论、工具评测、技巧笔记散落在公众号文章、微信收藏、本地笔记、飞书文档中，没有统一管理 |
| **Agent 用不了** | 知识存在 Notion/飞书里，人能看到但 Agent 看不到——每个新任务都要手动喂 context |
| **Skill 孤岛** | 当前 SKILL.md 只有本环境能用，无法分发到其他平台，也无法和其他作者共享 |
| **重复劳动** | 同一个模型评测、同一个技巧、同一个工作流模板，每换一个工具就要重新写一套 |

### 1.2 市场空白

从市场调研来看，现有产品都在各自维度上差一块：

| 品类 | 代表 | 面向人 | 面向Agent | 个人/团队 | 分发能力 | 创作领域特化 |
|------|------|--------|----------|----------|---------|------------|
| 文档知识库 | AFFiNE/Outline | ✅ | ❌ | ✅ | ❌ | ❌ |
| AI 记忆层 | Mem0 | ❌ | ✅ | ❌ | ❌ | ❌ |
| RAG 问答 | ChatWiki/Dify | ✅ | ⚠️ | ✅ | ❌ | ❌ |
| Skill 管理 | PromptHub | ✅ | ✅ | ❌ | ✅ | ❌ |
| **PopWave 知识库** 🎯 | — | ✅ | ✅ | ✅ | ✅ | ✅ |

**结论：把"知识管理"从"给人看的知识库"延伸到"Agent 也能消费的云端结构化知识插件"，加上多租户 + 权限控制 + 创作领域特化，这个组合在市场上没有直接竞品。**

## 2. 一句话定义

> **PopWave 知识库系统：一个面向创作领域的知识资产管理平台。**

底层统一存结构化条目 + SQLite，上层通过 Schema 模板区分品类（Skill/知识/方法论/评测），两端消费（人看 + Agent 查），权限可控（私有/团队/公开），兼有分发能力。

## 3. 核心价值主张

### 3.1 三个"既是…又是…"

```
既是「个人知识管理工具」   又是「团队协作知识库」
既是「人类可读的文档库」   又是「Agent 可查询的结构化数据库」
既是「从外部采集的收纳器」 又是「向外部输出的分发站」
```

### 3.2 一句话对三类用户的承诺

| 用户 | 承诺 |
|------|------|
| **个人作者** | 你的创作方法论、工具评测、技巧笔记，一个地方管好，人和 Agent 都能用 |
| **创作团队** | 共享知识资产，权限可控，协作不打架 |
| **Agent** | 需要的时候直接 SELECT，不用等人喂上下文 |

### 3.3 和 PopWave Gen 4 的呼应

> Gen 4 的核心是**多节点协作网络**——多人 + 多 AI 共享空间协同。
>
> 知识库就是这个共享空间的**知识底座**。没有这个底座，每个 Agent 和每个人都是信息孤岛。

## 5. 核心能力设计

### 5.1 知识条目引擎

知识库的最小单位不是"文档"，而是**条目（Entry）**。

一个条目 = 结构化元数据 + Markdown 正文 + 关联关系

```
Entry 结构示意：
─────────────────────────────────────
id: kb_20260529_001
title: "Gemini 3 Pro 网文评测"
schema: "tool_review"            ← 对应 Schema 模板
created_by: user_xxx
visibility: team                 ← 权限：私有/团队/公开
tags: [模型评测, Gemini, 2025]
relations:
  - type: "references" → kb_20260528_002  (关联文章原文)
  - type: "contradicts" → kb_20250801_001 (DeepSeek V3.1 评测)
content_md: |
  ## 一句话结论
  Gemini 3 Pro 是年度最大惊喜...

  ## 评测结果
  | 维度 | 评分 | 说明 |
  | ... | ... | ... |
─────────────────────────────────────
```

**关键设计原则**：同一个底层 `entries` 表，通过 `schema` 字段区分品类。不需要为 Skill、知识条目、方法论分别建表。

### 5.2 Schema 模板系统

Schema 模板 = 这个品类的条目应该有哪些字段 + 每个字段的消费方式

```
table: templates
├── id: "skill"
│   ├── fields: title, content_md, platform_compatibility, model_test_results
│   └── consumer: "分发"（终点是导出到 Cursor/Claude Code）
│
├── id: "knowledge"
│   ├── fields: title, content_md, source_url, insight_level, related_entries
│   └── consumer: "查询"（Agent 在写作过程中 SELECT）
│
├── id: "methodology"
│   ├── fields: title, steps[], principles[], examples[]
│   └── consumer: "注入"（Agent 作为 System Prompt Context 注入）
│
├── id: "tool_review"
│   ├── fields: tool_name, score, strengths[], weaknesses[], comparison[]
│   └── consumer: "决策"（Agent 选择工具时参考）
│
└── id: "custom" ← 用户自定义
    ├── fields: 用户定义
    └── consumer: 用户选择
```

**内置品类 vs 自定义**：

| 品类 | 内置/自定义 | 说明 |
|------|-----------|------|
| Skill | 内置 ✅ | SKILL.md 标准格式，对接分发通道 |
| 知识条目 | 内置 ✅ | 通用知识，Agent 可查询，人可阅读 |
| 方法论 | 内置 ✅ | 步骤/原则/示例结构化，Agent 可注入 |
| 工具评测 | 内置 ✅ | 评分/对比表格，辅助决策 |
| 自定义 | 自由创建 | 用户自己定义字段和消费方式 |

### 5.3 权限与协作模型

#### 三级可见性

```
私有 ───→ 仅自己可见
  │
团队 ───→ 所属团队的所有成员可见
  │
公开 ───→ 所有人可见（包括未登录用户，只读）
```

#### 四级操作权限

| 角色 | 查看 | 创建/编辑 | 删除 | 管理权限 |
|------|------|----------|------|---------|
| **所有者**（创建者） | ✅ | ✅ | ✅ | ✅ |
| **管理员**（团队指定） | ✅ | ✅ | ✅ | ✅ |
| **编辑者**（团队成员） | ✅ | ✅ | ❌ | ❌ |
| **查看者**（公开/分享） | ✅ | ❌ | ❌ | ❌ |

#### 协作能力

| 功能 | 说明 |
|------|------|
| **批注/评论** | 在条目上逐段批注，支持 @ 成员 |
| **修改建议** | 类似 Google Docs 的建议模式，所有者确认后生效 |
| **变更历史** | 每次修改自动记录，可 Diff 对比，可回滚 |
| **收藏/订阅** | 收藏到个人列表，订阅某品类/标签的更新通知 |
| **协作编辑** | 多人同时编辑一个条目（CRDT 实时同步 / 锁定编辑） |

### 5.4 Agent 消费接口

这是知识库区别于传统文档库的核心能力。

#### 接口一：同步查询（SELECT）

```
Agent 写文过程中直接查询：

GET /api/v1/kb/query
  ?q="Gemini 3 Pro 写网文效果"
  &tags=模型评测
  &schema=tool_review
  &mode=semantic          ← 语义搜索 / 关键词 / 混合
  &limit=3

→ 返回最相关的 3 个条目（结构化字段 + content_md）
```

#### 接口二：上下文注入（Context Injection）

```
Agent 启动时自动注入相关知识：

# Agent 的 System Prompt 中自动注入：
## 相关知识上下文

根据当前任务 [写一部仙侠文的大纲]，已从知识库加载以下条目：
1. 仙侠世界观构建方法论（methodology）
2. 等级体系设计模板（knowledge）
3. 作者过往作品的文风样本（custom）
```

#### 接口三：关联推荐（Relation Suggest）

```
Agent 检测到当前创作涉及某个知识点时，自动推荐相关条目：

[知识库] 检测到你在写「追妻火葬场」情节
→ 推荐阅读：「追妻火葬场 节奏控制方法论」(methodology)
→ 推荐阅读：「知乎风女频短篇 情绪曲线模板」(knowledge)
→ 是否加载为参考上下文？ [是] [否]
```

#### 接口四：订阅推送（Push）

```
Agent 或人可以订阅某品类/标签：
- "当知识库新增'模型评测'类条目时通知我"
- "当 DeepSeek 标签下的条目有更新时通知我"
```

### 5.5 人类消费界面

知识库同时为人类提供阅读体验：

| 视图 | 说明 |
|------|------|
| **浏览视图** | 按品类/标签/时间浏览条目 |
| **阅读视图** | Markdown 渲染，类似 Notion 阅读体验 |
| **搜索视图** | 关键词 + 语义搜索，支持过滤 |
| **关联图谱** | 条目间的引用/矛盾/补充关系可视化 |
| **收藏夹** | 个人收藏的条目 |
| **团队动态** | 团队成员最近创建/更新的条目流 |

### 5.6 分发与发布

知识库的条目可以"发布"到外部：

| 分发目标 | 支持格式 | 说明 |
|---------|---------|------|
| Cursor | `.cursorrules` | Skill 模板→Cursor 规则文件 |
| Claude Code | `CLAUDE.md` + `SKILL.md` | Skill 模板→Claude Code 技能 |
| Windsurf | `.windsurfrules` | Skill 模板→Windsurf 规则 |
| PromptHub | 兼容格式 | 导出为 PromptHub 可导入格式 |
| Web 公开页 | HTML/Markdown | 单条目生成独立公开页面（可分享链接） |
| 开放 API | JSON | 第三方通过 API 查询 |

## 6. 在 PopWave 代际框架中的位置

```
Gen 1（人→AI）──  不需要知识库，AI 无记忆
Gen 2（人→AI→交付）── 知识在人的脑子里，AI 用完即忘
Gen 3（人⟷AI）── 知识在 prompt 里，每次都要喂
Gen 4 🎯（人⟷AI⟷AI⟷人）
    │
    └── 知识不在 prompt 里，在知识库里
        知识是人积累的、Agent 自动查询的、团队共享的
        知识是活的——不是静态文件，是结构化资产
```

**知识库系统 = Gen 4 协作网络的"知识底座"。**

在 PopWave Creative IDE 架构图中，知识库既是：
1. **Agent 角色系统中的一个角色**（"知识库 Agent"——提供外部知识参考）
2. **执行引擎层文件系统的一部分**（设定集/知识库/经验库）
3. **跨作品、跨用户的共享层**（个人积累的知识，可以带到下一个项目）

## 7. 与 Agent 角色系统的融合

### 7.1 知识库 Agent

在 Agent 角色矩阵中新增「知识库 Agent」角色：

| 角色 | 职责 | 输入依赖 | 产出 | 协作对象 |
|:-----|:-----|:---------|:-----|:---------|
| **知识库** | 管理知识条目、提供查询服务、自动关联推荐 | 用户的查询/Agent 的上下文信号 | 知识卡片、关联推荐、一致性检测报告 | 所有角色 |

**知识库 Agent 不是普通角色——它是元角色**。其他所有 Agent 在写作过程中自动调用它：

```
情节师在写大纲时：
  → 自动询问知识库 Agent："当前世界观设定中，等级体系的常见写法有哪些？"
  → 知识库 Agent 从知识库中检索相关方法论条目返回

逻辑师在做一致性检测时：
  → 自动询问知识库 Agent："用户的知识库中有没有这个作品的设定约束？"
  → 知识库 Agent 返回该作品相关的设定条目
```

### 7.2 与经验库的关系

知识库 = 主动管理的、结构化的知识资产
经验库 = 自动积累的、跨项目的 Lessons Learned

两者可互相转化：
```
Agent 自动积累的经验 → 用户审核确认 → 沉淀为知识库条目
知识库条目 → Agent 消费 → 验证 → 更新条目（形成闭环）
```

## 8. 和现有产品的对比

### 8.1 与 PromptHub 对比

PromptHub 做的是**Skill 层面的管理+分发**，我们多了三个维度：

| 维度 | PromptHub | PopWave 知识库 |
|------|----------|---------------|
| 品类 | 只有 Skill/Prompt | 多品类：Skill/知识/方法论/评测/自定义 |
| 消费端 | 分发到 AI 平台 | 人看 + Agent 查 + 分发 |
| 权限 | 无（本地优先） | 私有/团队/公开三级 |
| 协作 | 单人 | 多人：批注/建议/历史 |
| 生态 | 独立工具 | PopWave Creative IDE 原生插件 |

### 8.2 与 AFFiNE/Outline 对比

| 维度 | AFFiNE/Outline | PopWave 知识库 |
|------|---------------|---------------|
| 受众 | 人类知识工作者 | 创作者 + Agent |
| 数据结构 | 文档/页面 | 结构化条目 + Schema 模板 |
| Agent 接口 | 无 | 原生 API（查询/注入/推荐/推送） |
| 领域特化 | 通用 | 创作领域：Skill/设定集/工具评测 |
| 分发能力 | 导出 Markdown | 分发到 15+ AI 平台 |

### 8.3 与 Mem0 对比

| 维度 | Mem0 | PopWave 知识库 |
|------|------|---------------|
| 定位 | Agent 的长期记忆层 | 用户主动管理的知识资产 |
| 数据来源 | 对话自动积累 | 用户主动创建/采集 |
| 结构化 | 记忆片段 | 结构化条目 + 自定义 Schema |
| 人工参与 | 不直接面向用户 | 双端：人管理 + Agent 消费 |

---

## 9. 关键设计决策

### 决策 1：存储选型

| 选项 | 优劣 |
|:----|:-----|
| **A. SQLite + Markdown（推荐 ✅）** | ✅ 本地优先，离线可用<br>✅ md 文件人类可直接阅读和版本管理<br>✅ 与现有 novel.db 技术栈一致<br>⚠️ 云端需同步层 |
| B. PostgreSQL（纯云端） | ❌ 不能离线<br>❌ 和本地优先理念冲突 |
| C. 纯向量数据库 | ❌ 结构化查询能力弱<br>❌ 不适合权限这种精确过滤 |

### 决策 2：Schema 模板是内置还是可扩展

| 选项 | 优劣 |
|:----|:-----|
| **A. 内置品类 + 可自定义（推荐 ✅）** | ✅ 开箱即用<br>✅ 用户可定义自己的品类<br>⚠️ 自定义模板的设计成本 |
| B. 纯内置品类 | ❌ 限制用户使用场景 |
| C. 纯自定义 | ❌ 上手门槛太高 |

### 决策 3：Agent 消费方式

| 选项 | 优劣 |
|:----|:-----|
| **A. REST API + WebSocket（推荐 ✅）** | ✅ 标准化，任何 Agent 可调用<br>✅ 异步支持（订阅推送）<br>⚠️ 需设计合理的 rate limit |
| B. 仅本地 SQLite 直连 | ❌ 仅 PopWave 本地 Agent 可用 |
| C. 仅 RAG 接口 | ❌ 丢失结构化查询能力 |

### 决策 4：权限模型复杂度

| 选项 | 优劣 |
|:----|:-----|
| **A. 三级可见 + 四级角色（推荐 ✅）** | ✅ 简单清晰，用户理解成本低<br>✅ 覆盖 90% 场景<br>⚠️ 不适合企业级复杂组织架构 |
| B. 细粒度 ACL（每人每条目可配置） | ❌ 太复杂，用户管理成本高 |
| C. 仅公开/私有 | ❌ 不能满足团队场景 |

## 10. 技术架构

| 层 | 选型方向 | 说明 |
|:---|:--------|:-----|
| **客户端** | React + Tailwind | Web 端阅读/管理界面 |
| **本地引擎** | SQLite + Python | 结构化条目管理、全文搜索、权限计算 |
| **向量搜索** | SQLite VSS / LiteLLM | 语义搜索能力 |
| **云端同步** | 类 Git 同步协议 | 本地优先→云端备份+团队共享 |
| **Agent API** | FastAPI + WebSocket | REST API + 订阅推送 |
| **发布通道** | Markdown 生成器 | 按目标平台格式渲染 |
| **权限引擎** | RBAC + Row-Level Filter | 查询时自动拼入 SQL WHERE |
| **Schema 引擎** | JSON Schema | 模板定义+校验 |

### 数据模型核心表

```sql
-- 条目表（核心）
CREATE TABLE entries (
    id TEXT PRIMARY KEY,
    schema_type TEXT NOT NULL,       -- 对应 templates.id
    title TEXT NOT NULL,
    content_md TEXT NOT NULL,
    owner_id TEXT NOT NULL,
    team_id TEXT,
    visibility TEXT CHECK(visibility IN ('private','team','public')),
    tags TEXT,                       -- JSON array
    created_at TEXT,
    updated_at TEXT,
    version INTEGER DEFAULT 1
);

-- 模板表
CREATE TABLE templates (
    id TEXT PRIMARY KEY,             -- 'skill','knowledge','methodology','tool_review'
    name TEXT NOT NULL,
    schema_definition TEXT NOT NULL, -- JSON Schema
    consumer_type TEXT,              -- 'dispatch','query','inject','decision'
    is_builtin INTEGER DEFAULT 0
);

-- 权限表
CREATE TABLE permissions (
    entry_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    role TEXT CHECK(role IN ('owner','admin','editor','viewer')),
    PRIMARY KEY (entry_id, user_id)
);

-- 关联表
CREATE TABLE relationships (
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    rel_type TEXT NOT NULL,          -- 'references','contradicts','extends','inspired_by'
    created_by TEXT,
    PRIMARY KEY (source_id, target_id, rel_type)
);
```

## 11. 演进路线图

### Phase 1：个人知识库（MVP）

**目标**：一个人能用，存/查/管自己的知识条目

```
成果：
  - 知识条目引擎（CRUD + Markdown 渲染）
  - 内置 3 个 Schema 模板：Skill、知识条目、方法论
  - 全文搜索 + 标签过滤
  - 私有/公开二级权限
  - Agent API 基础查询接口
  - 人类 Web 浏览界面
```

**验证指标**：
- 用户能否在 5 分钟内创建第一条知识条目
- 用户是否愿意把现有笔记迁移进来
- Agent 查询准确率 > 80%

### Phase 2：团队协作

**目标**：团队可用，权限可控，协作出得来

```
成果：
  - 团队空间（创建/邀请/成员管理）
  - 三级权限（私有/团队/公开）
  - 批注/评论/修改建议
  - 变更历史 + Diff + 回滚
  - 订阅推送（条目更新通知）
  - 关联图谱可视化
```

**验证指标**：
- 团队场景的使用率
- 协作冲突率（期望：低）
- 用户对权限模型的满意度

### Phase 3：分发与生态

**目标**：知识不仅能存，还能发出去

```
成果：
  - Skill 分发通道（Cursor/Claude Code/Windsurf）
  - 单条目生成公开分享页
  - 开放 API（第三方可查询）
  - Schema 模板可自定义
  - 与 PopWave Creative IDE 的 Agent 角色系统深度集成
  - 知识库 Agent 角色上线
```

**验证指标**：
- 分发通道的使用量
- 公开分享页的访问量
- 自定义模板的创建数

### Phase 4：智能化

**目标**：知识库自己也会成长

```
成果：
  - Agent 自动建议新条目（从创作过程中提取知识）
  - 自动关联推荐
  - 知识冲突检测（两个条目矛盾时报警）
  - 知识缺口分析（你经常查但知识库里没有的东西）
  - 经验库 → 知识库的自动沉淀管道
```

## 12. 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|:----|:------|:-----|:-----|
| **用户没有持续维护知识库的习惯** | 高 | 高 | 降低录入门槛：支持从公众号/网页一键采集；Agent 从创作中自动建议条目；浏览器插件 |
| **云端同步复杂度超预期** | 中 | 高 | Phase 1 纯本地优先，不同步；同步作为 Phase 2 的独立交付 |
| **权限模型在实际使用中不够用** | 中 | 中 | 从三级起步，预留扩展接口；用户反馈驱动是否细化 |
| **Agent 知识查询增加 token 消耗** | 高 | 中 | 查询结果摘要化；用户可控制每次查询的最大上下文量；高频条目在本地缓存 |
| **用户担心数据隐私（上传云端）** | 中 | 高 | 数据本地优先 + 端到端加密；用户可自托管；公开分享是对称加密链接 |
