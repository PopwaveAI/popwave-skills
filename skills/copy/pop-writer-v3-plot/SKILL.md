# pop-writer-v3-plot · 幕纲设计 v1.5.0

核心隐喻：幕纲 = 跨章叙事单元的活文档（4-10章/卡），是给创作子agent看的**导演指令集**。v1.5核心变化：**先推演故事，再提取结构**。单元故事推演是幕纲核心内容——先把故事讲一遍，节拍表/子线/红线全从推演中读出，不再凭空填表。

## 设计理念（v1.5）

- **先演后析**：不直接填表格。先写散文体故事推演（卷级beat级+单元级场景级），再从推演中提取结构化内容。解决机械填表根因
- **网文铁律前置**：铁律9条不作为事后检查清单，而是推演时必须遵守的创作约束，推演中即标注即检查
- **约束驱动涌现**：不预设路径，但预设约束；给方向不给路径
- **信息注入优先**：plot 设计 幕纲时从 library 匹配同类 幕纲作为结构标杆
- **判断前移**：信息需求判断在 plot 设计时一次性完成

## 三层推演架构

| 层级 | 推演产出 | 粒度 | 产出者 |
|:-----|:---------|:-----|:-------|
| 卷级 | 500-1000字beat级故事大纲 | 3-5个转折beat+高潮+收束 | step-1 |
| 单元级 | 每章200-400字散文体预演 | 场景级：画面/感官/情绪/决策/对白锚 | step-2 §3 |
| 章级 | 导演意图（从节拍表提取） | 字数段级：事件/情绪/爽点/钩子 | expert-writer Step0 |

> 卷级推演是骨架，单元级推演是血肉。节拍表/子线/红线从单元推演中提取，不是凭空设计。

## ❌ 质量红线

| 编号 | 红线 | 检查方式 |
|:-----|:-----|:---------|
| ❌1 | 创作节拍表每章必须含"设定引用"列（路径指针真实存在） | 逐章核对 |
| ❌2 | 每章必须有≥1个爽感释放点，且爽点类型明确 | 逐章核对 |
| ❌3 | 每章章末钩子必须是主角主动决策型，禁止被动接收型 | 抽查章末 |
| ❌4 | 幕纲活跃嵌套子线≤5条 | 计数嵌套子线条目 |
| ❌5 | plot设计幕纲时必须从library匹配同类幕纲（≥2张）作为参考 | 确认library参考卡路径已记录 |
| ❌6 | 单元红线2-4条且每条可验证 | 检查红线表述 |
| ❌7 | **单元故事推演必须产出（每章200-400字散文体）** | 检查推演段 |
| ❌8 | **节拍表内容必须从推演提取（非凭空填写）** | 抽查节拍表与推演一致性 |
| ❌9 | 幕纲设计前必须完成篇幅确认 | step-1 Step 0 篇幅确认环节未执行=硬阻塞 |
| ❌10 | **卷级故事大纲必须产出（500-1000字beat级）** | 检查 卷纲/推演/卷N-故事大纲.md |

## 路由

upstream `pop-writer-v3-seed`（产出写作参考/设定/）+ `pop-trope-library`（同类幕纲参考）→ **本skill** → downstream `expert-writer`

> plot 设计 幕纲如遇信息缺口，委托 `pop-research`（独立调研skill）补缺。

## 产出

- `卷纲/卷N-方向锚.md`（≤300字）：卷定位+起点终点四维度+主冲突
- `卷纲/推演/卷N-故事大纲.md`（500-1000字）：beat级故事骨架+幕纲切分表
- `卷纲/幕NNN-{名称}.md`（幕纲）：创作节拍表+嵌套子线+单元红线（从推演提取）
- `卷纲/推演/幕NNN-{名称}-推演.md`（推演过程文件）：每章200-400字散文体故事预演
