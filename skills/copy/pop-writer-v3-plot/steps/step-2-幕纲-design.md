# Step 2：幕纲设计

> 定位：v3.5 plot 的核心。设计跨章叙事结构——幕纲（4-10章/卡）。幕纲是唯一运行时活文档。
>
> **关键转向**：幕纲不是给架构师看的"设计文档"，而是给创作子 agent 看的"导演指令集"。用读者视角写节奏，用作者视角给约束。每行都要回答："读者在这一刻是什么感受？为什么不会滑走？"
>
> **v1.5 核心变化**：新增"单元故事推演"阶段——先把故事讲一遍，再从推演中提取节拍表/子线/结构。表格是提取产物，不是创作起点。

## 消费

- `卷纲/卷N-方向锚.md`（step-1 产出）
- `卷纲/推演/卷N-故事大纲.md`（step-1 产出，beat级骨架+幕纲切分表）
- **优先级1**：前一个 幕纲实际内容 `卷纲/幕NNN-{名称}.md`（arc 更新后）——必须（L2-001 除外）
- **优先级2**：`写作参考/设定/`（主角引擎/金手指/压力矩阵/冲突轴/核心卖点）——必须
- **优先级3**：library 匹配的同类 幕纲（2-3张）——必须
- **优先级4**：`卷纲/L3-NNN.md`（如有）——可选（L2-001 除外）
- `写作参考/索引.md`
- 用户输入

## 产出

- `卷纲/幕NNN-{名称}.md`（幕纲，唯一运行时活文档）
- （如有缺口）`写作参考/知识沉淀/{主题}.md`（由 pop-research 落盘）

## 做什么

### 1. 多层信息依赖读取（按优先级）

| 序 | 动作 | 说明 |
|:---|:-----|:-----|
| 1.1 | 读卷级故事大纲+L2切分表 | 从step-1产出的故事大纲中定位本卡覆盖哪些beat，知道本卡在整卷故事中的位置 |
| 1.2 | 读前一个 幕纲实际内容 | L2-001 跳过。读取 arc 更新后的物理坐标+子线状态，掌握主角当前状态、未回收伏笔 |
| 1.3 | 读写作参考/设定/ | 全局设定 ground truth：主角引擎、金手指、压力矩阵、冲突轴、核心卖点 |
| 1.4 | library 匹配同类 幕纲 | 确定当前 L1 大类，从 library 剧情库匹配 2-3 张同类 幕纲。**只参考结构骨架（情绪节奏/爽点分布/章末钩子类型），不复刻具体内容** |
| 1.5 | 读 L3 卡（如有） | 宏观参考。无则跳过 |

> **library 参考防污染**：参考"这组章节让读者经历什么情绪曲线"，不参考具体角色名/势力名/事件。自问：这个设计维度是结构性的还是内容性的？只参考前者。

### 2. 单元顶层设计

#### 2.1 元数据

单元编号 / 单元名称 / 单元类型（建置型/推进型/高潮型/余波型）/ 章节范围 / 叙事功能 / 核心成就 / 覆盖beat（来自卷级大纲切分表）。

#### 2.2 一句话单元（读者视角）

用一句话概括本单元给读者的核心体验，不是剧情简介：

- ❌ 不合格："索伦调查血祭现场，发现教会牵连。"
- ✅ 合格："让读者相信主角接手的案子不是普通谋杀，而是会咬人的东西。"

#### 2.3 物理坐标

- 时间轴：每章时间节点
- 地点轨迹：场景空间变化
- 人物表：角色+身份+本章功能+状态变化
- 力量体系现状：单元结束时的数值快照

### 3. 单元故事推演（核心！先讲故事，再填表格）

> **这是v1.5的核心变化**。不做这步直接填节拍表 = 机械填表。
>
> 推演不是列事件，是**把这个单元的故事从头到尾讲一遍**。像你给朋友安利一本小说——"第一章讲啥，第二章讲啥，超精彩的地方在……"——那种讲法。推演完，节拍表/子线/情绪弧线全从推演中"读"出来，不需要再编。

#### 3.1 推演输入

- 卷级故事大纲中本卡覆盖的beat（§1.1已读）
- 前幕纲的状态快照（§1.2已读）
- 设定文件（§1.3已读）
- library参考结构（§1.4已读）

#### 3.2 推演方法

对单元内**每一章**，写一段200-400字的散文体故事预演。要求：

**每章预演必须包含**：

- **开场画面**：这章第一个镜头是什么？空间/时间/感官信号。不是"主角在酒馆"，而是"主角推开酒馆门，烟味和劣质麦酒味扑面，角落里坐着两个人"
- **故事流动**：这章发生了什么？不是事件列表，是"故事讲一遍"。2-3个场景的叙事流动，有起承转合
- **情绪转折**：主角/读者情绪从X到Y。标注转折触发点
- **角色决策**：主角在这章做了什么主动决策驱动了剧情？不是"主角被攻击"，是"主角决定去暗杀巴拉斯，因为……"
- **对白锚点**：1-2个关键对白/数据的锚——不是写完整对白，是"薇薇安说了一句关于卖身的话"或"面板弹出背刺12点伤害"这种画面级锚点
- **章末画面**：这章最后一个镜头是什么？留了什么悬念？

#### 3.3 网文铁律在推演中的落地

推演时必须对照以下铁律，违反即改：

**① 情绪碾压逻辑**
推演时先确保每章有情绪高潮点（爽/燃/虐/悬念），再管逻辑。如果某个情节逻辑不够严但情绪到位——保留，在预演中标记`[逻辑待圆]`。写完再圆比写不出来强。

**② 默认快节奏**
推演中不写"平淡的几天过去了"。每章的事件密度要够——如果一章预演中只有1个事件转折，这章节奏太慢了，合并或加速。目标：每章2-4个事件转折。

**③ 爽感驱动**
每章预演中必须标注爽感释放点，用括号标注类型：
- `[爽点：压力→突破]` 主角被围困→找到弱点反杀
- `[爽点：弱→强]` 主角被看不起→一击秒杀
- `[爽点：未知→揭示]` 谜团揭开→真相震惊
- `[爽点：被欺→反击]` 被欺压→暴力反击
- `[爽点：行动→回报]` 行动后即时获得数值/技能/物质

**④ 主角主动**
每章预演中的核心转折必须由主角决策驱动。推演时自问："这章是谁在驱动故事？如果答案是'敌人来找主角'，改写成主角主动出击或主动反应。"

**⑤ 危机不间断**
推演中章与章之间的衔接不能有"安全区"。第一章的危机解决必须直接触发第二章的危机。在预演的章末画面中标注`[下章危机：XXX]`。

**⑥ 行动即回报**
推演中主角的每个重要行动必须有即时反馈。自问："主角这章做了什么？他得到了什么？"如果答案是"什么都没得到"，这章行动没有回报，读者会觉得没推进。

**⑦ 悬念不闭合**
每章预演的章末画面必须包含"揭示了一个谜 + 抛出了一个更大的谜"。自问："读者看完这章知道了什么新信息？但又冒出了什么更大的问题？"

**⑧ 章末追读**
每章预演的最后一个画面必须是"读者想看下一章"的时刻。情绪到位就收，禁止在预演中写"然后主角回去休息了"这种收尾。

**⑨ 设定浸泡**
推演中需要引入设定时，不独立抛出，融入事件自然呈现。标注`[设定浸泡：通过XXX事件呈现YYY设定]`。禁止在预演中写"本章需要交代金手指升级机制"这种设定灌输。

#### 3.4 推演产出

推演产出为**独立文件** `卷纲/推演/幕NNN-{名称}-推演.md`，格式：

```markdown
# 幕NNN 故事推演

### ch001 · {情绪目标}

{200-400字散文体预演}

**爽点**：[爽点类型]
**对白锚点**：{1-2个关键对白/数据锚}
**章末画面**：{最后一个镜头}
**[下章危机：XXX]**

### ch002 · {情绪目标}

{200-400字散文体预演}

...
```

> 推演是**过程性文件**，不是正式幕纲的一部分。推演完成后，节拍表/子线/红线从推演中提取写入幕纲；推演文件保留供create子agent写作时参考感受故事流动。
>
> **推演完成后**，以下所有结构化内容都从推演中**提取**，不再凭空填写。

### 4. 创作节拍表（从推演提取）

> 节拍表不是"设计"，是"提取"——从§3推演中读出每章的字数段节奏。

**每章一张节拍表**，按字数分段。

#### 4.1 节拍表格式

```markdown
### chXXX · {本章情绪目标}

**情绪目标**：本章想让读者感受到什么？（如"紧张→震惊→期待"）
**爽点类型**：压力→突破 / 弱→强 / 未知→揭示 / 被欺→反击 / 行动→回报（选1-2个）
**章末钩子**：{主角主动动作} + {悬念}（必须是主角决策型，禁止被动接收型）

| 字数段 | 场景/动作 | 读者情绪 | 设定引用 | 备注 |
|:-----|:----------|:---------|:---------|:-----|
| 0-300字 | 开篇画面 | 好奇/紧张 | 路径指针 | 禁止大段铺垫 |
| 300-700字 | 事件推进 | 投入 | 路径指针 | 主角开始行动 |
| 700-1200字 | 转折/冲突 | 爽感释放 | 路径指针 | 爽点具体位置 |
| 1200-1800字 | 余波/新信息 | 满足+好奇 | 路径指针 | 抛出更大的新谜团 |
| 1800-2500字 | 章末钩子 | 必须看下一章 | 路径指针 | 主角主动决策 |
```

#### 4.2 提取规则

节拍表的每个字段从推演中提取：

- **情绪目标** ← 推演中的情绪转折标注
- **爽点类型** ← 推演中的`[爽点：XXX]`标注
- **场景/动作** ← 推演中的故事流动，按字数段拆分
- **读者情绪** ← 推演中每个段落读出的情绪
- **设定引用** ← 推演中的`[设定浸泡：XXX]`标注对应的设定文件路径
- **备注** ← 推演中的对白锚点/章末画面/下章危机标注

#### 4.3 设定引用指针

节拍表"设定引用"列，标注本字数段涉及的设定文件路径。格式：

```
写作参考/设定/金手指.md#升级机制, 写作参考/设定/主角引擎.md#行为准则
```

要求：
- 每章至少标注 1 个设定文件路径
- 路径必须真实存在
- 是路径指针，不是内容副本

### 5. 嵌套子线与咬合（从推演提取）

> 子线不是"设计"出来的，是推演中故事自然形成的线索——提取它们。

#### 5.1 嵌套子线

从推演中读出每条故事线索的推进：

```markdown
### 子线A：{名称}（状态：延伸/闭合）
| 章节 | 事件 | 不可逆变化 |
```

要求：活跃子线≤5条（红线❌4）。

#### 5.2 子线咬合

从推演中读出子线如何互相触发：

```
ch003咬合: ⑤铁钳帮反杀 → 产生①系统升级 → ①系统升级 → 产生④灵能耳鸣 → 因果链: ⑤ → ① → ④
```

### 6. 单元红线

本单元特有的、必须遵守的创作约束。不是通用铁律（铁律已内联在推演阶段），而是**本单元独有的硬性要求**。

从推演中提取：推演中哪些情节是后续单元的伏笔？哪些事件是不可逆的？哪些状态必须在单元结束时达成？

格式：

```markdown
## 单元红线
1. ch001必须完成：系统觉醒 + 第一次战斗 + 第一次击杀反馈
2. 每章章末必须是主角主动决策型钩子，禁止被动接收型
3. 本单元结束前必须建立"DEA体制不可信"的认知
```

要求：2-4条，每条含一个可验证的结果（不是态度要求）。

### 7. 信息需求判断

#### 7.1 盘点本地已有

扫描 `写作参考/索引.md` + library 设定库。

#### 7.2 识别缺口

对照推演和节拍表的事件设计，判断哪些场景/设定本地没有。

#### 7.3 调研补缺

调用 pop-research，返回路径指针写入节拍表"设定引用"列。

### 8. 产出 幕纲存盘

将完整幕纲写入 `卷纲/幕NNN-{名称}.md`。幕纲结构：

```
# 幕NNN · {名称}

## 元数据
## 一句话单元
## 物理坐标
## 创作节拍表（§4产出，从推演提取）
## 嵌套子线与咬合（§5产出，从推演提取）
## 单元红线（§6产出，从推演提取）
```

> 推演文件 `卷纲/推演/幕NNN-{名称}-推演.md` 是独立的过程性文件。create子agent写作前读推演感受故事流动，读幕纲获取结构指导（节拍表/子线/红线）。

## 门禁（8条红线）

| 编号 | 红线 | 检查方式 |
|:-----|:-----|:---------|
| ❌1 | 节拍表每章"设定引用"列非空且路径存在 | 逐章核对 |
| ❌2 | 每章有≥1个爽感释放点，且类型明确 | 逐章核对 |
| ❌3 | 每章章末钩子必须是主角主动决策型，禁止被动接收型 | 抽查章末 |
| ❌4 | 嵌套子线≤5条 | 计数 |
| ❌5 | library 参考已记录≥2张同类幕纲路径 | 确认路径 |
| ❌6 | 单元红线≤4条且每条可验证 | 检查红线表述 |
| ❌7 | 单元故事推演已产出（每章200-400字） | 检查推演段 |
| ❌8 | 节拍表"场景/动作"列内容从推演中提取（非凭空填写） | 抽查节拍表与推演一致性 |
