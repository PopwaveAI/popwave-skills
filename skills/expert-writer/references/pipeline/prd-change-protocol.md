# 种子变更传播协议

> **用途**：当种子文档的六要素与正文/活记忆不一致时，定义变更标记、传播路径和硬边界违反阻断机制。
>
> **核心矛盾**：种子被设计为"活种子"（可生长/修剪），但真实创作是迭代探索。emerge 在涌现写作中可能发现需要改变种子要素的决策（压力源变化/主角弧线调整/世界规则演进）。本协议让这种变更有序传播，而非无声漂移。

---

## 一、种子核心要素清单

以下要素的变更必须触发传播协议：

| 类别 | 要素 | 种子文档中的位置 | 典型变更场景 |
|:---|:---|:---|:---|
| **压力矩阵** | 压力源/机制/杀人方式/显性表现 | 种子文档·要素1 | emerge 涌现出新压力源 |
| **主角引擎** | 驱动力三层/缺陷/决策风格 | 种子文档·要素2 | 主角弧线在写作中发生偏移 |
| **金手指** | 核心创意/能力解锁/限制/数值风格 | 种子文档·要素3 | emerge 发现能力解锁节奏需调整 |
| **冲突轴+活跃线索** | 四层生态/活跃线索清单 | 种子文档·要素4 | arc 校准发现线索失效 |
| **成长路径** | 能力里程碑/心态变化/资源积累 | 种子文档·要素5 | emerge 涌现出新的成长节点 |
| **目的地** | 5-10个模糊路标 | 种子文档·要素6 | arc 校准发现路标需调整 |

## 二、变更传播路径

### 2.1 emerge→种子生长（种子版本管理）

```
emerge 涌现出新要素（新压力源/新角色弧线节点/新伏笔等）
    ↓
emerge Step 5（记忆更新）：机械写入种子文档
  - 要素内容追加/更新
  - version+1
  - changelog 追加：版本号 + 时间 + 变更内容 + 触发章号
  - last_updated_ch 更新为当前章号
    ↓
emerge 继续（新种子版本立即生效，下一章消费更新后的种子）
```

### 2.2 arc→种子修剪（失效要素处理）

```
arc 校准发现种子要素已失效（线索断裂/角色弧线偏移/世界规则矛盾）
    ↓
arc 修剪种子文档：
  - 失效要素内容移入 closed_zones（已关闭区）
  - version+1
  - changelog 追加：版本号 + 时间 + 修剪内容 + 触发原因
    ↓
emerge 重新消费修剪后的种子（下一章按新种子涌现）
```

### 2.3 用户→种子修改（用户主动变更）

```
用户要求修改种子要素（加穿越者/改力量体系/换核心矛盾）
    ↓
expert-writer 退回 v3-seed 修改种子文档：
  - 要素内容更新
  - version+1
  - changelog 追加
  - 一致性检查：活记忆 + 已有正文是否与新种子矛盾
    ↓
如有矛盾 → 退回 v3-emerge 修正受影响章节
如无矛盾 → 继续涌现写作
```

## 三、一致性检查（阶段间快检）

> 进入下一阶段前，对比种子文档六要素 + 活记忆状态。

| 检查项 | 检查方式 | 不一致处理 |
|:---|:---|:---|
| 种子要素 vs 活记忆 | 六要素的 last_updated_ch vs 活记忆最近 event 章号 | 种子要素休眠过久（>15章）→ 标记需关注 |
| 种子要素 vs 正文 | 正文中的设定引用 vs 种子要素内容 | 不一致 → 退回 v3-emerge 修正 |
| 活记忆 baseline vs events | baseline 章号 + event 数量 vs 正文章号 | 不一致 → 补齐活记忆 |

## 四、变更标记

在项目总控.md 中维护种子变更日志：

```markdown
## 种子变更日志

| 版本 | 时间 | 变更类型 | 变更内容 | 触发者 | 触发章号 |
|:---|:---|:---|:---|:---|:---|
| v{版本} | {时间} | 生长/修剪/用户修改 | {变更描述} | emerge/arc/用户 | ch{编号} |
```
