# 核心原则（3 条）

> 从 SKILL.md 下沉。设计决策参考，不是执行步骤。

---

## 原则 1：上瘾优于工程

**含义：** 读者上瘾 > 结构完美。当工程严谨性和阅读爽感冲突时，选爽感。

**实践：**
- 章末钩子比章中伏笔更重要 — 读者翻页的动力来自"下一章会怎样"
- 信息密度 > 信息精度 — 每段推进两件事，比每段精确描述一件事更上瘾
- 爽点频率：1 个中等爽点 + 5 个微爽点 per chapter，缺爽点的章必须重写
- 节奏 > 逻辑 — 逻辑可以有小漏洞，节奏不能断

**反例：**
```
❌ WRONG: 为了设定严谨，用 3 段解释力量体系的运作原理
✅ CORRECT: 用 1 句话带过原理，用 2 段写主角用这个力量打脸
```

---

## 原则 2：升级优先于新建

**含义：** 已有产出升级 > 推倒重写。修改不推倒重来，只动直接受影响的部分。

**实践：**
- 用户追加设定 → 融合到写作参考/设定/对应文件，不打补丁不重写
- 修改设定 → 只改该文件 + 联动检查活记忆一致性，不重写所有设定
- 弧线校准修剪 → 只移除失效要素到写作参考/已废弃/，不重写L2卡
- 三步执行：影响评估 → 局部修改 → 联动验证

**反例：**
```
❌ WRONG: 用户要加一个反派 → 重写整个L2卡
✅ CORRECT: 评估影响范围 → 只改受影响的设定文件 → 验证活记忆一致性
```

---

## 原则 3：集中合同优于分散声明

**含义：** 管线顺序由 expert-writer 集中管理 > 各子 skill 自己声明上下游。

**实践：**
- 管线顺序的单一真相源 = `pipeline/manifest.md`（本 skill 持有）
- 子 skill 的 pipeline 字段 = 知识依赖 + 产出流向，不是执行顺序
- 首次加载时灌入上下文，不依赖子 skill 自己推导顺序
- 项目进度以文件系统为准，不以声明为准

**反例：**
```
❌ WRONG: 各子 skill 在自己的 SKILL.md 声明 upstream/downstream，agent 拼接推导顺序
✅ CORRECT: expert-writer 持有 pipeline/manifest.md 硬编码顺序，首次加载灌入
```

**事故复盘：** 6.18 项目 B 中 agent 跳过种子设计阶段，因为 expert-writer 的 SKILL.md 声明 upstream 缺中间节点。修复方案就是集中合同。
