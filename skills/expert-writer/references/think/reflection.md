# 四层审视框架（通用层）

> expert-writer 只做通用三问 + 风险标记 + 退回决策。
> L1-L4 详细清单由各子 skill 自管。

---

## 通用检查三问

| # | 问 | 检查方式 |
|:-:|:---|:---------|
| 1 | **产出回答了用户问的问题吗？** | 对比用户原始需求和产出结果 |
| 2 | **有没有超出 scope 的多余产出？** | 用户没要求的东西写多了 = 浪费 |
| 3 | **有没有明显的盲点被忽略？** | "看起来对但站不住"的结论 |

## 质量信号

| 信号 | 行动 |
|:-----|:-----|
| 完美覆盖需求 | 直接交付，一句话总结 |
| 小细节偏了 | 标记偏差，再问用户是否接受 |
| agent 偷懒 | 人工检查 |
| 南辕北辙 | 不退——复盘路由错在哪 |

## 风险标记

| 优先级 | 含义 | 行动 |
|:------|:-----|:------|
| P0 | 不改会出大问题 | 立刻退回子 skill 重做 |
| P1 | 影响后续写作质量 | 建议用户修，但不强制 |
| P2 | 可以放着，以后再说 | 记录到项目总控.md |

## 退回决策

| 问题类型 | 退回到 |
|:---------|:-------|
| 产出文件格式/位置错误 | 原子 skill |
| 一致性问题（entity-snapshot/角色卡） | 原子 skill |
| 质量不达标 | pop-writer-qa（质检）→ 原子 skill（修复） |
| 活人感不足 | pop-writer-prose（局部重写） |

> **expert-writer 不自己做 L1-L4 详细检查。** 子 skill 有自己的落盘检查点、一致性校验、QA 三层审查。expert-writer 只做通用三问，发现问题就退回子 skill。
