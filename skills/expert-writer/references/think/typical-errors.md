# 典型错误防范（调度层）

> 只列 expert-writer 调度层面的错误。子 skill 内部的错误由各子 skill 自管。

---

## 错误 1：不读子 SKILL.md 就路由

```
❌ WRONG: 用户说"写正文"→ 直接跳到 expert-writer(7步循环)，不读 expert-writer 的 step-2-* 文件
✅ CORRECT: 先 Get-Content -Encoding UTF8 -Raw 加载 steps/step-2-execute.md 全文，再按其 SOP 执行
```

## 错误 2：用 Read 工具读取 skill 文件被截断

```
❌ WRONG: 用 Read 工具读 SKILL.md，文件超过 2000 行被截断
✅ CORRECT: 用 skill_view 或 Get-Content -Encoding UTF8 -Raw，读取后做截断检测
```

## 错误 3：决策点跳过用户确认

```
❌ WRONG: 种子设计闸门未等用户确认就继续推进到涌现写作
✅ CORRECT: 闸门（种子设计/弧线校准）必须等待用户点头
```

## 错误 4：管线跳步

```
❌ WRONG: 种子未完成就直接路由到 expert-writer(涌现写作环)
✅ CORRECT: 种子设计 → 涌现写作环 ↔ 弧线校准，种子文件夹未产出不可进入涌现写作
```

## 错误 5：长文全量贴入对话

```
❌ WRONG: 写完 3000 字正文后全量贴入对话窗口
✅ CORRECT: 文件写入后对话只留摘要（≤200字）+ 文件路径
```

## 错误 6：不更新项目总控.md

```
❌ WRONG: 阶段完成后不更新项目总控的进度条和产出物清单
✅ CORRECT: 每次阶段完成后更新项目总控.md 的 current_stage + 产出物状态 + 执行顺序日志
```

## 错误 7：引导语不基于项目实际状态

```
❌ WRONG: Reflect 末尾给固定模板引导语，不检查项目实际进度
✅ CORRECT: 读项目总控.md + 活记忆/活记忆.yaml，基于实际状态给引导
```

## 错误 8：expert-writer 自己做详细质检

```
❌ WRONG: 涌现写作环完成后 expert-writer 自己做五问反思+种子生长判断
✅ CORRECT: expert-writer 只做通用三问+宏观检查，详细质检由 质检子skill(pop-writer-v3-qa)处理
```

## 错误 9：不捕获用户心智模型就开工

```
❌ WRONG: 用户说"设计境界命名"→ 直接抛 3 个花哨方案让用户选
          → 用户否定 3 次后才说"我要传统的"
          → 最终方案和初版完全不同，浪费 5+ 轮对话
✅ CORRECT: 先问两个问题再给方案：
   ① "你心中这个体系的总体框架是几层几阶？"
   ② "命名风格偏好什么方向？（传统/暗黑/兵器系/自创）"
   抓住结构/风格两个锚点再出方案，方案不超过 2 个方向。
```

**适用范围：** v3-seed（种子设计中的结构性选择）、任何需要用户做"结构性选择"的场景。

**判断方法：** 如果这个问题可以拆成"先定结构再填充名字"，那就是结构性选择。先问结构，再问风格，最后给填充方案。

## 错误 10：重大设定变更不做全链影响声明

```
❌ WRONG: 用户说"加穿越者设定+圣者浩劫升级"
          → 只更新种子文件夹中 1 个要素
          → 问"需要我继续更新吗"
          → 用户发现多个下游文件全部过时 → 用户发作
✅ CORRECT:
   ① 先扫描种子六要素 + 活记忆 + 已有正文
   ② 输出"影响范围声明"给用户（逐要素标注 受影响/不受影响/不确定）
   ③ 用户确认范围后再逐要素更新种子（version+1）
   ④ 不询问"要不要更新"——机械传播是 agent 的责任，不是用户的决策
```

**原理：** 穿越者、圣者浩劫这种"框架级设定"，会影响种子六要素中的多个要素（压力矩阵/主角引擎/世界规则/成长路径），不存在"改 1 个要素就够了"的可能性。如果有，说明你的变更声明有遗漏。

## 错误 11：把机械传播问成用户决策

```
❌ WRONG: 改完穿越者设定后问"需要我继续更新活记忆和方向提示吗？"
✅ CORRECT: 活记忆和方向提示显然受穿越者影响，直接更新。
            只在该问的地方问：种子要素的具体内容需要用户确认 → 问。
            明显受波及的部分 → 直接改，不改才需要说明原因。
```

**判断方法：** 问自己——"这个更新需要用户的创意决策吗？"
- 需要创意决策（主角名选什么/感情线走向）→ 问
- 只需要机械传播（穿越者改了→活记忆要体现穿越者影响）→ 直接做

**用户偏好锚定：** 用户明确说过要"act don't ask"——能直接做的事不要去问。
