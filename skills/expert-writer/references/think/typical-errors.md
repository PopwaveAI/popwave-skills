# 典型错误防范（调度层）

> 只列 expert-writer 调度层面的错误。子 skill 内部的错误由各子 skill 自管。

---

## 错误 1：不读子 SKILL.md 就路由

```
❌ WRONG: 用户说"写正文"→ 直接跳到 pop-writer-prose，不读 prose 的 SKILL.md
✅ CORRECT: 先 Get-Content -Encoding UTF8 -Raw 加载 prose/SKILL.md 全文，再按其 SOP 执行
```

## 错误 2：用 Read 工具读取 skill 文件被截断

```
❌ WRONG: 用 Read 工具读 SKILL.md，文件超过 2000 行被截断
✅ CORRECT: 用 skill_view 或 Get-Content -Encoding UTF8 -Raw，读取后做截断检测
```

## 错误 3：决策点跳过用户确认

```
❌ WRONG: creative 闸门未等用户确认就继续推进到 world
✅ CORRECT: 4 个闸门（creative/plot/chapter/prose）必须等待用户点头
```

## 错误 4：管线跳步

```
❌ WRONG: creative 完成后直接跳到 character，跳过 world
✅ CORRECT: creative → world → character → plot → chapter → prose → qa，硬性不可跳跃
```

## 错误 5：长文全量贴入对话

```
❌ WRONG: 写完 3000 字正文后全量贴入对话窗口
✅ CORRECT: 文件写入后对话只留摘要（≤200字）+ 文件路径
```

## 错误 6：不更新项目总控.md

```
❌ WRONG: 阶段完成后不更新项目总控的进度条和产出物清单
✅ CORRECT: 每次阶段完成后更新项目总控.md 的 current_stage + 产出物状态 + 执行顺序日志
```

## 错误 7：引导语不基于项目实际状态

```
❌ WRONG: Reflect 末尾给固定模板引导语，不检查项目实际进度
✅ CORRECT: 读项目总控.md + state-log.yaml，基于实际状态给引导
```

## 错误 8：expert-writer 自己做详细质检

```
❌ WRONG: prose 产出后 expert-writer 自己做 L1-L4 详细检查
✅ CORRECT: expert-writer 只做通用三问，详细质检退回 pop-writer-qa
```

## 错误 9：不捕获用户心智模型就开工（★新增）

```
❌ WRONG: 用户说"设计境界命名"→ 直接抛 3 个花哨方案让用户选
          → 用户否定 3 次后才说"我要传统的"
          → 最终方案和初版完全不同，浪费 5+ 轮对话
✅ CORRECT: 先问两个问题再给方案：
   ① "你心中这个体系的总体框架是几层几阶？"
   ② "命名风格偏好什么方向？（传统/暗黑/兵器系/自创）"
   抓住结构/风格两个锚点再出方案，方案不超过 2 个方向。
```

**适用范围：** pop-writer-world（境界命名）、pop-writer-plot（主线结构）、pop-writer-character（角色定位）、任何需要用户做"结构性选择"的场景。

**判断方法：** 如果这个问题可以拆成"先定结构再填充名字"，那就是结构性选择。先问结构，再问风格，最后给填充方案。

## 错误 10：重大设定变更不做全链影响声明（★新增）

```
❌ WRONG: 用户说"加穿越者设定+圣者浩劫升级"
          → 只更新 3 个文件（角色卡/起点快照/L1-03）
          → 问"需要我继续更新吗"
          → 用户发现 14+ 下游文件全部过时 → 用户发作
✅ CORRECT:
   ① 先扫描全链文件清单（PRD→L1 七件套→数值→宪法→卷纲→剧情线→L2单元卡→设计包→正文）
   ② 输出"影响范围声明"给用户（逐文件标注 受影响/不受影响/不确定）
   ③ 用户确认范围后再逐文件更新
   ④ 不询问"要不要更新"——机械传播是 agent 的责任，不是用户的决策
```

**原理：** 穿越者、圣者浩劫这种"框架级设定"，会穿透 creative → world → character → plot → chapter → prose 全链，不存在"改 3 个文件就够了"的可能性。如果有，说明你的变更声明有遗漏。

## 错误 11：把机械传播问成用户决策（★新增）

```
❌ WRONG: 改完穿越者设定后问"需要我继续更新动态升级表和世界蓝图吗？"
✅ CORRECT: 动态升级表和世界蓝图显然受穿越者影响，直接更新。
            只在该问的地方问：起点快照中的主角名需要用户确认 → 问。
            明显受波及的文件 → 直接改，不改才需要说明原因。
```

**判断方法：** 问自己——"这个更新需要用户的创意决策吗？"
- 需要创意决策（主角名选什么/感情线走向）→ 问
- 只需要机械传播（穿越者改了→动态升级表要体现穿越者成长节奏）→ 直接做

**用户偏好锚定：** 用户明确说过要"act don't ask"——能直接做的事不要去问。
