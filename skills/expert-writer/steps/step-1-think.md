## 1. 管线锚定与会话恢复（每次新会话优先执行）

> **模式检查（★v8.0.0）**：读取项目总控.md 的「管线模式」字段，决定走 v2 或 v3 逻辑。
> - **v2 雪花式** → 下方步骤 1-5（8阶段线性逻辑，保持原有行为）
> - **v3 涌现式** → 下方步骤 1-5（3阶段循环逻辑）+ §1.v3 补充状态感知
> - 字段不存在 → 按 v2 处理（兼容旧项目）

1. **加载管线合同**：`Get-Content -Encoding UTF8 -Raw references/pipeline/manifest.md`
   - v2 管线顺序：creative → world → character → plot → chapter → prose → qa
   - v3 管线顺序：种子设计(pop-writer-v3-seed) → 涌现写作环(pop-writer-v3-emerge) ↔ 弧线校准(pop-writer-v3-arc)
2. **加载项目总控**：检查项目根目录 `项目总控.md`
   - 存在 → 读取，获取当前阶段和执行顺序日志
   - 不存在 → 推断阶段并初始化：
     a. 按管线顺序扫描文件存在性（`全书立项PRD.md` → `素材储备池/` → `L1-01~06` → `角色卡` → `剧情设计/` → `章节设计包/` → `正文/`）
     b. 最后一个产出齐全的阶段 = 当前已完成阶段，下一阶段 = 路由目标
     c. 从 `references/project/master-control.tpl.md` 初始化项目总控.md（扫描结果填充项目现状）
     d. 输出 ⚠️ "项目总控.md 已自动初始化（基于文件扫描推断，请确认准确性）"
3. **版本漂移检测**：项目总控的管线版本 vs 当前 SKILL.md 版本不一致 → 刷新下半屏
4. **管线断裂检测**：执行顺序日志中的阶段在 manifest 顺序上是否连续
   - 不连续 → ⚠️ 告知用户断裂风险，询问是否回退补齐
5. **进度摘要**：`📊 [项目] | 第N章 | 幕M，上次停在 [阶段]。继续吗？`

## 2. 意图识别

### v2 模式（原有逻辑不变）

1. 从用户消息提取关键词，对照 SKILL.md v2 路由表定位目标子 skill：`pop-writer-{阶段}`
2. 检查管线前置条件：上游阶段是否完成（查项目总控.md 的执行顺序日志）
3. 上游未完成 → 告知用户缺什么阶段，建议先补齐
4. 上游已完成 → 进入 Execute

### v3 模式（★v8.0.0新增）

1. 从用户消息提取关键词，对照 SKILL.md v3 路由表定位目标子 skill：`pop-writer-v3-{seed|emerge|arc}`
2. **v3 信息需求判断**：emerge 路由时，根据上一章方向提示 + 活记忆状态，判断本章写作前需要补充哪些信息（套路库查询/文风锚定/伏笔状态等），形成信息需求清单传递给 Execute 层
3. **v3 智能调度判断**：

| 调度条件 | 路由目标 | 说明 |
|:---------|:---------|:-----|
| 正常写作 | pop-writer-v3-emerge | 未触发弧线校准、反思通过 |
| 弧线校准触发点（每10-20章 / 事件触发 / 用户触发） | pop-writer-v3-arc | 检查章号差值 + 用户指令 + 重大剧情转折 |
| emerge 反思不通过 | 回退重写当前章 | emerge 内五问反思未过 → 回退到 emerge 重写该章 |

**精简模式**：用户说"直接写/快一点/跳过解释" → 少解释、多执行、保留闸门确认。

> **前置条件详细检查由子 skill 自管。** expert-writer 只检查"上游阶段是否完成"，不检查具体文件是否存在。子 skill 自己有前置阻断检查。

## 3. 闸门确认

路由到子 skill 前，如果是以下阶段，必须等用户确认：

### v2 模式闸门（原有）

| 阶段 | 闸门内容 |
|:-----|:---------|
| creative | 方案A：当前设定 / 方案B：调整方向 / 4. 我重新来 |
| plot | 方案A：当前卷+幕 / 方案B：调整某卷 / 4. 我重新来 |
| chapter | 方案A：当前骨架 / 方案B：调整某事件 / 4. 我重新来 |
| prose | 方案A：当前渲染 / 方案B：调整某段 / 4. 我重新来 |

### v3 模式闸门（★v8.0.0新增）

| 阶段 | 闸门内容 |
|:-----|:---------|
| v3-seed（种子设计完成） | 种子七要素总览 → 方案A：确认种子 / 方案B：调整某要素 / 4. 我重新来 |
| v3-arc（弧线校准完成） | 校准报告六项检查结果 → 方案A：确认校准 / 方案B：调整某项 / 4. 我重新来 |

> v3-emerge 每章闸门由 emerge 内部五问反思处理，expert-writer 不重复——只做宏观检查（见 step-3-reflect）。

跳过闸门 = 违规。
