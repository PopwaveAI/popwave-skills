# Step 0：本章规划

> 参考: expert-writer 涌现写作环
> 消费: 种子文件夹(六要素) + 活记忆 + 上一章正文 + 方向提示 + 网文爽感机制
> 产出: `章节规划/chXXX-plan.md`（落盘+用户确认后再进Step 1）

## 目的

写之前先想明白。看种子+上一章正文，判断这章要达成什么——定位/目标/信息释放/信息需求。规划只定义"达成什么"，把"怎么达成"（场景设计）留给 create 环节涌现。规划做完，再进 Step 1 强制化信息获取。

## 读什么

| # | 文件 | 为什么 |
|:-:|:-----|:-------|
| 1 | `种子/文件夹`（当前版本） | 六要素：压力矩阵/主角引擎/金手指/冲突轴+活跃线索/成长路径/目的地——这章可用的故事引擎 |
| 2 | `活记忆/活记忆.yaml`（最后baseline+event） | 七组件当前状态——压力倒计时到哪了、哪些线索超期、节奏趋势 |
| 3 | `正文/ch{上一章}.md`（末尾500字） | 衔接——上章留了什么悬念，本章从哪里接 |
| 4 | 上一章方向提示（活记忆event.direction_hint） | 上章质检标记的待处理项（必须给奖励/必须加压/必须触面某线索） |
| 5 | `references/emerge-loop/网文爽感机制.md` | **写作法则**——本章规划必须对照检查 |

## 做什么

### 1. 故事状态感知

从活记忆七组件提取当前状态：

| 组件 | 提取什么 | 用于判断 |
|:-----|:---------|:---------|
| 压力状态 | 哪些倒计时在走？剩多少？ | 本章必须推进哪个压力源 |
| 种子追踪 | 哪些线索活跃？哪些超期预警？ | 本章该触面哪条线索 |
| 节奏日志 | 最近3章的事件密度趋势 | 本章该加速还是减速 |
| 战力曲线 | 主角当前能力水平 | 本章能力使用边界 |
| 目的地进度 | 最近的目的地进展 | 本章是否推进 |
| 角色状态 | 主角心理/资源/关系 | 驱动力哪层在激活 |

### 2. 本章规划（5个决策点）

基于状态感知+种子+方向提示，做5个规划决策。规划只定义"达成什么"，不定义"怎么写场景"——场景设计交给 create 环节涌现。

**① 本章定位**：这章在全书的位置/作用？
- 类型判断：开篇 / 转折 / 收束 / 过渡
- 作用陈述：这章在整体叙事弧中承担什么功能（不写具体场景）

**② 主角目标**：主角这章要干嘛？
- 元目标陈述：主角本章想达成什么（一句话，不写具体场景与动作）
- 目标来源：种子冲突轴哪层生态？哪个压力源驱动？哪条活跃线索牵引？

**③ 读者获得**：读者读完能获得什么？
- 至少1项，从以下维度选：信息（世界/规则/秘密）/ 爽感（技能/装备/金钱/地位）/ 情感（共鸣/宣泄）/ 悬念（钩子）
- 对照`references/emerge-loop/网文爽感机制.md`的奖励类型，爽感类必须即时兑现

**④ 剧情线目标**：推进哪≥3条线索？
- 从活记忆种子追踪选≥3条活跃线索，标注推进方式（触面/深化/回收）
- 不写具体场景，只标注每条线本章的目标状态变化

**⑤ 危机与钩子方向**：压力源怎么走 + 章末钩子方向？
- 至少1个压力源在本章逼近或爆发（给方向，不写具体事件）
- 章末钩子只给方向：即时危机型（下章开头需解决），不是远期悬念型（如"7天后决战"）

### 3. 网文爽感机制对照检查

将本章规划对照`references/emerge-loop/网文爽感机制.md`的10条法则。每条法则必须附 plan 中的设计证据，不能空泛打勾：

| 法则 | 本章设计证据（取自5决策点，非空泛打勾） | 如不满足的调整 |
|:-----|:--------------------------------------|:-------------|
| 压力机制 | {压力源如何逼近/爆发——引决策点⑤} | 补充压力源逼近 |
| 即时奖励循环 | {读者获得中的爽感项如何即时兑现——引决策点③} | 补充爽点 |
| 不间断危机链 | {章末危机钩子如何承接上章压力——引决策点⑤} | 补充章末危机钩子 |
| 高事件密度(8-12) | {本章定位决定的事件密度取向——引决策点①} | 补充密度 |
| 主角主动决策 | {主角目标如何驱动本章——引决策点②} | 改被动为主动 |
| 养成感可见 | {读者获得/剧情线带来的能力资源变化——引决策点③④} | 补充能力/资源变化 |
| 信息披露节奏 | {信息释放清单如何分层释放——引读者获得/信息释放} | 补充世界信息 |
| 80%展示20%内省 | {展示与内省配比预期} | 标注内省点 |
| 多线交叉(≥3线) | {≥3条线索如何交叉——引决策点④} | 补充线索推进 |
| 情绪闭环 | {读者获得的情感维度如何闭环——引决策点③} | 检查情绪弧线 |

无设计证据或证据空泛的法则，回到步骤2调整规划。

### 5. 落盘+用户确认

将任务 list 写入 `章节规划/chXXX-plan.md`（直接落盘步骤4的 Markdown 模板内容，不要再包一层 YAML）：

**向用户展示规划摘要并请求确认：**
- 展示任务 list 的5个决策点（定位 / 目标 / 读者获得 / 剧情线 / 钩子方向）
- 展示法则对照结果（每条附设计证据，非空泛打勾）
- 等待用户确认

**用户确认后：**
- 状态改为 `已确认`
- 进入 Step 1 信息获取

**用户要求修改：**
- 按用户反馈调整规划
- 更新文件后重新确认

## 门禁

| 检查项 | 失败动作 |
|:-------|:---------|
| 种子文件夹不存在 | 终止，提示先完成pop-writer-v3-seed |
| 活记忆不存在 | 终止，提示先初始化活记忆 |
| 网文爽感机制10条法则全部对照且每条附设计证据（红线❌5） | 未对照/证据空泛的法则回到步骤2调整 |
| plan 为任务 list 格式（Markdown，无 scenes 字段） | 删除 scenes，按步骤4模板重写为 Markdown 任务 list |
| ≥3条线索推进 | 补充线索 |
| ≥1个即时爽点（读者获得中含爽感项且即时兑现） | 补充爽点 |
| 章末即时危机钩子（只给方向） | 补充钩子方向 |
| **用户未确认** | **不进入Step 1，等待确认** |
