## 通用检查三问

| # | 问 | 检查方式 |
|:-:|:---|:---------|
| 1 | **产出回答了用户问的问题吗？** | 对比用户原始需求和产出结果 |
| 2 | **有没有超出 scope 的多余产出？** | 用户没要求的东西写多了 = 浪费 |
| 3 | **有没有明显的盲点被忽略？** | "看起来对但站不住"的结论 |

## 阶段级读者体验验收

> 每个子 skill 完成后，除了文件级检查，必须做一次"读者视角"验收。
> 不是检查文件格式对不对，是检查"读者读到这个产出会想继续吗？"

| 阶段完成 | 验收问题 | 不通过动作 |
|:---------|:---------|:-----------|
| creative | PRD 的爽点承诺，读者读到会期待什么？这个期待合理吗？ | 退回 creative 调整爽点锚点 |
| world | 世界的核心矛盾，读者读到这里会想看什么？设定是否服务爽点？ | 退回 world 调整设定优先级 |
| plot | 卷一的爽点密度，读者读完卷一会想继续吗？哪几章可能弃书？ | 标注弃书风险章，退回 plot 调整节奏 |
| chapter | 这章的钩子，读者读完会想看下一章吗？情绪弧线是否到位？ | 退回 chapter 调整事件链 |
| prose | 这章的文风，读者读起来流畅吗？有没有想跳的段落？ | 退回 prose 局部重写 |

**弃书风险章标注**（plot 完成后强制执行）：
- 标出最可能让读者弃书的 3 章
- 每章给出弃书原因 + 修复建议
- 写入项目总控.md 的风险日志

## 质量信号

| 信号 | 行动 |
|:-----|:-----|
| 完美覆盖需求 | 直接交付，一句话总结 |
| 小细节偏了 | 标记偏差，再问用户是否接受 |
| agent 偷懒 | 人工检查 |
| 南辕北辙 | 不退——复盘路由错在哪 |

> **L1-L4 详细审视清单由各子 skill 自管。** expert-writer 不重复子 skill 的落盘检查、一致性校验、QA 质检、活人感检查。如果子 skill 产出有质量问题，退回子 skill 重做。

## ⛔ 项目总控回写（强制执行 — 不更新 = 阶段未完成）

> **这是 reflect 步骤的硬性执行项，不是可选建议。**
> 复盘案例（2026-06-22 东方版深渊主宰）：53 轮对话中项目总控.md 从未更新，角色卡已产出但总控显示"目录就绪无产出"，plot已完成但总控显示"下一阶段：character"。
> 根因：reflect 步骤写了"更新总控"但不够指令式，agent 跳过了。

**每次子 skill 执行完成后，必须立即执行以下操作：**

1. **读取 `项目总控.md`**（用 `Get-Content -Encoding UTF8 -Raw`）
2. **更新上半屏（📊 项目现状）**：
   - 首屏仪表盘：将刚完成的阶段标记从 ⏳/⬜ 改为 ✅
   - 实际阶段执行表：追加已完成阶段行
   - 产出物清单：扫描文件系统，更新状态列
   - 当前进度锚点：调用 pop-state-engine `project-status` 命令获取引擎聚合数据（entity-snapshot.yaml 作为 fallback）
     ```bash
     python {engine_scripts}/command_executor.py -p {项目路径} -a project-status
     ```
     将返回 JSON 中的字段映射到总控进度锚点段：
     - `total_chapters` → entity-snapshot 章号
     - `current_arc.title` → 当前幕
     - `protagonist_state` → 主角状态
     - `critical_hooks` → 关键伏笔
     - 引擎查询失败时退回读 `状态/entity-snapshot.yaml`
   - 追加执行顺序日志：`| {序号} | {阶段名} | {时间戳} | {备注} |`
3. **写入更新后的 `项目总控.md`**
4. **确认写入成功**（重新读取验证）
5. **下半屏（🗺️ 所属管线）不动** — 只有版本漂移时才刷新

**⛔ 门禁检查：**
- 项目总控.md 是否已更新？→ 未更新 = 阶段未完成，禁止进入下一阶段
- 产出物清单是否与实际文件系统一致？→ 不一致 = 补齐后再进入下一阶段

## 完成后引导

### 引导纪律

1. 每轮先问「需要修改或调整吗？」，再建议下一步
2. 修改完成后重新读取文件状态再做引导
3. 引导是建议不是催促
4. qa 后只问是否修改

### 引导模板（概要）

| 项目状态 | 引导语 |
|:---------|:-------|
| world 完成 | 「设定已完成。需要调整吗？需要我帮你设计角色吗？」 |
| character 完成 | 「角色卡已完成。需要调整吗？需要我帮你规划剧情吗？」 |
| plot 完成 | 「剧情已规划。需要调整吗？需要我帮你写开头几章吗？」 |
| prose 第 N 章完成 | 「第 N 章已完成。需要修改吗？继续写第 N+1 章吗？还是先质检？」 |
| qa 完成 | 「质检完成。需要我根据反馈修改吗？」 |
| decon 完成 | 「参考书分析已完成。可以基于分析结果开始开书设定，要开始吗？」 |

> 完整引导表 → `references/think/completion-guide.md`
