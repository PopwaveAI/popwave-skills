# step-3-reflect.md — 通用审视 + 项目总控回写 + 引导

> **读什么**：子 skill 产出物
> **产出什么**：审视结论 + 项目总控.md 更新 + 引导语

```
工具选择：skill_view（首选）或 Get-Content -Encoding UTF8 -Raw
❌ 禁止用 Read 工具读取 skill 文件（有行数限制，会截断）
```

## 通用检查三问

> v2/v3 模式共用此三问。

| # | 问 | 检查方式 |
|:-:|:---|:---------|
| 1 | **产出回答了用户问的问题吗？** | 对比用户原始需求和产出结果 |
| 2 | **有没有超出 scope 的多余产出？** | 用户没要求的东西写多了 = 浪费 |
| 3 | **有没有明显的盲点被忽略？** | "看起来对但站不住"的结论 |

## 阶段级读者体验验收

> 每个子 skill 完成后，除了文件级检查，必须做一次"读者视角"验收。
> 不是检查文件格式对不对，是检查"读者读到这个产出会想继续吗？"

| 阶段完成 | 验收问题 | 不通过动作 |
|:---------|:---------|:-----------|
| creative | PRD 的爽点承诺，读者读到会期待什么？这个期待合理吗？ | 退回 creative 调整爽点锚点 |
| world | 世界的核心矛盾，读者读到这里会想看什么？设定是否服务爽点？ | 退回 world 调整设定优先级 |
| plot | 卷一的爽点密度，读者读完卷一会想继续吗？哪几章可能弃书？ | 标注弃书风险章，退回 plot 调整节奏 |
| chapter | 这章的钩子，读者读完会想看下一章吗？情绪弧线是否到位？ | 退回 chapter 调整事件链 |
| prose | 这章的文风，读者读起来流畅吗？有没有想跳的段落？ | 退回 prose 局部重写 |

**弃书风险章标注**（plot 完成后强制执行）：
- 标出最可能让读者弃书的 3 章
- 每章给出弃书原因 + 修复建议
- 写入项目总控.md 的风险日志

### v3 模式阶段验收（★v8.0.0新增）

> v3 模式替换上方 v2 验收表。expert-writer 只做宏观检查，细节由 v3 子 skill 自管。

| 阶段完成 | 验收问题 | 不通过动作 |
|:---------|:---------|:-----------|
| v3-seed | 种子七要素能否驱动前 3 章？（核心压力+主角弧线+世界规则+爽点承诺是否自洽） | 退回 v3-seed 调整种子 |
| v3-emerge（每章） | 五问反思已在 emerge 内完成，expert-writer 只做宏观检查：章号连续性+活记忆一致性+方向提示存在性 | 退回 v3-emerge 重写该章 |
| v3-arc | 校准报告六项检查是否通过？（节奏/伏笔/角色弧线/世界一致性/爽点密度/读者体验） | 退回 v3-arc 重新校准 |

**v3 方向提示检查**（emerge 每章完成后）：
- 确认 emerge 已在活记忆中生成方向提示（下一章走向建议）
- 方向提示缺失 → 退回 emerge 补写

**v3 弧线校准触发检查**（emerge 每章完成后）：
- 章号差值 ≥ 10（距上次校准）→ 建议触发 v3-arc
- 发生重大剧情转折（emerge 标注）→ 建议触发 v3-arc
- 用户明确要求 → 触发 v3-arc
- 未触发 → 继续涌现写作

## 质量信号

| 信号 | 行动 |
|:-----|:-----|
| 完美覆盖需求 | 直接交付，一句话总结 |
| 小细节偏了 | 标记偏差，再问用户是否接受 |
| agent 偷懒 | 人工检查 |
| 南辕北辙 | 不退——复盘路由错在哪 |

> **L1-L4 详细审视清单由各子 skill 自管。** expert-writer 不重复子 skill 的落盘检查、一致性校验、QA 质检、活人感检查。如果子 skill 产出有质量问题，退回子 skill 重做。

## ⛔ 项目总控回写（强制执行 — 不更新 = 阶段未完成）

> **这是 reflect 步骤的硬性执行项，不是可选建议。**
> 复盘案例（2026-06-22 东方版深渊主宰）：53 轮对话中项目总控.md 从未更新，角色卡已产出但总控显示"目录就绪无产出"，plot已完成但总控显示"下一阶段：character"。
> 根因：reflect 步骤写了"更新总控"但不够指令式，agent 跳过了。

**每次子 skill 执行完成后，必须立即执行以下操作：**

1. **读取 `项目总控.md`**（用 `Get-Content -Encoding UTF8 -Raw`）
2. **更新上半屏（📊 项目现状）**：
   - 首屏仪表盘：将刚完成的阶段标记从 ⏳/⬜ 改为 ✅
   - 实际阶段执行表：追加已完成阶段行
   - 产出物清单：扫描文件系统，更新状态列
   - 当前进度锚点：读 `状态/state-log.yaml` 最后一条 baseline + 之后的 event，提取：当前章号（最后一条 event 的 chapter）、当前弧线（baseline 的弧线段）、主角状态（最后一条 event 的变化段）、关键伏笔（baseline 伏笔段中未回收的）
   - 追加执行顺序日志：`| {序号} | {阶段名} | {时间戳} | {备注} |`
3. **写入更新后的 `项目总控.md`**
4. **确认写入成功**（重新读取验证）
5. **下半屏（🗺️ 所属管线）不动** — 只有版本漂移时才刷新

**⛔ 门禁检查：**
- 项目总控.md 是否已更新？→ 未更新 = 阶段未完成，禁止进入下一阶段
- 产出物清单是否与实际文件系统一致？→ 不一致 = 补齐后再进入下一阶段

### state-log 压缩检查

每次 reflect 时检查 state-log.yaml 是否需要压缩：

1. 读 `last_compacted_at` 和最后一条 event 的 chapter
2. 如果差值 ≥ 20（即上次压缩后已累积 20+ 章 event），执行压缩：
   - 读最后一个 baseline + 它之后的所有 event
   - 合并成一条新 baseline（角色变化、伏笔状态、世界状态全部体现在叙事中）
   - 删掉旧 baseline 及其后的所有 event
   - 追加新 baseline
   - 更新 `last_compacted_at` 为新 baseline 的 chapter
3. 如果差值 < 20，跳过

> 压缩是可选的——即使不压缩，state-log 也能正常工作（只是文件稍大）。20 章是建议值，可根据项目节奏调整。

### v3 项目总控回写（★v8.0.0新增）

> v3 模式下，项目总控回写内容与 v2 不同：更新 3 阶段仪表盘而非 8 阶段。

**每次 v3 子 skill 执行完成后，必须执行：**

1. 读取 `项目总控.md`，更新 v3 仪表盘：
   - **3 阶段仪表盘**：种子设计 / 涌现写作 / 弧线校准 的状态标记
   - **当前章号**：活记忆最后一条 event 的 chapter
   - **弧线计数**：已完成弧线校准次数
   - **种子版本**：种子文档当前版本号
2. 追加执行顺序日志
3. 写入并验证

**v3 活记忆压缩检查**（emerge 每章完成后）：
1. 读活记忆最后一条 baseline 的章号 + 最后一条 event 的章号
2. 差值 ≥ 20 → 执行压缩：合并 baseline + event 为新 baseline，更新压缩计数
3. 差值 < 20 → 跳过
4. 压缩计数写入项目总控.md

## 完成后引导

### 引导纪律

1. 每轮先问「需要修改或调整吗？」，再建议下一步
2. 修改完成后重新读取文件状态再做引导
3. 引导是建议不是催促
4. qa 后只问是否修改

### 引导模板（概要）

#### v2 模式引导

| 项目状态 | 引导语 |
|:---------|:-------|
| world 完成 | 「设定已完成。需要调整吗？需要我帮你设计角色吗？」 |
| character 完成 | 「角色卡已完成。需要调整吗？需要我帮你规划剧情吗？」 |
| plot 完成 | 「剧情已规划。需要调整吗？需要我帮你写开头几章吗？」 |
| prose 第 N 章完成 | 「第 N 章已完成。需要修改吗？继续写第 N+1 章吗？还是先质检？」 |
| qa 完成 | 「质检完成。需要我根据反馈修改吗？」 |
| decon 完成 | 「参考书分析已完成。可以基于分析结果开始开书设定，要开始吗？」 |

#### v3 模式引导（★v8.0.0新增）

| 项目状态 | 引导语 |
|:---------|:-------|
| v3-seed 完成 | 「种子设计完成。需要调整吗？可以开始涌现写作了。」 |
| v3-emerge 第 N 章完成 | 「第 N 章已完成。需要修改吗？继续写第 N+1 章吗？」 |
| v3-arc 完成 | 「校准完成。需要调整吗？继续写作吗？」 |

> 完整引导表 → `references/think/completion-guide.md`
