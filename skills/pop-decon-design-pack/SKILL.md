# pop-decon-design-pack · 章节设计包 v3.0.0

> **定位：Phase 1 of deconstruction. 逐章单文件提取，v3 精度模式。**
> - 每章独立 `chXXX-设计包.md`，4层结构（骨架+爽点+角色+感官）
> - 一次 LLM 调用完成单章全量提取，不跨章聚合
> - 同步提取套路归档到 `套路库/`
> - 中文网文「第X章」格式走手动 ETL（见 `references/chinese-novel-etl.md`）
> - **无 v2 批量模式。产出目录只有 `设计包v3/`，不创建 `设计包v2/`。**

## ❌ 质量红线

| # | 红线 |
|:-:|:-----|
| ❌1 | **ETL 未执行就写设计包** — 章节文件不存在 → 退回 |
| ❌2 | **跳步拆分** — 全本 txt 没按章切 → 退回先拆分 |
| ❌3 | **凭空发明事件** — 事件链中出现了原文不存在的事件 |
| ❌4 | **evidence 空** — 事件必须有原文引用（chXX-XX段格式），null 不退回 |
| ❌5 | **设定信息蒸发** — 章内有设定/物品/重要对话释放但未提取到对应区块 |
| ❌6 | **章型误判** — 混合章型放宽检查；纯战斗章中交锋占比< 60% → 退回 |
| ❌7 | **广告未去** — 设计包事件中混入非正文内容 → 退回 |
| ❌8 | **v3格式缺少4层结构** — 必须包含全部4层（骨架+爽点+角色+感官），缺一层 → 退回重写 |
| ❌9 | **每事件缺少精度锚点字段** — 每事件必须有 scene/POV/关键对白/感官锚点。见 `references/precision-anchor-format.md` |
| ❌10 | **产出只留摘要** — 写入后只说「已产出 {N} 份设计包。事件{M}+设定{P}+物品{Q}项」不贴全文 |
| ❌11 | **中文网文硬跑 extract.py** — extract.py 只识别英文 Chapter N，中文「第X章」走手动 ETL |
| ❌12 | **不跳过套路提取** — 每章至少提取1个套路并更新 `套路库/`。纯过渡章标"无"显式标注 |
| ❌13 | **不编造套路名** — 套路名必须来自已知套路列表或原文自然命名 |
| ❌14 | **四段式省略** — 即使某阶段为空也要显式标注 |
| ❌15 | **合并多章到同一文件** — 每章独立文件，文件名必须为 `chXXX-设计包.md` |

## 核心流程

### Step 1: ETL + 按章拆分

详见 `steps/step-1-etl-split.md`

**读什么：** TXT 原文文件
**做什么：** 手动 ETL → 按章正则拆分到 `_temp/chapters/chXXX.txt`
**中文 TXT 替代：** extract.py 不支持中文。参考 `references/chinese-novel-etl.md`

**产出：**
- `_temp/metadata.json` + `_temp/full_text.txt` + `_temp/chapters/ch001.txt ~ chNNN.txt`

**❌ 门禁：** 拆分后的文件数与实际章节数不匹配 → 退回。
**产出命名：** ch{三位数}.txt，如 ch001.txt, ch037.txt。

### Step 2: 逐章 v3 设计包提取（唯一模式）

详见 `references/设计包v3-格式规范.md`

**读什么：** 单章 chXXX.txt
**做什么：** 一次 LLM 调用，产出单章 v3 格式设计包（4层结构）：

1. **L1 骨架层** — 事件链（8-12事件，含字数预算/POV/原文证据/不可替换标记 🔒）
2. **L2 爽点层** — 套路识别+情绪弧线+钩子强度(L1-L5)+爽点机制
3. **L3 角色层** — 人格特质触发+关键对白(语气+潜台词)+关系变化
4. **L4 感官层** — 环境基线6维+动作六段式+氛围渲染要点+DNA衔接映射

**配套产出（同步）：** 每章设计包完成后，对每个识别的套路执行套路归档：
1. 检查 `写作资产/套路库/{套路名}.md` 是否存在
2. 存在 → 追加原文案例
3. 不存在 → 新建套路文件（含四段式+关键数据+首个案例）
4. 套路文件头部维护使用次数计数器

**产出路径：**
- 设计包：`写作资产/设计包v3/chXXX-设计包.md`
- 套路归档：`写作资产/套路库/{套路名}.md`

**产出命名强制规则：** 文件名必须为 `ch{三位数}-设计包.md`（如 `ch001-设计包.md`, `ch020-设计包.md`）。严禁以下变体：
- ❌ `chXXX_v3设计包.md`
- ❌ `chXXX-v3设计包.md`
- ❌ `v3_设计包_chXXX-chYYY.md`
- ❌ `chXXX-设计包.md`（无三位数补零）

**❌ 门禁：**
- 4层结构缺一 → 退回重写
- 每事件至少有精度锚点字段 → 缺则退回
- 每章至少提取1个套路并更新套路库 → 缺执行退回

**格式规范：** 详见 `references/设计包v3-格式规范.md`

### Step 3: 验证

详见 `steps/step-3-verify.md`

**做什么：** 对比 `_temp/chapters/` 和 `写作资产/设计包v3/` 的文件数，确保全覆盖。

**全量验证清单：**

| # | 检查项 | 通过标准 | 失败处理 |
|:-:|:-------|:---------|:---------|
| 1 | **命名一致性** | 全部文件名为 `chXXX-设计包.md` | 统一重命名 |
| 2 | **首行格式** | 全部文件首行匹配 `# 设计包 — chXXX「` | 标记差异文件 |
| 3 | **4层结构完整** | 全部文件包含 4 个小节标题 | 标记缺层文件 |
| 4 | **事件数下限** | 每章事件数 ≥8（后30%章≥5） | 标注低密度警告 |
| 5 | **覆盖率** | 设计包文件数 = 章节文件数 | 缺文件退回补充 |
| 6 | **套路归档覆盖率** | `套路库/` 中有文件对应各章提取的套路 | 缺归档退回补充 |

**验证通过后** 通知 orchestrator（pop-decon）Phase 1 完成。

### Step 3.5: 跨目录路径归并（delegate_task 并行提取后强制）

**执行时机：** 使用 `delegate_task` 并行提取后，且 Step 3 发现文件数 < 章节数时强制执行。
**为什么需要：** 不同子 agent 可能用不同项目根目录名（如 `让仙门再次伟大` vs `让魔门再次伟大`），导致产出分散。

**做什么：**
1. 扫描 `D:/workspace/` 下所有含 `写作资产/设计包v3/` 的目录
2. 将所有设计包文件 cp（非 mv，保留源备份）到目标项目目录
3. 重新执行 Step 3 验证

```bash
# 归并命令示例
find "D:/workspace" -path "*/设计包v3/ch*-设计包.md" -exec cp {} "D:/workspace/{目标项目}/写作资产/设计包v3/" \;
# 套路库同样处理
find "D:/workspace" -path "*/套路库/*.md" -exec cp {} "D:/workspace/{目标项目}/写作资产/套路库/" \;
```

**❌ 门禁：** 归并后设计包文件数仍 < 章节文件数 → 退回 Step 2。
**验证：** 归并后 `ls | sort` 确认文件号从 ch001 到 chNNN 连续无缺失。

## 边界条件（补充）

| 条件 | 处理方式 |
|:-----|:---------|
| **中文 TXT 无换行（单行长文本）** | `read_file` 会显示 total_lines=0。子 agent 必须用 `cat` 或 `head -c` 读取全文，不要依赖 `read_file` 的行号判断 |
| **子 agent 路径不一致** | 子 agent 可能将文件写到 `让仙门再次伟大` 而非 `让魔门再次伟大`。全部提取完成后强制执行 Step 3.5 路径归并 |
| **并行提取后套路库重复** | 不同子 agent 可能新建同名套路文件（大小写/空格差异）。归并后扫描套路库去重 |

## 边界条件

| 条件 | 处理方式 |
|:-----|:---------|
| 章节标题无法正则匹配（如无编号） | 用「chX-首句前20字」作为文件名 |
| 某章只有寥寥几句 | 标注「本章过短」，仍然产出设计包 |
| 广告与正文难以区分 | 保留疑似行，标注「⚠️ 可能为广告」|
| 中文 TXT 编码不确定 | 依次尝试 GBK → GB18030 → UTF-8 |
| **大规模并行提取（50+章）** | ⚠️ 事件衰减风险：子 agent 越往后越简化。每批 delegate_task 指令必须要求事件数≥8/章，完成后立即验证密度。后1/3批次的指令应比前1/3更详细（对抗惰性）|
| **多 batch 命名一致性** | 每个 delegate_task 指令第一行写明「产出文件名格式：ch{三位数}-设计包.md」，不得让子 agent 自行决定命名样式 |
| **并行提取后归一化** | 不同子 agent 可能使用不同命名格式。全部完成后扫描目录，统一为 `chXXX-设计包.md` |

## ❌ WRONG 示例

| 场景 | 错误做法 | 正确做法 |
|:-----|:---------|:---------|
| ETL 未运行 | 跳过 extract.py 直接写设计包 | 先手动 ETL 得到按章拆分的文件 |
| 不拆分逐章 | 直接读 full_text.txt 一次 LLM 跑全书 | 按章拆分后逐章处理 |
| 中文 TXT 硬跑 extract.py | chapter_count=0 仍继续 | 走手动 ETL |
| 广告混入 | 设计包事件包含「下载APP 查看更多」| LLM 调用时指示去掉非正文内容 |
| 多章合并到同一文件 | 子 agent 将多章写到1个文件（如 `v3_设计包_ch111-ch115.md`） | 指令中明确「每章独立文件，文件名 chXXX-设计包.md」|
| 跳过套路归档 | 写了事件链不提取套路 | 每章设计包完必须更新套路库 |
