# pop-decon-design-pack · 章节设计包 v3.3.0

> **定位：Phase 1 of deconstruction. 逐章单文件提取，v3 精度模式。**
> - 每章独立 `chXXX-设计包.md`，4层结构（骨架+爽点+角色+感官）+ 价值点分流清单
> - 一次 LLM 调用完成单章全量提取，不跨章聚合
> - 同步提取价值点，按 `pop-trope-library` 五库分流：立项库 / 设定库 / 文风库 / 剧情库 / 套路库
> - 中文网文「第X章」格式走手动 ETL（见 `references/chinese-novel-etl.md`）
> - **无 v2 批量模式。产出目录只有 `设计包v3/`，不创建 `设计包v2/`。**

## 核心流程

### Step 1: ETL + 按章拆分

详见 `steps/step-1-etl-split.md`

**读什么：** TXT 原文文件
**做什么：** 手动 ETL → 按章正则拆分到 `_temp/chapters/chXXX.txt`

**产出：**
- `_temp/metadata.json` + `_temp/full_text.txt` + `_temp/chapters/ch001.txt ~ chNNN.txt`

**❌ ❌ ❌ 硬门禁（不可跳过）：**
1. **预拆分是强前置条件** — `_temp/chapters/` 目录下必须存在独立章节文件（ch001.txt ~ chNNN.txt），每个文件只包含单章内容。
2. **禁止将大文件传给 delegate_task** — 绝对不允许子 agent 从 32000 行的单一 TXT 中扫描定位章节。这会耗尽子 agent 的上下文预算，导致定位消耗 25K tokens 后产出严重缩水。
3. **验证拆分完整性** — 拆分后的文件数与实际章节数不匹配 → 退回。用 `ls _temp/chapters/ch*.txt | wc -l` 确认数量。
4. **未拆分 = 不得进入 Step 2（delegate_task 模式）** — 如果用户跳过拆分直接要求 delegate_task 提取，必须拒绝并解释：「必须先预拆分为独立章文件。子 agent 从单一大文件中定位章节会消耗大量上下文，导致前 2-3 章后输出质量断崖式下跌。」

**产出命名：** ch{三位数}.txt，如 ch001.txt, ch037.txt。

**中文 TXT 替代：** extract.py 不支持中文。参考 `references/chinese-novel-etl.md`

### Step 2: 逐章 v3 设计包提取（唯一模式）

详见 `references/设计包v3-格式规范.md`

**读什么：** 单章 chXXX.txt（必须已预拆分为独立文件）+ `references/价值点采集-入库分流SOP.md`
**做什么：** 一次 LLM 调用，产出单章 v3 格式设计包（4层结构 + 价值点分流清单）。

**⚠️ delegate_task 模式下的 Token 预算（关键）：**

> 子 agent 的上下文不是无限的。每章原文（80-120行）+ 4层分析输出 = ~8-12K tokens/章。5章一批 = ~50-80K tokens 仅输出部分，加上格式规范/DNA/套路归档指令后，总上下文消耗接近 150-250K。**如果使用 25章/批，上下文在 ch003 就爆了，后面全是干条。**

| 批次大小 | 预估 token 消耗 | 可行性 |
|:---------|:---------------|:-------|
| **3章/批** | ~100-130K | ✅ 充裕，子 agent 有充足空间写完整4层 |
| 5章/批 | ~150-200K | ⚠️ 临界，后期章节可能缩水 |
| 8章/批 | ~200-280K | ❌ 高风险，ch3+ 退化 |
| 10章/批 | ~250-350K | ❌ 不可接受 |
| 25章/批 | ~500K+ | ❌ ❌ ❌ 自杀 |

**⚠️ 格式规范注入陷阱（严重 — 源自海贼法典32章5种格式的复盘）：**

当 orchestrator 在多批次并行提取中，token 压力会导致格式规范被悄悄从 context 中剪裁。典型退化路径：

```
批次1（ch001-003）：完整格式模板嵌入 context → 格式A（✅ 标准 8 列表格）
批次2（ch004-006）：格式被替换为"参见 ch001-003 格式说明" → 格式B（场景#表）
批次3（ch007-012）：同上，格式引用 → 格式C（维度表/信息单）
补跑批次1（ch013-024）：context 压缩 50%，格式截断 → 格式D（简化裁剪表）
补跑批次2（ch025+）：context 仅剩摘要，格式零注入 → 格式E（L1/L2/L3 自由发挥）
```

**硬性约束：** 从第一批到最后一批，`references/v3-format-quick-reference.md` 的全文（约 2K chars）必须 **100% 嵌入每个 delegate_task 的 context body**。禁止写「参见之前批次」「格式同 ch001」「详见 references/xxx.md」——子 agent 是独立上下文，这些引用是空话，子 agent 看不到其他批次 context。格式规范必须物理嵌入，不能靠引用传递。这是格式一致性的唯一保证。

**每章输出的优先级（上下文不足时按此顺序裁剪）：**
1. **L1 骨架层 + 原文证据**（绝不裁剪——这是下游 Phase 2-4 的唯一数据源）
2. **L2 爽点层**（套路识别+钩子——优先保留）
3. **L3 角色层**（关键对白+关系变化——可简化为要点）
4. **L4 感官层**（环境基线+氛围——上下文不足时可精简为一行）

> ⛔ **如果子 agent 上下文不足，裁剪顺序永远是 L4→L3→L2，绝对不允许裁剪 L1 骨架层的事件链和原文证据。** 事件链缺失 = 该章对下游完全无用。

1. **L1 骨架层** — 事件链（8-12事件，**每事件必须表格格式**，含 scene/POV/字数预算/原文证据/不可替换标记 🔒）
2. **L2 爽点层** — 套路识别+情绪弧线+钩子强度(L1-L5)+爽点机制
3. **L3 角色层** — 人格特质触发+关键对白(语气+潜台词)+关系变化
4. **L4 感官层** — 环境基线6维+动作六段式+氛围渲染要点+DNA衔接映射
5. **L5 价值点分流** — 把本章可复用资产标注为立项/设定/角色关系/剧情结构/章法/套路/文风/QA红线，并给出推荐入库路径

**⚠️ 格式锁——事件链只能用表格，禁止段落式事件描述。** 表格列名固定为：
```markdown
| # | 事件 | 类型 | scene | POV | 参与角色 | 字数估计 | 原文证据 |
```

**「原文证据」列不是原文转录！** 只写定位信息：`ch003-¶12` 或 `地窖对峙段·"阿金手指抹过喉咙"`。禁止摘录完整段落——原文证据是索引指针，不是引用块。详见红线 ❌18。
每事件完成后标注不可替换标记 🔒 和节奏分布图。

**配套产出（同步）：** 每章设计包完成后，对每个识别的价值点执行入库分流：

0. **写价值点分流清单** → `写作资产/价值点分流/chXXX-价值点.md`
   - 立项经验 → 推荐 `pop-trope-library/立项库/`
   - 设定/角色关系 → 推荐 `pop-trope-library/设定库/`
   - 剧情结构/章法/角色弧线 → 推荐 `pop-trope-library/剧情库/`
   - 具体桥段套路 → 推荐 `pop-trope-library/原始素材/`，成熟后晋升 `套路库/`
   - 文风/scene/句段手法 → 推荐 `pop-trope-library/文风库/`

1. **写原始套路卡** → `原始素材/{书前缀}_{套路名}.md`（含触发条件+公式+核心张力+情绪收益+运作机制+使用注意+原文案例）
2. **检查是否属于已有抽象模式** → 读 `套路库/` 下的抽象模式卡，看核心机制是否匹配某个变体
   - 匹配 → 在对应抽象模式卡的「跨书案例对比」表新增一列，在「案例积累」区追加一条
   - 不匹配 → 这是新变体或新模式。暂不创建抽象模式卡（等≥2本书验证后再提炼）
3. **更新覆盖计数** → 对应抽象模式卡的「覆盖书数」+1

> ⚠️ 质量陷阱：**不得用批量启发式推断元爽点家族。** 深渊主宰案例证明，文件名关键词匹配对严肃题材完全不可靠。每张卡必须人工逐张判断核心机制，再根据机制分配情绪类别。

**产出路径：**
- 设计包：`写作资产/设计包v3/chXXX-设计包.md`
- 价值点分流：`写作资产/价值点分流/chXXX-价值点.md`
- 套路归档：`写作资产/套路库/{套路名}.md`

**产出命名强制规则：** 文件名必须为 `ch{三位数}-设计包.md`（如 `ch001-设计包.md`, `ch020-设计包.md`）。严禁以下变体：
- ❌ `chXXX_v3设计包.md`
- ❌ `chXXX-v3设计包.md`
- ❌ `v3_设计包_chXXX-chYYY.md`
- ❌ `chXXX-设计包.md`（无三位数补零）

**❌ 门禁：**
- 4层结构缺一 → 退回重写
- 每事件至少有精度锚点字段 → 缺则退回
- 每章必须输出价值点分流清单；每章至少识别1个价值点。纯过渡章可标“无新价值点”，但必须说明原因

**格式规范：** 详见 `references/设计包v3-格式规范.md`

### Step 3: 验证

详见 `steps/step-3-verify.md`

**做什么：** 对比 `_temp/chapters/` 和 `写作资产/设计包v3/` 的文件数，确保全覆盖。

**全量验证清单：**

| # | 检查项 | 通过标准 | 失败处理 |
|:-:|:-------|:---------|:---------|
| 1 | **命名一致性** | 全部文件名为 `chXXX-设计包.md` | 统一重命名 |
| 2 | **首行格式** | 全部文件首行匹配 `# 设计包 — chXXX「` | 标记差异文件，执行 post-hoc 归一化脚本修复 |
| 3 | **4层结构完整** | 全部文件包含 4 个小节标题（骨架层/爽点层/角色层/感官层） | 标记缺层文件 |
| 4 | **事件表字段完整** | 抽检后10%（至少3章），事件使用表格格式，含 scene + POV + 字数估计 + 原文证据 四字段 | 缺字段 → 退回 |
| 5 | **🔒不可替换标记** | 抽检后10%，关键对白/数据标注 🔒 | 缺标记 → 警告 |
| 6 | **事件数下限** | 每章事件数 ≥8（后30%章≥5） | 标注低密度警告 |
| 7 | **覆盖率** | 设计包文件数 = 章节文件数 | 缺文件退回补充 |
| 8 | **价值点分流覆盖率** | `写作资产/价值点分流/` 中有文件对应各章设计包 | 缺分流退回补充 |
| 9 | **套路归档覆盖率** | `套路库/` 中有文件对应各章提取的套路；纯过渡章显式标"无" | 缺归档退回补充 |

**⚠️ 首行格式漂移是持续顽疾：** 实测 69/150 文件的格式在 batch 提取后偏离标准格式，尽管每个 delegate_task context 都明确了格式要求。子 agent 不受格式锁定充分约束——因此首行格式检查必须是全量扫描（不能仅抽检），修复策略必须是 post-hoc 脚本化归一化（`references/post-hoc-format-normalization.md`），不能指望子 agent 自修正。**tag: drifting-first-line-format**

**验证通过后** 通知 orchestrator（pop-decon）Phase 1 完成。

### Step 3.5: 跨目录路径归并（delegate_task 并行提取后强制）

**执行时机：** 使用 `delegate_task` 并行提取后，且 Step 3 发现文件数 < 章节数时强制执行。
**为什么需要：** 不同子 agent 可能用不同项目根目录名（如 `让仙门再次伟大` vs `让魔门再次伟大`），导致产出分散。

**做什么：**
1. 扫描 `D:/workspace/` 下所有含 `写作资产/设计包v3/` 的目录
2. 将所有设计包文件 cp（非 mv，保留源备份）到目标项目目录
3. 重新执行 Step 3 验证

```bash
# 归并命令示例
find "D:/workspace" -path "*/设计包v3/ch*-设计包.md" -exec cp {} "D:/workspace/{目标项目}/写作资产/设计包v3/" \;
# 套路库同样处理
find "D:/workspace" -path "*/套路库/*.md" -exec cp {} "D:/workspace/{目标项目}/写作资产/套路库/" \;
```

**❌ 门禁：** 归并后设计包文件数仍 < 章节文件数 → 退回 Step 2。
**验证：** 归并后 `ls | sort` 确认文件号从 ch001 到 chNNN 连续无缺失。

## 边界条件（补充）

| 条件 | 处理方式 |
|:-----|:---------|
| **中文 TXT 无换行（单行长文本）** | `read_file` 会显示 total_lines=0。子 agent 必须用 `cat` 或 `head -c` 读取全文，不要依赖 `read_file` 的行号判断 |
| **子 agent 路径不一致** | 子 agent 可能将文件写到 `让仙门再次伟大` 而非 `让魔门再次伟大`。全部提取完成后强制执行 Step 3.5 路径归并 |
| **并行提取后套路库重复** | 不同子 agent 可能新建同名套路文件（大小写/空格差异）。归并后扫描套路库去重 |
| **子 agent 合并多章输出** | 子 agent 倾向将5章全部写入1个文件和/或产出合并的「套路归档总表」。必须在 delegate_task prompt 最顶部（格式规范之前）显式写明「每章必须产出 INDIVIDUAL 独立文件，文件名 chXXX-设计包.md，不得合并！每套路一个独立文件，不得创建总表」。详见 `references/batch-scaling.md` |

## 边界条件

| 条件 | 处理方式 |
|:-----|:---------|
| 章节标题无法正则匹配（如无编号） | 用「chX-首句前20字」作为文件名 |
| 某章只有寥寥几句 | 标注「本章过短」，仍然产出设计包 |
| 广告与正文难以区分 | 保留疑似行，标注「⚠️ 可能为广告」|
| 中文 TXT 编码不确定 | 依次尝试 GBK → GB18030 → UTF-8 |
| **大规模并行提取（50+章）** | ⚠️ 事件衰减风险：子 agent 越往后越简化。每批 delegate_task 指令必须要求事件数≥8/章，完成后立即验证密度。后1/3批次的指令应比前1/3更详细（对抗惰性）。⚠️ 格式注入陷阱：token 压力下格式规范是第一个被砍的。必须从第1批到最后一批 100% 嵌入 `references/v3-format-quick-reference.md` 全文。见 `references/cn-novel-format-injection-failure.md` |
| **多 batch 命名一致性** | 每个 delegate_task 指令第一行写明「产出文件名格式：ch{三位数}-设计包.md」，不得让子 agent 自行决定命名样式 |
| **并行提取后归一化** | 不同子 agent 可能使用不同命名格式。全部完成后扫描目录，统一为 `chXXX-设计包.md` |

## ❌ WRONG 示例

| 场景 | 错误做法 | 正确做法 |
|:-----|:---------|:---------|
| ETL 未运行 | 跳过 extract.py 直接写设计包 | 先手动 ETL 得到按章拆分的文件 |
| 不拆分逐章 | 直接读 full_text.txt 一次 LLM 跑全书 | 按章拆分后逐章处理 |
| 中文 TXT 硬跑 extract.py | chapter_count=0 仍继续 | 走手动 ETL |
| 广告混入 | 设计包事件包含「下载APP 查看更多」| LLM 调用时指示去掉非正文内容 |
| 多章合并到同一文件 | 子 agent 将多章写到1个文件（如 `v3_设计包_ch111-ch115.md`） | 指令中明确「每章独立文件，文件名 chXXX-设计包.md」|
| 跳过套路归档 | 写了事件链不提取套路 | 每章设计包完必须更新套路库 |
