# 设计包批量处理：高章数场景的并行策略

> 适用场景：100+ 章大规模拆解。单批 5 章 × 20+ 批时，串行太慢，必须并行。
>
> ⚠️ **上下文预算硬约束（v3.3.0+）**：子 agent 的有效上下文约 100-130K tokens。每章原文 ~3-5K + 4层输出 ~5-8K。**批大小超过 8 章将导致 ch3+ 输出质量断崖下跌。详见 `references/token-budget-delegate.md`。**

## 核心策略

`delegate_task` 每次最多 3 个并发子任务（该用户上限），每个子任务处理 1 个 **3 章**批次。

### 批次数计算

```python
import math
total_chapters = 187
batch_size = 3  # ⛔ 硬上限，不得超过
batches = math.ceil(total_chapters / batch_size)
rounds = math.ceil(batches / 3)  # 3 concurrent per round
# 187 / 3 = 63 batches → 21 rounds
```

### 批次对齐规则

- 前 N 个完整批次：3 章/批（ch001-003, ch004-006, ...）
- 最后一批可能不足 3 章（ch185-187），直接按实际章数处理
- 不跨卷对齐（卷边界由 Phase 2 识别）

## delegate_task 调用模板

每个子任务接收：
1. **工作目录**（固定路径）
2. **3 个章节文件路径**（显式列出）
3. **上批末尾状态**（batch_link 上下文，首批传「（首批）」）
4. **输出目标路径**

### 提示上下文结构

```
工作目录: D:/workspace/{项目}/

读取这些章节文件:
- _temp/chapters/ch{N}.txt
- _temp/chapters/ch{N+1}.txt
...

{书名}第X卷(ch{N}-{N+2})，作者{作者}。

输出到: 写作资产/设计包/设计包-ch{N:03d}-{N+2:03d}.md

设计包格式(3章合一)，每事件有原文evidence。
包含全部10个模块：基础信息、智慧实体、事件链、设定提取、物品一览、地点一览、重要信息/对话、情绪节拍、钩子、批次衔接
```

### 给子任务的约束

- 每个子任务只读自己的 5 个文件，不跨批引用
- 子任务必须用 `write_file` 或 `terminal` 写入输出文件
- 无需返回 summary 以外的信息（写入即完成）

## 按轮次调度

```python
batches = [(1,5), (6,10), ..., (186,187)]  # 38 batches
rounds_of_3 = [batches[i:i+3] for i in range(0, len(batches), 3)]
for round_batches in rounds_of_3:
    # 并发启动 1-3 个 delegate_task
    results = delegate_task(tasks=[{b} for b in round_batches])
    # 检查全部 completed，失败批次退回重跑
```

### 批量完成后的命名归一化（关键！）

不同子 agent 可能采用不同命名约定。全部轮次完成后，必须执行一次目录扫描和统一：

```bash
# 1. 扫描所有异常命名变体
ls 写作资产/设计包v3/ | grep -vE '^ch[0-9]{3}-设计包\.md$'

# 2a. 处理 _v3设计包 变体（ch036_v3设计包.md）
# ❌ 不要用 grep -P（Windows git-bash 不兼容）
# ✅ 用 explicit mv 逐个处理
mv ch036_v3设计包.md ch036-设计包.md

# 2b. 处理 -v3设计包 变体（ch171-v3设计包.md）
mv ch171-v3设计包.md ch171-设计包.md

# 2c. 处理批次级合并文件（v3_设计包_ch111-ch115.md）
# → 必须逐章重新提取，不能手动拆分
# → 先检查该文件内容是否含多章设计包，如是：重跑 delegate_task

# 3. 最终验证
for i in $(seq -w 1 $TOTAL); do
  if [ ! -f "ch${i}-设计包.md" ]; then
    echo "MISSING: ch${i}"
  fi
done
```

**⚠️ 常见错误命名变体：**

| 变体 | 示例 | 根因 | 预防 |
|:-----|:-----|:-----|:-----|
| 下划线分隔 | `ch036_v3设计包.md` | 子任务指令未写明目标文件名 | 指令中写明 `ch036-设计包.md` |
| -v3 后缀 | `ch171-v3设计包.md` | 另一个子任务用了不同命名模式 | 所有 batch 统一模板 |
| 批次级合并 | `v3_设计包_ch111-ch115.md` | 子任务误用了 v2 模式输出 | v3 指令强调「每章独立文件」|
| **多章合并为单一文件** | `ch136-140-设计包.md` | 子 agent 将5章输出到1个文件 | prompt 最顶部写明「每章必须产出 INDIVIDUAL 独立文件」|
| **合并套路归档总表** | `套路归档_ch146-150.md` | 子 agent 将所有套路合并为一个总表文件 | prompt 明确「每套路一个独立文件，路径套路库/{套路名}.md，不得创建总表文件」|
| 缺位补零 | `ch1-设计包.md` | 三位数编号规则未传达 | `ch{三位数}` 在指令中显式写明 |

### v3 精度模式的注意事项

v3（单章独立）的并行策略与 v2（5章合一）不同：

| 维度 | v2 批量模式 | v3 精度模式 |
|:-----|:-----------|:------------|
| 输出粒度 | 5章合一 1 个文件 | 每章独立 5 个文件 |
| 套路归档 | 无 | 每章提取套路 + 更新 `套路库/` |
| 子任务上下文 | ~80-130K tokens/批 | ~150-250K tokens/批（含 style DNA + v3 格式规范） |
| 文件操作 | 1× `write_file` | 5× `write_file` + N× `patch`（套路库追加） |
| 并行冲突 | 无（文件不共享） | 套路库存在并发写冲突风险 |

#### v3 子任务提示模板（已验证模板，10轮150章实测有效）

> ⚠️ **关键陷阱：子 agent 倾向合并多章到单一文件。** 必须将「每章独立文件，不得合并生成总表」放到提示词最顶部。如果在格式规范区才提及，部分子 agent 会忽略并产出合并文件。经验证，在 context 第一段写明「【重要】每章必须产出 INDIVIDUAL 独立文件，文件名严格为 ch{三位数}-设计包.md，不得合并多章到同一文件！每套路必须单独一个文件到套路库/，不得创建套路归档总表！」可有效防止合并。
>
> ⚠️ **第二陷阱：首行格式漂移。** 即使子 agent 正确产出独立文件，首行格式仍可能漂移至 `# 第X章·标题 — 设计包 (v3)` 而非标准 `# 设计包 — chXXX「标题」`。149章实测：44/44 抽检文件首行均不合格。需后处理归一化（见 `references/post-hoc-format-normalization.md`）。

**验证过的提示结构（10轮150章实测，0合并事故）：**

```
【重要】每章必须产出 INDIVIDUAL 独立文件，文件名严格为 ch{三位数}-设计包.md，不得合并多章到同一文件！
每套路必须单独一个文件到套路库/，不得创建套路归档总表！

【上下文预算】
你的有效工作上下文约 100-130K tokens。本批次 3 章，每章原文 ~80-120 行。
输出裁剪优先级（上下文不足时）：L4 感官层 → L3 角色层 → L2 爽点层。⛔ 绝不裁剪 L1 骨架层！

项目根目录: D:/workspace/{项目}/

读取这些章节文件:
- D:/workspace/{项目}/_temp/chapters/ch{N}.txt
- D:/workspace/{项目}/_temp/chapters/ch{N+1}.txt
- D:/workspace/{项目}/_temp/chapters/ch{N+2}.txt

{书名}({作者})ch{N}-ch{N+2}。

无预先DNA档案时，感官层DNA映射部分标注"首章，后续补充"。

产出路径:
- 设计包: D:/workspace/{项目}/写作资产/设计包v3/ch{三位数}-设计包.md
- 套路库: D:/workspace/{项目}/写作资产/套路库/{套路名}.md

每章设计包v3格式(4层结构):

## 首行
```
# 设计包 — chXXX「章节标题」
```

## 1. 事件链(骨架层) - 表格格式
至少8个事件。表格列: | # | 事件 | 类型 | scene | POV | 参与角色 | 字数估计 | 原文证据 |
原文证据列只写定位指针（ch003-¶12 / 首句·关键词），禁止摘录完整段落。
禁止在文件末尾追加「## 原文证据」代码块。
🔒 不可替换标记(关键对白/数据)。

## 2. 爽点设计(爽点层)
- 套路识别(≥1个) + 四段式
- 情绪弧线
- 章末钩子(L1-L5)

## 3. 角色与人设(角色层)
- 登场角色行为锚定
- 关键对白(语气+潜台词)

## 4. 感官与画面(感官层)
- 环境基线6维
- 战斗章动作六段式
- 氛围渲染要点

## 5. 套路归档
每章≥1套路 → 套路库/{套路名}.md。纯过渡章标"无套路适用(纯过渡章)"

红线: 不发明事件/不编造套路名/不跳过归档/每章独立文件/用 write_file 写入
```

**注意区别：**
- 旧模板把格式规范写在 context 中间 → 子 agent 可能忽略首行格式约束
- 新模板在 context **第一段**（连标题之前）就写明独立文件约束
- 旧模板用相对路径 → 新模板用**绝对路径**防止路径解析失败
- 旧模板假设有 DNA 档案 → 新模板说明无DNA时的 fallback

#### v3 并行冲突处理

套路库是共享文件，多个并行子任务同时修改同一文件时会覆盖。对策：

| 策略 | 做法 | 适用场景 |
|:-----|:-----|:---------|
| **后处理合并** | 子任务只写自己批次的套路案例到独立文件 → 主任务收集后再更新套路库 | 套路库已积累较多时 |
| **预分配** | 主任务先读取套路库状态 → 子任务只允许新建文件 + 追加到不冲突的套路 | 套路名可预判时 |
| **按顺序串行** | 套路库更新阶段切成多轮，每轮只有1个子任务允许写套路库 | 套路名重叠度高时 |

##### 套路库计数器同步陷阱

**症状**：两个子任务同时读取 `使用次数: 3`，各自加1并写回 `使用次数: 4`，但应该是 5。

**根因**：并行子任务不能原子化地读→改→写共享文件。

**对策（从最优先开始）：**
1. **主任务后合并** — 子任务只输出自己的案例列表（不修改套路库），全部完成后由主任务统一读取套路库文件并追加，计数器自然准确
2. **基于 patch 的增量追加** — 如果必须并行更新，子任务不要读→写整个文件，而是用 `patch(action='replace', old_string='使用次数: N', new_string='使用次数: N+1')` + 追加案例段。这依赖于 patch 的原子性
3. **接收偏差** — 如果高精度计数器不重要（如参考性计数），允许 ±1-2 的误差，最终人工修正

#### 跨子任务上下文保障

子任务无记忆，需在提示中携带所有必要上下文：

```
1. 文件路径（绝对路径，不要用相对路径）
2. 完整 v3 格式规范（4层结构 + 精度锚点 + 套路归档）
3. 套路库已有文件列表（避免新建同名文件）
4. 质量红线清单（不要凭空发明事件/不要编造套路名）
```

### 路径安全

Windows git-bash 下 `grep -P` 不兼容，mv/重命名操作应使用纯 bash 替代或用 `write_file` 直接写入正确路径。最佳实践：

```bash
# ❌ 不安全 - grep -P 会失败
mv "v3设计包/ch011.md" "写作资产/设计包v3/ch011-设计包.md"

# ✅ 安全 - 直接写入正确路径
# 在子任务中设置 write_file 目标为: 写作资产/设计包v3/ch011-设计包.md
# 不要在事后 mv
```

## 质量注意事项

| 风险 | 对策 |
|:-----|:------|
| 并行度 3 封顶 | 无法加速到 3 以上，接受该上限 |
| 子任务总结不可靠 | 不依赖总结验证输出，直接 stat 输出文件 |
| 批次间上下文丢失 | 首批提供「（首批）」标记；后续批次独立，无跨批依赖 |
| 令牌预算不够单批 | v2: 5章 ~15KB，~80-130K tokens；v3: 5章 + 格式规范 + DNA，~150-250K tokens |
| LLM 产出不一致 | 每批使用同一提示模板；场景卡值、evidence 格式、套路名风格一致 |
| 套路库并发写冲突 | 见上方「v3 并行冲突处理」对策 |
| 子 agent 合并多章到同一文件 | 在 prompt 最顶部（格式规范之前）写明「重要: 每章必须产出 INDIVIDUAL 独立文件，文件名 chXXX-设计包.md，不得合并！」，子 agent 会选择性忽略格式区内的「每章独立文件」约束 |
| Windows git-bash 兼容性 | 不要依赖 grep -P；文件操作直接用 write_file/terminal 写入最终路径 |

## 性能参考（实测数据）

| 量级 | 批数 | 总耗时（3并行） | 实测验证 |
|:-----|:----|:---------------|:---------|
| 20 章（v3精度模式） | 4 | ~5-9 分钟 | — |
| 20 章（v2批量模式） | 4 | ~3-4 分钟 | — |
| 100 章 | 20 | ~15-20 分钟 | — |
| 149 章 | 30 | ~35-45 分钟 | ✅ 实测 10轮×~4.5min/轮 = ~45分 |
| 187 章 | 38 | ~30-40 分钟 | — |
| 500 章 | 100 | ~1.5-2 小时 | — |

注：基于 deepseek-v4-flash，v3单批 ~90-120s（含套路库读写操作）。149章实测中，首轮较慢（~250s）因子 agent 需加载格式规范 + 读5章内容，后续轮次稳定在 ~170-220s/轮。

| # | 检查项 | 实测结果 | 说明 |

**结论**: 文件级约束（命名/独立文件/覆盖率）可靠执行；内容级约束（首行格式/小节命名）在并行抽取中系统性漂移。必须增加格式化后处理步骤。
