# 格式规范注入失败：海贼法典复盘

> **定位：** 记录 2026-06-23 海贼法典 32 章设计包出现 5 种不同格式的根因分析。
> **用途：** 供后续 agent 在 multi-batch 并行提取前阅读，避免重复踩坑。

---

## 现象

32 章设计包出现 5 种不同输出格式，尽管 pop-decon-design-pack 只有一个 v3 标准格式：

| 格式 | 章数 | 关键区别 |
|:-----|:----|:---------|
| A 详尽事件表 | 15 | 8列表格(事件/类型/scene/POV/角色/字数/证据) + 🔒 + 套路 + 情绪弧 |
| B 场景#表 | 6 | 场景#/场景名称/POV/证据 + 逻辑流ASCII图 |
| C 维度表 | 6 | 维度/内容双列 + "二号信息单" |
| D 裁剪表 | 3 | 场景/POV/证据三列 + 裁剪影响表 |
| E L1/L2/L3 | 2 | 全新命名体系，事件链分层 |

---

## 根因链条

### 链 1：格式规范只在第一批物理嵌入

```
sessions_spawn 调用轨迹：

ch001-003: context=2026 chars ✅ 含完整格式模板
ch004-006: context=1423 chars ⚠️ 格式替换为"参见 ch001-003"
ch007-012: context=1184 chars ⚠️ 同上引用
ch004-006-v2 (补跑): context=899 chars ❌ 格式被截断
ch013-015: context=930 chars  ❌ 只有章节摘要
ch025-027 (补跑2): context=523 chars ❌ 完全无格式信息
```

每个子 agent 是独立上下文窗口——"参见之前"是伪引用。

### 链 2：Token 压力下格式是第一个被砍的

orchestrator 跑了 42 轮 sessions_spawn（全部在同一个 run 内），随着上下文累积，每轮省略"已经写过的"内容。退化曲线近似指数：

```
第一批 context size: ~2000 chars（含完整格式）
第二批 context size: ~1400 chars（格式变引用）
第四批 context size: ~900 chars（格式截断）
最后批 context size: ~520 chars（零格式信息）
```

### 链 3：无后置表格结构校验

现有的 Step 3 验证只检查：
- 4 层结构标题是否存在（硬性 ✅）
- **不检查事件表字段数是否一致**（❌ 缺失）
- **不检查表格列名是否匹配标准**（❌ 缺失）

因此 B/C/D/E 格式全部通过了 Step 3 验证，直到人眼查看才被发现。

---

## 教训

| # | 教训 | 措施 |
|:-:|:-----|:-----|
| 1 | 格式规范必须物理嵌入每个子 agent context，禁止引用 | ❌19 + Step 2 注入陷阱节 |
| 2 | "参见之前"在任何子 agent 上下文中都无效 | 显式禁止 |
| 3 | Token 短缺时，先用后处理归一化脚本兜底，而非省略格式 | 格式规范 2K chars < 上下文 1% |
| 4 | Step 3 必须检查事件表字段数一致性（8列 vs 4列 vs 3列） | 格式快照 + table-structure 校验 |
