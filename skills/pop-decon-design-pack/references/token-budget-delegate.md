# delegate_task 上下文预算指南

> 子 agent 的上下文窗口是有限的。本指南说明如何避免上下文溢出导致的产出质量断崖下跌。

---

## 问题根因

```
32000行单体 TXT → 25章/批 → ch003 上下文爆满 → ch004+ 干条
        ↑              ↑              ↑
     原因①          原因②          结果
```

三层失误链：
1. **未预拆分**：子 agent 从单体大文件中定位章节 → ~25K tokens 仅用于扫描定位
2. **批次过大**：25章 × (~10K tokens/章 原文+输出) = ~250K tokens 仅输出
3. **无预算指导**：子 agent 不知道上下文不足时优先保留什么

---

## 批次大小 vs 上下文消耗

> 每章原文 80-120 行（~3-5K tokens），v4 3层+1区分析输出 ~3-5K tokens（比 v3 的 4层 ~5-8K 降约 40%）。加上格式规范 ~15K tokens 固定开销（v4 无 DNA指令，比 v3 的 ~25K 更轻）。

| 批次大小 | 输入(原文) | 输出(3层+1区) | 固定开销 | **总消耗** | 可行性 | 退化风险 |
|:---------|:---------|:-------------|:---------|:----------|:-------|:---------|
| **3章/批** | ~12K | ~12K | ~15K | **~39K** | ✅ 充裕 | 无 |
| 5章/批 | ~20K | ~20K | ~15K | **~55K** | ⚠️ 临界 | ch4-5 可能缩水 |
| 8章/批 | ~32K | ~32K | ~15K | **~79K** | ❌ 高风险 | ch3+ 退化 |
| 10章/批 | ~40K | ~40K | ~15K | **~95K** | ❌ 不可接受 | ch2+ 退化 |
| 25章/批 | ~100K | ~100K | ~15K | **~215K** | ❌❌❌ 自杀 | ch003 已是干条 |

> 注：以上基于 v4 精简格式（3层+1区）实际测试。总消耗 = 输入 + 输出 + 固定开销。子 agent 的 dialogue 还会叠加 tool call 结果和 system prompt，实际可用上下文约 100-130K。因此 **3章/批** 是硬上限。

---

## 输出裁剪优先级

当上下文不足时，子 agent 必须按以下顺序裁剪输出（v4: 3层+1区）：

```
永不裁剪 ─── L1 beat链层（beat表 + 原文证据 + 🔒 + 感官锚点）
              ↓ 证据只用 chXX-¶X 格式，禁止摘录完整段落
              ↓ 这是下游 Phase 2-4 的唯一数据源
              ↓ 缺失 = 该章对下游完全无用
              
优先保留 ── L2 爽点层（情绪弧线 + 钩子）
              ↓ 情绪设计影响下游分析，但缺失不致命

可简化 ──── L3 角色层（关键对白 + 关系变化）
              ↓ 可简化为要点列表

最先裁剪 ── S1 设定/物品提取区（设定 + 物品）
              ↓ 可简化为要点，但不得完全省略
```

> ⛔ **红线：绝不裁剪 L1 beat链层的beat表和原文证据。S1 可简化但不得完全省略。**

---

## delegate_task context 模板（含预算约束）

在委派子 agent 时，context 参数必须包含以下预算段：

```
【上下文预算】
你的有效工作上下文约 100-130K tokens。本批次包含 3 个章节，每章原文 ~80-120 行。
请按以下优先级分配输出（v4: 3层+1区）：
1. L1 beat链层（beat表+原文证据+🔒+感官锚点）→ 必须完整，每beat≥1条原文引用
2. L2 爽点层（情绪弧线+钩子）→ 优先保留
3. L3 角色层（关键对白+关系变化）→ 可简化为要点
4. S1 设定/物品提取区（设定+物品）→ 可简化为要点，但不得完全省略

⚠️ 如果发现上下文接近上限，裁剪顺序永远是 S1→L3→L2。
⛔ 绝对不允许裁剪 L1 beat链层的beat表和原文证据。
```

---

## 预拆分的强制要求

```
❌ 错误：将 32000 行 single_file.txt 传给子 agent
   → 子 agent 必须用 read_file(offset=0, limit=2000) 扫描定位章节边界
   → ~25K tokens 仅用于定位，还没开始分析
        
✅ 正确：先 ETL 拆分为 _temp/chapters/ch001.txt ~ chNNN.txt
   → 子 agent 直接 read_file('_temp/chapters/ch045.txt') 读取单章
   → ~3-5K tokens 用于读取，立即开始分析
```

**预拆分节省的子 agent 上下文：~20K+ tokens，足够完整写 2 章的 L3/L4。**

---

## 三层防护总结

| 层 | 防护 | 不执行后果 | 实施位置 |
|:--|:-----|:---------|:---------|
| ① 预拆分 | `_temp/chapters/ch*.txt` 必须存在 | 子 agent 浪费 25K tokens 定位 | pop-decon Step 2 前置检查 |
| ② 批次≤3章 | ceil(总章数/批次数) ≤ 3 | 上下文爆满，ch1+ 干条 | pop-decon Step 2 前置检查 |
| ③ 预算优先级 | context 含裁剪顺序 | 子 agent 不知保留什么，可能裁剪 L1 | delegate_task context 模板 |
