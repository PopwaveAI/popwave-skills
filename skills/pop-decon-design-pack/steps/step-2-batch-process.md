# Step 2: 5章一批 LLM 提取设计包

> **方向**：每 5 章一次 LLM 调用，去广告 + 事件提取 + 角色扫描，一步完成。
> **核心约束**：这是拆书管线唯一的**质量瓶颈**。设计包烂 = volume/setting 全烂。宁退不回。

---

## I/O 注解

| 维度 | 内容 |
|:-----|:------|
| **读什么** | `_temp/chapters/ch{N}.txt` ~ `ch{M}.txt`（每批 5 章，~10-20k 字） |
| **做什么** | 一次 LLM 调用：去广告 → 逐章结构化提取 → 写入 1 份 5 章合一设计包 |
| **产出** | `写作资产/设计包/设计包-ch{N}-{M}.md`（每批 1 个文件，覆盖 5 章） |
| **门禁** | 质检未通过 → 退回该批重跑 |

---

## 0. 前置认识：不同章型的事件粒度

同一套"5-8 个事件"不能套所有章。必须先判断章型再定粒度：

| 章型 | 特征 | 每章事件数 | 事件粒度 |
|:-----|:-----|:----------|:---------|
| **战斗** | 回合制对抗、有胜负有转折 | 8-12 | 每一轮交锋 = 一个事件 |
| **对话/信息** | 两人/多人交谈、信息交换 | 5-8 | 每轮信息释放 = 一个事件 |
| **探索/发现** | 新地点、新发现、世界观披露 | 6-10 | 每个地点/每个发现 = 一个事件 |
| **过渡/日常** | 赶路、修炼、日常 | 3-5 | 每段日常/每段修炼 = 一个事件 |
| **高潮/转折** | 重大事件、多线汇合 | 10-15 | 可拆到每人每线一个事件 |

**章型判定规则（必须明确，不留模糊空间）：**
- 章内首段提战斗/交战/对决 → 战斗章
- 章内首段提探索/密室/未知 → 探索章
- 章内 60%+ 篇幅为对话 → 对话章
- 章内包含重大死亡/背叛/揭秘 → 转折章
- 上述均不满足 → 过渡章

---

## 1. 批次划分

固定窗口：ch1-5, ch6-10, ch11-15 … chN-4 ~ chN。
最后一批不足 5 章按实际章数处理，不补空白章。
**不跨卷对齐。** 卷边界在 Phase 2 (pop-decon-volume) 中识别。

---

## 2. LLM 调用：完整 Prompt

```
【角色】
你是一个网文拆解分析器。你的任务是从原文中提取事实，不做任何创作。

【输入】
以下是 5 章原文。每章以 `=== chXXX.txt ===` 分隔。
总字数约 10,000-20,000 字。

【处理原则（按优先级排序）】
P0 — 去非正文：跳过每章头部/尾部的广告、导航、作者感言、版权声明、更新通知、QQ群/公众号推广。这些内容不得出现在任何事件中。
P1 — 只提取不创作：事件内容必须来自原文。不得概括、不得推断、不得添加原文中没有的细节。"主角感到愤怒"必须来自"他攥紧了拳头，指甲嵌进掌心"这样的原文，不是你自己判断的。
P2 — 保持时间顺序：事件按原文叙事顺序排列。倒叙/插叙的事件标注[倒叙]。
P3 — 角色名一致性：同一角色在同一批 5 章内使用统一名称。别名用括号标注（如 "赵铁柱（铁子）"）。

【输出格式——严格 JSON】

你必须输出一个 JSON 对象（不要 markdown 包裹，不要代码块标记，直接输出 JSON）。
这个对象包含 5 章的全部信息，不拆为 5 个文件：

```json
{
  "batch_label": "ch001-005",
  "chapter_range": [1, 5],
  "core_arc": {
    "name": "跨章弧线标题",
    "span": [1, 5],
    "total_events": 35,
    "summary": "一句话概括本批5章的核心弧线"
  },
  "chapters": [
    {
      "chapter": 1,
      "title": "完美人生模拟器",
      "chapter_type": "探索",
      "word_count_estimate": 3300,
      "summary": "一句话概括本章核心事件",
      "cleaned": true,
      "ad_removed": ["广告具体内容"],
      "events": [
        {
          "seq": 1,
          "title": "事件标题（10字以内）",
          "type": "交锋/探索/社交/成长/披露/转折/日常",
          "content": "事件内容描述——一个独立的因果单位",
          "participants": ["角色A", "角色B"],
          "evidence": "「证明此事件存在的原文短句引用」",
          "location": "发生地点（如原文无则 null）"
        }
      ],
      "emotional_beat": "起点→推进→高潮→终点",
      "hook": {"type": "悬念/信息/情绪", "content": "钩子描述"}
    }
  ],
  "entities": [
    {
      "name": "智慧实体名（不限人类，如影子诡/大公鸡/灯笼诡）",
      "role": "主角/敌对存在/辅助者",
      "trajectory": {
        "ch1_start": "ch1开始时的状态",
        "ch5_end": "ch5结束时的状态",
        "arc": "本批5章中的变化轨迹描述"
      },
      "key_moments": [
        {"chapter": 2, "event": "关键事件简述", "evidence": "原文证据"}
      ],
      "personality_traits": [
        {"trait": "特质量描述", "evidence": "证据"}
      ]
    }
  ],
  "worldbuilding": [
    {
      "item": "设定项名称",
      "detail": "本批揭示的设定内容",
      "source_chapters": [1, 3, 5],
      "evidence": "最核心的原文证据"
    }
  ],
  "items": [
    {
      "name": "物品名",
      "info": "本批中物品的新信息",
      "importance": "1-3星",
      "evidence": "原文证据"
    }
  ],
  "locations": [
    {
      "name": "地点名",
      "function": "本批中地点的作用",
      "chapters": [1, 2, 4]
    }
  ],
  "key_info": [
    {
      "content": "重要信息内容",
      "speaker": "说话者（如适用）",
      "source_chapter": 1,
      "why_important": "重要性说明"
    }
  ],
  "batch_link": {
    "prev_batch_end": "前一批末尾的状态",
    "this_batch_start": "本批起始位置（通常零间隔衔接）"
  }
}
```

【事件类型速查】

| 类型 | 定义 | 示例 |
|:----|:-----|:-----|
| 交锋 | 有胜负的对抗 | 决斗、战斗、谈判对抗 |
| 探索 | 发现新事物 | 进入新地点、发现宝物、触发隐藏 |
| 社交 | 角色间互动 | 结盟、对话、交易、背叛 |
| 成长 | 能力/境界变化 | 升级、领悟新技能、突破瓶颈 |
| 披露 | 信息揭示 | 揭开真相、历史揭秘、设定说明 |
| 转折 | 剧情转向 | 死亡、重大抉择、命运翻转 |
| 日常 | 过渡内容 | 赶路、修炼、日常起居 |

【事件质量铁律】
- 每个事件是一个**独立的因果单位**，不是一段剧情。"A 拔剑攻向 B，B 闪避反击"是一个事件（交锋）。
  "A 和 B 打了三个回合"是三个事件（交锋×3）。
- combat 章事件粒度：每一刀/每一轮 = 一个事件。不是"整场战斗" = 一个事件。
- dialogue 章事件粒度：每一轮信息交换 = 一个事件。不是"整段对话" = 一个事件。
- state/evidence 必须指向可验证的原文——不说"主角很强"而是"「一拳轰碎了城墙，碎石飞溅」（ch5）」。

【Few-shot 示例】

输入片段（ch1）：
"赵铁柱从床上弹起来。窗外阳光刺眼——又迟到了。他胡乱套上衣服，抓起书包冲出门。楼道里撞到邻居王婶，王婶笑骂了句什么他没听清。"

正确输出（ch1 events 中该片段对应的部分）：
```json
{
  "chapter": 1,
  "events": [
    {
      "seq": 1,
      "title": "起床迟到",
      "type": "日常",
      "content": "赵铁柱起床发现已经迟到，匆忙穿衣出门",
      "participants": ["赵铁柱"],
      "evidence": "「从床上弹起来…阳光刺眼——又迟到了」",
      "location": "赵铁柱卧室"
    },
    {
      "seq": 2,
      "title": "撞见王婶",
      "type": "社交",
      "content": "楼道撞到邻居王婶，被笑骂但未回应",
      "participants": ["赵铁柱", "王婶"],
      "evidence": "「撞到邻居王婶，王婶笑骂了句什么他没听清」",
      "location": "居民楼楼道"
    }
  ]
}
```

错误输出（常见问题）：
```json
{
  "seq": 1,
  "title": "早上日常",
  "type": "日常",
  "content": "赵铁柱开启了忙碌的一天",  // ❌ 概括而非提取
  "evidence": null  // ❌ 无证据
}
```

请开始处理以下 5 章：
=== ch{N}.txt ===
{原文}
=== ch{N+1}.txt ===
{原文}
...
```

## 3. 输出后的质量检验（LLM 返回后必须运行）

每批 LLM 返回后，agent **必须**执行以下检查。**全部通过才能进入下一批。**

### 3.1 完整性检查

- [ ] JSON 对象包含 `batch_label` + `core_arc` + `chapters` + `entities` + `worldbuilding` + `items` + `locations` + `key_info` + `batch_link` 全部 9 个顶级字段
- [ ] `chapters` 数组长度 == 批内章数（5 或最后一组不足值）
- [ ] 每章 `events` 数组长度 >= [章型对应最小值]
- [ ] 每章 `summary` 非空且不是"本章讲述了……"这种套路话
- [ ] 每章 `cleaned` == true
- [ ] `entities` 数组非空——至少包含主角 entity
- [ ] `worldbuilding` 存在（可空数组但不能缺失）

### 3.2 证据检查

- [ ] 每事件 `evidence` 非 null、非空、引用了原文内容
- [ ] 证据是原文中的短句引用（5-30 字），不是 agent 的概括
- [ ] 事件 `content` 和 `evidence` 内容对应
- [ ] `worldbuilding` 每项的 `evidence` 非空
- [ ] `items` 每项的 `evidence` 非空
- [ ] `entities` 每项的 `key_moments` 中每条有 `evidence`
- [ ] `entities` 每项的 `personality_traits` 中每条有 `evidence`

### 3.3 类型检查

- [ ] 没有事件类型为 null 或空字符串
- [ ] 混合型章型（如"战斗/转折"）在质检中放宽类型占比检查
- [ ] 对话章没有把整段对话塞成一个"社交"事件（应拆为多轮）

### 3.4 广告检查

- [ ] 没有任何事件 content 出现广告文案
- [ ] "ad_removed" 字段说明了实际移除了什么内容
- [ ] 每章的 `ad_removed` 中列出的内容不在 events 中出现

### 3.5 智慧实体一致性检查

- [ ] 同一 entity 在 5 章内使用统一名称
- [ ] entity 的 `trajectory` 与 `chapters[].events` 中该实体的行为一致——ch1_start → ch5_end 的演化方向合理
- [ ] 不同 entity 之间的 `key_moments` 无冲突（如苏午和影子诡在同一章的事件互斥则有问题）

---

## 4. 质检未通过 → 退回策略

| 失败类型 | 对应检查 | 处理 |
|:---------|:---------|:-----|
| **结构缺失** | 3.1 — events < 下限 | 退回整批重跑，增加 prompt 中该章型的事件数示例 |
| **证据空** | 3.2 — evidence null | 退回整批重跑，强调"每事件必须引原文" |
| **类型错配** | 3.3 — 战斗章无交锋 | 退回，重新判断章型后再跑 |
| **广告残留** | 3.4 — 出现广告 | 退回重跑，在 prompt 中明确指出残留的广告位置 |
| **重跑 3 次仍不过** | 全部 | 拆成 3+2 两批单独跑，降低上下文长度 |

```

## 质量红线

| # | 红线 |
|:-:|:-----|
| ❌1 | **5章必须全产** — 读了 5 章必须产出含 5 章 events 的设计包，不能少 |
| ❌2 | **evidence 必须非空** — 事件/设定/物品/entity 每条必须有原文引用。null 不退库 |
| ❌3 | **广告未去混入事件** — 设计包中出现非正文内容 → 退回该批 |
| ❌4 | **不跨批引用** — 不准出现"如 ch6 所述"、"详见上批"等跨批关联 |
| ❌5 | **设定信息蒸发** — 章内有设定/物品/重要对话释放但未提取到对应 `worldbuilding`/`items`/`key_info` 区块 |
| ❌6 | **entity 轨迹缺失** — entities 缺少主角 entity 或主角跨章状态轨迹未标注 → 退回 |
| ❌7 | **批次衔接为空** — 非首批时 `batch_link` 缺少衔接信息 → 退回 |
| ❌8 | **连续 3 次退库后不分拆** — 同一批重跑 3 次仍不过，拆 3+2 两批 |
