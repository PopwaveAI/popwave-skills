# Step 5: 价值点分流批量 pass

> **定位**: 全部设计包完成后执行，取代原单章同步分流。
> v4 将价值点分流从 Step 2 的每章同步操作 defer 到此处批量执行，支持跨章去重与合并，提升分流质量。

---

## I/O 契约

| 维度 | 内容 |
|:-----|:------|
| **读什么** | `写作资产/设计包v4/ch*-设计包.md`（全文） |
| **做什么** | 扫描可复用资产 → 按 SOP 分流规则分类 → 产出分流清单 |
| **产出** | `写作资产/价值点分流/chXXX-价值点.md`（按批）或 `写作资产/价值点分流/汇总.md` |
| **门禁** | 每条有证据位置 / 不服务写书侧不入库 / 不新增一级库名 |

---

## 流程

### 1. 批量扫描

每批 ~10 章，只读设计包不读原文。

```
输入: 写作资产/设计包v4/ch001-设计包.md ~ chNNN-设计包.md
批次: 每批 ~10 章（设计包已精简为 3 层 + 1 区，单章 ~3-5K tokens，10 章 ~30-50K tokens）
操作: 逐章扫描全部区块，识别可复用资产
输出: 每批一份 chXXX-价值点.md
```

扫描时关注以下信号：
- 事件链中的结构模式（信息释放节奏、枪链设伏回收）
- 爽点层中的套路组合与情绪弧线设计
- 角色层中的身份张力、关系变化、说话风格
- 设定/物品提取区中的世界规则、力量体系
- 章末钩子的类型与压力梯度

### 2. 跨章去重与合并

全部批次扫描完成后，跨章去重并合并相似价值点。

```python
# 伪代码
all_valuepoints = []  # 从所有批次收集
for batch_file in batch_files:
    all_valuepoints.extend(parse(batch_file))

# 去重: 同一价值点在多章出现时合并，保留最早证据 + 全部章节引用
deduped = {}
for vp in all_valuepoints:
    key = (vp.name, vp.type)
    if key in deduped:
        deduped[key].chapters.extend(vp.chapters)
        deduped[key].evidence_locations.extend(vp.evidence_locations)
    else:
        deduped[key] = vp
```

- 同一价值点在不同章节以不同变体出现 → 合并为一条，变体说明写入「复用说明」列
- 跨章去重后，每条价值点应标注全部涉及的章节

### 3. 入库标记

根据分流判定规则，为每条价值点标注推荐入库位置和是否入库。

- 「是」→ 该价值点可直接入库，后续由人工或脚本写入 pop-trope-library 对应库
- 「否」→ 不服务写书侧任何阶段，不入库
- 「待复盘」→ 价值点存在但复用价值不确定，留待人工判断

---

## 分流判定规则表

| 类型 | 判定问题 | 入库位置 |
|:-----|:---------|:---------|
| 立项 | 它回答"这本书为什么成立"吗？ | `立项库/` |
| 设定 | 它回答"这个世界怎么运转"吗？ | `设定库/{书名}/` |
| 角色关系 | 它回答"角色/关系如何长期产生戏"吗？ | `设定库/角色与关系/` 或 `剧情库/角色弧线/` |
| 剧情结构 | 它回答"剧情线/幕如何排布"吗？ | `剧情库/结构模型/` |
| 章法 | 它回答"章如何调度、信息如何释放"吗？ | `剧情库/章法模型/` |
| 套路 | 它回答"具体桥段怎么爽"吗？ | `原始素材/`，成熟后晋升 `套路库/` |
| 文风 | 它回答"语言怎么写"吗？ | `文风库/{书名}.md` |
| QA红线 | 它揭示"什么会崩坏/失败"吗？ | `立项库/失败案例/` 或对应 `套路库/{模式名}.md` |

> 判定顺序从上到下，命中即止。全部不命中 → 不入库。

## 价值点分流清单格式

```markdown
## 价值点分流清单

| 价值点 | 类型 | 推荐入库 | 证据位置 | 复用说明 | 是否入库 |
|:-------|:-----|:---------|:---------|:---------|:---------|
| {一句话命名} | 立项/设定/角色关系/剧情结构/章法/套路/文风/QA红线 | {库路径} | chXXX-¶YY | {如何服务写书侧} | 是/否/待复盘 |
```

**字段说明：**
- **价值点**：一句话命名，不超过 20 字
- **类型**：8 类枚举之一，见上方分流判定规则表
- **推荐入库**：pop-trope-library 中的具体库路径
- **证据位置**：`chXXX-¶YY` 格式，指向设计包中的原文证据指针
- **复用说明**：该价值点如何服务写书侧（哪个阶段、解决什么问题）
- **是否入库**：是/否/待复盘

## 红线

| # | 红线 |
|:-:|:-----|
| 1 | 不把价值点全塞套路库——人物、节奏、文风、立项经验必须分流到对应库 |
| 2 | 不写无证据价值点——每条必须有 `chXXX-¶YY` 格式的证据位置 |
| 3 | 不服务写书侧不入库——不回答任何写书阶段问题的观察，标记为「否」 |
| 4 | 不新增一级库名——只能使用 `立项库/设定库/剧情库/套路库/文风库` 五个一级库 |

---

## 独立性

- **可与 Step 4 并行**：套路归档和价值点分流读取设计包的不同部分，互不依赖
- **跳过不影响 Phase 2/3**：价值点分流服务的是 pop-trope-library 的长期积累，不阻塞当前拆书管线的后续阶段
- 如果跳过本 Step，设计包中仍包含全部原始数据，后续可随时补跑
