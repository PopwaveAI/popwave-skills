# pop-decon-volume · 叙事结构提取 v3.0.0

> **定位**：Phase 2 of deconstruction. 提取 L2 剧情单元卡、追踪 L3 剧情线、识别 L4 全书事件。
> 物理结构（卷/幕/段）降级为参考坐标系，不再作为产出层级。
> **格式变更**：v3.0.0 起从「卷纲/幕纲」改为「L2 单元卡 / L3 剧情线 / L4 全书事件」三级产出。
> **方向**：逆 `plot` + `pop-writer-chapter`。只聚类原文已呈现的结构，从不「设计」剧情。

---

## 层级定义（L2 单元 vs L3 线 vs L4 全书事件）

**决定层级归属的核心判据不是物理篇幅，而是叙事结构的抽象层级。**

| 层级 | 功能 | 典型规模 | 核心边界信号 | 物理坐标（参考） |
|:-----|:-----|:--------|:------------|:----------------|
| **L2 剧情单元** | 一个有始有终的行动弧，内部含嵌套子线 | 5-30章 | ①穿越入口/出口 ②地点切换 ③力量跃迁 ④叙事基调转变 ⑤事件类型突变 ⑥POV 变化 | 对应原书的「幕」或「跨幕弧线」 |
| **L3 剧情线** | 跨多个 L2 单元持续演进的主题线 | 跨多个单元，贯穿全书或大半 | ①同一承压者/压力源持续施压 ②有阶段变化（触发→推进→复杂化→反转→高潮→收束） ③有叙事债务（前文递送后文回收） | 对应原书的「主线/支线」 |
| **L4 全书事件** | 跨多条 L3 线交汇的全局性事件 | 贯穿全书 | ①多条 L3 线在此交汇 ②引发人物命运/世界格局/主题表达的质变 ③有明确的演化阶段 | 对应原书的「卷级大事件」或「全书级转折」 |

**物理结构降级说明：**
- 原书的「卷/幕/段」不再作为产出层级，仅作为 L2 单元卡的「物理坐标」参考字段
- 一个 L2 单元可能跨越原书的多个幕，也可能小于一个幕
- 一条 L3 剧情线可能贯穿多个卷，也可能只在一个卷内活跃
- 一个 L4 全书事件可能对应原书的一卷，也可能跨多卷演化

**实际操作原则：**
- 当你在设计包中看到"主角升级了" → 这是 L2 单元内事件
- 当主角的**战力定位**从"城市级冒险者"变成"区域级战力" → 可能是 L2 单元边界
- 当同一条主题线（如"复仇"）跨多个 L2 单元持续演进 → 这是 L3 剧情线
- 当多条 L3 线在一个节点交汇并引发全局质变 → 这是 L4 全书事件

**scope 声明（❗必须执行）：**
所有产出文件必须在首行之后添加以下声明：
```markdown
> ⚠️ **基于前X章数据推断**：本章只读到了全书的一部分。
> 所有后续结论（剧情线走向/全书事件估计/套路偏好）均基于此有限窗口，
> 不代表全书终态。如需更新，可在后续章节补充后修正。
```
其中 X = 实际读到的章节数。
「全书事件」命名需标注置信度（候选/确认），前 N 章只做候选识别。

| 4 | 套路聚合 | 扫描 `写作资产/套路库/` | `设计/L3剧情线/` 套路分析段 | 套路库为空时跳过 |

## 产出目录树

```markdown
设计/
├── L2单元卡/
│   ├── L2-001-{单元名}.md         ← Markdown，L2 剧情单元卡（物理坐标+结构分析+嵌套子线+可迁移要点）
│   ├── L2-002-{单元名}.md
│   └── ...
├── L3剧情线/
│   ├── L3-001-{线名}.md           ← Markdown，L3 剧情线卡（核心驱动力+阶段推进+起终点快照+套路分析段）
│   ├── L3-002-{线名}.md
│   └── ...
└── L4全书事件/
    ├── L4-001-{事件名}.md         ← Markdown，L4 全书事件卡（演化阶段+组成线+全局影响）
    └── ...

写作资产/
└── 剧情库/                        ← Step 5 产出（可迁移单元/线入库）
    ├── {标签}/
    │   ├── {书}-L2-{编号}-{名称}.md   ← 可迁移 L2 单元卡
    │   └── {书}-L3-{编号}-{名称}.md   ← 可迁移 L3 剧情线卡
    └── ...
```

**命名规范（所有子agent严格执行）：**
- L2 单元卡文件：`# L2-{编号} · {单元名}（chXXX-chYYY）` — 文件名为 `L2-{编号}-{单元名}.md`
- L3 剧情线文件：`# L3-{编号} · {线名}（chXXX-chYYY）` — 文件名为 `L3-{编号}-{线名}.md`
- L4 全书事件文件：`# L4-{编号} · {事件名}（chXXX-chYYY）` — 文件名为 `L4-{编号}-{事件名}.md`
- 剧情库文件：`{书}-L2-{编号}-{名称}.md` / `{书}-L3-{编号}-{名称}.md`
- 编号从 001 开始，按章节顺序递增

**delegate_task 时的格式锁定：**
委派子agent 时，必须在 context 中明确给出 heading 格式，例如：
```
产出文件首行必须严格为：「# L2-001 · 少林蛰伏与因果了结（ch001-ch011）」
禁止使用「单元1:」「unit 1」「chapter 1」等变体
```

**⚠️ 结构分析格式漂移陷阱**：当用 delegate_task 并行产出多个 L2 单元卡时，子 agent 倾向于将结构分析表（阶段/功能/章节/事件brief/情感设计/节奏）简化为「阶段一览」或「事件流水账」。此漂移源于子 agent 天然压缩冗余信息的倾向。

**对策（二选一，优先A）：**
- **A. 指令锁定** — 在 delegate_task context 中写明：「结构分析必须按 L2 模板逐阶段产出，每阶段一行含全部字段（阶段/功能/章节/事件brief/情感设计/节奏）。不得将多阶段合并为「阶段一览」或「事件流水账」。嵌套子线必须逐条产出，不得省略。」
- **B. 后处理补充** — 子 agent 先产出单元元数据+结构摘要（节省 token），主 agent 收到后用一轮委托逐阶段补充结构分析+嵌套子线。适用于≥20章的宽单元。

## ❌ 质量红线

| # | 红线 |
|:-:|:-----|
| ❌1 | **Phase 1 未完成就聚类** — 设计包不存在 → 退回 |
| ❌2 | **L2 单元边界无跨章证据** — 必须有事件类型突变/场景切换/力量跃迁等跨章证据 → 退回补充 |
| ❌3 | **L3 阶段推进不引用 L2 单元** — L3 的每个阶段必须标注「承载单元」（L2 编号），不得跳过 L2 直接从事件聚类 |
| ❌4 | **前N章只做N章** — **每文件首行之后必须**标注「基于前X章推断，后续将修正」。L4 全书事件前 N 章只做候选（标注置信度），不得标注为确认 |
| ❌5 | **L4 候选标注为确认** — 前 N 章拆解时，L4 全书事件必须标注「候选（置信度：低/中/高）」，全书拆解时才可标注「确认」。误标 → 退回 |
| ❌6 | **套路聚合不编造** — 仅从设计包 L2 爽点层读取数据，不得凭空归纳 |
| ❌7 | **L2 单元卡缺结构分析** — 每个 L2 单元卡必须包含结构分析表（阶段/功能/章节/事件brief/情感设计/节奏）+ 嵌套子线表 → 缺任意一项退回 |
| ❌8 | **L3 剧情线卡缺阶段推进表** — L3 线卡必须包含阶段推进表（阶段/承载单元/事件brief/不可逆变化/读者收益），且每阶段引用 L2 单元 → 缺退回 |
| ❌9 | **产出格式为 Markdown** — 严禁输出 YAML 格式。所有产出为 Markdown |
| ❌10 | **设计包是唯一数据源，不是原始小说** — 子 agent 必须从 `写作资产/设计包v4/chXXX-设计包.md` 读取数据，不得绕过设计包直接从原始 TXT 提取。委派时在 context 中给出**绝对路径**防止相对路径解析失败 |
| ❌10b | **⛔ 设计包格式异常不标记** — 如果设计包事件链是段落式（非表格格式），子 agent 必须先记录「⚠️ 本批设计包为段落式，缺 scene/POV 字段，L2 单元卡中的场景分布/POV分布数据不完整」再作业。不得静默消费格式异常的设计包 |
| ❌11 | **L2/L3/L4 层级混淆** — L2 是单元（有始有终的行动弧），L3 是线（跨单元持续演进），L4 是全书事件（跨线交汇）。误把 L2 单元当 L3 线产出，或误把 L3 线当 L4 事件产出 → 退回。参见[层级定义](#层级定义l2-单元-vs-l3-线-vs-l4-全书事件) |
| ❌12 | **命名不规范** — L2 文件必须 `# L2-{编号} · {名称}（chXXX-chYYY）`，L3 文件必须 `# L3-{编号} · {名称}（chXXX-chYYY）`，L4 文件必须 `# L4-{编号} · {名称}（chXXX-chYYY）`。首行格式不匹配 → 退回。参见[产出目录树](#产出目录树) |
| ❌13 | **delegate_task 未锁定格式** — 委派子agent时必须在其 context 中给出精确的 heading 格式范例。子agent 间命名不一致 → 追责到调度 agent |
| ❌14 | **L2 单元边界从单章设计包直接判断** — 单元边界必须有跨章证据（至少 2 章的设计包对比），不得从单章设计包的事件直接判断单元边界。单章判断 → 退回 |
| ❌15 | **剧情单元写原文叙事** — 事件链每行必须是结构描述（"内鬼提议全体自裁以自证"），不能是原文叙事（"王平站起来说"）。原文叙事=污染下游复用 |
| ❌16 | **剧情单元写方法论** — 不写"这个单元好在哪""读者为什么会爽"。只写事件链+结构解析+可迁移要点 |
| ❌17 | **Wiki 数据覆盖精度数据** — 当 `_temp/wiki-skeleton.md` 存在时，不得用 Wiki 的卷边界直接替代精度数据的聚类结果。Wiki 是参考，设计包是权威。冲突时以精度数据为准，标注冲突点 |

## 产出格式规范

### L2 剧情单元卡 — L2-*.md

详见模板 `templates/L2-剧情单元卡.tpl.md`

必须包含以下区块：

| 区块 | 必须 | 内容 |
|:-----|:----:|:-----|
| 元数据 | ✅ | 来源书/章节范围/所属线/嵌套子线数/单元类型 |
| 物理坐标 | ✅ | 时间/地点/人物/力量 |
| 一句话单元 | ✅ | 用一句话概括本单元的叙事弧 |
| 结构分析 | ✅ | 表格：阶段/功能/章节/事件brief/情感设计/节奏 |
| 主题表达 | ✅ | 本单元在讲什么，主题与结构的对应 |
| 嵌套子线 | ✅ | 表格：子线/类型（闭合或延伸）/在本单元内的推进/与其他子线的咬合 |
| 可迁移要点 | ✅ | 结构模式/情感设计/节奏控制/嵌套策略等可复用模式 |
| 不可迁移部分 | ✅ | 原书专属设定/角色/因果 |
| 单元边界信号 | ✅ | 起点信号/终点信号（事件类型突变/场景切换/角色状态/心态状态/叙事功能） |

| 物理坐标 | ✅ | 时间/地点/人物/力量跃迁/世界冲突质变 |
| 一句话剧情线 | ✅ | 谁，因为哪个压力/诱惑/缺口，被迫做什么，结果把局面推向哪里 |
