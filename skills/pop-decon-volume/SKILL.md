# pop-decon-volume · 叙事结构提取 v3.0.0

> **定位**：Phase 2 of deconstruction. 提取 L2 剧情单元卡、追踪 L3 剧情线、识别 L4 全书事件。
> 物理结构（卷/幕/段）降级为参考坐标系，不再作为产出层级。
> **格式变更**：v3.0.0 起从「卷纲/幕纲」改为「L2 单元卡 / L3 剧情线 / L4 全书事件」三级产出。
> **方向**：逆 `plot` + `pop-writer-chapter`。只聚类原文已呈现的结构，从不「设计」剧情。

---

## 层级定义（L2 单元 vs L3 线 vs L4 全书事件）

**决定层级归属的核心判据不是物理篇幅，而是叙事结构的抽象层级。**

| 层级 | 功能 | 典型规模 | 核心边界信号 | 物理坐标（参考） |
|:-----|:-----|:--------|:------------|:----------------|
| **L2 剧情单元** | 一个有始有终的行动弧，内部含嵌套子线 | 5-30章 | ①穿越入口/出口 ②地点切换 ③力量跃迁 ④叙事基调转变 ⑤事件类型突变 ⑥POV 变化 ⑦**过渡/收束章**：当一章的功能是「战后复盘+权力碾压展示」而非「推进新冲突」，它通常是前单元的终点而非后单元的起点 | 对应原书的「幕」或「跨幕弧线」 |
| **L3 剧情线** | 跨多个 L2 单元持续演进的主题线 | 跨多个单元，贯穿全书或大半 | ①同一承压者/压力源持续施压 ②有阶段变化（触发→推进→复杂化→反转→高潮→收束） ③有叙事债务（前文递送后文回收） | 对应原书的「主线/支线」 |
| **L4 全书事件** | 跨多条 L3 线交汇的全局性事件 | 贯穿全书 | ①多条 L3 线在此交汇 ②引发人物命运/世界格局/主题表达的质变 ③有明确的演化阶段 | 对应原书的「卷级大事件」或「全书级转折」 |

**物理结构降级说明：**
- 原书的「卷/幕/段」不再作为产出层级，仅作为 L2 单元卡的「物理坐标」参考字段
- 一个 L2 单元可能跨越原书的多个幕，也可能小于一个幕
- 一条 L3 剧情线可能贯穿多个卷，也可能只在一个卷内活跃
- 一个 L4 全书事件可能对应原书的一卷，也可能跨多卷演化

**实际操作原则：**\n- 当你在设计包中看到"主角升级了" → 这是 L2 单元内事件\n- 当主角的**战力定位**从"城市级冒险者"变成"区域级战力" → 可能是 L2 单元边界\n- 当同一条主题线（如"复仇"）跨多个 L2 单元持续演进 → 这是 L3 剧情线\n- 当多条 L3 线在一个节点交汇并引发全局质变 → 这是 L4 全书事件\n- **单单元 L3 线判据**：当一条情感/主题线虽只跨越一个 L2 单元，但叙事权重足够（在该单元内完成了完整的起承转合 + 为全书提供不可替代的情感锚点），可升格为 L3。典型场景：卷末单元内的情感羁绊线（如苏夫人线，ch023-ch032 仅在 L2-005 内活跃但构成全书赤土篇的情感锚点）。升格条件：①有独立的情感弧（建置→升温→冲突→抉择→收束）②承担「读者对卷的终极情感记忆」功能③与至少 2 条其他 L3 线有实质性咬合。不满足任一条件则降级为 L2 嵌套子线

**scope 声明（❗必须执行）：**
所有产出文件必须在首行之后添加以下声明：
```markdown
> ⚠️ **基于前X章数据推断**：本章只读到了全书的一部分。
> 所有后续结论（剧情线走向/全书事件估计/套路偏好）均基于此有限窗口，
> 不代表全书终态。如需更新，可在后续章节补充后修正。
```
其中 X = 实际读到的章节数。
「全书事件」命名需标注置信度（候选/确认），前 N 章只做候选识别。

## 步骤速查

| 步骤 | 操作 | 读什么 | 产出 | 门禁 |
|:----:|:-----|:-------|:-----|:-----|
| 1a | **L2 边界预扫描** | 设计包标题/元数据（每章前20行：章名、核心行动、章末钩子） | `步骤1产出/L2-unit-segmentation.md` — 边界信号表 + 提案分段 | 边界必须有跨章证据；提案覆盖全部章节范围 |
| 1b | **L2 剧情单元生产**（并行） | 步骤1a的边界提案 + 各批次设计包全文 | `设计/L2单元卡/L2-{编号}-{名称}.md` | 每卡结构分析表完整；嵌套子线逐条产出；**批产量 ≤3 卡/轮** |
| 2a | **L3 剧情线分析**（全量预读） | 所有已产出的 L2 单元卡 | `设计/L3剧情线/L3-analysis-plan.md` — 跨单元子线追踪 + L3 线提案 | 从 L2 卡嵌套子线表的「延伸」标记追踪，不从记忆推断 |
| 2b | **L3 剧情线生产**（并行） | L3-analysis-plan（阶段递进表）+ 对应 L2 卡 | `设计/L3剧情线/L3-{编号}-{名称}.md` | 阶段推进每阶段必须引用 L2 编号；批产量 ≤3 卡/轮 |
| 3 | L4 全书事件识别 | L3 剧情线（跨线交汇识别） | `设计/L4全书事件/L4-{编号}-{名称}.md` | 前 N 章只做候选（标注置信度）；组成线必须引用 L3 线卡 |
| 4 | 套路聚合 | 扫描 `写作资产/套路库/` | `设计/L3剧情线/` 套路分析段 | 套路库为空时跳过 |

## 产出目录树

```markdown
设计/
├── L2单元卡/
│   ├── L2-001-{单元名}.md         ← Markdown，L2 剧情单元卡（物理坐标+结构分析+嵌套子线+可迁移要点）
│   ├── L2-002-{单元名}.md
│   └── ...
├── L3剧情线/
│   ├── L3-001-{线名}.md           ← Markdown，L3 剧情线卡（核心驱动力+阶段推进+起终点快照+套路分析段）
│   ├── L3-002-{线名}.md
│   └── ...
└── L4全书事件/
    ├── L4-001-{事件名}.md         ← Markdown，L4 全书事件卡（演化阶段+组成线+全局影响）
    └── ...

写作资产/
└── 剧情库/                        ← Step 5 产出（可迁移单元/线入库）
    ├── {标签}/
    │   ├── {书}-L2-{编号}-{名称}.md   ← 可迁移 L2 单元卡
    │   └── {书}-L3-{编号}-{名称}.md   ← 可迁移 L3 剧情线卡
    └── ...
```

**命名规范（所有子agent严格执行）：**
- L2 单元卡文件：`# L2-{编号} · {单元名}（chXXX-chYYY）` — 文件名为 `L2-{编号}-{单元名}.md`
- L3 剧情线文件：`# L3-{编号} · {线名}（chXXX-chYYY）` — 文件名为 `L3-{编号}-{线名}.md`
- L4 全书事件文件：`# L4-{编号} · {事件名}（chXXX-chYYY）` — 文件名为 `L4-{编号}-{事件名}.md`
- 剧情库文件：`{书}-L2-{编号}-{名称}.md` / `{书}-L3-{编号}-{名称}.md`
- 编号从 001 开始，按章节顺序递增

**delegate_task 时的格式锁定：**
委派子agent 时，必须在 context 中明确给出 heading 格式，例如：
```
产出文件首行必须严格为：「# L2-001 · 少林蛰伏与因果了结（ch001-ch011）」
禁止使用「单元1:」「unit 1」「chapter 1」等变体
```

### 🧠 递送上下文增强策略（v3.0.1 新增）
当委托 L2 单元卡生产时，**子 agent 需要比 heading 格式更多的引导**。经验表明，以下上下文显著提升产出质量：

1. **提供预扫描的边界信号**：在 context 中写明「chN→chM 的边界信号是事件类型突变/场景切换/力量跃迁 X」—— 子 agent 有了边界锚点后不会自行位移
2. **提供提案的阶段递进骨架**：例如「本单元的阶段预估：铺垫(chA)→推进①(chB-C)→触发(chD)→高潮(chE)→余波(chF)」—— 骨架降低子 agent 从零构建结构分析的成本
3. **说明本单元叙事类型**（建置型/对抗型/探索型/关系型/复合型）—— 不同单元类型对应不同的阶段模式偏好
4. **指明设计包格式（表格 vs 段落式）** —— 段落式格式会缺少 scene/POV 字段，子 agent 提前知道就不会静默消费（见红线❌10b）

**反模式（已踩过的坑）：**\n- 不给边界信号，子 agent 会自行发明边界 → 返工\n- 不给阶段骨架，子 agent 会产出过简的结构分析 → 违反红线❌7\n- 不给章节 title 列表，子 agent 从设计包索引章节可能出错 → 章节编号漂移\n- **L2 卡用中文文件名，子 agent 的 `read_file` 可能失败** — 当子 agent 按 context 中的路径调用 `read_file('.../设计/L2单元卡/L2-004-巫医崛起·医神之路.md')` 时，中文文件名可能导致解析失败。**对策**：在 context 末尾加一行「读取 L2 卡前先用 `search_files(pattern='L2-*.md', target='files', path='{项目根}/设计/L2单元卡/')` 获取实际文件路径，再用 `read_file` 读取 resolved path。不要直接用 context 中给出的文件名路径调用 read_file。」

### 🧠 L3 两阶段生产模式（v3.0.1 新增）

L3 剧情线生产遵循「先分析，再生产」的两阶段模式，不可跳过分析阶段直接生产：

**阶段 2a — L3 剧情线分析**（全量预读）
- 读取所有已产出的 L2 单元卡（每卡的「嵌套子线」区块）
- 追踪标记为「延伸」的子线如何在 L2 间流动
- 合并同类线、拆分多义线、识别跨单元主题
- 产出 `设计/L3剧情线/L3-analysis-plan.md`：含每条 L3 线的阶段递进度、L2×L3 对应矩阵、生产优先级

**阶段 2b — L3 剧情线生产**（并行）
- 消费阶段 2a 的分析计划（每个子 agent 拿到该 L3 线的阶段递进表 + 对应的 L2 卡列表）
- 子 agent 读取对应 L2 卡中的事件数据填写阶段表
- 各 L3 卡并行产出，但共享同一套分析框架

**为什么不能跳过 2a：**
- 如果每个 L3 子 agent 独立读全部 L2 卡 → 每个子 agent 花大量 token 重复读同一批数据
- 如果每个 L3 子 agent 独立识别 → 不同子 agent 对同一 L2 卡的事件解读不一致 → 跨线矛盾
- 2a 的分析计划提供了统一的解读基线，确保各 L3 线在相同的 L2 事件锚点上推导

### 🧠 卷末单元特殊处理（v3.0.1 新增）

当分解至最后一卷的最后一个 L2 单元时（卷末单元），在标准 L2 卡内容之外追加以下区块：

1. **跨卷未解钩子汇编**：按类别（战争线/势力线/角色线/世界观·神性线）分类整理所有开放伏笔，标注每条钩子的状态（展开/潜伏/演进中）
2. **卷末身份跃迁核心解读**：主角称号/地位的对比（起点→终点），解析递进逻辑
3. **卷末状态总结表**：所有子线闭合 vs 延伸状态清单，标注「跨卷敞口」
4. **下一卷预测锚点**：基于当前窗口的延伸子线方向，标注哪些 L3 线明确延续到下一卷

### 🚨 L3 生产特有的格式漂移陷阱（v3.0.3 新增）

L3 剧情线生产在格式漂移之外，还有两种特有的漂移形态：

**① 模板替换漂移** — 子 agent 不使用 skill 自带的 L3 模板（`templates/L3-剧情线.tpl.md`），改用自己记忆中的其他模板结构。典型表现：
- 使用「元信息」代替「元数据」
- 缺少「scope 声明」行
- 使用非标结构词如「Canvas投放合同」「质量自检」
- _对策_：在 context 末尾加一行 `⚠️ 必须使用 skill 自带的 L3 模板（templates/L3-剧情线.tpl.md），禁止替换模板结构。每项区块必须与模板完全对应。产出后提供首行+元数据区块 sample 供主 agent 验证`

**② 输出路径漂移** — 子 agent 不按 context 中的 output path 写文件，而是默认写到 `C:\Users\<user>\` 或子 agent 的临时工作目录。
- _表现_：主 agent 需要额外调用 `cp` 或 `mv` 将文件从错误路径修正到产出目录
- _对策_：在 context 末尾加一行 `⚠️ 文件必须写入 context 中指定的绝对路径。产出后运行 ls -la <output_path> 验证文件存在且非空`

### 🧠 多卷项目目录组织（v3.0.3 新增）

当项目包含多卷设计包（如 vol1 + vol2），产出的目录组织应遵循以下规则：

```
设计/
├── L2单元卡/          ← 当前精度窗口的**主卷**（最近作业的卷）
├── L3剧情线/          ← 同上
├── L4全书事件/        ← 同上（跨卷交汇）
├── 卷1/               ← 其他卷（非主卷）
│   ├── L2单元卡/
│   ├── L3剧情线/
│   └── L4全书事件/
└── 数值体系反拆/      ← 跨卷共享
```

- 当前正在作业的卷产出放在 `设计/` 根目录，避免路径过深
- 历史卷或非主卷放在 `设计/卷{N}/` 下
- L4 全书事件优先放在主卷的 `L4全书事件/` 下（因为跨卷交汇由主卷视角识别）
- 如果后续切换作业卷，对应目录提升到 `设计/` 根目录，原根目录内容移入 `设计/卷{N}/`

### 🧹 旧格式废弃与归档（v3.0.3 新增）

当旧格式文件（act-*.yaml / volume-*.md / 卷X.md）需要清理时，按以下流程操作：

1. **创建废弃目录**：在 `vol{N}/废弃文件/` 下创建子文件夹（按文件类型分类：`幕纲旧YAML/`、`旧卷架构/`、`旧剧情单元/`、`旧卷{N}幕纲/`、`旧卷{N}架构/`）
2. **移动文件**：将旧格式文件移入对应子文件夹（使用 `mv`，保留历史版本）
3. **清理空目录**：移除原空目录（`rmdir`）
4. **验证**：确认 `设计/` 下只剩新格式产出，废弃文件在 `vol{N}/废弃文件/` 下可回溯

**⚠️ 注意**：即使已为旧文件添加过 `⛔ 废弃文件` 头注，迁移时仍直接 mv 即可（头注随文件保留）。不需要先移除头注再移动。

---

**⚠️ 结构分析格式漂移陷阱**：当用 delegate_task 并行产出多个 L2 单元卡时，子 agent 倾向于将结构分析表（阶段/功能/章节/事件brief/情感设计/节奏）简化为「阶段一览」或「事件流水账」。此漂移源于子 agent 天然压缩冗余信息的倾向。

**对策（二选一，优先A）：**
- **A. 指令锁定** — 在 delegate_task context 中写明：「结构分析必须按 L2 模板逐阶段产出，每阶段一行含全部字段（阶段/功能/章节/事件brief/情感设计/节奏）。不得将多阶段合并为「阶段一览」或「事件流水账」。嵌套子线必须逐条产出，不得省略。」
- **B. 后处理补充** — 子 agent 先产出单元元数据+结构摘要（节省 token），主 agent 收到后用一轮委托逐阶段补充结构分析+嵌套子线。适用于≥20章的宽单元。

## ❌ 质量红线

| # | 红线 |
|:-:|:-----|
| ❌1 | **Phase 1 未完成就聚类** — 设计包不存在 → 退回 |
| ❌2 | **L2 单元边界无跨章证据** — 必须有事件类型突变/场景切换/力量跃迁等跨章证据 → 退回补充 |
| ❌3 | **L3 阶段推进不引用 L2 单元** — L3 的每个阶段必须标注「承载单元」（L2 编号），不得跳过 L2 直接从事件聚类 |
| ❌4 | **前N章只做N章** — **每文件首行之后必须**标注「基于前X章推断，后续将修正」。L4 全书事件前 N 章只做候选（标注置信度），不得标注为确认 |
| ❌5 | **L4 候选标注为确认** — 前 N 章拆解时，L4 全书事件必须标注「候选（置信度：低/中/高）」，全书拆解时才可标注「确认」。误标 → 退回 |
| ❌6 | **套路聚合不编造** — 仅从设计包 L2 爽点层读取数据，不得凭空归纳 |
| ❌7 | **L2 单元卡缺结构分析** — 每个 L2 单元卡必须包含结构分析表（阶段/功能/章节/事件brief/情感设计/节奏）+ 嵌套子线表 → 缺任意一项退回 |
| ❌8 | **L3 剧情线卡缺阶段推进表** — L3 线卡必须包含阶段推进表（阶段/承载单元/事件brief/不可逆变化/读者收益），且每阶段引用 L2 单元 → 缺退回 |
| ❌9 | **产出格式为 Markdown** — 严禁输出 YAML 格式。所有产出为 Markdown |
| ❌10 | **设计包是唯一数据源，不是原始小说** — 子 agent 必须从 `写作资产/设计包v4/chXXX-设计包.md` 读取数据，不得绕过设计包直接从原始 TXT 提取。委派时在 context 中给出**绝对路径**防止相对路径解析失败 |
| ❌10b | **⛔ 设计包格式异常不标记** — 如果设计包事件链不是标准表格格式，子 agent 必须先识别格式变体并记录格式异常再作业。已知格式变体（均缺标准 scene/POV 表头字段）：①**段落式 E1/E2 标记**（事件以段落 + 原文证据形式呈现，如 `ch006-设计包.md`）②**叙事层格式**（以「叙事层（Narrative Layer）」为区块头，事件以表格但无标准 scene/POV，如 `ch022-设计包.md`）③**事件N 标记**（以「事件N：」段落形式呈现，如 `ch021-设计包.md`）。子 agent 必须记录「⚠️ 本批设计包为 {变体名} 格式，缺标准 scene/POV 字段」再作业。不得静默消费格式异常的设计包 |
| ❌11 | **L2/L3/L4 层级混淆** — L2 是单元（有始有终的行动弧），L3 是线（跨单元持续演进），L4 是全书事件（跨线交汇）。误把 L2 单元当 L3 线产出，或误把 L3 线当 L4 事件产出 → 退回。参见[层级定义](#层级定义l2-单元-vs-l3-线-vs-l4-全书事件) |
| ❌12 | **命名不规范** — L2 文件必须 `# L2-{编号} · {名称}（chXXX-chYYY）`，L3 文件必须 `# L3-{编号} · {名称}（chXXX-chYYY）`，L4 文件必须 `# L4-{编号} · {名称}（chXXX-chYYY）`。首行格式不匹配 → 退回。参见[产出目录树](#产出目录树) |
| ❌13 | **delegate_task 未锁定格式** — 委派子agent时必须在其 context 中给出精确的 heading 格式范例。子agent 间命名不一致 → 追责到调度 agent |
| ❌14 | **L2 单元边界从单章设计包直接判断** — 单元边界必须有跨章证据（至少 2 章的设计包对比），不得从单章设计包的事件直接判断单元边界。单章判断 → 退回 |
| ❌15 | **剧情单元写原文叙事** — 事件链每行必须是结构描述（"内鬼提议全体自裁以自证"），不能是原文叙事（"王平站起来说"）。原文叙事=污染下游复用 |
| ❌16 | **剧情单元写方法论** — 不写"这个单元好在哪""读者为什么会爽"。只写事件链+结构解析+可迁移要点 |
| ❌17 | **Wiki 数据覆盖精度数据** — 当 `_temp/wiki-skeleton.md` 存在时，不得用 Wiki 的卷边界直接替代精度数据的聚类结果。Wiki 是参考，设计包是权威。冲突时以精度数据为准，标注冲突点 |

## 产出格式规范

### L2 剧情单元卡 — L2-*.md

详见模板 `templates/L2-剧情单元卡.tpl.md`

必须包含以下区块：

| 区块 | 必须 | 内容 |
|:-----|:----:|:-----|
| 元数据 | ✅ | 来源书/章节范围/所属线/嵌套子线数/单元类型 |
| 物理坐标 | ✅ | 时间/地点/人物/力量 |
| 一句话单元 | ✅ | 用一句话概括本单元的叙事弧 |
| 结构分析 | ✅ | 表格：阶段/功能/章节/事件brief/情感设计/节奏 |
| 主题表达 | ✅ | 本单元在讲什么，主题与结构的对应 |
| 嵌套子线 | ✅ | 表格：子线/类型（闭合或延伸）/在本单元内的推进/与其他子线的咬合 |
| 可迁移要点 | ✅ | 结构模式/情感设计/节奏控制/嵌套策略等可复用模式 |
| 不可迁移部分 | ✅ | 原书专属设定/角色/因果 |
| 单元边界信号 | ✅ | 起点信号/终点信号（事件类型突变/场景切换/角色状态/心态状态/叙事功能） |

### L3 剧情线卡 — L3-*.md

详见模板 `templates/L3-剧情线.tpl.md`

| 区块 | 必须 | 内容 |
|:-----|:----:|:-----|
| 元数据 | ✅ | 来源书/线型/证据范围/组成单元/parent_saga |
| 物理坐标 | ✅ | 时间/地点/人物/力量跃迁/世界冲突质变 |
| 一句话剧情线 | ✅ | 谁，因为哪个压力/诱惑/缺口，被迫做什么，结果把局面推向哪里 |
| 核心驱动力 | ✅ | 承压者/压力源/行动方向/失败代价/终局变化 |
| 阶段推进 | ✅ | 表格：阶段/承载单元（L2编号）/事件brief/不可逆变化/读者收益 |
| 起点/终点快照 | ✅ | 表格：维度/线起点/线终点/变化幅度 |
| 叙事债务与回收 | ✅ | 债务/前置递送/为什么必须回收/回收方式 |
| 与其他线的咬合 | ✅ | 关联线/咬合方式/本线给对方什么/对方给本线什么 |
| 可迁移/不可迁移 | ✅ | 可迁移部分+不可迁移部分（原书专属卷/幕节奏/设定前提/角色关系） |
| 套路分析段 | ⚠️（套路库非空时强制） | 本线使用频率Top套路（基于当前窗口） |

详见模板 `templates/L4-全书事件.tpl.md`

| 区块 | 必须 | 内容 |
|:-----|:----:|:-----|

| 一句话事件 | ✅ | 用一句话概括本全书事件 |

| 组成线 | ✅ | 表格：线（L3编号）/阶段/功能/与其他线的咬合 |
| 全局影响 | ✅ | 人物命运/世界格局/主题表达 |
| 可迁移要点/不可迁移部分 | ✅ | 可复用模式+原书专属内容 |

## 落盘检查点

| 文件 | 检查项 |
|:----|:-------|
| `设计/L2单元卡/L2-*.md` | 首行格式正确（# L2-{编号} · {名称}（chXXX-chYYY））；有scope声明；结构分析表完整（阶段/功能/章节/事件brief/情感设计/节奏）；嵌套子线逐条产出；单元边界信号有跨章证据 |

| `设计/L4全书事件/L4-*.md` | 首行格式正确（# L4-{编号} · {名称}（chXXX-chYYY））；有scope声明；置信度标注（候选/确认）；组成线引用 L3 线卡编号；演化阶段完整 |
| `写作资产/剧情库/{标签}/{书}-L2-*.md` | 区分原书切片与可迁移单元；可迁移要点完整；不可迁移部分标注原书专属内容 |
| `写作资产/剧情库/{标签}/{书}-L3-*.md` | 核心驱动力完整；阶段推进引用 L2 单元；叙事债务有回收方式 |
