# Step 1：L2 剧情单元识别

> **方向**：从设计包事件链（L1）中识别 L2 剧情单元边界，产出每个单元的结构化卡片。
> **级别覆盖**：前N章 / 全书

---

## 数据来源

- `写作资产/设计包v4/chXXX-设计包.md` → 事件链（L1）+ 爽点层（L2：套路类型+情绪弧线+爽点机制）+ 章末钩子
- `_temp/chapter-index.json` → `chapters[].tags` + `chapters[].chapter` + `chapters[].charCount` + `chapters[].firstSentence`
- `_temp/plot-data.json` → `majorEvents[]` / `actBoundaries[]`
- **（可选）** `_temp/wiki-skeleton.md` → 全局骨架，辅助识别跨章边界

## L2 单元边界判断信号

| 信号 | 判断方式 | 示例 |
|:-----|:---------|:-----|
| **穿越入口/出口** | 章末钩子标注"穿越"类，下一章进入/离开副本世界 | ch001 穿越·附身 → 单元起点；ch011 遁入险地 → 单元终点 |
| **地点切换** | 连续 N 章后首句出现新地名（不在之前 10 章中） | ch011 离开少林寺入霍山 → 单元边界 |
| **力量跃迁** | 等级/阶位从一阶进入二阶，或战力定位质变 | ch005 炼精圆满→真气自生→化气阶 → 单元内突破节点 |
| **叙事基调转变** | 情绪密度从压抑→爆发→反思的完整弧线收束 | ch010 爆发（复仇）→ ch011 反思（通缉）→ 基调转变 |
| **事件类型突变** | 连续 worldbuilding 标签章后出现 battle 密集，或反向 | 蛰伏修炼章 → 复仇战斗章 → 类型突变 |
| **POV 变化** | 叙事视角切换（主角→配角→主角） | POV 从方明切到封羽（追捕者视角）→ 可能是单元边界 |
| **章末钩子回收** | 章末钩子标注的"预期回收章"集中回收 | ch001 钩子在 ch006 回收 → ch006 可能是单元内节点 |

## 初始分类：利用设计包 L2 爽点层的套路类型与情绪弧线

设计包v4 的 L2 爽点层标注了每章的套路类型和情绪弧线。利用这两个字段做 L2 单元的初始分类：

| 套路类型/情绪特征 | 对应 L2 单元类型 | 说明 |
|:-----------------|:----------------|:-----|
| 成长/突破类套路 | 事件弧·成长型 | 以力量跃迁为核心的单元 |
| 冲突/对抗类套路 | 事件弧·对抗型 | 以正面对抗为核心的单元 |
| 探索/发现类套路 | 事件弧·探索型 | 以信息获取/地图扩展为核心的单元 |
| 关系/情感类套路 | 事件弧·关系型 | 以人物关系变化为核心的单元 |
| 设定/世界观类套路 | 事件弧·建置型 | 以世界观披露为核心的单元 |
| 混合型 | 事件弧·复合型 | 多种套路类型交替驱动的单元 |

> **初始分类只是起点**：最终单元类型由结构分析决定，不由单章套路类型决定。

## 单元识别流程

### 1. 扫描事件链，标记边界候选

逐章扫描设计包事件链，标记所有边界信号出现的位置：
```
ch001: 穿越·附身 ← 单元起点候选
ch005: 力量跃迁（炼精→化气） ← 单元内节点
ch010: 事件类型突变（修炼→复仇）+ 力量跃迁（化气→后天） ← 单元内高潮
ch011: 地点切换（少林寺→霍山）+ 叙事基调转变（爆发→反思） ← 单元终点候选
```

### 2. 跨章验证边界

对每个边界候选，检查至少 2 章的设计包数据：
- 起点信号：前一章的章末钩子是否指向新单元的开始？
- 终点信号：本章的事件是否完成了某个叙事弧的收束？
- 跨章证据：前后 2-3 章的事件类型/场景/POV 是否有连续性变化？

### 3. 划定单元范围

将验证后的边界候选连成单元范围：
```
L2-001: ch001-ch011（少林蛰伏与因果了结）
  起点证据：ch001 穿越·附身（事件类型突变）
  终点证据：ch011 地点切换+叙事基调转变（跨章证据：ch010 爆发→ch011 反思）
```

### 4. 识别嵌套子线

在每个 L2 单元内部，识别同时运行的子线：
- 标注「闭合」= 在本单元内完成起承转合
- 标注「延伸」= 超出本单元继续追踪
- 每条子线标注在本单元内的功能（为高潮提供什么/催生什么/验证什么）

### 5. 填写单元卡

按 `templates/L2-剧情单元卡.tpl.md` 填写每个 L2 单元卡。

> **参考示例**：`prd/03-设计规范/03-L2剧情单元卡-示例-少林蛰伏与因果了结.md`

## 产出

| 文件 | 内容 |
|:-----|:-----|
| `设计/L2单元卡/L2-001-{单元名}.md` | 按 `templates/L2-剧情单元卡.tpl.md` 填写 |
| `设计/L2单元卡/L2-002-{单元名}.md` | ...（覆盖全部已读章节） |

每个 L2 单元卡必须包含：
- **物理坐标**：时间/地点/人物/力量（参考坐标系，非产出层级）
- **结构分析**：阶段/功能/章节/事件brief/情感设计/节奏
- **嵌套子线**：子线/类型（闭合或延伸）/在本单元内的推进/与其他子线的咬合
- **可迁移要点**：结构模式/情感设计/节奏控制/嵌套策略等可复用模式
- **单元边界信号**：起点信号/终点信号

---

## 红线

| # | 红线 |
|:-:|:-----|
| 1 | **单元边界必须有跨章证据** — 至少 2 章的设计包对比，不得从单章设计包直接判断单元边界 |
| 2 | **不得从单章设计包直接判断单元边界** — 单章事件只能标记为"边界候选"，需跨章验证后确认 |
| 3 | **事件链写结构描述** — 事件 brief 必须是结构描述（"内鬼提议全体自裁以自证"），不能是原文叙事（"王平站起来说"） |
| 4 | **嵌套子线不得省略** — 每个 L2 单元至少识别 1 条子线，标注闭合/延伸类型 |
| 5 | **Wiki 数据不覆盖精度数据** — Wiki 骨架是参考，设计包是权威。冲突时以设计包为准 |
