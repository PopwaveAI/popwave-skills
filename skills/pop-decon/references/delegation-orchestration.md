# 大规模拆书 · 委派编排模式

> **用途**：当设计包数量≥50且需要跑 Phase 2（L2单元卡/L3剧情线/L4全书事件）或 Phase 4（创意溯源）时，使用委派编排模式并行处理。
> **设计原则**：分块扫描 → 并行产出 → 集中合成。避免单线程逐章处理（不可行）或一次性全量塞入一个子 agent（Token 爆炸）。

---

## 一、分块策略

| 量级 | 分块数 | 每块大小 | 推荐方案 |
|:-----|:------:|:---------|:---------|
| 50-100 章 | 2-3 | ~30-50 | 按时间序三等分 |
| 100-200 章 | 3 | ~50-70 | 按时间序三等分 |
| 200+ 章 | 4-5 | ~40-50 | 按卷/幕自然边界分块 |

## 二、Phase 2 编排流程

### 2.1 卷边界扫描（3×并行委派）

每个子 agent 的任务：
1. 读取本块设计包（每5章至少读1章，关键章节全读）
2. 记录6维卷边界信号：地理位置变化、时间跳跃、力量层次变化、情绪基调转变、叙事弧线完成/开始、角色池变化
3. 输出初步卷划分建议（含 chXX 证据链）

> ⚠️ **注意区分卷边界和幕边界**：卷边界需要力量体系代际跃迁/冲突类型质变/地理维度切换信号，幕边界仅需换地图/换配角/情绪转变。不确定时优先识别为幕边界，后续可合并。

```python
# 伪代码模式
tasks = [
  {"goal": "卷边界扫描 ch001-ch065", "context": "...", "toolsets": ["terminal", "file"]},
  {"goal": "卷边界扫描 ch066-ch130", "context": "...", "toolsets": ["terminal", "file"]},
  {"goal": "卷边界扫描 ch131-ch187", "context": "...", "toolsets": ["terminal", "file"]},
]
results = delegate_task(tasks=tasks, description="并行扫描设计包识别卷边界")
```

### 2.2 合成 + 卷架构

从3份扫描报告中合成统一的卷结构。如果只读到全书的一部分（如前N章），产出命名为 `设计/卷1-架构.md`，每文件首行之后添加 scope 声明。**禁止使用"全书架构"命名**。

### 2.3 卷+幕文件批量产出（3×并行委派）

按卷分组委派。每批子 agent 产出该批次所有幕的以下文件（注意：15-30章规模的叙事弧是「幕」不是「卷」）：

| 文件 | 路径格式 | 说明 |
|:-----|:---------|:-----|
| 卷设计 | `设计/卷/卷X.md` | 含剧情线≥5阶段/幕级摘要/分支线跟踪表/伏笔回收/套路分析 |
| 幕文档 | `设计/幕/卷1/幕X.md` | 含卷级定义改编的幕级元数据+章级切片+分支线跟踪 |
| 段文档 | `设计/幕/卷1/幕X/段Y.md` | 含段级元数据+章级切片+节奏自检（可选） |
| 伏笔表 | `设计/幕/卷1/chekhov-tracker.md` | 合并所有幕的伏笔为一张总表 |

**命名规范**（子agent context 中必须给出精确格式）：
```
# 卷X · 名称（chXXX-chYYY）
# 幕X · 名称（chXXX-chYYY）
# 段X · 名称（chXXX-chYYY）
```
禁止使用 `volume-XX:` / `act-YY:` / `vol-XX/act-YY.md` / `L2-{编号}:` 等变体。

典型分块（以187章为例，均为「幕」级，归入同一个卷1）：
- **批1**：幕1-3（ch001-ch065，3幕×2-3文件=10文件） — cp. 原volume-01~03
- **批2**：幕4-7（ch066-ch130，4幕×2文件=12文件） — cp. 原volume-04~07
- **批3**：幕8-10+间幕（ch131-ch187，4幕×2-3文件=12文件） — cp. 原volume-08~10+interlude

### 2.4 产出验证

每个委派批次完成后立即验证：
```bash
设计/卷/卷1.md                          → 存在
设计/幕/卷1/幕{所有编号}.md              → 存在
设计/幕/卷1/幕{编号}/段{编号}.md          → 存在且≥1（如有段级拆分）
设计/幕/卷1/chekhov-tracker.md           → 存在
```

## 三、Phase 4 编排流程

### 3.1 溯源扫描（3×并行委派）

按同样分块，每个子 agent 扫描设计包中的跨域参考来源：

| 维度 | 扫描内容 |
|:-----|:---------|
| DND 3.5e | 职业/法术/怪物/专长/阵营/神祇 |
| 被遗忘的国度 | 位面/神系/幽暗地域 |
| 博德之门 | 神子体系/巴尔之子 |
| 冰雪奇缘 | 阿伦黛尔/安娜/长公主/冰堡放逐 |
| 指环王 | 索伦/半身人/精灵文化 |
| 战锤40k | 原体/混沌/恶魔 |
| 克苏鲁 | 旧日支配者/深渊/不可名状 |
| 其他 | 暗月马戏团/网文传统/系统流 |

每份产出写入 `写作资产/创意溯源/chXXX-chYYY-溯源.md`。

**格式锁定**：在子agent context中必须给出精确的条目格式，禁止不同子agent使用表格/列表/编号等不同格式。

### 3.2 合成跨域参考索引

从多份扫描报告中提取：
- 参考密度总览表
- 按参考来源分节的详细条目（去重+合并证据章号范围）
- 作者原创贡献分析
- 创意转化版图

写入 `写作资产/创意溯源/跨域参考索引.md`。
- scope 声明放在文件第二行
- 条目全部使用统一格式（与子agent格式一致）

## 四、常见陷阱

### ❌ 子 agent 使用错误数据源
**现象**：子 agent 无法找到设计包文件，转而从原始 TXT 重新提取。
**原因**：设计包路径是 `写作资产/设计包v3/chXXX-设计包.md`，但子 agent 的 workdir 或相对路径解析失败时会 fallback 到原始文本。
**解决方案**：
1. 在 context 中给出**绝对路径**：`D:\\workspace\\项目名\\写作资产\\设计包v3\\ch001-设计包.md`
2. 要求子 agent 先验证文件存在：`ls 'D:/workspace/项目名/写作资产/设计包v3/' | head`
3. 如果不确定，让子 agent 使用 `search_files(target='files', pattern='*设计包*', path='绝对路径')` 先确认

### ❌ 产出被写入错误路径
**现象**：子 agent 声称"文件写入成功"，但 `design/卷/volume-03.md` 不存在。
**原因**：子 agent 使用相对路径（如 `design/卷/volume-03.md`），解析到的是子 agent 的临时工作目录。
**解决方案**：
1. 在 goal 中明确指定目标路径格式：`使用 write_file 产出到 D:\\workspace\\项目名\\设计\\卷\\volume-03.md`
2. 写入后验证：在 context 中要求子 agent 用 `ls` 或 `read_file` 验证写入
3. orchestrator 在委派返回后，必须亲自用 `search_files` 或 `read_file` 验证关键输出

### ❌ 子 agent 产出命名格式不一致

**现象**：不同子agent产出 `卷1 · 琥珀城风云`、`volume-02 — 远行之路`、`# 卷4·白马城的烽火` 三种不同格式。
**原因**：每个子agent各自解释模板，没有统一的首行格式指令。
**解决方案**：
1. 在 context 中给出精确的 heading 范例：`文件首行必须严格为：「# 幕1 · 名称（ch001-ch020）」，禁止 volume-/act- 变体`
2. 产出后验证：`head -1 各个目标文件` 检查一致性
3. 包含 scope 声明的行也要统一：`文件第二行为："> ⚠️ **基于前X章数据推断**：本文件只读到了卷1的章节。"`

### ❌ 设计包事件链格式不一致

**现象**：有的子agent产出了表格格式（含scene/POV/原文证据），有的产出了段落式（仅文字描述）。
**原因**：子agent没有统一的格式样板，各自按理解选择事件描述方式。段落式事件链无法被下游消费（pop-decon-volume 需要按 scene/POV 字段聚合数据）。
**解决方案**：
1. 在 context 中嵌入完整格式样板（直接复制 `pop-decon-design-pack/references/v3-format-quick-reference.md` 的内容）
2. 在 context 第一行写明：`「产出设计包时事件链必须使用表格格式。表格列： # | 事件 | 类型 | scene | POV | 参与角色 | 字数估计 | 原文证据」`
3. 要求子agent产出后在回复中确认：`「本章设计包已使用表格格式」`
4. orchestrator 在子agent返回后抽检首章确认格式

### ❌ scope 声明缺失
**现象**：只读到全书一部分时，产出使用"全书架构""全书主题"等绝对化表述。
**原因**：子agent不知道全书总量，默认当前数据=全书。
**解决方案**：
1. 委派时在 context 中告知全书总量：`全书共约9卷1000+章，当前只读到了前187章（卷1）`
2. 要求每个产出文件在第二行声明 scope
3. orchestrator 最终合入 `卷1-架构.md` 而非 `全书架构.md`
