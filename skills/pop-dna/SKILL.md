# pop-dna · 文风DNA蒸馏引擎 v4.0

> **给 prose-render 的不是指令——是原文。** prose-render Step 1 全量加载此文件，对照设计包的 scene 字段直接定位场景卡。

---

## ❌ 质量红线

```
□ 场景卡原文 < 500 字 → 退回。原文不够，render 无法感受作者笔触。
□ 场景卡 < 6 个 → 退回。场景类型覆盖不足。
□ 通用维度 < 4 个 → 退回。
□ 任何场景卡无"时间演变"段 → 退回——不知道早期/中期/末期差异的文风认知是残废的。
```

---

## 一、核心方法论

### v4.0 相比 v3.0 的核心变化

| v3.0 | v4.0 |
|:-----|:-----|
| "战斗" = 1 个维度 | 战斗拆为 4 个场景卡（早期/中坚/后期/暗杀） |
| 每条维度 2-3 处 1-2 行原文 | 每个场景卡 500+ 字完整原文 |
| 场景类型由 prose-render 自己判断 | 设计包 scene 字段 → DNA 场景卡 1:1 映射 |
| 产出一个"观察"文件 | 产出 prose-render 可直接对照的场景参考 |

### 做什么

从原文中**按场景类型**提取作者在每种战斗/对话/描写/叙述形态下的笔触。每个场景卡 = 1-2 句观察 + 500+ 字原文 + 时间演变。产出的不是分析报告，是 prose-render 渲染时旁边打开的参考页。

## 二、执行流程

```
步骤0 · 取样
  → 全书扫描，确认章节边界和全卷分布
  → 均匀采样：前 20% + 中段 40-60% + 后 20%，每段 ≥10 章
  → 目标：≥30 章，≥20,000 行
  → 不到 20,000 行 → 退回，继续读

步骤1 · 逐章精读采样章
  → 读完整章全文（不跳段、不只看开头）
  → 标记章内的场景类型和边界（一个章可能含多个场景类型）
  → 为每种场景类型标记候选原文段

步骤2 · 全书搜索验证
  → 对每个可能写成绝对结论的判断，搜全书验证
  → 搜索数字标注在产出中

步骤3 · 写出风格文件 (★ v4 核心)
  → 按 `templates/style-dna-profile.md` v4 产出
  → Step 3a：选出 ≥4 个通用维度 + ≥6 个场景卡
  → Step 3b：为每个场景卡截取 500+ 字原文（保持段落的完整性，不截断对话/不截断动作链）
  → Step 3c：写观察 + 时间演变，并确认场景卡名与设计包 scene 字段对齐
  → 全文约 20-25K

步骤4 · 试写验证
  → 选 2 个场景卡 → 对同一个骨架事件各写 300 字 → 对比是否捕捉到了差异
```

---

## 三、场景卡选择法则

> ∴ 从实际存在的场景出发选择卡名。不凑数、不硬套、不拉虚的。

```
① 读采样章 → 标记所有场景类型（战斗/对话/描写/叙述 × 阶段）
② 去重 → 合并同类型的候选（"ch1 小巷揍流氓"+"ch4 暗杀巴拉
斯"→ 选后者，因为它更典型、目标更明确）
③ 选 ≥6 个最典型的 → 每个配 500+ 字原文
④ 卡名对齐 prose-render 的 scene 字段命名（场景卡矩阵见模板）
```

---

## 四、与 prose-render 的集成

```
prose-render Step 1（读入输入）:
  ① 全量加载 styles/{书名}.md (~20-25K，Get-Content 安全范围内)
  ② 读设计包 → 获取本章 scene 字段（如 combat_early_skirmish）
  ③ 在 DNA 文件内定位对应的场景卡
  ④ 带这个场景卡的原文 + 观察进 Step 2 渲染

为什么全量加载而非按需：
  - 文风不是场景的简单拼接 — 过渡段/情绪缓冲的写法
    需要整体感知
  - 20-25K 在 Get-Content 安全范围内，不触发截断
  - 文件全量驻留在 context 中，允许 agent 自由对照维度
  - 单一 DNA 文件结构简单，维护成本低
```
