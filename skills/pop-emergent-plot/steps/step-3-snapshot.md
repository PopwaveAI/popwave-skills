# Step 3 - 事实快照（正文写作阶段的动态状态载体）

> **定位**：事实快照是正文写作阶段的信息一致性保障机制。幕纲是静态的剧情设计，快照是动态的事实记录。
> **维护者**：API维护，主agent调度。主agent不直接提取事实（避免上下文污染）。

---

## 一、为什么需要事实快照

正文写作中每章都会涌现新信息：NPC的具体行为/战斗的具体结果/角色受伤情况/新出现的设定细节。这些信息在幕纲和施工卡中不存在。

如果没有快照：
- ch004写"尸鼠王10级"，但没标注阶位→ch005引用时可能写错阶位
- ch006写"薇拉姐妹会"，但幕纲写的是血帮→矛盾无法被发现
- ch002写"索伦左臂受伤"，ch005写到换药时忘记了这个伤

快照解决：每章写完后提取新事实→下一章写作时注入→确保信息一致。

---

## 二、快照文件结构

文件路径：`产出/正文/事实快照-幕NNN.md`

```markdown
# 幕NNN 事实快照

## 全局设定（从幕纲初始化，正文不可覆盖）
- 帮派名称：血帮
- 索伦标签：杀戮神子
- 薇薇安标签：恐惧神子
- 阶位体系：一阶1-5/二阶6-10/三阶11-15/四阶16-20/传奇21-25
- 其他幕纲锁定的硬约束...

## 已确立事实（每章追加）
### ch001
- [事实条目1]
- [事实条目2]

### ch002
- [新事实条目]

## 当前角色状态（每章更新）
- 索伦：X级盗贼，[装备]，[状态]，[伤势]
- 薇薇安：[状态]
- [其他有名角色]：[状态]

## 待确认偏离
- [描述正文与幕纲不一致的地方。无则填"无"]
```

---

## 三、快照生命周期

### 3a. 初始化（幕纲确认后、施工卡生成前）

**执行者**：主agent

幕纲确认后，主agent从正式幕纲中提取全局设定，初始化快照文件。初始化内容：
- 从幕纲第八部分（prose-render交接）提取不可改动事件 → 写入"全局设定"
- 从幕纲第三部分（六要素）提取人物列表 → 写入"当前角色状态"初始值
- 从种子文档/世界圣经提取阶位体系等基础设定 → 写入"全局设定"

**红线**：初始化时不得从正文提取信息（此时还没有正文）。

### 3b. 注入施工卡（施工卡生成时）

**执行者**：主agent拼入API prompt

施工卡生成时，user prompt中必须包含当前快照全文。施工卡API能看到所有已确立的设定和事实，确保施工卡的"不可改动事件"与快照一致。

### 3c. 注入正文（每章正文写作时）

**执行者**：主agent拼入API prompt

每章正文写作时，user prompt中必须包含当前快照全文。正文API能看到所有已确立事实，避免hallucination。

### 3d. 更新快照（每章正文写完后）

**执行者**：API（主agent调度）

每章正文写完后，走一次轻量API调用更新快照：

- system prompt：
```
你是事实提取器。读取本章正文和当前快照，提取本章新确立的事实，更新快照。

规则：
1. 只提取客观事实（谁做了什么/谁受了什么伤/什么设定被确立），不提取主观描写
2. 全局设定区不可覆盖——如果正文出现了与全局设定矛盾的内容，标到"待确认偏离"
3. 已确立事实区追加新事实，不修改已有事实
4. 当前角色状态区用最新状态替换旧状态
5. 输出完整的更新后快照（不是增量）
```

- user prompt：当前快照全文 + 本章正文全文
- 输出：更新后的完整快照

### 3e. 偏离处理（主agent检查）

**执行者**：主agent

每章快照更新后，主agent检查"待确认偏离"区：

| 偏离类型 | 例子 | 处理 |
|---------|------|------|
| 正文hallucination | 幕纲写"血帮"，正文写"薇拉姐妹会" | 重写该章（在prompt中强调快照全局设定） |
| 正文涌现好创意 | 幕纲写"鼠王野生"，正文写"鼠王被召唤" | 老板决定是否采纳→采纳则更新幕纲+快照 |
| 正文补充细节 | 幕纲写"杀猎犬"，正文写"用匕首切眼睛补刀" | 快照记录细节，不算偏离 |

**红线**：主agent发现"待确认偏离"后不得忽略——必须处理（重写或上报老板）。

---

## 四、完整管线

```
幕纲确认
  → 3a 初始化快照（主agent从幕纲提取）
  → 3b 注入施工卡生成（API）
  → 施工卡产出
  
ch001正文
  → 3c 注入快照（API写正文）
  → ch001产出
  → 3d 更新快照（API提取事实）
  → 3e 检查偏离（主agent）
  
ch002正文
  → 3c 注入更新后快照
  → ch002产出
  → 3d 更新快照
  → 3e 检查偏离
  
...循环到最后一章
```

---

## 五、Token消耗估算

快照更新API调用每次约2000-4000 tokens（取决于快照大小和正文字数）。12章幕约增加12次API调用，总计约24,000-48,000 tokens。

相比正文写作的~400,000 tokens，快照更新成本约6-12%，可接受。

---

## 六、快照与幕纲的关系

| 维度 | 幕纲 | 事实快照 |
|------|------|---------|
| 性质 | 剧情设计（要写什么） | 事实记录（已经写了什么） |
| 变更频率 | 幕纲确认后原则上不变 | 每章写完后追加更新 |
| 信息层级 | 宏观（弧线/爽点/节奏） | 微观（NPC名字/伤势/具体数值） |
| 读者 | prose-render | prose-render + 快照更新API |
| 矛盾处理 | 幕纲是源头真相 | 矛盾时以幕纲为准，快照标记偏离 |

**红线**：
- 快照不可覆盖幕纲的全局设定
- 幕纲不可在正文写作中途修改（除非老板批准的"好创意"偏离）
- 快照必须每章更新，不得跳过
