# 剧情架构 v2.7 — 情绪驱动·爽点分布设计

> 版本：**v2.7** · 2026-05-25
>
> **v2.7 核心变化**：
> - 新增节奏自检3项基线检查：事件数基准（连续2章≥3核心事件）、scene_type多样性（同类型不超过2章）、字数基线（2000-2500字）
>
> **v2.6 核心变化**：
> - 新增节点B导演判断——幕设计前先思考"这个幕的第一性是什么"，对齐上游设计后再出文件

>
> **v2.5 核心变化**：
> - 新增 HARD-GATE 过程铁律——无开书启动禁止进入剧情架构
>
> **v2.4 核心变化**：
> - 新增「卷末高潮·Boss战设计通则」——三阶段结构、质量区分、情绪温度计、环境作为第二对手
>
> **v2.3 核心变化**：
> - 新增「核心三要素密度检查」——战斗/恐怖/成长每连续2章至少出现1次，防止节奏偏科

> 旧假设：小说的本质是把信息传给读者 → 幕纲的功能标签=信息分类
> 新假设：小说的本质是让读者经历情绪变化 → 幕纲的第一性=爽点分布与情绪弧线
>
> 这个改动波及全文。它是根因层的修正——不是加新字段，是换"这层在做什么"的假设。

## 平台与读者画像

从 v2.1 起所有幕纲设计必须先定义目标读者画像。同v1.0。

番茄用户的阅读本质：**每章都要有微爽点。** 这不是"节奏快"的问题——是读者打开一本书的核心期待是"这章能让我爽一下"。

## 爽点系统总览

### 爽点四级体系

| 等级 | 频率（番茄） | 读者反应 | 铺垫-释放比 | 案例 |
|:---:|:---|:---|:---:|:---|
| **微爽点** | 每章1-2个 | "还行，有点意思" | 2:1 | 一句骚话、一个小打脸、一个小获得 |
| **中爽点** | 每3-5章1个 | "好，这章值了" | 4:1 | 打脸一个配角、获得新能力、阶段性胜利 |
| **大爽点** | 每10-15章1个 | "卧槽！" | 8-10:1 | 卷内高潮、逆袭大敌、能力质变 |
| **终极爽点** | 每卷末1个 | "这本书太牛了" | 20:1 | 卷末Boss战、全书级转折、不可逆质变 |

### 爽点的底层动机

爽点的本质不是"主角变强了"，是**"读者希望发生的事情发生了"**。

| 动机类型 | 读者共鸣 | 诡异游戏适用场景 |
|:---|:---|:---|
| **打脸爽** | "让看不起主角的人后悔" | 玩家轻视规则→被规则反杀 |
| **获得爽** | "我也想要这个" | 获得新道具/新能力/新信息 |
| **升级爽** | "我也想变强" | 能力质变、规则理解突破 |
| **守护爽** | "保护我在乎的人" | 保护队友、保护自己的秘密 |
| **复仇爽** | "让伤害我的人付出代价" | 回头解决前期的威胁 |
| **解谜爽** | "原来如此！" | 规则真相揭晓、伏笔回收 |

### 情绪基调与爽点的配合

不是每章都需要"爽"。但**每章都需要让读者经历一次情绪变化**——即使是压抑章，也要让读者感受到"压抑是有目的的"。

| 情绪基调 | 配合的爽点类型 | 读者读完后 |
|:---|:---|:---|
| 紧张 | 无（蓄力中） | "我靠，要出事" → 想翻页 |
| 压抑 | 微爽点在中段 | "还有希望" → 继续追 |
| 释放 | 中爽点 | "舒服了" → 满足但想看更多 |
| 期待 | 信息差钩子 | "下一章会怎样" → 强烈追更 |
| 震撼 | 大爽点 | "卧槽" → 记住这本书 |

## 核心能力

1. **全书爽点节奏设计** — 微/中/大三级爽点在全书时间轴上的分布
2. **卷级情绪弧线** — 每卷的读者情绪起点→顶点→终点
3. **幕纲生成（情绪优先）** — 每章以读者情绪目标+爽点类型为第一性，功能标签降级为辅助信息
4. **爽点版场景卡输出** — 幕纲设计完成后，为每个爽点等级各写 1 段 300-500 字正文片段，验证爽点设计的落地效果
5. **场景类型锚定** — 幕纲切片的 `emotional_goal` 和 `payoff` 字段决定下游 emergent-writer 的场景类型识别，进而决定选哪个场景模板。因此每章切片的 emotional_goal 和 payoff 必须准确描述该章的核心情绪基调（识别规则见 emergent-writer 的模板池）。
6. **偏离检测** — 与参考书情绪曲线对比（保留v1.0）
7. **爽点库引用** — 设计每章payoff时参考学习资料的 `外部干货/爽点设计库.md`，从类型/触发方式/铺垫释放比/情绪叠加等维度选取组合

## 卷末高潮·Boss战设计通则（v2.4 新增）

> 卷末Boss战是"终极爽点"最典型的释放形式。
> 以下原则为通用方法论——不绑定任何特定题材或力量体系。

### 通则1：三阶段结构

普通战斗与Boss战的本质区别是有无「阶段转换」。没有阶段转换的战斗不该被当作Boss战对待。

| 阶段 | 功能 | 读者情绪 | 战斗面貌 |
|:---:|:----|:--------|:--------|
| **P1 环境压制** | 展示Boss的压迫感，主角试探性交手 | "这东西不好打" | Boss控制环境/地形限制主角/主角未全力出手 |
| **P2 全力对抗** | 主角全力输出，发现常规手段不够用 | "他打不过——那怎么办？" | Boss形态变化/攻击模式改变/主角受伤 |
| **P3 打破极限** | 借助非常规手段完成逆转（代价/外力/牺牲/规则活用） | "卧槽——但他付出了什么？" | 战斗逻辑改变/不再拼谁更强/代价明确呈现 |

**关键规则**：
- 三个阶段缺一不可。P1+P2=白打一场，P2+P3=缺少压迫感铺垫。
- 每个阶段结束时必须有一次"读者认知刷新"：P1结束→"这东西比想象中强"，P2结束→"主角打不过它"，P3结束→"代价值不值得"。

### 通则2：Boss战 vs 普通战斗的质量区分

| 维度 | 普通战斗 | Boss战 |
|:----|:--------|:------|
| 阶段数量 | 单阶段 | 三阶段 |
| 主角胜算 | 读者默认主角能赢 | 读者不确定主角能赢 |
| 战后代价 | 喘口气就好 | 影响至少后续3章的决策和状态 |
| 情感弧线 | 无变化 | 推动角色弧线（认知/心态/关系发生不可逆变化） |
| 配角参与 | 可独自完成 | 必须有外部视角反映战况 |

**检验问题**：「删掉这场战斗，剧情弧线是否受影响？」
- 普通战斗的答案是"删掉后从下一章开始依然成立"
- Boss战的答案是"删掉后角色的某个不可逆变化就没了"

### 通则3：次级角色作为"情绪温度计"

Boss战的主角处于战斗"内部"——读者只能看到他的动作和反应，无法感知战况的"严重程度"。

解法：**通过次级角色的反应来标定战况的温度**。

| 战况 | 观众角色的反应 |
|:----|:-------------|
| 常规压力 | 正常沟通、"你还行吗" |
| 开始危险 | 语气变化、简短催促 |
| 极限状态 | 打破角色惯常的语气模式（如从不正经的角色突然认真说话） |
| 濒死边缘 | 沉默——不说话的反应比任何台词都重 |

**核心技法**：情绪温度计的角色必须是读者熟悉其"正常状态"的。读者需要知道"他平时说话什么调调"才能感知到"他这次说话不一样了"。

### 通则4：环境作为"第二对手"

Boss战期间，环境不应是中性的。环境必须与Boss的威胁联动，制造**双重压迫感**。

| 手法 | 效果 | 例子 |
|:----|:----|:----|
| 环境随Boss状态变化 | 阶段转换时，环境也同步改变 | Boss暴怒→房间开始震动/灯光闪烁 |
| 限制主角的活动空间 | 迫使主角在不利条件下战斗 | 地板塌陷/墙壁收缩/门锁死 |
| 环境本身成为陷阱 | 战斗时一手应对Boss一手应对环境 | 地面出现漩涡/天花板滴落腐蚀性液体 |

**关键规则**：Boss战的"场地"应该像另一个角色一样被设计——有自己的"攻击方式"和"阶段变化"。

## 情绪叠加模型

有效的章节往往是混合情绪，而不是单一情绪。设计幕纲的 emotional_goal 时，推荐选择 2 种混合情绪：

| 叠加 | 效果 | 示例场景 |
|:---|:---|:---|
| **紧张+好奇** | 知道有危险，不知道什么危险 | 新威胁登场 |
| **爽+不安** | 变强了，但代价没兑现 | 拿到能力但副作用未知 |
| **绝望+希望** | 困境到了底，但有反杀线索 | 逆境章，卷中转折 |
| **释放+期待** | 难题解决了但更大挑战在前 | 卷中小高潮后 |
| **震撼+敬畏** | 世界观被颠覆，读者心跳加速 | 终极真相揭晓 |

**铁律**：幕内不允许连续 3 章使用同一种情绪叠加组合。同一组合在整幕中不超过 3 次。

## 高强度节奏设计（叠加压缩技巧）

普通章节的节奏是线性的：A发生→B发生→C发生。高强度章节的节奏是叠加的：A没解决→B叠加→C同时发生。

设计幕纲时，以下位置**必须使用叠加压缩**：
- 每幕的第 1 章（开篇定调）
- 中爽点释放章（叠加制造峰值）
- 大爽点/终极爽点前的铺垫章（蓄力到极限）
- 情绪弧线图中的 intensity ≥ 8 的关卡

**叠加不超过 3 层**，否则读者会麻木。每叠加一层，字数应缩短（制造压迫加速感）。

## 爽点版场景卡

### 为什么要写爽点版场景卡

已有的场景卡解决的是"本书的质感是什么样"（风格种子），爽点版场景卡解决的是**"本书的爽长什么样"**。

L0 只锁定了爽点类型（解谜爽/获得爽），但 LLM 和作者都不知道这个"解谜爽"写成正文是什么感觉。爽点版场景卡就是验证工具——在幕纲设计完成后、转入章纲生成之前，先写几段出来看看"这个爽点类型配上这个设定、这个主角，写出来到底爽不爽"。

### 写法

幕纲设计完成后，为每个爽点等级各写 1 段 300-500 字的正文片段：

| 等级 | 写几段 | 字数 | 用途 |
|:---:|:---:|:---:|:---|
| 微爽点 | 1 段 | 300-400 字 | 验证"最日常的爽"能不能写出来 |
| 中爽点 | 1 段 | 300-500 字 | 验证"阶段性成果"的情绪释放 |
| 大/终极爽点 | 1 段 | 400-500 字 | 验证"卷末高潮"的冲击力 |

### 格式

```markdown
# 爽点场景卡 · 微爽点

## 元数据
- 对应幕纲章节：ch4
- 爽点类型：获得爽
- 触发方式：预期差（读者以为失败，其实成功了）
- 铺垫：尝试了三次萃取都失败
- 释放段字数：约 350 字

## 正文（释放段）

[以主角视角写的 300-400 字正文片段，从铺垫的临界点写到爽点释放完成]
```

### 产出后的用途

1. **验证** — 作者读完："这爽对了"→ 进入章纲阶段；"这不对，不够爽"→ 回到幕纲调整 payoff_distribution
2. **风格种子** — 首章写作时，微爽点场景卡可替代"上一章结尾"作为 style_seed 注入（与已有场景卡机制共用）
3. **下游参照** — 正文写作时，可以引用爽点版场景卡的释放方式作为节奏参考

## 幕纲设计完成后的节奏自检

幕纲输出后，逐条检查以下指标：

```
□ 连续无爽点章节 ≤ 3 章？（超过说明节奏崩了）
□ 连续无新信息章节 ≤ 2 章？（超过读者会腻）
□ 高强度章间距 ≥ 3 章？（太密读者疲劳）
□ 情绪低谷长度 ≤ 2 章？（超过 3 章读者弃书）
□ 终极爽点预留了整卷铺垫？（20:1 的释放比，不能临时起意）
□ 大爽点之前的 2-3 章是否用叠加压缩蓄力了？
□ 幕的情绪弧线是否有 "起"？（不是一直平）

□ ⭐ 核心三要素密度检查（v2.3 新增）：
   □ 每连续2章中，战斗(⚔️)/恐怖(🔴)/成长(🟢) 至少出现1次？
     扫描全幕，标记所有战斗/恐怖/成长的分布。
     如果存在连续2章同时缺三者 → 那2章需要重新设计。
     允许单章缺三者，不允许连续2章同时缺三者。
   □ 每卷出现"零核心三要素"的纯推进章节 ≤ 3 章？
     （纯解谜/纯社区感/纯过渡章节属于辅助，不能成为主力）

□ ⭐ fun_level 零爽点硬检查（v3.1 新增）：
   □ 所有 chapters 的 fun_level 字段已填写（不得空缺）
   □ 连续 2 章 fun_level="零" → 必须修改其中至少 1 章
     参考：番茄基准每章至少 1 个微爽点

□ ⭐ scene_time 多样性检查（v3.1 新增）：
   □ 同一种 scene_time（白天/黄昏/入夜/深夜/黎明）连续不超过 3 章
     超过 → 至少替换 1 章的时间设置，增加时间轴变量

□ ⭐ payoff trigger 代价多样性检查（v3.1 新增）：
   □ 同一类型代价（如"纸人被撕碎"）不在 2 个以上的 payoff trigger 中重复使用
     物理代价/情感代价/信息代价/环境代价 应交替出现

□ ⭐ 事件数基准检查（v2.7 新增）：
   □ 每连续2章中，核心事件数是否 ≥ 3？
   □ 如果存在连续2章事件数 < 3 → 那2章需要合并或追加场景
□ ⭐ scene_type 多样性检查（v2.7 新增）：
   □ 同一场景类型（战斗/对话/探索/日常）连续不超过2章
     超过 → 至少替换1章的场景类型，增加场景变量
□ ⭐ 字数基线检查（v2.7 新增）：
   □ 每章字数应在 2000-2500 字范围内
   □ 低于 1800 或高于 2800 → 标记为异常

任意一项不通过 → 回到幕纲调整对应章节的 payoff_distribution 或 emotional_arc
```

## 平台节奏校准检查（v2.2 新增）

**为什么需要**：剧情架构设计时通常以"什么样的节奏最舒服"为标准——但这个"最舒服"在不同平台上完全不同。番茄读者要求每2-3章一个强爽点，起点可以接受更深的铺垫。幕纲完成后必须做平台专项校准。

**触发条件**：幕纲设计完成 + 节奏自检通过后。

### Step 1：确定平台标准

| 节点 | 番茄 | 起点 |
|------|------|------|
| **第1章钩子** | 必须有（冷开场/反常识/强悬念） | 第5章前有即可 |
| **核心卖点亮相** | 第5章内 | 第10章内 |
| **第一次战斗** | 第10章内 | 第20章内 |
| **第一个副本闭环** | 第15章内 | 第30章内 |
| **最大纯铺垫区间** | ≤ 1章 | ≤ 3章 |

### Step 2：逐项校准

```
## 平台适配检查清单

目标平台：[番茄/起点/其他]

- [ ] 第1章有钩子吗？（目标平台要求的时间节点内）
- [ ] 核心卖点亮相达标？
- [ ] 第一次战斗达标？
- [ ] 第一个副本闭环达标？
- [ ] 有无超过标准值的纯铺垫区间？
- [ ] 最强反差爽点在前15章内（番茄）？
- [ ] 卷末有悬念钩子吗？

任意一项不通过 → 执行节奏重排（不删内容，只重排顺序）。
重排后必须做时间线一致性校验（开书启动 Step 1.6.6）。

**附加检查：核心三要素密度**
重排后的版本必须再次跑一次核心三要素密度检查（本文件 节奏自检 中的 ⭐ 项）。
```

### Step 3：节奏重排原则

- **不删内容**：所有设计元素保留，只调换章节顺序
- **核心逻辑不改**：角色的成长线/因果关系不改变
- **前置检查**：重排后必须检查因果链有无断裂、能力解锁时间有无错乱

### 设计时的自检问题（v2.0更新）

```
□ 这一章的核心爽点是什么类型？（打脸/获得/升级/守护/复仇/解谜）
□ 读者读完这章后，情绪锚定在哪？（而不是"知道什么信息"）
□ 如果这章没有爽点（纯过渡），读者为什么不会弃书？
  答案不能是"因为后续会爽"——必须是"这章本身提供了情绪价值"（好奇/紧张/期待）
□ 这一章的钩子是"情绪驱使"还是"信息驱使"？
  - 信息驱使："读者想知道答案" — 弱
  - 情绪驱使："读者想知道主角会怎样" — 强
□ 这章结束时的情绪，与下一章开头的承接是否一致？（如果ch5结尾紧张，ch6开头不能直接轻松）
```

## 与v1.0的关键变化

| 维度 | v1.0 | v2.0 |
|:---|:---|:---|
| 设计假设 | 小说=信息传输 | 小说=情绪设计 |
| 章级第一性 | function_tags（功能标签） | emotional_goal + payoff（爽点类型） |
| emotional_goal | 辅助标注（2个字） | 明确描述（50-100字情绪路径） |
| 钩子 | type+content | type+driver（情绪驱动力分析）+content |
| 功能标签 | 核心分类 | 辅助标注 |
| 爽点 | 无 | 四级体系（微/中/大/终极）+ 情绪叠加模型 |
| 节奏设计 | 无（靠经验） | 叠加压缩技巧（高强度节奏设计方法） |
| 设计后自检 | 无 | 7项节奏诊断指标（通过才进入下一阶段） |
| 自检问题 | 信息导向（"读者知道X吗"） | 情绪导向（"读者感受到什么"） |

## 剧情架构在管线中的位置

```
【开书启动】Phase 0 → L0灵魂对齐（赛道+参考书+爽点+主角契约+三大引擎）
    ↓

    ↓
【节点B · 幕设计前思考】（v2.6新增）
  ─ 思考内容：
    ① 对齐检查：上游L0/L1设计与这个幕的设计一致吗？
    ② 核心判断：这个幕的第一性是什么？（不是填模板，是想清楚"这个幕在全书里扮演什么角色"）
    ③ 情绪弧线：幕的情绪起点→顶点→终点合理吗？
    ④ **双轴检查：对抗模型是否有"强度+深度"两个维度？**
       - 幕内敌人的来源层（民俗层/神话渗入层/源流层）和当前叙事阶段匹配吗？
       - 参考 `_shared/universal-knowledge/timeline-tripartition.md` 的时间轴三分原则
    ⑤ 风险扫描：平台节奏要求是否满足？爽点分布是否有连续空白？
  ─ 产出：【节点B设计说明】（散文式判断链，非结构字段）
  ─ 如发现上游设计冲突 → 输出「待确认/疑问」返回上游
    ↓
【黄金三章】← ★ 独立阶段（CH1-CH3）
  │  幕纲爽点提级（中爽点起步，三章内有大爽点）
  │  场景卡全量验证（每章核心事件各1段，≥6段）
  │  简略章纲分析强制（每章做）
  │  写完 → 转入正常管线
    ↓
【剧情架构】→ 幕纲设计（情绪弧线 + 爽点分布 + 情绪叠加模型）
    ↓
【剧情架构】→ 爽点版场景卡（每等级写 1 段验证爽点落地效果）
    ↓
【剧情架构】→ 平台节奏校准检查（v2.2新增 — 幕纲完成后做平台适配校准）
    ↓ 验证通过
【正文写作】→ 上下文组装（读幕纲+global_summary+character_state+style_seed）
            → 节点C · 导演判断 → 单次章节生成 + 轻量自检
```
