## 一、设计起源

### 动机

诡异游戏v2 CH001-CH010 实战中暴露三个根因问题：
1. 字数前重后轻（CH003+004占31%，CH008-010仅19%）
2. "不是而是"反复出现（3章累计清理40+处，QC修了又出）
3. 后段收束无力（CH009单章1,220字，仅大纲预期60%）

底因：小说正文可拆为"事实层（剧情骨架）"和"描述层（感官渲染）"，但当前AI一次生成两件事都干，token预算争抢，总有一个被牺牲。

### 核心假设

```
正文 = 事实骨架（选择性的情节事件序列 + 实体状态变换）
     + 描述渲染（感官细节 + 修辞 + 情绪着色 + 节奏控制）
```

## 三、关键决策记录

### 决策1：实体系统用YAML，不用DB

原因：YAML可读性强、人类可直接编辑、与git diff兼容。DB作为可选持久层（04-数据库/novel.db），不替换YAML。

### 决策2：一实体一文件，禁止合并

原因：每个实体的状态层独立更新。巴拉斯死了卡诺波不一定同时死。文件合并导致状态追踪耦合。

### 决策3：ESM脚本化，不靠LLM读YAML

原因：19个文件时LLM还能勉强读，50-80个文件时会读错路径、格式漂移、跑得慢。ESM只做文件I/O+校验，不做LLM调用。

### 决策4：Pass 2不做剧情决策

确立于v0.3的编排层vs渲染层责任清单。Pass 2不能新增实体/概念、不能改变钩子类型、不能改动核心事件序列。

### 决策5：世界观插槽放在Pass 1

确立于CH012推演失败的复盘。最初考虑让Pass 2自主决定插入世界观，后被否决——因为"在哪里插世界观"是编排层判断，不是渲染层的事。
