## 一、在管线中的位置

### 全貌：从开书到正文的完整流程

```
【开书启动】——— skill-project-bootstrap
    ↓ L0 灵魂对齐 + L1 设定 + 宪法
【情节架构】——— skill-plot-architecture
    ↓ act-XX.yaml + 幕设计思考链（节点B产出）
【黄金三章】——— skill-opening-arc（CH1-CH3 独立流程）
    ↓
【正文写作】——— skill-emergent-writer
    │
    ├── ① 素材准备（纯数据搬运 / 零LLM）
    │     读：幕纲切片 + 全局摘要 + 角色状态 + 上一章结尾 + 知识包 + 经验日志
    │     输出：导演 Agent 输入包
    │
    ├── ② 导演判断（1次LLM调用 / 新增）
    │     读：输入包（含知识+经验+素材）
    │     思考：6个战术问题
    │     输出：本章设计说明（散文式判断链）+ 待确认/疑问
    │
    ├── ③ 用户确认（可选 / 人机交互）
    │     无反问题 → 跳过
    │     有反问题 → 作者回答 → 记录到经验日志
    │
    ├── ④ Prompt 组装（纯数据搬运 / 零LLM）
    │     合并：6块素材 + 导演设计说明 → 写入 output-template.md
    │
    ├── ⑤ 单次章节生成（1次LLM调用）
    │
    ├── ⑥ 轻量自检（作者执行 / 7项）
    │
    ├── ⑦ QA 审稿（1次LLM调用 / novel-qa SOP-R 子Agent）
    │
    └── ⑧ QA 反馈回收（新增 / 关键！）
           将 QA 输出的关键定性语句 → 摘要 → 存入经验日志
           → 下一章的导演 Agent 能看到上一章的 QA 反馈
```

### 调用成本变化

| 版本 | 导演 | 生成 | QC/自检 | QA | 合计 |
|:---|:---:|:---:|:---:|:---:|:---:|
| v7.7 | 1 | 2-3（碎片） | 1 | — | 4-5 |
| v7.8 | 0 | 1 | 0 | 1 | 2 |

## 二、知识系统：让导演 Agent 真正懂

这是最关键的设计。导演 Agent 的核心输入除了素材，还有**两层知识**。

### 知识架构

```
┌────────────────────────────────────────────────────┐
│                 导演 Agent 输入包                    │
├─────────── 本章素材（6块） ────────────────────────┤
│  ① 幕纲切片 / ② 全局摘要 / ③ 角色状态              │
│  ④ 上一章结尾800字 / ⑤ 节点B思考链 / ⑥ 用户纠正   │
├─────────── 知识包（静态·随Prompt注入） ────────────┤
│  K1 核心理论（约800字）                             │
│  K2 平台画像（约300字）                             │
│  K3 量化基准线（约200字）                            │
│  K4 题材感知（约200字，按章节类型）                  │
├─────────── 经验日志（动态·随章节增长） ────────────┤
│  累积的用户干预 + QA 反馈摘要                       │
└────────────────────────────────────────────────────┘
```

### K1 · 核心理论（持久注入，约 800 字，来自外部干货）

```
网文底层原理：
  · 三大引擎：爽点（即时获得感） + 代入感（读者能代入） + 期待感（想翻下一页）
  · 黄金节奏：每章至少1个微爽点，每3-5章1个中爽点，每10-15章1个大爽点
  · 前三章决定留存——ch1拉人、ch2压住、ch3释放+新钩子

爽点设计原则：
  · 爽点六步法（ibiling六步）：渴望建立→阻碍出现→积蓄力量→突破时刻→释放满足→余韵回味
  · 爽点底层动机：打脸爽/获得爽/升级爽/守护爽/复仇爽/解谜爽
  · 每章1-2个微爽点，每章结尾必须有一个"翻页理由"

钩子设计：
  · 章末三种钩子：事件断裂型（关键信息打断）/ 悬念收束型（答案在下一章）/ 代价提示型（变强有代价）
  · 不能连续2章使用相同类型的钩子
  · 章节内部每800-1000字至少1个微钩子（反常行为/信息推进/角色变化）

节奏控制：
  · 不允许连续3章同一种节奏类型（紧张/喘息/推进）
  · 高强度章之间必须有低强度章缓冲
  · 对话占比建议≥25%，番茄读者偏好对话>旁白
  · 新概念每章≤2个（番茄读者认知负荷上限）
```

### K2 · 平台读者画像（持久注入，约 300 字）

```
番茄读者画像（当前项目目标平台）：
  · 平均阅读时长：15-30分钟/次
  · 决策时间：前300字决定是否划走
  · 爽点忍耐力：连续无爽点章节不能超过1章
  · 信息处理力：一次性新概念超过2个就开始困惑
  · 情绪需求：即时满足 > 长线铺垫，每一章都要有"这一章值了"的感觉
  · 弃书Top3原因：(1) 300字内没抓住人 (2) 连续2章无爽点 (3) 信息密度太大看不懂

节奏基准线（番茄）：
  · 第1章钩子：必须有（中爽点起步）
  · 核心卖点亮相：第5章内
  · 第一次战斗/副本：第10章内
  · 第一个副本闭环：第15章内
  · 最大纯铺垫区间：≤1章
```

### K3 · 量化基准线（持久注入，约 200 字）

```
量化检测基准（来自问题池+QA数据）：
  · 对话占比：≥25%（不够则增加对话碎片）
  · 新概念/章：≤2个（超过则删减或推迟）
  · 章末钩子：必须（事件断裂/悬念收束/代价提示）
  · 连续无爽点章节：≤1章
  · 连续独白段：不宜超过15行（插入威胁/悬念/冲突/感官异常）
  · "不是…而是…"句式：0容忍（否定式描写的标志）
  · "他感到/他意识到/他评估"类冗余前缀：尽可能减少

如果检测到异常，导演Agent必须在「待确认/疑问」区提出。
```

### K4 · 题材感知注入（按章节类型选择性注入，约 200 字）

```
诡异/恐怖（当前章类型匹配时注入）：
  · 让读者自己拼出真相比直接告诉真相恐怖十倍
  · 不要解释原理，让异常保持异常
  · 信息差是关键——读者比主角知道多一点点=焦虑；读者和主角同步发现=好奇

战斗/动作（当前章类型匹配时注入）：
  · 读者通过主角的视线看世界——描写主角眼中的天空，不是描写天空
  · 战斗的本质是信息差博弈，不是数值比拼

情感/暧昧（当前章类型匹配时注入）：
  · 暧昧的升华靠"有意无意"——调情、触碰、牵手，每一步都是放大的
  · 留白比控制更有张力
```

## 三、经验记忆：让导演 Agent 持续成长

这是"让导演Agent活起来"的关键机制。每次用户干预和 QA 反馈都不应该丢失。

### 存储机制：`director-experience-log.md`

放在项目 `00-总控/` 或 `05-引擎/` 下（取决于项目结构），每章写一条。

```
# director-experience-log.md
> 自动累积。每次用户干预 / QA 反馈后追加一条。
> 写入后被下一章的导演 Agent 读取。

---
## ch005 | 2026-05-20

### 用户干预
类型：节奏修正
内容：开局太慢，前500字全是环境描写
修正方向：后续同类章节开局直接给画面 + 动作，不铺垫环境
生效：ch006 已调整

### QA反馈摘要
- "第5章中段节奏不错，但前段太拖"
- "对话占比刚好25%，能再多一点更好"
- 关键决策：下一章开局直接用动作场景切入

---
## ch006 | 2026-05-20

### 用户干预
类型：信息密度
内容：新概念出现3个，有点多
修正方向：世界观术语推迟到下一章用对话带出
生效：ch007 已调整
```

### 写入触发时机

| 触发点 | 记录什么 | 谁写 |
|:---|:---|:---:|
| 用户确认导演反问时说"不，改这里" | 用户的选择 + 原因 | 脚本或主Agent |
| QA 审稿发现严重问题 | QA 报告中的关键定性语句 + 决策 | QA 收口脚本 |
| 用户直接干预正文 | 用户说了什么 + 怎么改的 | 主Agent |
| 复盘会议产生的规则（如"纸人巷太长了"） | 规则内容 + 适用范围 | 作者 |

### 为什么这个机制重要

没有这个机制，导演 Agent **每次都被重置**——LLM 每次调用都是独立上下文，不知道上一章发生了什么、用户纠正了什么。有了经验日志：

```
第5章 → 用户说"开局太慢"
    ↓ 记录到经验日志
第6章 → 导演 Agent 读日志时看到"开局太慢"
    ↓ 开局会主动检查："上一章被批评开局慢，这章要注意"
第10章 → 经验日志已有 5 条用户干预记录
    ↓ 导演 Agent 不只是做判断——它在判断时会参考历史模式
```

## 四、思考任务与输出格式

### 导演 Agent 的 Prompt 结构（概念设计）

```
你是这章的导演。你的工作是思考，不是执行。

【你的身份】
你是一个资深网文编辑，对番茄小说的节奏了如指掌。
你读过足够多的爆款诡异流小说，知道什么样的开局能留住人、什么样的爽点能让读者翻页。
你不只是一个分析工具——你是一个有判断力的教练。

【整体知识（K1-K3，每次注入）】
{网文底层原理 + 番茄读者画像 + 量化基准线}

【题材感知（按章节类型选择性注入，K4）】
{诡异/战斗/情感……等题材特有要求}

【项目背景（可变）】
项目：{项目名称}
目标平台：{目标平台}
读者画像：{reader_persona}

【本章素材】
幕纲本章切片（emotional_goal / payoff / reader_emotion_path / end_hook）：
{act-XX.yaml 对应切片}

全局摘要：
{global_summary}

角色状态：
{character_state}

上一章结尾（最后800字）：
{prev_chapter_excerpt}

节点B设计思考链（如果有）：
{design_chain_from_plot_architecture}

上一轮 QA 反馈 / 用户纠正（如果有）：
{qa_feedback_summary}

经验日志（过去 5 条相关记录）：
{recent_experience_log_entries}

【你的思考任务】
请按照以下顺序思考，输出一份「本章设计说明」：

```

### 思考任务和输出格式（即设计说明模板）

```markdown
## 本章设计说明

### ① 上一章锚定校准
我读了上一章结尾的正文。
幕纲说本周起点是【X】，但实际上章结尾读者锚定在【Y】。
【评估：对齐 / 偏离】
如果偏离：本章需要调整的核心点在哪里？具体怎么调？

### ② 核心目的
这章如果只让读者记住一件事，是哪一件？

### ③ 笔墨优先级
幕纲列出了以下事件/场景。我的判断：
- 核心场景 —— 【事件名】：建议写【约XX字】。因为【原因】
- 过渡场景 —— 【事件名】：建议写【约XX字】。因为【原因】
- 可砍内容 —— 【事件名】：建议【砍掉/合并到核心/精简为一句话】。因为【原因】
- 补充建议 —— 如果有幕纲没列但我觉得本章需要的内容（比如开局画面、情绪缓冲段等）

### ④ 读者情绪时刻表
开局（前300字）：【具体情绪】
事件①：【具体情绪】
章中转折：【具体情绪】
事件②：【具体情绪】
章末锚定：【具体情绪】

### ⑤ 开局策略
上一章结尾：【具体画面+情绪】
我的策略：【直接延续 / 先喘息 / 反差切入】
具体建议：【这章第一句话或者第一个画面是什么？为什么这么开场？】

### ⑥ 风险自查
【无风险 或 列出】：
- 信息密度检查：本章涉及【N】个新概念 → 【OK/过多】
- 对话占比预估：【高/中/低】 → 【OK/偏低，建议增加对话场景】
- 节奏类型：和上一章【相同/不同】 → 连续相同类型时警告
- 经验参考：经验日志中有【N】条类似位置的记录 → 具体建议

### ⑦ 待确认 / 疑问
【无】或逐条列出：
- 问题1（说明异常 + 建议方向 + 请作者确认原因）
- 问题2……

### ⑧ 关键提醒（给写作层的建议）
[自由文本，1-3条]

```

## 五、反问机制

### 触发规则

| 触发条件 | 反问内容 | 格式 |
|:---|:---|:---|
| 上一章实际偏离幕纲计划 | 偏离的幅度 + 建议调整方案 | 应当怀疑 |
| 本章信息密度 ≥ 3 新概念 | 列出概念 + 建议砍哪些 | 应当建议 |
| 本章对话占比预计偏低 | 预估比例 + 建议增加对话场景 | 应当建议 |
| 经验日志中有同位置翻车记录 | 历史记录 + 避免重复的建议 | 应当提醒 |
| 幕纲设计有潜在缺陷 | 缺陷分析 + 替代方案 | 可以建议 |
| 检测到"不是A而是B"句式风险 | 具体位置 + 替代写法 | 应当警告 |
| 连续 N 章异常（同类节奏/低对话/无钩子） | 趋势数据 + 调整建议 | 应当建议 |

### 用户确认后的处理

用户确认后，回答内容写入经验日志。如果用户驳回反问，也要记录——"用户选择不使用这个调整"本身也是数据。

## 六、与现有 Skill 的配合逻辑

```
                skill-mapping.yaml（入口路由）
                        │
          ┌─────────────┼──────────────┐
          │             │              │
          ▼             ▼              ▼
  skill-plot-     skill-             skill-
  architecture    emergent-writer    market-test
  （设计幕纲）     （写正文）          （验证市场）
          │             │              │
          │     ┌───────┼───────┐      │
          │     │       │       │      │
          │     ▼       ▼       ▼      │
          │  素材   导演    Prompt     │
          │  准备   判断   组装+生成    │
          │         │                 │
          │         ▼                 │
          │     经验日志 ← QA反馈 ──┘
          │         │
          └─────────┼──────────────┘
                    ▼
             director-experience-log.md
```

### 具体配合规则

| Skill | 与导演 Agent 的配合 | 发生节点 |
|:---|:---|:---|
| **skill-plot-architecture** | 产出的 act-XX.yaml 含情绪目标+爽点分布+情绪路径+章末钩子 → 导演 Agent 读这些做判断 | 幕纲设计完成后，每章写作前 |
| **skill-opening-arc** | CH1-CH3 用独立流程，但导演 Agent 在 CH1-CH3 中做更激进的知识注入和更多的反问（因为前三章最危险） | 每章写作前 |
| **skill-emergent-writer** | 导演 Agent 的产出（设计说明）作为一块额外素材嵌入 Prompt → 写作层消化后自建内容 | 每章写作前 |
| **skill-market-test** | 市场验证的反馈 → 修正在 act-XX.yaml 中 → 后续章节的导演 Agent 看到修正 | 市场测试后 |
| **novel-qa（审稿）** | QA 反馈 → 摘要 → 经验日志 → 下一章导演 Agent 参考 | 每章 QA 后 |

### 关键配合要点

1. **导演 Agent 不取代任何 Skill** — 它是 emergent-writer 内部的一个子步骤，不是独立 Skill
2. **导演 Agent 的输入依赖前面的 Skill** — plot-architecture 的产出越精细，导演的判断越准
3. **导演 Agent 的产出影响后面的 Skill** — 写作层读到的不仅仅是结构化字段，还有判断链
4. **QA 反馈通过经验日志回传给导演** — 形成闭环

## 七、用户干预沉淀机制

用户干预是最高质量的输入——因为它直接修正了 LLM 的错误判断。必须被捕获、摘要、可回溯。

### 记录流程

```
用户说："这章开局太慢了"
    ↓
脚本/主Agent捕获：
  识别干预类型（节奏/信息密度/对话占比/人物表现……）
  提取关键语句（"开局太慢"）
  判断适用范围（仅本章 / 同类章节通用 / 全管线规则）
    ↓
写入经验日志：
  ch005 | 2026-05-20 | 节奏修正
  用户的原话：……
  作者的解读：……
  调整了什么：……
    ↓
下一章导演 Agent 读到了这条日志
    ↓
```

### 结构化的存储格式

```yaml
# 每条用户干预存为一个结构化的块
- chapter: 5
  date: 2026-05-20
  type: "节奏"  # 节奏 / 信息密度 / 对话占比 / 人物表现 / 情节逻辑
  severity: "P1"  # P0 必须改 / P1 建议改 / P2 可记可不记
  user_verbatim: "前500字全是环境描写，太慢了"
  analysis: "开局铺垫过长，番茄读者前300字没看到进展就会划走"
  resolution: "后续同类章节开局直接给画面+动作，不铺垫环境"
  applied_to: "all_chapters"  # this_chapter_only / similar_chapters / all_chapters
  status: "active"  # active / superseded / expired
```

这个结构化数据可以存在 YAML 中，也可以直接嵌入 Markdown 的经验日志中。选 YAML 的好处是可以被脚本自动解析。
