# 导演层重构设计 · 节点 C（每章正文前）

> 设计日期：2026-05-20  
> 基于：emergent-writer v7.9、复盘遗漏问题清单  
> 状态：设计阶段（待确认后实施）

---

## 一、定位

节点 C 是**写每章正文前的那一次思考**。不是工序执行（拆碎片/选场景卡），不是格式转换（YAML→文本），是**真正的战术设计判断**。

### 它和现有节点的关系

```
情节架构设计（节点B）
    产出 act-XX.yaml + 幕设计思考链
        ↓
  导演 Agent（节点C）← 新增
    思考这章怎么写 → 产出设计说明（含反问）
        ↓
  上下文组装 → 写作 → 自检
```

它不是取代上下文组装 —— 上下文组装仍然做纯数据搬运（读文件、拼模板）。  
节点 C 的产出**作为一块素材**喂到上下文组装 → 嵌入写作 Prompt。

---

## 二、导演 Agent 的输入

每次调用时，系统给它以下素材：

| # | 素材 | 来源 | 用途 |
|:---:|:---|:---|:---|
| ① | 幕纲本章切片 | act-XX.yaml | emotional_goal + payoff + reader_emotion_path + end_hook |
| ② | 全局摘要 | global-summary.md | 全书进展，理解故事当前位置 |
| ③ | 角色状态 | character-state-anchor.md | 每个角色当前状态 |
| ④ | 上一章正文结尾 | 实际读取 **最后 800 字** | 验证读者真实锚定位置 |
| ⑤ | 节点B思考链（如果有） | 幕设计阶段留存的设计说明 | 理解"这章为什么这么设计" |
| ⑥ | 用户方向纠正（如果有） | 上一轮 QA 反馈/用户直接指令 | 避免重复踩坑 |

---

## 三、导演 Agent 的思考任务

六个问题，按顺序思考：

### ① 上一章实际锚定校准

```
幕纲说本章起点是 X 情绪——上一章真实写出来了吗？
  → 如果对齐：继续按计划
  → 如果偏离：本章怎么调整才能弥合断裂？
```

**价值**：这是 v7.8 上下文组装器做不到的事。组装器只读字段（"上章结尾情绪锚定"），不验证字段描述是否匹配实际正文。

---

### ② 这章的核心目的

```
这一章如果只让读者记住一件事，是哪一件？
  → 一句话说清楚
  → 接下来所有判断以此为准绳
```

**价值**：确立优先级锚点。如果多个判断冲突（"这边多写点" vs "那边节奏要快"），回溯到这个核心目的做取舍。

---

### ③ 笔墨优先级判断

```
幕纲列出了 N 件事——它们在读者体验维度上的权重一样吗？
  → 核心场景：哪个事件是本章的魂？要写多少字？
  → 过渡场景：哪个事件是推进作用？可以写少吗？
  → 可砍内容：哪个事件其实是灌水？能砍掉或合并吗？
```

**价值**：这是导演层最核心的判断——v7.8 把判断砍掉后，LLM 对 3 个核心事件均分笔墨，导致"该浓的地方不够浓，该淡的地方不够淡"。

---

### ④ 读者情绪时刻表

```
不是"起点→终点"两段式——是每一次关键节点上的读者状态：

  · 前 300 字读完，读者什么情绪？
  · 第一个关键事件发生，读者什么情绪？
  · 章中转折点，读者什么情绪？
  · 章末读完，读者什么情绪？想翻下一页吗？
```

**价值**：v7.8 的"情绪路径"只有起点+终点+3个锚点，太粗。5-6 个节点更接近实际阅读体验。

---

### ⑤ 开局策略

```
上一章结尾的实际情绪状态 + 本章第一个画面 = ?
  → 是直接延续（不中断情绪流）？
  → 还是先喘息（高强度后给读者放松）？
  → 还是反差切入（情绪急转）？
  → 具体用哪个画面/对话/动作打开？
```

**价值**：开局决定读者是否停留。这个判断依赖"真正读上一章结尾"，不是看字段。

---

### ⑥ 风险自查

```
从这根线的整体经验看，这章容易踩什么坑？
  → 信息密度过高？对话太少？解释过多？
  → 有没有上一章类似位置翻过车的记录？
  → 写法提示：特定题材的节奏要求（诡异/战斗/暧昧）
```

**价值**：把复盘经验（遗漏问题池）带进每章的判断，而不只是写在文档里。

---

## 四、导演 Agent 的输出格式

不是 YAML，不是字段列表，是**散文式的设计说明**。

```markdown
## 本章设计说明

### 核心判断
这章如果只让读者记住一件事：__________
（一句话说清本章不可替代的价值）

### 情绪时刻表
开局（前300字）：__________
事件①触发：__________
章中转折：__________
事件②落地：__________
章末锚定：__________

### 笔墨分配
核心场景 —— {事件名}：约 800 字。因为__________
过渡场景 —— {事件名}：约 300 字。因为__________
可砍内容 —— {事件名}：建议砍掉或合并到核心场景。因为__________

### 开局策略
上一章结尾锚定在：__________
本章开局选择：__________
具体画面建议：__________

### 关键提示
- __________（写法的具体建议）
- __________（容易踩的坑）
- __________（题材特有的节奏要求）

### 待确认 / 疑问
[无] 或 [列出需要作者确认的判断点]
```

### 为什么要用散文式而不是结构化字段

1. **思考链可见** — LLM 不只看到结论（"核心场景 800 字"），还看到原因（"因为这段是本章的情绪峰值，前面 3 章都在铺垫它"）
2. **LLM 理解更深** — 散文式判断链比字段列表更接近人类决策过程，LLM 消化后生成的正文更精准
3. **灵活性** — 不需要预定义所有字段，导演可以根据章节类型自由组织优先级
4. **反问自然融入** — "待确认"区可以直接用自然语言提问，不需要单独设计提问格式

## 五、反问机制

导演 Agent 在思考过程中，如果发现以下情况，必须输出到「待确认/疑问」区：

| 触发条件 | 反问语气 | 例子 |
|:---|:---|:---|
| 上一章实际偏离幕纲计划 | 疑惑型 | "上一章结尾实际锚定在'迷惘'，幕纲计划的是'紧张'。如果按幕纲走本章开局，可能会造成情绪断裂。是否临时调整本章的 emotional_goal？" |
| 幕纲设计有潜在缺陷 | 建议型 | "核心事件②的区分度和①不够高，两个事件都是'发现线索'类型。建议合并为一个，省出的 300 字给核心事件③深化写——是否同意？" |
| 信息密度过高/过低 | 风险型 | "本章涉及 5 个新概念（含 2 个世界观术语），超过番茄平台的上限。建议砍掉 2 个或在对话中带过。需要我直接在设计中调整吗？" |
| 发现可优化的机会 | 推荐型 | "上一章 QA 反馈提到'第 5 章的节奏偏慢'。本章的开局适合直接给画面不铺垫，可以弥补上一章的节奏问题。我已在设计中做了这个调整。" |

**让作者在导演输出后、写作开始前，有机会做一次确认。** 这不是"每次都要问"——是"有问题才问"。

## 六、在管线中的位置

```
① 素材准备（纯数据搬运）
    → 读 6 块素材
    → 拼成导演 Agent 输入包
    → 零 LLM 调用

② 导演判断（新增·1 次 LLM 调用）
    → LLM 做 6 个思考任务
    → 输出本章设计说明（散文式）
    → 输出待确认/疑问（如果有）

③ 作者确认（可选·人机交互）
    → 如果有反问 → 作者确认/回答
    → 如果有建议 → 作者决定
    → 无反问题 → 跳过

④ Prompt 组装（纯数据搬运）
    → 拿 6 块素材（含设计说明）+ 写入模板
    → 零 LLM 调用

⑤ 单次章节生成（1 次 LLM 调用）
    → 完整 Prompt → LLM → 正文

⑥ 轻量自检 + QA
    → 和现有流程一致
```

**管线总调用成本**：
- 旧 v7.8：0 次（导演层） + 1 次（生成） + 0 次（自检） + 1 次（QA）= **2 次**
- 新方案：1 次（导演） + 1 次（生成） + 1 次（QA）= **3 次**
- 对比旧 v7.7：1 次（导演） + 2-3 次（碎片）+ 1 次（QC）= **4-5 次**

**增加 1 次调用换回真正的判断能力**，比 v7.7 省 1-2 次。

## 八、对比：新导演层 vs v7.7 旧导演层

| 维度 | v7.7 旧导演层 | 新导演层（节点C） |
|:---|:---|:---|
| 做的事情 | 拆碎片、选场景卡、写碎片目标 | 核心目的判断、笔墨优先级、情绪时刻表、开局策略 |
| 输出性质 | 指令（"碎片A写什么、碎片B写什么"） | 设计说明（"这章什么重要、怎么走情绪"） |
| 对写作层的控制方式 | 指令卡 → 子Agent 执行 | 判断链嵌入 Prompt → LLM 消化后自建 |
| 反问机制 | 无 | 有（待确认/疑问区） |
| 调用成本/章 | 1次 LLM | 1次 LLM（成本相同） |
| 信息损耗 | 高（指令卡过滤了大量上下文） | 低（设计说明嵌入完整 Prompt，作为增量上下文） |
| 灵活性 | 低（字段固定、模板固定） | 高（散文式，根据章节类型自由组织） |
