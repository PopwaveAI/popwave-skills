> 版本：v1.0 | 生成日期：2026-05-23 | 基于：项目复盘交叉分析 + 诡异游戏v2实战复盘 + 用户反馈

## 一、问题全景

### 1.1 问题分类框架

所有问题分 **六个域** 归类，每个域对应一条根因方向：

```
域A · 剧作能力 —— 小说不懂剧情设计，不知道什么是爽点，无法操控读者情绪
域B · 管线可靠性 —— 启动、引用、胶水代码质量差，任务经常不按 skill 走
域C · 场景覆盖 —— 只覆盖了开书场景，未覆盖续写/连载迁移
域D · 编排体系 —— pop 作为中枢的设计是否合理，Harness 架构待定
域E · 执行质量 —— 上下文判断弱，不知道丢哪些上下文
域F · 锚定书效果 —— 从剧情设计到正文产出均不达预期
```

### 1.2 全量问题清单（37条）

> 标记说明：🟥 = P0（必须修） | 🟧 = P1（重要） | 🟨 = P2（可延后） | ⬜ = P3（观察/待定）

#### 域A · 剧作能力（8条）

| # | 问题 | 优先级 | 来源 | 说明 |
|:-|:-----|:------:|:----|:-----|
| A-01 | 🟥 **不懂剧情设计的底层原理** | P0 | 用户反馈 | 小说不知道什么是爽点、情绪曲线、节奏控制——当前"剧情设计"依赖模板匹配，不是真正的剧作理解 |
| A-02 | 🟥 **爽点设计停留在分类层面** | P0 | 用户反馈 | 能说出"这是中爽点"但不知道它为什么爽、什么时候该放、怎么构建前摇-释放-余韵 |
| A-03 | 🟥 **无法有意识地操控读者情绪** | P0 | 用户反馈+复盘 | 情绪曲线在幕纲层面有设计，但正文执行后情绪终点经常偏移。CH002绝望感积累不足即典型案例 |
| A-04 | 🟧 **战斗场景无参与感** | P1 | 问题池③ | 读者=主角视角，没有信息差。读者知道主角能赢，丧失紧张感 |
| A-05 | 🟧 **对话占比系统性偏低（~15%）** | P1 | 问题池⑦ | 旁白占85%，导致场景缺乏节奏变化和人物鲜活度 |
| A-06 | 🟧 **章末钩子系统性失效** | P1 | 遗漏① | 多章节末尾以"旁白总结"收束，不是"事件断裂"。诡异游戏v2跨章验证 |
| A-07 | 🟧 **信息密度失衡** | P1 | 问题池⑥ | 前期过密（诡异游戏首章13个新概念 vs 诡异人生3个），中期过稀 |
| A-08 | 🟨 **修炼/升级段不恐怖** | P2 | 问题池⑫ | 升级没有代价感，缺乏仪式感 |

**根因**：所有 A 域问题都指向同一个方向——**小说引擎没有"剧作理解"，只有"写作模板"。** 它知道"怎么写"但不知道"为什么这么写"。爽点分类、场景卡模板、情绪曲线都是外挂约束，不是引擎内化的能力。

| B-03 | 🟧 **输出端QC双Layer从未真正落地** | P1 | 遗漏③ | P0三个问题（一眼看不懂/信息密度/对话太少）的方案全部"已设计待实测"，从未在真实章节上跑通 |

#### 域D · 编排体系（5条）

| # | 问题 | 优先级 | 来源 | 说明 |
|:-|:-----|:------:|:----|:-----|
| D-01 | 🟧 **pop作为中枢的设计是否合理** | P1 | 用户反馈 | pop 承担了"身份声明""思考引导""编排控制""质量责任"四重角色，职责边界不清晰 |
| D-02 | 🟧 **POP-CALL 模板与实际使用脱节** | P1 | 实战观察 | POP-CALL.md 定义"先声明后做事"，但在连续多章的实践中，pop 声明在新任务开头被遵守，但中间环节逐渐被省略——说明模板没有形成肌肉记忆 |
| D-03 | 🟨 **Harness 编排层 vs AI直觉的冲突** | P2 | 扩展 | 当用户说"继续"时，AI绕过 Harness 直接写。Harness 设计成"可选入口"而非"必经路由" |
| D-04 | 🟨 **多 skill 协作时的上下文冲突** | P2 | 扩展 | 不同 skill 的 prompt 模板对同一概念（如"情绪曲线"）的定义不完全一致，跨 skill 传递时信息劣化 |
| D-05 | ⬜ **三棍调度器是否成立存疑** | P3 | 核心争议 | 节奏定量化的想法对，但方法（三棍）的有效性从未验证。可能走错方向 |

**根因**：**编排层是在已经有大量 skill 之后"补"出来的，不是从头设计的。** pop 是后加的编排者角色，Harness 是后加的路由层。后加的东西天然有缝合痕迹。

| # | 问题 | 优先级 | 来源 | 说明 |
|:-|:-----|:------:|:----|:-----|
| E-01 | 🟧 **任务进行中不知道丢哪些上下文** | P1 | 用户反馈 | 当前上下文选择依赖 AI 的自我判断，缺乏标准化的"上下文注入工序" |
| E-02 | 🟧 **跨章情绪承接无系统性方案** | P1 | 遗漏⑦ | CH003→CH004 的情绪断层修复了，但修复方式是加了一条人工铁律（"上一章正文权重 > 大纲权重"），不是系统能力 |

| E-04 | 🟧 **场景核心写成事件序列** | P1 | 问题池⑤ | 场景卡的任务是"制造情绪体验"，但执行结果是"报告事件发生" |

| E-06 | 🟧 **后段压缩（后半本书质量显著下降）** | P1 | 实战发现 | CH008-CH010 字数严重不足、收束无力。这不是巧合——是管线在"没有新内容要揭示"时缺乏驱动 |
| E-07 | 🟨 **连续写作时的状态一致性** | P2 | 实战发现 | 印的位置矛盾（八仙桌底 vs 灶房架子）说明跨章追踪能力弱 |
| E-08 | 🟨 **零容忍约束的执行痛点** | P2 | 实战发现 | "不是而是"和"拍"字在QC中反复出现——生成侧没有内化约束，全靠事后修复 |

**根因**：**管线的"执行层"没有问题，但"上下文装配层"有问题。** 每章写了什么、角色状态是什么、哪些线索已揭示——这些信息存在于 runtime 文件中，但没有被标准化工序自动注入写作 prompt。

## 二、迭代路线图

### 阶段划分逻辑

```
Phase 0：问题修复 —— 补最短的板，让现有管线能稳定产出
Phase 1：能力建设 —— 解决"不会"的问题（剧作/爽点/情绪）
Phase 2：架构优化 —— 解决"不稳定"的问题（编排/上下文/胶水）
Phase 3：场景扩展 —— 解决"只覆盖一种场景"的问题（续写/迁移）
Phase 4：正反馈循环 —— 解决"不进化"的问题（自学/回馈/数据驱动）
```

---

### Phase 0：止血（当前 ~ 2周）

**目标**：让现有管线在诡异游戏v2的CH011+能稳定产出，不再出现QC反复修"不是而是"的情况

| 任务 | 对应问题 | 交付物 |
|:----|:--------|:------|
| **P0-1 零容忍约束内化** | E-08 | 在 emergent-writer SKILL.md 的生成 prompt 中直接嵌入"禁止使用'不是A而是B'和'拍'字"的硬约束，不是靠事后QC修 |
| **P0-2 过程铁律HARD-GATE** | B-01/B-02 | 在 emergent-writer 管线入口加硬闸门：① 检查 runtime 文件是否存在且最新 ② 没有幕纲不允许写章 ③ 没有导演判断不允许生成 |
| **P0-3 自动上下文装配** | E-01/E-02 | 写一章前自动读上一章正文+当前角色状态锚+全局摘要+当前幕纲，打包成"上下文包"注入写作 prompt |
| **P0-4 "读原文"工序标准化** | B-05/E-06 | 每章写作 prompt 第一段必须是"上一章发生了什么"的 100-150 字缩写，由 AI 读完后自写（不是复制） |

**Phase 0 完成标志**：可以连续写5章，每章QC一次通过，不需要事后修复"不是而是"和"拍"字。

---

### Phase 1：剧作能力建设（2 ~ 6周）

**目标**：让引擎从"知道怎么写"进化到"知道为什么这么写"

| 任务 | 对应问题 | 交付物 |
|:----|:--------|:------|
| **P1-1 爽点设计库结构化** | A-01/A-02 | 现有"爽点设计库"从卡片集升级为可检索的"爽点分类 + 前摇模式 + 释放模式 + 余韵模式"结构化库 |
| **P1-2 情绪曲线执行检查** | A-03 | 在QC中新增情绪检查项：写作完成后验证章末情绪是否命中预设，偏差超过1级（如"绝望"写成了"恐惧"）标记并提示 |
| **P1-3 信息密度配额表** | A-07 | 首章新概念 ≤ 5个，单章新概念 ≤ 3个，超过自动预警。通过章级配额表控制 |
| **P1-4 对话配比检查** | A-05 | QC新增对话占比检查：战斗章≥20%，日常章≥35%，低于阈值标记 |
| **P1-5 章末钩子类型检查** | A-06 | QC新增钩子类型检查：不允许连续2章以"旁白总结"收束 |
| **P1-6 战斗场景模版升级** | A-04 | 将"信息差"概念植入战斗场景模板——读者必须有一个主角不知道的信息才能产生紧张感 |

**Phase 1 完成标志**：5人试读后能准确说出每章的"情绪高潮点在哪""这一章爽在哪里"，而不是"写得还可以"。

---

### Phase 2：编排与架构优化（6 ~ 10周）

**目标**：让 pop、Harness、胶水层成为稳定可靠的"操作系统"

| 任务 | 对应问题 | 交付物 |
|:----|:--------|:------|
| **P2-1 pop职责重定义** | D-01 | 明确 pop 三层边界：① 身份声明（必须）→ ② 思考引导（轻量）→ ③ 质量担保（QC检查前签字）。移除编排控制职责 |
| **P2-2 Harness 路由强化** | D-03 | 将 Harness 从"可选入口"改为"必经路由"——pop声明后强制调用 orchestrate.py 匹配意图 |
| **P2-3 QC Layer 2 实测** | B-03 | 核心P0问题（一眼看不懂/信息密度/对话太少）的QC方案在10章实测中跑通 |
| **P2-4 管线格式自动化检查** | B-04 | 在 post_write.py 中集成格式检查（换行/编码/单行长度） |
| **P2-5 运行时文件自动维护** | B-06 | 每章完成后自动更新 global-summary 和 character-state-anchor（用 Python 脚本，不用人工） |
| **P2-6 多skill上下文一致性协议** | D-04 | 统一所有 SKILL.md 中"情绪曲线""爽点等级""场景类型"等跨 skill 概念的定义，建立术语表 |

**Phase 2 完成标志**：一个新项目的10章正文，从开书→大纲→写作→QC 全链路只需要作者的创意决策，不需要手动维护任何运行时文件。

---

### Phase 3：场景扩展（10 ~ 14周）

**目标**：从"只帮人开书"扩展到"帮任何人写任何阶段的网文"

| 任务 | 对应问题 | 交付物 |
|:----|:--------|:------|
| **P3-1 续写管线** | C-01/C-02 | _continuation/ 从"一次性适配层"升级为完整管线：① 读已有章节 → ② 重建角色状态锚+全局摘要 → ③ 识别当前剧情位置 → ④ 适配幕纲设计 → ⑤ 继续写作 |
| **P3-2 连载迁移工作流** | C-03 | 存稿拆解工序：① 分章 → ② 提取设定信息 → ③ 重建角色状态锚 → ④ 标记未回收伏笔 → ⑤ 适配现有管线 |
| **P3-3 平台格式抽象层** | C-04 | 番茄/起点/晋江的差异抽象成配置：章节长度范围、标点规范、敏感词表、段落分割规则 |
| **P3-4 涌现角色管理** | 遗漏⑨ | 写作过程中"涌现"的新角色（大纲没写但写着写着出现的配角）如何统一命名、记录初始描写、标记与主线关系 |

**Phase 3 完成标志**：用户可以把一本20万字的连载小说从起点迁移过来，管线自动完成拆解和适配，不丢失任何伏笔和角色状态。

---

### Phase 4：正反馈循环（14周+）

**目标**：引擎从"用完就忘"进化到"越写越懂"

| 任务 | 对应问题 | 交付物 |
|:----|:--------|:------|
| **P4-1 拆书产出回馈宪法** | F-03 | 拆书发现的规则自动生成"宪法修正建议"，经作者确认后注入书籍宪法 |
| **P4-2 文风DNA版本进化** | F-04 | 每写完一卷，从正文中提取 n-gram 模式和句式偏好，自动更新文风DNA文件 |
| **P4-3 创作数据仪表盘** | 扩展 | 每章产出数据（字数/QC通过率/修改轮数/情绪命中率/对话占比）可视化，辅助作者优化节奏 |
| **P4-4 三棍调度器实验** | D-05 | 如果前三个阶段验证了"节奏可以量化"的假设，则实验三棍调度器。如果否，则正式放弃"定量节奏"方向 |
| **P4-5 跨书学习** | 扩展 | 同一作者的不同书之间的"风格基因"共享——不共享具体文字，共享句式模式、对话比例偏好、场景类型使用频率 |

**Phase 4 完成标志**：引擎为作者写的每一本书都笔力更稳——同一作者的第二本书不需要修前10章，因为引擎已经理解了该作者的节奏偏好。

## 三、核心矛盾的解决方案

### 矛盾1：约束越多写得越差 vs 没有约束质量不可控

| 约束策略 | 效果 | 结论 |
|:---------|:----|:----|
| 14条红线（灰骑士旧版） | 均分4.6 | ❌ 死亡约束 |
| 4条红线+章纲+类型定位 | 均分7.1 | ✅ 关键转向 |
| 零容忍约束（不是而是/拍字） | 执行有效但成本高 | ⚠️ 事后修复 |
| 过程铁律（硬闸门） | 未实测 | ➡️ Phase 0目标 |

**解**：约束分两类——**内容铁律**（越多越差，限制创作空间）和**过程铁律**（越多越好，保障执行稳定性）。当前所有约束混在一起，需要分类清理。

### 矛盾2：对齐度100%≠质量好

复盘验证：v2 对齐度100% 均分5.5 vs v3 对齐度72.5% 均分7.0

**解**：大纲是"方向"不是"图纸"。从"检查是否对齐大纲"转向"检查是否达到情绪目标"。

### 矛盾3：幕纲设计好 vs 正文写出来不好

场景卡均分8.5 vs 正文均分5.5

**解**：Phase 1 的"展示技法内化"和"上下文装配"联合解决——不是加更多约束，是让引擎在写的时候知道"这个场景的目的是什么"。
