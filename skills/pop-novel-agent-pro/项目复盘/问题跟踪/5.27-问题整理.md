# 5.27 问题整理

## 概述

2026/5/26-5/27 期间尝试一次性自动化跑完全部50章，最终在ch031附近崩溃，失败3-4次后中断。最终回退到分段手动推进模式。本文档记录根因分析及改进方案。

## 二、根因分析

### 根因1：没有"已写/未写"状态过滤

pipeline不区分章节的实际完成状态——ch001和ch050在progress.json中完全同级。B01-B08对所有50章一视同仁地执行，未写章因缺乏Director/骨架等规划数据，产出正文质量极低。

**严重度**：🔴 致命

**关联故障**：故障2

### 根因2：没有上下文预算管理

单会话累计14万字历史文本，而每章实际有效输入仅需~1500字。未在写新章前主动清理上下文（压缩、总结、截断），导致模型被迫负载大量无用历史。

**严重度**：🔴 致命

**关联故障**：故障1、故障3

### 根因3：没有断点恢复机制

progress.json 中未记录"最后一次成功完成的步骤"，崩溃后不知道该从哪章继续。手动猜测恢复位置导致覆盖已写内容。

**严重度**：🟡 中

**关联故障**：故障4

### 根因4：单步WARN修复成本链式放大

B07五问审查触发WARN后，需要重跑B02-B05所有前置步骤。1次WARN ≈ 4次额外开销。B07有8次WARN记录，其中5次发生在ch026-ch030（空跑导致），3次为违禁句式修复。

**严重度**：🟡 中

**关联故障**：故障4

### 根因5：DB Schema 缺乏 SSOT

DB 建库 DDL 与 ESM 代码（loader.py/updater.py）中硬编码的列名从未对齐。两套 schema 各自独立演化：

```
DB 实际 (v3.db)              代码假设 (updater.py)
entity_name TEXT  ← 分歧 →   entity_id INT
chapter TEXT      ← 分歧 →   chapter_id INT  
field TEXT        ← 分歧 →   field_changed TEXT
(无)              ← 缺失 →   last_appearance
(无)              ← 缺失 →   notes
```

没有任何声明文件说"DB schema 长这样，代码按这个来"。这个模式与根因1-3完全相同——**信息分布在多个地方，彼此之间没有同步契约**。

**严重度**：🟡 中

**关联故障**：故障5

## 三、当前有效策略

| 策略 | 说明 |
|:----|:-----|
| 分段执行 | 每次处理2-5章，完成后汇报进度，再继续下5章。上下文不会膨胀到危险水平。 |
| 依赖外部记忆 | 不靠对话历史隐式记忆，依赖DB(v3) + bundle作为每章输入。 |
| 每10章复盘 | 检查暗线推进、角色弧光分配、设定一致性。 |
| session隔离 | 每5章可以视为一个独立工作单元，关键状态通过DB同步而非对话历史保持。 |

## 四、待改进项目（Pipeline代码层面）

| # | 改进项 | 优先级 | 预期效果 |
|:-:|:------|:-----:|:--------|
| 1 | progress.json 增加 `status: planned/written` 字段，pipeline只处理 written 的章节 | 高 | 杜绝空跑 |
| 2 | progress.json 增加 `last_checkpoint: chN` 字段，重启时直接跳转到下一章 | 中 | 崩溃后可恢复 |
| 3 | fail_count 接入实际计数器，当某步失败N次后自动标记 stuck 并跳过 | 中 | 避免死循环重试 |
| 4 | 在Director中嵌入违禁句式正面检查（"写作时规避：不是A而是B、直觉告诉、她/他没有"） | 高 | 减少事后修复工作量 |
| 5 | 战斗章/终章字数上限放宽标记（准许扩展到3300字） | 低 | 减少B06 WARN |
| 6 | 在`00-总控/`或`04-数据库/`下创建`schema.yaml`，声明每张表的字段名、类型、用途。loader.py/updater.py从此文件读列名，而非硬编码。 | 高 | 杜绝DB与代码的schema分歧 |
