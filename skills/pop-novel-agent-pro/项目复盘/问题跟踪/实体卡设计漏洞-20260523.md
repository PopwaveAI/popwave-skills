# 实体卡设计 · 两个待修漏洞

> 来源：ch392 生死战写作暴露 | 优先级：P1（不影响骨架剧情保真，但影响战斗渲染一致性）

---

## 漏洞一：交叉引用缺乏双向约束

### 当前状态

```yaml
# 威廉.yaml（角色卡）
state:
  skills:
    mastered: ["蒸气果实", "御心流剑术", ...]
  inventory:
    - item: 黑刀秋水
    - item: 和道一文字
```

```yaml
# 蒸气果实.yaml（技能卡）
permanent:
  owner: "{威廉}"
  abilities: ["元素化", "高温汽化", ...]
```

两个问题：
1. **蒸气果实卡和威廉卡没有验证一致性** — 威廉说已掌握，果实卡说主人是威廉。但 ESM 不校验"威廉有没有写已掌握"。如果威廉改卡漏了更新，就出现了"果实主人是威廉，但威廉的 mastered 列表没写"的断裂。
2. **黑刀秋水没有实体卡** — 威廉的 inventory 写了，但 `00-实体卡/item/` 下没有对应的 YAML 文件。所以 ESM check-refs 不会报错（因为它只校验声明过的实体引用），但数据是不完整的。

### 修复方案

**增加 `owns` 和 `owner` 双向字段规范，并让 ESM check-refs 做双向校验。**

角色卡新增：
```yaml
# 威廉.yaml
state:
  owns:          # ← NEW：明确列出持有的所有实体
    skills: ["蒸气果实"]
    items: ["黑刀秋水", "和道一文字"]
    relationships: ["阿金:心腹", "丹彼尔:智囊"]
```

技能/物品卡统一用：
```yaml
# 蒸气果实.yaml
permanent:
  owner: "威廉"          # ← 必须和角色卡 owns 中的名字一致
```

**ESM check-refs 新增校验规则：**
- 对每个 `entry.owner`，找到对应角色卡 → 检查角色卡 `owns` 列表中是否包含该实体
- 对每个角色卡的 `owns` 条目，找到对应实体卡 → 检查实体卡的 `owner` 是否指向该角色
- 双向不一致 → 报错

## 漏洞二：战斗/能力/修炼缺乏"案例集"

### 当前状态

对话有 `speech_style` 带具体语言样本，所以 Pass 2 写对话时知道"威廉不说话时怎么说话"。

战斗/能力只有名称列表，没有任何"这个能力在战斗中具体怎么表现"的参考数据。

```yaml
# 蒸气果实.yaml（当前）
permanent:
  abilities:
    - "元素化（身体化为蒸汽）"
    - "高温汽化"
    - "压力控制"
    - "过热形态（蒸气+生命归还+铁块融合技）"
```

Pass 2 收到这个 → "元素化我知道，但它的战斗节奏是什么样的？防御型还是位移型？消耗大吗？能用多久？强者怎么破解它？"→ 不知道，自己编。

### 修复方案

**增加 `combat_profile` 字段——能力和修炼的"案例集"，对标 `speech_style.examples`。**

```yaml
# 蒸气果实.yaml（升级后）
permanent:
  combat_profile:              # ← NEW
    combat_style: "游走/爆发/环境操控"    # 一句话概括战斗风格
    capabilities:
      - name: "元素化"
        type: "被动·常驻"       # vs "主动·消耗" vs "爆发·冷却"
        effect: "身体化为白色蒸汽，物理攻击无效"
        limits: "武装色霸气可命中；霸王色可强制解除；海楼石触碰失效"
        combat_use: "位移规避主力技"
        power_scale: "七武海级标准"
        
      - name: "高温汽化"
        type: "主动·持续消耗"
        effect: "蒸汽温度可熔钢铁，接触即造成烫伤"
        limits: "需保持元素化状态才有效；海楼石或海水直接接触会失效"
        combat_use: "范围压制/破除障碍"
        power_scale: "对人强，对大型目标效果递减"
        
      - name: "压力控制"
        type: "主动·瞬间爆发"
        effect: "压缩蒸汽产生推力，可实现高速位移或冲击波"
        limits: "连续使用3次后体力明显下降"
        combat_use: "加速/弹射/追击"
        power_scale: "单体强控"
        
      - name: "过热形态"
        type: "爆发·有限持续"
        effect: "蒸气果实+生命归还+铁块融合技，全身蒸汽包裹并硬化"
        limits: "维持≤3分钟；结束后虚弱期30秒"
        combat_use: "决战底牌"
        power_scale: "短暂达到准四皇级输出"
    typical_tactics:
      - "开局元素化规避试探 → 压力控制近身 → 高温汽化逼退"
      - "被武装色预测时 → 放弃元素化转为环境陷阱（蒸汽配合地形）"
      - "绝境 → 过热形态搏命"
    weakness:
      - "武装色霸气可中和元素化优势"
      - "海楼石/海水直接接触即失效"
      - "过热形态有严格的时限，时间差战术可破解"
```

**对于没有战斗能力的实体卡（汉库克/战国等），战斗章节时 ESM before_write 额外加载他们对应的战斗描述。** 如果实体卡没有 `combat_profile`，就搜索该世界观的通用战斗规则。

**跨书复用：** `power_scale` 和 `combat_style` 字段按各自世界观的值域填充。海贼法典用"凡人→四皇级"，深渊主宰用"凡人→传奇级"。ESM 不校验 value 本身，只校验字段是否存在。

### 同时增加「修炼/成长」的案例集

对标战斗的 `combat_profile`，增加 `training_profile`：

```yaml
# 御心流剑术.yaml（如果有）
permanent:
  training_profile:
    style: "奥山流·实战型"
    philosophy: "不重招式重预判"
    techniques:
      - name: "防守反击"
        description: "后发先至，利用对手的攻击节奏找破绽"
        level_requirement: "入门"
      - name: "流水"
        description: "连续攻击中无缝切换攻防姿态"
        level_requirement: "熟练"
    current_level: "熟练"
    progress_track: "已掌握防守反击，流水掌握中"
```

## 本次不做、但应当记录的设计约束

1. **关系三方以上（如"威廉持有秋水，秋水由莫利亚锻造，莫利亚已被威廉杀死"）** — 当前不处理。`relationships` 只处理二元关系。三方以上关系链由 `state_history` 的 timeline 覆盖。需要更多案例后再设计方案。
2. **战斗案例的 QC 标准** — `combat_profile` 由人写，但 ESM validate 应该增加一条：写战斗章时，检查本章涉及的所有实体是否都有 `combat_profile`，缺失则告警。不阻塞，但提醒。
