# 项目复盘遗漏问题清单（P0 / P1）

> 基于全部 12 份复盘文档交叉比对，提取现有「问题池与根因库」未覆盖或未正确定级的遗漏问题。
> 日期：2026-05-20

---

## 判定标准

| 级别 | 定义 |
|:---:|:---|
| **P0** | 影响读者体验/系统稳定性的硬伤，必须修复。不修则下一轮写作会重复踩坑。 |
| **P1** | 重要问题，影响质量上限或效率，但短期有绕行方案，可延后处理。 |

---

## P0（遗漏丨必须修复）

### 漏-① 章末钩子系统性失效（跨项目共同问题）

**来源**：灰骑士复盘沉淀 §三-核心问题、问题池 §四-跨项目共同问题

**问题**：多章节末尾以"旁白总结"收束，不是事件断裂。诡异游戏同样存在"无事件断裂"的章末问题。现有 v7.6 QC 层 **没有检查章末类型的规则**。

**证据**：
- 灰骑士审稿："多章节末尾以旁白总结收束，不是事件断裂"
- 问题池共同问题表中列了"章末钩子失效"，但 **修复优先级表中完全没出现**（P0/P1/P2 里都没有）
- AG 管线有"下一章前瞻锚定"机制可缓解此问题，但 v7.7 只加了"下一章情绪承接方向"，未落地章末钩子类型质检

**为什么是 P0**：章末钩子决定读者是否"翻下一页"。连续 3 章以旁白总结收尾 = 读者流失风险。

**建议方案**：QC Layer 2 新增章末钩子类型检查（事件断裂/悬念收束/代价提示/过渡缓冲），且不允许连续 2 章"旁白总结"。

---

### 漏-② "读原文"工序缺失（管线级信息损耗）

**来源**：密藏域拆解与开局重构复盘 §二-发现3

**问题**：当前管线假设"好正文 = 好宪法 → 好L1 → 好L2 → 好章纲 → 好正文"，每层都是文件接力。但没有任何一个步骤是**"读一遍参考书或旧版正文"**。

**关键实验证据**：
- v3 写了 6 章进入密藏域，质量稳定（写前读过原文）
- v2 写了 5 章进入密藏域，ch003 开始崩塌（写前只读过设定文件）
- **唯一变量是"是否读过原文"，压倒了所有其他变量**

**为什么是 P0**：这是当前管线信息损耗的最大源头。不修复，每本书的开局都会重复 v2 的崩塌模式——设定文件说得清楚，但正文写出来就是"感觉不对"。

**建议方案**：在章纲→正文之间新增"上下文注入工序"，核心职责是"读原文/旧版正文 → 提取功能清单 + 锚点状态 + 用户纠正方向 → 打包成写作输入包"。详见密藏域复盘 §三-3.3 的设计。

---

### 漏-③ 输出端 QC 双 Layer 从未真正落地

**来源**：问题池与根因库 §三-P0（① ⑥ ⑦ 均标注"待实测"）

**问题**：问题池中标 P0 的三个问题（①一眼看不懂、⑥信息密度失衡、⑦对话太少），**解决方案全部标注"已设计，待实测"**。设计文档写好了，但在真实写作中从未走通过 QC Layer 2 的闭环。

**当前状态**：
- QC Layer 1（场景卡入库 5 问质检）→ ✅ 实测通过
- QC Layer 2（正文输出端质检的 ① ⑥ ⑦ 规则）→ ❌ 从未在真实章节上跑通

**为什么是 P0**：三个 P0 问题的修复方案卡在"已设计未实测"阶段——相当于医生开了药方但病人没吃药。

**建议方案**：在下一次真实写作中强制执行 QC Layer 2 全检，跑通至少 3 个章节的完整闭环，根据实测结果修正规则。

---

### 漏-④ 过程铁律缺失（工序可跳步，无硬闸门）

**来源**：superpowers-融合分析 §1.1、§1.3

**问题**：novel-agent-pro 目前只有"内容铁律"（世界观设定不可违反），**没有"过程铁律"**（写作工序不可跳过）。AI 收到"写一章"指令时，依赖自身判断该做什么——无幕纲直接写章、无自检就声称完成，这些跳步行为没有被任何机制阻止。

**Superpowers 的对照**：
```
HARD-GATE: Do NOT invoke implementation until design is approved
Iron Law: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

**为什么是 P0**：没有过程铁律 = 管线随时可以被跳过 → 所有复盘沉淀的经验都白费。

**建议方案**（已含具体设计，直接实施）：
1. 在 `skill-plot-architecture`、`skill-emergent-writer` 头部加 `<HARD-GATE>` 硬闸门文本
2. 在 `skill-emergent-writer` 尾部加"完工 4 项检查"声明模板（自检通过 + global_summary 更新 + character_state 更新 + 下一章钩子写入）
3. 在 `skills/skill-mapping.yaml` 加 Red Flags 表（AI 理性化自查）

## P1（遗漏丨重要可延后）

### 漏-⑤ 场景卡调用率长期偏低

**来源**：灰骑士复盘沉淀 §四-P2写作复盘

**问题**：场景卡调用率仅 30%，QC 阶段退回率约 40%（因信息密度过高）。这意味着碎片写作的核心优势（场景卡驱动节奏）没有充分发挥。

**证据**：灰骑士 CH11-CH20 复盘数据——"只有 30% 的碎片用了场景卡"。

**建议**：查明调用率低的根因（是导演层没分配？还是场景卡类型不匹配？），针对性优化。

---

### 漏-⑥ 对话类型场景卡存量不足

**来源**：灰骑士复盘沉淀 §四-P2写作复盘

**问题**："对话类型碎片仍不足"被直接提及。对话占比低的根因之一是**缺乏高质量对话型场景卡供注入参考**。当前场景卡库以战斗/探索/恐怖场景为主。

**建议**：在建场景卡时，优先补齐"谈判/交易/情报交换/情感释放"等对话密集型场景的节奏样本。

---

### 漏-⑦ 跨章情绪承接系统性方案未落地

**来源**：AG管线经验吸收 §1.2-特征二、灰骑士复盘 §三

**问题**：虽然修复了 CH03→CH04 断层（读上一章正文），但这是修复了个例，不是系统性方案。AG 管线在每次写作 Prompt 中都带"下一章信息"（标题+定位+作用+简述），让 LLM 在写本章结尾时天然考虑下一章的承接。

v7.7 虽在导演层加了"下一章情绪承接方向"字段，但：
- 没有在 QC 层检查本章结尾是否真的做到了这个承接
- 没有在写本章时注入下一章的预期信息（AG 的做法）

**建议**：借鉴 AG 的"下一章前瞻锚定"——在写作 Prompt 中携带下一章的基本信息（标题+情绪基调+核心事件1句话），让 LLM 写本章结尾时有方向感。QC 层增加"本章结尾是否与下一章预期承接"的检查。

---

### 漏-⑧ dialogue_ratio 连续 N 章监测缺失

**来源**：qa对比pilot案例、诡异游戏复盘沉淀 §二

**问题**：QA 对比 pilot 建议"如果连续 3 章对话占比都低于 25%，则需调整"。这是一个明确的定量规则，但当前没有任何机制自动监测**连续多章**的对话占比变化——QC 层只检查单章。

**建议**：在 QC 层或 global_summary 中维护一个"最近 3 章对话占比滚动窗口"，当连续 3 章低于阈值时触发告警。

---

### 漏-⑨ 涌现角色管理机制未标准化

**来源**：V2设计迭代复盘 §三-3.3

**问题**：V2 复盘提出了涌现角色的完整管理流程（涌现 → 发现 → 补 L1 角色卡 → 锁定行为模式 → 下一章可用），但这个流程**仅存在于复盘文档中，未写入任何 SKILL.md 或标准化工序**。

下次遇到涌现角色时，大概率会重复"有戏 → 没补卡 → 下一章行为不一致 → 浪费了好角色"的路径。

**建议**：将涌现角色管理流程写入 `skill-emergent-writer/SKILL.md` 或项目的 `00-总控/process_rules`（新增）。

---

### 漏-⑩ 拆书产出回馈宪法的闭环未建立

**来源**：密藏域拆解与开局重构复盘 §三、问题池根因库

**问题**：密藏域拆解产出了 12 条 P0 写法规则、7 个可复用设计策略，它们在 v3 写作中"在潜意识层面起了作用"——但**没有一条被正式写入宪法或 QC 规则**。下一次拆书时，大概率会重复"拆出好东西 → 写在报告里 → 后续写作没用到"的路径。

**建议**：拆书完成后设一个强制步骤："识别可入库规则 → 写入 constitution.yaml 或 QC 检查清单 → 确认写入后再推进下一阶段"。

---

### 漏-⑪ 结构层 QA 数据基础设施缺失

**来源**：qa对比pilot案例 §数据支持情况

**问题**：`plot_library.db` 从未建成，节拍/情绪曲线结构化数据不存在。当前 QA 对比的"结构层"（对话占比、新概念数、因果边数）可以自动化，但因为缺数据库，每次都需要 Agent 在线判断，成本高且不可复用。

**建议**：补建 plot_library.db，至少包含"节拍时间轴表"和"钩子类型表"。优先级低于 P0，但不建这张表，QA 的精密度和效率上不去。
