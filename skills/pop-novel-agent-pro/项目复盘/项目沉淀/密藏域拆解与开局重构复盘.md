# 阶段性复盘：关于写作质量来源的追问

> 时间：2026-05-19
> 背景：对《我的诡异人生·密藏域》大规模拆解 → 开局重构实验 v1/v2/v3 → 输入注入实验 → AG对标
> 问题：一个"稳定产出高质量正文"的系统，核心变量到底是什么？

---

## 一、本轮做了什么

### 阶段1：密藏域拆解（模式B对标拆解）

全本970+密藏域关键词命中，提取12条P0写法规则、7个可复用设计策略。产出了《密藏域剧情完整拆解与复盘报告》。

在这个阶段确认了几个事实：密藏域是全书质量最高的板块，丹加线是情感锚点，精莲反派的悲剧性塑造，双驼峰结构，以及——**前25章原文开局其实是全书的短板**（零CL、零密藏域风味、大量工具行为）。

### 阶段2：开局重构实验 v1→v2→v3

基于拆解结论，试图修复原文开局的问题。经历了三轮：

| 版本 | 核心错误 | 致命伤 |
|:---:|:--------|:------|
| v1 | 密藏域冷开场，砍掉模拟器系统 | 用户纠正："模拟器和CL才是最大卖点" |
| v2 | 5章赶进密藏域，ch03起质量崩塌 | 过度追求压缩比，丢失呼吸感 |
| v3 | 不限章节预算，每章串行写 | 质量稳定，但依赖我读过原文的记忆 |

### 阶段3：输入注入实验（子Agent验证）

用Task子Agent复现写作，验证稳定性：

| 实验 | 结论 |
|:---|:------|
| 子agent写ch001 | 5项结构化输入+0前情 → 方向一致但犯了"影诡第1章现实实体化"的错误 |
| 子agent写ch002 | 5项输入+前章正文 → 质量和主agent持平，且独创了"印记"设定 |
| **对比验证** | 输入可控→输出可复现；但丢失长篇连贯性 |

### 阶段4：产品层PE反向拆解

尝试还原主agent写ch001时的真实输入，发现：
- 无法dump system prompt（架构不可见性）
- echo实验的上下文声明是"事后叙事"而非一手材料
- 调研了mitmproxy反向代理方案——理论上可行，依赖Trae是否开放自定义endpoint

## 二、核心发现链

### 发现1：约束 → 锚点的转换

v7.2实验曾证明"约束越多写得越差"。本次实验发现：**障碍是"禁令"性质的约束（不要/禁止/避免），不是"锚点"性质的约束。**

v3写作时真正生效的约束是：
- "这一章要交付4样东西"（要做什么）
- "模拟器是P0卖点"（什么重要）
- "结尾不能是'我再模拟一次'"（要避免什么，但只有1条）

而不是宪法里的8条红线。禁令创造"要避免的心理空间"，消耗认知带宽；锚点创造"要抵达的心理位置"，分配注意力。

但用户提出了另一个可能性——也许不是约束/锚点的区别，只是上下文足够多时，约束的权重自然降低了。我在写ch001时不会主动去想"R-01不能断"——因为我脑子里有太多其他信息（原文记忆、纠正方向、节奏表），约束语句被稀释了。

两者可能都是对的：**锚点的效率更高，但即使全是禁令，在足够多的"正事"信息面前，禁令也会被自然稀释。**

### 发现2：上下文密度的决定性作用

AG版诡异游戏ch001在同一章内塞了7件诡异事件（纸人动→血被吸→符裂→拇指移位→呼吸声→血符烧成灰球→无嘴自笑）。v3 ch001只塞了2件（激活+死）。

AG的质量不一定更高（AG叙事效率低、节奏平），但它的做法揭示了一个规律：**正文写作的核心不是"少犯错"，是"多塞有分量的信息"。**

v3 ch001质量可以的原因也不是"没犯错"——是它在一个高密度的骨架里放对了信息（系统完整、影诡副本、CL一闪、现实紧迫感）。如果换成"节奏正确但每件事都轻描淡写"的写法，质量会是另一个故事。

### 发现3：读原文 > 读设定文件

v3能写的根本前提是 **我看过原文**。不是读过分析报告，不是读过设定文件——是像读者一样从第一章开始读到结尾。这个阅读给我提供了以下无法被任何设定文件替代的东西：

1. 节奏感——知道什么时候快、什么时候慢
2. 骨架识别——知道"手表→出血→系统音→天赋→死→评分"这个序列是对的，不用自己编
3. 冗余识别——知道"云龙观找道士"是灌水，可以删
4. 关键信息位置——知道"罡洞"在未来出租屋里、知道"死老鼠味"是预警信号

回头对比：v3写了6章进入密藏域，质量稳定。v2写了5章进入密藏域，从ch003开始崩塌。**两套同样的约束、同样的节奏表、同样的目标——唯一的区别是写v3前我读过原文，写v2前我只读过设定文件。**

这个差异压倒了所有其他变量。

### 发现4：AG的做法和我们的差异不是对错，是分工

AG管线的优势不是"设计得好"，是它**清晰地区分了两类工作**：
- 元数据管理（角色状态、世界观事实、章节摘要）→ 由文件/DB持续维护
- 写作（把元数据转化为叙事）→ 由Prompt一次完成，不依赖对话历史

我们skill的做法是混合的：L0/L1/L2的设计文档试图完成第一类工作，但运送到写作层时已经丢失了大量上下文。写作层不得不依赖自身的历史记忆来补全——这就是为什么v3 ch003开始崩（记忆不可靠）。

### 发现5：主agent vs 子agent的适用场景

子agent实验的结论是：

| 方式 | 单章稳定性 | 长篇连贯性 | 输入可查性 |
|:----|:---------:|:---------:|:---------:|
| 主agent直写 | ⚠️ 依赖输入精度 | ✅ 好（有上下文） | ❌ 不可查 |
| 子agent独立 | ✅ 好（输入可控） | ❌ 差（无上下文） | ✅ 完全可查 |

子agent在ch001犯了"影诡第1章现实实体化"的错误——不是因为能力不够，是因为它不知道ch003的设计意图。这个信息的丢失不是注入格式的问题，是**子agent没有阅读过原文**。

## 三、对skill的反思

### 3.1 skill当前做对了什么

1. **宪法层**的方向没错：在开始写作前确认"这本书要卖给谁、卖什么、不能碰什么"——这是必要的决策框架。问题不在"不该有宪法"，在宪法被当成了写作时的红宝书而不是设计时的锚点。

2. **多层隔离**的思路没错：把设定设计、节奏设计、剧情设计、写作执行分给不同的职责——问题在层与层之间的信息传递方式（文件接力），而不是分层本身。

3. **拆书反哺**的方向对了：密藏域拆解产出12条P0规则，这些规则在v3写作中确实在潜意识层面起了作用（丹加前置、认知升级每章一次、伞状揭露设定）。

### 3.2 skill当前做错了什么

1. **假设写作是"分步组装"**：当前管线假设"好正文=好宪法→好L1→好L2→好章纲→好正文"，通过每层增加约束来保证质量。但v3实验和v7.2实验共同证明：这个链路的每一步都是信息损耗，不是信息增益。

2. **没有给"读原文"留位置**：skill管线中没有任何一个步骤是"读一遍参考书或旧版正文"。它假设作者已经知道自己在写什么——但这个假设在当前场景（AI辅助写作）中不成立。

3. **上下文运送方式是文件而非记忆**：当前每一层的产出是文件（PRD.md→宪法.md→act.yaml→章纲.md），下一层读文件。这种方式确保了"文档完整"，但丢失了"写作手感"——而手感恰好是高质量正文的来源。

4. **约束写在了错误的位置**：R-01/R-02/R-03写在宪法里，但v3写正文时它们从未进入工作记忆。如果它们应该影响正文写作，它们应该在"写作输入包"里，而不是在00-总控/constitution.yaml里。

### 3.3 迭代方向

**不是把现在已有的层合并得更少——是新增一个"上下文注入"工序，放在章纲→正文之间。**

```
当前：
设定设计 → 节奏设计 → 剧情设计 → 章纲 → 写单章
                                       ↕ 文件接力
                
建议新增的工序（在"写单章"前）：
读原文或旧版正文 → 提取这一章的功能清单
                  → 提取上一章的锚点状态
                  → 提取用户的方向纠正
                  → 打包为上下文注入包 → 写单章
```

这个工序的核心职责不是"产生新设计"——是**把已有设计中与本单章相关的部分，打包成一个适合写作的格式**。

这个工序不是一个文件产出另一个文件——是一个判断力密集型环节。它需要：
- 决定哪些信息全量注入、哪些摘要、哪些丢弃
- 识别用户纠正的方向是否已被消化
- 判断"原文这一段好在哪里，要在自己的正文里保留什么"

如果要做工具化，这个工序需要的不是YAML模板——是一张checklist，让写作者在动笔前确认"我这章的上下文包里打包了A/B/C/D/E/F/G了吗？哪块我跳过了？"

## 四、悬而未决

1. **三棍调度器**：它的可验证部分（代价对位、信息密度配额）已被吸收到独立规则中，不可验证部分（数值系统、差距区间）仍存疑。本次实验没有为它提供新的支撑或证伪证据。

2. **"锚点 vs 禁令"还是"信息量稀释了约束"**：两种解释都能覆盖本次实验数据，需要新的实验设计来区分。

3. **mitmproxy抓包方案**：技术可行，依赖Trae是否开放自定义endpoint。如果落地，可以拿到产品层PE原文——但走了一圈，发现"知道自己被注入了什么"对正文质量的直接影响有限。更重要的事情还是：**读原文、打包上下文、写好单章**。

4. **这个"上下文注入工序"是应该作为skill的一个新步骤，还是作为写作环境的一个内置能力**：如果是前者，它需要在每章写作前执行一次Read+Read+Read+Write的流程。如果是后者，它应该是写作工具自身就带的功能（"开始写新章前，自动打包上下文"）。
