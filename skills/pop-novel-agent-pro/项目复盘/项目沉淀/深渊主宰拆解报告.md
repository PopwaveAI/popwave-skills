# 深渊主宰 · 前三章拆解报告

> 测试目的：验证「事实骨架 + 实体变量系统」能否有效拆解一部已完结的参考小说
> 拆解方法：草案 v0.1 规则——读原文 → 识别实体 → 建YAML卡 → 写事实骨架
> 拆解章节：CH001-CH003
> 拆解日期：2026-05-23

---

## 一、测试结论

**体系可用。** 事实骨架 + 实体变量卡确实能拆出小说的结构化数据。但暴露出几个需要调整的问题。

## 三、事实骨架的关键发现

### 3.1 字数分配合理

每章事实骨架 ≈ 1,100-1,800字，正文原文 ≈ 2,500-3,600字。描述层占比约55%-65%。

### 3.2 实体变化驱动明确

每章都有清晰的实体状态变化清单。CH003最高（7个实体的状态变了），CH001最低（3个）。这符合"黄金三章"的节奏——第一章是建立世界，变化少但信息密度高。

### 3.3 新概念数普遍偏高

| 章节 | 新概念数 | 评价 |
|:----|:-------:|:-----|
| CH001 | 4个 | 勉强过线 |
| CH002 | 5个 | 超标（数据流/属性面板/职业体系/传奇技能/灵魂复活） |
| CH003 | 6个 | 显著超标（但部分为CH002的延续解释） |

**症状**：穿越+游戏面板+西方奇幻的设定组合，首三章自然有大量世界观要交代。这不一定是问题——深渊主宰的原著成绩证明了读者能消化——但需要在QC中区分"必要概念"和"冗余概念"。

### 3.4 视角转换模式

原著采用了 **非常规的视角切换**：
- CH001：全章 {薇薇安} 视角（营造无助感/代入感）
- CH002：全章 {索伦} 视角（穿越+觉醒）
- CH003：{索伦} 主视角 + {薇薇安} 局部视角

**效果**：CH001从弱势方切入，让读者先建立对薇薇安的共情，再引出索伦的穿越+金手指。这是一个高级的设计手法——比直接从索伦视角开场更有效。

## 四、实体卡设计的反思

### 4.1 合并 vs 拆分

当前把多个小实体合并在一个文件里（如 `巴拉斯和卡诺波.yaml`、`盗贼技能组.yaml`、`琥珀城-贫民区-墓园区.yaml`），在草案阶段图省事。

但**正式用应该拆开**——因为每个实体的状态变化是独立的。巴拉斯死了卡诺波还活着（虽然差距不大），潜行技能20点但没变过而万能巧手有"封印→解封"的完整变化线。文件合并会导致状态追踪耦合。

### 4.2 description_palette 的有效性

给索伦写了 description_palette：
```yaml
primary: "昏暗/冷色"
texture: "泥泞/粗糙"
motion: "迅速/无声/致命"
metaphor_domain: ["阴影", "匕首", "冷雨"]
```

给薇薇安写了截然不同的一套：
```yaml
primary: "温暖/脆弱"
metaphor_domain: ["小动物", "泪珠", "烛光"]
```

这两套如果用于Pass 2的描述渲染，能确保同一个场景中不同角色的视角产生不同的感官描写——这是锚定书想实现但一直做不到的。

### 4.3 游戏数值应该字段化

深渊主宰有具体数值（力量12/盗窃35/平民5→10级）。当前我把这些放在 YAML 的 `attribute_base` 和 `skills.mastered` 字段里——但如果有升级/加点剧情，这些数值会频繁变动。**需要建立一个"数值变更历史"字段**，否则追不出属性增长的轨迹。

## 五、对草案提出的修改建议

| 问题 | 建议修改 |
|:-----|:---------|
| **单个文件多实体** | 强制一实体一文件。不支持合并文件 |
| **数值无变更历史** | state 中加 `attribute_history: [{chapter: 002, change: "平民5→10"}]` |
| **description_palette 有效性未验证** | 下一步测试：用索伦的 palette 重写 CH001 的第一段场景渲染，看是否符合原著文风 |
| **新概念数QC规则需细化** | 区分"必须交代的世界观设定"和"可延后的概念"。前者不影响阅读门槛，后者才需要警告 |
| **实体引用粒度** | 一个事实骨架至少引用8-15次实体标记。管线需要高效的批量加载机制 |
