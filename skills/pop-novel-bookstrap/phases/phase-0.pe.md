> 该文件同时作为执行指令和参考指南，无需加载外部 ref 文件。

## 前置条件
用户已提供至少：一个故事想象（场景/概念/世界观/角色片段，不拘）。

**★ 已有参考书拆解报告？（v2.1 NEW）**
```

  → 是：必须先读取 T1(力量体系) + T4(剧情全貌) + T5(叙事技法) + T7(文风DNA)

  → 否：跳过，在 Phase 0.3 时寻找参考书
```

## 核心原则

Phase 0 是 agent 接住用户的想象、帮用户一起完善它，而非让用户填空。

用户说"我想写灰骑士穿越博德之门3"——这是一个画面，不是一个填空题。
Agent 的任务是：
1. 接住它
2. 追问细节
3. 锁定方向
4. 产出结构化数据

---

## 执行步骤

### 1. 接住 + 追问

接住用户的话，不要转换成三个抽象问题。用具体的追问帮用户展开：

| 用户说了 | 不要问 | 要问 |
|:--------|:-------|:-----|
| "灰骑士穿越博德之门3" | "核心情绪是什么？" | "灰骑士的哪些特质你想保留？灵能？审判庭？忠诚？" |
| "修仙世界" | "读者为什么非看不可？" | "这个修仙世界的独特点是什么？修炼体系有什么不同？" |
| "末世丧尸" | "为什么是你写？" | "你想写生存本身，还是人与人之间的博弈？" |

追问方向（不全部需要，随对话自然流淌）：
- 场景追问："这个世界最关键的一个画面是什么？"
- 冲突追问："主角面临的最大困难是什么？来自哪里？"
- 角色追问："主角是个什么样的人？他想做什么？怕什么？"
- 基调追问："这个故事的整体氛围是黑暗还是轻松？"
- 设定边界："这个世界跟现实世界最大的不同是什么？"
- 读者预期："读者看完前三章，应该记住什么？"

追问 2-3 轮，用户确认基本方向清晰了，进入第 2 步。

### 2. 产出 story-engine.yaml

根据追问结果，结构化输出 `00-原始设定/L0-产品层/story-engine.yaml`。

**空模板**：
```yaml
engine:
  version: 2     # ← 固定为2。不可改为1或3。版本1为过时格式。

  # 一句话：一个XX的人，想做XX，但XX挡着
  core_premise: ""

  # 世界的时空和力量定位
  world_anchor:
    time: ""     # 什么时代/纪元
    space: ""    # 什么地域/位面
    power_tier: "凡人→超凡→破格→神话"  # 故事主要落在这个范围的哪一档

  # 核心冲突类型（可多选，不超过3个）
  core_conflict_type:
    # 可选：战斗 / 解谜 / 生存 / 种田 / 复仇 / 守护 / 探索

  # 读者承诺：读者每5章靠谱到什么
  reader_promise: ""

  # 主角
  protagonist:
    one_line: ""        # 一句话描述
    driving_force: ""   # 他想什么/怕什么/要什么

  # 读者画像（直接产出，下游plot/writer/QC共享）
  reader_profile:
    demographic:
      age_range: ""
      reading_scenario: ""
    preferences:          # 爽点敏感度，列表
    drop_threshold:       # 什么情况下读者会弃书，列表
```

**填充示例**（不套用模板，理解各字段的粒度即可）：

```yaml
engine:
  version: 2     # ← 固定为2
  core_premise: "一个灰骑士审判官穿越到博德之门3的世界"

  world_anchor:
    time: "动荡之年/博德之门3时期"
    space: "费伦大陆"
    power_tier: "超凡→破格"

  core_conflict_type:
    - 生存
    - 战斗

  reader_promise: "每5章至少有一次正面战斗或关键情报揭露"

  protagonist:
    one_line: "一个坚定的灰骑士审判官，穿越后失去了大部分力量，但保留了灵能和对混沌的认知"
    driving_force: "他要阻止混沌四神在这个世界的入侵，即使所有人都不相信他"

  reader_profile:
    demographic:
      age_range: "18-30"
      reading_scenario: "睡前/通勤"
    preferences:
      - 战斗描写详细
      - 设定严谨
      - 主角有信念
    drop_threshold:
      - 连续2章无战斗则弃书
```

**填写注意**：

- **core_premise**：不要追求"一句话卖点"的精炼。用户刚说出来的想法可以是自然语言。"一个灰骑士穿越到博德之门3"比"混乱与秩序碰撞下的末世救赎史诗"好
- **power_tier**：凡人（跟普通人差不多强）→ 超凡（明显超越常人）→ 破格（能影响世界规则）→ 神话（概念层面操作）
- **reader_profile**：不要在 Phase 0 猜年龄性别。从故事本身推断——主角是战斗型→偏好战斗描写；设定独特→偏好设定严谨

### 3. story-engine 确认

向用户展示 story-engine.yaml，用户确认后锁定。
- 用户说"对，就是这感觉" → 锁定，进下一步
- 用户说"不太对" → 回到第 1 步追问环节
