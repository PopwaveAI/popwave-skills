# Phase 5：数值体系模板升级

## 这份文档存在的意义

有战力升级系统的小说需要一个共同的问题：**力量体系不是"作者觉得应该差不多"——是需要量化设计的。**

数值体系存在的意义是：回答"主角在第 X 章时到底有多强？能打什么级别的敌人？"这个问题。不是真的让小说充满数字——是在作者（和 agent）的脑子里建立一个清晰的**战力参照系**，让战力增长有迹可循。

**没有数值体系的后果**：前期主角打刀虫艰难，50 章后打刀虫依然艰难——读者感觉不到成长。或者前期一刀秒小怪，100 章后一刀秒 Boss——读者感觉不到挑战。

---

## 必须回答的问题

1. **主角在每一卷每章的战力上限是什么？** — 不是一个大概范围，是"第 X 章他能打败什么级别的敌人、不能打败什么"。
2. **同级别的敌人之间有什么区别？** — 不是"所有 3 星敌人都一样强"——同级别也有类型差异（速度型/力量型/法术型）。
3. **每次升级读者的"体感"是什么？** — 升级不只是在文档里加一行"战力+50%"，是在正文中有一个**明确的画面**告诉读者"他变强了"。

---

## 执行步骤

### combat_capability 模板
定义角色在每个段位的战斗能力边界。
- 等级名、突破瓶颈机制
- 能力类型（物理/法术等）
- 攻击范围（近战/远程/范围）
- 弱点或冷却条件

### monster_rank_map 模板
将每个敌人类型映射到玩家的段位体系。
- 怪物等级与主角战力的对应关系
- Boss 设计规范

### act_rank_schedule 模板
规划每卷结束时主角应达到的段位。
- 每卷等级节奏安排
- 关键里程碑章节
- 段位跨度合理（不能第一幕就从凡人到半神）

详细模板见 `templates/L1-数值体系-template.md`

**产出**：
- `00-总控/数值体系/combat_capability.yaml`
- `00-总控/数值体系/monster_rank_map.yaml`
- `00-总控/数值体系/act_rank_schedule.yaml`
