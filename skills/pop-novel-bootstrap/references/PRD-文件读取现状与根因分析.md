## 二、根因分析

### 根因 1：文件读取完全依赖 tool call，无自动加载机制

agent 读取文件的唯一方式是显式执行 `read` 或 `Get-Content` 工具调用。没有任何机制在特定阶段（如"写下一章正文前"）**自动触发**一系列文件的读取。

**现状 vs 预期对比：**

| 阶段 | agent 实际读的内容 | 期望读的内容 |
|:-----|:-----------------|:-------------|
| 写 ch020 正文前 | 上一章(ch019)结尾 + chapter-state.yaml | 上一章结尾 + act-01.yaml + L1-04 金手指 + constitution.yaml + global-summary |
| 写 ch005 正文前 | 上一章(ch004)结尾 | 上一章结尾 + act-01.yaml + L1-01 世界观 + 锚定章库 |

### 根因 2：上下文窗口替代了文件读取

LLM 的上下文窗口中保留了大量对话历史，agent 倾向于**从对话历史中"回忆"设定**，而非从磁盘读取最新版本。当对话轮次增多、上下文窗口被新内容覆盖时，旧的设定记忆就丢失了。

**典型表现：**
- 用户发现 ch02 数据和 ch03 对不上
- ch03 正文和 act-01.yaml 的大纲描述不一致
- 这是因为写 ch03 时 agent 没读 act-01.yaml，凭"我记得大纲是这么写的"在工作

### 根因 3：project.yaml 索引未被用于自动加载

`project.yaml` 中维护了一个完整的文件索引（paths 段），列出了所有关键文件的路径。但 agent **只把它当 "元信息"看**，没有机制说"既然 project.yaml 索引了这些文件，我每次写正文前就把它们全读一遍"。

### 根因 4：管线被设计为"按需读取"，但 agent 不知道哪些"需要读"

bootstrap 的 phase 执行文件（.pe.md）写了"加载参考文件"，但在 writer 的写作流程中，**没有任何文件告诉 agent"每次写正文前必须读 L1 设定 + act-yaml + constitution"**。agent 的默认行为是"能少读就少读，上下文里有的就不读新的"。

## 三、影响评估

### 3.1 直接后果

| 问题 | 严重性 | 发生频率 | 影响 |
|:-----|:------:|:--------:|:-----|
| 章节与大纲不一致 | P1 | 中 | 读者发现剧情走向和期待不符 |
| 跨章数据断裂 | P1 | 中 | 前后章节的数值/状态对不上 |
| 设定前后矛盾 | P2 | 低 | 中期写回头发现设定不一致 |
| 金手指约束被遗忘 | P2 | 中 | 主角用金手指没付出应有代价 |
| 宪法原则被违反 | P1 | 低 | 作者承诺读者的"永不"被打破 |

### 3.2 对两个项目的具体影响

| 项目 | 已验证的问题 |
|:-----|:------------|
| 新书-诸天万界 | ch02 和 ch03 数据对不上；ch03 和大纲描述不一致；前三章爽点不够被用户要求重写 |
| 北帝镇诡录 | 未暴露严重问题（21章正文一致性较好），但无机制保障后续不出现 |

## 四、问题定义

> **在 paopao/openclaw 环境下，pop agent 每次写正文前不会自动加载设定/大纲/宪法等关键文件，依赖对话历史回忆设定，导致跨章数据断裂、正文与大纲要一致性无法保障。**

这个问题的根因是**文件读取机制的缺失**，不是某个 skill 编写质量的问题——无论 writer 的 SKILL.md 写得再好，如果 agent 实际执行时不去读该读的文件，设计上的"管线必读"就是空话。
