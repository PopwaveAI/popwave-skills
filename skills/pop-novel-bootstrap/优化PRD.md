# Bootstrap 优化 PRD

> 版本：v1.0 | 2026-06-05
>
> 本文档系统性记录 pop-novel-bootstrap skill 当前的所有问题、根因分析及优化方向。
> 后续每次修改前参照此 PRD，确保改动有目标、不跑偏。

---

## 一、整体定位问题

### 1.1 bootstrap 到底产出什么？给谁用？

| 当前状态 | 问题 |
|:---------|:-----|
| 产出 PRD.md + L1 六件套 + 宪法 + 数值体系 + 卷级展开 + Boss检查 + ... | 产出物过多，部分越界到 plot skill |
| 不知道下游管线怎么消费这些产出 | PRD.md 写出来就封档，plot/writer/QC 没人读 |
| Forward 和 Reverse 两种模式的产出格式不一致 | 同一本书走 forward 和 reverse 得到的数据结构不同 |

**根因**：bootstrap 没有一个明确的问题定义——"什么产出物是后面管线必须的、什么是一次性就没人看的"。

### 1.2 Phase 2（L2 卷级展开）和 Phase 6（硬检查）越界

| Phase | 做的事 | 应该是谁的 |
|:------|:-------|:----------|
| Phase 2 | 卷01 的 6 个文件设计（地理/势力/角色/剧情/装备/爽点） | **plot skill** |
| Phase 6 | Boss 战设计（铺垫≥10章/可量化代价/叙事推进） | **plot skill** |

**根因**：bootstrap 做"开书"，plot 做"剧情架构"——但 bootstrap 把自己的终点画到了 plot 的地盘上。边界没定过，谁做都一样。

### 1.3 Reverse 管线格式与 Forward 不共通

事件日志的字段（情绪账户/文风指纹/信息不对称）在 forward 中无人使用。如果一本书前 200 章 reverse 后 200 章 forward，两种数据无法融合。

**根因**：reverse 的数据格式是为"提取分析"设计的，不是为"复用"设计的。两种模式各自设计，没考虑互操作性。

## 二、Phase 0：灵魂对齐（最严重的环节）

### 2.1 核心问题：产出物没有下游消费方

| 当前设计 | 问题 |
|:---------|:-----|
| 灵魂三问（为什么是我写/读者为什么非看不可/核心情绪） | 三问是作者**向内自省**，不是**结构产出**。答案没有任何下游 phase 读取 |
| PRD 六要素（书名/卖点/画像/体验/差异/风险） | 产出后无人读取。plot 不查，writer 不查，QC 不对着查 |
| 压力测试（电梯/长篇/平台/差异） | 全部是主观判断。谁来判断"能否撑 200 万字"？|

**根因**：Phase 0 把自己定位成"商业文档编写"，不是"故事引擎设计"。它产出的是一份给人看的文档，不是给下游管线用的数据。

### 2.2 灵魂三问不对——用户应该说什么？

用户说"我想写灰骑士穿越到博德之门 3"。这不是"书名+核心爽点"——这是一个**具体的想象画面**。

当前流程要求用户把它拆成三个问题填空 → 劝退。正确流程应该是 agent 接住这个想象，**帮用户一起完善它**。

### 2.3 PRD 本身是个死文档

一行"一句话卖点"——但 plot skill 做幕纲设计时不会来读这句模糊字符串。L1 设定写完了也没人拿着 PRD 去对。

## 三、Phase 0.3：参考书甄别

### 3.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 搜索同类 → 拆解 → 差异化三列表 | 没有"好的差异化分析长什么样"的示例 |
| "调用 pop-novel-deconstructor" | 这个 skill 在磁盘上是否存在？如果不存在就是死路径 |
| 差异化三列（借鉴/偏离/空白） | 没有模板、没有粒度标准。一个点写到什么程度算够了？|
| 与 Phase 0 无关联 | "差异化"需要跟故事核心驱动对齐，phase 文件没提|

## 四、Phase 0.4：金手指设计

### 4.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 5 个必答问题 + 四级评估 + 番茄视角检查 | **最扎实的 phase——但有短板** |
| 没有 before/after 示例 | 好金手指长什么样？坏金手指怎么识别？只有标准没有参照 |

**好评**：四级评估（L1 核心功能→L2 约束→L3 成长→L4 叙事功能）有递进感。番茄视角前 3/10/50 章检查是少数可横竖检验的标准。

---

## 五、Phase 0.5：跨域素材聚合

### 5.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 卖点词拆解 → 跨域搜索 → 素材蒸馏 → 设定转化 | 流程顺序合理，但执行层面量化是数字游戏 |
| 量化门槛：领域≥4 / 种子≥8 / L1转化≥3 / 字≥1500 | 8 个弱种子不如 2 个强种子。数字达标不等于质量达标 |
| "标注来源/关联度/可转化性" | 字段没有格式模板。一句话"可转化性高"跟一段分析差距很大 |

**根因**：把"跨域"等同于"搜索多个关键词"。真正的跨域素材聚合应该是在不同领域中找到**可迁移的结构**，不是凑够 4 个领域的搜索记录。

## 六、Phase 1：L1 设定层推演（最关键·最薄）

### 6.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| "依次产出六个文件"——但每份文件一句定义 | 15 字定义一个文件。底层逻辑包含什么没说 |
| 六件套分类的合理性未验证 | "世界观底层逻辑/表层规则/种族/金手指/势力/物品"——这个分类从哪来？没人质疑过 |
| 没有示例 | 没有"好的世界观底层逻辑长什么样"的参考 |
| 01 和 02 的边界不清 | "基本法则"和"公开规则"的边界在哪？不同 LLM 跑同一本小说可能产出完全不同的结构 |
| "这个设定服务 PRD 里的什么爽点？"逻辑对但不可执行 | PRD 里是模糊句子字符串，LLM 判断不了"服务没服务" |

**根因**：L1 分类是拍脑袋没经过验证的。需要拆书反推来验证。

### 6.2 如何验证

选 2-3 本设定有特色的书（如诡秘之主/凡人修仙传），按现有六件套框架填：
- ✅ 填得进去 → 分类合理，只是模板太薄
- ❌ 填不进去或大量重复 → 分类本身需要重新设计

## 七、Phase 1.2：L1 深度展开

### 7.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 字数门槛（01≥2000/6维度） | "6 个维度"没说是什么维度 |
| 展开节点（定义+表现形式+剧情影响+画面锚点） | 方向正确但缺示例。什么样的记忆点算画面锚点？|
| "素材注入"——从跨域素材提取种子嵌入 | 跨域素材本身可能就有质量问题，嵌套依赖 |

## 八、Phase 1.3：L1 交叉关联

### 8.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 逐对检查 14 对关联 | 数字游戏。检查了 14 对不代表 14 对真的一致 |
| 交叉引用格式 `【关联：XX】——...` | 反模式。在每篇 L1 文件里塞对其他文件的引用标注，污染了文件本身的独立可读性 |
| "冲突>2处退回 Phase 0" | 一刀切。有些冲突改一下就好，不需要退回重来 |

---

## 九、Phase 1.5：世界稳定性检验

### 9.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| 三个自问环节（一致/闭环/可扩展） | **全线最弱的 phase**。没有标准、没有检查项、没有模板 |
| "每个设定可回溯至 PRD" | 怎么回溯？回溯不到就算不稳定？没有操作定义 |
| "写死自己的设定陷阱" | 什么算陷阱？怎么预测？没有常见陷阱清单 |
| 产出为 `world-stability-check.md` | 没有输出模板。对比 plot 的节奏自检有 7+3+3 条可 check 项 |

**对比**：plot skill 的节奏自检表格化、可勾选。Phase 1.5 是 3 句天问。

## 十、Phase 3：项目骨架创建（最好的一个）

### 10.1 评价

| 维度 | 评价 |
|:-----|:------|
| project.yaml schema | ✅ 有具体 yaml 结构+字段说明 |
| chapter-state.yaml schema | ✅ 同上 |
| constitution.yaml 模板 | ✅ 有框架 |
| 角色卡标准（主角满配/配角中配/龙套不建） | ✅ 有分层标准 |

**问题**：
- 7 个产出物塞在一个 step（目录/配置/宪法/角色卡/数据库/追踪面板/章状态），没有子步骤引导
- constitution.yaml 的模板太浅——"核心情绪基调：[一句话]"——一句话能概括吗？

## 十一、Phase 4：reader_profile 嵌入

### 11.1 核心问题

| 当前设计 | 问题 |
|:---------|:-----|
| "从 PRD 提取读者画像 → 嵌入 project.yaml" | reader_profile 已经在 Phase 0 的 PRD 里写过了。绕了一圈 |

**根因**：Phase 0 产出的是"给人读的 PRD.md"，是模糊字符串。Phase 4 需要结构化 reader_profile。**如果 Phase 0 直接产出结构化数据就不用绕路。**

---

## 十二、Phase 5：数值体系

### 12.1 评价

方向对。`力量体系大全.md` 是 bootstrap 最好的参考文档。

**问题**：三个 yaml 文件没有字段模板，只有名字。LLM 不知道每字段的填写粒度。

## 十三、Reverse 管线（r1-r6）

### 13.1 相对是最好的部分

r1 的事件日志格式有详细字段模板（情绪账户/文风指纹/信息不对称/跨批伏笔追踪），是整条管线中**少数 LLM 拿到就能产出结构一致内容的 phase**。

### 13.2 核心问题

| 问题 | 说明 |
|:-----|:------|
| 事件日志格式与 forward 不共通 | 如果一本书 200 章 reverse + 200 章 forward，两套数据无法合并 |
| r2-r5 过于简化 | r2~r5 都是重复的"从事件日志提取 → 用户确认"模式。每个 10 行 |

---

## 十四、横向问题

### 14.1 量化标准全是数字游戏

| 数字 | 问题 |
|:-----|:------|
| 素材种子≥8 | 8 个弱种子不如 2 个强种子 |
| 字数≥2000 | 2000 字废话不如 500 字精华 |
| 14 对关联检查 | 检查了是数字，检查质量是另一回事 |
| 领域≥4 | 凑 4 个领域名比找到可迁移结构容易 |

**根因**：用数量代替质量。因为质量不好定义，所以用容易 check 的数字凑数。

### 14.2 references/ 混入无关文件

`PRD-文件读取现状与根因分析.md`——内部审计报告（读取率/文件统计），跟开书毫无关系。

### 14.3 缺少"失败案例"参考

去 AI 味 skill 每条规则有 before/after 对比。bootstrap 的 phase 文件几乎全部是"做什么"的说明，没有"坏的长什么样、对的应该长什么样"的对比。

## 十五、优化方向汇总

| # | 方向 | 涉及 phase | 优先级 |
|:-:|:-----|:-----------|:------:|
| A | Phase 0 从"灵魂三问+PRD"改为"故事完善+结构化引擎数据" | P0 | P0 |
| B | 删掉 Phase 2（交 plot）和 Phase 6（交 plot） | P2, P6 | P0 |
| C | 拆书反推验证/优化 L1 六件套分类 | P1 | P0 |
| D | Phase 1-1.5 全部加具体模板+示例（before/after） | P1, P1.2, P1.5 | P1 |
| E | 量化标准从数字游戏改为质量导向 | P0.3, P0.5, P1.2, P1.3 | P1 |
| F | Phase 0.3 差异化三列表加模板+示例 | P0.3 | P1 |
| G | Phase 0.4 金手指加 before/after 示例 | P0.4 | P2 |
| H | Phase 0 直接产出结构化 reader_profile | P0, P4 | P1 |
| I | Phase 1.3 交叉引用从"注水格式"改为"实际交叉检查" | P1.3 | P2 |
| J | Reverse 事件日志格式与 forward 对齐 | r1 | P2 |
| K | 删除 references/ 中的无关文件 | references/ | P2 |
| L | Phase 3 constitution.yaml 加深度模板 | P3 | P2 |

## 十六、建议试点顺序

```
Phase 0 重构（方向 A）
  ↓
拆书反推验证 L1 分类（方向 C）
  ↓
确认分类 → Phase 1-1.5 模板化（方向 D）
  ↓
删除 Phase 2 + Phase 6 + 优化余下量化标准（方向 B + E）
  ↓
差异化模板 + reader_profile 直出 + 剩余优化（F → H → I~L）
```
