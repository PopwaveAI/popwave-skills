# Step 2：事件链设计

> 管线: pop-novel-chapter-design v1.2
> 模板: `templates/fact-skeleton.md`

---

## 目的

将本章拆解为故事节拍序列。事件链定义「这场戏发生了什么、为什么发生」——不定义「怎么演、什么质感」。

**Design 管的是故事节拍层，Render 管的是渲染节拍层。Design 不要降维到 Render 的层次去代劳。**

---

## Design 事件 vs Render 微节拍

```
                 Design 事件（本文件设计）            Render 微节拍（Render 自主生成）
                ─────────────────────────          ──────────────────────────────
                
 粒度            故事节拍：有叙事意义的剧情节点         渲染节拍：句子级的动作/感觉/观察切换
                "这章发生了什么，为什么"                "这几句话怎么写"
                
 每章数量         10-20 个                              40-60 个（Design 不关心）
                
 示例             "魔能超导脉冲炮命中恶魔群               Render 拿到这个事件后，自己决定：
                 ——50米蒸发，地面玻璃化"                  从谁的角度看 →
                                                         用什么感官切入 →
                 "索伦开启神灵形态——百丈金色巨人             写几段 →
                  降临战场"                               怎么收束这个场景
                
 语气             描述性：发生了什么、结果是什么             执行性：具体的词、句、节奏、感官
                
 情绪标注          有——每个事件有情绪目标 ←                Render 据此选句式和节奏
                
 信息释放          标注 info_release item_id ←             Render 按 release_method 执行
```

## 事件密度基线

### 核心理念：不是定死数字，是定「情绪密度 → 推导事件数」

同一类型的两章战斗，事件数可以差一倍——暗杀渗透密度高但每个事件轻，城市攻防战密度低但每个事件重。**数字不决定质量，情绪密度决定质量。**

### 情绪密度基准（来自番茄编辑 · 黄金看点公式）

| 指标 | 要求 | 映射到 Design |
|:-----|:-----|:-------------|
| **300 字 = 1 个情绪节点** | 读者每 300 字经历一次情绪波动，超过则感觉平淡 | Design 每个事件未必都是情绪节点，但「情绪节点总数 ÷ 章字数」必须 ≥ 1/300 |
| **500 字 = 1 个钩子** | 读者每 500 字遇到一个「然后呢？」 | Design 事件中至少有 N 个带有悬念/钩子属性，N ≥ 章字数÷500 |
| **1 章 = 1 个悬念** | 章末必须有让读者「必须点下一章」的钩子 | Design 最后一个事件必须是钩子事件 |

### 从情绪密度反推 Design 事件数

```
第 1 步：确定本章需要的情绪节点数
  情绪节点数 ≥ 章预期字数 ÷ 300

第 2 步：确定本章需要的 Design 事件数
  每个 Design 事件必须至少是一个「情绪节点」或一个「悬念节点」
  → Design 事件数 ≥ max(情绪节点数, 悬念节点数)
  → 即：Design 事件数 ≥ 章预期字数 ÷ 300

第 3 步：根据场景复杂度上浮
  单线叙事（暗杀/独白/追踪）→ 基线 × 1.0-1.3（情绪节点约等于事件数）
  双线叙事（对话+动作/探索+判断）→ 基线 × 1.3-1.6
  多线叙事（城市战/军团战/多视角）→ 基线 × 1.6-2.5（大量事件是纯叙事推进，非情绪节点）

第 4 步：下限校验
  Design 事件数 ≥ 6（低于此值说明一章没有足够的剧情推进——即使是纯过渡章）
```

### 不同类型章节的参考范围

| 章节类型 | 情绪节点数（≥） | Design 事件参考范围 | 说明 |
|:---------|:------------:|:----------------:|:-----|
| **常规章** | 2500÷300≈8 | **8-16** | 单线叙事为主 |
| **战斗章·单线** | 2500÷300≈8 | **12-20** | 暗杀/决斗/追踪——动作节拍多但情绪节点 = 动作节点 |
| **战斗章·多线** | 3000÷300≈10 | **16-25** | 城市战/军团战——每条线都需要独立的结构推进 |
| **对话章** | 2500÷300≈8 | **8-14** | 每次关键信息交换或情绪摊牌是一个事件 |
| **探索章** | 2500÷300≈8 | **8-16** | 每次信息跃迁是一个事件 |
| **高潮章** | 3000÷300≈10 | **15-25** | 决战/大爽点，多线汇聚 |
| **过渡章** | 2000÷300≈6 | **6-10** | 衔接为主 |

> **关键原则**：`Design 事件数`是下限，不是上限。多一个事件不扣分——只要它是叙事推进需要的结构点。**上限由「事件链流暢度」决定——事件之间因果链断裂、情绪反复跳极时，就是太多了。**

## 跨章剧情弧线识别（★ v1.2）

### 什么时候一个剧情弧线需要跨章

如果本章的核心剧情在给定字数内无法完成一个完整的「起→承→转→合」，**不要压缩事件**，而是：

1. **判定为跨章弧线** → 标注 `arc_span: [N, M]`
2. **为每章标注弧内位置** → `决战①` / `决战②` / `决战③`
3. **为每章标注情绪分工** → 铺垫→交战→高潮收束

### 跨章弧线的事件分配

1. **对整个弧线设计完整的故事节拍链**（可能 30-60 个 Design 事件）
2. **按章节字数预算切分**：第一刀在第 N 章、中段在 N+1 章、终局在 N+2 章
3. **每个切分点找一个情绪落点**——章末事件 = 该章的情绪止点

## 执行

### 1. 判断：本章是否属于跨章弧线

```
检查 act-XX.yaml 中本章及相邻章的章节标题和 emotional_goal：
  - 标题连续且相关（如 "决战①" "决战②" "决战③"）→ 跨章弧线
  - 情绪目标连续递进（紧张→临界→释放）→ 跨章弧线
  - 本章字数预算 > 3500，剧情复杂度 > 大纲预估的一章量 → 跨章弧线
```

### 2. 设计事件链

每个 Design 事件是一个**有叙事意义的剧情节点**，回答五个问题：

| 问题 | 约束 |
|:-----|:-----|
| **在哪？** | 必须可追溯到 act-XX-地图.md |
| **谁？** | 必须可追溯到 act-XX-人物.md |
| **发生了什么？** | 用描述性语言写出结果——"魔能炮命中→50米蒸发"。不写具体感官/动作细节/微节拍拆解 |
| **为什么发生？** | 这个事件在叙事中的作用——推进情节/释放设定信息/建立情绪/展示角色 |
| **读者感受到什么？** | 情绪目标——指导 Render 的句式和节奏选择 |
| **估计字数** | 100-250 字（Render 在此范围内自主拆 3-5 个微节拍） |

### 3. Design 事件示例（深渊主宰·第十六章 "摩多城攻防战"，3400 字）

```
事件1: 十二月初冬·索伦防错方向 — 他一直在防备南海岸，但奥喀斯真正的目标是摩多城 — 铺垫+反差 — 120
事件2: 警报拉响 — 奥术塔探测到南部20公里异常空间波动→一级备战 — 紧张升级 — 100
事件3: 水晶球投射画面 — 暗红色空间门撕裂天际，恶魔/亡灵如潮水涌出 — 恐惧 — 150
事件4: 黑土镇陷落 — 六臂蛇魔三分钟攻破守军，数百平民被屠 — 惨烈+压迫 — 160
事件5: 索伦愤怒 — 奥喀斯进攻摩多城=对他根基的直接挑衅，不死不休 — 愤怒→决意 — 150
事件6: 索伦接管奥术塔最高权限 — 下达一系列战斗命令(科琳娜→熔炉/歌莉娅→升空/塔灵→迷锁/爱丽丝→全部魔像) — 指挥官模式 — 180
事件7: 魔像军团激活 — 密瑟能核全力运转，地下室中6000+魔像同时睁开眼睛 — 宏大 — 160
事件8: 全城动员 — 广播征召所有战斗单位(城防军1-7/牧师团/巫师/飞龙骑士/海军舰队) — 史诗感 — 160
事件9: 幽灵船参战 — 摩多港海水左右分开，亡语者号破浪而出，浮空而起 — 视觉奇观 — 140
事件10: 魔能超导脉冲炮 — 充能锁定巫妖群→发射→50米蒸发，地面玻璃化 — 爽点释放 — 180
事件11: 索伦开启神灵形态 — 神力汇聚，百丈金色巨人浮现神殿上空 — 神级视觉 — 150
事件12: 神灵介入战场 — 一步跨出20公里，一拳砸下→神力冲击波×3，亡灵割麦般成片倒地 — 力量碾压 — 200
事件13: 海军/陆军/飞龙骑士团三线出击 — 掩护平民后撤，遏制亡灵推进 — 转入正面战 — 160
事件14: 代价 — 三个城镇受袭，5000+平民死亡。索伦消耗大量神力 — 沉重 — 120
事件15: 钩子 — 亡灵还在涌入，索伦不敢关空间门——一个强大恶魔领主的气息笼罩战场。奥喀斯本人已降临摩多城 — 悬念 — 100
```

**15 个 Design 事件 × ~153 字均值 = ~2290 字。** Render 在 153 字/事件的空间内，自主生成上下文氛围、角色微反应、环境细节、动作拆分——这 15 个事件到 Render 手里，自然长出 40-60 个渲染微节拍。

### 4. 这个事件链告诉 Render 什么，又不告诉什么

```
✅ 告诉了：
  - 这场戏的结果（"脉冲炮命中→50米蒸发"）
  - 情绪曲线（铺垫→紧张→恐惧→愤怒→宏大→爽点→代价→悬念）
  - 场景变换（奥术塔→黑土镇→地下室→摩多港→神殿→战场）
  - 信息释放节点

❌ 没告诉（留给 Render 自己决定）：
  - "炮口先从哪个角度亮起来的"
  - "索伦说命令时的语气和措辞"
  - "魔像睁眼的机械音是什么样的"
  - "幽灵船出水时海水溅到了谁身上"
  - "百丈金色巨人是先看到拳头还是先看到身形"
```

## 红线

- ❌ Design 事件写成渲染级微节拍 → 退回（"刺出第一刀"→"他侧身闪过"→"血溅到袖口"是 Render 的活，不是 Design 的）
- ❌ Design 事件写成模糊剧情 → 退回（"发生了激烈的战斗"——说了等于没说）
- ❌ 战斗章 ≤ 8 事件 → 退回（颗粒太大，Render 会收到过于模糊的创作需求）
- ❌ 普通章 ≤ 6 事件 → 退回
- ❌ 跨章弧线未标注 → 退回
- ❌ 跨章弧线的章末事件不是自然的情绪落点 → 警告
