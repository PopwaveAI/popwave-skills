# 续写适配 — 从已有正文到标准输入（v3.3）

> 版本：**v2.0** · 2026-05-25
>
> **v2.0 升级（v3.3 对齐）**：
> 1. Phase 2 L0 新增 reader_profile 提取 → 写入 project.yaml
> 2. Phase 3 L1 新增 07-数值体系.md（combat_capability / monster_rank_map / act_rank_schedule）
> 3. 事件日志路径从 04-事件日志/ 改为 01-写作资产/，对齐 v3.3 写作资产目录
> 4. 交接验证新增 reader_profile + 数值体系 + 经验日志检查项
> 5. 下一步指引新增 QC Agent + autocheck
>
> 职责：把"已有 N 章正文但无标准设计层"的项目，适配进 bootstrap → plot → writer 的写作管线。
>
> **核心设计**：不改任何已有 skill。本模块产出的文件路径/格式与 bootstrap/constitution/plot 的输入规范完全一致。下游 skill 感知不到"这是续写项目还是新项目"。

---

## 使用场景

用户说：
- "我有一本写了 XXX 章的小说，想续写"
- "这本书有草稿/正文但没做过 L1，帮我初始化"
- "把已有正文转成 novel-agent 标准格式"

## 不适用的场景

- 从零开书 → 走 bootstrap
- 已有标准设计层（L0/L1/L2/宪法齐全）→ 不需要本模块
- 只有想法没有正文 → 走 bootstrap

## 六阶段执行流程

### Phase 0：环境检测（每次启动必须执行）

读项目目录，检查已有文件。

**检查清单**：

| 检查项 | 标准路径 | 说明 |
|-------|---------|------|
| 正文章节 | `03-正文/` 或用户指定路径 | 计数 + 范围 |
| 已有事实骨架 | `01-写作资产/事实骨架/` | 有/无 |
| 全局摘要 | `01-写作资产/global-summary.md` | 有/无 |
| 经验日志 | `01-写作资产/experience-log.md` | 有/无 |
| L0 产品层 | `00-原始设定/L0-产品层/PRD.md` | 有/无 |
| L1 元设定层 | `00-原始设定/L1-元设定层/` | 有/无 + 文件数 |
| 宪法 | `00-总控/constitution.yaml` | 有/无 |
| 卷大纲 | `02-大纲/卷XX/剧情脉络.md` | 有/无 |

**输出**（向用户展示）：

```
续写适配 · 环境检测报告
─────────────────────────────
已有正文：    [N] 章（ch001 ~ chNNN）
已有事实骨架：[M] 章（需处理 ch(M+1) ~ chNNN）
全局摘要：    [✓ / ✗]
经验日志：    [✓ / ✗]
L0 产品层：   [✓ / ✗]
L1 元设定层：  [✓ / ✗]
宪法：         [✓ / ✗]
卷大纲：       [✓ / ✗]
─────────────────────────────
建议执行路径：
  → Phase 1: 逆向工程 [未处理章节数] 章 → 01-写作资产/事件日志/
  → Phase 2: L0 提取（含 reader_profile）
  → Phase 3: L1 提取（含 07-数值体系）
  → Phase 4: 宪法提取
  → Phase 5: 卷大纲确认
  → Phase 6: 交接验证
```

### Phase 1：逆向工程（10章一批 → 事件日志）

这是最耗时但最重要的步骤。所有后续提取（L0/L1/宪法）都依赖它产生的数据。

#### 批次划分

```
总待处理章数 = N
批次数 = ⌈N / 10⌉

Batch 1：ch(start) ~ ch(start+9)
Batch 2：ch(start+10) ~ ch(start+19)
...
Batch K：最后一批（不足10章也处理）
```

#### 单批处理：输入

| 输入 | 来源 | 约 |
|-----|------|----|
| 本批10章原文 | 用户提供的 .txt 或 .md 文件 | 2-4万字 |
| 前序批次摘要 | `04-事件日志/batch{N-1}-summary.md` | ~1000字 |
| 用户确认的 L1 候选（如有） | 前一批输出的 L1 候选 | ~5条 |

#### 单批处理：输出 A — 逐章事件日志（v3.3 → 01-写作资产/事件日志/）

每章一个独立文件 `01-写作资产/事件日志/ch{NNN}-eventlog.md`：

```markdown
# ch{NNN} 事件日志

## 1. 关键场景
- [场景名]：[核心事件，1-2句]

## 2. 状态变化
- [角色名]：[关键获得/失去/关系变化]

## 3. 剧情动力
- 推进线索：
- 新开线索：
- 关闭线索：

## 4. 信息不对称
- 读者知道但角色不知道：
- 角色知道但读者不知道：
- 无明显信息差：

## 5. 情绪账户
- 本章走向：[上升/下降/反转]
- 章末读者状态：[期待/担心/愤怒/满足]
- 已兑现承诺：
- 新建承诺：

## 6. 文风指纹
- 平均句长（估）：~[X] 字
- 对话占比（估）：~[X]%
- 本章特色句式/高频词：[1-2个]
```

#### 单批处理：输出 B — 批次摘要（v3.3 → 01-写作资产/事件日志/）

`01-写作资产/事件日志/batch{N}-summary.md`（临时文件，最终合并入 global-summary）：

```markdown
# Batch {N} 摘要（ch{NNN}-ch{NNN}）

## 弧线概述
[本批10章的故事弧，3-5句]

## 主角状态（批次结束时）
- 关键资源/装备：
- 核心关系变化：
- 心理状态：

## 活跃势力
| 势力 | 立场 | 与主角关系 | 当前动向 |

## 跨批伏笔追踪
| 伏笔 | 埋入章 | 本批状态 |

## L1 候选（从本批提取的永久规则）
- [规则1]
- [规则2]

## 文风统计（批次均值）
- 平均句长：~[X] 字
- 对话占比：~[X]%
- 高频词/标志性表达：[Top3-5]
```

#### 用户确认节点

每批完成后暂停，向用户展示关键发现，确认后继续下一批。

### Phase 2：L0 产品层提取（v3.3 含 reader_profile）

**依赖**：Phase 1 全部完成（所有事件日志 + 批次摘要齐全）

**做法**：基于全文事件日志 + 与用户对话，提炼以下内容：

1. **一句话卖点** — 这本书到底卖什么
2. **核心爽点类型** — 打脸/获得/升级/守护/复仇/解谜，各占多少比例
3. **读者定位** — 谁在看这本书，他们想要什么
4. **目标平台** — 番茄/起点/其他
5. **reader_profile** — 从正文用户画像 + 用户确认，产出结构化读者画像

**reader_profile 产出格式**（写入 project.yaml）：

```yaml
reader_profile:
  platform: "番茄"           # 平台
  gender: "男频"             # 性别方向
  age_range: "22-28"        # 年龄区间
  reading_habit: "日刷30-50章，每章停留≤40秒"  # 阅读习惯
  expectation: "每章至少一个微爽点"            # 读者期望
```

这个读者画像将作为第 0 项注入全管线——从 Director 到 Pass 2 到 QC 共享同一个心智模型。

**产出**：
- `00-原始设定/L0-产品层/PRD.md`（格式与 bootstrap 的 PRD.md 完全一致，见 templates/L0-PRD-template.md）
- `project.yaml` 中嵌入 reader_profile 字段

**用户确认**：PRD 和 reader_profile 都必须用户点头，不可跳过。

---

### Phase 3：L1 元设定层提取

**依赖**：Phase 1 全部完成

**做法**：基于全文事件日志 + 批次摘要中的 L1 候选，合并去重后，按以下六个维度输出：

| 文件 | 内容 |
|------|------|
| 01-世界底座.md | 位面结构、时间线、地理范围、核心物理/魔法法则 |
| 02-对抗模型.md | 敌人类型、战力阶位、冲突层级、死亡/失败代价 |
| 03-成长体系.md | 能力演进路径、关键里程碑、质变节点 |
| 04-势力底层.md | 各势力的底层逻辑/利益诉求/不可妥协的底线 |
| 05-主角契约.md | 红线（永不做什么）、语言风格、成长弧线承诺 |
| 06-力量体系.md | 具象化原理、技能树、战斗美学特征 |
| **07-数值体系.md** | **combat_capability（战斗能力约束） + monster_rank_map（敌我段位对照） + act_rank_schedule（幕级升级时间表）** |

**产出**：`00-原始设定/L1-元设定层/01-07.md`

格式与 bootstrap 的 L1 层完全一致（见 templates/L1-元设定层-template.md 和 templates/L1-数值体系-template.md）。

**用户确认**：逐文件确认。用户可以修改/补充。

---

### Phase 4：宪法提取

**依赖**：Phase 1 + Phase 3 完成

**做法**：从正文中归纳"不可违反的规则"，以 constitution 规范格式输出。

**产出**：`00-总控/constitution.yaml`

**关键**：宪法记录的是"永远不变的约束"，不是状态。状态在事件日志里。

**用户确认**：不可跳过。宪法是后续写作的铁律底线。

---

### Phase 5：卷大纲确认

**依赖**：Phase 0 检测结果

**分情况**：

| 情况 | 处理 |
|------|------|
| 已有完整卷大纲 | 确认当前进度 → 标记续写起点幕 |
| 只有粗略规划 | 从事件日志提取卷级事件链 → 整理为剧情脉络.md |
| 完全无大纲 | 从事件日志反向推导卷级结构 → 产出剧情脉络.md |

**产出**：
- `02-大纲/卷XX/剧情脉络.md`（补充/修正）
- `02-大纲/伏笔追踪.md`（从事件日志汇总）

**用户确认**：大纲方向用户必须认可。
