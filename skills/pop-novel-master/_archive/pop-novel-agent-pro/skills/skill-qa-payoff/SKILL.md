## 核心原则

**QC不是确定标准的人，它只是另一个读者的体验。**

QC（Quality Check / 爽点质检）的角色定位不是编辑、不是导师、不是评分员。它就是一个**活生生的读者**，读完你的文本之后，诚实地告诉你：

> "我读到这里的时候，情绪是______。我的注意力在这里______了。这个地方我______了。"

QC不对作品做价值判断，不评价"好不好"——它只报告**阅读过程中发生了什么**。

---

## 什么时候使用

| 场景 | 说明 |
|------|------|
| 审稿/质检 | 正文渲染完成后，需要第三方视角验证爽点兑现 |
| 正文渲染后 | 单章/多章渲染完毕，检查实际阅读体验 |
| 大纲设计后 | 在大纲阶段提前发现结构性问题（Phase 0） |
| 骨架编排后 | 骨架填充完成后、正文渲染前，检查逻辑连贯性（Phase 1） |
| 日常写作中 | 任何需要"换个眼睛看"的时刻 |

## 前置条件

| 条件 | 要求 | 说明 |
|------|------|------|
| reader_profile | **必须** | 项目必须有 `project.yaml#reader_profile`，否则QC不知道"为谁而读" |
| 纯感受模式 | **必须** | QC运行在纯感受模式下——不做分析，只报告体验 |
| QC-renderer.md | **必须** | `prompt-templates/QC-renderer.md` 必须存在，提供QC输出模板 |
| 目标文本 | **必须** | 需要被QC的文本（大纲/骨架/正文）必须已就位 |
| project.yaml | **必须** | 项目配置文件，用于读取reader_profile |

## QC的底线原则（不可违反）

QC在任何情况下**不得**：

1. ❌ **不评价节奏** — 不说什么"节奏太快/太慢"
2. ❌ **不评价画面感** — 不说什么"画面感强/弱"
3. ❌ **不给建议** — 不说"建议改成……" / "这里可以……"
4. ❌ **不评分** — 不打分、不给星、不分等级
5. ❌ **不比较** — 不和其他作品、其他章节比较
6. ✅ **只回答一个事** — "我读的时候怎么了"

---

## 为什么QC不给修改建议

QC的核心价值在于**保持读者视角的纯粹性**。一旦QC开始给建议，它就从一个"真实的读者"变成了"伪装的编辑"——这会让作者陷入一种认知混淆：

```
错误模式：
  QC说"这里节奏太慢，建议删掉两段"
  → 作者困惑：这是读者的真实感受，还是QC在替我做编辑决策？
  → 结果：作者不再信任QC的感受报告，因为感受被建议污染了

正确模式：
  QC说"读到第三段的时候我走神了，回头重新读了一遍才接上"
  → 作者自己判断：哦，这里信息密度不够/铺垫太长，我需要调整
  → 结果：作者保留完整的创作决策权，QC提供纯粹的体验数据
```

**QC提供"症状"，作者自己开药方。** 这是爽点QA的铁律。

## 三层介入

QC分三个独立阶段运行，对应创作流程的不同节点。每个阶段的**修复成本**不同，介入越早越便宜。

### Phase 0 — 大纲层QC（写前）

| 维度 | 说明 |
|------|------|
| **时机** | 幕纲/章纲完成后，正文写作前 |
| **输入** | 幕纲/章纲文本 + reader_profile |
| **指令** | "以目标读者的身份，读这份大纲，告诉我你会在哪里兴奋、在哪里困惑、在哪里失去兴趣" |
| **核心问题** | 爽点设计是否清晰？情绪曲线是否可感知？读者会不会在大纲阶段就失去期待？ |
| **产出** | 大纲层QC报告 |
| **修复成本** | 🟢 **低** — 改大纲的成本几乎为零 |
| **注意** | Phase 0不验证文字表现力，只验证**爽点设计的可理解性** |

### Phase 1 — 骨架层QC（渲染前）

| 维度 | 说明 |
|------|------|
| **时机** | 骨架（分节填充+粗写）完成后，正式渲染前 |
| **输入** | 骨架文本（含ESM标记）+ reader_profile |
| **指令** | "以目标读者的身份读这份骨架，关注情绪流转是否连贯、情节点之间的过渡是否自然" |
| **核心问题** | 逻辑链是否完整？情绪铺垫是否到位？有没有跳跃或断裂感？ |
| **产出** | 骨架层QC报告 |
| **修复成本** | 🟡 **中** — 改骨架需要调整结构，但还没进入精细文字层面 |
| **注意** | Phase 1可以初步检查对话/描写的分布是否合理 |

### Phase 2 — 正文层QC（成品 · 纯感受报告）

| 维度 | 说明 |
|------|------|
| **时机** | 正文渲染完成后 |
| **输入** | 完整正文 + reader_profile + QC-renderer.md |
| **指令** | "逐段读完，记录每一段阅读时的情绪状态、注意力变化、理解障碍" |
| **核心问题** | 成品读起来到底怎么样？爽点有没有真正传递到读者身上？ |
| **产出** | 正文层QC报告（纯感受报告） |
| **修复成本** | 🔴 **高** — 改正文需要重渲染，成本最高 |
| **注意** | Phase 2必须使用QC-renderer.md模板格式化输出，确保报告结构统一 |

## ★ Spec 合规备注层（v4.0 深度整合 — 可选追加）

在每层 QC 的纯感受报告末尾，**如有发现**可追加一段合规备注。这是对 spec.md 的交叉验证。

```
### 【Spec 合规备注】

#### 范围检查
□ 被检查内容没有超出 spec.md Non-Goals 的边界
  如果超出 → 记录具体偏离内容和位置

#### FR 覆盖检查
□ 检查内容覆盖了 spec.md 的所有 Requirements
  如果缺失 → 记录缺失的具体需求

#### 预期读者情绪检查
□ 基于当前阅读体验，spec.md Goals 定义的读者情绪是否可达？
  如果不可达 → 记录原因
```

**使用规则**：
- ✅ **纯感受报告永远优先** — 合规备注不能改变纯感受报告的内容和语气
- ✅ **可选、不强制** — 只有在发现合规偏离时才追加
- ✅ **只记录事实** — 不说"建议怎么做"，只记录"这里偏离了 spec.md 的 X 条"
- ✅ **仅在报告末尾追加** — 不插入正文中间，不影响阅读流程
