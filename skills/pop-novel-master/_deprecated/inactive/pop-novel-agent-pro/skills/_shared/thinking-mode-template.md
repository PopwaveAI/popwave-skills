# Skill 通用 · 思考模式注入规范

> v1.0 | 2026-05-20
> 注入目标：所有 novel-agent-pro 活跃 Skill
> 核心原则：**先思考，再出文件。跳过思考 = 放弃质量。**

---

## 一、为什么需要思考模式

v8.0 导演 Agent 在正文写作前的实践验证了一个规律：

> **"先想清楚再动手"这个模式本身是对的。无论在哪一层，跳过思考直接产文件，质量就会打折扣。反过来，停下来想清楚再动，哪怕只多花一次 LLM 调用的成本，产出也明显不一样。**

在诡异游戏 v2 的 CH001-CH003 写作中，导演 Agent 做了三件纯数据管道做不到的事：
1. **发现文件冲突** — 章纲单线 vs 剧情脉络双线，主动提出来
2. **修正设计缺陷** — act-01.yaml 的 CH002 映射已过时，主动修正
3. **风险预警** — 识别出"不是A而是B"句式的 22 处高风险位置

这些不是"写得好"——是"先想了再写"。

## 二、通用思考模式格式

每个 Skill 的核心产出步骤前，插入一个**思考节点**。

### 思考节点的结构

```
思考节点名称
  ├── ① 对齐检查 —— "上一步的设计和这一步要做的，一致吗？"
  │   检查维度：L0→L1→L2→章纲→正文的继承链有无断裂
  │   冲突处理：发现冲突 → 输出到「待确认/疑问」
  │
  ├── ② 核心判断 —— "这一步要产出的东西，第一性是什么？"
  │   一句话说清楚这一步不可替代的价值
  │   后续所有决策以这个判断为锚点
  │
  ├── ③ 风险扫描 —— "这一步容易踩什么坑？"
  │   基于项目经验日志 + 该 Skill 的历史翻车记录
  │
  ├── ④ 策略说明 —— 输出判断链（散文式，不是字段列表）
  │
  └── ⑤ 待确认/疑问 —— 需要上游或用户确认的问题
```

### 产出格式

```markdown
## 【节点名称】思考说明

### ① 对齐检查
{上一步和这一步的一致性评估}

### ② 核心判断
{这一步的核心目的：一句话说清}

### ③ 风险扫描
{基于经验和规则的潜在风险}

### ④ 策略说明
{判断链——文字描述，不缩写为字段}

### ⑤ 待确认/疑问
{需要上游/用户确认的问题，没有则写"无"}
```

## 四、如何注入现有 SKILL.md

### 注入方式

在现有流水线的"输入→执行→输出"之间插入思考节点：

```
旧：输入 → 执行 → 输出
新：输入 → 【思考节点】→ 执行 → 输出
```

### 不需要改的部分

- ✅ 现有的 HARD-GATE（保留，思考节点在其之后）
- ✅ 现有的设计原则和检查清单（保留）
- ✅ 现有的产出格式（保留，思考节点的产出作为额外上下文）

### 需要改的部分

- 在核心步骤前新增"思考节点"步骤描述
- 新增"思考节点"所需的输入清单（除了常规输入外，还需要上游决策链）
- 新增「待确认/疑问」处理流程
- 版本号 +1

## 五、经验日志集成

每个 Skill 在完成核心产出后，应将其思考节点的关键判断写入项目的经验日志（director-experience-log.md），供下游 Skill 和后续决策参考。

写入内容：
- 做了哪些关键判断
- 发现了什么冲突
- 有什么待确认事项
