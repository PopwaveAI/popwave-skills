# Step 2 - 幕纲

把卷纲幕序列中的一幕展开为分章施工卡。逐幕循环执行，每幕产出一个幕纲。

按 `templates/幕纲-模板.md` 输出到 `卷纲/幕纲-幕NNN-{标题}.md`。

## 2a. 定位确认

从卷纲幕序列生成/继承当前幕定位，必须使用模板六项：

- 幕定位（一句话：这幕在卷中干什么）
- 主爽点（1-2个）
- 读者期待（读者读完这幕应该得到什么）
- 主角弧线：起点→终点
- 上幕衔接：怎么进来的
- 下幕衔接：最后怎么勾住下一幕

输出当前幕定位后暂停，请用户确认或修改。用户确认前，不进入选种子和幕级外部燃料台。

## 2b. 选燃料

从卷纲可选剧情燃料池中摆出与当前幕功能位相关的燃料，供用户挑选。此步才开始决定当前幕要开采哪些燃料，但仍不直接写分章。

1. 列出相关主线燃料（3-5条）
2. 列出相关支线/暗线燃料（2-3条）
3. 列出相关世界观/设定燃料（1-2条）
4. 标明每条燃料的主角操作点、可外显爽点和风险/禁区
5. 询问用户是否要补充燃料（"我想加个电影里的剧情""这段我看过类似的处理"）

用户确认选定的燃料列表后，进入故事推演。

## 2c. 幕级外部燃料台

幕纲阶段窄用外部信息：只精读当前幕对应的主母版单元；辅助桥段最多1-2个，并说明补哪个缺口。

### 资料覆盖声明

- 主母版当前单元：已读范围（完整L2/章节设计包/片段）
- 辅助桥段：已读范围
- 未覆盖但可能影响本幕：

最低要求：主母版当前单元必须读完整；若只有片段，幕纲只能标记为草案。

### 主母版当前单元

允许短复述原剧情，保留具体事件形状：

- 原剧情短复述：谁想要什么、发生了哪几个动作、爽点在哪里兑现。
- 本幕要保留的功能：练级/清算/地牢/撤离/大战等。
- 本项目变形方式：用本书设定和主角解法替换原解法。

### 辅助桥段

只在当前幕缺少新鲜感、反转、战斗调度、地下规则、追踪逃亡等具体燃料时使用：

| 来源 | 补的缺口 | 原桥段短复述 | 本项目变形 | 采用章 |
|------|----------|--------------|------------|--------|

不要输出大型证据绑定清单。每个辅助桥段1-3句即可。

## 2d. 故事推演

基于选定的燃料，展开为分章beat链。幕纲负责把燃料排成剧情，不得反向要求卷纲补分章剧情。

### 登场人物

从卷纲本卷选角表中选取本幕登场人物，补强剧情耦合信息：本幕是否具名、幕初/幕末状态、关系变化、具体功能。反应模式和说话方式继承角色储备池/卷纲，不重写。

### 地点 + 时间

本幕覆盖的地点和时间跨度。

### 分章剧情展开

每章一个块：

```
### ch001 · {章名}
- 字数基准：XXXX字
- 章卡边界：本章只写什么；禁止吞并下一章什么
- POV：主角限制视角/配角短视角/妹妹视角等
- scene：1-3个文风DNA场景标签
- 情绪目标：一句话
- 主爽点：一句话说明读者爽在哪里
- 宏观剧情骨架：3-5个功能beat（只写本章大方向）
- 微beat施工表：10-14个可写动作beat（短章8-10个）
- 主角主动动作：
- 可数收益/明确损失：
- 爽点外显：面板/数字/敌人反应/旁人态度/资源到账/身份变化等
- 系统/战斗节点：
- 章末钩子：
- 禁改项：
```

节奏要求：
- 每章必须同时有“宏观剧情骨架”和“微beat施工表”。宏观骨架回答这一章干什么；微beat回答正文每一小段具体写什么。
- 2500-3500字章节至少10-14个微beat；1800-2200字短章至少8-10个微beat；低于这个数量只能标记为草案。
- 微beat必须是可写成正文的动作/阻力/反馈，不得只写“推理”“转场”“情绪铺垫”“战斗”“解释设定”。
- 每个微beat必须至少包含四项中的两项：角色动作、阻力/风险、信息变化、读者反馈。
- 每个微beat只消费本章内容，不得提前写下一章的坐标、战斗、揭示或结算。
- 事件间隔≤20行
- 每章必须有主爽点（压力→突破/弱→强/未知→揭示/被欺→反击），不要只写情绪目标
- 每章必须有 POV 和 scene；scene 用于 create 读取文风DNA
- 每章主角至少1个主动决策+1个行动
- 每章必须有可数收益或明确损失：钱/经验/装备/名声/情报/地盘/关系/代价
- 每章必须有爽点外显方式，不能只写"爽"
- 章末钩子必须是主角主动决策型

### 微beat 设计指引

**微beat 不只有事件描述——8种 beat 交替使用才有节奏：**

| beat 类型 | 作用 | 频率参考 |
|:---------|:-----|:---------|
| 情绪beat | 共鸣/动机/张力 | 每章1-2个，不能多 |
| 行动beat | 主角主动出击 | 每章2-4个，主力 |
| 收获beat | 面板/升级/结算/资源到账 | 系统流每章1-2个 |
| 信息beat | 揭示/反转/线索 | 每章1-2个 |
| 冲突beat | 对峙/被欺/打脸 | 每章1-2个，爽点载体 |
| 损失beat | 挫折/代价/危机升级 | 每章0-1个，制造压力 |
| 社交beat | 结盟/背叛/身份暴露 | 每章0-1个 |
| 日常beat | 喘息/互动/彩蛋 | 大战后1个，不能多 |

微beat施工表必须显式标注 beat 类型，方便 create 控制字数和节奏。纯行动beat链=干瘪；纯情绪beat链=拖沓；交替=节奏。开场可以情绪beat起手建立动机，但第2-3个beat必须切到行动或收获。

**主角反应模式必须匹配角色设定：**

设计 beat 时，主角的反应方式必须对齐卷纲舞台表中定义的反应模式（从 seed 主角引擎继承）：
- 前世高手/精锐特工 → 冷静评估→专业战术→致命执行（不是慌乱/蛮力/乱跑）
- 少年成长型 → 热血冲撞→吃瘪→领悟→突破
- 腹黑谋略型 → 隐忍布局→设套→收网

配角和反派的反应模式同样从卷纲舞台表继承。beat 中的对峙场景必须体现双方反应模式的碰撞——"冷静评估型主角 vs 暴怒失控型反派"和"冷静评估型主角 vs 阴沉算计型反派"写法完全不同。

如果主角设定是"前世高手"，beat 里不能出现"慌乱""无头苍蝇""混乱中捅死"——应该是"观察三人站位→判断威胁顺序→阴影中绕到最弱者身后→一击毙命→立即转向第二人"。

**网文节奏锚点：**

- 开场 beat 不能纯氛围——第一个 beat 必须有信息钩子（系统弹窗/异常信号/悬念抛出），让读者在第120字就知道"这本书有看点"
- 情绪 beat 必须带钩——不是单纯渲染情绪，是情绪中埋一个"为什么"（妹妹为什么不怕黑？），让情绪beat同时承担信息功能
- 行动 beat 必须带收益——主角主动行动后，后续 beat 必须给出可数反馈（经验/装备/情报/名声），不能行动完没回响
- 章末 beat 必须是主动决策——不是"脚步声逼近"这种被动悬念，是"他握紧匕首走向出口"这种主动选择

### 章卡边界

每张章级施工卡必须写清：

- 本章允许写到哪里结束。
- 下一章开场要保留什么。
- 本章禁止提前兑现什么。
- 如果 create 写作时自然滑到下一章，必须停下，不得合并章节。

### 子线追踪

本幕内推进的子线（从卷纲可选剧情燃料池中选取的相关项）：

| 子线 | 幕初状态 | 各章推进 | 幕末状态 |
|------|----------|----------|----------|

## 2e. Create 交接

幕纲末尾输出给 create 的施工指令：

- 文风注意：
- 不可改动事件：
- 可自由发挥空间：
- 爽点外显要求：
- 角色行为锚点：主角=（从卷纲舞台表反应模式继承）；反派A=（反应模式）；配角B=（反应模式）。create 写正文时对白和行动必须对齐这些锚点。

## 2f. 用户确认门

输出幕纲到 `卷纲/幕纲-幕NNN-{标题}.md`，暂停。

向用户确认：
- 分章beat链是否合理
- 节奏是否需要调整
- 幕级外部燃料是否需要追加或替换
- 是否交 create 写正文

用户确认后交 create。写完正文后回来做下一幕幕纲（回到 2a）。
