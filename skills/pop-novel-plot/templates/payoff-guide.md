# payoff-guide.md — 爽点四档定义 + 设计方法 + 责任归属

> 管线: pop-novel-plot v6.1 · chapter-design v1.5 · prose-render v2.0
> 定位：全链路唯一的爽点权威定义。所有 skill 涉及爽点设计/检查时以此为基准。

---

## 一、定义总表

| 级别 | 密度 | 是什么 | 读者反应 | 一个例子 |
|:-----|:----:|:------|:--------|:--------|
| 微爽点 | 每章 ≥ 5 | 段落级。读者在一两个句子内感到"哦有点意思" | 嘴角动一下、继续往下扫 | 主角推开门的瞬间看到一把刀悬在门框上——他低了头 |
| 中爽点 | 每章 ≥ 1 | 事件级。读者读完一小段后确认"刚才确实发生了点什么" | 眼睛睁大、调整了坐姿 | 审讯到第三句时，对方突然笑了——"你以为你抓的是谁？" |
| 大爽点 | 间隔 ≤ 5 章 | 情绪高潮。读者等了几天/几章，终于到了——放下手机缓一口气 | 想截图、想跟人分享 | 主角把手伸进裂缝，拉出来的不是武器，是他以为死了三年的妹妹 |
| 终极爽点 | 每幕 ≥ 2 | 幕级极值。这一幕最炸裂的两段——看完这一章，读者知道"这幕值了" | 看完坐着不动、然后翻回去重读 | 全书第一卷最后一章：主角把世界树连根拔起，天空裂成了两半 |

## 二、微爽点详解

### 2.1 定义

**微爽点 = 文本层面的一次微脉冲。** 不是情节推进，是阅读体验中的"小快感"。读者不会停下来想，但会因为这些微脉冲持续翻页。

### 2.2 五种类型 + 案例

| 类型 | 说明 | 案例 |
|:-----|:-----|:-----|
| **小反转** | 上一个句子的预期被下一个句子推翻 | "他以为门锁了——一推就开。" "刀拔出来的一瞬间，他看到了刀背。" |
| **信息碎片** | 一个细节让读者"啊，原来如此" | "血的流向不对。不是从伤口往外流，是从门口往伤口流。" |
| **角色微表情** | 一句对话配一个动作，暴露角色的真实状态 | '"你走吧。"她没抬头，但手里的茶杯抖了一下。' |
| **环境锐化** | 一个感官细节让场景突然变立体 | "风从北边来，带的不只是沙子——还有铁锈味。" |
| **节奏剪刀** | 一个极短句切断前面的长描述，制造冲击 | "三。二。一。火光亮了。" |

### 2.3 如何设计

微爽点**不在 plot / chapter-design 中规划**。它们诞生于 prose-render 的写作过程。

Render 阶段如何确保 ≥ 5/章：

```
写完正文后，逐段扫：
- 每 200-400 字内，读者有没有遇到"微脉冲"？
- 是否有超过 500 字连续无反转/无感官/无信息碎片？
- 章末倒数 500 字是否密度最高？（读者最后看到的决定"下一章点不点"）

自检方式：把正文给 reader_profile 身份读一遍，标记出 5 个"嗯，这里我多看了一秒"的位置。
找不够 5 个 → 回到最平淡的段落，插入小反转/微表情/环境锐化。
```

### 2.4 责任归属

| 负责 | 不负责 |
|:-----|:-----|
| ✅ prose-render — 写作时创造 | ❌ plot — 不规划具体位置 |
| ✅ prose-render Step 3 — 扫描验证密度 | ❌ chapter-design — 不涉及段落级文本 |

## 三、中爽点详解

### 3.1 定义

**中爽点 = 一次可感知的情节推进或情绪转折。** 读者读完这一段后，对故事的认知发生了变化——知道了新的信息、角色关系变了、局势升级了。

### 3.2 四种类型 + 案例

| 类型 | 说明 | 案例 |
|:-----|:-----|:-----|
| **信息揭示** | 读者知道了之前不知道的关键信息 | "那个一直不说话的小孩，是城主十年前丢的儿子。" |
| **关系转折** | 两个角色的关系发生质变——升级/破裂/反转 | "她收了刀。不是因为打不过——是因为看到了他脖子上的旧疤。" |
| **障碍突破** | 主角闯过了一道关卡 | "第三扇门打开时，整个地下室亮了——他找到了。不只是武器。还有地图。" |
| **情绪落地** | 积蓄的情绪终于释放——读者和角色一起感受到了 | "他蹲下来，把戒指放在墓碑前。没说一句话。但走的时候，背直了。" |

### 3.3 如何设计

中爽点**在 chapter-design 的事件链中确定**。

```
Step 2 逐事件设计时：
- 每个事件问：读者读完这个事件后的认知有没有变化？
- 有变化 → 标记为中爽点事件
- 无变化（纯过渡、纯环境、纯铺垫）→ 不标记

全部事件写完后验证：中爽点事件数 ≥ 1？
不够 → 把至少一个过渡事件升级为有信息揭示或情绪转折的事件。
```

### 3.4 责任归属

| 负责 | 不负责 |
|:-----|:-----|
| ✅ chapter-design Step 2 — 逐事件设计时标记中爽点 | ❌ plot — 不规划具体事件级爽点 |
| ✅ chapter-design Step 3 — 验证每章 ≥ 1 个中爽点事件 | |

## 四、大爽点详解

### 4.1 定义

**大爽点 = 读者等了几章的高潮终于到了。** 需要铺垫——前面至少 2-3 章在蓄力。读者知道"快了快了"，然后在某一章爆发。

### 4.2 四种类型 + 案例

| 类型 | 说明 | 案例 |
|:-----|:-----|:-----|
| **战斗高潮** | 本章的核心战斗。不是清扫杂兵，是有名字的对手、有赌注的输赢 | "终于到了擂台上。对面站着的不是别人——是十年前废了他的那个人。" |
| **重大揭示** | 一个信息改变了读者对整个故事的理解 | "那些信不是威胁。是求救。每一封都是——她从来没有背叛过。" |
| **关系质变** | 两个核心角色的关系完成一次不可逆的升级 | "她拉住了他。不是拦——是跟他一起跳。" |
| **副本通关** | 主角通过了一片完整的区域/关卡/试炼，获得了明确的成果 | "石碑碎了。不是他打的——是他通过了试炼之后，碑自己碎的。" |

### 4.3 如何设计

大爽点**在 plot 的幕纲编排中确定位置**。

```
plot Step 2 设计 act-XX.yaml 时：
1. 在 Canvas 矩阵中标记大爽点章节号
2. 大爽点前 2-3 章 → 标记为蓄力章（连续叠加压力/悬念/冲突升级）
3. 用 rhythm_check 自检：大爽点间隔 ≤ 5 章？
4. 把章节号填入 payoff_distribution.大爽点.positions
```

### 4.4 责任归属

| 负责 | 不负责 |
|:-----|:-----|
| ✅ plot Step 2 — 决定大爽点在哪章 + 铺垫蓄力 | ❌ chapter-design — 按大爽点章号设计事件链（不自己决定位置） |
| ✅ plot 自检 — rhythm_check 验证间隔 ≤ 5 章 | ❌ prose-render — 不判断是不是大爽点 |

---

## 五、终极爽点详解

### 5.1 定义

**终极爽点 = 这一幕中情绪最高的两个点。** 至少 2 个：一幕至少需要一个"中段爆发"和一个"幕末收束"。读者看完终极爽点章后，会觉得这一整幕没有白追。

### 5.2 两种位置 + 案例

| 位置 | 说明 | 案例 |
|:-----|:-----|:-----|
| **幕中爆发** | 幕约 1/2-2/3 处。全幕情绪的第一次极值 | "他跪在神殿中央，不是认输——是终于懂了那句预言。站起来的时候，他的影子在往下长。" |
| **幕末收束** | 幕最后一章。全幕的最终爆发，同时给下一幕留钩子 | "裂缝闭上了。不是他封的——是里面的东西自己关的门。但在门合上之前，他看到了一只眼睛。那只眼睛认识他。" |

### 5.3 如何设计

终极爽点**在 plot 的幕级定义中确定位置**。

```
plot Step 2 设计 act-XX.yaml 时：
1. 在 Canvas 矩阵中标记终极爽点章节号（必须 ≥ 2 个位置）
2. 幕中爆发：幕约 1/2-2/3 处
3. 幕末收束：幕最后一章（同时也是下一幕的情绪起点）
4. 填入 payoff_distribution.终极爽点.positions
```

### 5.4 责任归属

| 负责 | 不负责 |
|:-----|:-----|
| ✅ plot Step 2 — 决定终极爽点在哪两章 | ❌ chapter-design — 按位置设计事件链 |
| ✅ plot 自检 — rhythm_check 验证 ≥ 2 个 | ❌ prose-render — 不判断是不是终极爽点 |

---

## 六、全链路责任归属总表

```
plot (幕纲编排)
  ├─ 大爽点位置规划（payoff_distribution.positions）
  ├─ 终极爽点位置规划（幕中+幕末）
  ├─ 中爽点密度声明（每章 ≥ 1，不指定具体事件）
  ├─ rhythm_check: 大爽点间隔 ≤ 5 章？终极 ≥ 2 个？
  └─ Canvas 矩阵: 标记各章 payoff 类型

chapter-design (事件链设计)
  ├─ 逐事件标记中爽点（哪些事件是"可感知的情节推进"）
  ├─ 大爽点章 = 事件链高潮结构设计（按 plot 指定的章号）
  ├─ 终极爽点章 = 事件链全幕最高密度设计
  └─ Step 3 自检: 本章中爽点事件 ≥ 1？

prose-render (正文渲染)
  ├─ 创造微爽点（写作技艺层面）
  └─ Step 3 自检: 每 500 字内至少 1 次微脉冲？全章 ≥ 5 个微爽点？
```
