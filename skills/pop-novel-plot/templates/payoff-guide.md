## 一、定义总表（plot 负责的部分）

| 级别 | 密度 | 是什么 | 读者反应 | 案例 |
|:-----|:----:|:------|:--------|:-----|
| 大爽点 | 间隔 ≤ 5 章 | **一条剧情线经过多章蓄力后的释放**。读者等了几章，终于到了——放下手机缓一口气 | 想截图、想跟人分享 | 主角把手伸进裂缝，拉出来的不是武器，是他以为死了三年的妹妹 |
| 终极爽点 | 每幕 ≥ 2 | **本幕情绪最高的两个点**。全幕所有剧情线的交汇爆发。看完这一章，读者知道"这幕值了" | 看完坐着不动、然后翻回去重读 | 全书第一卷最后一章：主角把世界树连根拔起，天空裂成了两半 |

## 二、大爽点详解

### 2.1 定义

**大爽点 = 一条 M 线或 S 线在做了 N 章蓄力铺垫后，终于释放的那一章。**

- 不是"这一章好看"——是"这条线从这章开始变了"
- 一条线前面几章都在埋伏笔、设悬念、蓄压力 → 这一章爆发
- M1/M2/M3 三条主线各有自己的释放节点

### 2.2 四种类型 + 案例

| 类型 | 说明 | 案例 |
|:-----|:-----|:-----|
| **战斗高潮** | 本章的核心战斗。不是清扫杂兵，是有名字的对手、有赌注的输赢 | "终于到了擂台上。对面站着的不是别人——是十年前废了他的那个人。" |
| **重大揭示** | 一个信息改变了读者对整个故事的理解 | "那些信不是威胁。是求救。每一封都是——她从来没有背叛过。" |
| **关系质变** | 两个核心角色的关系完成一次不可逆的升级 | "她拉住了他。不是拦——是跟他一起跳。" |
| **副本通关** | 主角通过了一片完整的区域/关卡/试炼，获得了明确的成果 | "石碑碎了。不是他打的——是他通过了试炼之后，碑自己碎的。" |

### 2.3 Canvas 联动设计方法

**核心逻辑**：Canvas 矩阵中，一条线从 chX 铺垫到 chY 释放 → chY 就是这条线的大爽点候选章。

```
plot Step 2 设计 Canvas 时，逐行逐线标记 payoff_level：

Canvas 填写流程（新增 ★ 第④步）：

① 遍历本幕所有章节（ch1 ~ chN）
② 对每章，逐一确认每条线（M1/M2/M3/S1/S2）是否推进：
   - 推进了 → 写一句话摘要
   - 没推进 → 留空
③ 填写节奏笔记
④ ★ 标注每条线在本章的 payoff_level：
   - 空 / null   → 铺垫/建设/埋伏笔。这条线在动，但没有释放
   - 微          → 小信息碎片/小推进。这条线动了一小步
   - 中          → 这条线在这里有一次可感知的释放（信息揭示/关系转折）
   - 大 ★        → 这条线在这里爆发——前面蓄力的全部在这一章释放
⑤ 扫描整张 Canvas：所有 大 的格子 → 那些章就是大爽点章号
```

**判断一条线"有没有释放"的方法**：

```
对 Canvas 矩阵中 M1 这条线的每章摘要，问：
- ch3: "发现裂缝扩大"         → 铺垫（只是发现，没释放）
- ch5: "追踪到裂缝源头"        → 铺垫（在追，没追到）
- ch7: "进入裂缝——看到了源头"   → 中（终于看到了，有了新认知）
- ch11: "封住裂缝，代价是右手"  → ★ 大（这条线从ch3蓄力到ch11，终于释放）
- ch15: "裂缝重开，更大"        → 铺垫（新的蓄力循环开始）
```

### 2.4 大爽点间隔保证

每章的 `payoff_map` 汇总后：
- 扫描本幕所有章的大爽点标记
- 大爽点间隔必须 ≤ 5 章（product rule）
- 如果两条相邻大爽点之间超过 5 章 → 把中间某条线的释放提前，或把前一个大爽点后移

### 2.5 责任归属

| plot 负责 | plot 不负责 |
|:----------|:-----------|
| Canvas 中逐线标注 payoff_level | 具体事件内容（归 chapter-design） |
| 从 Canvas 提取大爽点章节号 → payoff_distribution.大.positions | 大爽点章的事件链怎么设计（归 chapter-design，但 plot 给了"线+释放"的方向） |
| 大爽点前 2-3 章标记蓄力 | |

## 三、终极爽点详解

### 3.1 定义

**终极爽点 = 全幕多条线在同一章或连续两章集中爆发。** 它是"这一幕的顶点"。

与 大爽点 的区别：
- 大爽点 = 一条线释放
- 终极爽点 = **多条线同时释放** + 情绪极值

### 3.2 两种位置 + 案例

| 位置 | 说明 | 案例 |
|:-----|:-----|:-----|
| **幕中爆发** | 幕约 1/2-2/3 处。全幕情绪的第一次极值 | "他跪在神殿中央，不是认输——是终于懂了那句预言。站起来的时候，他的影子在往下长。" |
| **幕末收束** | 幕最后一章。全幕的最终爆发，同时给下一幕留钩子 | "裂缝闭上了。不是他封的——是里面的东西自己关的门。但在门合上之前，他看到了一只眼睛。那只眼睛认识他。" |

### 3.3 Canvas 联动设计方法

**终极爽点 = 多条线 payoff_level=大 同时出现在同一章或相邻章。**

```
扫描 Canvas.payoff_map 汇总表：
- 哪些章有 ≥ 2 条线的 payoff_level = 大？→ 终极爽点候选
- 幕中（约 1/2-2/3 处）选一章做幕中爆发
- 幕末最后一章做幕末收束
- 填入 payoff_distribution.终极.positions

如果整幕只有 1 个多线汇聚点 → 选它做幕末收束，同时在幕中约 1/2 处强行设计一个：
把一条快到释放点的线提前到那里释放，或者让两条中释放线汇聚
```

### 3.4 责任归属

| plot 负责 | plot 不负责 |
|:----------|:-----------|
| Canvas 多线汇聚章识别 → 终极爽点位置 | 具体事件内容 |
| 如果天然不足 2 个 → 调整 Canvas 线释放节奏 | |

## 四、Canvas payoff_map — 一章结束后的交付物

Canvas 填完后，plot 应自动产出一个小结表，交给 chapter-design 消费：

```
payoff_map:
  ch1: 总payoff=0           → ⚠️ chapter-design 须自行制造至少 1 个中爽点
  ch2: 总payoff=1(中/大)   → ✅ Canvas 已供给中爽点，design 按线号定位事件
  ch3: 总payoff=2(中/大/中) → ✅ Canvas 已供给 ≥2 个中爽点
  ch4: 总payoff=1(大)      → ✅ 大爽点章
  ...
  ch14: 总payoff=3(大/大/中) → ★ 终极爽点候补（3线汇聚）
```

**消费规则**：
- 总 payoff=0 的章 → chapter-design 必须自己制造 ≥ 1 个中爽点事件（方法见其 payoff-guide.md）
- 总 payoff≥1 的章 → chapter-design 按 `payoff_map` 指明的线号定位，设计对应的事件
- 大爽点章 → chapter-design 设计全幕最高密度事件链
- 终极爽点章 → chapter-design 设计全幕最大规模事件链
