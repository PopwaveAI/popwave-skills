# 爽点QA — 纯感受报告 · 四层介入

> 版本：v0.6.0 | 2026-06-11

## ❌ 质量红线（开工前→完工后自检）

执行 QC 前必须逐项确认，任何一条违反即标记 **REJECT**：

- [ ] reader_profile 已确认存在（project.yaml#reader_profile）—— 不知道为谁而读就不读
- [ ] 不行使节奏评价 —— 不说什么"节奏太快/太慢"
- [ ] 不行使画面感评价 —— 不说什么"画面感强/弱"
- [ ] 不给修改建议 —— 不说"建议改成……"/"这里可以……"
- [ ] 不评分 —— 不打分、不给星、不分等级
- [ ] 不比较 —— 不和其他作品、其他章节比较
- [ ] 任何红线违反 → 立即标记 REJECT，不输出报告

> 红线违反时**不得**尝试修正、不得折中、不得输出不完整报告。

## 什么时候用

| 场景 | 说明 |
|------|------|
| **Step 0 — 项目审计** | 开书后/卷完成后/怀疑管线不对齐时，做一次全项目体检 |
| 审稿/质检/爽点质检 | 正文渲染完成后，需要第三方视角验证爽点兑现 |
| 正文渲染后 | 单章/多章渲染完毕，检查实际阅读体验 |
| Step 1 — 大纲设计后 | 在大纲阶段提前发现结构性问题 |
| Step 2 — 骨架编排后 | 骨架填充完成后、正文渲染前，检查逻辑连贯性 |
| Step 3 — 正文成品 | 成品读起来到底怎么样 |

## 前置条件

| 条件 | 要求 | 说明 |
|------|------|------|
| reader_profile | **必须** | 项目必须有 `project.yaml#reader_profile`，否则 QC 不知道"为谁而读" |
| QC-renderer.md | **必须** | `prompt-templates/QC-renderer.md` 必须存在，提供 QC 输出模板 |
| 目标文本 | **必须** | 需要被 QC 的文本（大纲/骨架/正文）必须已就位 |
| project.yaml | **必须** | 项目配置文件，用于读取 reader_profile |
| 纯感受模式 | **必须** | QC 运行在纯感受模式下——不做分析，只报告体验 |

## 四层介入（4步）

### Step 0 — 全量对齐审计（项目体检）

| 维度 | 说明 |
|------|------|
| **时机** | 开书后 / 每卷完成后 / plot 完成后 / 怀疑文件不对齐时 |
| **输入** | 项目根目录 + project.yaml + `expert-writer/references/pipeline-arch.md` |
| **指令** | "以 pipeline-arch.md 为基准，逐目录比对项目文件树；逐文件读入检查文档质量；判定管线阶段是否对齐" |
| **核心问题** | 项目工程文件是否完整？每个工程级文件是否达到对应 skill 的标准？当前阶段缺什么？ |
| **产出** | 项目全量对齐审计报告（按 `audit-renderer.md` 模板输出） |
| **修复成本** | 🟢 **最低**（仅报告，不修改） |
| **边界** | **工程层扫描**：S 型静态文件 + project-level D 文件（project.yaml / 角色卡 / entity-snapshot / volume-XX / act-XX）。**不碰单章产物**（chXXX-设计包 / chXXX.md 留给 Step 2/3） |
| **注意** | 不评价设计好坏 — 只回答「有没有」和「对齐了吗」。不包含修改建议。 |

**审计四维度：**

```
A. 文件树完整性 — 以 pipeline-arch.md §二 规范目录树为基准，逐目录比对存在 vs 缺失
B. 文档质量审查 — 读取每个产出文件，检查内容是否达到对应 skill 的产出标准
C. 管线阶段对齐 — 当前 project.yaml#phase vs 实际文件完整性，找断点
D. 文件分类正确性 — 对照 pipeline-arch.md §一 S/D/M 矩阵，检查动态文件是否在维护
```

> 输出模板 → `prompt-templates/audit-renderer.md`
>
> 加载方式：`Get-Content -Encoding UTF8 -Raw`，不用 Read 工具。
>
> **与 Step 1/2/3 的根本差异**：Step 0 是**横向扫描**（全文件 × 全目录），不问阅读体验；Step 1/2/3 是**纵向切入**（单章 × 单幕），只问阅读感受。

### Step 1 — 大纲层 QC（写前）

| 维度 | 说明 |
|------|------|
| **时机** | 幕纲/章纲完成后，正文写作前 |
| **输入** | act-XX.yaml（当前幕切片）+ reader_profile |
| **指令** | "以目标读者的身份，读这份大纲，告诉我你会在哪里兴奋、在哪里困惑、在哪里失去兴趣" |
| **核心问题** | 爽点设计是否清晰？情绪曲线是否可感知？读者会不会在大纲阶段就失去期待？ |
| **产出** | 大纲层 QC 报告 |
| **修复成本** | 🟢 **低** |
| **注意** | 不验证文字表现力，只验证**爽点设计的可理解性** |

### Step 2 — 骨架层 QC（渲染前）

| 维度 | 说明 |
|------|------|
| **时机** | 骨架（分节填充+粗写）完成后，正式渲染前 |
| **输入** | chXXX-设计包.md（含 ESM 标记）+ reader_profile |
| **指令** | "以目标读者的身份读这份骨架，关注情绪流转是否连贯、情节点之间的过渡是否自然" |
| **核心问题** | 逻辑链是否完整？情绪铺垫是否到位？有没有跳跃或断裂感？ |
| **产出** | 骨架层 QC 报告 |
| **修复成本** | 🟡 **中** |
| **注意** | 可初步检查对话/描写的分布是否合理 |

### Step 3 — 正文层 QC（成品 · 纯感受报告）

| 维度 | 说明 |
|------|------|
| **时机** | 正文渲染完成后 |
| **输入** | 正文/chXXX.md + reader_profile + QC-renderer |
| **指令** | "逐段读完，记录每一段阅读时的情绪状态、注意力变化、理解障碍" |
| **核心问题** | 成品读起来到底怎么样？爽点有没有真正传递到读者身上？ |
| **产出** | 正文层 QC 报告（纯感受报告） |
| **修复成本** | 🔴 **高** |
| **注意** | 必须使用 QC-renderer.md 模板格式化输出，确保报告结构统一 |

## ★ Spec 合规备注层（可选追加）

在每层 QC 的纯感受报告末尾，**如有发现**可追加一段合规备注。这是对 `spec.md` 的交叉验证。

```
### 【Spec 合规备注】

#### 范围检查
□ 被检查内容没有超出 spec.md Non-Goals 的边界
  如果超出 → 记录具体偏离内容和位置

#### FR 覆盖检查
□ 检查内容覆盖了 spec.md 的所有 Requirements
  如果缺失 → 记录缺失的具体需求

#### 预期读者情绪检查
□ 基于当前阅读体验，spec.md Goals 定义的读者情绪是否可达？
  如果不可达 → 记录原因
```

**使用规则**：
- ✅ **纯感受报告永远优先** — 合规备注不能改变纯感受报告的内容和语气
- ✅ **可选、不强制** — 只有在发现合规偏离时才追加
- ✅ **只记录事实** — 不说"建议怎么做"，只记录"这里偏离了 spec.md 的 X 条"
- ✅ **仅在报告末尾追加** — 不插入正文中间，不影响阅读流程

## ❌ 错误示例

### WRONG 1：QC 越界给建议

```
QC："第三章的开头节奏太慢了，建议删掉第2-3段，把冲突提前。"

❌ 违规项：节奏评价 + 给修改建议
正确做法：只报告感受——"读到第三章开头第二段时我走神了，回头重新看了一遍才接上。"
```

### WRONG 2：QC 打分排名

```
QC："本章综合评分 7.5/10，在目前已渲染的章节中排名第三。"

❌ 违规项：评分 + 比较
正确做法：只报告阅读体验——"读这一章时我的情绪最高点在______，最低点在______。"
```

## 异常与边界条件

| 场景 | 处理方式 |
|------|----------|
| reader_profile 缺失 | **REJECT** — 无法执行 QC。告知用户必须先配置 `project.yaml#reader_profile` |
| QC-renderer.md 缺失 | **REJECT** — 无法格式化输出。告知用户需要先创建 `prompt-templates/QC-renderer.md` |
| 目标文本未就绪 | **REJECT** — 没有文本可读。告知用户需要先准备好目标文本（大纲/骨架/正文） |
| 用户坚持要求评分 | **温和而坚定地拒绝** — 重申 QC 不输出任何形式的分数。如需要量化评估，建议使用其他工具或人工编辑 |
| 多个阶段同时提交 | 按 Step 1 → Step 2 → Step 3 优先级依次处理，分别输出独立报告。不合并、不交叉 |
| Spec 合规检查与纯感受报告冲突 | **纯感受报告优先** — 合规备注永远附属于纯感受报告。合规问题如实记录在备注中，但不修改感受报告本身 |
| 用户要求改进建议 | **拒绝** — 重申 QC 只提供"症状"，不开药方。作者应根据 QC 报告自行判断如何改进 |
