# Pop Writer V3 Review

review 是用户主动唤起的编辑审稿室，不是 create 后的默认流水线，也不是 arc 式弧线管理。

## 审稿优先级

review 的首要任务不是找错字、查格式或泛泛评价“好不好”。除非用户明确另有要求，按以下优先级审稿：

1. 爽文兑现：读者有没有得到压迫、期待、主角主动、误判、反转、收益、围观反馈、章末追读。
2. AI味：正文是否出现抽象总结、情绪标签、假紧张、机械转折、严肃文学式空镜、同质句式和低信息密度。
3. 剧情可读性：目标、阻力、变化、信息差和场面推进是否清楚。
4. Bug/OOC：时间线、规则、资源、角色反应和上游承诺是否冲突。
5. 文风贴合：是否贴近文风DNA，但文风不得覆盖爽文兑现。

如果爽文兑现失败或 AI 味严重，即使 bug/OOC 很少，也不得给“可用”等级。

## 输入

- 待审正文：单章、多章、单幕或一卷片段。
- 可选上游资料：幕纲、章级施工卡、章节运行日志、文风DNA、角色储备池、本卷选角表、设定账本。
- 用户关注点：AI味、爽感、bug、OOC、节奏、文风、剧情是否好看、项目是否跑偏。

## 执行

1. 读 `steps/step-1-editor-review.md`。
2. 若当前环境支持子 agent，至少发起两个独立审读视角：
   - 爽文/AI味审稿员：只判断爽不爽、哪里像 AI、哪里无聊、读者奖励是否兑现。
   - 连续性审稿员：对照必要上游资料判断 bug、OOC、设定冲突、承诺断裂。
3. 主审 agent 汇总子 agent 结论，去重、分级、给处理建议。
4. 按 `templates/编辑审稿报告-模板.md` 输出。

## 子 Agent 规则

- 子 agent 必须读原文，不能只读摘要。
- 子 agent 只拿完成任务所需资料，不把全项目资料一股脑塞进去。
- 子 agent 之间不要共享结论；先独立判断，再由主审合并。
- 爽文/AI味审稿员允许主观，但必须引用具体段落或场面，并按爽文评分表和 AI味评分表打分。
- 连续性审稿员必须给出依据：正文事实、上游设定、角色锚点或运行日志。

## 审稿维度

- 爽感：主角主动性、压力、误判、收益、代价、反馈、章末钩子。必须判断爽点是否“发生在正文里”，不能只看文本是否声明它发生。
- AI味：套话、抽象解释、情绪标签、重复句式、假紧张、总结腔、机械转折、严肃文学式空镜、用细节代替直白信息、角色说话像说明书。
- 剧情吸引力：场面是否有目标、阻力、变化和读者期待；是否只是格式正确。
- Bug：时间线、空间、能力规则、资源数量、组织反应、信息已知/未知。
- OOC：角色欲望、底线、反应模式、说话方式是否偏离既定锚点。
- 文风贴合：是否贴近文风DNA，是否过度严肃文学、过度说明或过度平滑。
- 管线问题：问题来自 create 执行、plot施工卡、world设定、文风DNA，还是用户输入不足。

## 输出原则

- 先给总判断：可读、勉强可用、需要返工、严重跑偏。
- 总判断必须给“爽文兑现等级”和“AI味等级”；这两个等级优先于 bug/OOC。
- 问题按严重度排序，不平均用力。
- 每个严重问题必须包含：位置、问题、为什么伤害阅读、建议交给哪个环节处理。
- 每个爽文/AI味判断必须有正文证据或明确位置，不允许只写“感觉不够爽/有AI味”。
- 不直接改正文；如用户要求修改，再转 revise/create。
- 不把个人偏好包装成硬规则；主观体感要标注为体感。

## 红线

- 不替 plot 重新排剧情。
- 不替 revise 写修订稿。
- 不用“整体还可以”糊弄，必须指出最影响阅读的前 3 个问题。
- 不只查格式和设定一致性；必须检查读者体感和爽文商业性。
- 不为了找问题而否定有效场面；保留可用资产。
