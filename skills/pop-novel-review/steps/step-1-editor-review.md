# Step 1 - Editor Review

目的：像网文编辑和真实读者一样读正文，判断这段文字是否好看、爽不爽、像不像 AI、有没有 bug/OOC，并定位问题来自哪个上游环节。

## 固定审稿 SOP

不得自由发挥散评。每次审稿按四遍走：

1. 裸读体感：只读正文，不看上游资料，记录哪里想继续看、哪里想跳读、哪里像 AI、哪里不爽。
2. 爽文兑现审计：按“压迫/诱因 → 主角主动 → 阻碍升级 → 误判/信息差 → 可见反馈 → 可数收益/损失 → 章末追读”检查正文里是否真实发生。
3. AI味模式扫描：按 AI味评分表逐项找证据，区分“语言像AI”和“剧情施工像AI”。
4. 连续性检查：只对 S/A 问题和用户关注点读取必要上游资料，查 bug/OOC/施工卡偏离。

输出时按问题严重度汇总，但不得跳过第 2、3 遍。爽文兑现和 AI味是默认主轴。

## 资料读取

必须读取：

- 待审正文原文。

按需读取：

- 文风DNA：检查文风贴合和 AI 味。
- 幕纲/章级施工卡/章节运行日志：检查 create 是否按施工卡执行。
- 角色储备池/本卷选角表/角色锚点：检查 OOC。
- 设定账本/世界宪法/动态升级表/金手指：检查规则 bug。

不要为了审一章正文加载全项目资料。先按问题域取最小资料，再追加。

## 子 Agent 审读

若当前环境支持子 agent，主审 agent 发起独立审读，不泄露自己的判断。

### 爽文/AI味审稿员

只从正文阅读体验出发，回答：

- 最有效的爽点在哪里，为什么有效。
- 最失败的爽点在哪里，是没有压迫、没有主动、没有反馈，还是没有收益。
- 哪一段最像 AI 写的，属于哪种 AI味模式。
- 哪一段最无聊或最假紧张，为什么。
- 读完是否想看下一章，卡在哪里。

必须按下方两张评分表打分。打分用 0-5 分，0=完全缺失，3=勉强有效，5=强兑现。

### 连续性审稿员

对照必要上游资料，回答：

- 有没有设定 bug、数值 bug、时间线 bug、信息差 bug。
- 有没有 OOC：欲望、底线、反应模式、口吻是否偏。
- create 有没有漏掉 plot 施工卡要求。
- 问题应该回 create、revise、plot、world，还是只需用户确认。

如果不能使用子 agent，主审 agent 也必须分两遍审：第一遍只做读者体感，第二遍才查资料。

## 问题分级

- S级：正文不可用。核心事件、爽点、角色或设定严重跑偏，建议返工。
- A级：影响阅读快感。需要重写段落或回 create 补动作/反馈/压力。
- B级：局部瑕疵。可由 revise 修句子、节奏、衔接。
- C级：记录观察。不急于处理，只作为后续注意项。

## 归因

每个 A/S 问题都必须归因：

- create 执行问题：施工卡明确，但正文没写出来。
- revise 可修问题：句子、节奏、衔接、信息密度问题。
- plot 问题：施工卡本身平庸、缺爽点、缺外部燃料、角色功能不清。
- world 问题：设定浅、角色锚点弱、金手指或势力规则支撑不足。
- 输入缺口：用户目标或参考资料不足，需补信息。

## 爽文评分表

| 项 | 0分表现 | 3分表现 | 5分表现 |
| --- | --- | --- | --- |
| 压迫/诱因 | 没有明确压力或欲望 | 有压力但弱/慢/抽象 | 压力具体、紧迫、能逼主角行动 |
| 主角主动 | 主角被推着走/旁观 | 有动作但选择感弱 | 主角主动下注、判断、冒险或设计 |
| 误判/信息差 | 没有人低估/错判 | 有错判但反馈弱 | 错判清楚，读者期待打脸或反转 |
| 阻碍升级 | 顺滑完成/只有说明 | 有阻碍但不疼 | 阻碍升级迫使主角加码 |
| 可见反馈 | 没有他人反应/结果 | 有反馈但泛泛 | 围观、敌人、同伴或系统给出明确反馈 |
| 收益/损失 | 没有可感知收益 | 有收益但模糊 | 资源、地位、信息、能力或关系明确变化 |
| 章末追读 | 没钩子/自然停顿 | 有悬念但弱 | 新威胁、新收益或新误判让人想点下一章 |

判定规则：

- 任一正文若“收益/反馈/主角主动”有两项低于 3 分，爽文兑现不得高于“勉强”。
- 如果压迫很强但没有兑现，要标记“严肃文学式压抑”或“虐而不爽”。
- 如果只靠旁白说“他赢了/众人震惊”，但没有场面反馈，要标记“声明式爽点”。

## AI味评分表

| 模式 | 典型表现 | 伤害 |
| --- | --- | --- |
| 抽象总结 | 用“这意味着/某种意义上/他终于明白”替代事件 | 读者看不到动作和结果 |
| 情绪标签 | 直接写愤怒、震惊、压迫感，但没有触发动作 | 情绪像贴标签 |
| 假紧张 | 反复写危险、沉默、空气凝固，但风险不可数 | 紧张没有抓手 |
| 机械转折 | “然而/但/与此同时”频繁推动，缺少因果 | 场面像流程拼接 |
| 严肃文学空镜 | 雾、光、气味、沉默、手指、铁锈等细节承担过多信息 | 爽点被氛围稀释 |
| 说明书对白 | 角色用对白解释设定、动机、规则 | 人物不像人，像工具 |
| 低信息重复 | 多句表达同一层意思，事件没有新增 | 字数涨，阅读快感不涨 |
| 过度平滑 | 冲突不尖锐，敌人不蠢不坏，收益不外显 | 打脸和反转被磨平 |

判定规则：

- 同一模式出现 3 次以上，AI味至少 B 级。
- “严肃文学空镜”如果替代了主角动作或爽点反馈，按 A 级处理。
- “说明书对白”如果承担关键设定，要判断是 revise 可修，还是 world/plot 没给可戏剧化表达。

## 输出

按模板输出：

- 总判断。
- 爽文兑现评分。
- AI味评分。
- 最影响阅读的 3 个问题。
- AI味诊断。
- 爽感诊断。
- Bug/OOC 诊断。
- 可保留资产。
- 处理建议。
- 下一章修正规则。

## 下一章修正规则

review 不是只做事后吐槽。若用户后续继续 create，报告末尾必须给出一组可直接交给 create 的规则：

- 下章必须保留的有效资产。
- 下章必须避免的写法偏移。
- 下章必须加强的爽点反馈。
- 下章必须检查的 bug/OOC 风险。
- 若发现章卡吞并、字数不足或严肃文学偏移，明确是回 plot 补施工卡，还是 create 按现有卡继续。

这些规则必须短、可执行、可复制到 create 输入包；不要写成长篇点评。
