# Step 2 - 执行路由

按 Step 1 的判断路由到执行 skill。你只做上下文组装和交接，不在这里重新设计流程。

## 路由输入

给执行 skill 的上下文保持完整但不臃肿：

- 当前目标：用户要得到什么产物。
- 商业网文目标函数：本次任务采用的读者快感优先级、主爽文引擎、禁用的严肃文学偏移、必须兑现的读者奖励；即使是局部修改也必须传入。
- 项目位置：项目根目录和目标输出路径。
- 前置材料：相关幕纲、运行日志、设定账本、文风DNA、上章末尾、活记忆。
- 资料路由：优先读取 `library/项目资料总目录.md` 判断本任务该读哪些文件；缺失时必须说明资料路由缺口。
- 公共资料库：`pop-trope-library` 路径、任务标签、候选案例、库缺口声明。
- 多元素整合范式：当输入包含多个种子、参考书/对标书、题材母版、现实机制或金手指方向时，向 `seed` 传入 `skills/pop-novel/references/多元素整合范式.md`。plot 不使用此范式，plot 使用自身的外部燃料原则。
- 复刻协议：当用户表达“像/照着/复刻/一比一/高保真/拿某体系/嫁接某世界观”等意图时，向执行 skill 传入 `skills/pop-novel/references/复刻协议.md`；先确定复刻深度和复刻位置，再进入多元素整合。
- 资料覆盖率协议：当参考作品、library 案例、wiki、百科、书评、拆书卡或模型常识被用于立项/设定/剧情判断时，向执行 skill 传入 `skills/pop-novel/references/资料覆盖率协议.md`；先声明资料覆盖范围、可用范围和不可外推项。
- 外部类型母版：读取 `library/设定账本/跨媒介母版雷达.md`；用户提到的影视、美剧、漫画、游戏、电影或经典桥段必须额外传给 plot；即使用户没提，plot 也要在燃料台内主动扫描外部母版。
- 约束：用户刚刚提出的偏好、禁止项、必须保留项。

## 执行 skill 加载规则

Pop 通常只强制注入 pop-novel。路由到执行 skill 时，不允许只凭本文件概述执行。必须读取本地对应 skill 文件：

- seed：读 `skills/pop-novel-seed/SKILL.md`，再按需读它的 `steps/step-*.md`。
- world-foundation：读 `skills/pop-novel-world/SKILL.md`，再按需读它的 `steps/step-*.md`。
- plot：读 `skills/pop-novel-plot/SKILL.md`，再读它声明的 step 文件。
- create/revise/review/arc：读对应 `SKILL.md` 和 step 文件。

若运行环境无法读取本地 skill 文件，必须在回复里声明“执行 skill 未加载，只能做对齐草案”，不得落正式产物。

## 执行模式判定

每次路由先判定模式，并把模式传给执行 skill：

| 模式 | 条件 | 处理 |
| --- | --- | --- |
| formal | 目标执行 skill 的 `SKILL.md` 与必要 step 文件已读取；入口材料满足该 skill 的门槛 | 可以生成正式产物 |
| draft | 目标执行 skill 已读取，但入口材料有缺口 | 只能生成草案或缺口报告；不得称完成 |
| trial | 目标执行 skill 未读取，或用户明确要求快速试写 | 只能输出试写/对齐稿；不得写入正式目录 |

执行凭证必须包含：

```yaml
execution:
  mode: formal|draft|trial
  loaded_skills:
    - pop-novel
    - <target-skill>
  loaded_steps:
    - <step-file>
  required_inputs:
    present: []
    missing: []
```

如果产物模板已有 YAML 记录，把 `execution` 放入模板；如果没有模板，在最终回报中列出。不得用“严格来自/按流程执行”等自然语言替代执行凭证。

## 执行规则

- 调 `seed`：先做调研水下层、资料覆盖率声明、种子展开、外部母版、故事概念候选 PK 和用户确认，再做长篇承诺书（商业爽点金字塔、世界规模假设、卷级质变草案、外部尺度燃料），再锁定新书 PRD，最后输出 `seed-to-world交接包.md`。如存在复刻/嫁接意图，seed 必须先产出复刻路由表，把 seed/PRD、world、plot、create/revise 分开；再做多元素整合。seed 只负责立项、长篇承诺和下游路由，不产 L1、金手指机制、文风DNA、章节链。正式产物写入 `library/设定账本/`，过程文件写入 `library/设定账本/运行/` 或 `library/设定账本/研究档案/`，不得写入 `卷纲/运行/`。
- 调 `world-foundation`：消费新书 PRD、长篇承诺书、研究档案和 seed-to-world 交接包，先建立 Foundation 外部燃料台和结构工程，产出商业爽点金字塔复核、世界尺度金字塔、势力战力资源总表、卷级质变阶梯、主角穿透模型；再建立 `library/设定账本/设定涌现/` 全量追踪目录，产出 L1 世界包、起点快照、世界宪法、动态升级表、主角引擎、角色储备池、金手指候选竞技、开局契约、文风DNA、读者表达偏好、Library 查询方向、跨媒介母版雷达和项目资料总目录。缺 Foundation 结构工程、角色储备池、读者表达偏好、项目资料总目录或设定涌现目录时不得称为底盘完成。
- 调 `plot`：先消费项目资料总目录、world 产物索引、Foundation结构工程、角色储备池、设定涌现目录、新书 PRD、L1 世界包、起点快照、世界宪法、动态升级表、主角引擎、金手指、开局契约、读者表达偏好和文风DNA，再做卷纲（卷定位候选 → 本书涌现燃料扫描 → 卷级外部燃料筛选 → 本卷边界 → 本卷选角表 → 可选剧情燃料池 → 幕功能位 → 篇幅预算），然后逐幕做幕纲（定位确认 → 选燃料 → 幕级外部燃料台 → 故事推演 → 章级施工卡/Create交接）。章级施工卡必须含章卡边界和微beat施工表。除非用户明确说“无需中途确认/全自动跑完”，否则 plot 至少在“卷定位确认门”和“幕定位确认门”阶段停下让用户确认。
- 调 `create`：让它根据 plot 章级施工卡、章节运行日志、幕级外部燃料台/案例消化摘要、读者表达偏好和爽文审计结果写正文，不要求它重新设计剧情。缺 plot 章级施工卡、章卡边界或微beat施工表时只能标记为降级草案。create 不得合并下一章事件。
- 调 `revise`：只在用户明确要改正文、润色、重写或按审稿意见修订时调用。局部改稿也必须传入商业网文目标函数和当前章主爽点；不得把冲突写含蓄、把收益写模糊、把主角主动性磨平。
- 调 `review`：只在用户要求审稿、检查 AI 味、爽不爽、bug、OOC、项目体检或“像编辑一样读”时调用；它只输出编辑诊断和处理建议，不直接改正文。
- 调 `arc`：让它事后复盘真实发生的剧情线和设定变化。

## 写作规划交互规则

- 剧情规划不是一次性黑箱生成。plot 必须把中间过程落盘到 `卷纲/运行/`。
- 不要把 plot 的中间产物私下交给子 Agent 或隐藏分支，除非用户明确要求独立验证或后台并行。写作规划需要用户能看见进度、能中途干预。
- “完整执行一场”默认含义是“按完整流程推进”，不是“跳过所有确认”。只有用户明确说“不要中途确认”才允许一口气跑完。
- 如果用户表达不满意、要求复盘、要求重跑，优先读取 run 日志和过程文件，定位流程断点，不要只加 checklist。

## Seed 立项硬要求

调 `seed` 时必须传入这条要求：seed 不是设定百科，而是研究档案 + 故事概念 PK + 新书立项 PRD + world 交接包。

seed 输出必须包含：

- `library/设定账本/研究档案/00-研究索引.md`
- `library/设定账本/研究档案/01-种子展开图.md`
- `library/设定账本/研究档案/02-外部母版卡.md`
- `library/设定账本/研究档案/03-多元素整合记录.md`（多元素输入时必需）
- `library/设定账本/研究档案/04-复刻路由表.md`（复刻/嫁接输入时必需）
- `library/设定账本/研究档案/05-资料覆盖率声明.md`（使用参考作品/library/wiki/模型常识时必需）
- `library/设定账本/运行/00-故事概念候选PK.md`
- `library/设定账本/运行/00b-长篇尺度与商业爽点金字塔.md`
- `library/设定账本/长篇承诺书.md`

- `library/设定账本/seed-to-world交接包.md`
- `library/设定账本/seed-产物索引.md`

seed 必须通过三道门：

- 研究水下门：必须有输入消歧、证据绑定、资料覆盖率声明、种子展开、外部母版卡；多元素输入时必须有多元素整合记录；复刻/嫁接输入时必须有复刻路由表。
- 概念 PK 门：必须有 2-3 个互斥故事概念和用户确认/自动选择理由。
- 长篇承诺门：必须有商业爽点金字塔、世界规模假设、卷级质变草案、外部尺度燃料。
- 爽文目标门：必须明确目标读者、主爽文引擎、核心读者奖励、主角获利方式、禁用的严肃文学偏移；没有这项不得进入 world。
- 新书 PRD 门：必须有篇幅体量、主角起点/终点、世界机器商业约束、元爽点和禁区，且尺度规划追溯到长篇承诺书。
- 交接门：必须明确不可涌现项、允许涌现项、证据索引和风险清单。

如果缺这些，pop-novel 不得说“seed 完成”，只能说“当前为 seed 草案，缺研究/概念PK/长篇承诺/PRD/交接门”。

## Plot 外部燃料台硬要求

调 `plot` 时必须传入这条要求：外部燃料不是可选装饰，而是幕纲质量的硬门槛。每个参考必须先分配资料角色，禁止平权混合。

plot 输出必须包含：

- 卷纲内的卷级外部燃料筛选：资料覆盖声明 + 资料角色表 + 燃料采用表（短复述 + 卷级用途 + 适合幕类型 + 风险）。
- 卷纲内的本卷选角表：从角色储备池切片读取 S/A/B/C，不临场发明主要角色。
- 卷纲内的可选剧情燃料池：只写可触发事件、主角操作点、适合幕类型、可外显爽点和风险，不排剧情顺序。
- 幕纲内的幕级外部燃料台：主母版当前单元短复述 + 本幕保留功能 + 本项目变形方式 + 辅助桥段表（最多1-2个）。
- 每章章级施工卡：章卡边界 + 宏观剧情骨架 + 微beat施工表 + 本章角色锚点；2500-3500字章节至少10-14个微beat，短章至少8-10个。
- 每章必须有爽文发动机实例：压迫/误判/诱因 → 主角主动动作 → 阻碍升级 → 可见反馈 → 可数收益或明确损失 → 章末追读。缺实例时不得交给 create。
- 资料角色分四类：`骨架源`（卷级顺序/商业波峰，只能1个主源）、`发动机源`（主角解决问题的方法，只能1个主源）、`模块源`（局部桥段，可少量）、`气质源`（质感氛围，不决定剧情）。
- 燃料必须带具体事件形状或短剧情复述，禁止只输出"信息差""压力升级""身份跃迁"等抽象机制词。
- 幕纲分章展开必须包含：主爽点、主角主动动作、可数收益/明确损失。
- `卷纲/幕纲-幕NNN-{标题}.md` 作为正式产物。

如果缺这些，pop-novel 不得在最终回报中称"幕纲完成"，只能说"当前为草案，缺外部燃料台/选角表/可选剧情燃料池/章卡边界/微beat施工表/质量门"。

## 禁止

- 禁止恢复历史写作环术语。
- 禁止把旧 changelog、废弃 references 当流程来源。
- 禁止在 pop-novel 内部抢执行 skill 的活。
