# pop-qidian-character · 起点角色管理引擎

> v1.0.0：新建。参考 pop-fanqie-character v1.0.1 适配起点架构。定位 Phase 3.5——world完成后、plot之前的必经加工站。消费骨架.md（动力引擎组成3众生攀登方式分层）+ 主角设计.md + 全书设定（势力.md+全书配角.md），按分幕设计出场角色清单按需批量塑造，产出 设计/角色库/角色库.md。角色库是write的角色唯一源。
> 管线位置：Phase 3.5——world完成后、plot之前。由pipeline触发。

---

## 做什么

输入：骨架.md + 主角设计.md + 设计/全书设定/（势力.md+全书配角.md）+ 创意.md + ch001.txt
输出：`设计/角色库/角色库.md`（角色总表+角色详细卡+增量更新日志）

**定位**：character不前置建库——world完成后根据全书设定规划分幕，按分幕设计出场角色清单按需批量塑造。character是world→plot之间的必经加工站，把骨架.md+主角设计.md+全书设定中的毛坯角色塑造成有灵魂的角色卡。

**与world的分工**：world做设定（地图/势力/危机/全书设定），产出设计/全书设定/；character做角色（性格/小传/成长数据），产出设计/角色库/角色库.md。

**与plot的分工**：character先跑，产出角色库；plot消费角色库写剧情白描。character不消费plot的产出（character在plot之前），而是根据全书设定预判分幕需求来设计出场角色清单。

## 核心约束

1. **每个具名角色必须标注攀登方式类型**——与骨架.md动力引擎组成3（众生攀登方式分层）对齐：有资源型/没资源型/爬到顶层型/掉下来型。角色的行为逻辑由攀登方式决定。
2. **角色的等级坐标必须与力量体系对齐**——角色的等级坐标引用骨架.md力量体系坐标系（主养成线阶位），不自行发明等级。
3. **角色库是write的唯一角色源**——write写正文时所有角色信息以角色库为准，不得在正文中创造与角色库矛盾的角色设定。
4. **不前置建库**——按分幕设计出场角色清单按需批量塑造，不预设配角/反派数量。

## 产出文件

`设计/角色库/角色库.md`，包含：
- **角色总表**：角色名/定位/阵营/攀登方式类型/等级坐标/引入章/最后出场章
- **角色详细卡**：每个具名角色一张卡
- **增量更新日志**：write引入新角色或更新状态时追加

### 角色详细卡字段

| 字段 | 内容 | 要求 |
|:--|:--|:--|
| 姓名 | 角色名 | 具名角色必须有 |
| 攀登方式类型 | 有资源型/没资源型/爬到顶层型/掉下来型 | 与骨架.md动力引擎组成3对齐 |
| 等级坐标 | 当前阶位（骨架.md主养成线阶位） | 与骨架.md力量体系坐标系对齐 |
| 性格 | 3-5个标签+行为表现 | 禁止泛泛词（"善良""勇敢"），必须是具体行为倾向 |
| 小传 | 200字背景故事 | 角色的过去决定了他现在为什么这样做 |
| 口头禅 | 标志性用语 | 写对话时直接调用 |
| 成长数据 | 起点/终点/关键转折 | 起点阶位→终点阶位+关键转折事件 |
| 出场记录 | 首次出场章/最后出场章/出场频率 | 高频/中频/低频 |

## 红线

1. **每个具名角色必须标注攀登方式类型，行为逻辑由攀登方式决定**——攀登方式类型与骨架.md动力引擎组成3（众生攀登方式分层）对齐。不标注=废卡。
2. **角色等级坐标必须与力量体系（坐标系）对齐**——角色等级引用骨架.md主养成线阶位，不自行发明等级。不对齐=废卡。
3. **角色库是write的唯一角色源，write不得自行发明角色**——write写正文时所有角色信息以角色库为准。write引入新角色或更新状态时必须同步回写角色库。
