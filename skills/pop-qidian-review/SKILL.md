# review

> 正文渲染完成后的编辑审核层。检查正文与plot大纲的gap，判断谁更好，输出决策+摘要+状态快照。
> 三技能管线（seed→plot-design→prose-render）之后的第四步。

---

## 做什么

输入：本章施工卡（plot-design产出） + 本章正文（prose-render产出） + DNA笔触要求
输出：审核报告（gap分析+决策） + 本章摘要 + 状态快照

核心原则：**不是强制正文服从大纲，也不是放任正文偏离大纲，而是作为编辑判断"谁更好"。**

---

## 怎么操作（SOP骨架）

### Step 1：加载审核上下文

读取：
- 本章施工卡（来自plot-design的幕纲）
- 本章正文（来自prose-render的渲染产出）
- 种子文档（风格关键词/不可牺牲项/禁区）
- 流派skill的笔触DNA（如 `prose-render-dndlike/dna/深渊主宰-笔触DNA.md`——叙事距离/面板格式/章节过渡/战斗模式分层/POV规则等）
- 种子文档的不可牺牲项与禁区（用于红线检查）
- 前一章状态快照（如有——用于衔接检查）
- 世界圣经活跃层（本幕的NPC/地点/组织/风俗/资源——检查正文是否与活跃层一致）

### Step 2：逐beat对比——正文 vs 施工卡

将施工卡的7节拍硬锚点和微beat施工表，与正文实际内容逐项对比。

对比方法：
1. 施工卡的每个节拍/beat，在正文中找到对应段落
2. 标注匹配状态：✅匹配 / ⚠️偏移 / ❌缺失 / ➕新增

输出gap清单：
```
| beat# | 施工卡内容 | 正文实际 | 状态 | gap类型 |
|-------|-----------|---------|------|---------|
| 1 | 催收员隔门喊话 | 催收员隔门喊话 | ✅ | — |
| 7 | 决定今晚动手→磨刀 | 暗杀执行→翻尸体→面板结算 | ⚠️ | 提速（吞并ch04） |
| + | — | 索伦去十字巷获取情报 | ➕ | 新增（大纲无） |
```

### Step 3：gap分类与质量判断

对每个⚠️/❌/➕的gap，进行分类和判断：

#### gap分类

| 类型 | 含义 | 典型场景 |
|------|------|---------|
| **提速** | 正文把后续章节的内容提前写了 | ch03写了ch04的暗杀执行 |
| **延迟** | 正文在一个beat上花了过多篇幅，没写完本章该写的 | ch02花3000字写面板激活，没写到门上催收刻痕 |
| **偏离** | 正文走了和施工卡不同的方向 | 施工卡写"暗杀"，正文写了"谈判" |
| **遗漏** | 正文跳过了施工卡中的某个beat | 施工卡有"翻尸体发现纸条"，正文没写 |
| **新增** | 正文写了施工卡中没有的内容 | 正文新增了一条"格里芬势力线" |

#### 质量判断（核心步骤）

对每个gap，回答一个问题：**正文版本更好还是大纲版本更好？**

判断维度：
1. **叙事自然度**：正文版本是否比大纲分章更自然？（如暗杀布局+执行在一个叙事流中更自然→正文更好）
2. **读者体验**：正文版本是否给读者更好的阅读体验？（如吞并后节奏更紧凑→正文更好；如吞并后信息过载→大纲更好）
3. **爽感完整性**：正文版本是否保持了爽感的完整闭环？（如把暗杀和面板结算拆开=爽感断裂→大纲有问题）
4. **后续影响**：如果采纳正文版本，后续施工卡需要怎么调整？调整成本是否可接受？
5. **字数合理**：正文是否在3000-5000字范围内？如果因吞并导致超标→需要权衡

判断结论（每个gap一个）：
- **正文更好**：采纳正文，调整后续大纲
- **大纲更好**：标记正文需要修改，附修改建议
- **各有优劣**：标注优劣，建议人工裁决

### Step 4：红线检查

检查正文是否违反硬性红线（不管gap判断结果，红线违反必须修改）：

| 红线 | 检查方法 |
|------|---------|
| 隔离红线 | 正文中是否出现原作专有元素（非同人测试模式下） |
| 篇幅红线 | 正文字数是否在DNA要求的范围内 |
| 主动行为红线 | 本章是否有≥1个主角主动行为 |
| 面板格式红线 | 面板是否用引号包裹系统消息嵌入正文（非代码块） |
| 设定强调纪律 | 同一设定约束是否在一章中提及>1次 |
| 战斗多样性红线 | 是否连续3章使用同一种战斗模式 |
| 三层世界红线 | 正文中出现的地名/NPC/组织/风俗是否在活跃层有记录？正文是否现编了未落盘的世界细节？ |

### Step 5：笔触质量审核

对照DNA笔触要求，检查正文的笔触符合度：

| 维度 | 检查内容 |
|------|---------|
| 叙事距离 | 是否以主角视角为主体（≥60%）？镜头切换是否有标志且快速切回？ |
| 对话引导词 | 是否使用DNA指定的引导动词？（如深渊主宰DNA用"道"不用"说"） |
| 面板弹出方式 | 是否作为意识流数据流嵌入正文？是否不停顿动作节奏？ |
| 章节过渡 | 是否使用了正确的过渡模式？是否连续两章使用相同过渡？ |
| 感官序列 | 新场景首次出场是否使用了感官序列？非新场景是否避免了感官序列？ |
| 情绪收束 | 杀完人后是否有冷评收束（或首次杀人的递减式心理波动）？ |
| 参照物梯子 | 面板数值弹出时是否用参照物建立认知标尺？ |
| 主角定位 | 面对威胁时是否先算后动？情绪是否通过瞳孔/呼吸/手指表达而非面部？ |

### Step 6：输出审核报告

```
## 审核报告：ch00X

### 1. gap清单
[Step 2的逐beat对比表]

### 2. gap分析与决策
[每个gap的分类+质量判断+决策]

### 3. 红线检查
[通过/不通过，不通过项标注]

### 4. 笔触质量
[各维度符合度]

### 5. 总体决策
- 正文采纳 / 正文需修改 / 人工裁决
- 如果采纳正文：后续施工卡调整建议
- 如果需修改：修改项清单

### 6. 本章摘要
[2-3句话概括本章发生的事件，用于后续章节的剧情摘要]

### 7. 状态快照
[本章结束时的结构化状态]
- 主角面板：职业/等级/生命值/经验值/杀戮经验/核心属性
- 主角位置：
- 持有物品：
- 已建立情报：
- 已传递读者信息：
- 本章结尾场景：
- 下章起点：
- 活跃层新增项：本章正文中出现的、需要追加到活跃层的NPC/地点/组织/风俗
```

## 红线

1. **gap必须逐beat检查**：不能只看总体感觉，必须逐个beat对比。跳过beat对比=不合格。
2. **质量判断必须有理由**：每个"正文更好"或"大纲更好"的判断必须附理由（哪个维度更好、为什么）。无理由的判断=不合格。
3. **红线违反不可妥协**：即使正文在叙事上更好，如果违反了硬性红线（如面板用代码块而非引号），仍需修改。
4. **摘要必须客观**：摘要只记录"发生了什么"，不评价"写得好不好"。
5. **状态快照必须准确**：状态快照的数据必须和正文中的面板数据一致。如果正文有属性偷跑，快照标注"正文数据异常"。

## 速查表

### gap决策速查

| gap类型 | 正文更好的信号 | 大纲更好的信号 |
|---------|--------------|--------------|
| 提速 | 叙事流更自然、爽感闭环更完整 | 吞并后信息过载、字数超标、后续章节无内容可写 |
| 延迟 | 该beat值得展开（如首次面板激活） | 拖沓注水、读者会跳读 |
| 偏离 | 正文方向更有趣/更符合角色 | 偏离破坏了卷纲/幕纲的结构设计 |
| 遗漏 | 遗漏的beat本来就不需要 | 遗漏的beat是关键信息/爽点 |
| 新增 | 新增内容丰富了世界观/角色 | 新增内容偏离主题/增加无关支线 |

### 审核流程速查

| 步骤 | 输入 | 输出 | 耗时参考 |
|------|------|------|---------|
| Step 2 逐beat对比 | 施工卡+正文 | gap清单 | 核心 |
| Step 3 gap判断 | gap清单+DNA+后续大纲 | 逐gap决策 | 核心 |
| Step 4 红线检查 | 正文+种子禁区 | 红线通过/不通过 | 快速 |
| Step 5 笔触审核 | 正文+DNA | 笔触符合度 | 快速 |
| Step 6 输出报告 | 全部 | 报告+摘要+快照 | 自动 |

## 使用方式

### 在prose-render之后调用

每章正文渲染完成后，调用review skill进行审核：
1. 输入施工卡+正文+DNA
2. 输出审核报告+摘要+状态快照
3. 根据决策：正文采纳→进入下一章 / 正文修改→返回prose-render重写

### 状态快照的用途

状态快照存入 `审核/ch00X-状态快照.md`，在下一章渲染时作为"前章状态快照"传入user prompt，替代完整前文正文——解决信息量膨胀问题。

### 摘要的用途

摘要在10章后开始使用——前10章用完整前文+快照，第11章开始用"近5章摘要+前章快照+前章结尾800字"替代完整前文。

## execution.mode

| mode | 条件 | 落盘 |
|------|------|------|
| `formal` | 必读输入齐全且达标：本章施工卡、本章正文、DNA笔触要求、种子文档、前一章状态快照（如有） | 正式落盘审核报告 + 状态快照 + 更新 current-state.md |
| `draft` | 必读输入有缺口但可补全继续（如缺少前章状态快照，但正文和施工卡齐全） | 产出标记 draft，落盘 |
| `trial` | 快速试做，不称完成（如仅快速浏览正文给出初步判断） | 不落盘，仅对话内 |

formal 必读输入清单：
- 本章施工卡（来自 plot-design 的幕纲）
- 本章正文（来自 prose-render 的渲染产出）
- DNA 笔触要求
- 种子文档（风格关键词/不可牺牲项/禁区）

## current-state 更新（吸收自 pop-qidian）

审核完成后，除了输出审核报告和状态快照，还需更新 current-state.md：

### 更新规则
- 将状态快照（面板/位置/物品/情报/已传递信息/结尾场景/下章起点）压缩进 current-state.md 的对应字段
- 将下一章的硬推进信息写入 current-state.md 的"下一章硬推进"字段
- 将伏笔债务更新写入 current-state.md 的"伏笔债务"字段
- 将本幕活跃层的索引（已建NPC/地点/组织清单）写入 current-state.md 的"活跃层索引"字段
- 活跃层索引帮助 write 知道哪些世界细节已落盘可用，哪些需要通知 plot 补建
- current-state.md 总字数控制在 1000-2500 字

### 历史层规则
- 每次更新 current-state.md 前，先归档旧版到 `压缩归档/current-state-{YYYYMMDD}-{章位}.md`
- 每次审稿追加一段到 `review-沉淀.md`（append-only，不删改历史），格式：
  ```
  ## {日期} · 第{N}章审稿
  - 总判断：{通过/返工/重写}
  - 3个主要问题：1... 2... 3...
  - current-state 是否更新：是/否
  - 修正规则：{如有则记录}
  ```

### 沉淀分流
审计发现分两类处理：
- **更新库文件**：设定库/人物库/剧情线/燃料库 的待确认修改，标注后通知
- **压入 current-state**：下一章需要的即时信息，直接写入 current-state
