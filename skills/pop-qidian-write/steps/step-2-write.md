# Step 2：写正文 + 落盘 + 创作记录

本步骤消费本章写作包，产出正文并落盘，再用 templates/chapter-record.tpl.md 回复。引用 PRD §4.3（命名）、§4.7（回复格式）。

## 1. 正文规则

- 开场 120 字内出现目标、异常、压力、反馈或主动动作之一。
- 主角至少完成一次动作闭环：压力/诱因 -> 判断 -> 主动动作 -> 阻碍升级 -> 可见反馈 -> 收益/损失 -> 新压力。
- 本章必须兑现 current-state 的硬推进，不能只铺氛围。
- 可用燃料必须落到动作、物件、权限、组织反应、技术细节、钱、身份或信息差里。
- 关键爽点必须外显（旁人反应、资源变化、局势翻转、敌人误判等），不能只由作者总结"他赢了/他变强了"。
- 不提前吞并 current-state 标为中期/后期的事件。
- 结尾必须留下主动钩子或威胁逼近，不用自然停顿收尾。

## 2. soul 落地

| soul 项 | 在正文中的体现 |
| --- | --- |
| 主卖点/元爽点 | 本章的读者奖励，不是口号 |
| 主角主动方式 | 主角必须做出选择或动作 |
| 爽点外显方式 | 旁人反应、资源变化、局势翻转、敌人误判 |
| 叙事人格 | 旁白站位和判断顺序 |
| 句子气口 | 长短句、独句段、信息先后 |
| 信息释放 | 先给结果还是先给过程，哪些暂不解释 |

本章写作包里选定的 3-5 条 soul 约束必须能从正文里看出来。

## 2.5 文风DNA落地（启用时）

文风DNA不是事后润色，而是参与本章生成：

| DNA 项 | 在正文中的体现 |
| --- | --- |
| 笔触目标 | 叙事距离、句式、段落呼吸、感官顺序、对话引导方式 |
| 章内套路 | 起笔、第一信息变化、压力/诱因、主角动作或判断、阻碍升级、可见反馈、章末钩子 |
| 可见反馈 | 敌人失态、旁观者改判、资源/信息变化、组织/权限变化、威胁逼近 |

执行要求：

- 每个启用的章型至少改变一处"章内展开方式"，不能只替换形容词。
- 每章必须有一条可见反馈：旁观者反应、敌人失态、规则揭露、资源变化、组织重新判断、威胁逼近等。
- 强嫁接时只迁移章内套路和笔触手感，不迁移原作专有名词、设定词、角色口癖。
- 禁止把 DNA 源中的开篇专用机制、特殊高光章、丑角台词当成全书通用逻辑。

## 3. 落盘（修复问题 7）

正文写入项目目录，路径固定（PRD §4.1 骨架、§4.3 命名）：

```text
涌现/正文/{书名}-第{N}章-{标题}.txt
```

- 路径固定为 `涌现/正文/`，不得写到项目根或其他目录。
- 文件名格式固定，不得自创别名。
- 对话只回摘要、钩子和创作记录，不把正文全文放进对话。
- trial 模式不落盘，仅对话内（PRD §4.5）。

## 4. 新增事实处理（修复问题 10）

write 不在 write 内维护任何库文件（设定库/人物库/剧情线/review-沉淀）。写作过程中产生的新增事实、人物变化、设定变化：

- 只在对话创作记录里列"新增事实待 review"清单（声明）。
- 不落盘到任何库文件。
- 由后续 pop-qidian-review 读取本章正文 + 创作记录后，决定是否落库。

这消解了"write 无库读取权却要判断新增"的矛盾：write 只声明，不落库。

## 5. 回复

采用 PRD §4.7 统一回复骨架，专属产出摘要使用 `templates/chapter-record.tpl.md` 模板。

必填项：

- 本次采用 skill：pop-qidian-write
- execution.mode
- 第 N 章 - {标题} 已写入 {文件名}
- 摘要
- 章末钩子
- 创作记录：已读输入 / 兑现的硬推进 / 使用的燃料 / 执行的 soul 约束 / 执行的文风DNA约束 / 新增事实待 review 清单 / 下一步建议执行 review

## 6. 下一步

正文落盘并回复后，**必须执行 `pop-qidian-review`**。

- review是write的硬后继——未执行review不得写下一章
- review负责：四维审核 + current-state更新 + 小说快照更新（全书累计视图）
- write只在创作记录中声明"新增事实待review"清单，不自行落盘任何库文件
- pipeline Phase 5完成后自动路由到Phase 6 review，agent不得跳过
