# 玄鉴仙族拆解 · 坑点沉淀

> 基于第1-500章拆解过程中实际踩过的坑，按出现顺序排列。
> 每个坑点包含：现象 → 根因 → 已采取的措施。

---

## 坑①：用二手资料代替原文

**现象：** 第一次拆书产出是基于舆情报告+HTML格言集写的，没有打开txt原文。写出来的"田芸说稀罕是全书最温暖的两个字"是从二手资料抄的。

**根因：** 对"原文"的敬畏不够。觉得舆情报告已经总结好了，没必要自己读。

**措施：** 方法论写入核心纪律"不准用二手资料代替原文"。舆情报告/wiki/书评只能用于校对，不能用于替代。

---

## 坑②：没读完就写

**现象：** 第二次迭代读了80章（第1-86章），第87-100章没读，但产出标注的是"前100章"。李项平之死（第100章）这个全书级事件被完全遗漏。

**根因：** 赶进度。觉得"大概够了，先写吧"。

**措施：** 每份文件开头标注精确阅读范围。`_章节手记.yaml` 的文件头声明 "第1-100章" 或 "第1-300章"。读多少写多少。

---

## 坑③：产出形式是"文章"而不是"信息"

**现象：** 第二次迭代产出是"深度拆书文章"——有开头结尾、有主观判断、有"我觉的"。不可复用，查信息得从头翻。

**根因：** 习惯性认为"拆书=写一篇好文章"。没有意识到"排好序的信息"比"写好的文章"有价值。

**措施：** 转向 YAML 结构化实体文件。每一条记录独立可查，不依赖上下文。

---

## 坑④：预设聚类类别

**现象：** 第三次迭代定了7个聚类类别（器物谱/人物决策/势力演变/设定规则/暗线伏笔/叙事手法/索引），按框子填数据。叙事手法笔记一次没用过——它是我预设的，实际拆的过程中没有这个需求。

**根因：** 以为"所有小说都需要同样的维度"。

**措施：** 方法论改为"类别是读的过程中涌现的，不是读之前预设的"。实体模板体系也标注了"根据书类型调整"。

---

## 坑⑤：100章时设计 schema 太早

**现象：** 第十次迭代读了100章就设计了完整的10种实体模板。读到300章发现少了仙基变异/箓气追踪/法鉴能力分类。读到500章发现少了天命之子机制/洞天体系。

**根因：** 核心概念在100章时根本没出现。李渊蛟筑基仙基变异在第377章，步梓搜魂在第348章，李周巍出生在第475章。100章时无法预见这些。

**措施：** 将实体体系分为两层：
- **L1（稳定层）**：角色/事件/地域/物品——几乎所有叙事作品都需要，可以提前设计
- **L2（书特定层）**：仙基/箓气/天命之子/功法品级——根据书的特点在阅读过程中涌现

L1 提前设计，L2 留空，读到足够量再填充模板。

---

## 坑⑥：覆盖而非追加

**现象：** 每次读完新一批都直接重写旧文件。01_角色.yaml 在第100章时有11条，300章时直接覆盖重写。失去了第100章时对这本书理解的快照。

**根因：** 追加比重写麻烦——要判断哪些是新增、哪些是变更、哪些是删除。

**措施：** 约定追加模式。新批次的内容只在文件末尾追加，不修改已有条目。如需修正旧条目，在末尾追加修正条目并标注"修正：原条目X有误"。

---

## 坑⑦：跨文件 ref 链没有断裂处理

**现象：** 一个角色状态变更（李通崖第100章标记为存活，第320章已死），所有引用他的文件（战斗记录/事件/功法）不会自动更新。500章后回查300章写的文件，出现"李通崖存活"和战斗记录里"李通崖已死"的不一致。

**根因：** ref 引用是单向的，没有反查机制。角色状态变更时，引用方不会收到通知。

**措施：** ref 引用标注目标实体的版本号或章节号。例如 `ref:LJC-003@ch300` 表示引用的是第300章时的状态。当阅读范围推进时，明确哪些引用需要更新。

---

## 坑⑧：子代理阅读颗粒度不一致

**现象：** 第101-300章和301-500章是子代理并行读的。不同子代理输出的质量差异大——有的记了每章细节，有的只记了概要。三个代理对"重要事件"的判断标准不一致。

**根因：** 指令中"重要事件"表述太模糊。没有给子代理具体的记录模板和优先级规则。

**措施：** 设计标准化章节摘要协议（见下文）。

---

## 坑⑨：静态字段 vs 时间序列

**现象：** 角色模板的 `cultivation_realm.current` 只能保存最后修为状态。500 章跨度里一个角色修为可能变 5 次，中间状态丢失。势力的 `members_realm`、地盘的 `territory` 同样。

**根因：** 设计时默认字段是静态值，没有考虑时间维度。

**措施：** 对时间敏感字段明确标注 `chapter_as_of`。修为突破增加 `breakthrough_chapter` 字段记录时间线。不追求"当前值"，而是记录"在某一时间点是什么值"。

---

## 坑⑩：Phase A 的 desc 只有首次出现信息，丢失时间维度

**现象：** `_名词索引.yaml` 里李通崖的 desc 是"李木田次子，勇敏果断敢打敢拼，镇守冷静，后娶柳柔绚，突破练气。"——这条信息来自第2章首次出场时的描述。第320章他死了，第212章他筑基了，这些后续关键状态变化在名词索引里没有任何记录。同一个角色"李通崖"在索引里只有一条记录。

**根因：** Phase A 被设计为 append-only 的名词提取，每条名词只存"首次出现章节"。它天生没有"同一主体在不同时间点的状态变化"这个维度。合并去重时，后一批次读到 "李通崖" 不会追加更新，因为 key 已经存在了。

**措施：** 
- 引入 `events` 表记录状态变更事件（死亡/突破/联姻/获得法宝/加入势力）
- 从 events 推导 `entity_state`（最新状态）
- 名词索引保留"首次出现"的 snapshot 信息
- Phase A 子代理除了输出名词索引，还必须输出**结构化事件**

---

## 坑⑪：实体间横向关系完全没有建模

**现象：** 角色与势力、角色与器物、角色与角色之间的关系完全没有被记录。李玄锋和李青乌弓是什么关系？李渊蛟和萧归鸾是什么关系？李家与萧家是姻亲还是仇敌？这些信息散在原文里，但没有任何结构化记录。

**根因：** 名词索引是按"类型"分桶的（人物一个桶、器物一个桶、势力一个桶），桶之间没有连线。Phase A 只机械提取"名词+desc"，Phase B 聚类只统计"各个桶装了多少条"，没有做桶与桶之间的关联分析。

**措施：**
- DB schema 增加 `relationships` 表： `subject → relation_type → object → ch`
- 例：`李玄锋 → 持有 → 青乌弓 → 第X章`
- 例：`李渊蛟 → 娶 → 萧归鸾 → 第268章`
- 子代理在 Phase A 输出时增加 `relationships` 字段
- Phase B 聚类分析中增加关系网络扫描

---

## 坑⑫：HTML 生成后没有验证即交付

**现象：** 生成的"数据库浏览器.html"交付后无法使用——要么 CDN 加载失败不报错、要么 JSON 解析错误页面空白、要么拖拽功能在本地不可用。两次生成都需要重新发版。

**根因：** HTML 生成脚本执行后只检查了文件大小存在，没有打开文件验证：
1. JSON 数据是否完整内嵌（被截断）
2. HTML 语法是否完整闭合（`<script> 到 </html>`）
3. 有无外部依赖阻断加载
4. 侧栏预置查询是否能正常工作

**措施：** 生成 HTML 后必须跑验证脚本：
1. 验证 JSON 可以 `json.loads()` 解析
2. 验证无外部 CDN 引用（`https://` 只允许 Google Fonts 可选项）
3. 验证 `</html>` 结尾存在
4. 验证 `var DB =` 和 `var QUERIES =` 声明存在且格式正确
5. 验证可以在浏览器中直接打开（无 require/fetch/XHR 等运行时依赖）

---

## 坑⑬：子代理输出质量无验收

**现象：** 第101-200章的子代理只产出0条新名词（正常应有100+），说明漏了大半内容——但没有任何告警，数据直接合进索引了。

**根因：** Phase A 跑完后没有置信度检查。不知道"这个代理是否读完了、读全了"。

**措施：**
- 每批子代理输出后跑验收脚本（`方法/phase_a_验收脚本.py`）
- 检查项：名词总数是否在期望范围 / 类型分布是否异常 / desc是否有主观词 / events条数是否合理
- 异常标记为"可能漏提"并重跑

---

## 坑⑭：同一实体的多个事件无冲突检测

**现象：** 两个子代理对同一件事说法不一致（A说第405章突破，B说第406章突破）。events 表全收，没有任何冲突标记。

**措施：**
- 同一 subject + 同一 event_type 的事件如果 chapter 相差 ≤ 3 章，取较早的，较晚的标记为"冗余"
- events 表保留一条 source_of_truth 字段标注来源优先级

---

## 坑⑮：多人参与的大事件无法建模

**现象：** "青松观洞天之战"涉及6+个人。events 表只有 single subject，一条事件只能记一个主体。

**措施：** core_events 表增加 `participants` 字段（JSON数组），存所有参与方。查询时展开显示。

---

## 坑⑯：手工勘误 vs 自动化产出的冲突

**现象：** 用户发现 DB 里某条数据错了（如李通崖出场在第2章而非第3章），手工修正后再次跑自动化批量时可能又被覆盖回来。

**措施：** core_corrections 表记录所有手工修正。自动化批量时优先检查 corrections 表，跳过已修正条目。

---

## 坑⑰：HTML 数据量膨胀导致加载慢

**现象：** 500章数据的 JSON 已达 465KB，读到 2000 章预估 2MB+。JS 的 filter/sort 在 2000+ 条数据上会明显卡顿。

**措施：**
- HTML 采用分页加载：首次只加载预览数据（100条），点击"加载更多"再拉取
- 完整查询走 SQL 标签页

---

## 坑⑱：跨书复用时 schema 硬编码

**现象：** ext_realms 的 `境界` 字段、events 的 `修为突破` 类型是为修仙类网文量身定做的。换侦探小说/历史小说用不上。

**措施：**
- schema 分层：**core 层**（角色/事件/地域/关系）通用层，任何叙事作品都能用
- **ext 层**（境界/功法/箓气/果位）书特定层，按需添加
- 新书只带 core 层，ext 层从零设计

---

## 坑⑲：事件有先后顺序但无因果关联（伏笔线/剧情线缺失）

**现象：** Phase A 的 events 表记录了 281 条"谁+做了什么"，但它们只是按 ch 排序的列表。"李通崖为什么在第320章去围杀摩诃？"——events 里有李通崖坐化的记录，也有李项平第100章被咒杀的记录。但没有一条数据告诉读者：**这两件事之间有因果关系**。

伏笔线、剧情线、因果链——这些读者真正关心的"故事结构"在现有数据中完全不存在。

**根因：** 当前 Phase A 的设计是"机械提取"，它天然不理解"因果"。一条事件记的是"X 发生在第N章"。但"X 导致 Y"的判断需要读完才知道，属于 Phase B 的分析任务：

```
Phase A（机械提取）:
  events: 李项平第100章被咒杀
  events: 李通崖第320章坐化
  → 两者在同一个列表里，但无关联

Phase B（聚类分析 — 人的判断）:
  发现：李项平死 → 李通崖变得保守 → 强攻摩诃重伤 → 坐化
  → 产出："因果链条: 李项平之死 → 李通崖性格转变 → 李通崖坐化"
```

**措施：**
- 明确将"因果关联"定为 Phase B 的任务，Phase A 只负责堆事件
- events 表预留 `caused_by` 字段（引用另一条 event 的 id），Phase B 填充
- Phase B 输出"剧情线/伏笔线"作为独立文件，每条线 = 一组按因果连接的事件链
- HTML 展示时支持"按线查看"模式（而非单纯按 ch 排序）
