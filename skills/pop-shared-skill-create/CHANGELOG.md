# CHANGELOG — pop-skill-create

## v5.3.0 — 2026-06-22

### 新增 ❌D15 红线：重要行为必须有 SOP/模板约束

**问题**：skill 中的关键产出环节（搜索、生成选项、质检、推导等）如果只有一句"agent 生成 X"，没有 SOP 或模板约束，产出质量完全靠 agent 发挥 — 时好时坏。

**新增 ❌D15**：skill 中的关键产出环节必须有对应的 SOP（references/，约束过程质量）或模板（templates/，约束产出格式和质量标准）。不能只有指令没有质量约束。

**来源**：从 pop-writer-creative 重构中提炼 — 联网搜索缺 SOP 导致搜索质量不可控，故事概念选项缺模板导致选项太 low。补了搜索SOP + 选项模板后质量立刻有保障。

## v5.2.0 — 2026-06-22

### 文件拆分原则沉淀

**新增 `steps/step-1-design.md` "文件拆分原则"段落**：

从 expert-writer 和 pop-writer-creative 重构经验中提炼的拆分规则：

| 层级 | 拆分策略 | 核心判断 |
|:-----|:---------|:---------|
| references/templates | 按需拆（越细越好） | 只在特定步骤查阅 → 拆。不拆 = 每次全读浪费 token |
| steps/ | 按独立阶段拆，不按步骤拆 | 有用户确认断点 → 拆。连贯推导链 → 不拆 |

**拆分决策流程**：step > 150 行 → 有用户断点？→ 按断点拆 / 有知识层可下沉？→ 下沉 / 都没有 → 不拆

**反面案例**（从实际重构中提炼）：
- ❌ 9 步推导链拆成 9 个文件 = 9 次工具调用 + 9 次注意力断点
- ❌ 7 类追问链留在 step 里 = 每次全读但只用 1-2 个
- ✅ 追问链下沉到 references/ = 按需加载
- ✅ 9 步链按 Step 3 用户断点拆成 2 个文件 = 自然暂停等用户

**行数约束修正**：原标准 "50-150 行/文件" 只约束 references/，steps/ 改为 "目标 50-150 行，不设硬上限 — 连贯推导链拆碎反而有害"

## v5.1.0 — 2026-06-22

### references/ vs templates/ 定位明确化

**问题**：多个 skill 同时有 `references/` 和 `templates/` 两个目录，但 pop-shared-skill-create 没有明确定位区别，导致模板文件和 SOP 文件混放。

**更新 `steps/step-1-design.md`**：
- 目录结构部分新增 `references/ vs templates/ 定位区别` 对照表
- 三层架构原则从三层改为四层（路由层/执行层/知识层/模板层）
- 速查表模板新增 templates/ 行
- 落盘检查点新增 templates/ 检查项
- WRONG 示例新增"模板放错目录"案例
- 新增禁止规则：模板文件不放 references/，SOP 文件不放 templates/

| 目录 | 定位 | 使用方式 |
|:-----|:-----|:---------|
| `references/` | 知识层 — 读后理解 | 阅读理解，指导操作 |
| `templates/` | 模板层 — 复制填充 | 复制→填充→产出文件 |

## v5.0.0 — 2026-06-22

### 全量重构：三层架构 + 注意力前置 + 读取协议

**核心问题**：v4.x 的 SKILL.md 有 127 行，混合了路由决策、执行细节、知识参考三层内容。agent 读完 SKILL.md 后注意力已耗散，反而忽略了最关键的路由信息。同时缺少读取协议——没有明确告知 agent 该用什么工具读文件，导致截断风险。

**重构原则**：

| 原则 | 说明 |
|:-----|:-----|
| 三层架构 | 路由层（SKILL.md ≤60行）→ 执行层（steps/）→ 知识层（references/） |
| 注意力前置 | 最关键的信息（红线、速查表）放在第一屏 |
| 读取协议 | 明确要求用 skill_view / Get-Content -Raw，禁止 Read 工具 |
| 截断检测 | 读取后对比 content.length vs file_size，不完整则回退重读 |

## v4.0.0 — 2026-06-14

### 设计规范 v4 — 对齐 03-pop-dna v4.0 / decon-* v1.0 文档范式

**核心问题**：v3.0 的 SKILL.md 产出模板过于宽松。速查表没有指定列格式，SOP 步骤缺乏 I/O 标注，缺少落盘检查和越界防护——导致新创建的 skill（如 decon-* 首批版本）在第一次写出后需要手动回补这些缺失要素。

**新增 5 条质量红线（❌9 ~ ❌13）**：

| # | 红线 | 说明 |
|:-:|:-----|:-----|
| ❌9 | 速查表缺列或无列名 | 必须有：「步骤/操作/读什么/产出什么/门禁」五列，「读什么」列必须是文件路径 |
| ❌10 | SOP 步骤缺 I/O 标注 | 每步必须标 **读什么/做什么/产出/❌门禁** |
| ❌11 | 无落盘检查点 | 产文件型 skill 必须列出逐条文件路径 |
| ❌12 | 无阶段边界越界检测 | 多 step skill 必须有跨步骤违规+越界检测表 |
| ❌13 | 下游 skill 缺消费关系行 | 有消费者的 skill 必须在速查表后标注「消费关系：产出→XX Skill →...」 |

**设计模式 Step 4 完整重写**：

- 速查表：从"有就行"到"强制五列格式 + 读什么列必须是文件路径"
- SOP 步骤：从"步骤名+描述"到"**读什么/做什么/产出/❌门禁** 四标签强制"
- 新增「落盘检查点」区块（从 OPC 范式引入）
- 新增「阶段边界越界检测」区块（从 OPC 范式引入）
- 新增「消费关系行」强制标注
- 新增 WRONG 4-7（速查表缺列/步骤缺 I/O/缺落盘/缺越界检测）

**评估模式维度重新加权**：

- 维度2（工作流清晰度）：15→12分，新增 I/O 标注检查
- 维度3（边界条件覆盖）：10→10分，新增越界检测表检查
- 维度4（检查点设计→检查点与门禁）：7→10分，新增落盘检查点检查
- 维度5（指令具体性）：15→10分，新增速查表文件路径检查
- 维度6（资源整合度）：5→10分，新增消费关系行检查
- 维度7（整体架构）：+边界声明检测
