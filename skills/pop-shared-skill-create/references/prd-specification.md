# PRD 文档写作规范

> 来源：prd 资料库全量整合经验，基于 29 篇 PRD 文档的通用模式提炼。
> 适用于所有新建 PRD 文档的格式标准。

---

## 一、首屏元数据块

每份 PRD 必须在文档开头包含以下元数据块：

```markdown
> 版本：v{主}.{次} | 日期：{YYYY-MM-DD}
> 状态：{草案/方案讨论/可执行/已实施}
> 分类：{规范标准/管线架构/问题诊断/参考资料}
> 关联：[关联文档名](相对路径) | 飞书原文：[链接](url)（如有）
```

## 二、核心段落模板

| 段落 | 必须写 | 可选 |
|:-----|:-------|:-----|
| 〇、一句话范围 | **3 行内说清：改什么、不动什么** | — |
| 一、背景/问题诊断 | 问题表（问题+表现+影响）或根因表（根因+症状） | 证据链引用 |
| 二、方案 | 改动点表（文件+修改类型+具体操作） | 代码示例 |
| 三、产出物规范（如果产文件） | 文件路径+内容结构+门禁 | 模板示例 |
| 四、不作的事 | ❌ 列表：明确告知不做什么 | — |
| 五、验收条件 | 表格：#+条件+验证方式 | 验收命令 |

## 三、证据链规范

问题诊断类 PRD 必须包含证据链：

```
### 证据 {N}：{一句话概括}
{数据/引用/会话摘录}
{分析/推演}
```

**三条标准：**
- **有数据**（代码行数、Token 数、占比、时间线）而不是"看起来很多"
- **有对比**（修前 vs 修后、正常 vs 异常）
- **可复现**（明确说明在哪个文件/run/session 中可查阅）

## 四、修复状态标记

问题/方案必须有状态标记：

```
✅ 已修复（{版本号}）
⏳ 待执行
❌ 不做（{原因}）
⛔ 框架层问题（{等待什么}）
```

## 五、交叉引用规范

- 首次提及外部 skill 时用全名：`pop-writer-plot`（不用缩写）
- 引用管线关系时指向全链路依赖图
- 引用问题时指向对应的自文档

## 六、可执行性自检清单

```
□ 读完后知道改哪些文件、怎么改
□ 读完后知道不改哪些文件
□ 读完后知道改完后怎么验证
□ 读完后知道这个方案依赖什么前置条件
□ 不含未定义的缩写或内部术语
```

## 七、文件命名规范

- 文件名用 kebab-case：`{序号}-{简短描述}.md`
- 序号用两位数，接续已有文件
- 描述 ≤ 5 个中文词，不对齐到具体 PRD 编号

## 八、归档规则

| PRD 类型 | 归档目录 |
|:---------|:---------|
| 规范标准 | `prd/00-规范/` |
| 管线架构 / 改造方案 | `prd/01-管线架构/` |
| 问题诊断 / 复盘分析 | `prd/02-问题诊断/` |
| 外部素材 / 排查手册 | `prd/03-参考资料/` |
| 待实施方案 | `prd/99-储备方案/` |
