# Step 3：评估模式 — Review 存量 Skill

> **前置条件：** 目标 skill 完整文件（SKILL.md + skill.json + CHANGELOG.md）。
> **产出：** 8 维评分卡 + 改进建议（P0/P1/P2）。
> **核心约束：评估不改代码。** 只评分和给建议，不直接改文件。

---

## ❌ 质量红线（评估模式）

| # | 红线 |
|:-:|:-----|
| ❌C1 | 不读完整文件就评分 — 必须先读 SKILL.md + skill.json + CHANGELOG |
| ❌C2 | 评估维度不全 — 8 个维度全部打分，缺一个标注 Not Rated |
| ❌C3 | 不输出评分卡 — 必须输出格式化总分表和短板分析 |
| ❌C4 | 效果维度不跑测试 — 维度 8 必须设计至少 2 个测试 prompt |
| ❌C5 | 改后不重新评估 — 改造后必须重新跑完整 8 维评分 |

---

## 评估流程（5 步）

### Step 1：读文件

**读什么：** 目标 skill 的 SKILL.md 全文、skill.json、CHANGELOG.md
**做什么：** 用 `skill_view` 或 `Get-Content -Encoding UTF8 -Raw` 读取（❌ 禁止 Read 工具）
**产出：** 一句话定位（skill 做什么、在哪条管线、当前版本号）
**❌ 门禁：** 不读完全文不打分。

### Step 2：设计测试 prompt

**读什么：** skill 的 Triggers 段落
**做什么：** 设计 2-3 个典型用户 prompt
**产出：** 测试 prompt 列表
**❌ 门禁：** 测试 prompt 交用户确认后再打分。

覆盖场景：
- **happy path** — 最常见的使用场景
- **边界场景** — 异常输入/缺文件/缺前置条件

### Step 3：跑 8 维评分

#### 结构维度（60 分）— 静态分析

| # | 维度 | 权重 | 评分标准 |
|:-:|:-----|:----:|:---------|
| 1 | Frontmatter 质量 | 8 | name 规范、description 含做什么+何时用+触发词、≤1024字符 |
| 2 | 工作流清晰度 | 12 | 步骤明确可执行、有序号、每步有 I/O 标注 |
| 3 | 边界条件覆盖 | 10 | 有异常处理表、有 fallback、有越界检测、不编造数据 |
| 4 | 检查点与门禁 | 10 | 关键决策前有用户确认、红线在第一屏、有落盘检查点 |
| 5 | 指令具体性 | 10 | 不模糊、有判断标准、可直接执行、速查表文件路径列存在 |
| 6 | 资源整合度 | 10 | references/scripts 引用正确、路径可达、有消费关系行 |

#### 效果维度（40 分）— 需要实测

| # | 维度 | 权重 | 评分标准 |
|:-:|:-----|:----:|:---------|
| 7 | 整体架构 | 15 | 结构层次清晰、不冗余不遗漏、三层架构、有边界声明 |
| 8 | 实测表现 | 25 | 用测试 prompt 跑一遍，对比 with-skill vs baseline |

#### 评分规则

- 每个维度打 1-10 分，乘以权重后求和
- **总分 = Σ(维度分 × 权重) / 10**，满分 100
- 维度 8 子 agent 不可用时退化为干跑验证，标注 `dry_run`

### Step 4：输出评分卡

```markdown
| 维度 | 得分 | 短板判断 |
|:-----|:----:|:---------|
| 1. Frontmatter 质量 | X/10 | — |
| 2. 工作流清晰度 | X/10 | — |
| 3. 边界条件覆盖 | X/10 | — |
| 4. 检查点与门禁 | X/10 | — |
| 5. 指令具体性 | X/10 | — |
| 6. 资源整合度 | X/10 | — |
| 7. 整体架构 | X/10 | — |
| 8. 实测表现 | X/10 | — |
| **总分** | **XX/100** | — |
```

### Step 5：输出评估报告

```markdown
## 评估报告 — {skill 名}

### 测试 Prompt
- 1. {happy path prompt}
- 2. {边界场景 prompt}

### 评分卡
{上面的表格}

### 改进建议
P0: {必须改的，不改会出安全问题}
P1: {推荐改的，改了效果提升明显}
P2: {以后改的，优化空间}
```

## 评分等级参考

| 总分 | 等级 | 判定 |
|:----:|:-----|:-----|
| 90-100 | A | 生产就绪 |
| 80-89 | B | 可用，有优化空间 |
| 70-79 | C | 需要改造后再用 |
| <70 | D | 不合格，必须走 B 改造模式 |

---

## ❌ WRONG 示例

```
❌ WRONG: 评估 skill 时不打维度 8 实测 → 评分卡缺 25% 权重，结论不可信
✅ CORRECT: 每个维度必须打分，维度 8 子 agent 不可用时退化为干跑验证（标注 dry_run）

❌ WRONG: 用 Read 工具读 SKILL.md → 文件超过 2000 行被截断，基于不完整内容评分
✅ CORRECT: 用 skill_view 或 Get-Content -Encoding UTF8 -Raw 读取，截断检测后评分
```

---

## 评估→改造闭环

评估(模式C) → 发现短板 → 改造(模式B) → 重新评估(模式C) → 确认通过。
