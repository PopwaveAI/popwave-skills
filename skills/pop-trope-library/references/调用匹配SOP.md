# 调用匹配 SOP

> 用途：各 writer skill 按模块协议查询 pop-trope-library。
> 新结构为六库：立项库 / 设定库 / 文风库 / 剧情库 / 套路库 / 金手指库。
> 执行者：各 writer skill 消费素材时。

---

## 匹配模型

```text
立项层素材：查立项库 → 匹配书型/元爽点组合/题材焊接 → 辅助 creative 生成 PRD
设定层素材：查 PRD 卖点 → 匹配书目 → 读对应 L1 文件
套路/剧情层素材：三维查询（层×赛道×元爽点）→ 读套路库/剧情库剧情线资产
文风层素材：按书名定位文风库 → 按 scene 匹配场景卡
金手指层素材：查金手指库 → 按类型×赛道×元爽点筛选 → 读金手指设计卡（WHAT层）
```

---

## Part 0：立项层匹配（creative 前置）

> 立项库沉淀 PRD 级经验，不替代 creative 推导，只提供可复用的书型、爽点组合、题材焊接与失败案例。

### Step 0.1：查立项库索引

读：`立项库/00-索引.md`

| 用户需求 | 优先查 |
|:---------|:-------|
| 不知道书型怎么立 | `立项库/立项模式/` |
| 有爽点但不知道怎么组合 | `立项库/元爽点组合/` |
| 想把两个题材焊在一起 | `立项库/题材焊接/` |
| 需要确认加工原则 | `立项库/加工哲学/` |
| 想避开历史踩坑 | `立项库/失败案例/` |

### Step 0.2：输出立项参考

立项库只输出“可参考的决策依据”，不得直接覆盖本书 PRD：

```markdown
## 立项库参考
- 匹配条目：{条目名}
- 适用原因：{为什么适合当前项目}
- 可迁移部分：{可复用机制}
- 不可迁移部分：{赛道/人设/体量限制}
- 下游影响：{会影响 creative/world/plot/prose 的哪类输入}
```

### 立项层红线

| # | 红线 |
|:-:|:-----|
| 1 | 把立项库当模板库直接套 PRD |
| 2 | 只看爽点组合，不检查题材焊接成本 |
| 3 | 不声明失败案例风险，导致后续 world/plot 重复踩坑 |

## Part A：设定层匹配（先查 PRD 再读 L1）

> 设定库按书分目录，每本书有完整的 L1 六件套。不能用一条 tag 找到，先看 PRD 卖点找对书，再读 L1 文件。

### Step A1：查 PRD 卖点匹配书目

1. 读 `设定库/00-索引.md`，先看每本书的卖点摘要。
2. 根据当前项目需要，查对应书目的 `PRD.md`。
3. 用 `PRD.md` 的匹配逻辑定位 L1 文件，例如“需要名器分级 → 让魔门 PRD 卖点1 → 读 L1-02 + L1-06”。

| 你的书需要 | 看哪本书的 PRD |
|:----------|:-------------|
| 东方武道力量体系 / 名器分级 / 修仙经济 / 阶层压迫 | 让魔门再次伟大 |
| 硬规则西幻 / 法术复杂度量化 / 封神体系 / 随机奖励 | 深渊主宰 |
| 修仙金融化 / 人体商品化 / 阶级物理化 / 赛博修仙 | 没钱修什么仙 |

### Step A2：读 L1 文件

PRD 的每个卖点都标注了对应 L1 文件。按标注路径读取。

### Step A3：根据本书 PRD 改写转化

L1 文件是原书的设定，不是直接复制。根据本书 PRD 的加工哲学改写。

## Part B：套路/剧情层匹配（三维查询）

| skill | 所属层 | 读取模块 | 读取目标 |
|:------|:-------|:---------|:---------|
| creative | 立项层+全层+框架+质感 | `立项库/` + `设定库/` + `套路库/00-总索引.md` + `金手指库/00-索引.md`（★NEW） + `references/元爽点-变体映射表.md` | 立项经验、主元爽点、变体家族、金手指设计参考、跨域设定素材源与可复用机制 |
| world | 框架+质感 | `设定库/`（先看 PRD 卖点 → 读 L1 文件） | 设定创意池 |
| character | 框架+质感 | `设定库/角色与关系/` + `设定库/{书名}/` | 角色身份、组织、阶层、关系网、说话风格 |
| plot | 机制层 | `剧情库/{标签}/` + `套路库/` + `references/元爽点-变体映射表.md` | L3剧情线卡（六段格式）+ 套路链配套；L2单元卡可作为L2单元卡设计补充参考 |
| chapter | 机制层 | `套路库/{具体套路名}.md` | 事件链、钩子、信息释放、场景调度、套路公式 |
| prose | 调味层 | `文风库/{书名}.md` | 风格渲染场景卡 |
| qa | 全层 | `套路库/{具体套路名}.md` 的“使用红线”段 | L3 原文对照与风险检查 |

## Step 2：确定赛道维度（筛选同赛道）

1. 从 `全书立项PRD.md` 的“类型锚点”提取赛道（修仙/西幻/都市/科幻/武侠/...）。
2. 在目标模块内按赛道标签（frontmatter `track` 字段）筛选。
3. 匹配优先级：同赛道素材 > 通用素材 > 跨赛道素材。
4. 采用跨赛道素材时，必须标注“跨赛道化用”。
5. 无同赛道素材时，降级到通用素材，并标注“本赛道库缺，用通用素材”。

---

## Step 3：确定元爽点维度（筛选变体，适用于机制层 skill）

1. 从 `创意种子/爽点引擎.md` 取本书 2-3 个主元爽点。
2. 从 plot L2幕锚点 `L2-{编号}.md` 取本卷/本章爽点目标。
3. 查 `references/元爽点-变体映射表.md` 匹配变体家族。
4. 在 Step 2 筛选结果中按元爽点标签（frontmatter `meta_joy` 字段）进一步筛选。
5. 无元爽点匹配时，标注“本元爽点库缺变体”，回退到通用套路。

> 7 个元爽点：信息差 / 阶级革命 / 绝境反杀 / 身份跃迁 / 世界观震撼 / 智斗博弈 / 情感羁绊

---

## Step 4：输出匹配清单

```markdown
# pop-trope-library 查询结果 · {skill} · {日期}

## 查询条件
- 层维度：{立项/框架/质感/机制/调味}
- 赛道维度：{修仙/西幻/...}
- 元爽点维度：{信息差/.../无}

## 匹配结果
| # | 素材名 | 模块 | 层 | 赛道 | 元爽点 | 匹配度 | 推荐理由 |
|:-:|:-------|:-----|:---|:-----|:-------|:-------|:---------|

## 库缺声明
- {维度}方向无匹配素材，已降级到{通用/跨赛道}
```

## Part C：文风层匹配（prose 专用）

文风库不走三标签主检索。主键是书名，次键是 `scene`。

读取路径：

`文风库/{书名}.md`

匹配方式：

1. 读设计包事件的 `scene`。
2. 在文风档案定位同名 `scene:`。
3. 若无同名场景卡，执行 prose 外部 DNA 适配。
4. 不得因文风不匹配而改写剧情骨架。

## Part D：金手指层匹配（creative 专用）

> 金手指库回答"设计什么金手指"（WHAT层），套路库回答"怎么写金手指的叙事节奏"（HOW层）。creative 在生成故事概念选项前必须查金手指库。

### Step D1：查金手指库索引

读：`金手指库/00-索引.md`

| 用户需求 | 优先查 |
|:---------|:-------|
| 不知道选什么类型金手指 | `金手指库/00-索引.md` 的 6 类型分类表 |
| 有类型方向，需要设计参考 | `金手指库/{类型}/` 下对应设计卡 |
| 需要确认代价机制是否合理 | 设计卡的「代价设计」字段 |
| 需要确认成长路径 | 设计卡的「成长路径」字段 |
| 需要确认赛道适配 | 设计卡的「赛道适配分析」字段 |

### Step D2：三维筛选（类型×赛道×元爽点）

1. **类型维度**：从 PRD 爽点锚点推导金手指类型方向（系统型/血脉型/道具型/契约型/感知型/悖论型）。
2. **赛道维度**：按 frontmatter `track` 字段筛选同赛道设计卡。
3. **元爽点维度**：按 frontmatter `meta_joy` 字段筛选匹配元爽点的设计卡。
4. 匹配优先级：同类型+同赛道+同元爽点 > 同类型+同赛道 > 同类型+通用 > 跨类型参考。
5. 无匹配时，声明"金手指库缺{类型}赛道{赛道}设计卡"，降级到通用设计卡或 web 搜索。

### Step D3：输出金手指参考清单

```markdown
## 金手指库参考
- 匹配设计卡：{卡名}（{类型}）
- 核心机制：{能干什么/怎么运作/力量来源}
- 代价设计：{代价类型+变体}
- 成长路径：{4阶段概要+天花板}
- 赛道适配：{适配分析}
- 关联元爽点：{对应变体}
- 关联套路卡：{HOW层套路卡名}
- 库缺声明：{如有}
```

### 金手指层红线

| # | 红线 |
|:-:|:-----|
| 1 | 混淆 WHAT/HOW：用套路库的叙事节奏卡替代金手指库的设计卡 |
| 2 | 金手指库无匹配时不声明库缺，直接凭空设计 |
| 3 | 只读索引不读设计卡详情，导致机制/代价/成长路径缺失 |

## 调用红线

| # | 红线 |
|:-:|:-----|
| 1 | 跳过模块层直接读目录，导致立项/设定/剧情/文风/套路混用 |
| 2 | 设定库不读 PRD 卖点，直接复制 L1 原书设定 |
| 3 | 赛道不筛选就取素材，不同赛道的设定/套路不能直接套用 |
| 4 | 元爽点不匹配就定套路链，导致爽点偏离 |
| 5 | 文风库缺失时不声明库缺，导致 prose 误判无风格资产 |
| 6 | 库缺时不声明，导致下游不知道素材可靠性 |
| 7 | 直接消费原书卷/幕聚合页，导致新书被原书篇幅和节奏带偏。卷/幕切片已归档到 `_LEGACY/`，不消费 |
| 8 | 金手指库无匹配时不声明库缺，导致 creative 凭空设计平庸金手指 |
| 9 | 混淆金手指库（WHAT）与套路库（HOW），用叙事节奏卡替代设计卡 |
