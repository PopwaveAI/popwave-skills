# chXXX-设计包模板（事实骨架 + 登场人物卡 合并）

> 产出路径: `写作资产/设计包/chXXX-设计包.md`
> 生产方式: pop-writer-chapter v1.5
> 下游消费者: pop-writer-prose

---

## 模板说明

设计包 = 事实骨架 + 登场人物卡 合并为单文件。一份文件包含本章设计阶段的所有产出。

**结构**：基础信息 → 角色状态（事件链的输入）→ 事件链 → 情绪节拍 → 信息释放清单 → 钩子 → 与上章衔接

**关键原则**：
- 事件只描述**内容**（谁、在哪、做什么），不描述**方式**（怎么写的、什么语气）
- 每个事件 150-400 字渲染量
- 章预期字数 ÷ 200 ≈ 基准事件数
- 角色 before 状态必须从 entity-snapshot.yaml 取——不许凭记忆
- 「本章台词风格」是一个字词——给 Render 一个对话基调

## 填写自检

### 事件数量（硬性下限）

- [ ] **事件数 ≥ 章预期字数 ÷ 200** — 低于此值是设计缺陷，"靶心不够，Render 没材料"
- [ ] 每个事件 = 一个回合/一个因果单位（战斗章：一刀一事件；对话章：一轮信息交换一事件）

### 事件质量

- [ ] 每个事件标注了情绪目标 ← 这是 Render 的创作靶心
- [ ] 事件内容是「谁做了什么 → 什么结果」— "科尔转身拔剑——刀从侧面切入他右腕"，不是 "开始攻击"
- [ ] 事件内容**不是** Render 微节拍 — 不写具体动作细节/感官/氛围
- [ ] 事件链因果不断裂 — 每个事件和下一个事件之间有"所以"或"然后"
- [ ] 章末事件是钩子事件

### 角色状态

- [ ] 主角 before 状态与 entity-snapshot.yaml 一致
- [ ] 所有出场角色已在 volume-XX.md §三（角色池）中定义
- [ ] 每个配角有「本章台词风格」— 必须是具体的一个词，不是泛泛
- [ ] after 状态与对应事件的事件内涵一致

### Canvas 对齐

- [ ] 每个事件的「参与角色」在 volume-XX.md §三（角色池）中存在
- [ ] 每个事件的「地点」在 volume-XX.md §三（地点池）中存在
- [ ] 「信息释放落地清单」与 act-XX.yaml#info_release_plan 对齐
- [ ] 「与上章衔接」与 entity-snapshot 对上章的状态一致
- [ ] 情绪节拍「终点」能自然衔接到下一章的起点
- [ ] 跨章弧线如存在，arc_span + 本章位置 + 情绪分工 三个字段必填
- [ ] 每个事件有估计字数（100-250字）
