# PRD 缺口分析：元爽点(需求) ≠ PRD(解决方案)

> **来源**：2026-06-20 session，对比三本拆书项目（深渊主宰 / 让魔门再次伟大 / 没钱修什么仙）的实际 PRD 后发现的架构性缺口。
>
> **状态**：学习笔记，非正式修订。待用户确认是否重构 PRD 结构。

## 核心认知

**元爽点 = 用户需求**（读者想要什么感觉）
**PRD = 解决方案**（世界怎么设计才能让这种感觉自然产生）

当前 `step-prd.md` 的推导链（元爽点 → 美学基因 → 色彩 → 材料 → 篇幅 → 加工矩阵 → 核心承诺）覆盖了「需求」和部分「美学方案」，但缺少将需求转化为可执行世界规则的**机制层**。

## 关键缺口：需求→解决方案的 5 层

```
需求层:   用户想要什么爽感（元爽点）
  ↓
第1层:   世界宪法 —— 为了让这个爽感成立，世界必须遵循哪些不可违反的规则？
第2层:   美学基因 —— 两个最底层的设计前提是什么？从此推导全部设计决策。
第3层:   结构架构 —— 卷/幕怎么分？数值怎么爬？地图怎么换？（静态金字塔+动态升级表）
第4层:   加工矩阵 —— 什么素材能用、什么不能用、怎么用？（S/A/B/C 分类+四维约束）
第5层:   防崩机制 —— 每个爽点的阈值在哪？写崩了怎么拉回来？（可量化的防崩阈值）
  ↓
解决方案层: PRD —— 可执行的宪法
```

## 常见错误顺序

```
❌ 错误：先有世界设定（大陆/国家/势力）→ 再塞金手指 → 再想爽点
  结果：世界和爽点脱节，读者觉得设定是设定、故事是故事。

✅ 正确：先有爽点需求 → 推导世界宪法（让爽点成立的规则）→ 再推导具体设定
  结果：世界规则和读者体验咬合，每一个设定都有读者意图。
```
