# 正文渲染 / 上色表达 v3.2.0

消费 pop-writer-chapter 的设计包产物，全量加载 `写作资产/文风DNA/` 文风DNA（场景卡 v4 格式），对照设计包的 scene 字段定位场景卡后渲染。

**加载策略**：全量加载单文件（~20-30K），不拆分。过渡段/情绪缓冲/场景切换依赖整体风格感知。

**核心约束：不碰剧情，不编造。** 不读上游 Canvas、不验证设定、不判断角色出场是否合理。Design 说了这章发生什么 → 我只管写好。**设计包中标注 🔒 的关键对白/数据必须原文照写，不得替换或改写。**

**v3 设计包格式**：支持 `写作资产/设计包v3/chXXX-设计包.md`（4层结构 Markdown）作为输入源，优先级高于 v2 格式。单章独立消费，无跨章数据聚合。

**合成模式**：当 prose-render 无原文可校验（新写/同人二创/风格迁移测试），完全靠设计包+DNA渲染时，额外加载 `references/synthesis-mode.md`。该模式风险等级更高——设计包精度锚点是唯一约束源，🔒标记强度从"建议"升级为"唯一可靠性栅栏"。

## ❌ 错误示例

### WRONG 1：叙事者跳出来解释
→ 「他手里的剑很锋利——这把武器是他三年前从师门继承的，那场变故之后他一直在熟悉它的重量。」
→ ✅ 正确：「剑刃在月光下泛着冷白。缠绳握柄上被拇指磨出了一道光滑的凹痕。」

### WRONG 2：因为风格规则而跳过骨架事件
→ 骨架有「主角向盟友提问」事件，但 Render 觉得对话太长就直接删了。
→ ✅ 正确：骨架说了有对话就必须写对话。语言可以简洁，事件不可跳过。

### WRONG 3：读上游 Canvas
→ Render 读了 act-XX.yaml 或 L1 设定。
→ ✅ 正确：Render 只读骨架 + 人物卡 + 文风DNA。

### WRONG 4：重新提取风格契约
→ 先写一份"风格契约"，再用契约渲染。这是多此一举——文风DNA文件已经完整可读。  
→ ✅ 正确：Step 1 读完文风DNA，context 中已有。Step 2 直接渲染。

### WRONG 5：事件描述模糊时用语言模型补全
→ 设计包写"分配经验升至满级"（无精确数值），Render 自行补全"获得【平民大师专长】力量+1"。  
→ ✅ 正确：设计包事件缺少关键对白/数据字段时的处理——保留模糊部分，不编造具体数值和专长名。如果设计包没有标注精确增益值，在正文中写"一股暖流涌入身体，能感觉到四肢正在恢复力量"而非"获得XX专长属性+1"。宁可模糊不编造。

### WRONG 5：跳过风格验证直接输出
→ Step 2 写完直接进 Step 4，觉得"差不多符合文风"。
→ ✅ 正确：Step 3 不可跳过。逐项对照文风DNA原文标记偏差，≥2处退回修补。

### WRONG 6：重渲染前未清理旧文件
→ 文风DNA就位后重写已有正文，旧 chXXX.md 残留。
→ ✅ 正确：Step 0 先删除旧文件，再进渲染流程。

### WRONG 7：从 act-XX.yaml 直接渲染绕过设计包
→ CH2 直接从 act-01.yaml Canvas 矩阵渲染正文，跳过了 chapter-design 的设计包步骤。
→ ✅ 正确：prose-render 的输入源必须是 chXXX-设计包.md。act-XX.yaml 是给 chapter-design 看的，不是给 prose 当骨架用的。设计包不存在 → 退回 chapter-design。

### WRONG 8：文风DNA为空时静默降级
→ "写作资产/文风DNA/目录为空，使用默认风格" — agent 自行决定降级，用户写完后才发现风格不对。
→ ✅ 正确：空目录时必须输出 A/B 选项给用户，由用户决定是否跳过风格。

### WRONG 9：未加载 step 文件就凭记忆渲染
→ 只加载了 SKILL.md 描述（4000+ chars）就直接进 Step 2 逐事件渲染，没有读 steps/step-1-read-input.md ~ step-4-output.md。不知道文本脉冲密度检查、解说员句式检查、字数校准等具体规则 → 正文质量偏离 skill 标准。
→ ✅ 正确：每次路由到 prose 后，用 skill_view 加载全部 steps/*.md 文件，完整了解 Step 1-4 的具体执行指令后再渲染。Step 3（风格验证）和 Step 4（最终输出）的检查项必须在渲染前就知道。

### WRONG 10：数据面板自发作案
→ 设计包没有在 `关键对白/数据` 字段标注面板数据，但 prose 自己加了一行数据反馈（"击杀混沌狂战士，杀戮经验+30"）。违反 prose ❌1"设计包是唯一输入源"——prose 不编造骨架未标注的字段。
→ ✅ 正确：数据面板/系统消息/精确数值仅在设计包事件的 `关键对白/数据` 字段中标注时才写入正文。没有标注 → 写感官和行为反馈（"一股暖流涌入手掌，四肢的疲劳消退了一些"），不写系统面板。

### WRONG 11：合成模式当拆书模式用，无原文时填充默认值
→ 设计包写"希斯是条老狗"，prose 写"一条普通的杂种土狗，毛发暗淡"。原文希斯是哥拉斯猎犬（可搏杀豹子）——合成模式无原文，prose 用通用模板填充了"杂种土狗"。
→ ✅ 正确：合成模式下所有未精确标注的细节保持模糊。写"希斯的毛发暗淡，步伐蹒跚"，不写品种/来源/血统——因为这些不在精度锚点中。详见 `references/synthesis-mode.md`。

### 章末状态更新块格式规范

正文文件末尾的 `__state_update` 块**不是 prose 的私有注释**——它由 parent agent 读取，用于更新 `00-总控/entity-snapshot.yaml`。因此格式必须一致。

推荐格式（CH4/CH5 已验证）：

```markdown
> 【chapter_state_update】
> 位置: {精确地点（地区·场所）}
> 时间: {时间锚点}
> {主角}状态:
>   - 等级: Lv{N}（{职业}）
>   - 装备: {关键装备列表·有变化要标注}
>   - 种子觉醒度: {百分比}%
>   - 生命值: {当前/最大}
>   - 状态: {特殊状态（亚空间反馈后遗症等）}
>   - 心理: {情绪关键词3-5个}
> {核心配角}状态:
>   - 伤势: {恢复情况}
>   - 装备: {装备变化}
>   - 特殊物品: {重要物品的变化}
>   - 心理: {关系/认知变化}
> 外部威胁:
>   - {敌方}: {状态变化}
>   - {环境}: {裂缝等状态}
> 决策: {下一步行动方向}
> 钩子: {章末情绪/悬念摘述}
```
