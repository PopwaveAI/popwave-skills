# Project-Level Retrospective Methodology

> 适用场景：用户要求对比两个项目（"为什么A比B质量差距这么大"）、或复盘单个项目的管线执行健康度。
> 区别于 `pop-writer-qa`（逐章的正文/设计包质量审查），本方法审查的是**整个管线的执行质量**——不是内容有没有写对，而是管线有没有被正确执行。

---

## 触发条件

当用户说出以下任意一句时，启用本参考：

- "全链路复盘"
- "对比两个项目"
- "为什么这个项目gap这么大"
- "从结果物到执行，哪里出问题了"
- "为什么武林半侠做得好，东方版做不出来"
- 要求对一个已完成(或停滞)的创作项目做整体诊断

---

## 方法论：四步诊断

### Step 1 — 定量全景扫描

从文件系统获取两个项目的完整结构，统计以下指标：

| 指标 | 获取方式 |
|:-----|:---------|
| 总文件数 | `find ... -type f \| wc -l` |
| 管线各阶段文件数 | 按目录名分组统计 |
| 管线完成度 | creative → reservoir → world → character → plot → chapter → prose → qa，逐阶段标记✅/⏳/❌ |
| 每文件平均深度 | 核心文件（剧情线、设计包、L2单元卡）的字符数/行数 |
| 回滚/备份次数 | 统计 `.backup-*` 目录数量，或 conversations 中的回滚指令 |
| 会话量 | conversations/ 中的 .jsonl 数量和消息总数 |

**产出：** 一张对比表格，标注每个管线阶段的完成状态和资产深度。

### Step 2 — 逐阶段深度对比

对管线每个阶段（PRD / 素材储备 / 世界设定 / 角色 / 剧情线 / 卷纲 / L2单元卡 / 设计包 / 正文），逐个做样本读取：

- **PRD：** 结构完整性（10块是否齐全）、创意原创性（是移植还是创作）、爽点体系是否可执行
- **素材储备：** 每张卡的深度（k字/卡）、是否含元卖点+因果链+质量自检、卡数量 vs 实际剧情覆盖
- **剧情线：** 节点数（8节点 vs 12节点+）、是否含"因果闭环"、"咬合细节"、起点/终点切片的颗粒度
- **章节设计包：** 是否存在（全部核心）、事件链粒度（事件数）、爽点层是否含情绪弧+钩子分级、感官层是否含6维环境基线
- **套路库/创意溯源：** 是否存在、数量级（个位数 vs 百位数）、是否被剧情线实际引用

**产出：** 每个阶段一层"厚度评估"——是"设计级"（3-5k字，骨架信息）还是"执行级"（10k+字，可直接消费）。

### Step 3 — 执行过程追溯

读取 conversations/ 下的会话文件，提取：

- **会话总数和消息总数**（越少越可疑——说明没有深入迭代）
- **回滚/重置次数**（每次回滚删除已产出内容，重新开始→不可逆消耗）
- **子Agent失败记录**（如"未能读取到子Agent最终文本"）
- **用户重复纠正的次数**（如"为什么剧情线还是只有1条主线"——说明第一次没理解需求）
- **项目生命周期**（从创建到最后一次更新的间隔）

**模式识别：**

| 模式 | 典型表现 | 后果 |
|:-----|:---------|:-----|
| 重复回滚 | 2次以上的"全量回滚"+"删除所有无关文件" | 项目总时间2/3消耗在重做上 |
| 压缩执行 | 48小时内跑完6个管线阶段 | 每个阶段都蜻蜓点水，逐层不得加深 |
| 执行中断 | 停在plot层，不进入chapter-design | 无"可消费"资产产出 |
| 工具失败 | 子Agent返回空结果 | 关键产出可能丢失 |

**产出：** 执行过程健康卡——回滚次数、对话密度、中断位置。

### Step 4 — 根因诊断映射

将发现归入以下模式框架：

#### 模式A：移植思维 vs 创作思维
- **特征：** PRD写"核心机制全部保留，只换文化包装"；读者体验不是从故事推导，是从"原版对应"推导
- **后果：** 故事引擎是借来的，不是生成的——缺少"为什么这个设定在这本书里好使"的内在逻辑
- **修复方向：** 从"原版对应"清单改为"原版逻辑→本书用户需求→适配方案"三段式推导

#### 模式B：设计深度断层
- **特征：** PRD(10k) → L1(8k) → 剧情线(3k) 每一层深度一致或递减，没有"逐层加厚"
- **后果：** 所有产出都是"设计级"（骨架/方向/大纲）而非"执行级"（可以直接消费的章节素材）
- **修复方向：** 管线不是"流水线式通过"，而是"逐步放大"——后一层子文件数×深度应显著大于前一层

#### 模式C：可消费资产缺失
- **特征：** 有设定卡（素材储备池）但无套路库/创意溯源/设计包——设定是"知识"，套路是"工具"
- **后果：** 从设定到正文的转译成本极高，导致停在了plot层
- **修复方向：** 设计包、套路库、创意溯源是"可消费资产"——它们的缺失是管线断裂的最强信号

#### 模式D：回滚消耗
- **特征：** ≤5条消息的对话就被关闭，下次开新会话"回滚到某个层重做"
- **后果：** 回滚消耗时间 = 原工作时间的2-3倍
- **修复方向：** 回滚之前问：是方向错了（重来）还是细节不够（在当前文件上迭代）？

#### 模式E：管线完整性断裂
- **特征：** 管线在某个固定位置中断（如总是停在plot层，从不进入chapter-design）
- **后果：** 前面5个阶段的设计资产都无从执行
- **修复方向：** 如果需要人介入才能继续（如"等作者确认"），标记为阻塞点并记录阻塞原因
