# Step 5：活记忆压缩（自然语言合并）

> 管线: pop-writer-v3-arc v1.4.0
> 消费: 活记忆.yaml（最后baseline + 本单元全部event）
> 产出: 压缩后的活记忆.yaml + 压缩记录 + 校准报告落盘

## 读什么

- `活记忆/活记忆.yaml`（全量：最后baseline + 本单元全部event，自然语言格式）
- `项目总控.md`（当前章号+L2单元编号+arc计数）
- Step 3 产出的更新后幕纲（作为压缩时的状态参照）

## 做什么

v3.5 活记忆为自然语言格式（不再是7组件结构化YAML）。压缩策略从"逐组件合并"改为"自然语言段落合并"——将最后一个baseline + 之后全部event合并为一段新的baseline概括。

每个L2单元结束时arc完整执行5步，压缩是第5步，始终执行（红线❌1：不省token）。

### 1. 读取最后一个baseline

找到活记忆中最后一个 type=baseline 的条目，记录其章号和内容（一段自然语言）。

活记忆格式（v3.5自然语言）：
```yaml
- type: baseline
  chapter: 5
  timestamp: 2026-06-28T10:00:00
  summary: |
    主角索伦在ch001-005完成了铁钳帮反杀，系统从Lv1升到Lv2，解锁街头感知技能。
    当前状态：血量12/60，体力恢复至30%，经验46/100。铁钳帮威胁阶段性解除，
    但教团渗透线索浮现（ch003发现命运迷宫纹）。灵能觉醒开始（ch003耳鸣）。
    活跃伏笔：灰色注释来源（ch003埋，ch010前回收）、教团-帮派资助关系（ch004发现）。
    压力源：铁钳帮搜索圈（已解除）、教团学校渗透（5章窗口）、灵能觉醒临界（10章+）。

- type: event
  chapter: 6
  summary: ch006发生XXX状态变化...

- type: event
  chapter: 7
  summary: ch007发生XXX状态变化...
```

### 2. 读取之后全部event

收集该baseline之后的所有 type=event 条目（本L2单元的章节级状态变化日志）。

### 3. 自然语言合并

将最后一个baseline + 之后全部event合并为一段新的baseline概括：

**合并原则：**
- 不是简单拼接，而是提炼合并后的**当前状态**
- 保留所有活跃伏笔、压力源、角色状态变化、力量体系数值
- 丢弃已回收伏笔的中间过程（只保留结果）、已关闭压力源的细节
- 一段话高度概括"当前主角处于什么状态，面前有哪些未了事项"
- 控制在300-500字（自然语言段落）

**合并示例：**

```
旧baseline（ch005）：
  主角索伦在ch001-005完成了铁钳帮反杀，系统从Lv1升到Lv2，解锁街头感知技能。
  当前状态：血量12/60，体力恢复至30%，经验46/100。铁钳帮威胁阶段性解除，
  教团渗透线索浮现。灵能觉醒开始。活跃伏笔：灰色注释来源、教团-帮派资助关系。

event ch006：
  索伦潜入铁钳帮据点清理残党，发现账本，系统经验+20，解锁感知扩展。

event ch007：
  教团纹身线索深化，索伦在街头感知到第二个纹身者，灵能耳鸣升级。

event ch008：
  索伦通过账本发现教团资助铁钳帮的证据，灰色注释来源回收（ch003埋→ch008回收）。

event ch009：
  系统升级Lv2→Lv3（经验破百），解锁灵能定位技能。教团学校渗透压力升级。

event ch010：
  索伦决定主动调查教团，教团-帮派资助关系线索深化为正面对抗。

合并后的新baseline（ch010）：
  主角索伦在ch001-010完成铁钳帮清剿并转向教团调查。系统从Lv1升到Lv3，
  解锁街头感知+灵能定位技能。当前状态：血量18/60，体力恢复至50%，经验12/200。
  铁钳帮威胁已完全解除（据点清理+账本证据）。教团渗透升级为正面对抗，
  发现教团资助铁钳帮的证据链。灵能觉醒推进至二级耳鸣+灵能定位。
  活跃伏笔：教团真实目的（ch004发现，持续）、灵能觉醒临界（ch003开始，10章+）。
  已回收伏笔：灰色注释来源（ch003→ch008回收）。
  压力源：教团学校渗透（升级为正面对抗，3章窗口）、灵能觉醒临界（5章+）。
```

### 4. 替换

1. 删除旧baseline及其后全部event
2. 追加新baseline（自然语言段落）
3. 更新metadata：last_compacted_at = {当前章号}，last_compacted_l2 = {L2单元编号}

```yaml
- type: baseline
  chapter: 10
  timestamp: {当前时间}
  l2_unit: L2-002
  summary: |
    {合并后的自然语言段落，300-500字}
```

### 5. 压缩后验证

| 检查项 | 失败动作 |
|:-------|:---------|
| 新baseline包含当前主角状态（力量体系数值） | 缺数值=补充 |
| 新baseline包含活跃伏笔列表 | 缺伏笔=补充 |
| 新baseline包含当前压力源 | 缺压力源=补充 |
| 已回收伏笔只保留结果不保留中间过程 | 冗余过程=精简 |
| 已关闭压力源已移除 | 残留=移除 |
| 新baseline字数300-500字 | 超范围=精简或补充 |
| 活记忆文件不为空 | 空=回滚压缩 |

### 6. 压缩失败处理

如果验证不通过：
1. 回滚：恢复压缩前的活记忆.yaml（从备份）
2. 记录失败原因
3. 在校准报告中标记"压缩失败，需人工检查"

## 门禁

| 检查项 | 失败动作 |
|:-------|:---------|
| 压缩前备份活记忆.yaml | 无备份=先备份再压缩 |
| 新baseline包含主角当前状态+活跃伏笔+压力源 | 缺项=补充后重新验证 |
| 新baseline 300-500字（自然语言段落） | 超范围=精简或补充 |
| metadata.last_compacted_at 已更新 | 未更新=补更新 |
| 活记忆文件非空 | 空=回滚压缩 |

## 产出

1. 压缩后的 `活记忆/活记忆.yaml`（新baseline替换旧baseline+event）
2. 压缩记录（填入校准报告草稿）：

| 项目 | 内容 |
|:-----|:-----|
| 压缩前章号范围 | ch{旧baseline章号}-ch{当前章号} |
| 压缩前文件大小 | {XX} KB |
| 压缩后文件大小 | {XX} KB |
| 合并event数量 | {N}条 |
| 新baseline字数 | {XXX}字 |
| 压缩状态 | 成功 / 失败 |

### 7. 校准报告落盘

所有5步完成后，将校准报告草稿整理为正式报告：
1. 使用 `templates/弧线校准报告-模板.md` 格式
2. 落盘到 `弧线校准报告/arc-{XX}.md`（XX=校准序号，与L2单元编号对应）
3. 更新 `项目总控.md`：arc计数+1
