# 信息获取强制化SOP

> 所属: pop-writer-v3-emerge v1.2.0 · references/
> 消费: Step 1 信息获取强制化时参考
> 红线: ❌6 信息获取必须读索引——索引.md未读取=退回补读
> v3.1改动: 从v3.0的"信息需求判断SOP"重写，从"自主判断要不要查"改为"强制读索引→读/搜→写本地"

## 一、设计背景

v3.0的"自主判断要不要查"导致16章跳过信息获取（ch12-23持续跳过）。根因：自主判断=选择不查，跳过成本为零。

v3.1改为强制化：每章必须读知识库索引，读后判断有没有匹配，有则读知识库文件，无则WebSearch并写入知识库。信息获取不可跳过——索引是每章的必经之门。

**核心原则：允许"无需求"，但不允许"不读索引"。**

## 二、强制化流程

```
Step 1启动
  │
  ├── 1. 强制读索引（红线❌6门禁）
  │   └── 写作资产/知识库/索引.md
  │       ├── 索引不存在 → 首次执行，创建知识库目录+索引文件
  │       └── 索引已读 → 进入步骤2
  │
  ├── 2. 基于chapter_plan做4类检查
  │   ├── 制度/流程细节
  │   ├── 场景技法
  │   ├── 已埋伏笔/前文细节
  │   └── 创意参考
  │
  ├── 3. 索引匹配判断
  │   ├── 有匹配 → 读知识库文件 → 提取信息
  │   └── 无匹配 → WebSearch/library → 写入知识库 → 更新索引
  │
  ├── 4. 无需求处理（合法，但索引必须已读）
  │   └── needs: []
  │
  └── 5. 登记记录（info_acquired）
```

## 三、4类检查

### 检查1：制度/流程细节

| 维度 | 内容 |
|:-----|:-----|
| 判断问题 | 这章涉及什么我不熟悉的现实制度/流程？ |
| 信息源 | WebSearch（→写入知识库） |
| 触发条件 | 本章场景涉及不熟悉的现实世界制度、法律流程、社会机制 |
| 示例 | 美国催债法流程、庇护所申请流程、帮派等级制、医疗账单体系 |

**不触发的场景：** 已建立机制的正常运作、幻想世界的自设规则（已在种子中确立）。

### 检查2：场景技法

| 维度 | 内容 |
|:-----|:-----|
| 判断问题 | 这章的场景类型需要参考技法吗？ |
| 信息源 | library套路库 `pop-trope-library/套路库/{套路名}.md` |
| 触发条件 | 本章场景类型属于高难度渲染场景，或agent对该场景类型的技法不熟练 |
| 示例 | 帮派对峙→套路库/暗黑现实.md、系统面板→套路库/金手指体系、绝境反杀→套路库/绝境反杀.md |

**不触发的场景：** 日常生存、已熟练的常规场景类型。

### 检查3：已埋伏笔/前文细节

| 维度 | 内容 |
|:-----|:-----|
| 判断问题 | 这章需要呼应哪些已埋伏笔？ |
| 信息源 | 项目本身（活记忆种子追踪组件+前章正文） |
| 触发条件 | 本章需要回收或推进已埋伏笔，或需要呼应前文细节 |
| 示例 | MS-13在找谁、老约翰之死、前身背景、上章留下的悬念 |

**不触发的场景：** 本章是全新场景，与前文无直接关联。

### 检查4：创意参考

| 维度 | 内容 |
|:-----|:-----|
| 判断问题 | 这章有什么场景需要灵感参考？ |
| 信息源 | WebSearch（→写入知识库）/ library |
| 触发条件 | 本章有特殊场景需要创意灵感或节奏参考 |
| 示例 | 绝境反杀的节奏控制、围城心理战的写法、群体冲突的场面调度 |

**不触发的场景：** 常规场景，无需额外灵感。

## 四、知识库目录结构

知识库位于 `写作资产/知识库/`，按分类组织：

```

├── 制度流程/                  # 现实制度/流程调研
│   ├── 美国催债法流程.md
│   ├── 庇护所申请流程.md
│   └── 帮派等级制.md
├── 场景技法/                  # 场景写法技法（从library提取或自建）
│   ├── 帮派对峙写法.md
│   └── 绝境反杀节奏.md
├── 创意参考/                  # 创意灵感参考
```

## 五、写入流程

当WebSearch获取新信息时，执行以下写入流程：

### 1. 创建知识库文件

在对应分类目录下创建 `.md` 文件，写入搜索结果：

```markdown
# {文件名}

> 创建: ch{章号} · {timestamp}
> 来源: WebSearch "{搜索关键词}"
> 分类: {制度流程/场景技法/创意参考/伏笔追踪}

## 关键信息

{提取的关键信息：流程步骤、关键规则、现实约束}

## 详细内容

{搜索结果的详细整理}
```

### 2. 更新索引

在 `写作资产/知识库/索引.md` 的对应分类下追加条目：

```markdown
| {文件名}.md | ch{章号} | {关键词} | {简述} |
```

### 3. 记录到info_acquired

在info_acquired记录中标记 `source: WebSearch→知识库/{分类}/{文件名}.md`，并在`kb_new_files`中记录新文件路径，`index_updated: true`。

## 六、防过度保守机制

v3.0的问题是agent过度保守变成"每章都不查"。v3.1通过强制读索引解决——即使最终`needs: []`，索引也必须已读取。但为防止另一种极端（每章都查），以下原则仍然有效：

1. **允许"无需求"**：4类检查均无明确需求时，`needs: []`是合法的
2. **常规章节不查**：日常生存、已建立机制的运作不需要查
3. **查询纪律**：查到的是参考不是模板——获取的信息用于丰富上下文，不是照抄

## 七、与v3.0信息需求判断SOP的差异

| 维度 | v3.0 信息需求判断SOP | v3.1 信息获取强制化SOP |
|:-----|:---------------------|:----------------------|
| 核心原则 | agent自主判断"需不需要查" | 强制读索引，不允许跳过 |
| 索引 | 无 | 强制读取（红线❌6） |
| 知识库 | 无本地沉淀 | WebSearch结果写入知识库+更新索引 |
| 信息源 | WebSearch/library/项目本身 | 同左，但WebSearch结果必须本地沉淀 |
| 4类检查 | 制度/场景写法/文风渲染/伏笔 | 制度/场景技法/伏笔/创意参考（文风渲染移到修订层） |
| 跳过成本 | 零（自主判断=选择不查） | 非零（必须读索引，读完才能判断） |
