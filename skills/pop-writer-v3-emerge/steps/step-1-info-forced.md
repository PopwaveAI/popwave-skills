# Step 1：信息获取（强制化）

> 管线: pop-writer-v3-emerge v2.0.0
> 消费: Step 0本章规划 + 种子文档 + 活记忆
> 产出: 信息获取记录（info_acquired）
> 红线: ❌6 信息获取必须读索引——素材库/索引.md未读取=退回补读

## 目的

v3.0的"自主判断要不要查"导致16章跳过信息获取。v3.1改为强制化：每章必须读素材库索引，读后判断有没有匹配，有则读素材库文件，无则WebSearch并写入素材库/知识沉淀。信息获取不可跳过。v3.2新增搜索深度标准和无需求门槛提高。

**与v3.0的核心差异：** 不是"判断要不要查"，而是"先读索引再说"。索引是每章的必经之门。

## 读什么

| # | 文件 | 为什么 |
|:-:|:-----|:-------|
| 1 | Step 0产出的`chapter_plan` | 本章要写什么场景/推进什么线索/设计什么爽点——信息需求的来源 |
| 2 | `素材库/索引.md` | **强制读取（红线❌6）**——素材库已有内容清单，判断有没有匹配 |
| 3 | `种子文档.md`（当前版本） | 六要素细节——判断涉及的制度/场景是否需要补充 |
| 4 | `活记忆/活记忆.yaml` | 种子追踪组件——本章要呼应的已埋伏笔的细节 |
| 5 | `references/信息获取强制化SOP.md` | 强制化流程+素材库目录结构+索引格式+写入流程+搜索深度标准 |

## 做什么

### 1. 强制读索引（红线❌6门禁）

**第一步永远是读索引。** 打开 `素材库/索引.md`，查看已沉淀的素材库文件清单。

- 索引不存在 → 首次执行，创建素材库目录+索引文件（见SOP的目录结构说明）
- 索引已读 → 进入步骤2

**门禁：如果素材库/索引.md未被读取，退回补读。不允许跳过。**

### 2. 基于chapter_plan做4类检查

从chapter_plan的场景设计、线索推进、爽点设计、危机设计中提取信息需求：

| 检查类型 | 判断问题 | 信息源 | 示例 |
|:---------|:---------|:-------|:-----|
| 制度/流程细节 | 规划中的场景涉及什么我不熟悉的制度/流程？ | WebSearch | 催债人上门→查美国催债法流程 |
| 场景技法 | 规划中的场景类型需要参考技法吗？ | library套路库 | 帮派对峙→查套路库的压迫感营造 |
| 已埋伏笔/前文细节 | 规划中要推进的线索需要呼应哪些前文？ | 项目本身（活记忆+前章正文） | MS-13线索→查ch1的老约翰之死细节 |
| 创意参考 | 规划中有什么场景需要灵感参考？ | WebSearch/library | 绝境反杀→查类似场景的节奏控制 |

### 3. 索引匹配判断

对每个信息需求，先查索引中是否已有匹配的素材库文件：

```
信息需求 → 查索引
  ├── 索引中有匹配文件 → 读该素材库文件 → 提取关键信息 → 标记source: 素材库/知识沉淀/{文件名}
  └── 索引中无匹配文件 → WebSearch/library查询 → 提取关键信息 → 写入素材库/知识沉淀 → 更新索引
```

### 4. 信息获取（如有需求）

按需求清单逐项获取：

**4a. 索引中有匹配 → 读素材库文件**
- 从 `素材库/知识沉淀/{分类}/{文件名}.md` 读取
- 提取关键信息融入写作上下文
- 在info_acquired记录中标记 `source: 素材库/知识沉淀/{文件名}.md`

**4b. 索引中无匹配 → WebSearch并写入素材库/知识沉淀**
- WebSearch搜索制度/流程/创意参考
- 提取关键信息：流程步骤、关键规则、现实约束
- **写入素材库/知识沉淀**：创建 `素材库/知识沉淀/{分类}/{文件名}.md`，写入搜索结果
- **更新索引**：在 `素材库/索引.md` 追加条目
- 在info_acquired记录中标记 `source: WebSearch→素材库/知识沉淀/{文件名}.md`

**4c. library查询（套路库）**
- 查询 `pop-trope-library/套路库/{套路名}.md`
- 提取场景技法/渲染参考
- 在info_acquired记录中标记 `source: library套路库/{套路名}.md`

**4d. 项目本身（活记忆+前章正文）**
- 查阅活记忆种子追踪组件：已埋伏笔清单+状态
- 查阅前章正文：具体伏笔细节
- 提取待回收伏笔的内容和预期回收点
- 在info_acquired记录中标记 `source: 活记忆+前章正文`

### 4e. 搜索深度标准（v3.2新增）

WebSearch获取的信息必须满足搜索深度标准：

| 维度 | 标准 | 不达标处理 |
|:-----|:-----|:-----------|
| 文件字数 | 每篇知识沉淀文件≥500字 | 补充搜索结果至达标 |
| 内容深度 | 必须含具体案例/流程步骤/数据数值（至少其一） | 追搜补充具体细节 |
| 多轮搜索 | 首次搜索结果不足→追搜（换关键词/换角度），最多3轮 | 3轮仍不足→标注"信息有限"并继续 |
| 结构化 | 按知识沉淀文件模板结构化 | 重新整理 |

**多轮搜索策略：**
1. 首次搜索：用 chapter_plan 提取的核心关键词搜索
2. 首次结果评估：是否有具体案例/流程/数据？字数是否≥500？
3. 不足→追搜：换关键词（更具体）/ 换角度（从案例反查流程）/ 换信息源
4. 最多3轮，3轮后仍不足→标注"信息有限，创作时酌情处理"

### 5. 无需求处理（v3.2门槛提高）

4类检查均无明确需求 → `needs: []` 仍合法，但**必须逐类说明不需求理由**：

```yaml
info_acquired:
  needs: []
  no_need_reasons:
    制度流程: "{具体理由}"
    场景技法: "{具体理由}"
    已埋伏笔: "{具体理由}"
    创意参考: "{具体理由}"
```

**门禁：no_need_reasons 四项未全部填写 = 退回补填。**

**连续3章 `needs: []` → 标记信息获取退化预警。**

**核心原则：允许"无需求"，但不允许"不读索引"，也不允许"敷衍式跳过"。** 常规章节（日常生存、已建立机制的运作）可以无需求，但每章都必须读索引确认，且必须说明为何无需求。

### 6. 登记记录

按 `templates/信息获取记录-模板.md` 格式，填写info_acquired记录（YAML），写入活记忆event的`info_acquired`字段。

## 门禁

| 检查项 | 失败动作 |
|:-------|:---------|
| **素材库/索引.md未读取（红线❌6）** | **退回补读索引** |
| chapter_plan不存在 | 退回Step 0补做本章规划 |
| 信息获取记录未填写 | 补填后再进Step 2 |
| WebSearch获取的信息未写入素材库/知识沉淀 | 补写入后再继续 |
| 素材库新文件未更新索引 | 补更新索引后再继续 |
| **知识沉淀文件字数<500字** | **补充搜索结果至达标** |
| **needs:[]但no_need_reasons未全部填写** | **退回补填** |
| **连续3章needs:[]** | **标记信息获取退化预警** |

> 注意：本步骤有"必须读索引"门禁（红线❌6），允许"无需求"（needs: []），但无需求必须逐类说明理由（no_need_reasons）。

## 产出

信息获取记录（YAML格式），写入活记忆event的`info_acquired`字段。包含：
- chapter（章号）
- index_read（索引已读取标记）
- needs（4类检查结果清单，或空数组）
- no_need_reasons（无需求时4类不需求理由，有需求时省略）
- merged（是否已融入写作上下文）
- kb_new_files（本章新建的素材库文件列表）
- index_updated（索引是否已更新）
