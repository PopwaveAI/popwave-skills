# Step 1：信息获取（强制化）

> 管线: pop-writer-v3-emerge v1.2.0
> 消费: Step 0本章规划 + 种子文档 + 活记忆
> 产出: 信息获取记录（info_acquired）
> 红线: ❌6 信息获取必须读索引——索引.md未读取=退回补读

## 目的

v3.0的"自主判断要不要查"导致16章跳过信息获取。v3.1改为强制化：每章必须读知识库索引，读后判断有没有匹配，有则读知识库文件，无则WebSearch并写入知识库。信息获取不可跳过。

**与v3.0的核心差异：** 不是"判断要不要查"，而是"先读索引再说"。索引是每章的必经之门。

## 读什么

| # | 文件 | 为什么 |
|:-:|:-----|:-------|
| 1 | Step 0产出的`chapter_plan` | 本章要写什么场景/推进什么线索/设计什么爽点——信息需求的来源 |
| 2 | `写作资产/知识库/索引.md` | **强制读取（红线❌6）**——知识库已有内容清单，判断有没有匹配 |
| 3 | `种子文档.md`（当前版本） | 六要素细节——判断涉及的制度/场景是否需要补充 |
| 4 | `活记忆/活记忆.yaml` | 种子追踪组件——本章要呼应的已埋伏笔的细节 |
| 5 | `references/信息获取强制化SOP.md` | 强制化流程+知识库目录结构+索引格式+写入流程 |

## 做什么

### 1. 强制读索引（红线❌6门禁）

**第一步永远是读索引。** 打开 `写作资产/知识库/索引.md`，查看已沉淀的知识库文件清单。

- 索引不存在 → 首次执行，创建知识库目录+索引文件（见SOP的目录结构说明）
- 索引已读 → 进入步骤2

**门禁：如果索引.md未被读取，退回补读。不允许跳过。**

### 2. 基于chapter_plan做4类检查

从chapter_plan的场景设计、线索推进、爽点设计、危机设计中提取信息需求：

| 检查类型 | 判断问题 | 信息源 | 示例 |
|:---------|:---------|:-------|:-----|
| 制度/流程细节 | 规划中的场景涉及什么我不熟悉的制度/流程？ | WebSearch | 催债人上门→查美国催债法流程 |
| 场景技法 | 规划中的场景类型需要参考技法吗？ | library套路库 | 帮派对峙→查套路库的压迫感营造 |
| 已埋伏笔/前文细节 | 规划中要推进的线索需要呼应哪些前文？ | 项目本身（活记忆+前章正文） | MS-13线索→查ch1的老约翰之死细节 |
| 创意参考 | 规划中有什么场景需要灵感参考？ | WebSearch/library | 绝境反杀→查类似场景的节奏控制 |

### 3. 索引匹配判断

对每个信息需求，先查索引中是否已有匹配的知识库文件：

```
信息需求 → 查索引
  ├── 索引中有匹配文件 → 读该知识库文件 → 提取关键信息 → 标记source: 知识库/{文件名}
  └── 索引中无匹配文件 → WebSearch/library查询 → 提取关键信息 → 写入知识库 → 更新索引
```

### 4. 信息获取（如有需求）

按需求清单逐项获取：

**4a. 索引中有匹配 → 读知识库文件**
- 从 `写作资产/知识库/{分类}/{文件名}.md` 读取
- 提取关键信息融入写作上下文
- 在info_acquired记录中标记 `source: 知识库/{文件名}.md`

**4b. 索引中无匹配 → WebSearch并写入知识库**
- WebSearch搜索制度/流程/创意参考
- 提取关键信息：流程步骤、关键规则、现实约束
- **写入知识库**：创建 `写作资产/知识库/{分类}/{文件名}.md`，写入搜索结果
- **更新索引**：在 `写作资产/知识库/索引.md` 追加条目
- 在info_acquired记录中标记 `source: WebSearch→知识库/{文件名}.md`

**4c. library查询（套路库）**
- 查询 `pop-trope-library/套路库/{套路名}.md`
- 提取场景技法/渲染参考
- 在info_acquired记录中标记 `source: library套路库/{套路名}.md`

**4d. 项目本身（活记忆+前章正文）**
- 查阅活记忆种子追踪组件：已埋伏笔清单+状态
- 查阅前章正文：具体伏笔细节
- 提取待回收伏笔的内容和预期回收点
- 在info_acquired记录中标记 `source: 活记忆+前章正文`

### 5. 无需求处理

4类检查均无明确需求 → `needs: []`，这是合法的。但**索引必须已读取**（红线❌6）。

**核心原则：允许"无需求"，但不允许"不读索引"。** 常规章节（日常生存、已建立机制的运作）可以无需求，但每章都必须读索引确认。

### 6. 登记记录

按 `templates/信息获取记录-模板.md` 格式，填写info_acquired记录（YAML），写入活记忆event的`info_acquired`字段。

## 门禁

| 检查项 | 失败动作 |
|:-------|:---------|
| **索引.md未读取（红线❌6）** | **退回补读索引** |
| chapter_plan不存在 | 退回Step 0补做本章规划 |
| 信息获取记录未填写 | 补填后再进Step 2 |
| WebSearch获取的信息未写入知识库 | 补写入后再继续 |
| 知识库新文件未更新索引 | 补更新索引后再继续 |

> 注意：本步骤有"必须读索引"门禁（红线❌6），但允许"无需求"（needs: []）。

## 产出

信息获取记录（YAML格式），写入活记忆event的`info_acquired`字段。包含：
- chapter（章号）
- index_read（索引已读取标记）
- needs（4类检查结果清单，或空数组）
- merged（是否已融入写作上下文）
- kb_new_files（本章新建的知识库文件列表）
- index_updated（索引是否已更新）

## 产出自检

- [ ] 基于chapter_plan做4类检查
- [ ] 索引匹配判断已执行（有匹配→读知识库文件；无匹配→WebSearch）
- [ ] WebSearch获取的信息已写入知识库
- [ ] 知识库新文件已更新索引
- [ ] 需要查的信息已获取并记录result
- [ ] 不需要查的已标记needs: []
- [ ] info_acquired记录已填写
