# Step 5：记忆更新+种子生长+方向提示（主会话机械执行）

> 管线: pop-writer-v3-emerge v1.2.0
> 类型: **主会话**（机械执行，不做判断）
> 消费: Step 4质检报告 + 修订稿 + 活记忆 + 种子文档
> 产出: 活记忆event追加 + 种子文档（如生长则更新版本） + 方向提示
> 红线: ❌3 活记忆唯一写入者；❌4 种子生长时必须更新版本号+变更日志

## 目的

质检通过后，主会话**机械执行**记忆更新。不做任何判断——种子生长判断已在Step 4质检子skill完成，本步骤只是按质检报告的结论机械写入。

**v3.1核心改动：** v3.0的种子生长判断和写入都在Step 4同会话完成，判断在context污染下退化（27章0次生长）。v3.1把判断拆给质检子skill（Step 4），写入归主会话（Step 5）机械执行。判断和执行分离。

## 读什么

| # | 文件 | 为什么 |
|:-:|:-----|:-------|
| 1 | Step 4质检报告 | 种子生长建议+不达标处理标记+方向提示来源 |
| 2 | Step 3修订稿 | 提取本章状态变化（机械提取，不做判断） |
| 3 | `活记忆/活记忆.yaml`（最后baseline+event） | 追加event的基准 |
| 4 | `种子文档.md`（当前版本） | 种子生长写入的基础 |
| 5 | `references/活种子生长触发规则.md` | 生长执行流程+版本管理规则 |

## 做什么

### 1. 活记忆七组件更新（红线❌3：唯一写入者）

**机械执行：** 从修订稿中提取本章状态变化，向 `活记忆/活记忆.yaml` 追加本章event。

| 组件 | 更新内容 | 数据形态 |
|:-----|:---------|:---------|
| 种子追踪 | 本章新埋伏笔（章号+内容+预期回收点）；已回收伏笔标记状态变更 | 列表追加 |
| 压力状态 | 倒计时当前值变化；新压力源激活 | 键值对更新 |
| 角色状态 | 主角的数值/心理/资源/关系变化增量 | 章末追加 |
| 世界规则 | 本章新确立的规则（只增不改） | 规则表追加 |
| 节奏日志 | 本章事件密度/爽点数/钩子类型 | 章号+三项指标 |
| 战力曲线 | 主角能力值变化（章号+能力值） | 折线数据追加 |
| 目的地进度 | 目的地到达状态变化 | 状态更新 |

event条目格式：
```yaml
- type: event
  chapter: {章号}
  timestamp: ...
  chapter_plan: {}          # Step 0产出
  info_acquired: {}         # Step 1产出
  creation_log: {}          # Step 2产出（创作决策记录）
  revision_log: {}          # Step 3产出（修订记录）
  qa_report: {}             # Step 4产出（质检报告摘要）
  seeds_added: []           # 新埋伏笔
  pressure_changes: {}      # 压力变化
  character_changes: {}     # 角色变化
  world_rules_added: []     # 新世界规则
  rhythm_entry: {events: N, rewards: N, hook_type: ...}
  power_entry: {ch: N, ...}
  destination_updates: []
  seed_growths: []          # 种子生长记录（如有，来自质检报告）
  direction_hint: ""        # 本步骤填写
```

### 2. 种子生长执行（机械写入，红线❌4）

**机械执行：** 读Step 4质检报告中的"种子生长建议"，如有`追加`建议则执行：

1. 更新种子文档对应要素
2. **更新该要素的`last_updated_ch`字段为当前章号**
3. 版本号 minor+1（如 1.0.0→1.1.0）
4. 变更日志追加记录
5. 同步更新活记忆对应组件
6. 在event的`seed_growths`字段记录生长内容

变更日志格式：
```markdown
| 版本 | 章号 | 类型 | 变更内容 | 触发原因 |
|:-----|:-----|:-----|:---------|:---------|
| 1.1.0 | 5 | grow | 压力矩阵追加"强化剂" | ch5质检报告种子生长建议 |
```

**如质检报告无种子生长建议：**
- 不更新种子文档
- 不更新版本号
- 在event的`seed_growths`字段填空数组`[]`
- 活记忆照常更新七组件

### 3. 方向提示生成（机械转化）

**机械执行：** 读Step 4质检报告中的"不达标处理标记"和"last_updated_ch新鲜度检查"，转化为下章方向提示。

方向提示是1-2句话的方向感，**不是章纲**。不指定具体事件——只说"需要给奖励、需要加压、需要触面某线索、需要即时钩子"。

方向提示来源（从质检报告机械提取）：
- 五问反思中不达标项的处理标记
- 爽点终验中不达标法则的处理标记
- 种子追踪组件的超期预警伏笔
- 压力状态的倒计时停滞情况
- 活跃线索的长期未触及情况
- last_updated_ch > 15章的要素（需关注标记）

方向提示格式：
```
方向提示 ch{下一章号}:
- {方向项1}
- {方向项2}
```

方向项类型（非穷举）：
- **需要给奖励**：上章只有信息无实质获得
- **需要加压**：某倒计时已N章未推进
- **需要触面某线索**：某活跃线索已N章未出现
- **需要即时钩子**：上章钩子不够即时
- **需要加速/减速**：事件密度偏离趋势
- **需关注某要素**：某种子要素last_updated_ch>15章

### 4. 回写方向提示到活记忆

将方向提示写入本章event的`direction_hint`字段。

## 门禁

| 检查项 | 失败动作 |
|:-------|:---------|
| 活记忆event未追加（红线❌3） | 补追加后再继续 |
| 七组件未完整更新 | 补全后再继续 |
| 种子生长但未更新版本号（红线❌4） | 补更新版本号+变更日志 |
| 种子生长但未更新变更日志（红线❌4） | 补追加变更日志 |
| 种子生长但未更新last_updated_ch | 补更新后再继续 |
| 种子生长但活记忆未同步 | 补同步后再继续 |
| 方向提示未生成 | 补生成后再继续 |
| 方向提示写了具体事件（非方向感） | 退回，改为方向感 |
| 活记忆direction_hint未回写 | 补回写后再继续 |

## 产出

1. `活记忆/活记忆.yaml` — 追加本章event（七组件更新+direction_hint回写）
2. `种子文档.md` — 如质检报告有生长建议，更新要素+last_updated_ch+版本号+变更日志
3. event的`seed_growths`字段 — 记录生长内容（或空数组）
4. event的`direction_hint`字段 — 方向提示

- [ ] 如执行生长：活记忆对应组件已同步
