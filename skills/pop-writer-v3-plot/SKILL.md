# Pop Writer V3 Plot

plot 负责解决“如何正向写出逆向拆书级别的剧情单元卡”。它不是一次性出稿器，也不是把一个想法从头顺到尾的单线程推演器，而是可被用户中途干预的剧情规划工作台。

底层原则：**先发散事件源，再造故事引擎，再让互斥方案竞争。** 一个合格幕纲不是“我能解释它为什么成立”，而是“library 案例、外部类型母版、强制逻辑发散、角色视角事件池多路信息碰撞后，至少三套不同剧情引擎 PK 出来的胜者”。

plot 的核心失败模式是：先想到 A，再写一个弱化版 B，最后证明 A 对。此模式禁止。

候选竞技的核心失败模式是：A 写完整逐章运行，B/C 只写框架或差异标注。此模式禁止。三套候选必须同等颗粒度、同等章节数、同等质量门，再进入 PK。

卷纲的核心失败模式是：把“卷级战略锚”写成“弱版幕纲”。此模式禁止。卷纲只定义商业承诺、方向边界和资源预算；幕纲才负责具体事件运行。最终幕纲还必须包含章级导演包，供 create 直接施工。

## 核心流程

0. 全书/卷级尺度契约与卷级设计意图：先确认全书/当前卷体量、当前卷商业承诺、主角变化、主压力、主收益、禁区和不可牺牲项；禁止写幕细节。
1. 卷级战略锚、卷内幕序列与当前幕切片：先输出卷级战略锚和 5-7 个幕的功能位，再锁定当前幕编号、章节预算和功能位；仍禁止写逐章剧情。
2. 任务定位与案例消化：按任务读案例，提取结构、爽点证据、内容肌理、不可照搬项；同时主动做外部母版扫描，不等用户提醒。
3. 事件源发散：从 library 调研、外部类型母版、逻辑强制发散、角色视角事件池多路生成事件原料。
4. 事件碰撞与筛选：标注相似、相反、盲区、互斥、可融合、强因果事件，形成候选事件矿池。
5. 故事引擎铸造：从事件矿池中组装“压力引擎、收益引擎、敌人误判引擎、主角操作引擎、情绪/爽点曲线、长线债务”。
6. 互斥候选方案：至少生成 3 套不同剧情引擎驱动的方案。每套必须有不同主压力、不同第一主动动作、不同收益路径、不同敌人误判链。
7. 竞技场 PK：逐章对比三套方案，按追读、爽点、主角操作、压力升级、设定行动化、正文可写性打分，必须淘汰至少一套。
8. 胜者合成：只允许吸收败者的局部强点，不允许把三套方案堆叠成杂烩。
9. 反证式质量门：缺单章导演意图、状态快照、设定引用、可数收益、爽点外显、事件拍点、create交接，均不可输出最终稿。
10. 设定账本增量：运行中产生的新规则、新组织、新物品，只冻结必要部分。
11. 幕纲压缩：把胜出仿真整理成可供 create 直接写正文的唯一可用稿。

## 卷纲定位

卷纲不是剧情细纲，也不是幕纲的上级完整版。卷纲是战略契约层，只回答“这一卷承诺给读者什么、不能越过什么边界、当前幕在卷内承担什么功能”。

卷纲只能承担三类职责：

- 商业承诺：本卷读者追什么，主压力是什么，主爽感是什么，主角最终拿到什么。
- 方向边界：哪些收益不能提前兑现，哪些设定不能提前揭露，哪些敌人/组织/资源不能跳级解决。
- 资源预算：本卷约多少章，拆成几个幕，当前幕约多少章，哪些幕只做功能位不做细节。

卷纲不做这些事：

- 不写逐章剧情。
- 不生成事件矿池。
- 不做三候选竞技。
- 不替代幕纲给 create。
- 不把“以后可能发生的桥段”伪装成本卷承诺。

卷纲产物建议分两份：

| 文件 | 作用 | 可否给 create |
|:-----|:-----|:---:|
| `卷纲/运行/卷N-00-全书卷级尺度契约.md` | 用户确认全书/当前卷体量 | 否 |
| `卷纲/运行/卷N-00b-卷级设计意图契约.md` | 用户确认当前卷商业承诺、主角变化、禁区、不可牺牲项 | 否 |
| `卷纲/卷N-战略锚.md` | 用户确认本卷商业承诺、方向边界、章节预算、禁区 | 否 |
| `卷纲/卷N-幕序列.md` | 只列 5-7 个幕的功能位和预期收益，不写细节 | 否 |
| `卷纲/运行/幕XXX-01-当前幕切片确认.md` | 用户确认本轮深推演哪一幕、该幕功能位和边界 | 否 |

幕纲才是执行剧本。`卷纲/幕纲-幕XXX-{标题}.md` 是 create 的唯一剧情输入。

## 事件源发散框架

plot 不得直接从“我觉得该这样写”进入方案。必须先生成事件矿池，类似 pop-research 的三界调研：先多元发散，再交叉困境，再收束。

### 事件源四路

| 来源 | 产出 | 作用 |
|:-----|:-----|:-----|
| Library 调研 | 从同任务 L2/L3、套路库、金手指库提取事件结构、爽点证据、内容肌理 | 提供“优秀网文已经验证过”的结构标杆 |
| 外部类型母版 | 默认从 seed 母版雷达、用户指定母版、agent 主动扫描的影视/美剧/漫画/游戏/电影中提取类型骨架 | 提供 library 不足时的强剧情引擎，如倒计时、追踪链、竞技赛制、逃杀、潜入、复仇、抢劫、救援 |
| 逻辑强制发散 | 用相似、相反、盲区、升级、代价、资源、误判、制度、金手指限制等维度生成事件 | 防止 agent 只沿第一想法顺滑推进 |
| 角色视角事件池 | 让本幕重要角色各自从“我想要什么/我误判什么/我能做什么”出发生成事件 | 让剧情像多方行动碰撞，而不是作者推着主角走 |

### 外部类型母版移植

每次 plot 都必须产出外部类型母版卡，不等用户提醒。来源优先级：

1. 用户明确提到的外部作品或桥段。
2. `library/设定账本/跨媒介母版雷达.md`。
3. agent 根据当前任务标签主动扫描的经典影视/美剧/漫画/游戏/电影母版。
4. 若当前环境有搜索工具，可做简短检索；若无搜索工具，用模型已知经典母版，并声明“未联网验证”。

不要照搬角色、台词、专有设定，只抽取类型功能和节奏骨架。

| 字段 | 要求 |
|:-----|:-----|
| 母版名 | 如《飓风营救》、疾速追杀、谍影重重、异形、黑袍纠察队、魂类游戏、Roguelike等 |
| 核心类型承诺 | 读者/观众追什么：救人、复仇、逃杀、潜入、抢劫、揭谜、赛制晋级 |
| 关键机制 | 倒计时、追踪链、每个敌人释放下一地图、密室压迫、猎杀名单、规则漏洞等 |
| 可迁移事件 | 可进入本项目的事件原料，写成 `M-E01` 形式 |
| 不可照搬项 | 原作角色、地名、台词、专有桥段、版权化设定 |
| 本项目变体 | 如何变成当前世界观、金手指、主角性格和爽文反馈 |

外部母版不是“可选点评”。每次 plot 至少扫描 3 个候选母版，采用 1-2 个进入事件矿池；若全部淘汰，必须说明为什么本幕不适用。

### 母版本土化门

外部母版只能增强本幕设计目的，不能夺走本幕功能。采用前必须回答：

| 检查项 | 必须成立 |
|:-----|:-----|
| 本幕功能 | 母版是否服务当前幕功能位，而不是改写成另一部作品的第一幕 |
| 网文爽点 | 是否保留主角主动、可数收益、面板/资源/身份反馈、章末追读 |
| 项目核心引擎 | 是否保留本书主角引擎、金手指规则、世界压力和卷级商业承诺 |
| 情感锚点 | 情感角色不能只变成“被救目标”，除非本幕目的就是救援；必须保留其对主角选择的持续影响 |
| 长篇生长 | 母版不能一次性吃掉卷级长期压力、长期敌人、成长空间 |
| 本土化方式 | 外部机制必须变成当前世界的规则、资源、误判、收益或代价 |

如果母版导致“网文开局功能”被救援片/电影三幕式替代，必须降级为局部机制，只抽取倒计时、追踪链、线索解锁等部件，不允许作为唯一骨架。

### 角色视角事件池

每个重要角色/势力至少输出 3 个事件意图：

| 角色/势力 | 当前目标 | 掌握信息 | 误判 | 会主动做什么 | 会制造的事件 | 与主角碰撞点 |
| --- | --- | --- | --- | --- | --- | --- |

至少包含：

- 主角。
- 核心敌人或压力源。
- 情感锚点角色。
- 信息/资源中介。
- 隐藏势力或长线债务代表。

如果某角色在本幕“重要”，但不能主动制造事件，说明它只是背景，不应占用重要出场位。

### 事件碰撞分类

事件矿池必须按以下方式标注：

- 相似事件：多个来源都指向同一功能，说明它可能是本幕核心。
- 相反事件：两个事件方向互斥，适合生成 A/B 候选。
- 盲区事件：library/角色/逻辑有一方提到，其他来源没覆盖，可能是新鲜点。
- 强因果事件：不发生它，下一章无法成立。
- 诱饵事件：看起来有趣但不改变压力、收益或关系，优先淘汰。
- 可融合事件：两个事件融合后能同时推进至少 2 条子线。

没有事件矿池，不得进入故事引擎铸造。

## 故事引擎定义

故事引擎是“为什么这一幕会自己往前滚”的动力系统，不是主题说明，也不是设定摘要。

每个候选方案必须先写清自己的六个引擎：

| 引擎 | 必须回答 |
|:-----|:---------|
| 压力引擎 | 谁在逼主角马上行动？如果主角不动，下一章会损失什么？ |
| 收益引擎 | 主角每章拿到什么可数收益？这个收益如何改变下一章选择空间？ |
| 主角操作引擎 | 主角靠什么主动赢：规则知识、环境利用、信息差、话术、战术、资源交换还是牺牲代价？ |
| 敌人误判引擎 | 敌人根据什么合理情报误判主角？误判如何逐章升级而不是送人头？ |
| 情绪/爽点引擎 | 每章让读者爽什么：反杀、升级、到账、身份变化、旁人震惊、日常回暖、钩子恐惧？ |
| 长线债务引擎 | 本幕留下什么必须追的承诺、威胁或谜题？触发条件是什么？ |

如果一个方案不能写出六个引擎，它不是方案，只是剧情摘要。

## 候选方案竞技规则

优先使用真正独立的候选生成：

1. 如果运行环境支持子 Agent/并行写作工位，必须派出 3 个候选工位，分别生成 A/B/C，互不读取彼此草稿。
2. 如果当前环境不能实际调子 Agent，也必须在同一回复中模拟“隔离工位”：先写 A 完整引擎与逐章链，再写 B，再写 C；禁止边写边评、禁止让 B/C 只是 A 的微调。
3. 三套方案必须同等颗粒度：同样章节数，同样每章 5-8 个拍点，同样有状态、主动动作、误判链、收益、钩子、证据引用。禁止 A 是完整稿，B/C 是概览或差异标注。
4. 三套方案必须互斥：不能只改章节顺序或侧重点，必须在主类型母版、主压力来源、主角第一主动动作、收益路径、敌人误判链、幕终点收益中至少 4 项不同。
5. 如果三套候选共享同一首章、同一中段关键战、同一幕终点，只能算同一方案的变体，必须重跑至少 2 套。
6. PK 前不得预设推荐方案。审计必须允许 A 被淘汰。
7. 竞技过程必须落盘到 `卷纲/运行/幕XXX-03-候选竞技.md`，用户能看到三套方案为什么赢/输。

## 产物分层

plot 可以产出过程文件，但必须让用户一眼知道“哪个能用”。所有产物分三层：

| 层级 | 路径 | 用途 | 是否可直接给 create |
|:-----|:-----|:-----|:---:|
| 卷级战略稿 | `卷纲/卷N-战略锚.md`、`卷纲/卷N-幕序列.md` | 给用户确认卷方向、商业承诺、体量、禁区和幕功能位 | 否 |
| 过程文件 | `卷纲/运行/幕XXX-*` | 复盘、审计、中途干预、定位失败原因 | 否 |
| 唯一可用幕纲 | `卷纲/幕纲-幕XXX-{标题}.md` | create 的唯一直接输入 | 是，且必须通过质量门 |
| 章级导演包 | 包含在最终幕纲内，或另存 `卷纲/章节设计包/幕XXX-chYYY-设计包.md` | 每章正文施工卡 | 是 |

每次 plot 完成后必须写 `卷纲/运行/幕XXX-产物索引.md`，列出本轮所有文件状态：

```markdown
# 幕XXX 产物索引

| 文件 | 类型 | 状态 | 用途 | 可否给create |
| --- | --- | --- | --- | --- |
| 卷纲/运行/幕XXX-案例消化与证据绑定.md | 过程 | 过程资料 | 证明参考案例怎么被消化 | 否 |
| 卷纲/运行/幕XXX-运行日志.md | 过程 | 过程资料 | 展示仿真经过 | 否 |
| 卷纲/运行/幕XXX-审计.md | 过程 | 过程资料 | 记录淘汰和风险 | 否 |
```

## 最终幕纲质量门

最终幕纲如果缺以下任一项，只能标记为“草案”，不得写入 `library/单元剧情卡/`，不得声称“可直接写正文”：

- 每章有“导演意图”：叙事功能、事件动作链、情绪/爽点曲线、章末钩子。
- 每章有“开章状态快照”：人物位置、资源、伤势、压力、未兑现承诺。
- 每章有“设定引用”：本章需要读取哪些设定账本/文风DNA/金手指文件。
- 每章有“案例证据引用”：S/J/T 证据 ID 如何变成具体事件。
- 每章有“主角主动动作”：不是醒来、发现、被告知，而是主动选择推动局势。
- 每章有“可数收益/明确损失”：经验、钱、装备、线索、声望、关系、伤势、代价至少一种。
- 每章有“爽点外显方式”：面板、数字、敌人误判、旁人反应、资源到账、身份变化至少一种。
- 每章有“create 交接”：不可改事件、可自由发挥空间、文风注意、禁改设定。
- 每章有“章级导演包”：地点、在场人物、POV、事件链、关键对白方向、感官锚点、爽点外显、章末钩子、状态更新。

## 执行

- 读 `steps/step-0-scale-contract.md` 确认全书/卷级尺度和卷级设计意图。
- 读 `steps/step-1-volume-anchor.md` 确定卷级战略锚、卷内幕序列和当前幕切片。
- 读 `steps/step-2-case-digest.md` 做公共案例消化。
- 读 `steps/step-3-simulation.md` 做故事引擎铸造、互斥候选方案和竞技场 PK。

## 红线

- 不允许黑箱一次性合并 Step 1-4。过程文件必须落盘到 `卷纲/运行/`。
- 不允许把过程文件伪装成可用稿。过程文件只服务复盘和中途干预，不能直接交给 create。
- 不允许把卷纲写成弱版幕纲。卷纲不得包含逐章剧情、事件矿池、候选竞技、章节运行日志。
- 不允许越级设计。Step 0 只做全书/卷级，Step 1 才切当前幕，Step 2 之后才允许进入案例和剧情事件。
- 不允许跳过卷级设计意图契约。卷纲/幕纲深推演前必须明确本卷要服务什么读者承诺、主角变化、禁区和不可牺牲项。
- 不允许把未通过质量门的幕纲写入 `library/单元剧情卡/`。
- 不允许在最终回复里只说“全部完成”。必须明确“唯一可用稿是哪一个、哪些文件只是过程资料、当前是否通过质量门”。
- 不允许默认短篇化。常规目标按全文 500 章以上、卷 60-80 章、幕 10-15 章理解；压缩详细展开可以，但结构尺度不能塌。
- 不允许在用户未确认卷级战略锚和当前幕切片时直接深推演，除非用户明确说“无需确认/全自动跑完”。
- 不允许把写作规划交给隐藏子 Agent 后只回一个完成报告；用户需要能看见中间状态。
- 不用填表式写作环。
- 不允许“读了案例但没有案例消化卡”。
- 不允许只抽象结构，必须保留爽点证据和内容肌理。
- 不允许案例消化只留在表格里。章节运行日志必须能看见案例证据如何变成事件选择。
- 不允许只生成一个方案，也不允许生成“强A+弱B”的假多方案。至少三套互斥方案，且必须淘汰至少一套。
- 不允许把 PK 写成事后证明推荐方案正确。胜负必须来自逐章证据。
- 不允许用“审计全通过”代替证据。每个通过项必须给出具体事件证据；证据不足则重跑仿真。
- 不允许每章只有一句剧情摘要。每章必须有可写成正文的场面拍点和状态变化。
- 不允许外部母版夺舍。外部作品只能提供类型机制，不得替代本书的网文爽点、主角成长、金手指反馈和卷级长期压力。
- 不做空泛爽点标签，爽点必须来自局势变化。
- 不一次性设计全世界设定，只冻结本单元运行必需的设定。
- 每章必须能被 create 直接写成正文。
- 不允许上游 PRD/L1 世界包/动态升级表缺失时硬写正式幕纲。
