# Pop Writer V3 Plot

plot 负责解决“如何正向写出逆向拆书级别的剧情单元卡”。它不是一次性出稿器，而是可被用户中途干预的剧情规划工作台。先对齐体量和卷幕尺度，再确定当前写作任务，从公共库多案例消化“结构 + 爽点证据 + 内容肌理”，把案例证据带入多套剧情仿真，经过反证式审计淘汰，最后压缩成能直接指导 create 的幕纲。

## 输入

- seed 产出的世界观底盘、主角引擎、金手指规则、文风DNA、初始设定账本。
- 上一单元 arc 复盘或当前活记忆。
- 用户指定的目标：卷方向、单元目标、章节数、爽点偏好、禁忌。
- 公共资料库：优先读取 `pop-trope-library` 的 `任务索引/`，再按索引指向精读剧情库 L2/L3、套路库、设定库、金手指库。没有任务索引时，临时按当前任务反推候选案例，并输出任务索引补充建议。

## 公共知识库

路径：`$env:APPDATA\popwave\remote-skills\pop-trope-library`

| 子库 | 用途 |
|:-----|:-----|
| `任务索引/` | 写作任务入口，先回答“当前任务该读哪几张案例、读哪些段、学什么” |
| `剧情库/` | 按书保存的 L2 单元卡和 L3 剧情线原始高保真资产，由任务索引指向后精读 |
| `套路库/` | 场景技法（战斗/扮猪吃虎/绝境反杀等），部署套路时查节奏控制段 |

查库方式：先读 `$env:APPDATA\popwave\remote-skills\pop-trope-library\任务索引\00-索引.md`，进入匹配任务索引，再读取索引推荐的 L2/L3。任务索引缺失时，才回退到 `剧情库/00-索引.md` 按书筛选候选。

## 核心流程

0. 尺度契约：确认全文/卷/幕/章数尺度，不能默认把长篇网文压成短篇试跑。
1. 卷幕方向草案：先给 2-3 个方向，说明体量、主压力、主收益、风险，让用户确认。
2. 任务定位：给当前幕打任务标签，例如开局、死者还魂、系统觉醒、低资源求生、第一次升级反馈、追杀反杀、据点经营。
3. 案例消化：读取 3-5 个相关案例，保留结构骨架、爽点证据、内容肌理、不可照搬项、本项目映射。
4. 证据绑定：给可迁移结构、爽点证据、内容肌理编号，后续章节日志必须引用这些编号或说明为何不用。
5. 状态契约：人物位置、资源、能力限制、敌我压力、未兑现承诺、潜在种子。
6. 多方案仿真：至少生成 2 套完整幕运行方案，每套逐章跑物理坐标、事件拍点、准备链、误判链、收益、设定显影、钩子。
7. 反证式审计：淘汰主角不主动、收益不可数、压力不升级、敌人无误判、准备链不足、章节场面密度不足、案例证据未落地的方案。
8. 设定账本增量：运行中产生的新规则、新组织、新物品，只冻结必要部分。
9. 幕纲压缩：把胜出仿真整理成可供 create 直接写正文的单元剧情卡。

## 执行

- 读 `steps/step-0-scale-contract.md` 确认体量和用户对齐点。
- 读 `steps/step-1-volume-anchor.md` 确定卷级方向。
- 读 `steps/step-2-case-digest.md` 做公共案例消化。
- 读 `steps/step-3-simulation.md` 做多方案剧情仿真。
- 读 `steps/step-4-judge-compile.md` 审计并压缩幕纲。
- 按 `templates/幕纲-模板.md` 输出。

## 红线

- 不允许黑箱一次性合并 Step 1-4。过程文件必须落盘到 `卷纲/运行/`。
- 不允许默认短篇化。常规目标按全文 500 章以上、卷 60-80 章、幕 10-15 章理解；压缩详细展开可以，但结构尺度不能塌。
- 不允许在用户未确认卷幕方向时直接深推演，除非用户明确说“无需确认/全自动跑完”。
- 不允许把写作规划交给隐藏子 Agent 后只回一个完成报告；用户需要能看见中间状态。
- 不用填表式写作环。
- 不允许“读了案例但没有案例消化卡”。
- 不允许只抽象结构，必须保留爽点证据和内容肌理。
- 不允许案例消化只留在表格里。章节运行日志必须能看见案例证据如何变成事件选择。
- 不允许只生成一个方案，必须多方案仿真后淘汰。
- 不允许用“审计全通过”代替证据。每个通过项必须给出具体事件证据；证据不足则重跑仿真。
- 不允许每章只有一句剧情摘要。每章必须有可写成正文的场面拍点和状态变化。
- 不做空泛爽点标签，爽点必须来自局势变化。
- 不一次性设计全世界设定，只冻结本单元运行必需的设定。
- 每章必须能被 create 直接写成正文。
