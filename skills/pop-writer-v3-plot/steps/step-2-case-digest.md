# Step 2 - 案例消化

先解决“当前写作任务要向哪些成功案例学习”，再开始设计剧情。

## 1. 任务标签

给当前单元打 3-7 个标签，例如：

- 开局
- 死者还魂
- 系统觉醒
- 低资源求生
- 第一次升级反馈
- 信息差收益
- 追杀反杀
- 据点经营
- 势力初建
- 大势力博弈

任务标签优先于书名。写开局就找开局案例，写追杀就找追杀案例。

## 0. 输出路径

```text
卷纲/运行/幕XXX-02-案例消化与证据绑定.md
```

案例消化是可复盘过程文件，不要只塞进最终幕纲。

## 2. 公共库读取

优先查 `pop-trope-library`：

- `任务索引/`：写作任务入口，优先读 `任务索引/00-索引.md`，进入匹配任务索引。
- `设定库/`：世界如何运行。
- `剧情库/`：故事如何推进，由任务索引指向后精读 L2 单元卡和 L3 剧情线卡。
- `套路库/`：场面节奏和爽点模式。
- `金手指库/`：金手指的 WHAT 层机制。
- `文风库/`：只给 create/prose 阶段做语感参考，plot 不用它替代剧情判断。

执行顺序：

1. 读 `任务索引/00-索引.md`。
2. 进入最匹配的任务索引，读取“必读案例”和“配套素材”。
3. 按索引说明精读案例中的指定段落。
4. 如果 library 暂无匹配任务索引，临时按任务标签反查 `剧情库/00-索引.md` 和书名目录；同时在输出中写出“建议新增的任务索引条目”。

## 3. 案例数量

读取 3-5 个案例。至少包含：

- 1 个同任务案例。
- 1 个同爽点案例。
- 1 个同机制或同赛道案例。

案例不足时要声明库缺口，不得假装已经覆盖。

## 4. 案例消化卡

每个案例输出：

- 可迁移结构：事件动力学，不是剧情摘要。每条编号为 `S{序号}`。
- 爽点证据：主角主动、敌人误判、信息差、可数收益、章末追读分别在哪里发生。每条编号为 `J{序号}`。
- 内容肌理：低资源感、危险质感、系统反馈、社会制度讽刺、战利品质感等可迁移元素。每条编号为 `T{序号}`。
- 不可照搬项：专属角色、专属地名、专属世界规则、原书独有设定。
- 本项目映射：对应主角处境、资源、敌人、第一次收益、更大压力。

格式优先遵守 `pop-trope-library/references/案例消化卡-标准格式.md`。

## 5. 证据绑定清单

案例消化末尾必须输出“本单元必落地证据”，选择 6-10 条最关键证据：

| 证据ID | 来自案例 | 类型 | 本单元落地方式 | 预计章节 |
| --- | --- | --- | --- | --- |
| S1 |  | 结构 |  |  |
| J1 |  | 爽点 |  |  |
| T1 |  | 肌理 |  |  |

后续多方案仿真和章节运行日志必须引用这些证据 ID。没有被使用的证据，要在审计中说明删除原因。

## 5. 禁止

- 禁止只写“参考了某某案例”。
- 禁止只抽象结构而不保留爽点证据。
- 禁止不编号。没有证据 ID，后续无法检查案例是否真的落地。
- 禁止照搬案例内容。
- 禁止用单一案例支配整个单元。
