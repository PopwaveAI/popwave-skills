# Step 2：L2 剧情单元卡设计

> 定位：v3.5 plot 的核心。设计跨章叙事结构——L2 剧情单元卡（4-10章/卡）。L2 卡是唯一运行时活文档，吸收了种子文档的全部运行时功能。
>
> 粒度卡在"导演意图"层：定义"这组章节要达到什么效果"，不预设具体场景和台词。

## 消费
- `卷纲/卷N-方向锚.md`（step-1 产出）
- **优先级1**：前一个 L2 卡实际内容 `卷纲/L2-NNN.md`（arc 更新后）——必须（L2-001 除外）
- **优先级2**：`写作参考/设定/`（主角引擎/金手指/压力矩阵/冲突轴/核心卖点）——必须
- **优先级3**：library 匹配的同类 L2 卡（2-3张）——必须
- **优先级4**：`卷纲/L3-NNN.md`（如有）——可选（L2-001 除外）
- `写作参考/索引.md`（本地已有设定/知识沉淀总目录）
- 用户输入

## 产出
- `卷纲/L2-NNN.md`（L2 剧情单元卡，唯一运行时活文档）
- （如有缺口）`写作参考/知识沉淀/{主题}.md`（调研产出，由 pop-research 落盘）

## 做什么

### 1. 多层信息依赖读取（按优先级）

按优先级逐层读取信息，确保设计有足够信息支撑：

| 序 | 动作 | 说明 |
|:---|:-----|:-----|
| 1.1 | 读前一个 L2 卡实际内容 | L2-001 跳过。读取 arc 更新后的物理坐标段+嵌套子线状态，掌握主角当前状态、子线延伸状态、未回收伏笔 |
| 1.2 | 读写作参考/设定/ | 全局设定 ground truth：主角引擎（驱动力+行为准则）、金手指（能力+限制+行动驱动）、压力矩阵、冲突轴、核心卖点 |
| 1.3 | library 匹配同类 L2 卡 | 确定当前 L1 大类（建置/崛起/征服/...），从 library 剧情库匹配 2-3 张同类 L2 卡作为结构标杆。**记录参考卡路径** |
| 1.4 | 读 L3 卡（如已有） | 宏观参考：剧情线推进进度+待回收债务。无则跳过 |

> 红线❌3：library 参考是必须的。v2 雪花法失败根因是信息注入不足——agent 没见过好的结构，设计出来的东西是空洞的。参考的是结构骨架和设计维度，内容是当前项目自己的。
>
> **library 参考防污染指导（v1.3新增）：** 参考卡是"结构骨架标杆"，不是"内容模板"。读取library L2卡时，提取的是：
> - 该单元的叙事功能如何与前后单元咬合
> - 子线如何交叉产生戏剧性
> - 物理坐标如何记录状态变化
> - 事件brief的粒度标准
>
> **禁止从参考卡复刻到当前项目的内容：**
> - 参考卡里的具体角色名/势力名/地名（如"薇薇安""铁钳帮"是深渊主宰的，不是你的）
> - 参考卡里的具体能力/技能/物品名
> - 参考卡里的具体事件（如"战宠养成线"是我在美国搞内战的子线，不是你的）
> - 参考卡里的子线编号和名称
>
> 参考时自检：这个设计维度是结构性的（怎么咬合/怎么收束），还是内容性的（具体叫什么名字/什么能力）？只参考前者，不复刻后者。

### 2. 设计 L2 卡骨架

按标准格式（见 templates/L2卡-模板.md）设计：

#### 2.1 元数据
单元编号 / 单元名称 / 单元类型（建置型/推进型/高潮型/余波型）/ 章节范围（chXXX-chXXX，4-10章）/ 叙事功能 / 核心成就（单元结束时主角的状态变化，含数值）。

#### 2.2 物理坐标
- 时间轴：每章的时间节点
- 地点轨迹：场景空间变化
- 人物表：角色+身份+本章功能+状态变化
- 力量体系现状：单元结束时的数值快照

#### 2.3 一句话单元
一段话概括本单元发生了什么。

### 3. 结构分析表设计（核心）

每章一行，7列：

| 阶段 | 功能 | 章节 | 事件brief | 情感设计 | 节奏 | 设定引用 |
|:-----|:-----|:-----|:----------|:---------|:-----|:---------|

#### 3.1 事件brief——必须用动作链（红线❌2）

事件brief用**具体动作+因果链**，禁止纯功能描述：

| 粒度 | 示例 | 是否合格 |
|:-----|:-----|:---------|
| 功能描述（禁止） | "索伦趁乱清理铁钳帮据点" | 不合格——太抽象 |
| 动作链（要求） | "索伦从系统面板看到据点坐标→趁群龙无首潜入仓库→黑暗中逐个清除3个守卫→发现账本→带走战利品→撤离时看到教团资助文件" | 合格——有具体动作和因果 |

动作链要求：
- 用"→"串联具体动作
- 每个动作为可观察的具体行为（不是"清理""解决"这类抽象动词）
- 含因果（前一个动作导致/触发后一个动作）

#### 3.2 设定引用指针——必须每章标注（红线❌1）

结构分析表"设定引用"列，标注本章涉及的设定文件路径。这是 v3.5 核心机制——plot 设计时做信息需求预判，写作时 emerge 看到指针强制读取。

格式：`写作参考/设定/{文件}.md#{锚点}, 写作参考/设定/{文件}.md#{锚点}`

示例：
```
写作参考/设定/金手指.md#升级机制, 写作参考/设定/主角引擎.md#行为准则
```

要求：
- 设定引用是**路径指针，不是内容副本**
- 每章至少标注1个涉及的设定文件路径
- 路径必须真实存在（设计时验证）

### 4. 嵌套子线设计

每条子线一个表，标注延伸/闭合：

```markdown
### 子线A：{名称}（类型 | 状态：延伸/闭合）
| 阶段 | 章节 | 事件 | 不可逆变化 |
```

要求：
- 活跃子线≤5条（红线❌4，防膨胀，超出需说明必要性）
- 每条子线最后标注【延伸】或【闭合】
- 闭合子线：本单元内解决
- 延伸子线：跨单元延续，记录待回收状态

### 5. 子线咬合设计——因果推导（非矩阵填空）

子线咬合用**因果推导**，说明子线之间如何互相触发/因果连锁：

```
ch003咬合逻辑:
  ⑤铁钳帮反杀 → 产生①系统升级（经验来源）
  ①系统升级 → 产生④灵能耳鸣（升级副作用）
  ④灵能感知 → 引导③教团纹身发现
  因果链: ⑤ → ① → ④ → ③
```

要求：
- 用"A → B（原因）"格式说明因果
- 最后给出完整因果链
- 禁止矩阵填空（不要只画交叉表不说明因果）

### 6. 可迁移/不可迁移要点 + 单元边界信号

#### 可迁移要点
从本单元提炼的设计经验，供后续 L2 卡借鉴（如节奏模式、子线咬合手法、建置技法）。

#### 不可迁移部分
本单元专属的、不可复制到其他项目的要素。

#### 单元边界信号
- 起点信号：本单元开篇的触发状态
- 终点信号：本单元结束的收束状态（也是下一单元的起点）

### 7. 信息需求判断（三层流程）

设计完 L2 卡后，对照结构分析表的事件brief，执行信息需求判断：

#### Step 7.1 盘点本地已有
扫描 `写作参考/索引.md` + library 设定库，标记"本地有"的设定。

#### Step 7.2 识别缺口
对照 L2 卡事件brief，判断哪些场景/设定本地没有，标记"需要调研"的项。

#### Step 7.3 调研补缺——调用 pop-research

对识别出的缺口，调用 pop-research（独立调研skill）：

```
调用接口（传入）：
  - 调研需求：事件brief中涉及的具体场景/领域
  - 期望深度：≥500字，含具体案例/流程步骤/数据数值

pop-research 执行：
  - 三界框架定界 → 分界调研 → 压力源转化 → 交叉困境识别 → 质量检查
  - 存盘到 写作参考/知识沉淀/{主题}.md
  - 更新 写作参考/索引.md
  - 返回路径指针

plot 接收：
  - 将返回的路径指针写入 L2 卡结构分析表"设定引用"列
```

> 注意：pop-research 尚未创建时，本步骤预留接口。识别到缺口时记录缺口清单，标注"待调研"，待 pop-research 可用后补缺。设定引用列中本地已有的路径照常标注。

### 8. 产出 L2 卡存盘

将完整 L2 卡写入 `卷纲/L2-NNN.md`。

## 门禁（4条红线）

| 红线 | 要求 | 检查 |
|:-----|:-----|:-----|
| ❌1 设定引用列 | 结构分析表每章"设定引用"列非空且路径存在 | 逐章核对 |
| ❌2 动作链brief | 事件brief含"→"动作链+具体动作，禁止纯功能描述 | 抽查 |
| ❌3 library参考 | 已从 library 匹配≥2张同类L2卡并记录参考路径 | 确认路径 |
| ❌4 子线防膨胀 | 活跃嵌套子线≤5条 | 计数 |

## 产出自检

- [ ] library 参考卡路径已记录（红线❌3）
- [ ] L2 卡含全部标准段（元数据/物理坐标/一句话单元/结构分析表/嵌套子线/子线咬合/可迁移要点/不可迁移部分/单元边界信号）
- [ ] 结构分析表每章设定引用列非空（红线❌1）
- [ ] 事件brief用动作链（红线❌2）
- [ ] 嵌套子线≤5条（红线❌4）
- [ ] 子线咬合用因果推导
