# Step 3 - 故事引擎与候选竞技

把案例消化结果、项目设定和角色行动放进当前状态里运行。先生成事件矿池，再铸造故事引擎，再让至少 3 套互斥候选方案竞技。不要把第一套能跑的方案包装成唯一答案。

## 0. 输出路径

```text
卷纲/运行/幕XXX-03-候选竞技.md
```

候选竞技文件必须独立落盘，方便用户看到：事件从哪里来、怎么碰撞、每套方案的引擎是什么，为什么赢，为什么输。

## 1. 状态契约

所有候选方案共享同一开局硬状态，不允许通过偷偷改起点降低难度：

- 主角当前位置、资源、能力、伤势、身份风险。
- 敌人位置、目的、误判、行动速度。
- 环境压力、制度压力、资源压力。
- 金手指可用功能、限制、代价、反馈方式。
- 已有承诺、伏笔、未解决问题。

## 2. 事件源发散

先产出事件矿池，禁止直接写方案。

| --- | --- | --- | --- | --- | --- |

### 2.2 逻辑强制发散事件源

按以下维度各生成至少 1 个事件原料：

| 维度 | 生成问题 | 事件原料 |
| --- | --- | --- |
| 相似 | 与案例相似功能但换成本项目语境，会发生什么？ |  |
| 相反 | 如果反着写，主角不按案例路径解决，会发生什么？ |  |
| 盲区 | 当前方案没覆盖但设定必然会带来的压力是什么？ |  |
| 升级 | 当前压力解决后，下一层更近/更大的压力是什么？ |  |
| 代价 | 主角赢了要付出什么会影响下一章选择？ |  |
| 资源 | 本章能到账的资源如何改变下一章行动？ |  |
| 误判 | 敌人基于合理情报会怎么误判主角？ |  |
| 制度 | 社会/组织/规则如何把私人冲突放大？ |  |
| 金手指限制 | 面板/能力不能解决什么，反而制造什么麻烦？ |  |

### 2.3 角色视角事件池

让本幕重要角色/势力各自产生事件。每个角色必须像活人一样有目标、信息差和主动动作。

| 角色/势力 | 当前目标 | 掌握信息 | 误判 | 主动动作 | 制造事件 | 与主角碰撞点 |
| --- | --- | --- | --- | --- | --- | --- |

至少包含：

- 主角。
- 核心敌人或主压力源。
- 情感锚点角色。
- 资源/信息中介。
- 隐藏势力或长线债务代表。

如果某角色不能提供主动事件，只能提供情绪或设定说明，不允许把它列为本幕“重要角色”。

## 3. 事件碰撞与筛选

把三路事件源放进碰撞表：

| 事件ID | 来源 | 类型：相似/相反/盲区/强因果/诱饵/可融合 | 推动子线 | 可数收益/损失 | 追读价值 | 处理：采用/融合/淘汰 |
| --- | --- | --- | --- | --- | --- | --- |

筛选规则：

- 至少保留 12 个候选事件。
- 至少有 3 个相反/互斥事件，用来生成不同候选方案。
- 至少有 3 个角色主动制造的事件。
- 至少有 3 个可融合事件，每个融合事件要同时推进 2 条以上子线。
- 淘汰所有不改变压力、收益、关系或信息格局的诱饵事件。

## 4. 故事引擎铸造

先写“本幕必须解决的故事动力学”：

| 引擎 | 本幕答案 |
| --- | --- |
| 压力引擎 |  |
| 收益引擎 |  |
| 主角操作引擎 |  |
| 敌人误判引擎 |  |
| 情绪/爽点引擎 |  |
| 长线债务引擎 |  |

然后生成 3 套候选，每套都要重写自己的引擎答案。候选之间必须互斥。

### 互斥标准

A/B/C 三套方案至少在以下 5 项中有 3 项不同：

- 第一章外部压力来源。
- 主角第一主动动作。
- 第一笔可数收益。
- 敌人误判链。
- 章末钩子类型。
- 本幕终点收益。
- 情绪曲线主味型。

如果 B/C 只是 A 的节奏微调，视为无效，重写。

## 5. 候选方案要求

每套候选必须写完整幕运行链：

- 单元开局。
- 主角第一主动动作。
- 第一个可数收益。
- 第一次误判或认知差。
- 准备链：主角如何在弱势下获得“能赢的理由”。
- 敌人行动链：敌人为什么会来、为什么会误判、为什么会输。
- 阻力升级。
- 设定显影如何服务行动。
- 章末钩子如何推动下一章。
- 单元终点状态。
- 幕章节预算：默认 10-15 章，若少于 8 章必须说明是试跑压缩。

每套候选必须声明“使用了哪些事件矿池ID”。如果候选方案不引用事件矿池，视为凭空编造，重写。

## 6. 逐章运行日志

每章至少写：

- 物理坐标：时间、地点、在场人物、可用物件、资源状态。
- 开章状态。
- 触发事件。
- 主角主动选择。
- 事件拍点：5-8 个可直接扩写成正文的动作/信息/对白/反应节点。
- 准备链或代价链：主角赢/活下来的依据，或为此付出的明确代价。
- 敌人误判链：敌人基于什么信息做错判断。
- 阻力升级。
- 爽点落点。
- 爽点外显方式：本章用面板、数字、敌人反应、旁人反应、资源到账或身份变化中的哪一种让读者“当场爽到”。
- 可数收益或明确损失。
- 设定显影。
- 章末新压力。
- 案例证据引用：引用 Step 2 的 `S/J/T` 证据 ID。
- 事件矿池引用：引用本步骤的 `L-E/逻辑/角色事件ID`。

如果一章写不出 5 个事件拍点，说明它只是摘要，不是运行日志，必须重跑该章。

## 7. 子线咬合

每套方案必须给出 3-6 条嵌套子线，并标明每章推进：

| 子线 | 类型 | ch001 | ch002 | ch003 | ch004 | 状态 |
| --- | --- | --- | --- | --- | --- | --- |
| 主角生存线 | 主线 |  |  |  |  | 闭合/延伸 |

至少包含：

- 主压力线。
- 金手指成长线。
- 关系/情感/人性锚点线。
- 设定揭示或制度压迫线。
- 敌对势力线。

### 淘汰要求

- 必须淘汰至少 1 套。
- 每个被淘汰方案必须写“致命缺陷”，不是“略弱”。
- 胜出方案可以吸收败者局部强点，但最多吸收 2 个，且必须说明不会破坏主因果链。

## 9. 融合规则

胜者合成不是堆料。融合败者事件时必须满足：

- 融合后能增强主压力或主收益。
- 融合后不改变胜者的主因果链。
- 融合事件必须有明确落点章节。
- 最多融合 2 个败者强点。
- 被融合事件必须重新进入逐章运行日志，而不是只写在总结里。

## 10. 仿真原则

- 让人物目标、世界规则、组织利益、金手指限制、环境压力互相撞击。
- 设定必须通过行动显影，不能只解释。
- 每章至少有一个局势变化。
- 主角可以弱，但不能只被动挨打。
- 反杀必须有准备链，不能让敌人为了被杀而降智上门。
- 可数收益必须改变下一章的选择空间，否则只是装饰性奖励。
- 不能把爽点只写成氛围或设定理解。至少一半章节要有外显反馈：数字、资源、敌人反应、系统提示、地位变化。
- 不要用“复盘设定”当章节核心。设定信息必须被危机、交易、战斗、误判、收益或选择逼出来。
- 不要让日常缓冲吞掉压力。日常必须产生情感收益或新威胁感知。
