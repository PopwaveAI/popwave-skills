# Step 4 - 审计与压缩

先审计，后压缩。不要把第一套能跑的方案直接当幕纲。

## 0. 输出路径

```text
卷纲/运行/幕XXX-04-审计与压缩说明.md
卷纲/幕纲-幕XXX-{标题}.md
```

审计说明和最终幕纲分开保存。审计说明用于复盘，最终幕纲用于 create。

## 1. 反证式爽文审计

逐套方案打分。每项必须写“证据”和“反证问题”，不能只打 ✅。

| 项 | 通过标准 | 常见反证 |
| --- | --- | --- |
| 主角主动 | 每章都有主角选择推动事件 | 主角只是醒来、发现、被告知、被追 |
| 压力升级 | 解决一个问题后出现更大或更近的压力 | 压力只在嘴上升级，行动没有变化 |
| 可数收益 | 经验、资源、线索、身份、技能、地盘、关系至少一种明确到账 | 奖励不改变下一章选择空间 |
| 信息差 | 主角、读者、敌人之间存在可利用认知差 | 信息差只解释设定，没有改变行动 |
| 敌人误判 | 敌人基于合理信息做错判断 | 敌人无理由降智或为了送经验上门 |
| 准备链 | 主角弱势取胜有前置准备、环境利用或代价 | 反杀像作者安排，不像主角挣出来 |
| 场面密度 | 每章有 5-8 个可正文扩写的事件拍点 | 章节只有功能摘要 |
| 子线咬合 | 主线、金手指、设定、敌对、情感至少 3 条互相推动 | 各线并列罗列，没有互相造成事件 |
| 设定行动化 | 设定改变行动路径，而不是背景说明 | 设定只在“设定显影”里解释 |
| 爽点外显 | 关键爽点有面板/数字/敌人反应/资源到账/身份变化 | 爽点只靠氛围、隐喻或读者慢慢品 |
| 章末追读 | 每章结尾有下一步必须看的理由 | 钩子和本章行动无因果关系 |
| 长线债务 | 本单元给后续留下清晰承诺或威胁 | 债务只是名词，没有触发条件 |
| 案例证据落地 | Step 2 的关键 `S/J/T` 证据在章节中有对应事件 | 案例只出现在“案例消化摘要” |

低于 9 项通过的方案淘汰。任一项出现强反证时，必须修复或重跑 Step 3。若所有方案都失败，重跑 Step 3。

## 2. 选胜出方案

胜出方案不是“最复杂”，而是：

- 主角行动最清楚。
- 爽点闭环最硬。
- 设定显影最自然。
- 后续生长空间最大。

可以合并其他方案的局部强点，但必须保持因果链单一清晰。

## 2.5 用户确认

如果 Step 1 尚未获得用户确认，不允许输出最终幕纲。先输出草案和待确认项。

## 3. 幕纲压缩

把胜出方案压缩为：

- 状态契约。
- 案例消化摘要。
- 证据绑定清单。
- 物理坐标。
- 嵌套子线。
- 章节运行日志。
- 设定账本增量。
- Create 交接。
- 下一单元输入。

压缩不是删细节，而是删不能指导正文的解释。

## 4. 逆向卡对齐门槛

最终幕纲必须至少包含这些拆书卡级别的结构：

- 物理坐标：时间、地点、人物、可用物件、资源状态。
- 情绪/爽点曲线：每章情绪基调与峰值。
- 嵌套子线：3-6 条，标注闭合/延伸。
- 章节事件拍点：每章 5-8 个。
- 可迁移证据落地：案例证据 ID 对应到章节。
- 可数收益/明确损失：每章至少一个。
- 爽点外显：关键爽点不能只写成感受或设定理解。
- 敌人误判与准备链：高潮章必须有。

缺任一项，不输出最终幕纲，回到对应步骤补齐。
