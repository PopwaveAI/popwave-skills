# Step 4 - 审计与压缩

先审计，后压缩。不要把第一套能跑的方案直接当幕纲。

## 0. 输出路径

```text
卷纲/运行/幕XXX-04-审计与压缩说明.md
卷纲/幕纲-幕XXX-{标题}.md
卷纲/运行/幕XXX-产物索引.md
```

审计说明和最终幕纲分开保存。审计说明用于复盘，最终幕纲用于 create。

如果质量门未通过，不得输出“最终幕纲”，只能输出：

```text
卷纲/幕纲-幕XXX-{标题}-草案-不可写正文.md
```

并在产物索引里把状态标为“草案/不通过”。

## 1. 候选竞技复核

先读取 `卷纲/运行/幕XXX-03-候选竞技.md`，确认：

- 至少有 3 套互斥候选方案。
- 有四路事件源或明确缺席说明：library 事件源、外部类型母版事件源、逻辑强制发散事件源、角色视角事件池。
- 有事件碰撞与筛选表，并淘汰诱饵事件。
- 三套候选不是同一方案的节奏微调。
- 三套候选同等颗粒度：都不是“完整A + 概览B/C”。
- 三套候选至少在主类型母版/主剧情机制、主压力、第一主动动作、收益路径、敌人误判链、幕终点收益中 4 项不同。
- 每套候选都有六个故事引擎。
- 每套候选引用事件矿池 ID。
- 每套候选都有独立完整文件：`03a/03b/03c`，且章节数一致。
- 竞技场评分完整。
- 至少淘汰 1 套。
- 胜出方案分数不低于 75。
- 有 `卷纲/运行/卷N-00-全书卷级尺度契约.md`、`卷纲/运行/卷N-00b-卷级设计意图契约.md` 和 `卷纲/运行/幕XXX-01-当前幕切片确认.md`，并且候选方案没有偏离当前卷承诺与本幕切片目的。
- 若采用外部母版，已通过母版本土化门；没有出现“纯XXX版”夺舍项目核心引擎。

任何一项不满足，禁止进入压缩，回到 Step 3。

## 2. 反证式爽文审计

逐套方案打分。每项必须写“证据”和“反证问题”，不能只打 ✅。

| 项 | 通过标准 | 常见反证 |
| --- | --- | --- |
| 主角主动 | 每章都有主角选择推动事件 | 主角只是醒来、发现、被告知、被追 |
| 压力升级 | 解决一个问题后出现更大或更近的压力 | 压力只在嘴上升级，行动没有变化 |
| 可数收益 | 经验、资源、线索、身份、技能、地盘、关系至少一种明确到账 | 奖励不改变下一章选择空间 |
| 信息差 | 主角、读者、敌人之间存在可利用认知差 | 信息差只解释设定，没有改变行动 |
| 敌人误判 | 敌人基于合理信息做错判断 | 敌人无理由降智或为了送经验上门 |
| 准备链 | 主角弱势取胜有前置准备、环境利用或代价 | 反杀像作者安排，不像主角挣出来 |
| 场面密度 | 每章有 5-8 个可正文扩写的事件拍点 | 章节只有功能摘要 |
| 子线咬合 | 主线、金手指、设定、敌对、情感至少 3 条互相推动 | 各线并列罗列，没有互相造成事件 |
| 设定行动化 | 设定改变行动路径，而不是背景说明 | 设定只在“设定显影”里解释 |
| 爽点外显 | 关键爽点有面板/数字/敌人反应/资源到账/身份变化 | 爽点只靠氛围、隐喻或读者慢慢品 |
| 章末追读 | 每章结尾有下一步必须看的理由 | 钩子和本章行动无因果关系 |
| 长线债务 | 本单元给后续留下清晰承诺或威胁 | 债务只是名词，没有触发条件 |
| 案例证据落地 | Step 2 的关键 `S/J/T` 证据在章节中有对应事件 | 案例只出现在“案例消化摘要” |
| 故事引擎有效 | 压力/收益/操作/误判/情绪/长线六引擎互相咬合 | 六引擎只是口号，章节仍靠作者推着走 |
| 事件源有效 | 章节事件来自 library/外部母版/逻辑/角色四路事件源碰撞 | 事件仍是 agent 凭空顺剧情 |
| 多角色主动 | 重要角色都能制造事件，而不是只等待主角触发 | 角色是设定板或工具人 |
| 外部母版有效 | 影视/美剧/漫画/游戏等母版被抽取成类型机制并进入事件矿池或淘汰说明 | 外部作品只在对话里点评，没有影响方案 |
| 设计意图一致 | 方案同时服务卷级设计意图、当前幕切片功能位和不可牺牲项 | 方案好看但忘了是在写本书、本卷、本幕 |
| 母版本土化 | 外部母版增强网文爽点、主角成长、金手指反馈和长篇压力 | 外部母版夺舍，变成“纯某作品版” |
| 候选同等颗粒度 | A/B/C 均有完整逐章运行日志、拍点、误判链、收益、证据引用 | A 是完整稿，B/C 只是概览或差异标注 |
| 候选互斥度 | 三套方案换了故事引擎或类型机制，而不是同一主线的侧重点变化 | 三套共享同一首章、中段关键战、幕终点 |

低于 9 项通过的方案淘汰。任一项出现强反证时，必须修复或重跑 Step 3。若所有方案都失败，重跑 Step 3。

若“设计意图一致”“母版本土化”“候选同等颗粒度”或“候选互斥度”不通过，不允许靠修辞解释通过，必须回到 Step 0/Step 3 补齐或重跑。

## 3. 选胜出方案

胜出方案不是“最复杂”，而是：

- 主角行动最清楚。
- 爽点闭环最硬。
- 设定显影最自然。
- 后续生长空间最大。

可以合并其他方案的局部强点，但必须保持因果链单一清晰。

## 3.5 用户确认

如果 Step 1 尚未获得用户确认，不允许输出最终幕纲。先输出草案和待确认项。

## 4. 幕纲压缩

把胜出方案压缩为：

- 状态契约。
- 案例消化摘要。
- 证据绑定清单。
- 物理坐标。
- 嵌套子线。
- 章节运行日志。
- 设定账本增量。
- Create 交接。
- 下一单元输入。

压缩不是删细节，而是删不能指导正文的解释。

## 4.5 产物索引

每次结束前必须生成 `卷纲/运行/幕XXX-产物索引.md`。索引要告诉用户：

- 哪个文件是唯一可用稿。
- 哪些只是过程文件。
- 哪些文件不能给 create。
- 最终稿是否通过质量门。
- 如果不通过，缺哪几项，下一步应该补什么。

禁止只把文件散落在 `卷纲/运行/` 后让用户自己判断。

## 5. 逆向卡对齐门槛

最终幕纲必须至少包含这些拆书卡级别的结构：

- 物理坐标：时间、地点、人物、可用物件、资源状态。
- 卷级设计意图与当前幕切片摘要：当前卷读者承诺、主角变化、禁区、当前幕功能位、不可牺牲项、外部母版边界。
- 情绪/爽点曲线：每章情绪基调与峰值。
- 嵌套子线：3-6 条，标注闭合/延伸。
- 章节事件拍点：每章 5-8 个。
- 可迁移证据落地：案例证据 ID 对应到章节。
- 可数收益/明确损失：每章至少一个。
- 爽点外显：关键爽点不能只写成感受或设定理解。
- 敌人误判与准备链：高潮章必须有。
- 单章导演意图：每章要有“本章要让读者获得什么感受/爽点”的明确指令。
- 单章状态快照：每章开头必须知道人物、地点、资源、压力和未兑现承诺。
- 设定引用：每章必须列出需要 create 读取的设定文件或写明“无新增设定依赖”。
- Create 交接：每章必须有可交给正文写作的不可改事件和自由发挥空间。
- 候选竞技摘要：写明三套方案的核心差异、淘汰理由、胜出理由。
- 故事引擎摘要：保留胜出方案六引擎，供 create 和 arc 判断剧情是否跑偏。
- 事件矿池摘要：保留采用/融合/淘汰的关键事件，说明事件来自哪一路。
- 外部类型母版摘要：若采用影视/美剧/漫画/游戏母版，说明迁移了什么类型机制、如何本土化、哪些不可照搬项已避开。
- 角色主动事件表：重要角色/势力至少各有一个主动动作落入章节。

缺任一项，不输出最终幕纲，回到对应步骤补齐。
