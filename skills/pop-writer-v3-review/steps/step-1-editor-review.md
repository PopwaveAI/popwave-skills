# Step 1 - Editor Review

目的：像网文编辑和真实读者一样读正文，判断这段文字是否好看、爽不爽、像不像 AI、有没有 bug/OOC，并定位问题来自哪个上游环节。

## 资料读取

必须读取：

- 待审正文原文。

按需读取：

- 文风DNA：检查文风贴合和 AI 味。
- 幕纲/章级施工卡/章节运行日志：检查 create 是否按施工卡执行。
- 角色储备池/本卷选角表/角色锚点：检查 OOC。
- 设定账本/世界宪法/动态升级表/金手指：检查规则 bug。

不要为了审一章正文加载全项目资料。先按问题域取最小资料，再追加。

## 子 Agent 审读

若当前环境支持子 agent，主审 agent 发起独立审读，不泄露自己的判断。

### 读者体感审稿员

只从正文阅读体验出发，回答：

- 哪一段最像 AI 写的，为什么。
- 哪一段最无聊或最假紧张，为什么。
- 哪个爽点有效，哪个爽点没打出来。
- 读完是否想看下一章，卡在哪里。

### 连续性审稿员

对照必要上游资料，回答：

- 有没有设定 bug、数值 bug、时间线 bug、信息差 bug。
- 有没有 OOC：欲望、底线、反应模式、口吻是否偏。
- create 有没有漏掉 plot 施工卡要求。
- 问题应该回 create、revise、plot、world，还是只需用户确认。

如果不能使用子 agent，主审 agent 也必须分两遍审：第一遍只做读者体感，第二遍才查资料。

## 问题分级

- S级：正文不可用。核心事件、爽点、角色或设定严重跑偏，建议返工。
- A级：影响阅读快感。需要重写段落或回 create 补动作/反馈/压力。
- B级：局部瑕疵。可由 revise 修句子、节奏、衔接。
- C级：记录观察。不急于处理，只作为后续注意项。

## 归因

每个 A/S 问题都必须归因：

- create 执行问题：施工卡明确，但正文没写出来。
- revise 可修问题：句子、节奏、衔接、信息密度问题。
- plot 问题：施工卡本身平庸、缺爽点、缺外部燃料、角色功能不清。
- world 问题：设定浅、角色锚点弱、金手指或势力规则支撑不足。
- 输入缺口：用户目标或参考资料不足，需补信息。

## 输出

按模板输出：

- 总判断。
- 最影响阅读的 3 个问题。
- AI味诊断。
- 爽感诊断。
- Bug/OOC 诊断。
- 可保留资产。
- 处理建议。
- 下一章修正规则。

## 下一章修正规则

review 不是只做事后吐槽。若用户后续继续 create，报告末尾必须给出一组可直接交给 create 的规则：

- 下章必须保留的有效资产。
- 下章必须避免的写法偏移。
- 下章必须加强的爽点反馈。
- 下章必须检查的 bug/OOC 风险。
- 若发现章卡吞并、字数不足或严肃文学偏移，明确是回 plot 补施工卡，还是 create 按现有卡继续。

这些规则必须短、可执行、可复制到 create 输入包；不要写成长篇点评。
