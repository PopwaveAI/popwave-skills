# PRD 结构模板

以下为通用 PRD 结构，生成时根据产品类型调整各章节侧重点。
各章节具有逻辑递进关系：WHY → WHO → HOW → WHAT → UI → DATA → QUALITY

自由扩展原则：以下为最小必要章节。如产品涉及权限体系、多角色协作、合规要求等，
可自由追加相应章节（如「权限设计」「合规与安全」），不必拘泥于此模板。

---

## 1. 产品概述

回答：为什么要做这个产品？

### 1.1 业务背景
描述行业现状、市场趋势、用户痛点。引用真实数据支撑观点。

### 1.2 产品定位与目标
一句话定位 + 3-5 个可量化核心目标。

### 1.3 产品范围
V1.0 包含 / 不包含 清单。

---

## 2. 用户角色与场景

回答：谁来用，怎么用？

### 2.1 角色定义
3-5 类用户角色，含职责描述和核心关注点。

### 2.2 核心用户场景
3 个具体场景，用叙事方式描述。格式：「[角色] 在 [场景] 中，通过 [操作] 达成了 [目标]」。

---

## 3. 核心业务流程

回答：用户完成一件事经过哪些步骤？

如果产品有显式工作流（如审批、合作、交易），从用户视角描述完整的端到端流程。
如果产品是纯内容型或无显式流程（如数据看板），此章可改为「核心用户路径」或跳过。

### 3.1 主流程总览
ASCII 流程图，展示从起点到终点的所有阶段。

### 3.2 各阶段说明
表格：阶段名称 / 关键角色 / 输入 / 关键动作 / 输出

### 3.3 状态流转规则
谁 → 什么条件 → 可以推进到什么阶段 → 超时/异常处理

---

## 4. 产品架构

回答：系统分几大块？

### 4.1 功能模块总览
树状结构展示所有功能模块及其子模块。

### 4.2 系统逻辑架构
分层架构图：展示层 → 业务服务层 → 数据服务层 → 基础设施。

---

## 5. 功能详述

回答：每个模块具体做什么？

针对第 4 章的每个功能模块展开：
- 功能描述
- 交互要求（筛选条件、操作按钮、排序规则等）
- 页面结构（如有对应的 UI 页面）
- 关键字段/配置项

> 注意：如果某个功能会在第 6 章有独立页面设计，此处只需描述功能逻辑，
> 不要在此处重复页面的布局和交互细节。

---

## 6. 页面交互设计

回答：用户看到什么界面，怎么操作？

本章将第 5 章的功能逻辑转化为可视化的页面设计。
是生成 Demo HTML 的直接输入。

### 6.1 页面路由图
ASCII 图展示页面间的跳转关系。标注哪些跳转是新页面、哪些是 Tab 切换。

### 6.2 页面线稿
每个核心页面用 ASCII 网格描述布局：
- 顶部区域（标题、操作按钮、筛选栏）
- 主体区域（表格/卡片/图表 的网格布局）
- 底部/侧边区域（分页、统计等）

### 6.3 组件-状态矩阵
表格列：页面 / 组件 / 空态 / 正常态 / 异常态 / 触发条件

### 6.4 关键交互说明
筛选联动、Tab 切换、拖拽排序、弹窗触发等交互的规则描述。

---

## 7. 数据模型

回答：系统需要存什么数据？数据之间什么关系？

### 7.1 核心实体关系
ER 图（ASCII 方式），展示核心实体及其关联关系。

### 7.2 关键数据字段
每张核心表的字段定义：字段名 / 类型 / 必填 / 说明

### 7.3 数据契约（Demo 用）
TypeScript 风格的接口定义，描述 Demo 中使用的 JS 数据结构。
这是 页面交互设计 → Demo 填充 之间的桥梁。

```typescript
// 示例：列表项接口
interface ListItem {
  id: number; name: string; platform: string; category: string;
  score: number; riskLevel: string; status: string;
}
// 示例：详情接口
interface DetailData {
  id: number; name: string; grade: string;
  scores: { spread: number; content: number; /* ... */ };
  risks: { sev: string; date: string; title: string; detail: string; status: string }[];
  campaigns: { brand: string; year: string; result: string; roi: string }[];
}
```

---

## 8. 非功能需求

回答：系统要多快、多安全、多稳？

### 8.1 性能要求
首屏加载时间、列表查询响应时间、并发用户数等。

### 8.2 安全性要求
身份认证、权限校验、数据加密、操作日志。

### 8.3 可用性要求
系统可用性、备份策略、维护窗口。

### 8.4 兼容性要求
浏览器、分辨率、移动端支持。

### 8.5 扩展性要求
配置化能力、平台/数据源接入灵活性。

---

## 附录

### 术语表
产品相关的领域术语解释。

### 竞品参考
同类产品/平台及差异化定位。

### 待确认事项
技术可行性、对接方案、数据迁移等需进一步确认的事项。

---

> 结构设计原则：1章→2章→3章 建立用户视角（WHY→WHO→HOW），
> 4章→5章→6章 建立系统视角（WHAT→DETAIL→UI），
> 7章→8章 建立技术视角（DATA→QUALITY）。
> 各章自由添加/删除/跳过，核心逻辑链不断即可。
