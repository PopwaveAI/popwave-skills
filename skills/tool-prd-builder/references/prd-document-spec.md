# PRD 文档规范指南

> 参考：prd/整理-全量文档整合报告.md §第三部分
> 所有新 PRD 应遵循此规范，确保可读性、可审查性、可追溯性。

---

## 1. 首屏元数据块

每份 PRD 必须在文档开头包含以下元数据块：

```markdown
> 版本：v{主}.{次} | 日期：{YYYY-MM-DD}
> 状态：{草案/方案讨论/可执行/已实施}
> 分类：{规范标准/管线架构/问题诊断/参考资料}
> 关联：[关联文档名](相对路径) | 飞书原文：[链接](url)（如有）
```

### 状态定义

| 状态 | 含义 | 适用场景 |
|:-----|:-----|:---------|
| 草案 | 初稿阶段，待评审 | 首次产出，尚未获得回馈 |
| 方案讨论 | 核心方案已定，细节待确认 | 在讨论方向，未锁定改动 |
| 可执行 | 方案已确认，可以动工 | 改动范围明确，有验收条件 |
| 已实施 | 改动已完成，文档归档 | 已落地 |

### 分类定义

| 分类 | 含义 | 典型内容 |
|:-----|:-----|:---------|
| 规范标准 | 定义了怎么做 | SKILL.md 写作规范、命名规则、PRD 规范 |
| 管线架构 | 定义了做什么 | 全链路依赖图、skill 拆分、上下游关系 |
| 问题诊断 | 分析和复盘了什么问题 | 根因分析、修复记录、evidence chain |
| 参考资料 | 辅助理解的素材 | 外部方法论、排查手册、案例分析 |

## 2. 核心段落模板

| 段落 | 必须写 | 可选 |
|:-----|:-------|:-----|
| 〇、一句话范围 | 3 行内说清：改什么、不动什么 | — |
| 一、背景/问题诊断 | 问题表（问题+表现+影响）或根因表（根因+症状） | 证据链引用 |
| 二、方案 | 改动点表（文件+修改类型+具体操作） | 代码示例 |
| 三、产出物规范（如果产文件） | 文件路径+内容结构+门禁 | 模板示例 |
| 四、不做的事 | ❌ 列表：明确告知不做什么 | — |
| 五、验收条件 | 表格：#+条件+验证方式 | 验收命令 |
| 附录 | — | 证据索引、关联图、用户路径示例 |

---

## 3. 证据链规范

```
### 证据 {N}：{一句话概括}
{数据/引用/会话摘录}
{分析/推演}
```

标准：
- **有数据**（代码行数、Token 数、占比、时间线）
- **有对比**（修前 vs 修后、正常 vs 异常）
- **可复现**（文件/run/session 路径）

### 问题证据表格式

| # | 问题 | 严重性 | 影响范围 | 证据 | 修复 |
|:-:|:-----|:------|:--------|:-----|:-----|
| 1 | {问题名} | P0/P1/P2 | {范围} | {证据引用} | {版本号或方案} |

---

## 4. 修复状态标记

| 标记 | 含义 |
|:----|:-----|
| ✅ 已修复（{版本号}） | 改动已落地，关闭 |
| ⏳ 待执行 | 方案已确认，未动工 |
| ❌ 不做（{原因}） | 经评审决定不做 |
| ⛔ 框架层问题（{等待什么}） | 当前层不可修，等框架改动 |

---

## 5. 交叉引用规范

- 首次提及 skill 用全名：`pop-writer-plot`
- 引用管线关系时指向写作专家全链路文件依赖图
- 引用已归档问题时指向对应的问题诊断文档
- 引用外部来源标注来源+URL

---

## 6. 可执行性自检清单

每份 PRD 提交前逐项检查：

```
□ 有元数据块（版本/日期/状态/分类）
□ 读完后知道改哪些文件、怎么改
□ 读完后知道不改哪些文件
□ 读完后知道改完后怎么验证
□ 读完后知道这个方案依赖什么前置条件
□ 不含未定义的缩写或内部术语
□ 问题诊断类有证据链
□ 修复标记使用规范符号
```
