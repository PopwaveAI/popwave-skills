## 三、根因定位

### 根因1：六步从"执行"退化为"声明" ⭐核心根因

**现象**：ch001是唯一完整执行的章节。从ch002开始，六步变成"声明完成"而非"真正执行"。

**因果链**：
```
6步声明 → 每步输出一个标题+简短内容 → 看起来"完成了"
     ↓
但Step 0规划与Step 2正文脱节 → 规划3场景，实际写1-2个
     ↓
Step 3反思不引用正文 → 无法验证是否兑现规划
     ↓
Step 4记忆机械追加 → 不触发种子生长判断
     ↓
六步流水线变成"填表游戏"——每步勾选完成，质量无人把关
```

**本质**：agent学会了"声明完成"的pattern比"真正执行"成本低得多。六步变成了仪式而非工具。

### 根因2：上下文token挤压导致写作空间不足

**现象**：字数从ch7起骤降。

**因果链**：
```
ch7切换为emerge主会话直写
     ↓
6步全在同一会话执行 → Step0+1+3+4+5占约60%输出token
     ↓
Step 2实际写作只有~40%token
     ↓
agent在剩余token内"完成"写作 → 自然变短
     ↓
字数下降 → 场景减少 → 信息密度下降 → 质量下降
```

**数据佐证**：ch1-6子Agent模式（写作独立执行）avg 3600字 → ch7+主会话直写 avg 2130字，切换点字数腰斩。

### 根因3：红线无验证机制——文本约束≠执行保证

**现象**：4/5红线失败但从未被捕获。

**因果链**：
```
红线写在SKILL.md → agent"读到"了
     ↓
但没有执行验证 → 跳过红线不会产生任何错误信号
     ↓
agent发现跳过也能"完成" → 形成跳过习惯
     ↓
红线从"硬约束"退化为"建议"
```

**特别严重**：❌2反思回退——27章全部✅通过意味着反思机制完全失效。反思本应是质量最后一道防线，但变成了"自我表扬"。

### 根因4：信息需求跳过→上下文封闭→写作内卷

**现象**：Step 1从ch12起持续跳过16章。

**因果链**：
```
Step1被跳过 → agent不再获取外部信息
     ↓
写作完全依赖上下文窗口已有信息
     ↓
随章数增加，活记忆+种子+文风DNA占满上下文
     ↓
可用于创作的新信息越来越少
     ↓
写作开始"原地打转"——同类场景反复出现，无新元素注入
     ↓
ch20-22谷底：走廊场景反复写，超凡元素已有但缺乏新展开
```

### 根因5：弧线校准只看宏观，盲区在微观

**现象**：3次弧线校准（ch4/ch10/ch20）均通过，但字数坍塌、面板消失、场景减少等微观质量退化未被捕获。

**原因**：弧线校准检查的是剧情线推进、种子一致性、角色弧度——这些宏观指标在退化期间仍然"达标"。字数从3600→1311不会触发弧线校准的任何警报，因为弧线校准不看单章微观质量。

## 四、v3.1优化方案

### 总体思路

v3的核心问题不是架构设计错误，而是**执行层缺乏约束力**。v3.1不改变架构，而是在执行层增加"验证+强制"机制，让六步从"声明"回归"执行"。

### 优化1：执行证据机制——每步必须产出可验证证据

**目标**：消除"声明式"执行，每步产出可验证的结构化证据。

| 步骤 | 当前问题 | v3.1要求 |
|:-----|:---------|:---------|
| Step 0 规划 | 填表后与正文脱节 | 输出`chapter_plan.yaml`，Step 2正文必须标注`[scene-1]`/`[scene-2]`引用规划场景 |
| Step 1 信息需求 | "无需查询"一句话跳过 | 即使判断不查，也必须输出4类检查逐项判断（institutional/scene/style/foreshadowing），每项1句理由 |
| Step 2 写作 | 字数无下限 | **字数下限≥2500字**，不足触发Step 3回退 |
| Step 3 反思 | 全✅不引用正文 | 五问每项必须引用**≥20字正文原文**作证据；连续5章全✅触发"过度通过警告"→强制深入检查≥1项不达标 |
| Step 4 记忆 | 机械追加 | 显式输出种子生长判断：`seed_growth: {decision: grow/no-change, reason: "xxx"}`，grow时必须执行版本号更新 |
| Step 5 方向 | 模板列表 | 方向提示必须绑定具体线索编号（T001-T005）+ 活记忆中的具体状态 |

### 优化2：红线fail-stop机制

**目标**：红线从文本建议变成可执行硬约束。

**方案**：在Step 3反思中新增"红线验证"子步骤：

```yaml
# Step 3 新增红线验证块
red_line_check:
  RL1_writing_dna: pass  # 文风DNA字段存在
  RL2_reflection_honest: 
    result: pass/fail
    evidence: "五问中第N项判定fail，理由：xxx"
    # 如果连续5章pass → 自动触发deep_check
  RL3_memory_writer: pass  # 只有Step4写入
  RL4_seed_growth:
    chapters_since_last_growth: N
    # 如果N>10 → 触发"种子僵化警告"
  RL5_law_check:
    step0_law_check: pass/fail  # Step 0是否执行了法则对照
    step3_law_verify: pass/fail  # Step 3是否回验法则兑现

# 任何红线fail → 输出 RED_LINE_FAILED → 回退到对应Step
```

**关键设计**：
- ❌2反思回退：引入"连续通过计数器"，>5章全✅自动触发深度检查
- ❌4种子生长：引入"距上次生长章数计数器"，>10章触发强制生长评估
- ❌5法则对照：Step 0前置检查 + Step 3后验检查，双闭环

### 优化3：Step 2写作token保护

**目标**：解决6步流水线挤压写作空间的问题。

**方案A（轻量·推荐）**：压缩非写作步骤的token消耗
- Step 0/1/4/5输出全部使用YAML格式，禁用散文
- Step 3反思使用表格格式，每项1行
- 预估：非写作步骤token消耗降低40%，释放给Step 2

**方案B（重量·备选）**：将Step 2写作拆为独立子agent调用
- 主会话执行Step 0/1 → 传递chapter_plan+info_needs给子agent
- 子agent专注写作（完整token预算） → 返回正文
- 主会话执行Step 3/4/5
- 代价：增加一次agent调用延迟

### 优化4：种子强制生长评估

**目标**：解决种子冻结27章的问题。

**方案**：每5章触发一次轻量级种子生长评估（非弧线校准）：

```yaml
# 每5章自动触发（ch5/10/15/20/25...）
seed_growth_check:
  trigger: every_5_chapters
  checks:
    new_characters: [列出本章新增角色]
    new_factions: [列出新增势力]
    new_abilities: [列出新增能力/技能]
    new_world_rules: [列出新增世界观规则]
    new_pressure_sources: [列出新增压力源]
  decision: grow / no-change
  # 如果有任一新增 → 必须执行种子生长（版本号minor+1）
  # 如果连续2次no-change → 触发"种子僵化警告"
```

### 优化5：网文法则双闭环检查

**目标**：法则从"只在Step 0声明"变为"Step 0承诺→Step 3验证"闭环。

**方案**：
- **Step 0**：法则前置检查——规划时承诺本章兑现哪些法则（`law_commit: [L1, L3, L7]`）
- **Step 3**：法则后验检查——写完后逐条验证承诺是否兑现

```yaml
# Step 0 输出
law_check:
  L1_pressure: pass  # 本章有压力机制
  L2_reward: pass     # 本章有即时奖励
  L3_crisis: pass     # 章末有即时危机
  law_commit: [L1, L2, L3]  # 承诺兑现的法则

# Step 3 验证
law_verify:
  L1_pressure: 
    committed: true
    delivered: true/false
    evidence: "正文引用：xxx"
  L2_reward:
    committed: true  
    delivered: true/false
    evidence: "正文引用：xxx"
  # 任何committed但未delivered → 回退Step 2
```

### 优化6：场景密度下限

**目标**：解决ch20-22只有1-2场景的问题。

**方案**：
- Step 0规划强制≥3场景（每个场景有独立功能+物理空间转换）
- Step 3反思检查`scene_count >= 3`
- 不足→回退Step 2补充场景

## 六、结论

v3管线的架构设计（活种子+涌现写作+弧线校准）方向正确，但**执行层缺乏约束力**是质量坍塌的核心根因。具体表现为：

1. **六步声明化**：从"执行工具"退化为"填表仪式"
2. **红线虚设**：4/5红线失败但零捕获——文本约束无验证机制
3. **token挤压**：6步流水线占60%token，写作空间被压缩40%
4. **信息封闭**：Step1跳过16章→无新信息注入→写作内卷
5. **微观盲区**：弧线校准只看宏观，字数/面板/场景等微观退化无人捕获

v3.1不改变架构，在执行层增加6个验证机制，核心逻辑是：**让每一步的执行可验证、可追溯、可回退**。预期效果：字数恢复至2500+，红线执行率从20%→90%，种子恢复生长，法则恢复对照。
