# 案例消化卡标准格式

> 用途：writer skill 读 library 案例后，必须把“看过”转成“可迁移的写作决策”。

---

## 标准输出

```markdown
## 案例消化：{案例名}

- 来源路径：{path}
- 写作任务：{开局/成长/冲突/...}
- 原书任务：{原案例解决的问题}
- 本项目映射：{当前项目要解决的问题}

### 1. 可迁移结构
- 压力源：
- 主角主动判断：
- 行动链：
- 收益闭环：
- 代价/新压力：

### 2. 爽点证据
- 信息差：
- 主角能动性：
- 可数收益：
- 情绪释放点：
- 章末追读钩子：

### 3. 内容肌理
- 空间/物件：
- 社会生态：
- 数字/资源锚点：
- 角色关系锚点：
- 语言/画面质感：

### 4. 不可照搬项
- 原书专属设定：
- 原书专属角色关系：
- 原书专属文化梗：
- 篇幅/节奏限制：

### 5. 改写决策
- 我们保留：
- 我们替换：
- 我们强化：
- 我们删除：
```

---

## 合格标准

合格的案例消化必须能回答：

1. 为什么这个案例爽？
2. 爽点在文本里靠什么证据成立？
3. 哪些结构能迁移，哪些内容不能迁移？
4. 迁移到本项目后，主角会做什么主动选择？
5. 单元结束时，读者得到什么明确收益或新期待？

---

## 不合格示例

```markdown
这个案例结构是“开局绝境→系统觉醒→解决危机”，可以参考。
```

问题：只有结构名，没有爽点证据、内容肌理、不可照搬项和本项目映射。

---

## 压缩版

当 token 紧张时，允许输出压缩版：

```markdown
- 案例：{path}
- 学什么：{1-2句}
- 爽点证据：{3条}
- 可迁移：{3条}
- 不可照搬：{2条}
- 本项目改写：{3条}
```
