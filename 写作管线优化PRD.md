## 2. 根因分析（skill 文件级）

### 2.1 设计包过度设计

**根因文件**：`pop-writer-chapter/templates/fact-skeleton.md`

模板要求每个事件填写 **16 个字段**（地点/参与角色/scene/POV/关键对白/感官锚点/事件内容/驱动角色/角色决策/决策代价/不可逆后果/决策一致性/情绪目标/冲突层次/所属线/信息释放/估计字数）。v2.9.0 的 DQTC 质量层又增加了 4 个决策质量字段。12 事件 × 20 字段 = **240 个字段条目**。

prose 实际消费的字段只有 7 个（scene/POV/关键对白/感官锚点/事件内容/情绪目标/估计字数）。其余 13 个字段（驱动角色/角色决策/决策代价/不可逆后果/决策一致性/冲突层次/所属线/信息释放 + 润色层的套路表/爽点机制表/契诃夫枪表/对白语气分析/信息释放清单）是 chapter 自身的过程产物或 plot 的溯源要求，**prose 不消费**。

设计包还包含 8 个子表（套路表/爽点标注/情绪弧线/爽点机制表/章末钩子/契诃夫枪表/调味空间/对白语气分析）+ 信息释放落地清单 + 跨章弧线 + 质量自检表。设计包承担了 **10+ 种功能**，违反单一职责。

### 2.2 管线结构性缺陷的 5 个根因

**根因 1：事件链模式导致流水账**

根因文件：`pop-writer-chapter/steps/step-2-event-chain.md` + `pop-writer-prose/steps/step-1-render.md`

chapter 的 step-2 要求"逐事件设计骨架"——每个事件独立填写 16 个字段。prose 的 step-1 要求"逐事件渲染"——每个事件走 A→B→C→D 流程（理解意图→选POV→写段落→过渡）。

这导致正文变成了**事件清单的扩写**：每个事件写 200 字，12 个事件拼成 2400 字。事件之间只有 1-2 句过渡，没有场景内部的有机流动。

对标书的写法是**场景驱动**——一个场景自然包含多个事件（醒来→发现老约翰→老约翰说话→老约翰死亡→系统装载→搜尸→获得技能），事件在场景中自然流淌。

**根因 2：文风 DNA 从未设置**

根因文件：`expert-writer/references/project/master-control.tpl.md`

项目总控模板的"知识库路径"区块初始为 `{pop-trope-library绝对路径}` 占位符。world 阶段获取文风库的流程依赖此路径。路径未解析 → DNA 从未加载 → prose 输出 A/B 选项但用户未响应 → 默认 AI 文风。

prose SKILL.md 红线写的是"文风DNA已检查——缺失时输出 A/B 选项给用户，不自行降级"。但实际执行中 agent 忽略了红线直接降级渲染。**根本问题是路径未解析，而非 agent 忽略红线**。

**根因 3：调味禁忌过度限制**

根因文件：`pop-writer-chapter/templates/fact-skeleton.md` 第 152 行

模板的"调味空间"表有"调味禁忌"字段（"本章绝对不能出现的元素"），step-3 门禁要求"调味空间缺退回"。ch001 设计包填了 7 条禁忌，包括"主角不对任何人说话——本章不安排任何对白"。

这些禁忌的初衷是控制信息释放节奏，但叠加在一起把第一章变成了"零对白纯观察"模式。写作需要冲突、对话、行动——设计包把这些全禁了。

**根因 4：信息释放百科化**

根因文件：`pop-writer-chapter/templates/fact-skeleton.md` 第 159-160 行 + `references/info-release.md`

模板的"信息释放落地清单"要求逐项标注设定信息的释放方式和位置。ch001 设计包列了 12 项设定信息，平均每 200 字释放一个设定点。正文变成了设定展示的容器——每个事件的存在目的不是推进故事，而是"释放一个设定信息"。

对标书的设定是通过行动和交互自然展开的——老约翰的遗言揭示背景、搜尸获得技能展示系统机制、血量下降展示斩杀线紧迫感。设定是故事的副产品，不是故事的目的。

**根因 5：prose"不解释"导致情感空洞**

根因文件：`pop-writer-prose/steps/step-1-render.md` 第 60-61 行

```
| 不解释 | 不写「他意识到/他感到/他仿佛」 |
| 展示 | 情绪通过动作/环境/感官传递 |
```

这条规则的初衷是避免 AI 式解说，但执行过头变成了情感空洞。ch001 正文中主角看到有人死在面前，反应是"他咬了咬牙"——确实是"展示"而非"解释"，但没有情感深度。

对标书的写法是展示和内心独白交替："韦恩只感觉整个人瞬间被一股阴冷的感受包裹，仿佛有什么东西已经融入他的体内"——既有感官展示，又有内心体验，还有情绪传递。

### 2.3 plot 层 gap 的 3 个维度

**维度 1：剧情密度**

根因文件：`pop-writer-plot/steps/step-6-canvas-audit.md`

Canvas 验核标准是"每章≥1线 payoff≥中"+"连续3章空铺垫退回"。没有事件密度量化基准。L2-001 的七阶段设计明确写了"阶段①铺垫（ch1-3）：爽点密度：低——本章段不设反杀/爽点，专注建立绝望感"。

这是文学小说的节奏策略，不是网文的节奏策略。对标书 ch1 有 13 个事件+完整情绪闭环（恐惧→悲伤→震惊→紧张→得意→绝望→野蛮→满足→好奇）。

**维度 2：危机丰富度**

根因文件：`pop-writer-plot/steps/step-3-plotline-docs.md`

剧情线设计依赖储备池+library，但缺乏"危机来源多样性"检查机制。本项目 ch1-3 的 7 个事件全来自"世界观揭露"一个系统。对标书 ch1 的 8 个事件来自 5 个系统（金手指/社会环境/角色关系/世界观/外部威胁）的自然交叉。

**维度 3：金手指机制**

根因文件：`pop-writer-creative/SKILL.md` + `pop-trope-library/金手指库/00-索引.md`

creative v6.1.0 有金手指库消费链，但金手指库的分类（契约型/悖论型/系统型/血脉型）偏向"面板观察"设计范式。本项目的金手指被设计为"观察面板"（标注NPC等级/弱点/阵营），而非"行动引擎"（发布任务/获技/经验/血量/战宠）。

对标书的系统是行动引擎：发布任务→主角行动→获得奖励→推动下一步。系统是剧情发动机，不是观察窗口。

### 2.4 项目总控失效

根因文件：`expert-writer/steps/step-0-init.md` + `references/project/master-control.tpl.md`

总控模板有大量 `{待填充}` 占位符（文风DNA路径/知识库路径/管线进度）。step-0-init 的初始化流程未强制解析这些路径就进入 creative。

总控失效是"放大器"非"发起者"：ch1 质量差的原因是 skill 管线的结构性缺陷，但总控失效让 ch4 之后加速恶化（agent 无进度锚点、无文风 DNA、无知识库）。

### 2.5 生态调研深度不足

根因文件：`pop-writer-creative/references/种子展开法.md` + `references/深度域研究SOP.md`

域 DNA 档案的产出格式是：叙事语法 2-3 句 + 核心机制 2-3 句 + 美学基因 1-2 句 + 经典元素库≥5 个。这个深度是对"域"的摘要，不是对"社会生态"的穿透。

对标书对北美生态的调研涉及具体社会角色群（流浪汉/收尸人/强化剂黑市/帮派/巫毒教），需要的是"社会生态图谱"——谁在什么位置、靠什么生存、彼此什么关系。现有调研产出的是"美学参考"，不是"生态结构"。

## 3. 优化方案（文件级修改清单）

### 3.1 chapter skill 优化

| 文件 | 修改内容 | 目的 |
|:-----|:--------|:-----|
| `templates/fact-skeleton.md` | 事件字段从 16 个精简到 7 个核心字段（地点/参与角色/scene/POV/关键对白/事件内容/估计字数）；DQTC 字段（驱动角色/角色决策/决策代价/不可逆后果/决策一致性）移到独立的"决策溯源附录"，prose 不加载 | 设计包从 20K 字降到 ~8K 字 |
| `templates/fact-skeleton.md` | 移除质量自检表（23 项）——改为 step-4 的内存检查，不留在交付物中 | 减少 ~2K 字冗余 |
| `templates/fact-skeleton.md` | 信息释放落地清单从"逐项标注"改为"prose 渲染时自然嵌入"——设计包只标注"本章需释放的设定项清单"，不标注释放方式和位置 | 从设定百科回归叙事 |
| `templates/fact-skeleton.md` | 调味禁忌从"禁止清单"改为"引导方向"——从"本章绝对不能出现的元素"改为"本章创作方向建议" | 释放创作空间 |
| `steps/step-2-event-chain.md` | 从"逐事件设计骨架"改为"场景流设计"——以场景为单位组织事件，场景内部允许有机流动；一个场景可包含 3-5 个事件 | 解决积木式写法 |
| `steps/step-3-polish.md` | 情绪弧线/爽点机制表/契诃夫枪表保留但精简——从 8 个子表压缩到 3 个（爽点标注/情绪弧线/章末钩子） | 减少设计包体积 |

### 3.2 prose skill 优化

| 文件 | 修改内容 | 目的 |
|:-----|:--------|:-----|
| `steps/step-1-render.md` 第 60-61 行 | "不解释"原则修正：从"禁止他意识到/他感到"改为"限制使用+允许关键情感内省"——每章允许 1-2 处内心活动，用于情感高潮点；规则改为"80%展示+20%内省" | 解决情感空洞 |
| `steps/step-1-render.md` 第 52-69 行 | 从"逐事件渲染"改为"场景流渲染"——以场景为渲染单位，场景内部连续流动，不逐事件走 A→B→C→D | 解决 PPT 式拼接 |
| `SKILL.md` 红线 | 文风 DNA 缺失时从"A/B 选项给用户"改为"硬阻塞+自动从项目总控解析路径"——DNA 缺失 = 终止，非降级 | 解决无锚点渲染 |

### 3.3 plot skill 优化

| 文件 | 修改内容 | 目的 |
|:-----|:--------|:-----|
| `steps/step-6-canvas-audit.md` | 增加事件密度量化标准："每章≥8 个事件"+"每 3 章至少 1 个完整情绪闭环" | 解决剧情密度不足 |
| `steps/step-3-plotline-docs.md` | 增加"危机来源多样性检查"——每幕的危机事件必须来自≥3 个不同系统（世界观/角色/社会/金手指/外部环境） | 解决危机来源单一 |
| `references/prd-payoff-audit.md` | 增加爽点密度基准——参考对标书 ch1 的 13 事件密度作为校准线 | 量化爽点标准 |
| L2 剧情单元卡模板 | 阶段①铺垫的"爽点密度：低"改为"爽点密度：中——每章至少 1 个小爽点" | 解决网文节奏 |

### 3.4 creative skill 优化

| 文件 | 修改内容 | 目的 |
|:-----|:--------|:-----|
| `references/种子展开法.md` | 域 DNA 档案从"2-3 句摘要"升级为"社会生态图谱"——增加"生态角色清单"（谁在什么位置/靠什么生存/彼此什么关系）+"生态与剧情的交叉点" | 解决生态深度不足 |
| `references/深度域研究SOP.md` | L4 叙事语法层增加"社会生态结构"维度——不只问"怎么讲故事"，还要问"这个社会的生存结构是什么" | 深化调研穿透 |
| `SKILL.md` 金手指设计步骤 | 增加"行动引擎检查"——金手指是否提供任务/获技/经验/血量/战宠等行动驱动机制；若金手指只有"观察/标注"功能 → 标注风险 | 解决金手指是观察工具 |

### 3.5 expert-writer 优化

| 文件 | 修改内容 | 目的 |
|:-----|:--------|:-----|
| `steps/step-0-init.md` | 强制初始化项目总控的关键路径——文风 DNA 路径/知识库路径在 creative 启动前必须解析完成，否则阻塞 | 解决总控失效 |
| `references/project/master-control.tpl.md` | 减少待填充占位符，增加自动解析逻辑（pop-trope-library 路径自动检测、文风库自动扫描） | 减少人工遗漏 |
| `steps/step-3-reflect.md` | 增加管线进度更新门禁——每章 prose 完成后必须更新总控的进度锚点 | 解决进度追踪断裂 |

## 5. 验证方法

用对标书 ch1 做复现验证：优化后管线产出的 ch1 是否达到以下标准：

| 验收标准 | 当前状态 | 目标 |
|:--------|:--------|:-----|
| AC-1：ch1 事件密度 ≥ 8 个 | 12 个（但全是观察型） | ≥ 8 个且 ≥ 3 种类型（行动/对话/发现/冲突） |
| AC-2：ch1 有完整情绪闭环 | 绝望→绝望→更绝望 | ≥ 4 种情绪转换 |
| AC-3：ch1 危机来自 ≥ 3 个系统 | 1 个（世界观揭露） | ≥ 3 个系统 |
| AC-4：ch1 有对白 | 零对白 | ≥ 2 段对白 |
| AC-5：ch1 有内心活动 | 纯外部观察 | 1-2 处情感内省 |
| AC-6：ch1 文风有辨识度 | 默认 AI 文风 | 可感知的文风特征 |
| AC-7：设计包字数 ≤ 10K | 20K | ≤ 10K |
| AC-8：金手指驱动行动 | 仅标注信息 | 至少 1 次行动驱动（获技/任务/奖励） |
