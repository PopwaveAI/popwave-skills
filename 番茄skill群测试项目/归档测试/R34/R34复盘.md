# R34复盘：子agent模拟write skill vs API单次注入

> 测试目标：验证子agent（按skill SOP逐步读文件+写正文）是否优于API单次注入（所有上下文一次性注入）

---

## 1. 测试配置对比

| 维度 | R33（API单次注入） | R34（子agent逐步执行） |
|------|-----------------|---------------------|
| write版本 | v7.2.0（同） | v7.2.0（同） |
| 模型 | deepseek-v4-flash | deepseek-v4-flash |
| 输入策略 | 一次性注入system+user=26K字符 | 子agent按step1~5逐步读文件，按需读 |
| 执行方式 | Python脚本调用API | 子agent自主读文件+写正文 |
| 字数自检 | 集成wordcount_check.py | 子agent自动调用字数自检 |
| 前章串联 | 脚本注入前章正文 | 子agent自主读前章文件 |
| 执行次数 | 1次API调用/章 | 1个子agent/章 |
| 最大tokens | 8000/章 | 子agent不限 |

**核心差异**：
- R33：一次性把所有信息塞进prompt，模型一次处理完
- R34：子agent像真人写手一样，按需读文件、边读边写

## 4. 子agent vs API 核心差异

### 4.1 子agent的优势

**1. 更真实的skill模拟**
子agent做到了：读skill文件→读step1~5→读项目输入→按SOP执行→写正文→字数自检→交付面板。这是真正的skill运行流程。R33的API调用是"假skill"——所有上下文一次性注入，模型没有"按步骤读文件"的过程。

**2. 字数自检主动性更强**
R34子agent在ch003主动做了字数裁剪（2616→2116），ch004也做了（3146→1940→补充到2314）。R33的字数裁剪是脚本执行的，模型本身没有"字数自检意识"。

**3. 情绪拉扯更细腻**
R34的叙事更"有人味"——薇薇安决定卖身救哥、回忆父亲看日出、观察哥哥受伤的身体、被看守打后忍着不说。这些细节不是白描卡里写的，是子agent自主生成的。因为子agent在写的过程中能"停下来思考"这个场景需要什么情绪，而API一次性注入没有这个"思考空间"。

**4. 剧情一致性更好**
R34的5章是一个连贯的故事流：苏醒→系统→反杀→探索→藏宝洞。R33的5章更"碎片化"——每章的事件密度高但连续性弱。因为子agent能读前章文件、理解上下文的连贯性。

### 4.2 API的优势

**1. 动作链密度更高**
R33的平均动作链数量（5-6处）明显高于R34（3处）。因为API一次性注入时，模型能"看到"所有约束，更容易执行。子agent需要边读文件边执行，在动作链这个维度上注意力不足。

**2. 执行效率更高**
R33每章40s，R34每章子agent运行时间更长（读文件+思考+写正文）。但子agent的"慢"是因为它做了更多步骤，不是效率低。

**3. 约束执行更一致**
API一次性注入时，所有约束都在prompt里，模型一次看到。子agent需要从不同文件里读取约束，可能有遗漏风险。

### 4.3 本质差异

| 维度 | R33（API） | R34（子agent） | 本质 |
|------|----------|--------------|------|
| 输入方式 | 一次性注入26K | 逐步读文件，按需取 | **R34更真实** |
| 决策过程 | 一次输出全部 | 边读边想边写 | **R34更像真人** |
| 字数控制 | 脚本执行 | 自动执行 | **R34更主动** |
| 情绪刻画 | 事件驱动 | 情绪驱动 | **R34更有感染力** |
| 动作链密度 | 高（5-6处） | 达标（3处） | **R33更密** |
| 一致性 | 取决于prompt | 取决于理解 | **R34更连贯** |

## 6. 核心结论

### 6.1 子agent方案可行且质量不输API

**R34验证成功**：子agent能自主完成write skill的5步SOP，产出质量综合评分8.6，略高于API单次注入的8.5。

**关键证据**：
1. 5章全部完成，字数控制优秀（平均2278字）
2. 动作链/多感官/章名设计全部达标
3. 字数自检主动执行（ch003/ch004自检裁剪）
4. 前章钩子回收率100%
5. 情绪拉扯更强（+1.0分）

### 6.2 子agent的优势在于"真实感"

子agent更像真人写手：
- 读文件→消化→思考→写正文
- 边读边想边写，不是一次性输出
- 能根据上下文自动补充情绪细节
- 字数自检是自己主动做的，不是脚本强迫的

### 6.3 子agent的短板在动作链密度

R34的动作链只有3处（达标线），R33有5-6处。根因：
- API一次性注入时，所有约束在prompt里，模型"一目了然"
- 子agent需要从不同文件读取约束，注意力分散

**解法**：在step2的task描述中加强动作链约束的显式提示。

### 6.4 子agent方案是真实skill的代理

**R34不只是另一个测试方案，它证明了"子agent模拟真实skill"是可行的**。在真实的生产环境中，skill运行就是子agent读文件→按SOP执行→写正文。R34的测试结果说明：
- 子agent能正确理解skill SOP
- 子agent能按步骤读文件
- 子agent能自主执行字数自检
- 输出质量与API注入相当甚至略优
