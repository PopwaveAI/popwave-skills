## □ 面板/数据窗口分类统计

**本书系统的核心设定：**
- 系统不是"金手指外挂"——它是**神格具现化的产物**，只是将主角已有的力量以数据页面的方式呈现
- 主角前世的系统也可以变成书本形式——当前游戏化界面只是潜意识里最直观的理解方式
- **关键差异**：系统不给力量，只具现化力量

**面板类型：**

| 面板类型 | 出现频率 | 出现场景 |
|:---------|:--------:|:---------|
| 属性面板（职业/属性/能力） | 每1-2章 | 升级确认/状态检查/战斗前后 |
| 变形形态数据 | 变形场景 | 展示变形后的属性变化 |
| 能力获得提示 | 获得新能力时 | ch01获得野性启蒙、ch02获得月之祝福 |
| 战斗数据 | 战斗中 | 变形状态下属性加成展示 |

**面板长度分布：**
| 类型 | 长度 | 占比 |
|:-----|:----:|:----:|
| 短（3-5行） | 核心属性/职业信息 | ~50% |
| 中（8-15行） | 形态变化数据+技能列表 | ~40% |
| 长（16+行） | 新能力完整描述 | ~10% |

## □ 面板放置规律

```
- 变形形态前: 展示变形形态属性 → 预示战力变化
- 战斗高潮时: 穿插面板提示 → 确认战斗效果
- 获得新能力时: 完整面板展示 → 确认成长
- 日常检查时: 短面板 → 状态确认

核心规则：
- 面板永远在"需要读者知道主角状态变化时"出现
- 面板放置不打断叙事节奏——它嵌入在动作描述之间
- 章节中段最常用（变形/战斗场景），章末次之（获得新能力）
```

## □ 面板与叙事融合策略

**策略A - 面板确认+叙事解释（最常用 ~65%）：**
```
示例(ch01)：
"姓名：萧伯纳。种族：人类。阵营：绝对中立。
职业：5级德鲁伊（荒野试炼【未完成】）。"

↓ 紧接叙事
"系统。萧伯纳召唤出脑海中的系统页面，这是他来到这个世界最大的助力..."
```

**策略B - 叙事中嵌入数据（次常用 ~25%）：**
```
示例(ch03变形):
"——月影巨狼形态(野性变身)
——属性：力量22(+2) 敏捷24(+2) 体质20(+2) 智力12 感知20 魅力16"

↓ 紧接动作
"在月影巨狼的形态下，萧伯纳的战斗属性...有大幅提升"
```

**策略C - 面板单独展示（少用 ~10%）：**
```
新能力获得时的完整面板展示
```

## □ 数据显示/隐藏逻辑

```
总是显示的数据:
- 职业等级（核心定位）
- 主要属性（力量/敏捷/体质/智力/感知/魅力）
- 核心能力（德鲁伊法术/变形能力）

选择性显示的数据:
- 变形形态下的属性变化（仅在变形时显示）
- 法术详情（仅在施法/学习时显示）
- buff加成（获得时显示）

始终不显示的数据:
- 经验值/升级进度（非游戏，不显示）
- 技能熟练度（不量化显示）
- 详细技能树（非游戏）

选择逻辑：只显示读者"现在需要知道"的信息，不显示游戏化数值。
```

## □ 面板技法的演变（前100章）

```
阶段 | 面板长度 | 功能 | 占章节比
----|---------|------|---------
开局(ch01-03) | 中(8-15行) | 建立系统认知 | ~20%

成长(ch20+) | 短+中 | 确认新能力/升级 | ~15%
持续 | 短 | 维持"有数据支撑"的感觉 | ~5%
```

## □ 数据流体验评估

- **数据流打断叙事还是增强沉浸？** → 增强（因为系统设定为"神格具现化"，和世界观逻辑一致）
- **哪些位置提升了阅读体验**:
  - 变形前的数据展示：让战斗有"真实感"
  - 新能力获得时的面板：确认成长的成就感
- **哪些位置降低了阅读体验**:
  - 无——面板使用很克制，频率不高
- **可借鉴的设计**:
  - "系统不赠予力量"的设定解决了系统文的核心矛盾
  - 面板长度控制得当（最长不超过15行）
  - 面板嵌入叙事中，不单独成段
- **要避免的问题**:
  - 避免面板过长（本书做得好）
  - 避免面板频繁（本书控制得当）
